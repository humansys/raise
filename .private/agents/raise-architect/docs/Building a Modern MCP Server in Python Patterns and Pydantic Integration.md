# **Building a Modern MCP Server in Python: Patterns and Pydantic Integration**

## **1\. Introduction**

The **Model Context Protocol (MCP)** is an open standard that defines a **universal interface** for AI systems to connect with external tools and data sources ([Model Context Protocol (MCP): Integrating Azure OpenAI for Enhanced Tool Integration and Prompting | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/azure-ai-services-blog/model-context-protocol-mcp-integrating-azure-openai-for-enhanced-tool-integratio/4393788#:~:text=Model%20Context%20Protocol%20serves%20as,pattern%20enables%20AI%20systems%20to)) ([Constructed Futures](https://constructedfutures.com/episodes/ai-briefing-series-expanding-ai-agent-capabilities-with-model-context-protocol-mcp#:~:text=Anthropic%27s%20Model%20Context%20Protocol%20,aware%20approach%20compared%20to)). In essence, MCP acts as a **communication bridge** between large language models (LLMs) or AI agents and the services they need to access, enabling AI assistants to interact with files, databases, APIs, etc., through a **standardized protocol**. This approach addresses the fragmentation in AI integrations – instead of bespoke adapters for each tool, any MCP-compliant client can talk to any MCP server, much like a *“USB-C for AI applications,”* providing a plug-and-play connectivity model ([Model Context Protocol (MCP): Integrating Azure OpenAI for Enhanced Tool Integration and Prompting | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/azure-ai-services-blog/model-context-protocol-mcp-integrating-azure-openai-for-enhanced-tool-integratio/4393788#:~:text=MCP%20has%20the%20potential%20to,Ultimately%2C%20MCP%20aims%20to)) ([Everything You Need to Know About the Model Context Protocol (MCP) from Anthropic | by allglenn | Mar, 2025 | Stackademic](https://medium.com/@glennlenormand/everything-you-need-to-know-about-the-model-context-protocol-mcp-from-anthropic-84acdb3c1a2f#:~:text=The%20Model%20Context%20Protocol%20,enhances%20interoperability%2C%20and%20promotes%20scalability)).

The core architecture of MCP follows a **client–server model**: an AI application (the *host/client*) issues requests to an MCP *server*, which in turn executes specific actions (called **tools**) or fetches data and returns results in a structured format ([Model Context Protocol (MCP): Integrating Azure OpenAI for Enhanced Tool Integration and Prompting | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/azure-ai-services-blog/model-context-protocol-mcp-integrating-azure-openai-for-enhanced-tool-integratio/4393788#:~:text=Model%20Context%20Protocol%20serves%20as,formatted%20results%20that%20the%20AI)) ([Constructed Futures](https://constructedfutures.com/episodes/ai-briefing-series-expanding-ai-agent-capabilities-with-model-context-protocol-mcp#:~:text=%2A%20Client,servers%20and%20manages%20client%20permissions)). This design lets AI systems preserve their core reasoning capabilities while **extending their reach** into real-world data and actions via these tool calls ([Model Context Protocol (MCP): Integrating Azure OpenAI for Enhanced Tool Integration and Prompting | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/azure-ai-services-blog/model-context-protocol-mcp-integrating-azure-openai-for-enhanced-tool-integratio/4393788#:~:text=architecture%20of%20MCP%20consists%20of,with%20external%20systems%20and%20databases)). The latest MCP specification (circa 2025\) emphasizes secure, two-way connections and context-sharing between AI and tools, aiming to transform isolated AI models into **context-aware, interoperable systems** deeply integrated with their environment ([Model Context Protocol (MCP): Integrating Azure OpenAI for Enhanced Tool Integration and Prompting | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/azure-ai-services-blog/model-context-protocol-mcp-integrating-azure-openai-for-enhanced-tool-integratio/4393788#:~:text=MCP%20has%20the%20potential%20to,Ultimately%2C%20MCP%20aims%20to)). In summary, MCP’s importance in AI-driven systems lies in **standardization** (interoperability across many tools), **dynamic context injection** (providing LLMs with real-time information and actions), and **scalability of integrations** (solving the N×M integration problem with a single N+M standard) ([Constructed Futures](https://constructedfutures.com/episodes/ai-briefing-series-expanding-ai-agent-capabilities-with-model-context-protocol-mcp#:~:text=,%28Introduction%2C%20citing%20source%203)). This research overview will explore architectural and design patterns for building a modern MCP server, focusing on a Python implementation. We pay special attention to **scalable, modular architecture** choices and how **Pydantic AI** can be leveraged for model management and context-aware interactions.

## **2\. Design Pattern Landscape**

Building a robust MCP-compliant server requires applying proven software architecture patterns to achieve scalability, maintainability, and proper propagation of context through the system. Key patterns relevant to an MCP server include the **Command Pattern**, **Context Object Pattern**, **Dependency Injection**, and a **Layered (or Hexagonal) Architecture**, among others. These patterns work in concert to decouple components, enforce clear data flow of “context” information, and support the addition of new tools or features without breaking existing functionality. We also note that an MCP server essentially behaves like an **Adapter** between the AI agent and an external service – a concept that underpins many of these patterns ([MCP (Model Context Protocol): Simply explained in 5 minutes](https://substack.com/home/post/p-159139276?utm_campaign=post&utm_medium=web#:~:text=MCP%20,a%20message%20like%20%E2%80%9CGet)). Below, we describe each major pattern and its role in an MCP server’s design.

### **Command Pattern**

The Command Pattern is a behavioral design pattern that **encapsulates a request as an object**, allowing the system to parameterize and handle requests in a uniform way ([Encapsulating Requests: The Power of the “Command Design Pattern” | by Sumonta Saha Mridul | Bootcamp | Medium](https://medium.com/design-bootcamp/encapsulating-requests-the-power-of-the-command-design-pattern-d2f42b0f9d1d#:~:text=1,the%20inverse%20of%20the%20command)). In an MCP server, each tool invocation from the AI can be treated as a *command* object – containing all the information needed to perform the action (tool name, parameters, context, etc.). This approach decouples the component that receives and queues the request (the MCP interface or dispatcher) from the component that executes the action (the tool implementation). By representing tool calls as command objects, the server gains flexibility in scheduling, logging, or even undoing actions. For instance, a “SearchLogs” command or a “RunQuery” command can be created and passed to an executor without the invoker needing to know the execution details. This yields **loose coupling** between the MCP protocol layer and the business logic of each tool – the sender (AI client) and the receiver (tool logic) are decoupled by the command interface ([Encapsulating Requests: The Power of the “Command Design Pattern” | by Sumonta Saha Mridul | Bootcamp | Medium](https://medium.com/design-bootcamp/encapsulating-requests-the-power-of-the-command-design-pattern-d2f42b0f9d1d#:~:text=1,the%20inverse%20of%20the%20command)). New tools can be added easily by defining new command classes or functions, without altering the core server dispatcher. The Command pattern also aligns with the idea of *tools as independent actions*: it treats each tool call as a first-class object, which is useful for debugging (logging a history of commands), and for potential extensions like batching or reordering requests. In summary, using the Command Pattern in an MCP server supports **extensibility (adding new tool commands easily)** and **maintainability (each command’s code is isolated)** while cleanly separating the protocol handling from the execution of actions.

### **Context Object Pattern**

MCP by definition deals heavily with “context” – information that the AI and the server share to carry out tasks. The Context Object pattern is about encapsulating contextual data into a dedicated object that can be easily passed around to different parts of the system ([Context Object Pattern in Java: Simplifying Access to Contextual Data | Java Design Patterns](https://java-design-patterns.com/patterns/context-object/#:~:text=Intent%20of%20Context%20Object%20Design,Pattern)). Rather than passing numerous parameters or relying on global state, a **Context Object** holds all relevant state for a request or session (user identity, permissions, conversation history, etc.), and any component that needs this info can receive the context object. In an MCP server, this pattern ensures **consistent context propagation**: for example, if a tool needs to know the user’s ID or a session token, that can be included in a context object provided to every tool call. The pattern’s intent is to *decouple components from environmental complexity* – the business logic doesn’t need to know how to retrieve context (it’s just given an object) ([Context Object Pattern in Java: Simplifying Access to Contextual Data | Java Design Patterns](https://java-design-patterns.com/patterns/context-object/#:~:text=Intent%20of%20Context%20Object%20Design,Pattern)). This leads to cleaner APIs within the server. A real-world analogy is an airport using a “Passenger Context” record that multiple services (check-in, security, boarding) reference instead of each service independently gathering passenger info ([Context Object Pattern in Java: Simplifying Access to Contextual Data | Java Design Patterns](https://java-design-patterns.com/patterns/context-object/#:~:text=Real)). Likewise, an MCP server can maintain a **request context** object that travels through layers (from the FastAPI request handler, to the command executor, to the tool function) carrying data like request IDs, user metadata, or prior tool outputs. The Context Object pattern facilitates **efficient information sharing across system layers** and ensures that as the context “flows” through the server pipeline, no necessary information is lost ([Context Object Pattern in Java: Simplifying Access to Contextual Data | Java Design Patterns](https://java-design-patterns.com/patterns/context-object/#:~:text=Core%20J2EE%20Patterns%20says)). This is particularly important for AI interactions, where maintaining state (like conversation context or environment state) can be key. By designing a context object (using Pydantic, for example, to define its schema), the MCP server achieves protocol-independent context handling – the same context structure could be used for different transports or scenarios without rewriting integration code. In summary, this pattern improves **maintainability** (one structured context passed around) and supports **context-aware behavior** by making relevant data readily available wherever needed.

### **Dependency Injection**

**Dependency Injection (DI)** is a design principle in which components receive their dependencies from an external injector rather than instantiating them directly. This pattern is crucial for building **modular and testable** systems. In a Python MCP server, DI can be achieved via frameworks or patterns (for example, using FastAPI’s `Depends` mechanism or a manual factory pattern) to provide things like database connectors, external API clients, or configuration settings to the parts of the code that need them. The benefit is that the core logic of tools or services does not hard-code references to particular implementations – they ask for an interface or abstract class which is provided at runtime. This yields greater flexibility: the server can swap out, say, a logging backend or a vector database client without changing the tool code, simply by injecting a different implementation (e.g., a mock for testing or a different vendor’s SDK). DI **decouples components** and allows the code to **evolve easily** over time ([Using a Dependency Injection (DI) Container to decouple your code \- DEV Community](https://dev.to/jackmarchant/using-a-dependency-injection-di-container-to-decouple-your-code-11b2#:~:text=We%27ve%20seen%20how%20dependency%20injection,be%20passed%20into%20the%20constructor)). For example, if one tool relies on a repository class for data access, injecting that repository (configured via settings) means the tool logic isn’t tied to *how* data is stored. This decoupling also makes unit testing straightforward – one can inject stub dependencies to simulate external systems. Dependency Injection complements the MCP’s goal of modularity: each server capability can be developed and reasoned about in isolation, with clear contracts for what it needs (e.g., “I need a database session and an LLM client, give me those and I’ll do the job”). Modern Python tooling offers many DI conveniences: FastAPI’s declarative dependency system or external libraries (like `punq` or `dependency-injector`) can manage object lifetimes and injection. By using DI, we ensure the MCP server is built on **clean interfaces**, supporting changes and extensions with minimal ripple effects on the codebase ([Using a Dependency Injection (DI) Container to decouple your code \- DEV Community](https://dev.to/jackmarchant/using-a-dependency-injection-di-container-to-decouple-your-code-11b2#:~:text=Now%2C%20BookService%20has%20no%20idea,why%20a%20DI%20container%20exists)) ([Using a Dependency Injection (DI) Container to decouple your code \- DEV Community](https://dev.to/jackmarchant/using-a-dependency-injection-di-container-to-decouple-your-code-11b2#:~:text=We%27ve%20seen%20how%20dependency%20injection,be%20passed%20into%20the%20constructor)). Overall, DI enhances **maintainability, testability, and configurability** of the server.

### **Layered Architecture**

A **Layered Architecture** (often a multi-tier design) organizes the server into distinct layers with defined responsibilities. This separation of concerns is especially valuable for a complex system like an MCP server, which touches on networking/protocol handling, business logic, and possibly data storage or external API calls. In a typical layered design, one might have:

* a **Presentation/Interface Layer** (in this context, the MCP protocol interface, e.g. an API endpoint or STDIO handler that receives requests and returns responses),

* a **Service/Domain Layer** (the core logic that processes a tool call – e.g., interpreting parameters, applying business rules, orchestrating calls to models or data),

* a **Data/Integration Layer** (responsible for actual I/O with databases, file systems, external web APIs, or invoking an AI model if the server itself uses one internally).

Each layer communicates **only with the layer directly beneath or above it**, creating a clear structure that’s easier to maintain and expand ([Layered Architecture: Key to System Scalability](https://www.linkedin.com/advice/0/how-does-layered-architecture-enhance-system-7z1qe#:~:text=Layered%20architecture%2C%20a%20method%20of,easy%20to%20manage%20and%20expand)). For instance, the protocol layer shouldn’t need to know how to query the database – it just passes the request to a service which uses a repository class in the data layer. This yields a system where you can modify or replace one layer (say, switch from STDIO to HTTP+SSE transport on the interface layer, or swap out a logging database in the data layer) with minimal changes to others ([Layered Architecture: Key to System Scalability](https://www.linkedin.com/advice/0/how-does-layered-architecture-enhance-system-7z1qe#:~:text=Layered%20architecture%20divides%20a%20system,scaling%20as%20the%20application%20grows)). Layered architecture also improves **scalability**: each layer can be scaled or optimized independently (e.g., if the data layer is the bottleneck, one can add caching there or scale the database, without touching the protocol code) ([Layered Architecture: Key to System Scalability](https://www.linkedin.com/advice/0/how-does-layered-architecture-enhance-system-7z1qe#:~:text=2%20Scalability%20Boost)). In an MCP server, a *layered approach* might manifest as a **clean API module, a core logic module, and utility modules** for external integrations. For example, the **FastAPI** layer (or a custom TCP/stdio interface) would handle incoming MCP messages and convert them to internal calls; the core logic would interpret which tool to run (perhaps using the Command pattern dispatch) and manage context; and an integration layer would contain clients for things like cloud APIs (for a Google Drive server, this layer would call Google’s API). This strict separation means each concern is addressed in one place, aiding **maintainability and team collaboration** (different teams could work on different layers) ([Layered Architecture: Key to System Scalability](https://www.linkedin.com/advice/0/how-does-layered-architecture-enhance-system-7z1qe#:~:text=Layered%20architecture%2C%20a%20method%20of,easy%20to%20manage%20and%20expand)). Additionally, such layering aligns well with **Hexagonal (Ports and Adapters) architecture**, which is essentially a form of layered design where the MCP protocol is one “port” and the external service is another “port,” and the core logic is the independent hexagon in between. Indeed, many view MCP servers fundamentally as **adapters** that translate standardized MCP requests into domain-specific actions ([mcpadapt | Model Context Protocol](https://model-context-protocol.com/servers/mcp-server-adapt#:~:text=The%20project%20uses%20an%20adapter,ToolAdapter%20class%20that%20framework)). A layered/hexagonal design implements this by keeping the translation logic at the edges and the domain logic in the center. By following a layered architecture, our MCP server remains **modular, scalable, and easier to extend**, which is crucial given MCP is evolving and new features or tools will likely need to be integrated over time.

*(In summary, these patterns collectively ensure that an MCP server can handle context-rich interactions efficiently while being easy to extend with new capabilities. The Command and Adapter patterns help structure tool actions; the Context Object pattern and DI ensure context and dependencies are handled cleanly; and a Layered Architecture ties it all together with clear boundaries, supporting scalability and maintainability.)*

## **3\. Pydantic AI Integration**

**Pydantic AI** is an extension of the Pydantic library geared towards AI applications, and it proves extremely useful in building an MCP server. Pydantic’s core strength is in defining clear data models (schemas) and validating data against those models, which aligns perfectly with MCP’s need for structured context and tool I/O definitions. In an MCP server implementation, Pydantic can be leveraged in multiple ways:

* **Schema Definition for Tools and Context**: Each MCP tool can be defined with input and output schemas using Pydantic models. Pydantic models (classes inheriting from `BaseModel`) allow us to explicitly declare the fields and types that a tool expects and produces. This ensures that when an AI client calls the tool, the JSON payload is automatically parsed into a well-defined Python object (or rejected if it’s invalid). For example, if we have a tool `find_user`, we might define a Pydantic model `FindUserRequest` with fields like `username: str` or `user_id: int`; the tool function can then accept a parameter of type `FindUserRequest`. Pydantic will handle parsing the incoming request to that model and validating that all required fields are present and of correct type. Similarly, the output can be a Pydantic model (or built-in types), which guarantees the server’s response follows the MCP schema. By **using Pydantic models to define the structure of input and output data**, we get type safety and documentation for free ([PydanticAI In Practice: Agent Framework for Type-Safe Python LLM Development — Powering DaaiC.dev | by Fru | Medium](https://medium.com/@frulouis/pydanticai-in-practice-agent-framework-for-type-safe-python-development-powering-daaic-dev-cc20f4861a8a#:~:text=Pydantic%20AI%20Agents%20extend%20the,These%20agents)). This practice leads to **structured and consistent interactions** – no unstructured blobs, but rather well-defined contracts for each tool invocation.

* **Validation and Guardrails**: An MCP server often deals with inputs coming from AI agents, which might be generated or user-derived. Pydantic provides robust validation: beyond type checking, we can add constraints (like string length, regex patterns, numerical ranges) and custom validators (ensuring, for instance, a date string is not in the future, or an enum field is one of known values). This is critical in maintaining server robustness, because it prevents the AI (or a bug in the client) from triggering actions with malformed or dangerous inputs. Pydantic AI’s philosophy is to *enforce schemas so that AI outputs align with application requirements* ([PydanticAI In Practice: Agent Framework for Type-Safe Python LLM Development — Powering DaaiC.dev | by Fru | Medium](https://medium.com/@frulouis/pydanticai-in-practice-agent-framework-for-type-safe-python-development-powering-daaic-dev-cc20f4861a8a#:~:text=Bridging%20the%20Gap%20Between%20AI,and%20Structured%20Data)) – the same applies on the server side. If the AI sends something off-spec, the server can catch it early and respond with an error, rather than executing a flawed operation. This **reduces runtime errors and makes data handling seamless** ([PydanticAI In Practice: Agent Framework for Type-Safe Python LLM Development — Powering DaaiC.dev | by Fru | Medium](https://medium.com/@frulouis/pydanticai-in-practice-agent-framework-for-type-safe-python-development-powering-daaic-dev-cc20f4861a8a#:~:text=adheres%20to%20the%20expected%20structure%2C,and%20making%20data%20handling%20seamless)), acting as a safety net for the server’s operations. It also provides a clear contract; e.g., the AI developer knows exactly what JSON structure the server expects for each tool.

* **Model Management and Versioning**: As the MCP spec or our server’s capabilities evolve, we might need to change the schemas (perhaps add a field to a context or tool input). Pydantic supports **schema versioning strategies**. A straightforward approach is using model inheritance to create versioned schemas: e.g., `MyContextV1`, `MyContextV2` classes where V2 extends V1 with new fields ([Episode 13: Version Information & Annotations in Pydantic | by Kishanbabariya | Medium](https://medium.com/@kishanbabariya101/episode-13-version-information-annotations-in-pydantic-a7545976fcab#:~:text=Pydantic%20allows%20you%20to%20define,attributes%20from%20the%20previous%20version)). This way, the server could handle multiple versions of requests if needed by checking a version field and using the appropriate model class. Pydantic models can also be annotated or unioned for backward compatibility. The goal is to **maintain backward compatibility** while moving forward – something Pydantic facilitates by making it easy to declare new models that extend old ones ([Episode 13: Version Information & Annotations in Pydantic | by Kishanbabariya | Medium](https://medium.com/@kishanbabariya101/episode-13-version-information-annotations-in-pydantic-a7545976fcab#:~:text=Pydantic%20allows%20you%20to%20define,attributes%20from%20the%20previous%20version)) ([Episode 13: Version Information & Annotations in Pydantic | by Kishanbabariya | Medium](https://medium.com/@kishanbabariya101/episode-13-version-information-annotations-in-pydantic-a7545976fcab#:~:text=class%20UserV1,str)). Additionally, Pydantic’s strong typing aids with internal version control: if a tool changes its output schema, any part of code still expecting the old model will raise a type error in development. This explicitness helps senior engineers track and manage **context schema evolution** over time. In practice, you might also include a version field in your context or use the MCP protocol’s versioning (if any) to decide which Pydantic model to use for parsing a request.

* **Pydantic AI Agents within the Server**: Another interesting integration is that Pydantic AI allows using AI *agents* as part of the server’s functionality. For example, PydanticAI provides an `Agent` class that can wrap an LLM (OpenAI, Anthropic, etc.) with input/output models. The Pydantic docs show that you can embed an Agent in a tool – in the example, a “poet” tool calls a Claude model via a PydanticAI Agent to generate a poem ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=server%20%3D%20FastMCP,system_prompt%3D%27always%20reply%20in%20rhyme%27)). This means the server itself can leverage AI and still keep the interaction typed and structured. The agent’s input prompt and output are managed by Pydantic (the `result_type` of the Agent ensures the AI’s output conforms to a schema) ([PydanticAI In Practice: Agent Framework for Type-Safe Python LLM Development — Powering DaaiC.dev | by Fru | Medium](https://medium.com/@frulouis/pydanticai-in-practice-agent-framework-for-type-safe-python-development-powering-daaic-dev-cc20f4861a8a#:~:text=and%20Data%20Analytics%20events%3A)) ([PydanticAI In Practice: Agent Framework for Type-Safe Python LLM Development — Powering DaaiC.dev | by Fru | Medium](https://medium.com/@frulouis/pydanticai-in-practice-agent-framework-for-type-safe-python-development-powering-daaic-dev-cc20f4861a8a#:~:text=)). In an MCP server context, Pydantic AI could manage an internal language model that perhaps reformats or interprets data. By integrating at this level, one ensures that even when the server invokes AI internally, the exchange remains **structured, validated, and context-aware**.

* **Structured Context and Workflows**: MCP servers often need to handle **context objects** that encapsulate state for a request or session (as discussed in the pattern above). Pydantic is ideal for defining such context schemas. For example, you might have a `SessionContext` model with fields like `session_id`, `user_profile: UserProfile`, `permissions: List[Permission]`, etc. This model can be threaded through your calls (possibly via dependency injection in FastAPI). Pydantic will ensure that whenever a context is constructed (say at session initialization or request startup), it’s valid and all required info is present. Moreover, if the context needs to be enriched or transformed (e.g., adding new info after a tool executes), Pydantic models can be easily extended or updated (via `.copy(update={...})` or creating new instances). All these contribute to **context-aware interactions** – the server can confidently propagate a Pydantic-modeled context knowing it remains consistent. Also, Pydantic’s JSON serialization can be used to log or send parts of the context if needed (with sensitivity in mind).

In summary, Pydantic (and Pydantic AI) brings **type-safe, schema-driven development** to MCP servers. It manages model definitions for inputs/outputs, validates data at boundaries, assists with versioning of schemas, and even helps embed AI functionalities in a controlled manner. By enforcing that **all data crossing the MCP interface or flowing between components adheres to defined models**, we greatly reduce integration errors and improve clarity. As one source puts it, Pydantic helps *“bridge the gap between AI and structured data”*, eliminating the chaos of unstructured outputs ([PydanticAI In Practice: Agent Framework for Type-Safe Python LLM Development — Powering DaaiC.dev | by Fru | Medium](https://medium.com/@frulouis/pydanticai-in-practice-agent-framework-for-type-safe-python-development-powering-daaic-dev-cc20f4861a8a#:~:text=Bridging%20the%20Gap%20Between%20AI,and%20Structured%20Data)). This is invaluable in a production MCP server where reliability and correctness of each tool call matter. Senior engineers can leverage these capabilities to build a robust server that confidently handles complex interactions without sacrificing the guarantees that strong typing and validation provide.

*(Note: Using Pydantic in an async context is fully supported – Pydantic V2 is mostly synchronous for model creation, but since validation is fast and I/O is the main async workload, there’s no conflict. Also, Pydantic models can be used in FastAPI request bodies or as function parameters seamlessly, which likely applies if our MCP server uses FastAPI under the hood for an HTTP transport.)*

## **4\. Code Structure & Best Practices**

Designing a **modular codebase** for an MCP server is essential to handle growth in features and to keep the project maintainable. A clean separation of files and directories also enables multiple developers to work on different aspects (e.g., adding a new tool vs. tweaking core logic) without stepping on each other’s toes. Here we propose a production-grade Python project structure and highlight best practices:

```
mcp_server/
├── app.py               # Entry point, initializes the FastAPI or MCP server instance
├── api/                 # API layer (transport and routing)
│   ├── v1/              # Versioned API (if supporting versioned endpoints)
│   │   ├── endpoints.py # MCP endpoint implementations (HTTP routes or stdio handlers)
│   │   └── ...          
│   └── dependencies.py  # Dependency definitions for DI (e.g., get_db, get_config)
├── core/                # Core functionality and config
│   ├── tools.py         # Tool registration and dispatch logic (Command pattern)
│   ├── context.py       # Context object definitions and context manager logic
│   ├── settings.py      # Application settings (possibly using Pydantic BaseSettings)
│   └── utils.py         # Utility functions (common helpers)
├── models/              # Pydantic models for schema definitions
│   ├── requests.py      # Pydantic models for tool inputs and context schemas
│   ├── responses.py     # Pydantic models for tool outputs or common results
│   └── domain.py        # (Optional) Domain-specific models (e.g., data schemas)
├── services/            # Business logic layer (could also be named "services" or "actions")
│   ├── data_access.py   # Abstractions for database or external API access (repos, clients)
│   ├── ai_agent.py      # If using an internal AI agent (PydanticAI agent setup)
│   └── ...              # Additional service classes or functions
├── db/                  # Database integration (if applicable)
│   ├── connection.py    # Database connection initialization
│   └── models.py        # ORMs or raw SQL models if needed
├── tests/               # Test suite for the server
│   ├── test_api.py      # Tests for API endpoints
│   ├── test_tools.py    # Tests for tool logic
│   └── ...              
└── requirements.txt / pyproject.toml  # Dependencies and project metadata
```

This structure is inspired by common FastAPI project layouts and examples of MCP servers in the community ([Awesome MCP FastAPI | Glama](https://glama.ai/mcp/servers/2ysi73d5t3#:~:text=Our%20application%20follows%20a%20modular,architecture)). The **`api`** package represents the interface layer. For an MCP HTTP server, this might include FastAPI routers and endpoints; for an STDIO-based server, it could include an initializer that uses the official MCP Python SDK to handle stdio communication. We separate versioned API definitions (`v1`, `v2`, etc.) if we anticipate breaking changes in the protocol down the line. Within `api`, the `dependencies.py` can define reusable dependencies – for example, a function to provide the current context or database session to endpoints (using FastAPI’s `Depends`). This ties into our DI pattern: we declare what each request needs and FastAPI (or our framework of choice) injects it.

The **`core`** module contains central pieces:

* `tools.py` could maintain the **tool registry** and routing logic. For instance, if using a framework like *FastMCP* (which might wrap FastAPI for MCP), this is where we use `server = FastMCP(...)` and decorate tool functions with `@server.tool()` as in the Pydantic example ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=server%20%3D%20FastMCP,system_prompt%3D%27always%20reply%20in%20rhyme%27)). If not using that, we could implement our own dispatcher: e.g., a mapping from tool names to function objects, and a function `execute_tool(name, payload, context)` that finds the right function and calls it (after Pydantic validation). This is essentially the Command pattern manifest in code – mapping commands to executors.

* `context.py` might define a `RequestContext` Pydantic model and perhaps a context manager or middleware that populates it. In FastAPI, we might use `Depends` to create a context object per request (gathering auth info, etc.). In a simpler STDIO setting, we might not have HTTP layers, so context could be passed along via the protocol messages themselves. Either way, having a module dedicated to context handling centralizes that logic.

* `settings.py` uses Pydantic’s `BaseSettings` to define configuration (API keys, URLs, etc.) which can be loaded from env or config files. This is a best practice for any app, and especially useful here since our server likely needs to know credentials for external APIs (e.g., a Logfire API key) or model names. Using Pydantic for settings ensures type correctness and allows easily extending config.

* `utils.py` could hold miscellaneous helpers (e.g., formatting functions, common constants, etc.).

The **`models`** package is where all our **Pydantic schemas** live. It’s good to group them logically – e.g., request models vs response models – to keep clarity. For instance, if we have a tool `find_exceptions` (from the Logfire example) that requires `age: int` and returns a list of exceptions, we can define `FindExceptionsRequest(BaseModel)` and `FindExceptionsResult(BaseModel)` in these modules. Co-locating models makes it straightforward to manage changes and to generate documentation (these models could also be referenced in OpenAPI docs automatically via FastAPI). They also serve as the single source of truth for data structures used across the layers.

The **`services`** layer (or call it `business` or `domain`) contains the actual logic implementations for our tools and any supporting operations. For example, for a *weather* MCP server, a `services/weather_service.py` might have functions that call an external weather API. In the Logfire server, you might have `services/logfire_queries.py` that knows how to query the Logfire API given parameters. The idea is to keep this logic separate from the controller (API) and separate from low-level details. If using a database, `data_access.py` might define repository classes or direct queries; if using external SDKs, this layer would wrap those calls so that the rest of the server doesn’t deal with SDK specifics. We also included `ai_agent.py` to highlight that if the server uses an internal LLM (like the poem generator example), you’d encapsulate that here – e.g., initialize a `Agent` with model and system prompt in one place and expose a function `generate_poem(theme)`. The rest of the server can call `generate_poem` without knowing it’s using an Anthropics model under the hood. This follows **separation of concerns** and keeps things modular.

Some **best practices** in coding this out:

* **Modularity and Reusability**: Each tool should ideally be a function or class method that can be called independently (and thus tested independently). By decoupling tools into the `services` layer and making them pure functions (as much as possible), we allow reuse (maybe one tool can call another internally) and simplify testing.

* **Use Async IO**: For I/O-bound operations (which most MCP tools are, e.g., calling an API or database), define functions as `async def` and use `await` for network or disk operations. Using FastAPI or an async MCP framework ensures that multiple requests can be handled concurrently with high throughput. In practice, our endpoints or command dispatcher will be async and will call async service functions. This non-blocking design is crucial for scalability, as an MCP server might be handling many tool requests in parallel (especially if used by a busy AI agent).

* **Dependency Injection & FastAPI Depends**: As touched on, use FastAPI’s dependency system for things like injecting a DB session or a config object into your path functions. For example, a tool function might declare `def find_exceptions(age: int, db=Depends(get_db))` – FastAPI will provide a database session. This removes the need for global variables and keeps functions explicit in what they need. Similarly, you might have `def some_tool(params: SomeModel = Body(...), context: RequestContext = Depends(get_context))` to automatically get a validated body and a built context. This practice yields cleaner, self-documenting code.

* **Error Handling and Validation**: Use Pydantic validation errors or custom exception handling to return meaningful messages to the AI client. Since Pydantic will raise `ValidationError` if input is wrong, you can catch that in the API layer and format an MCP error response. Likewise, if a tool function raises an exception (say an HTTP 500 from an external API), you should catch it and translate it to an MCP response (perhaps an `error` field in the result). Encapsulate such error-handling logic in one place (maybe a middleware or in the command dispatcher) to avoid repetition.

* **Logging and Tracing**: Implement structured logging within your server. For each request (tool call), log an entry with the tool name, parameters (perhaps sanitized), and the outcome. This is useful for debugging interactions with the AI. Using Python’s `logging` library with a consistent format (JSON logging can be helpful) and possibly integrating with OpenTelemetry (as Logfire does) for distributed tracing can greatly aid observability (we discuss more in Advanced Considerations). But from a code structure perspective, having a `logging` setup configured in `app.py` (so all modules use the same logger) is a good practice.

* **Testing**: Organize tests mirroring the structure of the code. Unit test each tool function in `services` by injecting fake dependencies (for instance, simulate the external API responses). Also test the API layer with test clients (FastAPI provides a `TestClient` to simulate HTTP calls, or you might simulate stdio messages). By structuring code into layers, you can test each layer in isolation: e.g., test that the FastAPI endpoint correctly calls the service given a valid Pydantic model, and separately test that the service does the right thing when the external API returns X or Y. Use Pydantic’s `.dict()` method in tests to easily create payloads.

* **Documentation**: Thanks to Pydantic and FastAPI, a lot of documentation can be auto-generated. Ensure to add docstrings to your tool functions – these can be used as descriptions in OpenAPI docs or by MCP tool registries. For instance, the example in *Awesome MCP FastAPI* uses the docstring of the function as the tool’s description ([Awesome MCP FastAPI | Glama](https://glama.ai/mcp/servers/2ysi73d5t3#:~:text=from%20fastapi%20import%20FastAPI%20from,Cannot%20divide%20by%20zero)). Writing clear docstrings (including what the tool does, expected input and output) will benefit anyone using your MCP server (and even the AI might use those descriptions to decide when to call the tool).

* **Folder and Naming Conventions**: Keep names intuitive – e.g., `find_exceptions` tool corresponds to `find_exceptions` function, and perhaps a `FindExceptionsResult` model. Consistency helps avoid mistakes. Also, as the project grows, consider grouping tools by domain into submodules if needed (for example, `services/logs_tools.py` vs `services/db_tools.py` if those categories emerge).

By following this kind of organized structure, we create a codebase that is **easy to navigate** and **extensible**. If a new engineer joins, they can quickly locate where to add a new tool (e.g., add a function in `services` and an endpoint in `api`). If we need to support a new protocol transport (say, add a WebSocket interface), we can add a new module under `api/` without touching the core logic. The separation of config, models, and business logic means each can evolve on its own timeline (for example, upgrading a database library mostly affects `db/` and maybe `services/data_access.py`, without interfering with tool definitions). This modular approach was echoed by an example project which explicitly followed a **modular architecture** dividing API, core, DB, etc. ([Awesome MCP FastAPI | Glama](https://glama.ai/mcp/servers/2ysi73d5t3#:~:text=Our%20application%20follows%20a%20modular,architecture)). Emulating such proven layouts will ensure our MCP server is production-ready and manageable in the long run.

## **5\. Advanced Considerations**

Designing a modern MCP server isn’t just about fulfilling the basic protocol – to be truly **production-grade**, we must consider cross-cutting concerns and future evolution. Here we discuss a few advanced considerations:

**Extensibility & Modularity**: As new tools or features are added, the server should handle them with minimal refactoring. By using the patterns above, we largely achieve *Open/Closed Principle* – for example, new commands can be added without modifying the dispatcher significantly. It’s wise to design the tool registration system such that new tool modules can be auto-discovered or easily plugged in. Some frameworks allow dynamic loading of tool definitions (perhaps scanning a folder for functions decorated as tools). If we anticipate community or third-party extensions, we might even allow a plugin mechanism: e.g., drop-in Python packages that the server reads to register additional capabilities. Ensuring that internal APIs (between layers) are stable is also part of extensibility. Moreover, because MCP itself might introduce new primitives (for instance, the spec might add a notion of *server-initiated callbacks* or new message types), we should keep our **protocol handling logic abstracted**. If currently we handle a fixed set of message types (initialize, call tool, etc.), design that part to be data-driven (maybe based on message name) so that if a new type “requestCompletion” or similar appears, we can support it by adding a handler function rather than rewriting the parser. Essentially, treat the MCP spec as something that could evolve and keep our code flexible accordingly.

**Performance and Scalability**: Performance optimizations may be needed once the basic functionality is in place. One important aspect is to avoid blocking operations – use asynchronous calls for I/O, and if heavy CPU work is needed (like large data processing), consider offloading it to worker threads or processes (using libraries like `asyncio.to_thread` or background task queues like Celery). Caching is another tool: for instance, if a certain context data is expensive to fetch but used in many requests, caching it in memory for the session duration could save time (this echoes the idea from the context pattern discussion – *cache context data to avoid redundant retrievals* ([Exploring Model Context Protocol (MCP) – Matt's Blog](https://mswis.com/exploring-model-context-protocol-mcp/#:~:text=context%20%3D%20ModelContextProtocol,the%20context%20reuse%20cached%20data)) ([Exploring Model Context Protocol (MCP) – Matt's Blog](https://mswis.com/exploring-model-context-protocol-mcp/#:~:text=This%20simple%20illustration%20shows%20how,centralizing%20and%20caching%20the%20context))). However, be mindful of cache invalidation and memory use. The server should also be horizontally scalable: design it such that it can run behind a load balancer with multiple instances (meaning any in-memory session state is minimal or shared via an external store). If stateful interactions are needed (say a tool that maintains an environment), consider using a lightweight database or cache (Redis) to store that state so that any instance can retrieve it. MCP servers are typically *lightweight* by design ([Introduction \- Model Context Protocol](https://modelcontextprotocol.io/#:~:text=At%20its%20core%2C%20MCP%20follows,can%20connect%20to%20multiple%20servers)), focused on a specific domain capability – this helps scalability as well, since each instance handles a manageable scope of work.

**Observability (Logging, Monitoring, Tracing)**: Observability is crucial in a live environment, both for debugging and for understanding usage patterns. We should incorporate structured **logging** at key points: log each request from the client (with a request ID, timestamp, tool name, maybe part of the params), and log the response or any error. In Python, using the standard `logging` module with JSON output makes it easier to feed logs into systems like ELK or Datadog. Additionally, consider **metrics**: using something like Prometheus client library to count how many times each tool was called, how long they take (histograms of latency), how many errors occur, etc. These metrics can be exposed via an endpoint (if HTTP) or logged. For tracing, if using OpenTelemetry, one could propagate a trace context from the AI request (the MCP spec might include a trace context in headers or so) and then create spans for the work done in the server. For example, if the MCP server calls an external API, wrap that call in a traced span to measure that external call latency. Pydantic AI’s Logfire MCP server example itself is all about observability – it allows AI to query logs/traces ([GitHub \- pydantic/logfire-mcp: The Logfire MCP Server is here\! :tada:](https://github.com/pydantic/logfire-mcp#:~:text=Logfire%20MCP%20Server)). We can take inspiration from it to instrument our server similarly (perhaps even dogfooding by having our server use Logfire to log about itself\!). In summary, treat the MCP server like any critical microservice: employ **monitoring** (CPU, memory, etc.), set up **health checks** (FastAPI can have a health endpoint, or the stdio server could have a periodic heartbeat log), and prepare dashboards/alerts based on the metrics.

**Security**: Although not explicitly requested, security is an advanced concern worth noting. MCP servers often execute powerful actions (imagine one connected to a database or filesystem). We must enforce appropriate **authorization** – the MCP host typically manages permissions (the AI agent will only be allowed certain tools), but the server should still **validate credentials or tokens** provided. For example, if the host passes a user token in context, the server should verify it before executing the action. Ensure to sanitize any inputs if they might be used in commands (e.g., if a tool runs a shell command or SQL query, guard against injection). Running the server itself with least privileges (if it interacts with a filesystem, sandbox it to allowed paths) is wise. Since MCP is an open protocol, also consider **rate limiting** or abuse detection – you don’t want a rogue (or buggy) client to overwhelm the service. These facets go beyond architecture, but our architecture can accommodate them by having, say, a middleware layer for authentication/authorization and by designing tools to check permissions.

**Future-Proofing for MCP Evolution**: The MCP spec is evolving (Anthropic and others are actively improving it). One example is the notion of *“sampling”* – where an MCP server might ask the client’s LLM to generate text (for instance, to complete a prompt that is part of the tool operation) ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=Note%3A%20sampling%2C%20whereby%20servers%20may,not%20yet%20supported%20in%20PydanticAI)). Pydantic’s docs note that this feature (server-initiated LLM requests) was not yet supported in their library ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=Note%3A%20sampling%2C%20whereby%20servers%20may,not%20yet%20supported%20in%20PydanticAI)), but likely to come. We should architect our server in a way that adding such a bidirectional interaction is possible. This could mean our server’s core can handle both *incoming* calls and *outgoing* requests if needed (perhaps by maintaining a reference to the client connection). Another area is **transports** – currently, common transports are STDIO (spawn the server as a subprocess) and HTTP/SSE. In the future, there might be direct library calls or other IPC mechanisms standardized. Writing our server logic in a transport-agnostic way will help: for example, by separating the FastAPI app from the tool execution logic, we could later integrate a different frontend. In fact, there are projects to mount an MCP server on top of FastAPI automatically ([tadata-org/fastapi\_mcp: A zero-configuration tool for ... \- GitHub](https://github.com/tadata-org/fastapi_mcp#:~:text=tadata,app%20and%20it%20works)) ([FastAPI-MCP \- MCP Servers](https://mcp.so/server/fastapi_mcp/tadata-org#:~:text=FastAPI,with%20your%20FastAPI%20app)), which hints that if we write a standard FastAPI app for our tools, a lightweight adapter can expose it as MCP. Embracing such patterns (possibly by using that library or adhering to its conventions) can make our implementation future-proof and compatible with tooling improvements. Additionally, keep an eye on **MCP specification changes** – having a test suite that can be updated against the official MCP test cases or SDKs will catch when things break due to spec updates.

**Scalability & Distribution**: If our MCP server becomes heavily used, we might consider deploying it as a set of microservices (if one server binary is handling too much). For example, if one tool is extremely resource-intensive, it could be hosted on a separate service that our main MCP server calls internally. This starts to look like the AI Orchestrator pattern, but it’s conceivable for scaling. For now, assume one MCP server is one process, but design with **decoupling** such that calls to heavy backends (like a local ML model or a database) are abstract – if later that’s moved out-of-process, the rest of the server doesn’t change much (just the implementation of that service call changes to an RPC).

**Testing and CI/CD**: As advanced practice, set up continuous integration to run tests and perhaps use containers (Docker) to ensure the server runs in a reproducible environment. Because MCP may involve different OS-level concerns (especially for STDIO servers run as subprocesses by hosts), test in conditions similar to production. Write integration tests that simulate a full round-trip: a dummy MCP client connecting and calling a tool. This ensures compliance with the protocol.

By addressing these advanced considerations, we **future-proof** the MCP server and make it truly *production-ready*. We achieve a design that can adapt to new MCP features, handle increasing load gracefully, provide deep insight into its own operation, and remain secure and stable in the face of the unexpected. These qualities differentiate a simple working prototype from a system that can be confidently deployed in an enterprise environment where AI-driven systems are expected to be reliable and trustworthy.

The FastAPI integration also means you get an OpenAPI docs UI for your MCP server (useful for development) and can serve both “standard” REST endpoints and MCP interfaces side by side ([FastAPI | Glama](https://glama.ai/mcp/servers/integrations/fastapi#:~:text=FastAPI%20,MCP%20and%20standard%20web%20endpoints)).
