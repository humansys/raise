# Standardizing Cursor Rules (v0.47+) for a Team FastAPI/PydanticAI Project

## 1. Definition and Functionality of Cursor Rules (v0.47+)  
**What are Cursor Rules?** In Cursor IDE, “rules” are configuration files that act as project-specific or global instructions to guide the AI coding assistant. They essentially serve as a persistent system prompt or guidelines for the AI, telling it how to behave when writing or editing code ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Using%20rules%20in%20Cursor%20you,a%20system%20prompt%20for%20LLMs)). In other words, Cursor Rules define the context, best practices, and constraints for the AI’s interactions with your codebase, enabling *context-aware assistance* rather than a generic code generation ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=Cursor%20Rules%20are%20configuration%20files,behavioral%20patterns%20for%20AI%20interactions)). For example, a rule might inform the AI that your project uses FastAPI with Pydantic models and should follow certain coding conventions, so the AI will factor those instructions into its responses. 

**Types and Scope – Global vs Project Rules:** There are two main scopes for rules in Cursor 0.47+:

- **Global Rules:** These are user-specific settings that apply to *all* projects. They are configured in **Cursor Settings > General > “Rules for AI”** (within the IDE) and are always included in the prompt context for any chat ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Global%20Rules)) ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Global%20Rules)). Global rules are ideal for preferences that every project should follow (for example, always answer in English, or always use a certain coding style). However, because they aren’t tied to a repository, they are not easily shared with a team except by manual duplication.  
- **Project Rules:** These are rules stored with a particular codebase (in the repository itself). In Cursor v0.47, project rules live as markdown files (with extension `.mdc`) inside a directory called `.cursor/rules` at the root of your project ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Project%20Rules)). Project rules are loaded automatically by Cursor when you work on that project, providing *project-specific* context to the AI. This is the recommended approach for team environments, since the `.cursor/rules` files can be committed to version control and shared with all collaborators ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=1,everyone%20pulls%20the%20same%20rules)). (In older versions of Cursor, a single file named `.cursorrules` in the project root was used to hold all rules. As of 0.47 this legacy file is still supported for backwards compatibility, but it’s deprecated and will be removed in the future ([How to use Cursor Rules | Instructa Courses](https://www.instructa.ai/en/blog/how-to-use-cursor-rules-in-version-0-45#:~:text=Note%3A%20The%20,deprecated)) ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=)). Teams should migrate to the new `.cursor/rules/*.mdc` system for better flexibility.)

**What do rules actually do?** Cursor Rules primarily influence **AI behavior and code generation**. They do not enforce formatting or linting the way a traditional linter does; instead, they *instruct the AI to follow certain standards*. You can use rules to convey anything you want the AI to remember about your project or coding standards. For example, rules might cover: coding style guidelines (naming conventions, preferred libraries/APIs), architectural patterns (e.g. “use repository pattern for database access”), or team policies (“never commit secrets”, “write unit tests for new endpoints”). In practice, this means the AI will *attempt* to adhere to these instructions when generating code or answers. (It’s still an AI model following a prompt – so while it usually complies, it’s not a compiler-enforced rule system.) You can think of it as setting the “house rules” for your AI pair-programmer. For instance, you might include a rule that says: *“All API endpoints must use FastAPI’s dependency injection and Pydantic for request validation”* – then when you ask Cursor’s AI to generate a new endpoint, it will follow those guidelines in its output. Rules can also include high-level project context (e.g. a summary of the project’s purpose or tech stack) so the AI doesn’t hallucinate irrelevant solutions ([How to use Cursor Rules | Instructa Courses](https://www.instructa.ai/en/blog/how-to-use-cursor-rules-in-version-0-45#:~:text=work,Image)). They even can direct the AI’s *tone or role* (for example, you can instruct the AI to act as a Python/FastAPI expert or to always explain its reasoning). 

**How rules are applied (Context Injection):** Under the hood, Cursor uses a clever two-stage mechanism for rules ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=Rules%20are%20injected%20into%20the,rule%20is%20injected%20depends%20on)) ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=)):

- **Injection:** Based on certain criteria, the content of some rule files is injected into the AI’s system prompt context. Criteria include:
  - *Glob pattern matching:* If a rule’s configured file pattern (“globs”) matches the file(s) currently in context or being edited, that rule’s content is automatically loaded into the prompt ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=An%20,content%20that%20looks%20like%20this)). For example, a rule with `globs: ["app/routes/**/*.py"]` would inject whenever you’re working on files under `app/routes/` in a Python project.
  - *Always-apply flag:* A rule can be marked as “always apply,” meaning it is unconditionally injected for the project ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=,situation%20globs%3A%20alwaysApply%3A%20true)). Use this for truly global project rules that should be present in every AI interaction (like overall project conventions or tech stack info).
- **Activation:** After injection, the AI model decides *when* each injected rule is actually relevant, guided by the rule’s **description**. The description is a short natural-language hint (provided in the rule’s front-matter) telling the AI in what scenario to use that rule ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=1.%20,such%20as)). In the combined system prompt, Cursor presents a list of available rules and their descriptions, instructing the model to fetch or use a rule only if it’s relevant to the user’s query ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=Cursor%20Rules%20are%20extra%20documentation,the%20AI%20understand%20the%20codebase)) ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=That%E2%80%99s%20the%20key%20prompt%20,them%20if%20they%20seem%20unrelated)). For example, if a rule’s description says “**For FastAPI endpoints**: Use this rule when working on API route code,” the AI will interpret that it should apply that rule’s guidance only when the task at hand involves FastAPI endpoint code. This two-step process ensures the AI isn’t overwhelmed with all rules at all times – it can pick and choose contextually appropriate ones, somewhat akin to how a developer might reference relevant parts of a style guide as needed. 

**Rule Types in v0.47+:** To make rule behavior more intuitive, Cursor 0.47 introduced four rule “attachment” modes that you can set when creating or editing a rule ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=When%20creating%20a%20new%20Cursor,file%20to%20the%20AI%20context)):

- **Always:** The rule is always included, regardless of context (equivalent to setting `alwaysApply: true` in the rule). This is useful for fundamental project rules or framework guidelines that should *always* influence the AI (e.g. “All code must be Python 3.10+, use type hints everywhere”) ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=When%20creating%20a%20new%20Cursor,file%20to%20the%20AI%20context)).
- **Auto-Attached:** The rule will attach automatically when files matching its pattern are in use. This corresponds to providing specific `globs` in the rule. It’s great for monorepos or projects with multiple domains – for example, a rule that targets `**/frontend/**` could auto-attach only for front-end files ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=When%20creating%20a%20new%20Cursor,file%20to%20the%20AI%20context)).
- **Agent-Requested:** The rule is not automatically injected by file context, but it can be pulled in by the AI “intelligently” based on user intent. In practice this relies on the rule’s description and the model’s judgment. The AI will decide to use such rules if it thinks they’re relevant to your question or task (this is how more general or situational rules behave in Chat mode) ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=When%20creating%20a%20new%20Cursor,file%20to%20the%20AI%20context)).
- **Manual:** The rule will only apply if explicitly invoked – meaning you as the user would have to manually attach it or reference it (for example, by using an `@` mention of the rule file in a prompt). If a rule is set to Manual, Cursor won’t inject it unless you tell it to. This might be used for very niche rules or large documentation files that you only sometimes want to bring in ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=When%20creating%20a%20new%20Cursor,file%20to%20the%20AI%20context)).

Under the hood, these modes are implemented via the `alwaysApply` flag and the presence/absence of globs and descriptions in the rule file, but it’s helpful for a team to explicitly decide which category each rule falls into. For instance, you might mark your primary “Project Guidelines” rule as **Always**, set specific technology rules (FastAPI, Pydantic, database) as **Auto-Attached** with file globs, and maybe have an extra big “Architecture Overview” rule kept on **Manual** (so it’s available when asked for but not always flooding the prompt). By leveraging these settings, you ensure the AI gets the right guidance at the right time without irrelevant rules interfering ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=That%E2%80%99s%20the%20key%20prompt%20,them%20if%20they%20seem%20unrelated)).

## 2. Directory Structure Best Practices for Rule Organization  
Organizing Cursor rules in a clear, scalable way is important for a team. The recommended approach is to use a dedicated folder in your repository for all project rules. Cursor already expects project rules to reside in a folder named `.cursor/rules/` at the root of your project ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Project%20Rules)). You should add this entire `.cursor` folder to version control (e.g., Git) so that everyone on the team gets the same rules when they pull the codebase ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=1,everyone%20pulls%20the%20same%20rules)). Storing rules alongside code ensures that AI guidance stays synchronized with the project’s evolution and is accessible to all contributors.

**Basic Layout:** At minimum, you’ll have a structure like: 

```text
your-project/
├── .cursor/
│   └── rules/
│       ├── some_rule.mdc
│       ├── another_rule.mdc
│       └── ... 
├── app/
│   └── ... (your code) 
└── ...
```

Each `.mdc` file inside `.cursor/rules` is one rule. You can choose a naming scheme that makes sense (more on that below). For example, one common convention is to have an **“instructions” or “global” rule file** that contains broad project-wide guidelines (often marked Always) and then additional rule files for more specific concerns or contexts ([How to use Cursor Rules | Instructa Courses](https://www.instructa.ai/en/blog/how-to-use-cursor-rules-in-version-0-45#:~:text=As%20projects%20get%20more%20complex%2C,this%20kind%20of%20separation%20possible)). In the context of a FastAPI + Pydantic project, you might organize rules like this: 

```text
.cursor/rules/
├── global.mdc              # General project instructions (Always apply)
├── fastapi.mdc             # FastAPI framework usage rules (Auto-attached to FastAPI files)
├── pydantic.mdc            # Pydantic model usage rules (Auto-attached to model files)
├── db.mdc                  # Database access patterns (Auto or Agent as needed)
└── agent_tasks.mdc         # (Optional) Rules for autonomous Agent mode tasks
```

Such separation keeps each rule file focused. As a concrete example, your `fastapi.mdc` could apply to all FastAPI route or API files, while `pydantic.mdc` might target model and schema files – each will only come into play when relevant, ensuring the AI only gets the instructions it needs for the code at hand ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=User%20Rules%20define%20global%20preferences%2C,K)) ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=Cursor%20Rules%20are%20extra%20documentation,the%20AI%20understand%20the%20codebase)).

**Nested Rule Directories:** In Cursor 0.47, you can further organize rules into subdirectories under `.cursor/rules` if desired. This was a new feature – earlier versions did not support scanning subfolders, but 0.47’s March 2025 update *“allows nested `.cursor/rules` directories”* ([Scan for Project Rules in Subdirectories of .cursor/rules - Feature Requests - Cursor - Community Forum](https://forum.cursor.com/t/scan-for-project-rules-in-subdirectories-of-cursor-rules/61534#:~:text=AI%20Code%20Editor%20www,says%20this%20is%20implemented)). This means you can group rules hierarchically. For instance, you could have sub-folders by domain or module, like: 

```text
.cursor/rules/
   ├── 00_global/…
   ├── backend/
   │    ├── fastapi.mdc
   │    ├── pydantic.mdc
   │    └── database.mdc
   └── frontend/
        ├── react.mdc
        └── tailwind.mdc
```

This is similar to how some teams structure ESLint or Stylelint rules (e.g., grouping rules by front-end vs back-end). A domain-based structure makes it easier to navigate as the number of rule files grows ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=,mdc)). Each rule file’s glob patterns should be set so that they only target their intended scope (for example, `backend/fastapi.mdc` might have `globs: ["**/routers/**/*.py", "**/views/**/*.py"]` to match FastAPI-related code paths, whereas `frontend/react.mdc` would have globs for `.tsx` files, etc.). **Note:** If you use subdirectories, verify that Cursor is picking them up – the 0.47 implementation should handle it, as users confirmed it working in v0.47.5 ([Scan for Project Rules in Subdirectories of .cursor/rules - Feature Requests - Cursor - Community Forum](https://forum.cursor.com/t/scan-for-project-rules-in-subdirectories-of-cursor-rules/61534#:~:text=Have%20you%20found%20this%20feature,directory%20of%20.cursor%2Frules)) ([Scan for Project Rules in Subdirectories of .cursor/rules - Feature Requests - Cursor - Community Forum](https://forum.cursor.com/t/scan-for-project-rules-in-subdirectories-of-cursor-rules/61534#:~:text=Yes%2C%20it%E2%80%99s%20working%20for%20me%21,CursorAI%2FProject%20Rule%2FTest%2FAgent%2F%20%2FFind%20MDC%2Fin%20Subfolder)).

**Local vs. Global Rules in a Team Setting:** Within a repository, all rules in `.cursor/rules` are “local” to that project and shared by the team through version control. Global (user-level) rules are stored in each user’s Cursor app settings, not in the repo. To ensure consistency in a team, **favor project rules over personal global rules** for anything that affects code generation. Each developer should keep their global user rules minimal (perhaps just personal preferences like AI tone or default language), and encode all project-related standards in the shared `.cursor/rules` files. This way, when a new team member joins and opens the project in Cursor, the AI will behave the same for them as for everyone else.

If your organization has certain coding standards or AI prompt rules that span *all* projects (for example, a “company-wide style guide”), you have a few options to standardize those across repos: 

- **Manual Duplication:** maintain a base set of rule files and copy them into each project’s `.cursor/rules` folder. This is straightforward but can become hard to maintain if the base rules change frequently.  
- **Git Submodule or Shared Repo:** since Cursor now supports nested rule directories, one strategy is to include a submodule (or subtree) in each repo that points to a central “rules” repository. For example, you could have a repo `ai-rules-central` containing a set of `.mdc` files (e.g., Python conventions, security rules, etc.), and then include it under `.cursor/rules/shared/` in all your projects. This way, updates to the central rules can be pulled into all projects easily. The idea of sharing project rules in a centralized repository has been discussed in the Cursor community, as it enables teams to **“collaborate together on developing mature, open source cursor rules”** across multiple codebases ([Scan for Project Rules in Subdirectories of .cursor/rules - Feature Requests - Cursor - Community Forum](https://forum.cursor.com/t/scan-for-project-rules-in-subdirectories-of-cursor-rules/61534#:~:text=If%20Cursor%20did%2C%20then%20we,important%20step%20in%20that%20direction)). (Make sure to configure any submodule directories in `.cursor/rules` to be ignored by Cursor’s own indexing if necessary, to avoid double-counting them ([Serious performance issues Version: 0.47.5 #2873 - GitHub](https://github.com/getcursor/cursor/issues/2873#:~:text=cursor%20rules%20or%20the%20rules,made%20it%20unreliable%2C%20most)).)  
- **Cursor Settings Export:** In absence of a global file, a lightweight approach is to document the recommended global rules for your team (e.g., in your README or a wiki) and have each member add them to their Cursor Settings manually. This is less ideal for enforceability, but can cover things like “Always format answers in Markdown” if that’s a universal preference.

In terms of discoverability, treat the `.cursor/` directory as you would other configuration directories like `.vscode/` or `.github/`. It’s a well-known location once developers are aware of Cursor. You may want to include a short note in your project README to explain what Cursor Rules are and that the project contains custom AI helper rules – this helps developers (especially new ones) understand why the AI might be responding in a certain way.

**Comparison to Other Tools:** The practice of organizing Cursor rules draws inspiration from how other development tools manage configurations:

- *ESLint/Prettier:* These use config files (like `.eslintrc.json` or `.prettierrc`) often stored in the repo root (or in specific folders) to enforce coding standards. Similarly, Cursor’s `.mdc` rule files enforce coding practices but via AI guidance instead of compiler/linter errors. ESLint configs can extend shareable presets; in Cursor, we emulate this by sharing rule files or including common rules via `@` references (see Section 4).  
- *.vscode Folder:* Teams often check in a `.vscode/` folder with workspace settings or recommended extensions. The `.cursor/` folder is analogous – it’s a workspace-specific configuration (for AI behavior). Both are hidden directories intended to travel with the code.  
- *EditorConfig:* EditorConfig files define formatting rules per file type using glob patterns. Cursor rules also use glob patterns to target files, achieving a similar per-file-type customization ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=,when%20the%20rule%20is%20applied)). For example, just as EditorConfig might specify different indent settings for `*.py` vs `*.html`, you can have Cursor rules that apply only to `*.py` (perhaps enabling Python-specific AI tips) and different rules for `*.js` files, etc.  
- *Git Hooks/CI Config:* Many projects have a `.github/` or CI config files to ensure certain checks run for everyone. In a way, a set of Cursor rules is like an “AI pair programmer config” – ensuring every developer’s AI assistant follows the same playbook. Just as you’d keep CI config in the repo to standardize it, you keep Cursor rules in the repo for consistency.

**Tip:** Maintain a clean structure by grouping related rules and naming them clearly. Over time, your team might accumulate many rules (covering frameworks, testing, security, etc.), so organize them logically (by technology or by purpose). The example below shows a structured layout by domain, which is one of the suggested patterns ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=Structure%20rules%20by%20domain%3A)):

```text
.cursor/rules/
├── 01_general.mdc        # (Prefix numbers can be used to sort if desired)
├── 02_frontend/
│   ├── react.mdc
│   └── styling.mdc
├── 03_backend/
│   ├── fastapi.mdc
│   ├── pydantic.mdc
│   └── database.mdc
└── 04_devops/
    └── docker.mdc
```

This is just an example – choose what fits your project. The key is that any developer can open the `.cursor/rules` folder and quickly see what rules exist and for what purpose, much like they would with other config directories.

## 3. Rule Definition and Syntax (Writing Effective Rules)  
Each Cursor rule is defined in a Markdown file (`.mdc` extension) with a specific structure. The file consists of a YAML front-matter at the top (delimited by `---` lines) followed by the rule content in Markdown. Here’s a breakdown of the components:

**YAML Front Matter Fields:** The front matter defines metadata that Cursor uses to manage the rule. Common fields include: 

- **`description`:** A short sentence or phrase describing *when or how to apply this rule*. This is not just a comment – it’s shown to the AI to help it decide if the rule is relevant ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=)). Think of it as the rule’s “trigger” description. For example: `description: "Guidelines for FastAPI endpoints (use this in API code)"` or `description: "Always apply in any situation"` for an always-on rule ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=1.%20,such%20as)) ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=,situation%20globs%3A%20alwaysApply%3A%20true)). Writing a clear description is crucial, as many users initially assumed this was just for humans, but it actually guides the AI’s logic ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=,any%20situation)).  
- **`globs`:** One or multiple file pattern globs (wildcard patterns) indicating which files or paths this rule is associated with ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=An%20,content%20that%20looks%20like%20this)). If any currently open or linked file matches the glob, Cursor will inject this rule’s content automatically into context ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=An%20,content%20that%20looks%20like%20this)). Globs can be as broad or specific as needed, and you can include multiple patterns or even exclude patterns. For example: 
  ```yaml
  globs:
    - "app/api/**.py"
    - "app/schemas/**.py"
    - "!app/api/legacy/**"   # (exclude legacy folder)
  ``` 
  This might be used in a FastAPI rule to target all API and schema Python files except those in a legacy subfolder. Using strategic glob patterns helps scope rules precisely to relevant parts of the project ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=Create%20focused%2C%20hierarchical%20patterns%3A)). If you leave `globs` blank or omit it, the rule won’t auto-attach based on files (it could still be invoked by description or manually).  
- **`alwaysApply`:** A boolean (`true/false`) indicating if the rule should be injected into *every* AI conversation for this project, regardless of file context ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=,situation%20globs%3A%20alwaysApply%3A%20true)). Setting `alwaysApply: true` essentially makes it a global project rule. Use this sparingly for your most general rules, because too many always-on instructions can consume token context. Often a single “global” rule file with `alwaysApply:true` can serve as the universal project preamble, and more specific rules can remain conditional. If `alwaysApply` is false or not set, then the rule will only come into play via globs or by the AI picking it based on the description.  

These fields correspond to the “Rule Type” options discussed earlier. For example, an **Always** rule will have `alwaysApply: true`. An **Auto-Attached** rule will have specific `globs` (and typically `alwaysApply: false`). An **Agent-Requested** rule might have no globs and rely on its description. A **Manual** rule could simply not match anything unless called (one might even omit the description to ensure it doesn’t get pulled unless explicitly fetched). The Cursor UI’s rule editor sets these fields for you when you choose options, but you can also edit them by hand in a text editor (in fact, some users prefer external editors due to past UI quirks ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=Editing%20Limitations))).

**Rule Content (Body):** After the YAML front matter, you write the actual instructions/guidance in markdown format. This can include headings, bullet lists, code blocks, or plain text – whatever conveys the rule best. The content is what the AI will consider or follow when the rule is active. Here are some best practices for writing rule content:

- **Be Clear and Concise:** Write instructions in a straightforward manner, as if you’re writing guidelines for a human developer – because the AI will parse it similarly. Use bullet points or numbered lists for individual rules or steps, and use section headings (`##`, `###`, etc.) to organize if the content is long. Short, direct sentences work best. For example: *“Use def for sync endpoints and async def for async endpoints”*, *“Validate inputs with Pydantic models (no raw `dict` parsing) ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=Object%2C%20Return%20an%20Object%20,routes%2C%20utilities%2C%20static))”*, or *“Follow FastAPI’s naming conventions for path operations (e.g. use nouns for route names)”*. Each bullet should ideally be a single guideline or fact. This helps the AI easily digest the points.  
- **Include Essential Context:** If your project uses specific frameworks, libraries or patterns, mention them. E.g., a rule might start with a high-level blurb like: “You are working on a **FastAPI** web service with **Pydantic** for data models and **SQLAlchemy** for DB access” – essentially enumerating the tech stack. This primes the AI with relevant context. In fact, a common pattern in community examples is to literally begin the content with a phrase like *“You are an expert in Python, FastAPI, and scalable API development”* ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=You%20are%20an%20expert%20in,)) to set the AI’s persona/knowledge domain. This can be followed by key principles or project goals. A high-level overview at the top of the rule content provides valuable context and “sets the stage” for the more granular rules that follow ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=Image%3A%20Cursor%20Rules%20High%20Level,Overview)). For instance, you might have a section **“Project Overview”** that briefly describes what the system does (especially if it’s a specific domain like healthcare or finance – so the AI doesn’t suggest irrelevant solutions). Keep this overview to a few sentences or bullets.  
- **State Best Practices and Conventions:** The meat of the rule file will usually be a list of guidelines. These can cover coding style (e.g., *“Use type hints on all functions”*, *“Organize FastAPI routers in separate modules”*), architectural decisions (*“Prefer dependency injection over global variables in FastAPI”*, *“Use Pydantic `BaseModel` for request and response schemas ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=Object%2C%20Return%20an%20Object%20,routes%2C%20utilities%2C%20static))”*), or even team processes (*“All new endpoints must include at least one unit test”*). If you have an established style guide or if you’re following known best practices (like the official FastAPI docs or Pydantic usage recommendations), you can summarize those points here. For example, an excerpt of rule content might look like: 

  > **FastAPI Best Practices**  
  > - Use **async** endpoints for any operation that involves I/O or waiting (database calls, HTTP requests) ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=Object%2C%20Return%20an%20Object%20,routes%2C%20utilities%2C%20static)).  
  > - Leverage FastAPI’s **dependency injection** (`Depends`) for common logic (auth, DB sessions) instead of global objects.  
  > - **Pydantic Models**: Define request/response schemas with Pydantic models (avoid using raw dictionaries) ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=Object%2C%20Return%20an%20Object%20,routes%2C%20utilities%2C%20static)). This ensures validation and clear data contracts.  
  > - Follow HTTP semantics: use proper status codes and exception handling (e.g., FastAPI’s HTTPException for errors).  
  > - Document endpoints with docstrings or comments, so they appear in the automatic docs.

  Each of the above bullet points could be part of the `fastapi.mdc` rule content, for example. They give the AI concrete dos and don’ts. Notice the use of **imperative tone** (“Use X”, “Do Y”, “Avoid Z”) – this is intentional, as a direct tone helps ensure the AI takes them as instructions ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=4)). You can also include **rationale** in parentheses or following sentences if needed, which might help the AI understand why the rule exists (though brevity is usually preferred to save token space). For instance, the Pydantic bullet above could add “(to enforce data validation and clarity)” to reinforce the point.  
- **Highlight Deprecated or Forbidden Patterns:** If there are certain practices the project wants to avoid, explicitly call them out. For example: *“Do NOT use `sqlalchemy.orm.Session` directly in route functions – use the dependency-provided session instead.”* If some legacy approach exists in the codebase that new code should not imitate, mention it (and mention the correct approach). Marking deprecated patterns clearly helps the AI refrain from suggesting them ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=4)). Using **bold** or all-caps “NOT” can emphasize prohibition, but don’t overuse formatting or it might get lost; a concise “do not use X, instead use Y” is effective.  
- **Provide Examples (if applicable):** Cursor’s model can read and learn from examples you provide in the rules. If you have space, including a small code snippet example can be very powerful. For instance, you might show a mini-template of a FastAPI endpoint: 

  ```markdown
  ```python
  # Example: Proper FastAPI endpoint structure
  @router.post("/items/", response_model=ItemOut)
  async def create_item(item: ItemIn, db: Session = Depends(get_db)):
      # Validate and process the item
      ...
      return saved_item
  ```  
  ```  

  And perhaps a counter-example (if helpful) of what *not* to do, with a note. Examples should be clearly labeled and explained. According to best practices, use proper Markdown formatting for code and add comments to highlight important parts ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=Image%3A%20Cursor%20Rules%20Example%20Patterns)). However, keep examples succinct – they should illustrate one or two points, not an entire program. Realize that examples do consume context tokens, so balance the benefit. Often, one good example is worth ten bullet points of explanation. If your team has standard code templates or patterns (like how to structure a Pydantic model or how to open a DB transaction), including a short example in the rules can train the AI to produce code matching that style.  
- **Verification or Checklist:** Some advanced rule authors end their rule files with a “Checklist” or “Verification steps” section – essentially reminding the AI to double-check certain things in its output ([How to write great Cursor Rules | Trigger.dev](https://trigger.dev/blog/cursor-rules#:~:text=6)). For instance, you might have: “**Verification:** Before finalizing, ensure that all data models have type annotations and that no TODO comments remain in the code.” This is less about instructing the AI during generation and more about prompting it to review its answer for compliance. This idea is inspired by prompt engineering techniques (making the AI self-review). It’s optional, but in team settings it could reduce the need for human corrections. If you include such steps, prefix them in a way the AI will interpret as an instruction (e.g., “Before returning your answer, do X”).  

**Schema and Example:** To illustrate, below is an example of what a simple `.mdc` file might look like for our FastAPI/Pydantic scenario:

```yaml
---  
description: FastAPI service rules (attach to FastAPI-related files)  
globs:  
  - "app/api/**.py"  
  - "app/schemas/**.py"  
---  

# FastAPI Project Guidelines

- **Framework**: This project uses **FastAPI** for the web API and **Pydantic** for data models. Assume an expert knowledge of these.  
- **Async & Sync**: Use `async def` for endpoints that perform I/O or long operations; use normal `def` only for purely CPU-bound logic ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=Object%2C%20Return%20an%20Object%20,routes%2C%20utilities%2C%20static)).  
- **Routing**: Organize endpoints using APIRouter instances (one router per feature module). Include clear prefixes and tags for each router.  
- **Dependency Injection**: Use FastAPI’s `Depends()` for shared utilities (e.g., DB session, authentication) – *do not import global objects directly*.  
- **Data Models**: Define request and response schemas with **Pydantic** models (e.g., `ItemCreate`, `ItemOut` classes) ([Rules for FastAPI | Cursor Directory](https://cursor.directory/rules/fastapi#:~:text=Object%2C%20Return%20an%20Object%20,routes%2C%20utilities%2C%20static)). Avoid using untyped dicts – always use Pydantic for validation.  
- **Error Handling**: Use `HTTPException` for client errors and ensure correct HTTP status codes. Log server-side errors and do not expose internal tracebacks.  
- **Testing**: Every new endpoint should have a corresponding test (use pytest); the AI should prefer writing test stubs if asked.  

*(... any additional rules ...)*

```

In the above example, the YAML front matter has a description and globs (this rule would auto-attach when editing files in `app/api` or `app/schemas`). The content then lists a set of guidelines covering various aspects of FastAPI usage. Note how it mixes general statements with specific directives, and uses bold to highlight key terms. Real rule files can be much longer and more detailed – some projects have dozens of bullet points or multiple sections in a single rule file if needed. The content can also reference other files: for instance, you could include something like “See @README.md for high-level system architecture” – by prefixing a filename with `@`, you instruct Cursor that the file’s content is relevant context ([How to use Cursor Rules | Instructa Courses](https://www.instructa.ai/en/blog/how-to-use-cursor-rules-in-version-0-45#:~:text=Step%20,)). The AI can then fetch that file’s content when using the rule. This is a powerful way to avoid duplicating information: you might keep large documentation in actual markdown files and just refer to them in the rule, thus pulling them in only when needed (and saving token space most of the time). For example, if you have an architecture overview document or a database schema in your repo, you can do: “Refer to @docs/ARCHITECTURE.md for the overall system design.” When the rule triggers, the AI knows it can load that file if necessary to answer a question.

**Modularity and Reuse:** It’s wise to keep rules modular. Instead of one gigantic rule file, use multiple files and have them reference each other if there are dependencies. Cursor allows including one rule file’s content into another via the `@filename.mdc` syntax ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=,standards)). For example, you might have a `base_python.mdc` with general Python guidelines, and in your `fastapi.mdc` you put at the top of the content `@base_python.mdc` to pull those in (so you don’t have to repeat generic Python rules in every framework-specific file). The community recommends creating composable rule sets in this way ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=,standards)). Just be careful to avoid cyclical references. With a bit of planning, you can achieve a hierarchy: e.g., a `base.mdc` (language-agnostic fundamentals), `python.mdc` (Python-specific conventions), and then `fastapi.mdc` which brings in `python.mdc` and adds FastAPI-only rules. This approach is analogous to how one might extend configurations in ESLint or Prettier rather than rewriting everything per project.

**Maintainability:** Treat rule files as living documentation for your AI assistant. Revisit them whenever your tech stack changes or if you find the AI giving undesired output (that might be corrected by adding a new rule or tweaking an existing one). Encourage team members to contribute to rules. For instance, if someone finds the AI keeps using a deprecated function, add a rule bullet to not use that function and to use the new one instead. Over time you’ll refine rules to cover common pitfalls. Just as code evolves, so should your AI’s guidance. Keep an eye on Cursor’s updates too – as the AI models improve or the Cursor engine changes how it uses rules, you might adjust the wording or structure of rules for optimal results. In v0.47, the system is already quite advanced in understanding the natural language descriptions ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=,any%20situation)), so focus on clarity and let the AI handle the rest.

## 4. Analogous Systems and Inspirations for Rule Management  
When establishing a standardized system for Cursor rules, it helps to draw inspiration from other configuration and rule management systems developers are familiar with. Here are some analogous systems and how their best practices can inform a Cursor rules strategy:

- **Linters and Code Formatters (ESLint, Prettier, etc.):** These tools use config files to enforce style and best practices across a codebase. For example, ESLint has shareable rule sets and the ability to target specific files (via overrides or plugin configurations). Similarly, with Cursor rules, you can maintain a *“style guide for the AI”*. One practice from ESLint is to keep the config in source control and even publish shareable configs (like extending `eslint-config-airbnb`). For Cursor, while we can’t “extend” rules in the same way, we can share common rule files across projects (via copy or submodule, as mentioned in Section 2) and even community-source them. In fact, the Cursor community has begun creating **awesome-cursorrules** repositories – analogous to shareable linter configs – where a set of tried-and-true rules for certain languages or frameworks are made available to others ([Created a collection of 879 .mdc Cursor Rules files for you all - Showcase - Cursor - Community Forum](https://forum.cursor.com/t/created-a-collection-of-879-mdc-cursor-rules-files-for-you-all/51634#:~:text=I%E2%80%99m%20sure%20most%20of%20you,cursorrules)). Also, lint configs often have **version control** (to update rules as the language evolves); likewise, treat your Cursor rules as versioned config that evolves. Consider using semantic or descriptive commit messages when updating rules (e.g., “Update cursor rule for FastAPI v0.95 changes” akin to “Update ESLint rules for React 18”). This makes rule changes transparent to the team.

- **EditorConfig:** EditorConfig files allow defining coding style per file type in a simple, declarative manner. They often live in the project root and use a INI-like syntax with section per glob pattern (e.g., `[*.py] indent_style=space`). The analogous concept in Cursor is the use of **globs in rule front-matter** to target certain files ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=,when%20the%20rule%20is%20applied)). The best practice from EditorConfig is to be as specific as necessary with patterns and to put more specific rules lower in the file. In Cursor, each `.mdc` is separate, but you should still avoid pattern conflicts or duplications. If two rules target the same file with contradictory advice, the AI might get confused. Thus, design your glob patterns carefully (you can use include/exclude as shown earlier) to create a hierarchy (e.g., a broad rule for “all Python files” and a narrower one for “FastAPI Python files”, where the latter’s description and context make it supersede the former in relevant situations). This mirrors how EditorConfig or even stylelint would override rules based on specificity. Additionally, EditorConfig is intentionally simple (only key settings); by analogy, keep each Cursor rule focused on a single theme or area. Don’t mix front-end and back-end guidelines in one file; instead, split them so they can be applied independently.

- **Software Design Rules / Architecture Decision Records:** In large projects, teams often document architectural guidelines or decisions (ADR documents) to ensure consistency. Cursor rules can be seen as an executable form of those architecture guidelines. For example, an ADR might state “We will use Pydantic for data validation in all services.” In Cursor, a rule enforces that by telling the AI about Pydantic usage. Drawing from ADR practices: each rule should have a clear purpose and ideally a short rationale (though in the rule file we keep it brief). Just as ADRs are usually stored in a `docs/` folder and indexed, you can keep verbose explanations in docs and just reference them from rules (using `@docname`). This way, your documentation and your AI rules stay in sync. 

- **Prompt Engineering Frameworks:** Outside of Cursor, there are emerging frameworks for managing prompts and AI agents (LangChain, prompt templates, etc.). Those often emphasize modularity (breaking prompts into reusable parts), versioning of prompts, and testing prompt outcomes. For Cursor rules, modularity is achieved via multiple files and references, as discussed. **Versioning** is inherently handled by git if you commit the rule files – it’s good practice to review changes to .mdc files in code reviews, similar to how one would review changes in CI config or infrastructure-as-code. **Testing** Cursor rules can be informal – e.g., after adding a rule, ask the AI something that should trigger it and see if the answer reflects the rule. In a team, you might even script some automated checks (for example, a small script that uses Cursor’s CLI (if available) to query something and verify the output contains or follows the rule, though such automation is non-trivial and beyond typical usage). The key idea is borrowed from software testing: if you rely on certain rules to always be followed, verify them when you update those rules or update Cursor versions.  

- **Hierarchy and Overrides:** Many config systems allow hierarchy (user-level config, project-level config, etc.). Cursor similarly has user (global) and project (local) rules. A best practice from tools like Git or VSCode settings is *“project settings override user settings for that project.”* With Cursor, global (user) rules will always be present, but you might design them to be very general, and let project rules be more specific. If there’s potential conflict, assume the project-specific rule should win (and craft the descriptions such that the AI knows the context – e.g., your global rule might say “use polite tone” and a project rule says “use concise technical tone for this project’s docs”; the AI will try to reconcile, but context suggests in this project, concise technical is intended). If truly needed, you can explicitly tell the AI in a project rule to *ignore* certain global instructions. For example, “In this project, abbreviations are okay even if global rules prefer formal language.” Use such overrides carefully and sparingly.

- **Inspirations from Constitutional AI / Guidelines:** At a higher level, OpenAI and Anthropic have experimented with “constitutions” for AI – essentially a list of rules the AI should always follow (like “don’t produce harmful content”, etc.). Cursor rules are a much more domain-specific analogue: *you are writing a mini-constitution for your AI specific to your project.* The way Anthropic’s **Constitutional AI** is structured (a list of principles the AI reflects on) is similar to how you might structure core principles in a global rule file (e.g., “Core Principles: 1. Security first – no credentials in code. 2. Performance – prefer O(n) over O(n^2) algorithms,” etc.). The lesson here is to phrase these principles clearly and let the AI “reflect” them in its output ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=,Question%20every%20assumption%20and%20inference)) ([Mastering Cursor Rules: A Developer's Guide to Smart AI Integration - DEV Community](https://dev.to/dpaluy/mastering-cursor-rules-a-developers-guide-to-smart-ai-integration-1k65#:~:text=3.%20THINKING%20PROCESS%20,CORE_PRINCIPLES)). Of course, your rules are directly injected, not something the AI separately evaluates, but ensuring they are stated as principles can give the AI a sense of their importance.

In summary, manage Cursor rules as you would any important configuration for a project: keep them organized, make their intent clear, reuse and share where possible, and adapt practices from known config systems (like grouping, inheritance, version control). The ultimate goal is that any developer on the team can open the rules and understand the AI’s “playbook,” and that the rules remain maintainable as the project grows.

- **Naming Conventions and File Organization:** While not formally standardized, a few conventions have appeared:
  - Use a **`global.mdc` or `instructions.mdc`** file for broad rules that apply always. If you have a large project, prefixing with numbers or letters can sort them (e.g., `1_global.mdc, 2_backend_fastapi.mdc, 3_backend_pydantic.mdc, 4_frontend_xyz.mdc`), though alphabetical sorting is usually only for human convenience (Cursor doesn’t require a specific order, it loads all applicable rules).
  - Store any large text or reference materials as normal markdown/docs in your repo, and not as rules (to avoid bloating the rules). Then reference those in rules by `@filename`. Community examples often show rules referencing things like `@CONTRIBUTING.md` or `@architecture_diagram.png` (if images converted to text) – basically leveraging the Cursor context system. This way the rule files remain focused on *direct instructions*, and heavy documentation stays separate.

These indicate that the community is actively trying to streamline rule creation and sharing. For your team, this means you’re not alone – you can leverage these tools. A pragmatic approach could be: use a generator to create an initial rule file with common FastAPI guidelines, then iterate on it with the team’s specific decisions. The community-shared rules are a baseline, but your project’s “flavor” will dictate the final content.

**Emerging Conventions:** In summary, while formal standards are still forming, a few conventions can be noted:

- **Separation of Concerns:** one rule file per major concern or per file type (mirroring how one might have separate lint configs for separate packages).  
- **Descriptive Descriptions:** writing the rule’s description as an if-clause or context (e.g., “Use for X”, “When in Y context”) rather than something abstract. This helps the AI and also any human reading it.  
- **Committing and Sharing:** treating the `.cursor/rules` folder as a first-class citizen of the repo (most serious Cursor users do this now, given the emphasis in docs and community guides on sharing rules via Git ([A Deep Dive into Cursor Rules (> 0.45) - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/a-deep-dive-into-cursor-rules-0-45/60721#:~:text=1,everyone%20pulls%20the%20same%20rules))). It’s increasingly expected that if you clone someone’s project that uses Cursor, you look at `.cursor/rules` to understand custom behaviors – similar to checking for a `CONTRIBUTING.md` or `.env.example`.  
- **Continual Refinement:** Many users report tweaking rules as they work with the AI – this iterative refinement (sometimes even asking the AI to help improve its own rules!) is becoming a norm. It’s understood that writing good AI rules is an ongoing process, not a one-shot task.

Finally, keep an eye on official Cursor releases and community forums for updates. Cursor IDE is evolving rapidly; for instance, if version 0.48+ introduces changes to how rules are formatted or applied, you’ll want to adapt your standardized approach accordingly. The documentation and changelogs (and people like Cursor’s co-founders on social media) often highlight improvements to the rules system ([Cursor 0.47 - Reddit](https://www.reddit.com/r/cursor/comments/1j8pp1j/cursor_047/#:~:text=Rules%3A%20Allow%20nested%20,MCP%3A%20Added%20global%20server)). With version 0.47, we got a solid foundation for multi-file rules and better project integration. By following the practices above – clear definition of rules, proper organization, syntax mastery, borrowing config management wisdom, and learning from community examples – your team can establish a robust, shareable, and maintainable system of Cursor rules that greatly enhances your FastAPI + PydanticAI development workflow.
