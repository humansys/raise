# ADR-012: IDE Context Architecture

## Context

### Problem

AI-assisted development tools (Claude Code, Cursor, Windsurf, etc.) each provide a mechanism for "always-on" project instructions:

| IDE | Mechanism | Location |
|-----|-----------|----------|
| Claude Code | CLAUDE.md + CLAUDE.local.md | Project root |
| Claude Code | Auto-memory | ~/.claude/projects/.../memory/ |
| Claude Code | Hooks | .claude/scripts/ |
| Cursor | .cursorrules | Project root |
| Windsurf | .windsurfrules | Project root |

RaiSE was using 3 Claude Code-specific mechanisms simultaneously:
1. `.claude/scripts/session-init.sh` — hook to preload identity files
2. `~/.claude/projects/.../memory/MEMORY.md` — auto-memory with process rules
3. `~/.claude/projects/.../memory/cli-reference.md` — CLI reference

This created:
- **Platform lock-in:** None of this works outside Claude Code
- **Fragmentation:** 5 files across 3 mechanisms for context that serves one purpose
- **Brittleness:** Silent failure if any file/mechanism changes
- **Non-determinism:** AI editing memory files with inference (no guarantee of correctness across models/tools)

### Principle at Stake

**Platform Agnosticism** (RaiSE constitution): "RaiSE works wherever Git works. No dependency on GitHub, GitLab, Bitbucket, or any specific platform."

This extends to IDE tooling. RaiSE should not depend on Claude Code's proprietary mechanisms when a platform-agnostic alternative exists.

## Decision

### Two-Layer Architecture

```
CANONICAL SOURCE (platform-agnostic, in repo):
  .raise/rai/identity/     → Rich identity documentation
  .raise/rai/memory/        → Graph, patterns, calibration
  .raise/governance/        → Process rules, gates, constitution

PROJECTED PROMPT (IDE-specific, generated):
  CLAUDE.md                 → Projection for Claude Code
  .cursorrules              → Projection for Cursor (future)
  .windsurfrules            → Projection for Windsurf (future)
```

### Rules

1. **The canonical source lives in `.raise/`** — always. This is the single source of truth.

2. **IDE prompt files are projections** — generated from `.raise/`, not hand-edited. They contain compressed, AI-optimized representations of the canonical content.

3. **Projections are generated by `rai init`** — no new commands. `rai init` already generates project structure; it now also generates the IDE prompt file.

4. **No proprietary mechanisms** — no hooks, no memory directories, no platform-specific features for core functionality. If a mechanism only works in one IDE, don't depend on it.

5. **Updates are deterministic** — when canonical source changes, re-run `rai init` to regenerate the projection. The AI never edits the projection directly.

6. **On-demand context via CLI** — rich, dynamic content (patterns, coaching, session narrative) comes from `rai session start --context`. The projected prompt only contains stable, rarely-changing content.

### What Goes in the Projected Prompt (CLAUDE.md)

The projected prompt answers: **"What does Rai need to BE Rai before any command runs?"**

| Content | Purpose | Why Always-On |
|---------|---------|---------------|
| Identity (values, boundaries, principles) | Behavioral priming | Without this, Rai is generic Claude |
| Communication style | Collaboration calibration | Without this, Rai is verbose and polite |
| Work lifecycle + gates | Methodology adherence | Without this, Rai skips process |
| CLI reference (compact) | Operational capability | Without this, Rai fumbles commands |
| "Run /rai-session-start" | Memory bootstrap | Without this, Rai works without context |

Format: Markdown-KV compact where applicable (proven by RAISE-166 as top performer for AI consumption).

### What Does NOT Go in the Projected Prompt

| Content | Where Instead | Why |
|---------|---------------|-----|
| Accumulated patterns | CLI `--context` | Dynamic, grows over time |
| Coaching/calibration | CLI `--context` | Session-specific |
| Session narrative | CLI `--context` | Changes every session |
| Rich identity prose | `.raise/rai/identity/` | Canonical source, not for AI consumption |
| Current focus/active work | CLAUDE.local.md | User-maintained, personal |

### CLAUDE.local.md

CLAUDE.local.md remains user-maintained (gitignored) for personal/temporal context:
- Current epic/story focus
- Client-specific context
- Personal workflow notes

This file is NOT generated by `rai init` — it's the user's space.

## Consequences

### Positive
- **Platform agnostic:** Same canonical source generates prompts for any IDE
- **Deterministic:** `rai init` produces identical output from identical input
- **Simple:** 2 always-on files (CLAUDE.md + CLAUDE.local.md) + 1 CLI command
- **Reduced fragmentation:** From 5 files across 3 mechanisms to 1 generated file
- **Future-proof:** New IDEs supported by adding a projection template

### Negative
- **Manual regeneration:** When canonical source changes, user must re-run `rai init` (low frequency — identity/process rarely change)
- **Two representations:** Canonical (rich) and projected (compact) can drift if `rai init` isn't run

### Migration
- Delete `.claude/scripts/session-init.sh`
- Replace `~/.claude/projects/.../memory/MEMORY.md` with redirect
- Delete `~/.claude/projects/.../memory/cli-reference.md`
- Generate new CLAUDE.md with consolidated content
- Add CLAUDE.md generation to `rai init` (future story)

## Alternatives Considered

### Keep hooks + memory dir
Rejected: Platform lock-in, fragmentation, brittleness.

### Put everything in CLI (no always-on)
Rejected: Rai needs identity priming before first command. Without always-on behavioral primes, the first interaction in any session is with generic Claude, not Rai.

### Use .raise/ files directly (no projection)
Rejected: IDEs read specific files (CLAUDE.md, .cursorrules). We can't change their discovery mechanism. The projection is necessary to meet IDEs where they are.
