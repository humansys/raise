# ISSUE-003: Directory Structure Ontology Confusion

> **Status:** RESOLVED
> **Priority:** High (blocks clean F&F experience)
> **Created:** 2026-02-05
> **Resolved:** 2026-02-05
> **Resolution Commit:** 2229f5e
> **Related:** E14 Rai Distribution, ADR-022

## Resolution

Implemented `.raise/rai/` nested structure that separates concerns:

| Directory | Purpose | Contents |
|-----------|---------|----------|
| `.raise/` | Framework presence (per-project) | manifest.yaml, graph/, future config |
| `.raise/rai/` | AI partner state | memory/, telemetry/, identity/ |
| `~/.rai/` | Global developer profile | developer.yaml (unchanged) |

**Key insight:** RaiSE framework ≠ Rai AI partner. The framework is the methodology; Rai is the AI that uses it. Nesting `rai/` inside `.raise/` preserves this distinction while keeping a single visible directory.

**Implementation:**
- Centralized 17 path constants + 10 helper functions in `config/paths.py`
- Updated 16 source files to use path helpers
- Updated 11 test files to match new paths
- All 1106 tests pass, 90.47% coverage

## Problem Statement

The current directory structure conflates three concerns:

1. **RaiSE framework development** (raise-commons as the framework repo)
2. **RaiSE framework distribution** (what gets bundled in raise-cli)
3. **User project governance** (what a user project should have)

New users running `rai init` on their project will be confused about what directories they need and what each one means.

---

## Current Structure Issues

### 1. `.raise/` vs `.rai/` — Both feel like "RaiSE stuff"

| Directory | Purpose | User confusion |
|-----------|---------|----------------|
| `.raise/` | Methodology engine (katas, gates, templates) | "Is this config or engine?" |
| `.rai/` | AI partner state (identity, patterns) | "Why two dot-directories?" |

**Root cause:** `.raise/` was designed as per-project methodology config, but E14 bundles base methodology in raise-cli. Do users need `.raise/` at all?

### 2. `dev/decisions/` vs `governance/decisions/` — Split ADRs

| Location | Files | Status |
|----------|-------|--------|
| `dev/decisions/` | 22 ADRs | Actually used |
| `governance/decisions/` | 1 file | Nearly empty |

**Root cause:** We documented `governance/decisions/` as canonical but use `dev/decisions/` in practice.

### 3. `dev/` mixes framework-dev with project-dev

Current `dev/` contains:
- `epic-e*.md` — raise-cli specific epic scopes
- `components.md` — generic project artifact
- `architecture-overview.md` — generic project artifact
- `sops/`, `prompts/`, `agents/` — internal raise team tooling
- `parking-lot.md` — generic project artifact

**Root cause:** raise-commons is both the framework repo AND a dogfooding project.

### 4. `.claude/skills/` vs `.raise/skills/` — Same name, different things

| Location | Contents | Purpose |
|----------|----------|---------|
| `.claude/skills/` | Markdown files | Claude Code slash commands |
| `.raise/skills/` | YAML files | Framework operation contracts |

**Root cause:** "Skill" is overloaded. Claude Code skills ≠ RaiSE skills.

### 5. `framework/` only makes sense for framework repo

Users adopting RaiSE don't maintain the framework textbook — they READ it. Having `framework/` in a user project is confusing.

### 6. `governance/` feels enterprise-heavy

Most developers expect `docs/` for documentation, not `governance/`. The term "governance" implies bureaucracy.

## Proposed Clean Structure

### For User Projects (the common case)

```
user-project/
├── .rai/                      # AI partner + RaiSE config (ONE directory)
│   ├── identity/              # Rai's identity (from base)
│   │   ├── core.md
│   │   └── perspective.md
│   ├── memory/                # Patterns, calibration, sessions
│   │   ├── patterns.jsonl
│   │   ├── calibration.jsonl
│   │   └── sessions/
│   ├── config.yaml            # Project RaiSE configuration
│   ├── manifest.yaml          # Project detection result
│   └── telemetry/             # Local signals
│
├── .claude/                   # Claude Code specific
│   ├── skills/                # Slash commands (user-defined)
│   └── settings.local.json    # Local settings
│
├── docs/                      # Project documentation (familiar name)
│   ├── decisions/             # ADRs
│   │   └── adr-001-*.md
│   ├── architecture.md        # Architecture overview
│   └── README.md              # Docs index
│
├── CLAUDE.md                  # AI instructions (generated by raise)
├── CLAUDE.local.md            # Personal context (git-ignored)
│
└── src/, tests/, etc.         # Their actual code
```

**Key changes:**
1. **No `.raise/` for users** — methodology bundled in raise-cli
2. **`.rai/` is the single AI-related directory** — cleaner mental model
3. **`docs/` instead of `governance/`** — familiar to all developers
4. **ADRs in `docs/decisions/`** — standard industry practice
5. **No `framework/`** — users read docs, don't maintain them
6. **No `work/`** — users use their existing issue tracker

### For raise-commons (the framework repo)

```
raise-commons/
├── .rai/                      # Emilio's Rai (dogfooding)
├── .claude/                   # Claude Code settings
│
├── src/rai_cli/             # CLI source code
│   └── rai_base/              # Bundled base Rai (E14)
│
├── framework/                 # PUBLIC textbook (what we publish)
│   ├── reference/             # Constitution, glossary
│   ├── concepts/              # Core concepts
│   └── getting-started/       # Tutorials
│
├── methodology/               # Methodology SOURCE (what gets bundled)
│   ├── katas/                 # Process definitions
│   ├── gates/                 # Validation criteria
│   └── templates/             # Output formats
│
├── docs/                      # Project documentation
│   ├── decisions/             # ADRs (consolidated from dev/ and governance/)
│   ├── architecture.md
│   └── components.md
│
├── dev/                       # Framework DEVELOPMENT only
│   ├── epics/                 # Epic scopes (raise-cli specific)
│   ├── sops/                  # Internal SOPs
│   └── experiments/           # Experiments
│
├── work/                      # Active WIP (raise-cli development)
│
├── CLAUDE.md                  # Generated
└── tests/, etc.
```

**Key changes:**
1. **Rename `.raise/` to `methodology/`** — clearer that it's SOURCE, not config
2. **Consolidate ADRs to `docs/decisions/`** — single location
3. **Move generic artifacts from `dev/` to `docs/`** — architecture, components
4. **Keep `dev/` for framework-specific** — epics, experiments
5. **Delete `governance/`** — consolidate into `docs/`

## Migration Path

### Phase 1: Pre-E14 (Now)
- Document the target structure (this issue)
- Decision: Do we migrate raise-commons or just define user structure?

### Phase 2: E14 Implementation
- `rai init` creates the clean user structure
- Base Rai goes to `.rai/` (not `.raise/`)
- No `.raise/` created for user projects

### Phase 3: Post-F&F (Optional)
- Migrate raise-commons itself to clean structure
- Consolidate ADRs
- Rename `.raise/` to `methodology/`

## Questions to Resolve

1. **Should users have `.raise/` at all?**
   - If methodology is bundled, why per-project config?
   - Custom katas/gates could live in `.rai/overrides/`

2. **What about `work/` for user projects?**
   - Most teams use Jira/Linear/GitHub Issues
   - Do we need `work/` or let users use existing tools?

3. **Should we migrate raise-commons now or after F&F?**
   - Migration is disruptive
   - But dogfooding the clean structure validates it

4. **Keep "governance" terminology anywhere?**
   - Governance-as-code is a core concept
   - But `docs/` is more approachable
   - Could use `docs/governance/` as subdirectory

## Appendix: Naming Alternatives

### For the AI partner directory

| Option | Pros | Cons |
|--------|------|------|
| `.rai/` | Short, Rai identity | Not obvious it's RaiSE-related |
| `.raise/` | Clear RaiSE branding | Conflicts with current usage |
| `.raise-ai/` | Clear but longer | Verbose |
| `.partner/` | Generic | Not branded |

**Recommendation:** Keep `.rai/` — it's established and Rai IS the AI partner.

### For documentation directory

| Option | Pros | Cons |
|--------|------|------|
| `docs/` | Industry standard | Generic |
| `governance/` | RaiSE terminology | Feels enterprise/bureaucratic |
| `project/` | Clear scope | Uncommon |

**Recommendation:** `docs/` — familiar to all developers.
