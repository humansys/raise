## Structuring AI Instructions (Anthropic/OpenAI Guidance)

**Clear, sequential prompts:** Both Anthropic and OpenAI stress writing **specific, step-by-step instructions** in a logical order. For example, the Claude guide advises specifying the desired output format and using numbered or bullet lists for sequential steps[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=,or%20completeness%20of%20steps%20matters). OpenAI’s GPT-4.1 guide likewise recommends starting with sections like **Role/Objective**, **Instructions**, **Reasoning**, **Output Format**, **Examples**, and **Context**[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=). In practice this means: define the agent’s role (often in a system prompt), then list _exactly_ what to do in order. Short, clear sentences with explicit constraints tend to work best[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=,or%20completeness%20of%20steps%20matters)[aws.amazon.com](https://aws.amazon.com/blogs/publicsector/why-your-ai-agents-give-inconsistent-results-and-how-agent-sops-fix-it/#:~:text=instructions%2C%20requiring%20it%20to%20fill,in%20the%20gaps%20itself)[cline.bot](https://cline.bot/blog/the-worst-instructions-you-can-give-an-ai-coding-agent#:~:text=Vague%20objectives%20like%20,a%20lottery%20with%20your%20tokens) (Confidence: **HIGH**).

**Use examples and formatting tags:** Both providers encourage embedding examples and using structure tags. Anthropic explicitly recommends wrapping few-shot examples in `<example>` (or `<examples>`) tags and including ~3–5 representative examples[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Use%20examples%20effectively). OpenAI similarly advises an **“Examples”** section and notes that any behavior shown in examples should also be spelled out in the instructions[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=examples,also%20cited%20in%20your%20rules). Employing markup (XML tags or Markdown headers) to separate sections is also emphasized: e.g. use `<instructions>…</instructions>` or Markdown `# Instructions`, `# Context`, `# Examples` etc[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Structure%20prompts%20with%20XML%20tags)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each). This clarity helps the model parse content unambiguously. Anthropic’s docs state that using descriptive tags for `<context>`, `<input>`, `<output>`, etc. **“reduces misinterpretation”**[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Structure%20prompts%20with%20XML%20tags). OpenAI’s GPT-4.1 guide similarly finds Markdown headers highly effective (and notes XML tags also perform well)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=2,inputs%20and%20outputs%20for%20each). Combining both methods – headings for main sections and XML tags to delineate blocks – is a strong pattern. (Confidence: **HIGH**; evidence: Anthropic & OpenAI docs[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Structure%20prompts%20with%20XML%20tags)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each).)

**Context and motivation:** Include _why_ the task matters. Provider docs advise giving the model some background or rationale (a “motivation”) to focus it on the goal[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Add%20context%20to%20improve%20performance)[aws.amazon.com](https://aws.amazon.com/blogs/publicsector/why-your-ai-agents-give-inconsistent-results-and-how-agent-sops-fix-it/#:~:text=instructions%2C%20requiring%20it%20to%20fill,in%20the%20gaps%20itself). For instance, Anthropic notes that explaining the purpose (even in one sentence) helps Claude “understand your goals and deliver more targeted responses”[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Add%20context%20to%20improve%20performance). The AWS Agent SOP guidelines echo this: agents fail often when instructions are ambiguous[aws.amazon.com](https://aws.amazon.com/blogs/publicsector/why-your-ai-agents-give-inconsistent-results-and-how-agent-sops-fix-it/#:~:text=instructions%2C%20requiring%20it%20to%20fill,in%20the%20gaps%20itself). Thus state assumptions/constraints up front (e.g. data formats, deadlines, constraints) in a **Context/Preamble** section. This background should be brief – just enough to justify the steps.

**System or role prompt:** Begin with a **system prompt** that assigns the agent’s role or persona (e.g. “You are a research assistant specializing in XYZ”[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Give%20Claude%20a%20role)). OpenAI and Anthropic both advise giving a role to shape tone and expertise[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Give%20Claude%20a%20role)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=). This top-line framing (usually one sentence in system message) anchors the agent’s behavior.

**Brief, complete instructions:** In summary, expert guidelines converge on **“short but complete”** instructions: use as few words as necessary to be precise, and break tasks into numbered steps[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=,or%20completeness%20of%20steps%20matters)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each). Avoid long-winded prose or philosophical asides. (We will discuss brevity more under _Token Efficiency_ below.)

## Agent Framework Patterns

**CrewAI (YAML tasks):** CrewAI defines tasks in YAML with two main fields: `description` (the process steps) and `expected_output` (the deliverable format)[docs.crewai.com](https://docs.crewai.com/en/concepts/tasks#:~:text=research_task%3A%20description%3A%20,topic%7D%20agent%3A%20researcher)[docs.crewai.com](https://docs.crewai.com/en/concepts/tasks#:~:text=reporting_task%3A%20description%3A%20,Formatted%20as%20markdown%20without). Their docs stress **single-purpose tasks** (“one clear objective, one output”)[docs.crewai.com](https://docs.crewai.com/en/guides/agents/crafting-effective-agents#:~:text=1). Each YAML task includes a concise natural-language `description` (what to do) and an `expected_output` template (how results should look). For example, a task might say, “_research top 5 trends in AI for 2025_” with an expected markdown list format[docs.crewai.com](https://docs.crewai.com/en/concepts/tasks#:~:text=research_task%3A%20description%3A%20,topic%7D%20agent%3A%20researcher). CrewAI explicitly recommends this YAML approach as “cleaner, more maintainable” than raw text prompts[docs.crewai.com](https://docs.crewai.com/en/concepts/tasks#:~:text=There%20are%20two%20ways%20to,defining%20them%20directly%20in%20code). (Confidence: **HIGH**.)

**LangChain:** LangChain agents can be given an optional `system_prompt` (string or structured `SystemMessage`) when instantiated[docs.langchain.com](https://docs.langchain.com/oss/python/langchain/agents#:~:text=You%20can%20shape%20how%20your,be%20provided%20as%20a%20string)[docs.langchain.com](https://docs.langchain.com/oss/python/langchain/agents#:~:text=more%20control%20over%20the%20prompt,features%20like%20Anthropic%E2%80%99s%20prompt%20caching). In practice, one can supply a Markdown or JSON-based system prompt (LangChain even allows a list of message objects). If no system prompt is provided, the agent infers instructions from the first user message. Thus, LangChain underlines providing a clear system message to **“shape how your agent approaches tasks”**[docs.langchain.com](https://docs.langchain.com/oss/python/langchain/agents#:~:text=You%20can%20shape%20how%20your,be%20provided%20as%20a%20string). It also highlights that using a structured `SystemMessage` (e.g. a JSON list of roles and context) offers precise control[docs.langchain.com](https://docs.langchain.com/oss/python/langchain/agents#:~:text=more%20control%20over%20the%20prompt,features%20like%20Anthropic%E2%80%99s%20prompt%20caching). For example, LangChain uses YAML/JSON config to define agents with name, role, and tools, but the final agent prompt still follows the patterns above.

**Microsoft AutoGen:** AutoGen’s `AssistantAgent` class includes a `system_message` field that defaults to something like _“You are a helpful AI assistant. Solve tasks using your tools.”_[microsoft.github.io](https://microsoft.github.io/autogen/dev//reference/python/autogen_agentchat.agents.html#:~:text=str%5D%20,result%7D%27%2C%20tool_call_summary_formatter). Users typically override this with a custom string. In tutorials, agents are built with code such as `Agent(name="weather_assistant", system_message="You fetch weather info and return it.")`[clintgoodman27.medium.com](https://clintgoodman27.medium.com/a-practical-guide-for-using-autogen-in-software-applications-8799185d27ee#:~:text=weather_assistant%20%3D%20WeatherAgent%28%20name%3D,). This reinforces that AutoGen’s design expects a concise system instruction for each agent. Multi-agent setups in AutoGen often use human-readable YAML or code templates (with agents referencing YAML task specs), but the core “prompt” to each agent is the system message plus any tool/output context. (Confidence: **MEDIUM** — documentation-level evidence.)

**Claude Code (Anthropic tooling):** Claude Code’s design exemplifies many best practices. It uses a simple ReAct loop (model alternates reasoning and tool calls)[jannesklaas.github.io](https://jannesklaas.github.io/ai/2025/07/20/claude-code-agent-design.html#:~:text=At%20it%E2%80%99s%20core%2C%20Claude%20Code,agent%20waits%20for%20user%20input). Uniquely, Claude Code inserts _“system-reminder”_ messages around the user’s query to reinforce instructions. For instance, at start it sends a `<system-reminder>` block listing strict rules (e.g. “Do exactly what is asked… NEVER create files unless necessary… ALWAYS prefer editing existing files” etc.[jannesklaas.github.io](https://jannesklaas.github.io/ai/2025/07/20/claude-code-agent-design.html#:~:text=%3Csystem,explicitly%20requested%20by%20the%20User)). It then follows up with reminders about its TODO list status[jannesklaas.github.io](https://jannesklaas.github.io/ai/2025/07/20/claude-code-agent-design.html#:~:text=%3Csystem,reminder). These reminders are tagged in pseudo-XML and repeated on every tool call, effectively acting like persistent instructions. The lesson is that **reiterating critical rules explicitly (even via specially tagged blocks) helps consistency**. While this is a specialized case, it highlights that clearly labeling instructions (e.g. using a unique tag or header) can boost compliance. (Confidence: **MEDIUM**.)

## Prompt Length vs Compliance

Empirical studies suggest there is _no magic “ideal” length_ but that **conciseness and relevance matter**. Overly long prompts can exceed the model’s attention and cause it to “lose” mid-prompt details. In long-context models, performance is best when key info is at the _beginning or end_ of the context[aclanthology.org](https://aclanthology.org/2024.tacl-1.9/#:~:text=context,models%20use%20their%20input%20context). Anthropic calls this “context rot”: as context grows, the model’s recall of earlier instructions degrades[anthropic.com](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents#:~:text=we%E2%80%99ve%20observed%20that%20LLMs%2C%20like,information%20from%20that%20context%20decreases). Thus, extra filler or drifting content (e.g. redundant philosophy) can reduce adherence to the core instructions[anthropic.com](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents#:~:text=we%E2%80%99ve%20observed%20that%20LLMs%2C%20like,information%20from%20that%20context%20decreases)[aclanthology.org](https://aclanthology.org/2024.tacl-1.9/#:~:text=context,models%20use%20their%20input%20context). On the other hand, extremely terse prompts may omit needed detail. For example, one evaluation found that **very concise prompts** (while cheaper) often lowered accuracy for complex tasks[researchgate.net](https://www.researchgate.net/publication/392505938_What's_in_a_Prompt_A_Large-Scale_Experiment_to_Assess_the_Impact_of_Prompt_Design_on_the_Compliance_and_Accuracy_of_LLM-Generated_Text_Annotations#:~:text=sentiment%2C%20rumor%20stance%2C%20and%20news,using%20LLMs%20in%20CSS%20research).

**Takeaway:** Aim for the “Goldilocks zone” of prompt length[anthropic.com](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents#:~:text=we%E2%80%99ve%20observed%20that%20LLMs%2C%20like,information%20from%20that%20context%20decreases): include all **essential** instructions and context but strip out anything irrelevant. Combine related steps, avoid superfluous prose, and put key constraints (like formatting rules) early. If a task is inherently complex, it’s often better to break it into stages than to overload a single prompt. (Confidence: **MEDIUM**.)

## Examples vs Rules

Providers recommend using **both** explicit rules and illustrative examples. Anthropic notes that examples (few-shot) are “one of the most reliable ways to steer output format, tone, and structure”[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Use%20examples%20effectively). Embedding well-chosen examples clarifies the intent (especially for subtle or ambiguous tasks). OpenAI likewise advises adding example interactions (tool usage examples, sample outputs, etc.) in a dedicated section, while also **“citing in your rules”** any behavior shown in examples[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=examples,also%20cited%20in%20your%20rules). In other words, don’t rely solely on examples; explicitly state any important constraints or steps in prose as well. The Elements.cloud prompt guide echoes this: it calls adding examples for complex instructions a best practice (pattern 6)[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=6,Instructions).

**Recommendation:** When feasible, include a few concrete examples of the desired output or process steps (3–5 is common)[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Use%20examples%20effectively)[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=6,Instructions). Each example should be clearly tagged (e.g. under an “Examples” header or `<example>` block). Critically, ensure that the essential behaviors illustrated by the examples are also described in the textual instructions or rule clauses[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=examples,also%20cited%20in%20your%20rules).

## Conditional Logic (If/Else)

Logical branching should be spelled out clearly and _not_ conflated. A common anti-pattern is writing “If X then do A, otherwise do B” in one bullet – this can confuse the model about when each branch applies. Instead, **separate each branch into its own instruction**. For example, write **“If condition X, then do A. If condition Y, then do B.”** as two sequential steps. The Elements guide explicitly warns against embedding multiple conditions in a single line[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=5,One%20Step), noting this often leads to the AI choosing the wrong branch. Similarly, contradictory or compound constraints (e.g. “Add X but keep it Y”[cline.bot](https://cline.bot/blog/the-worst-instructions-you-can-give-an-ai-coding-agent#:~:text=The%20AI%20reads%20this%20and,choice%20and%20runs%20with%20it)) should be broken down and prioritized. If absolute rules exist (e.g. safety limits), consider framing them with RFC-2119 terms (MUST, SHOULD) as AWS recommends[aws.amazon.com](https://aws.amazon.com/blogs/publicsector/why-your-ai-agents-give-inconsistent-results-and-how-agent-sops-fix-it/#:~:text=,LLM%20is%20doing%20and%20why).

**Recommendation:** Use **distinct bullets or numbered steps** for each conditional rule. Phrase them positively (“Do A only if X”) rather than negatively, and avoid overloading a single step with “if/else.” This clarity in structure minimizes ambiguity and misinterpretation[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=5,One%20Step)[cline.bot](https://cline.bot/blog/the-worst-instructions-you-can-give-an-ai-coding-agent#:~:text=The%20AI%20reads%20this%20and,choice%20and%20runs%20with%20it). (Confidence: **MEDIUM- HIGH**.)

## Structural Formats (XML vs Markdown vs YAML)

Guidelines emphasize **markup clarity**. OpenAI’s newest guidance is clear: _Markdown headings and lists_ are the preferred default for organizing prompts[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each). They allow human-readable sectioning (e.g. `# Task`, `## Instructions`, `- Step 1`) and work well across models. OpenAI further notes that **XML-style tags** also perform very well with GPT-4.1 and GPT-5[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=2,inputs%20and%20outputs%20for%20each). XML can explicitly demarcate sections (`<instructions>`, `<example>`, `<input>`, etc.), nest content, and attach metadata (like IDs) to aid parsing. (Indeed, Anthropic’s docs strongly encourage this practice[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Structure%20prompts%20with%20XML%20tags).) JSON/YAML are less recommended _for the prompt itself_ because of verbosity and extra tokens[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=3,escaping%20that%20can%20add%20overhead). YAML is similar to JSON in this regard: while some agent frameworks use YAML _externally_ (e.g. CrewAI’s task config[docs.crewai.com](https://docs.crewai.com/en/concepts/tasks#:~:text=research_task%3A%20description%3A%20,topic%7D%20agent%3A%20researcher)), the model itself sees it as just text.

**Recommendation:** Use **Markdown** (or XML) to structure instructions. For example, begin each prompt with a high-level Markdown section (H1/H2) naming the task or objective, then use bullet lists or numbered lists for steps, and subheadings for sub-sections. If you embed tags (as Anthropic suggests), keep them consistent and descriptive (e.g. `<context>`, `<steps>`, `<tool>`)[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Structure%20prompts%20with%20XML%20tags)[developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each). In short, make sections “stand out” either via headers or tags.

(Confidence: **HIGH** for Markdown/XML; **LOW** for YAML since not covered by formal docs.)

## Instruction Anti-Patterns (What to Avoid)

Drawing on documented pitfalls, avoid these common mistakes:

-   **Vague or High-Level Goals:** Don’t use ambiguous phrases like _“Make it better”_ or _“improve efficiency”_ without specifics[cline.bot](https://cline.bot/blog/the-worst-instructions-you-can-give-an-ai-coding-agent#:~:text=Vague%20objectives%20like%20,a%20lottery%20with%20your%20tokens). Vague objectives force the model to guess your intent, often with poor results. Always define _exactly_ what “better” means in concrete terms (e.g. “refactor the function into two smaller functions, add error handling,” etc.).
    
-   **Contradictory Rules:** Don’t pair opposing instructions in one prompt (e.g. _“Add OAuth but keep it simple”_[cline.bot](https://cline.bot/blog/the-worst-instructions-you-can-give-an-ai-coding-agent#:~:text=The%20AI%20reads%20this%20and,choice%20and%20runs%20with%20it)). The AI will arbitrarily pick one interpretation. Instead, clarify priorities (e.g. “security is most important, simplicity secondary” or break into separate questions).
    
-   **Negative Framing:** Avoid phrases that tell the model _what not to do_. Elements.cloud warns that too many “do not” statements confuse agents[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=4). Preface with positive directives: e.g., say “Only send a summary if condition X is met” instead of “Don’t send a summary if not X.”
    
-   **Multi-branch Logic in One Step:** As above, putting multiple branches (“if/else”) in one sentence is an anti-pattern[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=5,One%20Step). Split into clear steps.
    
-   **Inconsistent Terminology:** Use the same words for the same concepts throughout. Switching synonyms (e.g. calling an object a “case” in one step and a “ticket” in another) can confuse the model[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=7). Pick one term and stick with it.
    
-   **Unnecessary Detail or Repetition:** Remove any content that isn’t strictly needed. Don’t copy large unrelated text dumps into the prompt. For example, Claude Code found that repeating fixed instructions in each tool output (via the `<system-reminder>`) dramatically improved adherence[jannesklaas.github.io](https://jannesklaas.github.io/ai/2025/07/20/claude-code-agent-design.html#:~:text=A%20small%20but%20clever%20trick,be%20in%20the%20system%20prompt) – suggesting that burying rules invisibly is an anti-pattern. Likewise, redundant templates or lengthy background belong in context sections, not mixed with the step list.
    

Confidence for these anti-patterns: **HIGH** (based on multiple guides[cline.bot](https://cline.bot/blog/the-worst-instructions-you-can-give-an-ai-coding-agent#:~:text=Vague%20objectives%20like%20,a%20lottery%20with%20your%20tokens)[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=4)[elements.cloud](https://elements.cloud/blog/agent-instruction-patterns-and-antipatterns-how-to-build-smarter-agents/#:~:text=5,One%20Step)).

## Token Efficiency and Conciseness

Minimizing tokens is crucial for performance and cost. Across sources the advice is consistent: **Cut the fluff.** Every unnecessary word is wasted budget. Tips from practice include replacing filler phrases with leaner language (“due to the fact” → “because,” “at this point” → “now,” etc.)[medium.com](https://medium.com/write-a-catalyst/token-efficient-llm-prompting-save-money-speed-up-workflows-08d8d5a45b1b#:~:text=Core%20Strategy%3A%20Cut%20the%20Fluff%2C,Add%20Structure)[medium.com](https://medium.com/@ThinkingLoop/d3-8-10-smart-ways-to-cut-ai-api-token-costs-d71cddfde433#:~:text=1,Fluff). Use short, active sentences. OpenAI notes newer models are inherently more concise, and will omit verbose summaries unless asked[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=self,for%20efficiency%20unless%20prompted%20otherwise); nevertheless, prompt authors should be equally concise.

Structuring prompts well also saves tokens. Medium and IBM blogs (and Anthropic) all emphasize that **clear structure prevents expensive reasoning loops**[medium.com](https://medium.com/write-a-catalyst/token-efficient-llm-prompting-save-money-speed-up-workflows-08d8d5a45b1b#:~:text=But%20word%20choice%20is%20just,boundaries%20cut%20that%20waste%20immediately)[medium.com](https://medium.com/@ThinkingLoop/d3-8-10-smart-ways-to-cut-ai-api-token-costs-d71cddfde433#:~:text=1,Fluff). For example, a disorganized or ambiguous prompt may cause the model to “think out loud” rather than give a crisp answer. By contrast, a well-bulleted prompt with explicit constraints yields faster, shorter outputs.

**Recommendations:** - **Shorten instructions:** Merge sentences when possible, drop polite preambles (“please”/“could you”), and avoid redundant clauses[medium.com](https://medium.com/write-a-catalyst/token-efficient-llm-prompting-save-money-speed-up-workflows-08d8d5a45b1b#:~:text=Core%20Strategy%3A%20Cut%20the%20Fluff%2C,Add%20Structure)[medium.com](https://medium.com/@ThinkingLoop/d3-8-10-smart-ways-to-cut-ai-api-token-costs-d71cddfde433#:~:text=1,Fluff). - **Use bullets/numbering:** They guide the model to produce similarly compact answers. - **Omit unneeded context:** Keep only what affects the task. If a skill has multiple common steps or policies, consider using shared templates or “include” directives rather than rewriting them each time. - **Re-use context smartly:** In multi-step workflows, use techniques like context caching or agent memory instead of re-sending the full history each time.

## Skill Contract Recommendations

Based on the above, a **Skill Contract** (canonical skill prompt template) should use the following structure and principles:

-   **Brief Purpose/Role (system prompt):** One-liner stating the agent’s role or the goal of the skill.
    
-   **Context (optional):** A short section (or `<context>` tag) giving any necessary background/motivation for the task.
    
-   **Inputs & Constraints:** Clearly list any inputs the agent has (parameters, files, etc.) and any strict constraints (format, length limits, safety rules). Use bullets.
    
-   **Step-by-Step Instructions:** Numbered or bulleted steps describing exactly what to do. Each step should be a concise imperative sentence. Use active voice (e.g. “Extract the top 3 entities from the text” rather than “You should try to extract…”). Ensure steps are in logical order.
    
-   **Conditional Branching:** If the task has branches, split each conditional rule into a separate sub-step (e.g. Step 3.1 _If X: do A_, Step 3.2 _If Y: do B_).
    
-   **Examples (if needed):** If output format or task is complex, include a “Examples” subsection (or `<example>` tags) showing sample input/output. Provide 2–3 examples.
    
-   **Output Format:** Clearly specify what the final answer should look like (e.g. “Provide a JSON object with keys `name`, `date`, `score`” or “a markdown report with sections and bullet points”). This can be a separate bullet or final step.
    
-   **Use Clear Markup:** Delimit sections with Markdown headers (e.g. `## Instructions`) or XML tags as shown above. Be consistent. E.g.:
    
    ```
    <div><div><p>Code</p></div><div id="code-block-viewer" dir="ltr"><p><span>## Instructions</span></p><br><p><span>1. Load the JSON from `input1.json`.</span></p><br><p><span>2. Filter records where `status = "active"`.</span></p><br><p><span>3. ...</span></p><br><p><span>## Output Format</span></p><br><p><span>Provide the result as a markdown table with columns ID and Name.</span></p></div></div>
    ```
    
    [developers.openai.com](https://developers.openai.com/cookbook/examples/gpt4-1_prompting_guide/#:~:text=1,inputs%20and%20outputs%20for%20each)[platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#:~:text=Structure%20prompts%20with%20XML%20tags).
    
-   **Explicit “Do’s” Rather Than “Don’ts”:** Phrase each requirement positively. List any forbidden actions only if absolutely necessary (and even then as a final note).
    
-   **Avoid Bloat:** Omit unrelated text, history, or verbose explanation. If certain guidance is shared across many skills, consider a **shared snippet** (to reduce repetition) or an included reference.
    
-   **Single-Task Focus:** Each skill should address one type of task. If more than one distinct action is needed, use multiple skills or steps.
    

By following these structured patterns—clear headers/tags, concise bullet steps, embedded examples, and minimal fluff—the Skill Contract will be **reliable, consistent, and token-efficient**.
