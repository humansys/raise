## R1: Domain-Agnostic Knowledge Graph Retrieval Architectures

### Research Question

How do existing systems solve "one retrieval engine, multiple knowledge domains" for structured (non-embedding) knowledge graphs? What architectural patterns enable domain-agnostic query interfaces while preserving domain-specific semantics?

### Context (for the researcher)

We're building a personal AI agent that ingests structured knowledge from multiple domains (business methodology, productivity systems, life management frameworks). Each domain has its own ontology (node types, relationship types, attributes), but we want a single retrieval interface that works across all domains.

Our current stack:
- **Graph engine:** NetworkX-backed property graph (MultiDiGraph) with typed nodes and edges
- **Node model:** Pydantic BaseModel subclasses with domain-specific fields stored in `metadata: dict[str, Any]`
- **Existing query:** Keyword search (text matching on content) + concept lookup (BFS neighbors with edge_type filter)
- **Constraint:** Retrieval must be symbolic (graph traversal), NOT vector similarity. No LLM calls in the retrieval path.
- **Scale:** Hundreds of nodes per domain, not millions. Latency target <100ms.

### Specific Questions to Answer

1. **Architecture patterns:** What are the proven patterns for multi-domain KG retrieval? (schema-mediated queries, ontology mapping, query federation, adapter pattern, query rewriting)
2. **Real systems:** How do GraphRAG (Microsoft), LlamaIndex Knowledge Graph, Neo4j domain plugins, Apache Jena, Wikidata Query Service, and similar systems handle domain-pluggability?
3. **Query abstraction:** What does a good "domain-agnostic query" look like? How do you express "give me relevant nodes for this context" without hardcoding domain attributes?
4. **Domain registration:** How do systems register new domains? Configuration-driven? Code-driven? Schema introspection?
5. **Trade-offs:** What are the failure modes? Over-abstraction that loses domain richness? Under-abstraction that duplicates logic?

### Expected Output

- 2-4 architectural patterns with pros/cons/when-to-use
- 3+ real system examples with how they solve this
- Recommended pattern for our constraints (small-scale, symbolic, Pydantic+NetworkX)
- Anti-patterns to avoid

### Evidence Standards

- Prioritize: peer-reviewed papers, official documentation, production postmortems
- Accept: well-regarded blog posts (engineering blogs from known teams), conference talks with slides
- Triangulate: each claim needs 2+ independent sources
- Rate confidence: Very High / High / Medium / Low per finding

## R2: Pluggable Domain Adapter Interfaces for Graph Traversal

### Research Question

What interface should a "domain adapter" expose so that a generic retrieval engine can delegate domain-specific query interpretation and traversal strategy without losing retrieval quality?

### Context (for the researcher)

We need to design the contract between a domain-agnostic retrieval engine and domain-specific plugins. The first domain is "Scaling Up" (business methodology) with these characteristics:
- 6 node types: decision, concept, tool, worksheet, stage, metric
- 6 relationship types: belongs-to, requires, feeds-into, measured-by, prerequisite-of, implements
- Key query dimensions: decision_area (people/strategy/execution/cash), stage (startup/growth/scaleup), difficulty level
- Nodes have structured fields (not just text blobs): decision area, relationships, tags, difficulty, source pointers

Future domains will have different node types, relationship types, and query dimensions. For example:
- GTD (Getting Things Done): projects, contexts, next-actions, waiting-for, someday-maybe
- Life Areas: health, finance, career, relationships — each with goals, habits, metrics

### Specific Questions to Answer

1. **Adapter interface design:** What methods/responsibilities should a domain adapter have? Query parsing? Traversal strategy selection? Result ranking? Node filtering?
2. **Design patterns:** Strategy pattern vs. Visitor vs. Query rewriting vs. Plugin architecture vs. Protocol/trait-based adapters. Which fits best for "interpret this query in my domain's semantics"?
3. **Query context model:** How to represent a domain-agnostic query that can carry domain-specific attributes? Union types? Open dict? Schema-validated extensions? Discriminated unions?
4. **Traversal delegation:** Should the adapter OWN the traversal (full control) or ADVISE the engine (suggest starting nodes, edge filters, depth)? What's the right balance between flexibility and consistency?
5. **Registration mechanism:** How do systems register adapters? Entry points? Registry pattern? Convention over configuration? How does raise-commons do plugin registration for GraphNode subclasses (it uses `__init_subclass__`)?

### Expected Output

- Recommended adapter interface (Protocol/ABC with method signatures)
- Query context model design
- Registration mechanism recommendation
- 2-3 real examples of similar adapter patterns in production systems
- Code-level sketch of the interface (Python Protocol preferred)

### Evidence Standards

- Prioritize: production codebases (open source), design pattern literature (GoF, PEAA), Python ecosystem patterns
- Accept: conference talks, architectural decision records from known projects
- Triangulate: pattern recommendation needs 2+ successful implementations as evidence
- Rate confidence: Very High / High / Medium / Low per finding

## R3: Symbolic Graph Retrieval Scoring Without Embeddings

### Research Question

What is the state of the art in ranking/scoring graph retrieval results using ONLY graph structure (no embeddings, no vector similarity)? How do you determine "this node is more relevant than that node" through symbolic operations?

### Context (for the researcher)

Our retrieval engine returns a set of ontology nodes from a knowledge graph. We need to rank them by relevance to the query context. Constraints:
- **No embeddings or vectors** — retrieval is purely symbolic (graph traversal + attribute matching)
- **No LLM calls** in the retrieval path — scoring must be deterministic and fast
- **Small graphs** — hundreds of nodes, not millions. Sub-100ms latency.
- **Rich structure** — nodes have types, attributes, typed relationships. Not just content blobs.
- **Domain-specific semantics** — relationship types have meaning (requires ≠ feeds-into ≠ belongs-to)

Current naive approach: BFS from starting nodes, rank by hop distance. This loses semantic richness.

### Specific Questions to Answer

1. **Graph-based scoring algorithms:** Personalized PageRank, spreading activation, Katz centrality, path-based relevance. Which work best for small, rich, typed property graphs?
2. **Edge-type weighting:** How do you score differently based on relationship type? (A node connected via "requires" is more relevant than one connected via "related-to")
3. **Type-aware ranking:** How to incorporate node types into relevance? (A "tool" node might be more relevant than a "metric" node when the query is about "what should I do?")
4. **Multi-hop scoring decay:** What's the right decay function for distance? Linear? Exponential? Should it differ by relationship type?
5. **Composite scoring:** How to combine structural relevance (graph distance) + attribute match (query context matches node attributes) + type relevance into a single score?
6. **Benchmarks and evaluation:** How do you evaluate retrieval quality for symbolic KG retrieval? Precision/recall against competency questions? Human evaluation?

### Expected Output

- Ranked list of scoring algorithms with applicability to our constraints
- Recommended composite scoring formula with rationale
- Edge-type and node-type weighting strategies
- Evaluation methodology for retrieval quality
- Anti-patterns (what doesn't work for small symbolic graphs)

### Evidence Standards

- Prioritize: peer-reviewed papers (SIGIR, WWW, ISWC, ESWC), textbook algorithms
- Accept: well-cited survey papers, implementation comparisons
- Triangulate: each algorithm recommendation needs empirical evidence from 2+ papers
- Rate confidence: Very High / High / Medium / Low per finding
- Note: much KG retrieval research focuses on large-scale or embedding-based systems. Flag explicitly when a technique doesn't scale DOWN to small graphs or doesn't work without embeddings.
