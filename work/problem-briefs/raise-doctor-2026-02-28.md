# Problem Brief: RaiSE Doctor

> Generated by `/rai-problem-shape` on 2026-02-28

---

## 1. Domain

**Calidad / retrabajo** — Configuraciones RaiSE incompletas o degradadas causan comportamiento deficiente del agente, generando retrabajo y pérdida de confianza.

## 2. Primary Stakeholder

**Desarrollador + Rai** — La dupla de trabajo que experimenta directamente el comportamiento degradado sin saber por qué.

## 3. Gap (Estado Actual)

El desarrollador + Rai **no pueden detectar que la configuración de RaiSE está incompleta o degradada** porque **no existe un diagnóstico que valide archivos, secciones y coherencia del setup — el problema solo se manifiesta cuando el agente ya se comporta mal**.

## 4. Root Cause (3 Whys)

1. **¿Por qué no existe diagnóstico?** → Siempre hubo features más urgentes que herramientas de diagnóstico.
2. **¿Por qué eran más urgentes?** → No había evidencia concreta de que config rota causara problemas reales.
3. **¿Por qué no había evidencia?** → Solo Emilio puede distinguir entre Rai funcionando bien y Rai funcionando mal. No hay observabilidad programática de la salud del setup — la calidad depende de que un experto note síntomas difusos.

**Raíz:** RaiSE no tiene observabilidad de su propia salud. El diagnóstico depende de un experto humano, no de validación programática.

## 5. Early Signal (4 semanas)

Dos modos complementarios:
- **On-demand:** `rai doctor` que cualquier usuario corra y vea rojo/verde si su setup está sano
- **Proactivo:** Rai detecta config incompleta y ofrece reparar con confirmación del usuario (e.g., al inicio de sesión)

## 6. Hypothesis (SAFe)

> Si implementamos diagnóstico programático de la salud de configuración RaiSE (determinista para archivos/secciones + inferencia para reparación guiada), entonces el desarrollador+Rai podrán detectar y corregir setup incompleto antes de que degrade el comportamiento del agente, medido por: **proyectos nuevos pasan `rai doctor` en verde sin intervención manual de Emilio**.

## Scope Hints (for epic design)

### Deterministic checks (CLI, `rai doctor`)
- Archivos existen: CLAUDE.md, MEMORY.md, manifest.yaml, methodology.yaml, identity/
- Secciones presentes: Process Rules, Gates, Branch Model, CLI Reference en CLAUDE.md
- Secciones no vacías: MEMORY.md tiene skills con prefijo `rai-`, branch model matches manifest
- Coherencia: skill names en MEMORY.md matchean skills instalados en .claude/skills/
- Staleness: MEMORY.md last_updated no es >30 días viejo

### Inference-assisted checks (skill or session hook)
- CLAUDE.md tiene instrucciones de proceso suficientes (no es placeholder)
- Reparación guiada: ofrecer regenerar CLAUDE.md, actualizar MEMORY.md, sincronizar skill names
- Integración con `/rai-session-start`: correr checks ligeros al inicio de sesión
