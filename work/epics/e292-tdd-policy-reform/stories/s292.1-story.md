# S292.1 — Governance: Update Guardrails + DoD

## User Story

**As** an AI agent implementing a RaiSE story,
**I want** consistent, quality-first testing guidelines in all governance documents,
**so that** I write tests that catch real bugs instead of tests that hit a coverage number.

## Acceptance Criteria

**Scenario: Agent reads guardrails-stack.md testing section**
- Given the agent opens `governance/guardrails-stack.md`
- When it reads the Testing section (§4)
- Then it sees no ">90% coverage" target
- And it sees explicit anti-patterns for test muda
- And it sees "coverage as diagnostic, not gate" philosophy

**Scenario: Agent reads Definition of Done**
- Given the agent reads the DoD in `governance/backlog.md §6`
- When it checks the testing criterion
- Then it sees quality-based criteria, not a coverage number

**Scenario: Agent reads CLAUDE.md**
- Given the agent reads CLAUDE.md Critical Rules
- When it reads the TDD rule
- Then it understands "each test justifies its existence"

**Scenario: Coherence check**
- Given guardrails.md (MUST-TEST-001) and guardrails-stack.md
- When both are read together
- Then they agree: coverage is informative, test quality is the gate
