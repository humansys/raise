## What to Improve

- **Design was incomplete**: Gemba Walk identified 4 files, but `grep -rn` in T3 found 5 more active docs with the same stale policy. The design should have inventoried all occurrences upfront.
- For coherence stories, the design step should always include a full `grep -rn` pass — same principle as "grep all call sites" for code refactors.

## Heutagogical Checkpoint

**1. What did I learn?**
Governance docs are like a distributed codebase — the same "variable" (`>90% coverage`) can appear in many places. `system-design.md`, `prd.md`, `sops/`, `vision.md`, `architecture/` are all load-bearing docs that agents may read. A partial fix is worse than no fix — agents would get contradictory signals.

**2. What would I change about the process?**
In `/rai-story-design` Step 3 ("Describe Approach"), add an explicit gate for coherence/policy stories:
> *"For coherence/policy stories: run `grep -rn <pattern>` across entire repo before defining scope. Document exact file count and locations."*

**3. Framework improvement?**
The `/rai-story-design` skill's Step 3 has the note *"For refactoring: grep all call sites of the target"* — this should be generalized to include documentation coherence stories.

**4. What am I more capable of now?**
Identifying when a "simple" 4-file doc story is actually a 9-file sweep. The pattern: if a policy is expressed as a number/metric, it will be duplicated across every doc that references quality standards.
