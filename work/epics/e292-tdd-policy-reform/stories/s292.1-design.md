# S292.1 — Design

## Problem & Value

**Problem:** `guardrails-stack.md` tells agents to target >90% coverage in 3 places, while `guardrails.md` already says "no fixed gate." This incoherence causes agents to write tautological tests (PAT-E-444).

**Value:** After this story, every agent reading governance docs gets consistent quality-first guidance. Future stories produce fewer, better tests immediately.

## Approach

Pure documentation coherence edits — no code changes. Four files, all mechanical. The heuristics are already defined in the epic design (D2); this story bakes them into governance.

**Components affected:**

| File | Change type | What |
|------|------------|------|
| `governance/guardrails-stack.md` | Modify | 3 targeted changes (Quick Ref, §4.7, Jidoka checklist) |
| `governance/guardrails.md` | Modify | Minor wording on MUST-TEST-001 for coherence |
| `governance/backlog.md` | Modify | §6 DoD: replace ">90% coverage on feature" |
| `CLAUDE.md` | Modify | 1-line addition to TDD rule |

### 2. `guardrails-stack.md` — §4.7 (lines 497-504)

**Before:**
```markdown
### 4.7 Coverage Is Not Enough

**Do:** Pair coverage with mutation testing (mutmut).

**Target:** >90% line coverage + >85% branch coverage.

**Don't:** Trust 100% coverage alone (tests may not assert correctly).
```

**After:**
```markdown
### 4.7 Test Quality Over Coverage

**Do:** Write tests that catch real defects. Each test must assert observable behavior.

**Coverage:** Diagnostic only — no fixed target. Floor: if coverage drops below 70%, investigate gaps in domain logic.

**Anti-patterns (muda):**
- Constant assertions: `assert "x" == "x"` (always passes)
- Mock-implementation tests: only verify an internal method was called
- Magic-number counts: `assert len(items) == 21` (brittle, not behavioral)
- Happy-path-only: no boundary tests (empty, one, many, error cases)

**Don't:** Write tests to hit a coverage number. If a test doesn't catch a real bug, it's waste.
```

**After:**

```markdown
### Testing

- [ ] No muda: no constant assertions, no mock-implementation tests, no magic counts?
```
