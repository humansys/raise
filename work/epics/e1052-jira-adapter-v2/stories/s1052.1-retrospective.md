# S1052.1: JiraClient Wrapper — Retrospective

**Story:** S1052.1 | **Epic:** E1052 | **Date:** 2026-03-31

---

## Summary

Delivered a thin Python wrapper (`JiraClient`) over `atlassian.Jira` with:

- 11 methods: get_issue, create_issue, update_issue, transition_issue, get_transitions, jql, create_link, set_parent, add_comment, get_comments, server_info
- Typed exception hierarchy: JiraAdapterError -> JiraAuthError, JiraNotFoundError, JiraApiError
- Auth resolution from environment variables with instance-specific overrides
- `from_config()` classmethod for multi-instance support
- `backoff_and_retry=True` with 5 retries, factor 1.0
- Full unit test suite + integration tests (skip when no credentials)

## What Went Well

1. **ConfluenceClient mirror pattern worked.** Having an existing reference (S1051.1) made design decisions near-automatic. All 7 design decisions were auto-decided from epic context or the ConfluenceClient precedent. Zero ambiguity, zero debate.

2. **TDD cycle was smooth.** 5 sequential tasks, each building on the previous, with clear RED-GREEN-REFACTOR cycles. The plan's data flow trace and exception mapping table directly translated to test cases.

3. **Estimate accuracy.** Planned 95-120 min, delivered in ~90 min. The M sizing was correct. The sequential task dependency chain was clear and didn't cause any blocking.

4. **Test-to-production ratio (2.8:1 LOC) is healthy.** 49 tests covering all error paths, auth resolution, and method delegation. The mock-based approach gives fast feedback without requiring live Jira.

---

## What Could Improve

1. **Coverage report output is noisy.** The full-project coverage report makes it hard to find the jira_client-specific line. Should configure pytest to only report on the packages under test, not the entire monorepo.

2. **Integration test coverage is thin.** Only 3 integration tests (server_info, get_issue, jql_search). Would benefit from integration tests for write operations (create, update, transition) — deferred to S1052.3 where the adapter will exercise these paths.

3. **Missing lines (92% vs 100%).** The 10 uncovered lines are the real atlassian.Jira call bodies inside each method — these are exercised by integration tests but not by unit tests (which mock one level up). Acceptable trade-off: testing the error mapping wrapper is more valuable than testing `self._client.issue(key)` delegation.

Both were minor — caught by review, not by gate failure.

---

## Patterns Worth Capturing

1. **Mirror pattern accelerates delivery.** When a second adapter follows the same shape as the first (ConfluenceClient -> JiraClient), design becomes mechanical. This confirms epic design decision D2 (unified client interface). Estimate: 30-40% time savings vs greenfield.

2. **Test-to-code ratio for wrapper classes.** Thin wrappers over external libraries need high test ratios (~3:1 LOC) because the value is in error mapping and auth resolution, not business logic. The tests ARE the specification of the contract.
