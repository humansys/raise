# S1052.3: PythonApiJiraAdapter — Design

## Integration Decisions

### D1: Lazy Client Cache Strategy
**Decision:** One JiraClient per instance name, cached in `_clients: dict[str, JiraClient]`.
`_client_for(project_key)` resolves project → instance name via config, then
creates or returns cached client via `JiraClient.from_config()`.

**Rationale:** Multi-instance support (e.g., humansys + client instances) without
creating redundant connections. Same pattern as config's `resolve_instance()`.

### D2: Status Resolution — Live Transition Lookup
**Decision:** `transition_issue` receives a CLI slug (e.g., "in-progress"), normalizes
to Jira display name ("In Progress") via convention, queries `get_transitions()`,
matches by name (case-insensitive), then calls `client.transition_issue(key, id)`.

**Rationale:** No hardcoded transition IDs. Works across different Jira workflow
configurations. Same `normalize_status()` convention as AcliJiraAdapter.

### D3: Response Parsing — Port from AcliJiraAdapter
**Decision:** Port `_extract_issue_fields`, `_parse_issue_detail`, `_adf_to_text`,
`to_jql`, `_quote_project_values`, `normalize_status`, and `build_url` directly.

**Rationale:** These are battle-tested parsing functions. Jira REST API format is
identical whether accessed via ACLI or atlassian-python-api.

### D4: Async Protocol with Sync Client
**Decision:** Methods are `async def` to satisfy `AsyncProjectManagementAdapter`
protocol, but internally call sync `JiraClient` methods directly (no await).

**Rationale:** atlassian-python-api is synchronous. The async signature enables
future async transport or SyncPMAdapter wrapping for CLI consumption. This is the
same pattern as if we wrapped with `asyncio.to_thread` but simpler — the sync
methods are fast (network-bound, not CPU-bound).

### D5: Entry Point Coexistence
**Decision:** Register `jira` entry point in raise-cli's pyproject.toml. Do NOT
modify raise-pro's entry points (that's S1052.5). Add `jira` optional dependency
group.

**Rationale:** During migration, both adapters exist. The resolver picks based on
`-a jira` flag or manifest default. Deletion happens in S1052.5.

## Key Interfaces

### Constructor
```python
def __init__(self, project_root: Path | None = None) -> None:
    root = project_root or Path.cwd()
    self._config = load_jira_config(root)
    self._clients: dict[str, JiraClient] = {}
```

### Client Resolution
```python
def _client_for(self, project_key: str) -> JiraClient:
    instance = self._config.resolve_instance(project_key)
    cache_key = instance.site  # unique per actual Jira instance
    if cache_key not in self._clients:
        self._clients[cache_key] = JiraClient.from_config(
            self._config, instance_name
        )
    return self._clients[cache_key]
```

### Transition Resolution
```python
async def transition_issue(self, key: str, status: str) -> IssueRef:
    client = self._client_for(key.split("-")[0])
    jira_status = normalize_status(status)
    transitions = client.get_transitions(key)
    match = next(
        (t for t in transitions if t["name"].lower() == jira_status.lower()),
        None,
    )
    if not match:
        available = [t["name"] for t in transitions]
        raise JiraApiError(f"No transition to '{jira_status}'. Available: {available}")
    client.transition_issue(key, match["id"])
    return IssueRef(key=key, url=self._build_url(key))
```
