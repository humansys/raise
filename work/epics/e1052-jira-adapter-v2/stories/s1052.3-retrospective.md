# S1052.3: PythonApiJiraAdapter — Retrospective

## Outcome
All acceptance criteria met. 46 tests, pyright clean, ruff clean.

## What Went Well
1. **Port from acli_jira.py was smooth** — parsing logic, JQL normalization, ADF conversion all ported cleanly. Same Jira REST API format regardless of transport.
2. **JiraClient + JiraConfig layering worked** — the adapter is a thin orchestration layer over well-tested lower layers. Most complexity lives in JiraClient (S1052.1).
3. **Live transition lookup** — querying get_transitions() instead of hardcoding IDs is more robust. Different Jira instances can have different workflow configurations.
4. **Lazy client cache** — elegant solution for multi-instance routing without redundant connections.

## What Could Improve
1. **JQL/ADF functions duplicated** — `to_jql`, `normalize_status`, `_adf_to_text` are duplicated between acli_jira.py and jira_adapter.py during coexistence period. S1052.5 (deletion) will resolve.
2. **Test escaped operators test** — initial test had wrong expectation for `\!` handling in to_jql. The `!` is a JQL operator char, so after unescaping it triggers the passthrough branch, not the text search branch. TDD caught this.

## Patterns
- **Lazy cache by site key** — using instance.site as cache key ensures one JiraClient per actual Jira instance, even if multiple project keys route to the same instance.
- **Live transition lookup** — always query available transitions rather than assuming IDs. More resilient to workflow configuration changes.
