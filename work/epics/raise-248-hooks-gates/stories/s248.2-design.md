# S248.2: Hook Protocol & Registry

## What & Why

**Problem:** S248.1 built the event emitter with raw `HandlerFn` callbacks. There's no typed contract for hook implementations and no mechanism to discover hooks from installed packages.

**Value:** Completes the hook pipeline (emit ŌåÆ discover ŌåÆ dispatch). After this, external packages can ship `LifecycleHook` implementations that auto-register via entry points ŌĆö same mental model as adapters (AD-2).

## Approach

Three components, all following established patterns:

1. **`LifecycleHook` Protocol** ŌĆö runtime-checkable Protocol in `hooks/protocol.py`. Defines `events`, `priority`, `handle()` per ADR-039 ┬¦1.

2. **`HookRegistry`** ŌĆö in `hooks/registry.py`. Uses `importlib.metadata` entry points (`rai.hooks` group), identical to `adapters/registry.py`. Discovers, validates (Protocol conformance), and caches hook instances. Priority-sorted dispatch.

3. **`EventEmitter` integration** ŌĆö extend emitter to accept `HookRegistry`, auto-register discovered hooks on construction. Add per-hook timeout (5s default) to `emit()`.

### Components Affected

| Component | Change |
|-----------|--------|

## Examples

### LifecycleHook Protocol

```python
from rai_cli.hooks.protocol import LifecycleHook
from rai_cli.hooks.events import HookEvent, HookResult

class TelemetryHook:
    """Example hook ŌĆö logs events to signals.jsonl."""
    events = ["session:start", "session:close", "graph:build"]
    priority = 0  # default

    def handle(self, event: HookEvent) -> HookResult:
        # write to signals.jsonl...
        return HookResult(status="ok")

# Protocol conformance check
assert isinstance(TelemetryHook(), LifecycleHook)
```

### HookRegistry Discovery

```python
from rai_cli.hooks.registry import HookRegistry

registry = HookRegistry()
registry.discover()  # loads from rai.hooks entry points

# List discovered hooks
for hook in registry.hooks:
    print(f"{hook.__class__.__name__}: events={hook.events}, priority={hook.priority}")

# Get hooks for a specific event, priority-sorted (highest first)
handlers = registry.get_hooks_for_event("session:start")
```

### Entry Point Registration (pyproject.toml)

```toml
[project.entry-points."rai.hooks"]
telemetry = "rai_cli.hooks.builtin:TelemetryHook"
```

### EventEmitter with Registry + Timeout

```python
from rai_cli.hooks.emitter import EventEmitter
from rai_cli.hooks.registry import HookRegistry

registry = HookRegistry()
registry.discover()

emitter = EventEmitter(registry=registry)
# Hooks are auto-registered from registry

result = emitter.emit(SessionStartEvent(session_id="SES-264"))
# ŌåÆ hooks dispatched in priority order, 5s timeout per hook
```

### Timeout Behavior

```python
class SlowHook:
    events = ["session:start"]
    priority = 0
    timeout = 2.0  # override default 5s

    def handle(self, event: HookEvent) -> HookResult:
        time.sleep(10)  # exceeds timeout
        return HookResult(status="ok")

# SlowHook times out ŌåÆ logged as error, other hooks continue
# EmitResult.handler_errors includes "TimeoutError: SlowHook exceeded 2.0s"
```

### Error Isolation (already in emitter, preserved)

```python
class BrokenHook:
    events = ["graph:build"]
    priority = 10

    def handle(self, event: HookEvent) -> HookResult:
        raise RuntimeError("oops")

# BrokenHook fails ŌåÆ warning logged, next hook runs, CLI doesn't crash
```

## Acceptance Criteria

**MUST:**
- `LifecycleHook` is a `@runtime_checkable` Protocol with `events: ClassVar[list[str]]`, `priority: ClassVar[int]`, `handle(event: HookEvent) -> HookResult`
- `HookRegistry.discover()` loads hooks from `rai.hooks` entry points
- Hooks dispatch in priority order (highest priority first)
- Per-hook timeout (5s default, overridable via `timeout` class attribute)
- Broken/slow hooks are isolated ŌĆö logged, skipped, CLI continues
- `EventEmitter` accepts optional `HookRegistry` and auto-wires hooks

**SHOULD:**
- `rai hooks list` CLI command showing discovered hooks (name, events, priority)

**MUST NOT:**
- Break existing `EventEmitter` API (bare `HandlerFn` registration still works)
- Import from `adapters/` ŌĆö hooks module is independent (can share pattern, not code)
- Add async or threading ŌĆö timeouts use `signal.alarm` or thread-based executor

## Design Decisions

### DD-1: Optional registry on EventEmitter (backward compat)

`EventEmitter(registry=None)` keeps S248.1 tests passing. When registry is provided, discovered hooks are auto-registered alongside any manually registered handlers. Manual registration still works for testing.

### DD-2: Timeout via `concurrent.futures.ThreadPoolExecutor`

`signal.alarm` is Unix-only and not thread-safe. `threading.Timer` is messy. `ThreadPoolExecutor` with `future.result(timeout=N)` is stdlib, cross-platform, and clean. One worker thread per hook invocation ŌĆö hooks are synchronous within the thread but timeout is enforced from outside.

### DD-3: Priority order ŌĆö highest first

Consistent with Python logging, middleware stacks, and `stevedore` conventions. PRO hooks with `priority=100` run before COMMUNITY hooks with `priority=0`.

### DD-4: `events` as ClassVar[list[str]], not subscriptions

The hook declares what events it cares about. The registry indexes by event name. This is simpler than a subscribe/unsubscribe model and matches the static nature of entry points (you can't unload a pip-installed package at runtime).
