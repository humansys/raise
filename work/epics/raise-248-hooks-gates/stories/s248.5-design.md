# S248.5 Design: WorkflowGate Protocol and Registry

## What & Why

**Problem:** Quality gates (tests, types, lint, coverage) are currently manual checks in skill content. No infrastructure to discover, invoke, or compose gates programmatically.

**Value:** Standalone gate infrastructure that `rai gate check` invokes directly, and that S248.6's GateBridgeHook wires into `before:` events for auto-enforcement. Completes M1 Walking Skeleton alongside hooks.

## Approach

Mirror the hooks infrastructure (S248.2) with gate-specific semantics:

- **WorkflowGate Protocol** — `gate_id`, `description`, `workflow_point`, `evaluate(context) -> GateResult`
- **GateContext/GateResult** — frozen dataclasses (internal infrastructure, not boundary objects)
- **GateRegistry** — entry point discovery from `rai.gates`, identical pattern to HookRegistry
- **CLI commands** — `rai gate check <gate-id>`, `rai gate check --all`, `rai gate list`

### Components

| Component | Action | Path |
|-----------|--------|------|
| `WorkflowGate` Protocol | Create | `src/rai_cli/gates/protocol.py` |
| `GateContext`, `GateResult` | Create | `src/rai_cli/gates/models.py` |
| `GateRegistry` | Create | `src/rai_cli/gates/registry.py` |
| `gates/__init__.py` | Create | `src/rai_cli/gates/__init__.py` |
| `gate` CLI group | Create | `src/rai_cli/cli/commands/gate.py` |
| CLI registration | Modify | `src/rai_cli/cli/commands/__init__.py` |
| Entry point group | Modify | `pyproject.toml` |

### Key Difference from Hooks

| Aspect | LifecycleHook | WorkflowGate |
|--------|---------------|--------------|
| Question | "What happened?" (react) | "Is this allowed?" (validate) |
| Failure | Log + continue | Block operation |
| Composition | Independent (one fails, others run) | All-must-pass |
| UX on failure | Warning | Actionable error message |
| Dispatch | Priority-sorted | All run, results aggregated |

## Examples

### WorkflowGate Protocol

```python
@runtime_checkable
class WorkflowGate(Protocol):
    gate_id: ClassVar[str]              # "gate-tests"
    description: ClassVar[str]          # "All tests pass"
    workflow_point: ClassVar[str]       # "before:release:publish"

    def evaluate(self, context: GateContext) -> GateResult: ...
```

**IMPORTANT:** `gate_id`, `description`, `workflow_point` are `ClassVar` — same pattern as `LifecycleHook.events` and `LifecycleHook.priority`. This enables Protocol conformance checking via `isinstance()`.

### GateContext and GateResult

```python
@dataclass(frozen=True)
class GateContext:
    gate_id: str
    working_dir: Path = field(default_factory=Path.cwd)

@dataclass(frozen=True)
class GateResult:
    passed: bool
    gate_id: str
    message: str = ""        # Human-readable (actionable for failures)
    details: tuple[str, ...] = ()  # Additional detail lines
```

### Gate Implementation (reference for S248.6)

```python
class TestGate:
    gate_id = "gate-tests"
    description = "All tests pass"
    workflow_point = "before:release:publish"

    def evaluate(self, context: GateContext) -> GateResult:
        result = subprocess.run(["pytest", "-x", "--tb=short"], capture_output=True)
        return GateResult(
            passed=result.returncode == 0,
            gate_id=self.gate_id,
            message="Tests pass" if result.returncode == 0 else "Tests failing",
            details=(result.stdout.decode(),) if result.returncode != 0 else (),
        )
```

### GateRegistry (mirrors HookRegistry)

```python
EP_GATES: str = "rai.gates"

class GateRegistry:
    def discover(self) -> None: ...       # Load from rai.gates entry points
    def register(self, gate) -> None: ... # Manual registration (testing)
    def get_gate(self, gate_id: str) -> WorkflowGate | None: ...
    def get_gates_for_point(self, workflow_point: str) -> list[WorkflowGate]: ...

    @property
    def gates(self) -> list[WorkflowGate]: ...
```

## Acceptance Criteria

**MUST:**
- `WorkflowGate` Protocol is `@runtime_checkable` with pyright validation
- `GateContext` and `GateResult` are frozen dataclasses
- `GateRegistry.discover()` loads from `rai.gates` entry points
- `rai gate check <gate-id>` runs one gate, returns exit code 0/1
- `rai gate check --all` runs all gates, returns exit code 0 (all pass) or 1 (any fail)
- `rai gate list` shows discovered gates with id, description, workflow_point
- Gate exception in `evaluate()` → `GateResult(passed=False)`, not CLI crash

**SHOULD:**
- `--format json` output for programmatic consumption
- `rai gate check --all` shows summary line

**MUST NOT:**
- Depend on event emitter (AD-5: gates are standalone)
- Import anything from `rai_cli.hooks` (complete independence)

## Notes

- **No built-in gates in this story.** S248.6 adds the 4 built-in gates + GateBridgeHook.
- **`workflow_point` as ClassVar:** Used by GateBridgeHook (S248.6) and `get_gates_for_point()`. For manual `rai gate check <id>`, the workflow_point is metadata, not a filter.
- **Error isolation pattern:** Same as hooks — `try/except Exception` around `evaluate()`, convert to `GateResult(passed=False, message=str(exc))`.
