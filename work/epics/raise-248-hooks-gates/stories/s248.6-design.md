# S248.6 Design: Built-in Gates + GateBridgeHook

## What & Why

**Problem:** S248.5 built the gate infrastructure (Protocol, Registry, CLI) but no actual gates exist. The `rai gate check --all` command discovers nothing. Gates and hooks are independent systems with no bridge.

**Value:** 4 quality gates become executable via `rai gate check`. GateBridgeHook wires them into `before:` events, enabling auto-enforcement before releases. Completes M2 (Auto-firing).

## Approach

Two independent deliverables that share no code:

1. **4 built-in gates** in `src/rai_cli/gates/builtin/` — each runs a subprocess and returns `GateResult`. Mechanical: same pattern as the `TestGate` reference in S248.5 design.

2. **GateBridgeHook** in `src/rai_cli/hooks/builtin/gate_bridge.py` — a `LifecycleHook` that subscribes to `before:` events, discovers gates for that workflow point, runs them, and returns `abort` if any fail. This is the bridge between the two independent systems (PAT-E-454).

### Gate Implementations

All 4 gates follow identical structure:

```python
class TestGate:
    gate_id = "gate-tests"
    description = "All tests pass"
    workflow_point = "before:release:publish"

    def evaluate(self, context: GateContext) -> GateResult:
        result = subprocess.run(
            ["pytest", "-x", "--tb=short"],
            capture_output=True, text=True,
            cwd=str(context.working_dir),
        )
        return GateResult(
            passed=result.returncode == 0,
            gate_id=self.gate_id,
            message="Tests pass" if result.returncode == 0 else "Tests failing",
            details=(result.stdout,) if result.returncode != 0 else (),
        )
```

| Gate | Command | gate_id |
|------|---------|---------|
| TestGate | `pytest -x --tb=short` | `gate-tests` |
| TypeGate | `pyright` | `gate-types` |
| LintGate | `ruff check .` | `gate-lint` |
| CoverageGate | `pytest --cov --cov-report=term-missing -q` | `gate-coverage` |

**Coverage note:** CoverageGate checks that `pytest --cov` succeeds (exit 0), not a specific percentage. Per PAT-E-444, coverage is diagnostic, not a hard gate.

### GateBridgeHook

```python
class GateBridgeHook:
    events = ["before:release:publish", "before:session:close"]
    priority = 100  # Run before other hooks (gates must pass first)

    def handle(self, event: HookEvent) -> HookResult:
        # Map event_name to workflow_point
        workflow_point = event.event_name  # "before:release:publish"
        registry = GateRegistry()
        registry.discover()
        gates = registry.get_gates_for_point(workflow_point)
        if not gates:
            return HookResult(status="ok")

        failures = []
        for gate in gates:
            context = GateContext(gate_id=gate.gate_id)
            try:
                result = gate.evaluate(context)
            except Exception as exc:
                result = GateResult(
                    passed=False, gate_id=gate.gate_id,
                    message=f"{type(exc).__name__}: {exc}",
                )
            if not result.passed:
                failures.append(result)

        if failures:
            summary = "; ".join(f"{f.gate_id}: {f.message}" for f in failures)
            return HookResult(status="abort", message=f"Gates failed: {summary}")
        return HookResult(status="ok")
```

**Key design decisions:**
- **Priority 100** — gates run before telemetry (priority 0) on `before:` events
- **Own GateRegistry** — bridge creates its own registry instance (no shared state)
- **Error isolation** — gate exceptions become `GateResult(passed=False)`, not hook crashes

## Acceptance Criteria

**MUST:**
- 4 gates discoverable via `rai gate check --all` and `rai gate list`
- Each gate runs its subprocess in `context.working_dir`
- GateBridgeHook returns `abort` when any gate fails on `before:` events
- GateBridgeHook returns `ok` when all gates pass or no gates match
- Error isolation: gate `evaluate()` exceptions → `GateResult(passed=False)`
- Entry points in pyproject.toml for all 4 gates + bridge hook

**SHOULD:**
- GateBridgeHook logs which gates it runs (DEBUG level)

**MUST NOT:**
- Gates must not import from `rai_cli.hooks` (AD-5: independence)
- Bridge hook must not hard-code gate IDs — discovers dynamically via registry
