# S248.3 Design: Built-in TelemetryHook

## What & Why

**Problem:** Skills manually call `rai signal emit-work` and `rai signal emit` for telemetry. This is ceremony (~1,000 tokens/invocation) that's fragile — if a skill forgets to emit, telemetry has gaps.

**Value:** First real LifecycleHook that proves the hook system works E2E. Automatic telemetry from CLI events without skill intervention. Zero-gap coverage once S248.4 wires events.

## Approach

Create a `TelemetryHook` class that implements `LifecycleHook` Protocol. It subscribes to all 9 after-events and writes `CommandUsage` signals to `signals.jsonl` via the existing `telemetry.writer.emit()` function.

**Components:**
- **Create:** `src/rai_cli/hooks/builtin/telemetry.py` — TelemetryHook implementation
- **Create:** `src/rai_cli/hooks/builtin/__init__.py` — package init
- **Modify:** `pyproject.toml` — register entry point under `rai.hooks`
- **Create:** `tests/test_hooks/test_telemetry_hook.py` — tests

**Why `CommandUsage`:** The lifecycle events (session:start, graph:build, etc.) represent CLI operations. `CommandUsage` is the natural fit — command name maps from event name (`"session:start"` → command=`"session"`, subcommand=`"start"`). WorkLifecycle signals (story/epic phases) are a different concern — those stay as manual `rai signal emit-work` until S248.7 evaluates what to do with them.

## Examples

### TelemetryHook usage

```python
from rai_cli.hooks.builtin.telemetry import TelemetryHook

hook = TelemetryHook()
assert hook.events == [
    "session:start", "session:close", "graph:build",
    "pattern:added", "discover:scan", "init:complete",
    "adapter:loaded", "adapter:failed", "release:publish",
]
assert hook.priority == 0  # default, runs after higher-priority hooks

result = hook.handle(SessionStartEvent(session_id="SES-265"))
assert result.status == "ok"
# → signals.jsonl now has: {"type":"command_usage","command":"session","subcommand":"start",...}
```

### Entry point registration (pyproject.toml)

```toml
[project.entry-points."rai.hooks"]
telemetry = "rai_cli.hooks.builtin.telemetry:TelemetryHook"
```

### Signal output format

```json
{"type":"command_usage","timestamp":"2026-02-23T22:00:00Z","command":"graph","subcommand":"build"}
{"type":"command_usage","timestamp":"2026-02-23T22:01:00Z","command":"session","subcommand":"start"}
```

### Error isolation

```python
# If emit() fails (disk full, permission error), hook logs warning and returns ok
# Hook never raises — returns HookResult(status="error", message=...)
result = hook.handle(event)
assert result.status in ("ok", "error")  # never raises
```

## Acceptance Criteria

**MUST:**
- TelemetryHook implements LifecycleHook Protocol (passes `isinstance` check)
- Subscribes to all 9 after-events
- Writes `CommandUsage` signals via `telemetry.writer.emit()`
- Event name maps to command/subcommand: `"session:start"` → `("session", "start")`
- Entry point registered and discoverable via `HookRegistry.discover()`
- Error isolation: `emit()` failure → `HookResult(status="error")`, never raises

**SHOULD:**
- priority=0 (default, lowest — telemetry observes, doesn't interfere)
- Logging at DEBUG level on successful emit, WARNING on failure

**MUST NOT:**
- Write WorkLifecycle signals (that's a different concern, stays manual until S248.7)
- Import or depend on anything outside `rai_cli.hooks` and `rai_cli.telemetry`
- Raise exceptions from `handle()`
