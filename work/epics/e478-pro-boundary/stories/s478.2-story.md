# S478.2 — Clean Entry Points and Dependencies

## User Story

**As a** community user installing raise-cli,
**I want** the package to contain zero pro-only entry points and dependencies,
**So that** I get a lean install without Atlassian/Jira/Confluence packages I don't need.

## Acceptance Criteria

```gherkin
Feature: Clean community package boundaries

  Scenario: Community install has no pro dependencies
    Given a fresh virtualenv
    When I run "pip install raise-cli"
    Then atlassian-python-api is NOT installed
    And authlib is NOT installed
    And cryptography is NOT installed

  Scenario: Pro entry points only register with raise-pro
    Given raise-cli is installed without raise-pro
    When I run "rai adapter list"
    Then only "filesystem" adapter is shown
    And no "jira" or "confluence" adapters appear

  Scenario: Pro entry points register when raise-pro is installed
    Given raise-cli is installed
    And raise-pro is installed
    When I run "rai adapter list"
    Then "jira" and "confluence" adapters are shown

  Scenario: Pro package declares its own entry points
    Given packages/raise-pro/pyproject.toml exists
    Then it declares jira adapter entry point
    And it declares confluence adapter entry point
    And it declares jira-sync hook entry point
    And it declares pro-only dependencies

  Scenario: All tests pass
    Given both community and pro code are present
    When I run the test suite
    Then all tests pass
```

## Context

S478.1 moved pro adapter code to `src/rai_pro/adapters/` and updated entry points
to reference `rai_pro` paths. However, those entry points are still declared in
`raise_cli`'s `pyproject.toml`, meaning a community-only install still tries to
register them (and fails). Pro-only dependencies also remain in the community
package's dependency list.

This story completes the separation by moving entry points and dependencies to
a dedicated `raise-pro` workspace package.
