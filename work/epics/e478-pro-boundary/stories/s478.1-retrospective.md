# S478.1 Retrospective — Move Pro Adapters to rai_pro

## Summary

Moved three pro-only files (mcp_jira.py, mcp_confluence.py, jira_sync.py) from `raise_cli` to `rai_pro`, updated all test imports and patch targets, and fixed entry points to point to the new locations. All 3715 tests pass, type checks and lint clean.

## What Went Well

- **Clean move:** The file moves were straightforward with no logic changes required. Internal imports to shared raise_cli infrastructure (models, bridge, events) stayed unchanged as planned.
- **TDD gate caught scope creep early:** Entry point discovery tests failed after the move, forcing entry point updates in the same story rather than deferring to S478.2. This was the right call — gates enforce correctness.
- **AR found real issues:** The acceptance review caught a stale `jira-sync` entry point and a missing `rai_pro` sdist include — both one-line fixes (commit e89d3154).
- **QR clean:** Zero issues found in quality review. The move was mechanical and well-planned.

## What to Improve

- **Plan should account for entry point updates:** The plan scoped only file moves and test import updates. Entry points are a known coupling in this codebase — should have been in the plan from the start.
- **Estimate was accurate overall:** 25 min estimated, ~25 min actual. The AR fix added minimal overhead.

## Heutagogical Checkpoint

### 1. What did you learn?
Entry point declarations are tightly coupled to module paths. Moving code between packages forces entry point updates in the same change — you cannot defer them without breaking discovery tests. This is actually a feature of the entry point architecture: it prevents stale references from shipping.

### 2. What would you change about the process?
Include entry point audit as an explicit step in any plan that moves modules registered as entry points. A simple `grep` for the module path in `pyproject.toml` during planning would have caught this.

### 3. Are there improvements for the framework?
Consider a planning checklist item: "Does this story move any entry-point-registered modules? If yes, include entry point update tasks." Could be part of a move-refactor kata.

### 4. What are you more capable of now?
Better at anticipating the full dependency chain of file moves in a plugin-architecture codebase. Entry points, sdist includes, and test patch targets all need updating atomically.

## Patterns

- **PAT-E-599 (candidate):** When moving code between packages, entry point discovery tests force updating entry points in the same story — can't defer without failing gates. Include entry point audit in any plan that moves registered modules.
