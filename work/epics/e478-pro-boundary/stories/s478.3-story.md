# S478.3: Clean pro-specific CLI logic

## User Story (Connextra)

**As a** community contributor,
**I want** the raise_cli source to contain zero Jira/Confluence-specific logic,
**So that** the open-source CLI is cleanly decoupled from pro adapters.

## Acceptance Criteria (Gherkin)

### Scenario: No Jira-specific logic in CLI source
```gherkin
Given the raise_cli source tree under src/raise_cli/
When I search for Jira or Confluence references
Then zero matches are found in production code
```

### Scenario: Adapter list still works
```gherkin
Given raise-cli is installed without raise-pro
When I run `rai adapter list`
Then the filesystem adapter is listed
And no Jira or Confluence adapters appear
```

### Scenario: Adapter list with pro installed
```gherkin
Given raise-cli is installed with raise-pro
When I run `rai adapter list`
Then filesystem, jira, and confluence adapters are listed
```

### Scenario: All tests pass
```gherkin
Given the Jira-specific CLI logic has been removed
When I run the full test suite
Then all tests pass
```
