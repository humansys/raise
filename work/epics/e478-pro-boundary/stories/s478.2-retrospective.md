# S478.2 — Retrospective: Clean Entry Points and Dependencies

## Summary

Created the `raise-pro` workspace package under `packages/raise-pro/`, moved pro entry points (jira, confluence, jira-sync) and 6 pro-only dependencies out of `raise-cli`, verified all gates pass.

## Key Risk Materialized

**R-478.2-1 (Hatch out-of-tree paths):** Confirmed. Hatch wheel config does NOT support `../../src/rai_pro` relative paths from `packages/raise-pro/`. ValueError on build. The design had anticipated this as a risk with a documented fallback: move `src/rai_pro` into `packages/raise-pro/src/rai_pro`. The fallback was executed, resulting in one extra commit for import sort order fix.

## Heutagogical Reflection

### What did I learn?
Hatch's wheel builder enforces that source packages must live inside (or at) the package root directory. Out-of-tree relative paths fail with a ValueError. This is not documented prominently — we discovered it empirically. The design's risk table (R-478.2-1) saved us: having a pre-planned fallback meant the deviation added ~5 minutes, not hours.

### What would I do differently?
Start with the fallback (physical file move) as the primary approach. The "keep files in place" approach was optimistic. When a tool's path resolution behavior is unknown, default to the conventional layout rather than testing the creative one first.

### What surprised me?
How clean the workspace pattern is with `uv`. Adding `packages/*` to workspace members and `raise-pro = { workspace = true }` to sources — two lines — and everything resolves. The uv workspace model is genuinely well-designed.

### What pattern should be captured?
Hatch wheel config requires source code inside the package directory. Out-of-tree relative paths fail with ValueError. Plan for physical file moves when creating workspace packages.
