### 3.3 Community retrofit from server design

| Retrofit | Server design informs | Community implementation |
|----------|----------------------|------------------------|
| **UUID generation** | Server uses UUIDs internally | CLI generates UUIDs offline — no counter files, no coordination needed |
| **Atomic write protocol** | Server uses DB transactions | CLI uses tmp+rename consistently (already proven in backlog/sync.py) |
| **Heartbeat file** | Server has session registry API | CLI writes `.raise/rai/personal/heartbeat` with PID + timestamp, session-start reads all heartbeats |
| **Graph node ID from Jira key** | Server uses Jira key as canonical ID | CLI uses `RAISE-{N}` as epic node ID instead of directory-derived `E{int(N)}` |

## 4. Recommended 2.4 Quick Wins (priority order)

1. **Fix epic ID normalization** (G6) — change `epic.py:85` from `E{int(N)}` to use full directory name or Jira key. Resolves 3 open bugs.
2. **Add collision check to epic-start** — validate directory uniqueness across worktrees. Resolves 1 open bug.
3. **Atomic writes everywhere** — apply tmp+replace pattern to all `write_text()` calls on state files. Prevents corruption class.
4. **File locking on JSONL appends** — extend telemetry pattern to all append-only files. Prevents data loss class.
5. **Session state from git** (RAISE-822) — biggest effort but eliminates the entire "manual state ≠ reality" failure mode.

## 5. Architecture Observations

### What's already good
- **Telemetry writer** (`telemetry/writer.py`) is the gold standard: file locking, session-scoped paths, clean separation
- **Backlog sync** (`backlog/sync.py`) uses proper atomic writes
- **Pattern reinforcement** (`memory/writer.py:512-517`) uses atomic tmp+replace
- **Session identity** (`session/identity.py`) is timestamp-based — no counter, no coordination
- **Prefix registry** (`session/prefix.py`) has collision detection built in

### What needs work
- **12 of 16 state files** use raw `write_text()` — no atomicity, no locking
- **5 of 9 ID generators** use scan-for-max — vulnerable to counter reset
- **Epic parser** uses `int()` normalization — the single most damaging design choice (4 bugs)
- **Session close** writes to 6 different files in sequence with no transaction — partial failure leaves inconsistent state
- **`_append_jsonl()`** is used by 3 callers but has no locking — the telemetry writer solved this but the pattern wasn't applied broadly
