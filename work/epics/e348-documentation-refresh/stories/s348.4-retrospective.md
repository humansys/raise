## What Went Well

1. **Source-first approach worked.** Reading `protocols.py`, `registry.py`, `catalog.yaml`, and `pre-compact-journal.sh` before writing guides ensured accuracy. Every code example references real patterns from the codebase.

2. **Consistent format.** Following the existing guide pattern (frontmatter + intro + sections) made the new pages feel native to the docs site.

3. **Astro build as gate.** Running the build after changes caught nothing (clean pass), but having it as a verification step is reliable.

## What Could Improve

1. **Hooks section is thin.** The `.claude/hooks/` directory does not exist in this project — hooks are configured differently (scripts in `.claude/scripts/`). The guide covers the general pattern but could be more specific once Claude Code's hook system stabilizes.

2. **No Spanish translations.** The Astro build generates stub pages for `es/` locale, but actual translations are out of scope. Should be a follow-up story.

## Learnings

- Entry point groups are the stable contract for adapter discovery — documenting these is more valuable than documenting individual adapter implementations.
- Skill frontmatter metadata fields are the key to understanding skill behavior — `raise.work_cycle`, `raise.frequency`, `raise.gate` control when and how skills run.
- MCP catalog format (`catalog.yaml`) is simpler than expected — 6 fields per server.
