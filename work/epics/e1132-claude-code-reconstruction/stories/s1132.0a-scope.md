# S1132.0a Scope — TypeScript Import/Export Extraction

**Jira:** RAISE-1170
**Epic:** E1132 (RAISE-1162)

---

## In Scope

- `TypeScriptAnalyzer` class in `packages/raise-cli/src/raise_cli/context/analyzers/typescript.py`
  - Follows same interface pattern as `PythonAnalyzer`: `detect()`, `analyze_modules()`, returns `list[ModuleInfo]`
  - Uses tree-sitter-typescript (already a dependency via scanner.py)
- Extract static import statements: named, default, namespace, re-exports
- Extract static export statements: named, default, re-exports, export declarations
- Handle `import type` / `export type` (include them, flag optional)
- Handle barrel files (`index.ts`) as module public API (analogous to `__init__.py`)
- Module boundary detection: top-level directories under `src/` are modules
- Return `ModuleInfo` with `language="typescript"`, populated imports/exports/component_count
- Full TDD with unit tests for each import/export pattern
- Integration-level test against a small fixture tree

## Out of Scope

- Dynamic imports (`import()` expressions) — runtime, not static structure
- CommonJS `require()` / `module.exports` — separate story if needed
- Path alias resolution (`@/foo` mapped in tsconfig) — future story
- Cross-package imports in monorepos — future story
- Modifications to `scanner.py` or `extract_typescript_symbols()` — this story creates a new analyzer
- Performance optimization beyond "completes in 60s on Claude Code source" — future story
- Changes to `ModuleInfo` model — reuse as-is

## Done Criteria

2. All Gherkin scenarios from s1132.0a-story.md pass as tests

4. Type checks pass (pyright strict)
5. Linting passes (ruff)
6. Tests pass with coverage for the new module
7. Works on .ts, .tsx, .js, .jsx files
