# S1132.0a Design: TypeScript Import/Export Extraction via Tree-Sitter

**Story:** S1132.0a | **Jira:** RAISE-1170 | **Epic:** E1132
**Complexity:** Moderate (new file + 2 modifications + tests)

---

## Problem

`rai-discover` can extract TypeScript symbols (classes, functions, interfaces) but cannot extract import/export relationships. Without imports and exports, the knowledge graph cannot represent module dependencies for TypeScript codebases. This blocks architecture reconstruction of Claude Code (1902 TS files, 512K LOC).

## Value

Enables automated dependency graphs for TypeScript repositories. The PythonAnalyzer already does this for Python -- this story brings the same capability to TypeScript/JavaScript.

---

## Approach

### New: `TypeScriptAnalyzer` class

File: `packages/raise-cli/src/raise_cli/context/analyzers/typescript.py`

Mirrors PythonAnalyzer's structure:

```
TypeScriptAnalyzer(src_dir: str)
  .detect(project_root: Path) -> bool
  .analyze_modules(project_root: Path) -> list[ModuleInfo]
```

**Key differences from PythonAnalyzer:**

| Aspect | PythonAnalyzer | TypeScriptAnalyzer |
|--------|---------------|-------------------|
| Parser | `ast.parse()` (stdlib) | tree-sitter-typescript (already installed) |
| Module marker | `__init__.py` required | Any dir with `.ts`/`.tsx` files |
| Barrel file | `__init__.py` | `index.ts` / `index.tsx` |
| Project detection | `pyproject.toml` / `setup.py` | `tsconfig.json` or `package.json` with TS files |
| File extensions | `.py` | `.ts`, `.tsx`, `.js`, `.jsx` (skip `.d.ts`) |

### Integration points

1. **`__init__.py`** -- Add `TypeScriptAnalyzer` to `__all__` and imports
2. **`builder.py`** -- Add `TypeScriptAnalyzer(src_dir="src")` to analyzers list in `load_code_structure()`
3. **New test file** -- `tests/context/analyzers/test_typescript.py`

### Tree-sitter node types used

**Imports:**
- `import_statement` -- contains the full import declaration
  - Child `string` node holds the source path (e.g., `'./core/runner'`)
  - `import_clause` children distinguish named/default/namespace imports

**Exports:**
- `export_statement` -- re-exports and named export lists
- `export_declaration` -- wraps `class_declaration`, `function_declaration`, etc. with `export` keyword
- `export_default_declaration` -- `export default class X {}`

**Components (counted):**
- `class_declaration`, `function_declaration`, `interface_declaration`, `enum_declaration`, `type_alias_declaration`

## Import Resolution

Relative imports are resolved to sibling module names, matching PythonAnalyzer's granularity:

```
File: src/commands/run.ts
Import: import { Runner } from '../core/runner'

1. Extract source string: '../core/runner'
2. Resolve relative to file: commands/../core/runner = core/runner
3. Extract first path segment: 'core'
4. 'core' != 'commands' (current module) -> add 'core' to imports
```

**Ignored:**
- External packages (no `./` or `../` prefix, e.g., `import { z } from 'zod'`)
- Self-imports (resolved module == current module)
- Dynamic imports (`import()` expressions)

---

## Barrel File Handling

```typescript
// src/utils/index.ts
export { formatDate } from './date';
export { parseArgs } from './args';
export type { ArgOptions } from './args';
```

- If `index.ts`/`index.tsx` exists in a module directory, use it as the export source
- Re-exports contribute both an export name AND an import dependency
- If no index file exists, aggregate exports from all `.ts`/`.tsx` files

## Examples

### Example 1: Various import styles

```typescript
// src/commands/execute.ts
import { CommandRunner } from '../core/runner';     // named import
import Config from '../config';                      // default import
import * as utils from '../utils';                   // namespace import
import type { Logger } from '../logging';            // type-only import
import { z } from 'zod';                            // external (ignored)

export class ExecuteCommand {}
```

**ModuleInfo contribution from this file:**
- imports: `{"core", "config", "utils", "logging"}` (all resolve to sibling modules; zod ignored)
- exports: `["ExecuteCommand"]`
- component_count: 1

### Example 2: Barrel file (index.ts)

```typescript
// src/utils/index.ts
export { formatDate } from './date';
export { parseArgs } from './args';
export type { ArgOptions } from './args';
export { DEFAULT_TIMEOUT } from './constants';
```

**Module-level ModuleInfo:**
- exports: `["formatDate", "parseArgs", "ArgOptions", "DEFAULT_TIMEOUT"]`
- component_count: sum of classes/functions/interfaces across all files in utils/

### Example 3: Full module analysis

```
src/
  commands/
    index.ts          <- barrel file: re-exports from execute.ts and help.ts
    execute.ts         <- imports from ../core, ../config
    help.ts            <- imports from ../core
  core/
    index.ts          <- barrel file
    runner.ts
    types.ts
```

**Result for "commands" module:**
```python
ModuleInfo(
    name="commands",
    language="typescript",
    source_path="src/commands",
    imports=["config", "core"],
    exports=["ExecuteCommand", "HelpCommand"],
    component_count=4,
    entry_points=[],
)
```

## Acceptance Criteria

### MUST
- AC1: TypeScriptAnalyzer implements CodeAnalyzer protocol
- AC2-AC4: Extract named, default, and namespace imports
- AC5-AC7: Extract re-exports, named exports, and default exports
- AC8: Handle `import type` / `export type` statements
- AC9: Barrel files (index.ts) define module public API
- AC10: ModuleInfo output matches PythonAnalyzer contract
- AC11: Tests, pyright strict, ruff all pass

### SHOULD
- Handle `.tsx` files (JSX in TypeScript)
- Handle `.js` / `.jsx` files (JavaScript with same import syntax)
- Skip `.d.ts` declaration files

### MUST NOT
- Extract dynamic imports (`import()` expressions)
- Extract CommonJS `require()` calls
- Resolve tsconfig path aliases
- Modify the existing `ModuleInfo` model
- Modify `scanner.py`

---

## Decisions

| ID | Choice | Rationale |
|----|--------|-----------|
| D1 | Reuse `_get_ts_parser` from scanner.py | Already handles TS/TSX/JS parser setup with ImportError wrapping |
| D2 | Top-level src/ subdirectories are modules | Mirrors PythonAnalyzer heuristic; works for Claude Code layout |
| D3 | index.ts as barrel file for exports | Standard TS convention, analogous to `__init__.py` |
| D4 | Import source resolves to sibling module name | Same granularity as PythonAnalyzer; `../core/runner` -> `"core"` |
| D5 | detect() checks tsconfig.json or package.json | tsconfig.json is canonical TS marker; package.json as fallback |
| D6 | Skip `.d.ts` files | Declaration files are ambient types, not runtime dependencies |
