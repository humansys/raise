# S1132.0a: TypeScript Import/Export Extraction via Tree-Sitter

**Jira:** RAISE-1170
**Epic:** E1132 (RAISE-1162) — Claude Code Architecture Reconstruction
**Size:** S | **Type:** code

---

## User Story

**As a** RaiSE user analyzing TypeScript codebases,
**I want** the discovery pipeline to extract import and export relationships from TS/JS files,
**So that** the knowledge graph accurately represents module dependencies for TypeScript projects like Claude Code.

---

## Acceptance Criteria

```gherkin
Feature: TypeScript import/export extraction

  Scenario: Extract ES module imports
    Given a TypeScript file with "import { Foo } from './bar'"
    When the TypeScriptAnalyzer processes the file
    Then it reports "bar" as an imported module

  Scenario: Extract namespace imports
    Given a TypeScript file with "import * as utils from '../utils'"
    When the TypeScriptAnalyzer processes the file
    Then it reports "utils" as an imported module

  Scenario: Extract default imports
    Given a TypeScript file with "import Config from './config'"
    When the TypeScriptAnalyzer processes the file
    Then it reports "config" as an imported module

  Scenario: Extract re-exports
    Given a TypeScript file with "export { Foo } from './bar'"
    When the TypeScriptAnalyzer processes the file
    Then it reports "Foo" as an export and "bar" as an imported module

  Scenario: Extract named exports
    Given a TypeScript file with "export class UserService {}"
    When the TypeScriptAnalyzer processes the file
    Then it reports "UserService" as an export

  Scenario: Extract default exports
    Given a TypeScript file with "export default class App {}"
    When the TypeScriptAnalyzer processes the file
    Then it reports "App" as an export (default)

  Scenario: Handle type-only imports
    Given a TypeScript file with "import type { Foo } from './bar'"
    When the TypeScriptAnalyzer processes the file
    Then it reports "bar" as an imported module (type-only flag)

  Scenario: Analyze a module directory
    Given a directory "src/utils/" with multiple .ts files
    When TypeScriptAnalyzer.analyze_modules() processes it
    Then it returns a ModuleInfo with aggregated imports, exports, and component_count

  Scenario: Handle barrel files (index.ts)
    Given an index.ts that re-exports from sibling files
    When TypeScriptAnalyzer extracts exports
    Then it uses the index.ts exports as the module's public API

  Scenario: Scale to Claude Code source
    Given the Claude Code source tree (1902 files, 512K LOC)
    When TypeScriptAnalyzer.analyze_modules() processes it
    Then it completes within 60 seconds and returns valid ModuleInfo list
```

## Specification by Example

### Example 1: Named imports and exports

```typescript
// src/commands/index.ts
import { CommandRunner } from '../core/runner';
import type { Config } from '../config';
export { CommandRunner };
export class CommandRegistry {}
```

**Expected ModuleInfo contributions:**
- imports: `["core"]` (sibling module, not "config" as type-only)
- exports: `["CommandRunner", "CommandRegistry"]`
- component_count: 1 (CommandRegistry class)

### Example 2: Barrel file (index.ts)

```typescript
// src/utils/index.ts
export { formatDate } from './date';
export { parseArgs } from './args';
export type { ArgOptions } from './args';
```

**Expected exports:** `["formatDate", "parseArgs", "ArgOptions"]`

### Example 3: Dynamic imports (not extracted)

```typescript
const mod = await import('./lazy-module');
```

**Expected:** Not extracted (dynamic imports are runtime, not static structure).
