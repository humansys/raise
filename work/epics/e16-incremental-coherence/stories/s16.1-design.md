## Problem

The unified graph builds module nodes from manually-maintained YAML frontmatter in `governance/architecture/modules/*.md`. When code changes (new imports, new exports, new classes), the graph doesn't know — it reflects what Rai wrote during the last discovery, not what the code actually is. This makes the graph unreliable for design decisions, context loading, and future diff detection.

## Value

Module nodes in the graph reflect real code structure. `rai memory context mod-memory` returns truth — real imports, real exports, real component counts. Every session starts from accurate data. The graph becomes a live model of the codebase, not a stale snapshot.

## Approach

Add a pluggable code analysis layer to `UnifiedGraphBuilder` that extracts module-level structure from source code using Python's `ast` module. Architecture uses Protocol pattern for future multi-language support.

### Components

| Component | Action | Location |
|-----------|--------|----------|
| `CodeAnalyzer` Protocol | **create** | `src/rai_cli/context/analyzers/protocol.py` |
| `ModuleInfo` model | **create** | `src/rai_cli/context/analyzers/models.py` |
| `PythonAnalyzer` | **create** | `src/rai_cli/context/analyzers/python.py` |
| `UnifiedGraphBuilder` | **modify** | `src/rai_cli/context/builder.py` |
| `context/__init__.py` | **modify** | Export new public API |

### Architecture

```
UnifiedGraphBuilder.build()
  │
  ├── load_governance()        (existing)
  ├── load_memory()            (existing)
  ├── load_work()              (existing)
  ├── load_skills()            (existing)
  ├── load_components()        (existing)
  ├── load_architecture()      (existing — reads frontmatter)
  ├── load_identity()          (existing)
  └── load_code_structure()    (NEW — reads actual source code)
        │
        ├── detect analyzers for project
        ├── PythonAnalyzer.analyze_modules()
        │     └── ast.parse() each module → ModuleInfo
        └── convert ModuleInfo → ConceptNode updates
              └── enrich existing mod-* nodes with real data
```

## Examples

### Protocol and Models

```python
from typing import Protocol, runtime_checkable
from pydantic import BaseModel

class ModuleInfo(BaseModel):
    """Language-agnostic module analysis result."""
    name: str
    language: str
    source_path: str
    imports: list[str]        # other modules this one imports from
    exports: list[str]        # public API (names exported)
    component_count: int      # classes + functions defined
    entry_points: list[str]   # CLI commands, if detectable

@runtime_checkable
class CodeAnalyzer(Protocol):
    """Contract for language-specific code analyzers."""
    def detect(self, project_root: Path) -> bool: ...
    def analyze_modules(self, project_root: Path) -> list[ModuleInfo]: ...
```

### PythonAnalyzer Usage

```python
analyzer = PythonAnalyzer(src_dir="src/rai_cli")
if analyzer.detect(project_root):
    modules = analyzer.analyze_modules(project_root)
```

### Builder Integration

```python
# In UnifiedGraphBuilder.build():
# After load_architecture() creates mod-* nodes from frontmatter,
# load_code_structure() enriches them with real data.

def load_code_structure(self) -> list[ConceptNode]:
    """Enrich module nodes with real code analysis."""
    analyzers = self._get_analyzers()
    code_modules: list[ModuleInfo] = []
    for analyzer in analyzers:
        if analyzer.detect(self.project_root):
            code_modules.extend(analyzer.analyze_modules(self.project_root))
    return self._merge_code_info(code_modules)
```

### What Changes in Graph Output

Before (from frontmatter only):
```
mod-memory:
  depends_on: [config, context]          ← what Rai wrote manually
  public_api: [append_pattern, ...]      ← what Rai listed
  components: 30                          ← what Rai counted
```

After (enriched with real code data):
```
mod-memory:
  depends_on: [config, context]          ← from frontmatter (preserved)
  code_imports: [config, context, core]  ← from ast analysis (NEW, real)
  code_exports: [append_pattern, ...]    ← from __init__.py (NEW, real)
  code_components: 28                    ← from ast count (NEW, real)
```

**IMPORTANT:** Code analysis data stored in `metadata` under `code_*` prefix to distinguish from manually-maintained frontmatter fields. This preserves backward compatibility — existing `depends_on` from frontmatter still works, `code_imports` is the ground truth from code.

## Acceptance Criteria

**MUST:**
- `CodeAnalyzer` Protocol with `detect()` and `analyze_modules()` methods
- `ModuleInfo` Pydantic model with all fields typed
- `PythonAnalyzer` correctly extracts imports from `ast` (handles relative, absolute, conditional)
- `PythonAnalyzer` correctly extracts exports from `__init__.py`
- Builder integrates code analysis into module node metadata
- Unit tests covering: normal imports, relative imports, `__init__` exports, no-source-found

**SHOULD:**
- Component count from ast (classes + top-level functions)
- Handle edge cases: `TYPE_CHECKING` imports, `try/except` imports
- `rai memory context mod-*` shows code analysis data

**MUST NOT:**
- Break existing graph build (backward compatible)
- Replace frontmatter data — enrich alongside it
- Analyze non-Python files in this story
