### G9: Parallel Coordination — Scoring Rationale

- **Value to Core (1):** RaiSE's current model is single-developer + single-agent. File-lock coordination matters at enterprise scale (multiple agents on same repo), not for current users. Memory and methodology (the moat) don't depend on parallel coordination.
- **Simplicity Cost (3):** High — bare-git, file-locks, merge conflict resolution, Docker containers. Art.7 used ~$20K and 2000 sessions. This is infrastructure-heavy.
- **Observability Gain (1):** Marginal — parallel agent work is harder to trace (EO.8: non-determinism). Observability actually *decreases* with parallelism unless heavy investment in tracing.
- **Effort (L):** Infrastructure: bare-git server, container orchestration, lock file protocol, conflict resolution patterns.

### Specialized Roles — Scoring Rationale

- **Value to Core (2):** Aligns with existing skill system. RaiSE already has 19 skills that provide specialization. The gap is giving agents *identity* per role (critic agent vs implementer agent), not creating new capabilities.
- **Simplicity Cost (1):** Low — Claude Code already supports custom subagents via `.claude/agents/` with role-specific instructions and tool restrictions. No new infrastructure needed.
- **Observability Gain (2):** Moderate — knowing which "role" produced which output improves auditability. "Security reviewer flagged X" is more traceable than "agent flagged X."
- **Effort (S):** Define role templates in `.claude/agents/`, map to RaiSE skill phases.

### Orchestrator Scaling — Scoring Rationale

- **Value to Core (1):** Single-developer + HITL model doesn't need an agent orchestrator. Human IS the orchestrator. Value emerges only at enterprise scale with multiple concurrent agents.
- **Simplicity Cost (3):** Very high — orchestrator logic, scaling rules, token budget management, failure recovery, non-determinism handling. Art.6 acknowledges this is complex and still synchronous.
- **Observability Gain (1):** Same as G9 — parallelism reduces observability unless heavily instrumented.
- **Effort (XL):** Full orchestration framework: task queue, agent pool, result aggregation, quality verification across agents.

## 3. P1 Deliberation — Orchestrator Scaling Under ADR-034

### The Question

Under ADR-034's reformulated P1, where does "agent dispatches work to other agents" fall in the Human/Machine/Shared classification?

### Analysis

Two distinct patterns emerged from the articles:

**Pattern A: Art.6 — Lead Agent as Orchestrator**

The lead agent decides how many subagents to spawn, what each investigates, and synthesizes results. The *human* defines the research question; the *agent* decides execution strategy.

| Activity | Classification | Test |
|----------|---------------|------|
| Defining the research question | Human-owned | Setting goals |
| Deciding how many subagents | Machine-negotiable | Execution optimization |
| Assigning subagent focus areas | Machine-negotiable | Task decomposition |
| Synthesizing results | Machine-negotiable | Execution within boundaries |
| Deciding if more research needed | Shared | Quality judgment |

**Pattern B: Art.7 — No Orchestrator (Flat + Environment)**

Humans design the environment (tests, CI, lock protocol). Agents autonomously select and execute tasks. No agent dispatches to other agents.

| Activity | Classification | Test |
|----------|---------------|------|
| Designing the test suite | Human-owned | Defining success criteria |
| Designing the coordination protocol | Human-owned | Setting boundaries |
| Selecting which task to work on | Machine-negotiable | Execution choice |
| Resolving merge conflicts | Machine-negotiable | Execution within git boundaries |
| Walking away during execution | Human — conscious delegation | Boundaries already set |

### Key Finding

Both patterns are compatible with P1 reformulated:

- **Pattern A** works because the human defines the goal and the agent negotiates how to distribute execution. The lead agent is doing task decomposition (Machine-negotiable per ADR-034).
- **Pattern B** works because the human defines the boundaries (tests, locks, CI) and agents operate within them. Even more conservative — no agent-to-agent delegation, just autonomous agents within human-designed constraints.

### The Scale Threshold

Under what conditions does human-as-orchestrator break down?

From the evidence:
- **Art.6 scaling rules:** Simple=1 agent, Comparison=2-4, Complex=10+. At 10+ agents, human orchestration becomes impractical — too many parallel contexts to track.
- **Art.7 example:** 16 agents, human "mostly walked away." Human designed the environment but didn't orchestrate execution.

**Threshold under P1 reformulated:** Human orchestration is natural for 1-3 agents (current RaiSE model). At 4+ parallel agents, the human should shift from orchestrator to *environment designer* (Pattern B) or accept agent-orchestration within human-defined boundaries (Pattern A). Both are compatible with P1 reformulated.

### Verdict

Orchestrator scaling is **compatible with P1 reformulated** at any scale, provided:
1. Human defines goals and success criteria (non-negotiable)
2. Human designs the coordination boundaries (locks, tests, CI)
3. Agent-to-agent dispatch is task decomposition, not goal redefinition
4. Results are human-reviewable (HITL gate on final output)

This provides the third piece of supporting evidence for ADR-034 (after G13/S789.2 and G11/S789.3).

## 4. Verdicts

### G9: Parallel Coordination — **Defer** (Enterprise)

**Priority score: -1** (Reject for OSS, plan for Enterprise)

RaiSE's single-developer model doesn't need file-lock coordination today. The infrastructure cost (bare-git, containers, lock protocol) is too high for current value.

**What to note for Enterprise (E9):**
- Art.7's pattern (bare-git + file-locks + autonomous agents) is proven at scale (16 agents, ~2000 sessions)
- Art.6's pattern (lead + subagents) is simpler but synchronous (bottleneck acknowledged)
- Worktree awareness (already planned in RaiSE) is the lightweight precursor — get that working first
- Non-determinism (EO.8) is a fundamental challenge that requires observability investment before scaling

**No implementation story for OSS.** Enterprise design note added.

### Specialized Roles — **Adapt** (Both OSS + Enterprise)

**Priority score: 3** (Plan for 2.4.x)

RaiSE already has the mechanism: skills provide specialization, Claude Code supports custom subagents. The gap is formalization.

**What to adopt:**
- Define role templates in `.claude/agents/` mapped to RaiSE concerns: security-reviewer, architecture-critic, test-writer
- Each role gets specific tool restrictions and evaluation criteria
- Roles align with existing skill phases: design agent, implementation agent, review agent

**What to adapt:**
- RaiSE's Rai identity is unified (one entity, not multiple personalities). Roles are *modes*, not separate identities. "Rai in review mode" vs "a review agent" — preserves identity coherence (ADR-013/014).
- Role templates reference RaiSE governance artifacts (code-standards, guardrails) as evaluation criteria

**Simplicity test (PAT-E-010):** Would a developer define agent roles mid-session? No — but they'd benefit from pre-defined roles that activate per skill phase. Framework-level change, transparent benefit. → **OSS Core** (templates) + **Enterprise** (configurable roles per team).

**Implementation story:** RAISE-827 — Agent role templates aligned with skill phases (S)

### Orchestrator Scaling — **Defer** (Enterprise)

**Priority score: -1** (Reject for OSS, plan for Enterprise)

Human-as-orchestrator works for RaiSE's current model (1-3 agents). Agent-orchestration is compatible with P1 reformulated (§3) but the complexity cost is too high for current value.

**Threshold documented:** At 4+ parallel agents, shift to environment-design (Pattern B from Art.7) or accept agent-orchestration (Pattern A from Art.6). Both require infrastructure RaiSE doesn't have yet.

**No implementation story for OSS.** Enterprise design note added.
