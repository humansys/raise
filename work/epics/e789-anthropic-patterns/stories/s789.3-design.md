# S789.3 Design — Tool & MCP Patterns Research

## Methodology

Same 4-phase approach validated in S789.1 and S789.2:
1. **Deep Read** — Extract claims from source articles
2. **Evidence Catalog** — Triangulate claims, assign confidence levels
3. **Benchmark Matrix** — Score gaps on 4+1 dimensions
4. **Verdicts & Backlog** — Adopt/Adapt/Reject per gap, create implementation stories

| Art.5 — Writing Tools for Agents | Deep read | Primary source for G6, G7, G11 |
| Art.8 — Claude Code Best Practices | Deep read | Primary source for G6, G7 (CLAUDE.md patterns) |

## Design Decisions

### D1: Article scope — Art.9 superficial only
Art.9 (Code Execution with MCP) is about sandbox execution, not tool design. Deep read would add ~15 min for low-relevance content. Extract the code-mode principle only.

### D2: G8 (Code-mode MCP) — principle extraction, not formal scoring
Epic scope already marks G8 as "Low relevance — RaiSE is Python CLI, not TypeScript." Instead of 4-dimension scoring, extract the underlying principle (tools-as-code vs tools-as-functions) and mark as "Not Applicable — principle noted." Saves time without losing insight.

### D3: G11 as P1 reformulation test case
G11 (agent self-improvement of tool descriptions) is the first gap that directly tests ADR-034's reformulated P1. Evaluate explicitly:
- Under P1 original: violation (machine changing its own tools = redefining execution)
- Under P1 reformulated: potentially acceptable IF within boundaries (git-tracked, human-reviewable)
- This provides concrete evidence for ADR-034 ratification

### D4: Analytics question out of scope
Research question 5 (tool usage analytics) has no associated gap. Include only if it emerges naturally from article evidence. No dedicated investigation time.
