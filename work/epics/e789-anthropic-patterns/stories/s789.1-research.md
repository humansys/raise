## Section 1: Evidence Catalog

### G4: Context Management Strategy

---

**E4.1 — Context rot: recall degrades as context grows**
- Source: Art.4 (needle-in-haystack benchmarks)
- Anthropic says: "As the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases." Termed "context rot." Caused by transformer attention budget stretched thin across n² token pairings.
- RaiSE reality: Session bundles are ~600 tokens (lean). CLAUDE.md is ~10.5K chars (high). No explicit policy to manage context during long tasks.
- Triangulation 2 (RaiSE operational data): Sessions regularly hit compaction after long implementation phases. Post-compaction recovery is fragile (bugs #12671, #15174). This friction is documented in blueprint G-CTX-2.
- Triangulation 3 (first principles): If attention budget degrades with token count, then our largest context load (CLAUDE.md at session start) may be less efficiently processed mid-session when context is full. The math supports the claim.
- **Confidence: High** — Mechanistically grounded + matches our observed fragility.
- Maps to: G4, G-CTX-1, G-CTX-2

---

**E4.2 — Compaction strategy: maximize recall first, then precision**
- Source: Art.4
- Anthropic says: "Maximize recall to ensure your compaction prompt captures every relevant piece of information from the trace, then iterate to improve precision." The "safest lightest touch" form is tool result clearing (now a platform feature).
- RaiSE reality: No explicit compaction policy. We rely on Claude Code's automatic compaction. The PreCompact hook exists but post-compaction injection is broken.
- Triangulation 2: RaiSE memory graph and journal were designed exactly for this — persist decisions outside context so they survive compaction. The mechanism exists; the explicit policy connecting them to compaction doesn't.
- Triangulation 3: The recall-first principle aligns with our MVC (Minimum Viable Context) design. We already optimize for signal density; we just haven't formalized compaction as a lifecycle event.
- **Confidence: High** — Principle is sound and aligns with existing mechanisms. Gap is policy formalization.
- Maps to: G4, G-CTX-1

---

**E4.3 — Just-in-time context retrieval outperforms pre-loading**
- Source: Art.4 (Claude Code example), Art.1 (SWE-bench agent)
- Anthropic says: Agents should maintain "lightweight identifiers (file paths, stored queries, web links)" and load data dynamically. Claude Code "drops CLAUDE.md files into context up front, while primitives like glob and grep allow it to navigate its environment and retrieve files just-in-time."
- RaiSE reality: `rai graph query` (MVC) is our just-in-time mechanism — 97% token savings vs reading files. Session bundles are pre-loaded (600 tokens). CLAUDE.md is always-on. The hybrid is already there.
- Triangulation 2: Our own PAT-E-591 confirms this pattern — module-level Path constants as test seams follow the same "reference, don't embed" principle.
- Triangulation 3: The architecture is correct. Skills call CLI tools on-demand rather than pre-loading all context. The gap is making this explicit in skill documentation.
- **Confidence: Very High** — RaiSE already implements this. Not a gap, a confirmation.
- Maps to: G4 (strength, not gap)

---

**E4.4 — Sub-agents return condensed summaries (1000-2000 tokens)**
- Source: Art.4
- Anthropic says: Sub-agents "might explore extensively, using tens of thousands of tokens or more, but return only a condensed, distilled summary of their work (often 1,000-2,000 tokens)."
- RaiSE reality: Skills fork subagents (Agent tool). Subagents return brief summaries to orchestrator. But the token budget for summaries is not explicitly specified in skill contracts.
- Triangulation 2: In this session, subagent outputs were several paragraphs (~300-500 tokens). No budget was specified; subagent self-compressed.
- Triangulation 3: Explicit token budgets for subagent returns would improve consistency and reduce context waste.
- **Confidence: High** — Directionally we do this, but without specification.
- Maps to: G4, G-CTX-3

---

**E4.5 — Structured note-taking outside context window for persistence**
- Source: Art.4 (Pokémon example — 1,234 steps), Art.2 (claude-progress.txt)
- Anthropic says: Agents write persistent notes outside the context window, retrieved later. Example: precise state tracking across thousands of game steps.
- RaiSE reality: Journal system (`rai session journal add`) does exactly this. Pattern storage in `patterns.jsonl`. Session narrative captured at close. This is a confirmed strength.
- Triangulation 2: The journal has been used for post-compaction recovery in practice. It works when the hook isn't broken.
- **Confidence: Very High** — This is RaiSE's core mechanism. Already implemented.
- Maps to: G4 (strength, not gap)

### G10: Effort Scaling

---

**E10.1 — "Simplest solution possible" is explicit Anthropic advice**
- Source: Art.1 ("finding the simplest solution possible, and only increasing complexity when needed"), Art.4 ("'do the simplest thing that works' will likely remain our best advice"), Art.6 (scaling rules: 1 agent for simple queries)
- Anthropic says: Three separate articles, different contexts, same principle. Not coincidental.
- RaiSE reality: RaiSE rule #3: "Full Skill Cycle — Use skills even for small stories." This is in direct tension.
- Triangulation 2: Our own rule #11 says "Simple First — Simple heuristics over complex solutions." We already have the principle — but rule #3 overrides it for process.
- Triangulation 3: The ShuHaRi mastery system was designed exactly for this — adapt verbosity and ceremony to context. But rule #3 doesn't allow it.
- **Confidence: High** — The tension is real and documented in three sources.
- Maps to: G10

---

**E10.2 — Explicit scaling rules prevent both over- and under-investment**
- Source: Art.6 (explicit guidelines from implementation: 1 agent = simple, 2-4 = comparison, 10+ = complex research)
- Anthropic says: Early system spawned "50 subagents for simple queries." Adding explicit scaling rules fixed both over-ambition and under-investment.
- RaiSE reality: "Full skill cycle always" addresses under-investment (don't skip). But creates over-investment for XS stories (e.g., a 1-line bug fix going through all 8 phases).
- Triangulation 2 (RaiSE operational data): PAT-E-592 noted "Dogfood stories need ~50% time buffer." This is a calibration problem that explicit effort scaling could address.
- Triangulation 3: ShuHaRi levels already encode effort scaling conceptually. The gap is that rule #3 overrides it.
- **Confidence: High**
- Maps to: G10

---

**E10.3 — Frameworks add abstraction that obscures underlying behavior**
- Source: Art.1 ("frameworks often create extra layers of abstraction that can obscure the underlying prompts and responses")
- Anthropic says: Build with basic components first; reduce abstraction layers for production.
- RaiSE reality: RaiSE *is* a framework. But it's a process framework (markdown skills), not a code abstraction. The critique applies to code frameworks (LangChain etc.), not methodology frameworks.
- Triangulation 2: RaiSE explicitly chose skills over engines (ADR-012) for exactly this reason — 85% complexity reduction by letting the agent read markdown instead of executing state machines.
- **Confidence: Medium** — Partially applicable. The anti-framework critique doesn't directly target RaiSE's approach.
- Maps to: G10 (context only)

### G12: Agent Test Design

---

**E12.1 — Automated tests are the validation mechanism for coding agents**
- Source: Art.1 (SWE-bench coding agent), Art.6 (production tracing)
- Anthropic says: Code solutions are validated via automated tests; agents iterate using test results as feedback. "Measuring performance and iterating on implementations."
- RaiSE reality: 237 tests, 80%+ coverage, pytest. Gate-code.md enforces this. TDD is rule #1. This is a confirmed strength.
- Triangulation 2: Every story commit runs `uv run pytest --tb=short`. Gates pass before merge.
- **Confidence: Very High** — RaiSE is ahead here.
- Maps to: G12 (strength)

---

**E12.2 — Small-scale testing (20 queries) outperforms waiting for large evals**
- Source: Art.6 ("small-scale testing right away with a few examples outperforms waiting for large evals")
- Anthropic says: Early stage changes have "dramatic impacts" making small samples sufficient. Start evaluating immediately.
- RaiSE reality: TDD (RED-GREEN-REFACTOR) embodies this. We write tests before code. But this specific recommendation is about *agent behavior* testing, not unit tests.
- Triangulation 2: We have no systematic agent behavior tests — no "20 canonical queries" that verify skill output quality.
- Triangulation 3: The gap is meta-evaluation (evaluating agent/skill performance), not code unit tests.
- **Confidence: High**
- Maps to: G12 (gap: no agent behavior eval corpus)

---

**E12.3 — Human testers find edge cases that evals miss**
- Source: Art.6 (human testers discovered agents choosing SEO farms over authoritative sources)
- Anthropic says: "Manual testing remains essential despite automation."
- RaiSE reality: HITL Default (rule #6) — we always pause for human review. This is a confirmed RaiSE strength.
- **Confidence: Very High** — RaiSE's HITL principle is the same recommendation.
- Maps to: G12 (strength)

---

**E12.4 — Production tracing for systematic agent failure diagnosis**
- Source: Art.6 (full production tracing in console simulations)
- Anthropic says: "Monitor agent decision patterns and interaction structures without accessing conversation content." Step-by-step watching revealed verbose queries, tool selection errors, unnecessary continuation.
- RaiSE reality: Telemetry signals exist (`rai signal emit-work`) but no step-level tracing of agent decisions. No dashboard.
- Triangulation 2: The blueprint identifies G-OBS-1 (no agent performance metrics) and G-OBS-2 (no dashboards). Confirmed gap.
- **Confidence: High**
- Maps to: G12 (gap: no agent decision tracing), G-OBS-1

### G14: Planner Ambition Calibration

---

**E14.1 — Two failure modes: over-ambition and under-ambition**
- Source: Art.2 (observed in building claude.ai clone)
- Anthropic says: Over-ambition: agent tries entire app in one session, runs out of context, leaves features undocumented. Under-ambition: later agents see partial progress and declare done.
- RaiSE reality: `/rai-story-plan` decomposes into tasks but has no explicit scope fence. Over-ambition is checked by HITL and story scope doc. Under-ambition is partially addressed by scope.md's "Done When" criteria.
- Triangulation 2 (RaiSE operational data): During E616 (License MVP), story scopes expanded mid-implementation. The scope doc existed but wasn't enforced as a fence during planning.
- Triangulation 3: The problem is at plan creation time — verifying that T1...TN cover scope exactly, neither more nor less.
- **Confidence: High**
- Maps to: G14

---

**E14.2 — Comprehensive feature list upfront prevents scope drift**
- Source: Art.2 (200+ feature list in JSON, explicitly structured with steps and pass/fail)
- Anthropic says: JSON preferred over Markdown because "model is less likely to inappropriately change or overwrite JSON files."
- RaiSE reality: Story scope.md documents "In Scope / Out of Scope / Done When." Design doc has AC in Gherkin. These are the right artifacts. The gap is connecting them to plan validation.
- Triangulation 2: Our scope.md + story.md together approximate the "feature list." The issue is the agent doesn't re-verify scope coverage during planning.
- **Confidence: High**
- Maps to: G14

---

**E14.3 — Lead agent uses extended thinking to calibrate subagent count and scope**
- Source: Art.6 (lead agent "uses thinking to plan its approach, assess which tools fit, determine subagent count")
- Anthropic says: Explicit resource allocation rules prevent spawning 50 agents for a simple query.
- RaiSE reality: `/rai-story-plan` decomposes tasks but doesn't explicitly size the scope or validate it against the story definition.
- Triangulation 2: Our plan template has a "Risks" section but no scope verification gate. The task count is not validated against AC coverage.
- **Confidence: High**
- Maps to: G14

## Section 3: G10 Effort Scaling Tension Resolution

### Position A: Full Skill Cycle Always (current RaiSE rule #3)

**Strongest arguments:**
1. Consistency builds discipline. "Small story" is a judgment call that accumulates into skipped steps over time. Threshold creep is real.
2. ShuHaRi is the adaptation mechanism — verbosity adapts, not process. Skills are already lightweight at Ha/Ri level.
3. PAT-E-592: "Dogfood stories need ~50% time buffer beyond task estimates." Stories we labeled "small" took longer. Under-estimation is a recurrent failure mode.
4. The overhead of a full skill cycle (start→design→plan→implement→review→close) for a small story is perhaps 15-20 minutes of ceremony. That's acceptable insurance against missed scope or regression.

**Evidence for A:**
- PAT-E-592: small stories take longer than estimated — premature lightness fails
- Rule works: we've shipped 616 stories with full cycles without the overhead becoming a blocker

---

**Position B: Scale Effort to Task Size (Anthropic)**

**Strongest arguments:**
1. Art.1, Art.4, Art.6 — three independent articles from the same source recommending "simplest thing that works." Convergent evidence.
2. RaiSE's own rule #11: "Simple First — Simple heuristics over complex solutions." We already value simplicity.
3. ShuHaRi mastery explicitly exists to adapt process to context. Ha/Ri levels allow compression. The rule as written prevents their use.
4. P7 (Lean Software Development): "Eliminate waste" is an explicit RaiSE principle. A full 8-phase ceremony for a 1-line fix is waste by our own standards.
5. Art.6 quantified the failure of not scaling: "spawning 50 subagents for simple queries" — they had to add explicit scaling rules to fix it.

**Evidence for B:**
- Three Anthropic articles converge on this recommendation
- Our own Lean principle (P7) supports it
- ShuHaRi was designed for this but is blocked by rule #3

---

### Testing Against RaiSE Principles

**P1 (Humans Define, Machines Execute):** Not affected by either position. Both preserve human oversight.

**P5 (Heutagogy — Learn to Learn):** Supports B. "Teach, don't just deliver." A learner who understands *when* to apply full ceremony is more capable than one who applies it blindly. Rule #3 as currently written prevents graduating from Shu to Ha/Ri on process judgment.

**P7 (Lean):** Supports B. "Eliminate waste, amplify learning, decide late." Fixed ceremony regardless of task size violates "eliminate waste." However, P7 also says "Deliver fast (validation gates, not batch)" — which means *gates* must never be skipped, only *ceremony*.

**The critical distinction:** Rule #3 conflates two things:
- **Ceremony** (design doc, planning doc, retrospective) — CAN scale to task size
- **Gates** (tests pass, linting, type checks, human review) — must NEVER be skipped

---

### Verdict for G10

**Decision: Refine the principle. Not reject, not keep as-is.**

The current rule #3 is correct in spirit (protect gates) but over-specified (mandates full ceremony). The refinement:

**Before:** "Full Skill Cycle — Use skills even for small stories (Structure helps; overhead is minimal)"

**After:** "Gates Always, Ceremony Scales — All quality gates are non-negotiable regardless of story size. Ceremony (design doc, plan doc, retrospective) scales with story size: M+ stories use full cycle; S/XS stories use adapted cycle at Ha level minimum."

**Why this is not an ADR:** This is a clarification, not a reversal. The underlying intent (protect quality, use skills) is preserved. ShuHaRi mastery already encodes this — we're making the policy explicit. An ADR would be warranted if we were removing a quality gate. We are not.

**What changes:**
1. `CLAUDE.md` critical rules section — update rule #3 wording
2. `methodology.yaml` — annotate principle with size guidance
3. Add note to `/rai-story-start` skill: "For XS/S stories, Ha-level ceremony is acceptable. Gates are non-negotiable."

**What doesn't change:** All 9 quality gates (tests, lint, types, format, coverage, AC coverage, guardrail compliance, human review, commit discipline) remain mandatory for every story of any size.

## Section 4: Verdicts

### G4 — Context Management Strategy

**Verdict: ADAPT**

**Rationale:** RaiSE has the right mechanisms (journal, MVC graph, session bundles) but no explicit *policy* connecting them to context lifecycle events. Anthropic's recommendation is to formalize compaction as a deliberate lifecycle event with recall-first strategy.

**What to adopt:**
- Explicit compaction trigger (skill-level: when to compact, how, what to preserve)
- Token budget specification per skill phase
- Sub-agent return size budget (explicit: aim for 300-500 tokens)
- `rai-progress.md` pattern — persistent state file for long stories (equivalent to Anthropic's claude-progress.txt)

**What not to adopt:**
- Tool result clearing as a platform feature (Claude Code already has this; we don't control it)
- Reinventing sub-agent architecture — ours already works

**RaiSE adaptation:** The journal + MVC graph + session bundles are the RaiSE implementation of structured note-taking + just-in-time retrieval. The adaptation is formalizing a context policy that invokes these mechanisms explicitly, not building new ones.

**Cross-reference to RAISE-783:** G4 findings directly inform Session Management. The "resume vs start" distinction in RAISE-783 is a compaction/reset policy problem in disguise — on resume, context should be loaded differently than on fresh start.

**Verdict: ADAPT (refine principle)**

See Section 3 above. Summary: gates non-negotiable, ceremony scales with story size.

**Implementation:** CLAUDE.md and methodology.yaml update. No new story needed — a commit directly on release/2.4.0 after human approval.

### G12 — Agent Test Design

**Verdict: ADOPT partially — quick wins now, full vision later**

**Quick wins (do now, XS effort):**
- Add "agent-friendly test naming" guidance to `code-standards.md` — tests named after behavior, not implementation, are more interpretable by agents
- Add 10-20 canonical regression scenarios for skill outputs (not unit tests — agent behavior specs)

**Defer:**
- Context-budget-aware tests (requires token counting infrastructure)
- Full production tracing dashboard (G-OBS-1, G-OBS-2 — enterprise roadmap)

**What RaiSE already has that Anthropic recommends:**
- HITL (human testers catch what evals miss) ✓
- TDD with automated gates ✓
- Structured testing with fixtures ✓

**Gap remaining:** No corpus of agent behavior specs ("given skill X and story Y, agent should produce output with properties Z"). This is meta-testing, not unit testing.

---

### G14 — Planner Ambition Calibration

**Verdict: ADOPT**

**Rationale:** The problem is documented, the solution is simple, the cost is minimal, and we have direct operational evidence (E616 scope drift). The solution: add a scope fence step to `/rai-story-plan`.

**Scope fence mechanism:**
1. Before finalizing the plan, explicitly re-read `s{N}.{M}-story.md` AC
2. Verify: does the task list cover ALL AC? (under-ambition check)
3. Verify: does the task list introduce work NOT in AC? (over-ambition check)
4. Document verdict: "Scope fence: N tasks cover M AC scenarios. No out-of-scope tasks."

**Inspired by Anthropic but RaiSE-native:** Their feature list JSON + coding agent restriction is their implementation. Ours is simpler — the scope doc already exists, we just need the verification step.

**Honest assessment of Anthropic's evidence quality:** Art.6 is the most empirically grounded (90.2% performance improvement, 40% time reduction, token cost data). Art.1, Art.2, Art.4 are primarily assertion-based with illustrative examples. Claims from these articles should be treated as strong hypotheses worth testing against RaiSE's specific context, not universal laws.

**One Reject:** G10 as originally framed ("adopt Anthropic's effort scaling pattern wholesale") is rejected. The refinement keeps RaiSE's core discipline (gates always) while adopting the spirit (don't waste ceremony on XS work). This is the only finding where we explicitly diverged from Anthropic's framing.
