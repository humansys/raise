### T3: Score Benchmark Matrix — 5 Gaps × 5 Dimensions (S)

**Description:** Score each of the 5 gaps on the 4 standard evaluation dimensions plus the OSS/Enterprise 5th dimension. Calculate priority using formula. Apply the G2 simplicity test explicitly.

**Dimensions:**
| Dimension | Scale | Question |
|-----------|-------|----------|
| Value to Core | 0-3 | Strengthens memory, methodology, or entity? |
| Simplicity Cost | 0-3 | Complexity added? (lower = better) |
| Observability Gain | 0-3 | More traceable/auditable? |
| Adoption Effort | XS-XL | Story points to implement |
| OSS/Enterprise | OSS/Ent/Both | Where does this belong? |

**Priority formula:** (Value to Core + Observability Gain) - Simplicity Cost
- 4-6: Do now (2.4.0 OSS)
- 1-3: Plan (enterprise or future OSS)
- 0 or negative: Reject or defer

**G2 Simplicity Test (D2):** For scoring G2, explicitly answer: "Would a developer apply this rubric mid-session, or skip it?" If "skip it" → Simplicity Cost = 3, Enterprise classification. If "use it" → score normally for OSS.

**TDD analog:**
- RED = empty matrix
- GREEN = all 20 cells (5×4) scored with rationale + OSS/Enterprise column
- REFACTOR = validate scores against evidence catalog (no score without evidence)

**Verification:**
- All 5 gaps scored on all 4+1 dimensions
- Each score has 1-sentence rationale
- G2 simplicity test result documented explicitly
- Scores consistent with evidence catalog confidence levels

**Dependencies:** T2.
**Size:** S (~20 min)

### T4: P1 Deliberation — Reformulation + ADR Draft (M)

**Description:** Execute the 4-step deliberation for G13 (contract negotiation) using D4 as framing: P1 may need reformulation based on 500+ operational sessions. The output is both a G13 verdict AND an ADR draft.

**Deliberation steps:**
1. State P1 as written and its original intent (what failure it was designed to prevent)
2. Gather operational evidence: what actually happens in 500+ sessions — is execution negotiation already agent-driven?
3. Test the proposed reformulation: "Humans Define Goals and Boundaries — Machines Negotiate Execution within them"
   - Does it still prevent the original failure modes?
   - Does it enable G13 patterns that were previously blocked?
   - Is it more honest about RaiSE's actual architecture?
4. Decide: keep P1 as-is / refine / reformulate. If reformulate: draft ADR.

**ADR structure:**
```
# ADR-NNN: Reformulate P1 — Humans Define Goals and Boundaries

## Status: Proposed
## Context: [operational evidence]
## Decision: [new P1 wording]
## Before/After: [table]
## Consequences: [what changes, what stays]
## Alternatives considered: [keep as-is, partial change]
```

**G3 HITL-compatible loop (AC6):** In this task, also define the OSS-compatible iterative eval pattern:
- Human defines AC
- Agent runs eval round
- Human reviews round verdict
- Agent iterates if human approves
vs. fully automated loop (Enterprise only).

**TDD analog:**
- RED = P1 tension unresolved, G13 verdict not possible
- GREEN = explicit deliberation with evidence, ADR drafted
- REFACTOR = ensure ADR is internally consistent with other architectural decisions

**Verification:**
- Both positions on P1 stated with strongest arguments
- Proposed reformulation tested against G2, G3, G13 (does it enable the right things?)
- ADR draft written with before/after table
- G3 HITL-compatible loop described concretely (not just "with human review")
- G13 verdict follows from P1 deliberation outcome

**Dependencies:** T2 (evidence), T3 (scoring context). Riskiest task — scheduled after context-building.
**Size:** M (~25 min)
