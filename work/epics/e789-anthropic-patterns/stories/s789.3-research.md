## 2. Benchmark Matrix

### Scored Gaps (G6, G7, G11)

| Gap | Value to Core | Simplicity Cost | Observability Gain | Priority | Effort | OSS/Enterprise |
|-----|:---:|:---:|:---:|:---:|:---:|:---:|
| **G6: Tool descriptions** | 2 | 1 | 1 | **2** | S | OSS Core |
| **G7: Progressive disclosure** | 3 | 1 | 2 | **4** | M | OSS Core |
| **G11: Agent self-improvement** | 2 | 2 | 2 | **2** | M | Both |

### G6: Tool Descriptions — Scoring Rationale

- **Value to Core (2):** Improves daily agent reliability. Better tool descriptions = fewer misuses = less wasted context. But CLAUDE.md reference already works reasonably well — this is optimization, not missing capability.
- **Simplicity Cost (1):** Low — improving help text and parameter names is incremental. No new abstractions needed. Main cost: maintaining dual-audience descriptions (human CLI + agent consumption).
- **Observability Gain (1):** Marginal — better descriptions don't directly improve traceability. Indirectly reduces debugging time when agent misuses tools.
- **Effort (S):** Audit 20+ command groups, improve descriptions, add examples. Mostly text changes.

### G7: Progressive Disclosure — Scoring Rationale

- **Value to Core (3):** High — directly addresses the fundamental constraint (context window). RaiSE already has 19 skills, 20+ CLI commands, MCP tools. Loading everything always wastes context. Skills declaring their tool dependencies would make each phase more focused. Blueprint already identifies this gap (§3.1).
- **Simplicity Cost (1):** Low — the mechanism already exists. Claude Code has skills, `.claude/rules/` with path scoping, and ToolSearch (deferred tools). RaiSE just needs to leverage them. No new infrastructure needed.
- **Observability Gain (2):** Moderate — knowing which tools were available vs used per skill phase improves auditability. "Agent had 5 relevant tools, used 3" is better than "agent had 40 tools, used 3."
- **Effort (M):** Each of 19 skills needs tool dependency declaration. Requires design decision on format (frontmatter? separate manifest?). Then integration with session/skill loading.

### G11: Agent Self-Improvement — Scoring Rationale

- **Value to Core (2):** Aligns with P6 (Kaizen) and P5 (Heutagogy). Agent improving its own tool descriptions is self-learning in action. But the value is incremental — tool descriptions don't change often.
- **Simplicity Cost (2):** Moderate — needs governance guardrails. Agent changes to tool descriptions must be committed (P2), reviewed (HITL), and bounded (can't change tool behavior, only descriptions). More process than the others.
- **Observability Gain (2):** Good — every improvement is a git commit. Audit trail of what changed, when, and why. Auto memory already provides a pattern for this.
- **Effort (M):** Design the improvement loop (when to trigger, what to change, how to review). Integrate with existing pattern system. Define boundaries.

### G8: Code-Mode MCP — Principle Extraction (Not Scored)

**Underlying principle:** Shift intermediate data processing from model context to code execution environment. Agent writes code to transform/filter results locally; model only sees synthesized output.

**Relevance to RaiSE:** The principle is language-agnostic, but the implementation requires a sandboxed code execution environment — infrastructure RaiSE doesn't have and doesn't need for its current use case (CLI framework, not data pipeline). The 98.7% token savings are compelling for data-heavy MCP use cases (spreadsheets, databases) but RaiSE's MCP usage is mostly structured queries with bounded output.

**Verdict preview:** Not Applicable for OSS. Principle noted for future reference if RaiSE supports data-heavy MCP integrations (Enterprise).

## 3. P1 Deliberation — G11 as ADR-034 Test Case

### The Question

Under ADR-034's reformulated P1 ("Humans Define Goals and Boundaries — Machines Negotiate Execution within them"), is agent self-improvement of tool descriptions acceptable?

### Analysis Under P1 Original

"Humans Define, Machines Execute" → Tool descriptions define the interface between human intent and machine execution. An agent modifying tool descriptions is changing its own execution interface = **redefining**, not executing. **Violation.**

### Analysis Under P1 Reformulated

The activity classification from ADR-034:

| Activity | Classification | Test |
|----------|---------------|------|
| Deciding WHAT tools exist | Human-owned | Changing goals/boundaries |
| Deciding tool behavior/implementation | Human-owned | Changing execution semantics |
| Improving tool descriptions for clarity | Machine-negotiable | Execution optimization within boundaries |
| Suggesting new parameters/defaults | Shared | Proposing boundary changes (human approves) |

**Key distinction:** Improving a description ("clarify that `user` means `user_id`") ≠ changing behavior ("now `user` accepts email too"). The first is execution optimization. The second is boundary change.

### Failure Mode Testing

1. **Agent makes description worse** → Git diff shows the change. HITL review catches it. Auto-revert possible.
2. **Agent changes behavior through description** → Description-only changes can't alter CLI implementation. Separation of concerns protects.
3. **Agent accumulates drift over time** → Pattern system (PAT-E-*) tracks changes. Periodic human audit of tool descriptions (same as CLAUDE.md review).

### Conditions for Acceptability

Under P1 reformulated, agent self-improvement of tool descriptions is acceptable IF:

1. **Changes are description-only** — no modification of tool behavior, parameters, or return types
2. **Changes are committed to git** — P2 (Governance as Code) satisfied
3. **Changes are human-reviewable** — HITL gate before merge
4. **Trigger is bounded** — improvement happens at review time (like retrospective), not continuously mid-task
5. **Auto memory provides the pattern** — Claude Code already does this. Agent notes "this tool description confused me" → human reviews → description updated

### Verdict

G11 is **compatible with P1 reformulated** under these 5 conditions. This provides concrete supporting evidence for ADR-034 ratification: the reformulation correctly distinguishes execution optimization from goal/boundary redefinition.

## 4. Verdicts

### G6: Tool Descriptions — **Adapt** (OSS Core)

**Priority score: 2** (Plan for 2.4.x)

**What to adopt:**
- Unambiguous parameter naming (E6.2) — audit and fix across CLI
- Actionable error responses (E6.7) — replace tracebacks with guidance
- Tool descriptions as "new hire onboarding" (E6.1) — rewrite help text with this lens
- Sensible defaults and pagination (E6.4) — already partially done

**What to adapt:**
- Response format control (E6.3): RaiSE already has `--format human|json|compact` on many commands. Extend to all commands, add `--format agent` option that strips visual formatting and includes only machine-relevant fields.
- Namespacing (E6.5): `rai` CLI already namespaces by command group (`rai session`, `rai pattern`, `rai backlog`). No change needed — our structure matches the recommendation.

**What to reject:**
- Nothing. All recommendations are compatible and incremental.

**Simplicity test (PAT-E-010):** Would a developer improve tool descriptions mid-session? Yes — it's a quick text fix. → **OSS Core**.

**Implementation story:** RAISE-821 — Audit and improve CLI tool descriptions for agent comprehension (S)

### G7: Progressive Disclosure — **Adopt** (OSS Core)

**Priority score: 4** (Do now — release 2.4.0)

**What to adopt:**
- Skills declare tool dependencies — each skill specifies which CLI commands and MCP tools it needs (E7.1)
- Leverage `.claude/rules/` path scoping for context-relevant instructions (E7.2)
- CLAUDE.md size discipline — keep under 200 lines, move details to skills (E7.3)
- Subagent pattern for investigation to protect main context (E7.5)

**What to adapt:**
- Skill frontmatter gets a `tools:` field listing relevant CLI commands. Session loader only surfaces these tools in context during that skill phase.
- CLAUDE.md audit: move command reference details to individual skill files. Main CLAUDE.md keeps only the essential reference table.

**What to reject:**
- Code-mode MCP as progressive disclosure mechanism (E7.7) — the token savings are real but the infrastructure cost is too high for RaiSE's current use case.

**Simplicity test:** Would a developer configure tool dependencies per skill? No — this is framework architecture. But would they benefit from leaner context per skill? Absolutely. The benefit is automatic once configured. → **OSS Core** (framework-level change, user-transparent benefit).

**Implementation story:** RAISE-822 — Skill tool dependency declarations + CLAUDE.md audit (M)

### G8: Code-Mode MCP — **Not Applicable** (Principle Noted)

**No priority score** — outside RaiSE's current architecture.

The underlying principle (shift data processing to code execution) is sound for data-heavy integrations. RaiSE's MCP usage is structured queries with bounded output — the overhead of sandboxed code execution doesn't justify the token savings for our use case.

**Principle captured:** When RaiSE supports data-heavy MCP integrations (e.g., analytics, large dataset queries), code-mode should be evaluated. File under Enterprise roadmap considerations.

**No implementation story.** Note in Enterprise Design Implications document.

### G11: Agent Self-Improvement — **Adapt** (Both OSS + Enterprise)

**Priority score: 2** (Plan for 2.4.x)

**What to adopt:**
- Auto memory pattern: agent notes tool description issues → human reviews at retrospective → descriptions updated (E11.1, E11.3)
- Evaluation transcript analysis: after story review, agent can suggest tool description improvements based on session friction (E11.2)

**What to adapt:**
- Scope to retrospective phase only — not continuous mid-task. Improvement happens during `/rai-story-review`, not during `/rai-story-implement`
- Changes must be committed as separate PRs (P2), not mixed with feature commits
- 5 conditions from P1 deliberation (§3) as guardrails

**What to reject:**
- Fully autonomous tool modification without human review — even under P1 reformulated, HITL gate is non-negotiable for tool interface changes
- Continuous improvement loop — too much overhead for marginal gain. Batch at retrospective boundaries

**Simplicity test:** Would a developer trigger tool improvement suggestions mid-session? No — retrospective-scoped. But would they review suggested improvements at story close? Likely yes, if well-formatted. → **Both** (OSS: manual review of suggestions; Enterprise: automated suggestion pipeline).

**Implementation story:** RAISE-823 — Tool description improvement loop at retrospective (S)
