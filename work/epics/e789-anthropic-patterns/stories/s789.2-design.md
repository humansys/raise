## What & Why

**Problem:** RaiSE's quality gate system is binary (pass/fail) and single-pass. Anthropic recommends scoring rubrics, iterative eval loops, and feature verification checklists — patterns that could strengthen AR/QR skills but may introduce complexity or conflict with HITL principles.

**Value:** A principled evaluation of these 5 gaps will clarify which evaluation improvements belong in OSS (improving the open source core), which belong in the enterprise edition, and which to reject as incompatible with RaiSE's P1 (Humans Define) principle. This directly informs RAISE-806 story creation.

## Research Approach

Same 4-phase methodology as S789.1, adapted for evaluation domain:

### Phase 1: Deep Read (evaluation-focused)
Read Art.2, Art.3, Art.6 targeting evaluation-relevant sections:
- Art.3: Evaluator-generator pattern, scoring rubrics, grading criteria
- Art.2: Feature verification checklist, test oracle, harness checkpointing
- Art.6: Small-scale testing, human testers, production tracing

**Reading order:** Art.3 first (most evaluation-dense), then Art.2 (harness/oracle patterns), then Art.6 (empirical eval data).

### Phase 2: Evidence Catalog
Triangulate each major claim:
1. Anthropic article (source)
2. RaiSE operational reality (current gates, AR/QR skill behavior)
3. First principles from RaiSE architecture (P1, P4, P5, P8)

### Phase 3: Benchmark Matrix (5 gaps × 4 dimensions)
Score each gap with **5th dimension: OSS vs Enterprise** (PAT-E-008):
- OSS Core: Improves daily practice for any developer
- Enterprise: Requires org-level infra, automated agents, or scale

### Phase 4: Verdicts + Backlog Impact
- Adopt / Adapt / Reject per gap with rationale
- OSS/Enterprise classification per verdict
- Stories for RAISE-806 (OSS) and E9 (Enterprise)

## Design Decisions

### D1: G13 Evaluation Framework — P1 as Hard Constraint

**Question:** Does evaluator↔generator contract negotiation violate P1 (Humans Define, Machines Execute)?

**Approach:** Evaluate G13 using P1 as a hard constraint, not a soft preference. The question is not "is this useful?" but "is it compatible with our architectural principle that humans define what gets built?"

**Positions to evaluate:**
- A: Reject — agent negotiation on *scope* violates P1. Scope is a human decision.
- B: Nuanced accept — agents can negotiate *implementation approach* within human-defined scope.
- C: Enterprise-only — agent negotiation acceptable when humans define the scope envelope.

**Expected outcome:** Some form of Reject or Enterprise-only — not full OSS adoption.

---

### D2: G2 Scoring Rubrics — Simplicity Gate

**Question:** Can we add calibrated grading to AR/QR skills without undermining their simplicity?

**Approach:** Apply the same simplicity test used in S789.1 — what is the minimum viable rubric that improves quality without adding ceremony? A 3-point scale per dimension (0-3) may be sufficient vs. complex LLM-as-judge.

**Simplicity test:** "Would a developer add this rubric mid-session, or would they skip it because it's too heavy?" If the answer is "skip it," it belongs in enterprise automated eval, not OSS manual gates.

---

### D3: G3 Iterative Eval — HITL Compatibility Test

**Question:** Is a "run N rounds until threshold met" loop compatible with RaiSE's HITL Default (rule #6)?

**Key distinction to probe:**
- Fully automated loops (no human in the loop): likely conflicts with HITL Default
- Human-gated iterations (human reviews each round verdict): compatible
- "Auto-fix, then human reviews" hybrid: probably the right OSS pattern

**Expected outcome:** Adapt — iterative loops are fine if each round requires human approval before proceeding.
