**E1.1 — Browser automation as end-to-end verification**

- Triangulation 2 (RaiSE operational): RaiSE produces no UI artifacts. All outputs are files, CLI output, and JIRA/Confluence updates. The "interactive" evaluator for RaiSE would verify artifact quality, not visual rendering.
- Triangulation 3 (first principles — P3 Platform Agnosticism): Playwright/Puppeteer ties evaluation to browser environments. RaiSE's gate system is platform-agnostic by design. Adding browser-specific evaluation violates P3.
- **Confidence: High** — the pattern is real but misaligned with RaiSE's domain.
- Maps to: G1
- OSS/Enterprise: **N/A for OSS** (no UI). Enterprise teams building web products could use this, but it's not a RaiSE framework feature — it's a project-level testing choice.

- Triangulation 2: Our gate-code.md already includes `uv run pytest --tb=short` which validates the system loads and passes. This is functionally equivalent.
- **Confidence: Medium** — the pattern exists, but RaiSE already covers this with test gates.
- Maps to: G1 (partial strength)
- OSS/Enterprise: N/A — already covered.

### G2: Calibrated Grading

---

**E2.1 — Feature list as structured evaluation rubric (binary, not scored)**
- Source: Art.2
- Anthropic says: JSON-based feature specification with binary pass/fail per feature. "We prompt coding agents to edit this file only by changing the status of a passes field." 200+ features in the claude.ai clone, each with steps and pass/fail status.
- RaiSE reality: Story AC in Gherkin format. Binary (scenario passes or not). Not connected to automated verification — human reviews AC coverage manually.
- Triangulation 2 (RaiSE operational): Our AC are more expressive than JSON feature lists (Gherkin supports Given/When/Then with context), but they're not used as automated test oracles. The gap is the automation connection, not the format.
- Triangulation 3 (first principles — P4 Validation Gates): Gates are binary pass/fail today. A feature list with structured steps is essentially what our Gherkin AC already are — but not connected to automated checking.
- **Confidence: High** — Anthropic's rubric is actually *less* expressive than our AC. The gap is automation, not scoring.
- Maps to: G2
- OSS/Enterprise: **OSS Core** — connecting AC to automated verification is a daily workflow improvement.

---

**E2.2 — Implicit quality criteria beyond binary pass/fail**
- Source: Art.2
- Anthropic says: Beyond feature pass/fail, quality criteria include "no major bugs," "code is orderly and well-documented," "a developer could easily begin work on a new feature." These are qualitative, not scored.
- RaiSE reality: Our AR/QR skills evaluate similar qualities (architecture proportionality, code quality, test coverage) but through human judgment, not rubrics.
- Triangulation 2 (RaiSE operational): The AR/QR skills produce narrative assessments. There's no structured rubric (e.g., "orderliness: 1-3, documentation: 1-3"). Adding one would make reviews more consistent and comparable across stories.
- Triangulation 3 (D2 Simplicity Test): Would a developer apply a structured rubric mid-session? **Answer: depends on weight.** A 3-point scale per dimension (0-2: poor/adequate/good) on 4 dimensions = 4 numbers to fill in. That's fast enough. A 10-dimension rubric with calibration examples = too heavy for OSS. **Simplicity test result: 3-4 dimensions × 3-point scale = OSS viable. More than that = Enterprise.**
- **Confidence: High** — Anthropic's quality criteria map to what AR/QR already evaluate, but without structure.
- Maps to: G2
- OSS/Enterprise: **OSS Core** (lightweight rubric), **Enterprise** (full LLM-as-judge scoring)

---

**E2.3 — Evaluator-optimizer workflow: LLM evaluates LLM output in a loop**
- Source: Art.1
- Anthropic says: "One LLM call generates a response while another provides evaluation and feedback in a loop." Works when there are "clear evaluation criteria" and "iterative refinement provides measurable value." Analogous to "the iterative writing process a human writer might go through."
- RaiSE reality: No LLM-as-judge pattern in the framework. AR/QR skills are human-driven reviews, not automated eval loops.
- Triangulation 2: The concept maps to using Claude to review its own output — which we do informally (asking Rai to self-check). But there's no structured evaluator pass with criteria.
- Triangulation 3 (P1 reformulation context — D4): Under original P1 ("Humans Define, Machines Execute"), LLM-as-judge is problematic — the machine is both executing and evaluating. Under the proposed reformulation ("Humans Define Goals and Boundaries, Machines Negotiate Execution"), LLM-as-judge within human-defined criteria is execution-level evaluation, which is compatible.
- **Confidence: High** — The pattern is well-described. Its P1 compatibility depends on the D4 outcome.
- Maps to: G2, G3
- OSS/Enterprise: **Enterprise** for automated loops; **OSS** if human approves each round.

### G3: Iterative Evaluation Loop

---

**E3.1 — Multi-context-window iteration as evaluation loop**
- Source: Art.2
- Anthropic says: Each coding agent follows a cycle: "Read progress files and git logs → Run baseline test → Select one incomplete feature → Implement and test → Commit with progress update." Multiple context windows, each doing one feature. Stopping condition: all features marked "passes: true."
- RaiSE reality: Our story lifecycle is one pass: plan → implement → review. No "run N rounds until all AC pass" loop. The closest analog is TDD (RED→GREEN→REFACTOR) which iterates within a single task, not across the whole story.
- Triangulation 2 (RaiSE operational): After `/rai-story-review`, if issues are found, the human decides to fix them. There's no automated "re-evaluate and iterate" loop. The review skill doesn't re-run gates — it produces a retrospective.
- Triangulation 3 (P5 Kaizen): Iterative improvement is a core principle. But today it's inter-session (patterns compound across sessions), not intra-story (multiple eval rounds within one story).
- **Confidence: High** — The gap is real. RaiSE iterates across sessions, not within stories.
- Maps to: G3
- OSS/Enterprise: **Both** — the pattern is valuable at any scale. The question is HITL compatibility.

---

**E3.2 — Agent self-correction via environmental feedback**
- Source: Art.1
- Anthropic says: Agents gain "ground truth from the environment at each step (such as tool call results or code execution) to assess its progress." Dynamic course correction, not static evaluation.
- RaiSE reality: TDD gives exactly this — test failures are environmental feedback that the agent uses to course-correct. `uv run pytest --tb=short` output is the "ground truth."
- Triangulation 2: This is how we already work in practice. The agent runs tests, reads failures, fixes code, runs tests again. It's iterative evaluation at the task level.
- **Confidence: Very High** — RaiSE already does this via TDD. Confirmed strength.
- Maps to: G3 (strength, not gap)
- OSS/Enterprise: Already implemented.

---

**E3.3 — HITL compatibility of iterative loops**
- Source: RaiSE principles analysis (no direct Anthropic source — derived from D3 design decision)
- Anthropic says: Their loops are fully automated (no human in the loop per round). The stopping condition is objective: all features pass.
- RaiSE reality: HITL Default (rule #6) says "pause after significant work for human review." A fully automated multi-round eval loop bypasses this.
- Triangulation 2 (D3 from design): Three variants exist:
  1. **Fully automated** (Anthropic's approach): agent iterates until all criteria pass. No human review per round. Conflicts with HITL Default.
  2. **Human-gated per round**: agent runs one eval round, presents results, human approves next round. HITL-compatible but slow.
  3. **Auto-fix mechanical, human-gate judgmental**: auto-iterate on objective criteria (tests pass, lint clean, types check). Pause for human only when subjective judgment needed (architecture, scope, quality). **This is the RaiSE-compatible hybrid.**
- Triangulation 3 (P7 Lean): "Eliminate waste" — human-gating every automated check is waste. But human oversight on judgment calls is not waste, it's quality.
- **Confidence: High** — The hybrid (variant 3) is the principled OSS position.
- Maps to: G3
- OSS/Enterprise: **OSS = variant 3 (hybrid)**. Enterprise = variant 1 (fully automated for mature teams with audit trail).

### G5: Feature Verification Checklist

---

**E5.1 — JSON feature list as test oracle**
- Source: Art.2
- Anthropic says: 200+ features in structured JSON. Each feature has category, description, steps, and binary pass/fail. "Claude was asked to write a JSON description of each feature, describing the steps a human tester would follow to verify it." Agents can only change the `passes` field — cannot modify criteria.
- RaiSE reality: Story AC in Gherkin exist but aren't structured as machine-readable test oracles. The AC live in `s{N}.{M}-story.md` as markdown, not as a checkable data structure.
- Triangulation 2 (RaiSE operational): During `/rai-story-review`, the reviewer manually checks AC coverage. There's no automated "did all AC pass?" verification. The gap is making AC machine-readable and checkable.
- Triangulation 3 (P2 Governance as Code): AC as checkable data structures aligns with "governance as code" — the acceptance criteria become verifiable artifacts, not just documentation.
- **Confidence: High** — The pattern is directly applicable. Our AC have the right content but wrong format for automation.
- Maps to: G5
- OSS/Enterprise: **OSS Core** — converting AC to a checkable format is a daily workflow improvement.

---

**E5.2 — Immutable criteria prevent agents from changing the definition of done**
- Source: Art.2
- Anthropic says: Agents can only toggle `passes` to true, never modify the criteria themselves. "We prompt coding agents to edit this file only by changing the status of a passes field."
- RaiSE reality: Our scope.md "Done When" criteria are editable by anyone. No immutability guarantee. In practice, the agent doesn't modify them — but there's no enforcement.
- Triangulation 2: Scope creep (E616 documented in S789.1) happens precisely when done criteria drift. An immutable feature list would have prevented it.
- Triangulation 3 (G14 from S789.1): The scope fence we proposed in RAISE-795 addresses this — verify tasks against AC without allowing modification. Complementary patterns.
- **Confidence: High** — Directly complements the scope fence from S789.1.
- Maps to: G5, G14
- OSS/Enterprise: **OSS Core** — immutable AC as verification target.

### G13: Contract Negotiation (Evaluator ↔ Generator)

---

**E13.1 — Initializer agent expands scope unilaterally, no negotiation**
- Source: Art.2
- Anthropic says: The initializer agent creates the feature list (200+ features) from a high-level prompt. "This meant over 200 features." There is no negotiation between evaluator and generator — scope is defined upfront by a dedicated agent, then handed to coding agents as immutable criteria.
- RaiSE reality: `/rai-story-plan` decomposes tasks from human-defined scope. The human defines AC; the agent decomposes into tasks. No agent-to-agent scope negotiation.
- Triangulation 2: What Anthropic calls "initialization" is actually what RaiSE calls `/rai-story-design` + `/rai-story-plan` — an agent expanding a human goal into structured, executable criteria. The difference: in Anthropic's model, a separate agent does this without human review. In RaiSE, the human reviews the plan (HITL).
- **Confidence: Very High** — Anthropic doesn't actually do evaluator↔generator negotiation either. They do unilateral expansion by an initializer agent.
- Maps to: G13
- OSS/Enterprise: See E13.2 for P1 analysis.

---

**E13.2 — Evaluator-optimizer as execution negotiation, not scope negotiation**
- Source: Art.1
- Anthropic says: The evaluator-optimizer pattern is described as "one LLM call generates a response while another provides evaluation and feedback in a loop." The evaluator doesn't negotiate *what* to build — it evaluates *how well* it was built. The scope was already defined.
- RaiSE reality: This maps to a post-implementation review, not pre-work negotiation. In RaiSE terms: the evaluator is `/rai-quality-review`, not `/rai-story-plan`.
- Triangulation 2 (D4 — P1 reformulation): Under the proposed P1 reformulation ("Humans Define Goals and Boundaries, Machines Negotiate Execution"), this pattern is clearly execution-level. The evaluator doesn't define goals — it verifies execution quality within human-defined criteria. P1-compatible.
- Triangulation 3 (operational evidence): In 500+ sessions, Rai already evaluates its own output quality when asked to review. The agent doesn't redefine scope — it assesses whether the output meets the scope. This is the pattern Anthropic describes.
- **Confidence: Very High** — G13 as originally framed ("contract negotiation") mischaracterized the pattern. It's execution quality evaluation, not scope negotiation.
- Maps to: G13
- OSS/Enterprise: **Both** — but the automation level differs (see G3).

---

**E13.3 — Lead agent calibrates resource allocation, not scope**
- Source: Art.6
- Anthropic says: Lead agent "uses thinking to plan its approach, assess which tools fit, determine subagent count." This is resource allocation within a defined task, not scope negotiation.
- RaiSE reality: `/rai-story-plan` decomposes tasks and assesses effort. Same pattern — but executed by the agent within human-defined boundaries.
- Triangulation 2: The "50 subagents for simple queries" failure was over-resourcing, not scope expansion. The fix was calibration rules, not removing agent autonomy.
- **Confidence: High** — Further evidence that "contract negotiation" is a misnomer. It's execution calibration.
- Maps to: G13
- OSS/Enterprise: Both — already how RaiSE works in practice.

## Section 3: P1 Reformulation Deliberation

### P1 as Written

> "Humans Define, Machines Execute" — Humans define what gets built; machines execute the definition.

**Original intent:** Prevent agents from autonomously deciding strategic direction, expanding scope beyond human intent, or making irreversible architectural decisions without human oversight.

### Operational Evidence (500+ Sessions)

What actually happens in RaiSE sessions:

| Activity | Who decides | P1 as written says | Reality |
|----------|-----------|-------------------|---------|
| Story scope (what to build) | Human | Human ✓ | Aligned |
| Epic strategy | Human | Human ✓ | Aligned |
| Architecture decisions | Human (with Rai input) | Human ✓ | Aligned |
| Task decomposition (/rai-story-plan) | Agent (human reviews) | Human? | Agent proposes, human approves |
| Execution approach (which files, what order) | Agent | Machine ✓ | Aligned |
| Self-evaluation (does this meet AC?) | Agent | ? | Not covered by P1 |
| Research methodology (what to read, how to triangulate) | Agent | ? | Not covered by P1 |
| Tool selection (which CLI commands) | Agent | Machine ✓ | Aligned |
| Quality judgment ("is this good enough?") | Agent proposes, human reviews | ? | Shared — P1 doesn't model this |

**Key observation:** The grey zone is "shared judgment" — activities where the agent proposes and the human reviews. P1 doesn't model this. It models a clean binary (human defines / machine executes) that doesn't match the collaborative reality.

### Testing the Proposed Reformulation

**Proposed:** "Humans Define Goals and Boundaries — Machines Negotiate Execution within them."

**Test 1: Does it still prevent the original failure modes?**
- Agent expanding scope beyond human intent? → Goals and boundaries are human-defined. Agent can't expand them. ✓ Protected.
- Agent making irreversible architecture decisions? → "Boundaries" includes architecture decisions. ✓ Protected.
- Agent autonomously deciding strategic direction? → "Goals" are human-defined. ✓ Protected.

**Test 2: Does it enable patterns that were previously blocked?**
- G13 evaluator↔generator quality negotiation? → Execution-level. ✓ Enabled.
- G3 automated eval loops within defined criteria? → Execution within boundaries. ✓ Enabled.
- G2 LLM-as-judge within human-defined rubric? → Execution assessment. ✓ Enabled.
- Agent unilaterally changing story scope? → Goal change. ✗ Still blocked.
- Agent removing quality gates? → Boundary change. ✗ Still blocked.

**Test 3: Is it more honest about RaiSE's actual architecture?**
- Task decomposition is agent-proposed, human-reviewed → "Machines negotiate execution" ✓
- Research methodology is agent-chosen within story scope → "Within boundaries" ✓
- Quality review is collaborative → "Negotiate" captures this better than "execute" ✓

### Verdict

**Decision: Reformulate P1. Propose as ADR.**

The current P1 is a useful simplification for explaining RaiSE to newcomers ("humans define, machines execute" is easy to understand). But as an architectural principle governing framework design decisions, it's inaccurate — it blocks patterns that are already operational and compatible with RaiSE's reliability promise.

The reformulation preserves ALL original protections (humans own goals, scope, and irreversible decisions) while honestly modeling the collaborative execution that makes RaiSE work in practice.

**What the ADR must clarify:**
1. Human-owned (non-negotiable): goals, scope boundaries, architectural decisions, principles, values
2. Machine-negotiable (within human boundaries): task decomposition, execution approach, quality assessment, tool selection, research methodology
3. Shared (agent proposes, human reviews): plans, designs, evaluations of significant work

## Section 4: Verdicts

### G1 — Interactive Evaluator

**Verdict: REJECT for OSS framework**

**Rationale:** RaiSE is CLI-first. Interactive visual evaluation (Playwright, Puppeteer) is a project-level concern for teams building web products, not a framework feature. Adding browser dependencies violates P3 (Platform Agnosticism). Score: -1.

**What stays:** Teams can add Playwright MCP to their project's `.raise/mcp/` config. The framework doesn't need to own this.

**OSS/Enterprise:** N/A — not a framework concern at either level.

---

### G2 — Calibrated Grading

**Verdict: ADAPT — lightweight rubric for OSS, LLM-as-judge for Enterprise**

**Rationale:** Anthropic's "rubric" is actually simpler than expected — implicit quality criteria, not formal scoring. RaiSE can add a 3-4 dimension structured rubric to AR/QR skills without heavy ceremony. The simplicity test (D2) passed: 4 dimensions × 3-point scale is fast enough for mid-session use.

**OSS implementation:**
Add to `/rai-architecture-review` and `/rai-quality-review`:
```
| Dimension | Score (0-2) | Notes |
|-----------|-------------|-------|
| Proportionality | _ | Is complexity justified? |
| Clarity | _ | Would a new team member understand? |
| Testability | _ | Are behaviors verifiable? |
| Consistency | _ | Does it follow established patterns? |
```
Score 0 = needs work, 1 = adequate, 2 = strong. Total 0-8, threshold ≥4 for pass.

**Enterprise extension:** LLM-as-judge automated scoring using the same rubric dimensions, with few-shot calibration examples per dimension. Human reviews only when score disagrees with automated assessment.

**OSS/Enterprise:** OSS = manual rubric in skills. Enterprise = automated scoring + human override.

---

### G3 — Iterative Evaluation Loop

**Verdict: ADAPT — hybrid variant for OSS, full automation for Enterprise**

**Rationale:** Fully automated eval loops conflict with HITL Default. But TDD already IS an iterative eval loop at the task level (E3.2 confirmed strength). The gap is at the *story* level — no "re-evaluate all AC after implementation, iterate if needed" loop.

**OSS implementation (hybrid variant 3):**
1. After `/rai-story-implement` completes, run automated gates (tests, lint, types, format)
2. If automated gates fail → agent auto-fixes without human intervention (mechanical)
3. After automated gates pass → run AR/QR review (human reviews)
4. If AR/QR identifies issues → human decides: fix now or accept with note
5. Loop back to step 1 if fixes made

**What this changes:** Today, gate failures block the commit (correct) but there's no "re-evaluate AC coverage" step. The loop adds an explicit "all AC verified?" checkpoint after implementation.

**Enterprise extension:** Steps 3-4 become automated (LLM-as-judge reviews, auto-fix for non-judgment issues, human gate only for disagreements).

**OSS/Enterprise:** OSS = hybrid (auto-fix mechanical + human-gate judgment). Enterprise = fully automated with human override.

---

### G5 — Feature Verification Checklist

**Verdict: ADOPT — connect AC to automated verification**

**Rationale:** Score 5 — highest in this story. The content (Gherkin AC) already exists. The gap is making it machine-readable and checkable. This directly strengthens P2 (Governance as Code) — acceptance criteria become verifiable artifacts, not just documentation.

**OSS implementation:**
1. Add `ac_status` structured section to story scope or a new `s{N}.{M}-checklist.yaml`:
```yaml
acceptance_criteria:
  - id: AC1
    description: "User can create a new session"
    status: pending  # pending | passed | failed
    verified_by: null  # test name, manual check, or gate
    verified_at: null  # timestamp
```
2. During `/rai-story-implement`, agent updates checklist status as tasks complete
3. During `/rai-story-review`, reviewer verifies all AC have `status: passed`
4. Criteria are immutable (agent cannot modify description, only status) — enforced by skill instruction

**Complements:** RAISE-795 (scope fence from S789.1) — the scope fence validates task↔AC coverage at plan time. The checklist validates AC↔status at implementation and review time. Together they close the full loop.

**OSS/Enterprise:** OSS Core — daily workflow improvement.

---

### G13 — Contract Negotiation

**Verdict: RECLASSIFY — not a gap, a mischaracterization. P1 ADR required.**

**Rationale:** Deep analysis reveals G13 was mischaracterized in the epic scope. Anthropic doesn't describe evaluator↔generator scope negotiation. They describe:
1. An initializer agent expanding a goal into criteria (≈ our `/rai-story-design`)
2. An evaluator assessing output quality against criteria (≈ our `/rai-quality-review`)
3. A lead agent calibrating resources for execution (≈ our `/rai-story-plan`)

None of these are "contract negotiation" in the scope-defining sense. They are all execution-level activities within human-defined goals.

**What's actually needed:** Not a new feature — a P1 reformulation (ADR) that honestly models the collaborative execution already happening in RaiSE. This unblocks G2 and G3 patterns for OSS by clarifying that execution-level evaluation is not a P1 violation.

**OSS/Enterprise:** Both — the ADR applies to the framework principle, not to a specific feature tier.
