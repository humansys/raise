# S789.1 Design — Context & Harness Patterns Research

## Complexity: Moderate

Research story with 4 gaps to evaluate across 4 source articles, producing structured analysis artifacts. No code produced.

---

## What & Why

**Problem:** RaiSE has 4 identified gaps in the Context & Harness domain (G4, G10, G12, G14) relative to Anthropic's published patterns. We lack data to decide what to adopt, adapt, or reject. Specifically, G10 (effort scaling) creates a tension with RaiSE principle #3 ("Full skill cycle always") that must be resolved with evidence, not opinion.

**Value:** Informed architecture decisions for 2.4.0 roadmap. Prevents both blind adoption (losing RaiSE identity) and blind rejection (missing real improvements). Directly informs RAISE-783 (session management) via G4 findings.

---

## Approach

### Research Methodology

Conduct a structured read of 4 Anthropic articles, extracting claims and cross-referencing against the RaiSE blueprint. Each gap gets a 4-dimension score and a verdict.

**Phase 1: Deep Read & Claim Extraction** (~40 min)

Read each article sequentially. For each article, extract:
- Concrete claims (what they say works/doesn't)
- Evidence level (do they provide data, examples, or just assertions?)
- RaiSE mapping (what mechanism does this relate to?)

Articles in reading order (dependency-first):
1. Art.4 — Context Engineering (foundational for G4)
2. Art.2 — Effective Harnesses (foundational for G10, G14)
3. Art.3 — Harness Design (refines G10, G14)
4. Art.6 — Multi-Agent Research (secondary evidence for G12, G14)

**Phase 2: Evidence Catalog Assembly** (~20 min)

Consolidate claims into an evidence catalog. Triangulate each major claim against:
- Source 1: The Anthropic article itself
- Source 2: RaiSE's own operational data (sessions, patterns, known friction)
- Source 3: First-principles reasoning from RaiSE's architectural principles

Confidence levels:
- **Very High**: 3+ sources agree, empirical data available
- **High**: 2-3 sources agree, consistent with experience
- **Medium**: 1-2 sources, plausible but unverified in RaiSE context
- **Low**: Single source, assertion without data

**Phase 3: Benchmark Scoring** (~20 min)

Score each gap on the 4 dimensions defined in epic scope:

| Dimension | Scale | Question |
|-----------|-------|----------|
| Value to Core | 0-3 | Strengthens memory, methodology, or entity? |
| Simplicity Cost | 0-3 | Complexity added? (lower = better) |
| Observability Gain | 0-3 | More traceable/auditable? |
| Adoption Effort | XS-XL | Story points to implement |

Priority = (Value to Core + Observability Gain) - Simplicity Cost
- 4-6: Do now (2.4.0)
- 1-3: Plan for enterprise
- 0 or negative: Reject or defer

**Phase 4: Verdict & Recommendations** (~20 min)

For each gap, produce:
- **Adopt**: Use Anthropic's pattern as-is
- **Adapt**: Use with RaiSE modifications (specify what changes)
- **Reject**: Do not adopt, with principled rationale

At least one recommendation must include a concrete implementation story.

## G10 Tension Resolution Framework

The effort scaling tension (G10) is the highest-stakes question in this story because it may change a RaiSE principle. The resolution follows a structured deliberation:

1. **State both positions clearly**
   - RaiSE: "Full skill cycle always" (rule #3) — consistency builds habits, overhead is minimal, structure helps
   - Anthropic: "Scale effort to task size" — light process for simple tasks, full ceremony for complex

2. **Gather evidence for each**
   - Pro full-cycle: session data showing small-story skill cycles completed in <5 min, patterns discovered in "simple" stories
   - Pro scaling: session data showing ceremony overhead on trivial fixes, cases where full cycle added friction without value

3. **Test against RaiSE principles**
   - P1 (Humans Define): Does scaling violate human control? (No, human decides task size)
   - P7 (Lean): Does full cycle create waste for trivial tasks? (Possibly)
   - P5 (Heutagogy): Does skipping steps lose learning opportunities? (Sometimes)

4. **Decide and document**
   - If changing principle: propose ADR with before/after and migration path
   - If keeping principle: document why with evidence
   - If refining (most likely): articulate the nuanced position (e.g., "full cycle with ShuHaRi-scaled depth")

## Examples (Specification by Example)

### Example 1: Evidence catalog entry

```markdown
### Claim: Context compaction requires task-specific policies
- **Source**: Art.4, "Managing Long Contexts" section
- **Anthropic says**: "Different task types need different compaction strategies — coding tasks preserve recent edits, research tasks preserve key findings"
- **RaiSE reality**: No compaction policy. Relies on Claude Code defaults. Post-compaction recovery fragile (bugs #12671, #15174). CLAUDE.md has manual recovery instructions.
- **Triangulation**: (1) Art.4 assertion, (2) RaiSE session data — 3+ sessions lost context after compaction with no recovery, (3) First principles — MVC (P7) implies context should be managed, not left to defaults
- **Confidence**: High
- **Maps to gap**: G4
```

### Example 2: Benchmark matrix row

```markdown
| G4: Context mgmt | 3 (directly improves daily reliability) | 1 (policy layer, not new subsystem) | 2 (compaction events become observable) | M | 4 | Adapt |
```

Rationale: Anthropic's compaction policy pattern is sound but needs RaiSE adaptation — our skills define natural compaction boundaries, and our session bundle already carries resumption context. Adapt by adding a per-skill compaction policy to skill frontmatter.

### Example 3: G10 verdict

```markdown
**Verdict: Adapt** — Refine "full skill cycle always" to "full skill cycle with ShuHaRi-scaled depth"

**Rationale**: Both positions have merit. The data shows that skill cycles for XS tasks complete in <5 minutes but sometimes surface unexpected insights (PAT-E-592). However, for truly trivial tasks (typo fix, config change), the full cycle adds ceremony without learning. The ShuHaRi mechanism already exists but only scales verbosity, not process steps.

**Proposed change**: Allow Ri-level practitioners to skip design for XS stories (already in skill — "When to skip" section). Formalize this as a principle refinement, not a principle change. No ADR needed — the skill already supports it.
```

### Example 4: Error case — insufficient evidence

```markdown
| G12: Agent test design | 1 (nice-to-have) | 1 (minor changes) | 1 (marginally more observable) | S | 1 | Defer |

**Note**: Insufficient evidence to score confidently. Anthropic's claims about "grep-friendly tests" are assertions without comparative data. Our 237 tests work well for agent-driven TDD today. Recommend gathering usage data before investing.
```

## Acceptance Criteria

### MUST
1. Evidence catalog exists with 12-20 entries covering all 4 gaps, each with source, triangulation, and confidence level
2. Benchmark matrix scores all 4 gaps on all 4 dimensions with 1-sentence rationales
3. Each gap has an Adopt/Adapt/Reject verdict with supporting rationale
4. G10 tension has an explicit resolution following the 4-step framework above
5. At least 1 implementation story drafted for the backlog

### SHOULD
1. Evidence catalog identifies at least 1 surprising finding (something that contradicts initial assumptions)
2. Recommendations cross-reference downstream stories (RAISE-783, enterprise roadmap)

### MUST NOT
1. Produce code or modify production files
2. Adopt any pattern without principled evaluation (no "Anthropic says so" as sole rationale)
3. Change RaiSE principles without proposing an ADR
