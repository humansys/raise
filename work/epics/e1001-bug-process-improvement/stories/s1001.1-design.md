# S1001.1 Design: Jira Infrastructure

## Problem & Value

No hay estructura para clasificar bugs — sin campos custom ni workflow formal, no se pueden analizar patrones ni extraer mejoras de proceso. Esta story habilita todo el epic.

## Approach

Configuración manual en Jira admin UI. Cero código. Tres pasos:
1. Crear 4 custom fields (select) con valores constrained
2. Activar workflow de 7 estados para Bug issue type
3. Validar con bug de prueba

## Custom Fields

| Campo Jira | Tipo | Valores exactos |
|------------|------|-----------------|
| `Bug Type` | Single Select | `Functional`, `Interface`, `Data`, `Logic`, `Configuration`, `Regression` |
| `Severity` | Single Select | `S0-Critical`, `S1-High`, `S2-Medium`, `S3-Low` |
| `Origin` | Single Select | `Requirements`, `Design`, `Code`, `Integration`, `Environment` |
| `Qualifier` | Single Select | `Missing`, `Incorrect`, `Extraneous` |

**Importante:** los nombres de campo y valores deben ser exactamente estos — S1001.2 (rai-bugfix v2.0) los referenciará por nombre.

## Workflow

```
Open → Triage → Diagnose → Fix → Verify → Retro → Done
```

Transiciones:
- Lineal: Open → Triage → Diagnose → Fix → Verify → Retro → Done
- Reopen: cualquier estado → Open

Aplica solo a issue type **Bug**. Story y Task no se tocan.

## Validación (bug de prueba)

```
1. Crear issue: type=Bug, summary="Test E1001 — workflow validation"
2. Clasificar:
   - Bug Type = Functional
   - Severity = S3-Low
   - Origin = Code
   - Qualifier = Incorrect
3. Transicionar: Open → Triage → Diagnose → Fix → Verify → Retro → Done
4. Verificar: campos persisten en cada transición
5. Cerrar o eliminar bug de prueba
```

## Acceptance Criteria

- **AC1:** 4 custom select fields visibles en Bug con valores exactos
- **AC2:** Workflow de 7 estados activo para Bug issue type
- **AC3:** Bug de prueba transicionado end-to-end con campos persistentes
- **AC4:** Nombres de campos documentados para S1001.2

## MUST NOT
- Tocar workflow de Story/Task
- Agregar post-functions ni automations
