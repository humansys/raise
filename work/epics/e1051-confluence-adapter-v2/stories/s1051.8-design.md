# S1051.8: Story Lifecycle Skills → Docs Adapter — Design

## Problem
Skills write artifacts as local files only. No Confluence publication.
Manual `rai docs publish` is error-prone and requires governance/ convention.

## Value
Every story artifact auto-published to Confluence. Dogfood from this story onward.

## Complexity: Simple
1 CLI flag + 4 skill text updates. No new models or architecture.

## Approach

### CLI change: `--file` flag

```python
@docs_app.command()
def publish(
    artifact_type: str,
    title: str | None = None,
    file: Path | None = None,   # NEW — read content from any file
    target: str | None = None,
) -> None:
    if file:
        content = file.read_text()
        path = str(file)
    else:
        # existing behavior: governance/{type}.md
        resolved = _resolve_artifact_path(artifact_type)
        content = resolved.read_text()
        path = str(resolved)
    ...
```

### Skill change pattern

Each skill that writes an artifact gets a publish step after the Write:

```markdown
## Output

After writing the artifact file, publish it:
\`\`\`bash
rai docs publish {doc_type} --file {path} --title "{title}"
\`\`\`
```

### Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Write file first, then publish | Reliable: local copy always exists, publish is additive |
| D2 | `--file` flag, not stdin | Simpler, no escaping issues with markdown |
| D3 | Publish is non-blocking | If Confluence fails, skill continues (composite handles warning) |
| D4 | Title = story ID + artifact type | Consistent, searchable in Confluence |
