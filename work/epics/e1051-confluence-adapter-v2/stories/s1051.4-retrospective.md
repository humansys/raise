## What Went Well

1. **Clean design-to-implementation pipeline.** The design doc specified flows for every method (`discover_page_tree`, `_walk_children`, `build_space_map`, `_wrap_error`). Implementation followed them almost verbatim -- one deviation noted below (T2 using `get_page_by_id` for homepage title).
2. **V1/V2 coexistence was zero-friction.** Placing `ConfluenceDiscoveryService` alongside existing `ConfluenceDiscovery` in the same module required no changes to V1 code or tests. All 9 original tests passed unmodified.
3. **TDD caught a real issue early.** Pyright's strict mode + self-referential `PageNode` required `model_rebuild()` AND a type-ignore comment. The RED phase surfaced this before implementation got complex.
4. **Primed patterns were directly applicable.** PAT-E-595 (granular methods), PAT-E-597 (TYPE_CHECKING guard), PAT-E-598 (error cause logging) all applied and were confirmed. 75% acceptance rate.

## What to Improve

1. **Graph unavailability in worktrees.** All 3 tier1 queries across design/plan/implement returned 0 results because the graph index doesn't exist in worktrees. This is a systemic gap -- worktrees are the standard work context. Consider: (a) symlink graph index from main repo, or (b) make `rai graph query` fall back to main repo's index.
2. **Design didn't specify homepage title retrieval.** `build_space_map` needed to call `get_page_by_id` for the homepage title, which wasn't in the design flow. Minor, but shows design should trace ALL client calls needed per method.

## Heutagogical Checkpoint

### What did I learn?
Pyright strict mode interacts poorly with Pydantic's self-referential model pattern (`model_rebuild()`). The resolution -- `pyright: ignore[reportUnknownVariableType]` -- is a targeted suppression, not a blanket ignore. This is now documented as PAT-E-711.

### What would I change about the process?
The graph index gap in worktrees appeared in every story of this epic (S1051.1 through S1051.4). It should be addressed as infrastructure before the next epic rather than noted repeatedly.

### Are there improvements for the framework?
Yes: `rai graph query` should have a fallback path for worktrees. Either resolve to the parent repo's index or surface a clear "worktree mode -- using stale index" warning instead of silently returning 0 results.

### What am I more capable of now?
Building service layers that compose low-level client methods into domain-meaningful operations. The discovery service transforms flat API calls (get_children, get_labels) into a recursive tree with inverted label index -- the value is in the composition, not the individual calls.
