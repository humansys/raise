# S479.2 Retrospective — Git Evidence Extractor

**Size:** M | **Tasks:** 4 | **Commits:** 4

## Summary

Built a subprocess-based git evidence extractor that pulls commits, merge commits, tags, and branches as structured `EvidenceItem` records with date-range filtering. TDD throughout, shared model for future extractors.

## What Went Well

- Clean TDD cycle: EvidenceItem model first, then extractors layered on top
- Subprocess mocking pattern worked cleanly for git command testing
- Pipe-delimited format with `maxsplit` handles commit messages containing delimiters
- EvidenceItem Pydantic model is reusable across future extractors (Jira, Confluence)
- Integration test validates end-to-end extraction pipeline

## What to Improve

- **Branch date filtering bug (R1):** Tag extraction had date filtering but branch extraction initially missed it. Caught by review, not by tests. Tests happened to use data within range, masking the gap.
- **Lesson:** When implementing the same filtering logic across multiple extraction methods, test with data OUTSIDE the expected range to verify filtering actually works.
- **Potential commit/merge-commit duplication (R6):** Merge commits appear in both commit and merge-commit results. Future story should consider `--no-merges` flag or deduplication.
- **Approvals semantics (R7):** Branch protection / approval evidence is inferred from merge commits, not directly extracted. Semantics need clarification in a future story.

## Known Debt

| ID | Description | Priority |
|----|-------------|----------|
| R1 | Branch date filter was missing, fixed in review | Resolved |
| R6 | `--no-merges` for commit extraction to avoid duplication | Low |
| R7 | Approvals semantics — merge commits as proxy for approvals | Medium |

## Heutagogical Reflection

1. **Learned:** Subprocess-based git extraction is effective for compliance evidence. The pipe-delimited parsing pattern is robust.
2. **Would change:** Copy date-filtering pattern to ALL methods from the start. Write boundary tests (data outside range) before happy-path tests.
3. **Framework:** QR caught a semantic bug that tests missed due to convenient test data. Reinforces: test boundaries, not just happy paths.
4. **Capability gained:** Building evidence extraction pipelines from version control data.
