# S479.1 Design: Control Mapping Configuration

## Problem

Teams preparing for ISO 27001 audits have no structured mapping between Annex A.8 controls and the evidence artifacts RaiSE already produces. Evidence collection is manual, ad-hoc, and takes days per audit cycle.

## Value

A validated YAML config establishes the foundation for automated evidence collection. All downstream extractors (S479.2-4) and report generators (S479.5-7) depend on this mapping to know what to collect and where to find it.

## Approach

Create a `compliance` package under `raise_cli` with three components:

1. **YAML config** (`controls.yaml`) -- Default mapping of 8 ISO 27001:2022 Annex A.8 controls to RaiSE evidence sources (git, gate, session)
2. **Pydantic models** (`models.py`) -- `EvidenceSourceConfig`, `ControlConfig`, `ControlMapping` with strict validation
3. **Loader** (`loader.py`) -- `load_control_mapping(path?)` reads YAML, validates via Pydantic, returns typed `ControlMapping`

### Components

| File | Action | Purpose |
|------|--------|---------|
| `src/raise_cli/compliance/__init__.py` | CREATE | Package init |
| `src/raise_cli/compliance/controls.yaml` | CREATE | Default mapping: 8 A.8 controls to evidence sources |
| `src/raise_cli/compliance/models.py` | CREATE | Pydantic models for config validation |
| `src/raise_cli/compliance/loader.py` | CREATE | Config loader with default + custom path support |

## Models

```python
from pydantic import BaseModel
from typing import Literal

class EvidenceSourceConfig(BaseModel):
    type: Literal["git", "gate", "session"]
    extractor: str
    description: str

class ControlConfig(BaseModel):
    id: str
    name: str
    description: str
    evidence_sources: list[EvidenceSourceConfig]

class ControlMapping(BaseModel):
    version: str
    standard: str
    controls: list[ControlConfig]
```

## YAML Config Structure

```yaml
version: "1.0"
standard: "ISO 27001:2022"
controls:
  - id: "A.8.32"
    name: "Change management"
    description: "Changes to information processing facilities and systems shall be subject to change management procedures."
    evidence_sources:
      - type: git
        extractor: commits
        description: "Commit history with messages, authors, timestamps"
      - type: git
        extractor: pull_requests
        description: "PR reviews and approvals"
      - type: git
        extractor: backlog_references
        description: "Issue/ticket references in commits and PRs"
```

## Loader API

```python
from raise_cli.compliance.loader import load_control_mapping

# Default: loads bundled controls.yaml
mapping = load_control_mapping()

# Custom: loads user-provided config
mapping = load_control_mapping(Path("custom-controls.yaml"))

# Access controls
for control in mapping.controls:
    print(f"{control.id}: {control.name}")
    for source in control.evidence_sources:
        print(f"  - {source.type}/{source.extractor}")
```

The loader:
- Reads YAML from the given path (or default package path)
- Validates via `ControlMapping.model_validate()`
- Raises `ValidationError` with clear messages on invalid config
- Raises `FileNotFoundError` if path doesn't exist

## Acceptance Criteria

**MUST:**
- AC1: YAML config with all 8 A.8 controls and their evidence source mappings
- AC2: Pydantic models validate config on load (invalid YAML raises ValidationError)
- AC3: Default config ships with package and loads without arguments
- AC4: Custom config path supported via `load_control_mapping(path)`

**SHOULD:**
- AC5: Helpful error messages on invalid control config

**MUST NOT:**
- No evidence extraction logic (that's S479.2-4)
- No report generation

## Complexity

**Simple** -- 3 new files + 1 YAML config, no integration with existing code, no side effects. Standard Pydantic + YAML loading pattern.
