# S479.2 Design: Git Evidence Extractor

## Problem

No automated way to extract ISO 27001 evidence from git history. Teams manually copy-paste git logs into audit spreadsheets, which takes days per audit cycle.

## Value

Covers 3+ Annex A.8 controls (A.8.4, A.8.9, A.8.32) with traceable EvidenceItem instances extracted from existing git data. Eliminates manual evidence collection for all git-sourced controls.

## Approach

Create an evidence extractor that:

1. **Reads ControlMapping** (from S479.1) to identify controls with `type: "git"` evidence sources
2. **Calls git via subprocess** to extract commits, tags, branches, merge commits
3. **Maps extracted data** to `EvidenceItem` Pydantic model instances
4. **Supports date range filtering** for audit periods

## EvidenceItem Model

Defined in `compliance/evidence.py`, shared by all extractors (git, gate, session):

```python
from pydantic import BaseModel, ConfigDict
from datetime import datetime
from typing import Literal

class EvidenceItem(BaseModel):
    model_config = ConfigDict(frozen=True)
    control_id: str
    control_name: str
    evidence_type: Literal["git", "gate", "session"]
    title: str
    description: str
    timestamp: datetime
    source_ref: str
    url: str | None = None
```

This matches the contract from the epic design document.

## Extractor API

```python
from raise_cli.compliance.extractors.git import GitEvidenceExtractor
from raise_cli.compliance.loader import load_control_mapping
from datetime import date
from pathlib import Path

mapping = load_control_mapping()
extractor = GitEvidenceExtractor(repo_path=Path("."))

items = extractor.extract(
    mapping=mapping,
    start_date=date(2025, 1, 1),
    end_date=date(2025, 12, 31),
)

for item in items:
    print(f"[{item.control_id}] {item.title} ({item.timestamp})")
```

### GitEvidenceExtractor

- **Constructor**: `GitEvidenceExtractor(repo_path: Path)`
- **Method**: `extract(mapping: ControlMapping, start_date: date, end_date: date) -> list[EvidenceItem]`

Internally iterates over `mapping.controls`, filters for controls with git-type evidence sources, and dispatches to the appropriate git command based on the `extractor` field.

## Git Commands

| Artifact | Command | Format |
|----------|---------|--------|
| Commits | `git log --after={start} --before={end} --format="%H\|%an\|%aI\|%s"` | hash, author, ISO date, subject |
| Tags | `git tag --sort=-creatordate --format="%(refname:short)\|%(creatordate:iso-strict)\|%(subject)"` | tag name, date, subject |
| Branches | `git branch -a --format="%(refname:short)\|%(committerdate:iso-strict)"` | branch name, date |
| Merge commits | `git log --merges --after={start} --before={end} --format="%H\|%an\|%aI\|%s"` | hash, author, ISO date, subject |

Tags are filtered post-extraction by date range (git tag doesn't support `--after`/`--before`).

## Control-to-Evidence Mapping

| Control | Git Extractors | Evidence Demonstrates |
|---------|---------------|----------------------|
| A.8.4 | commits | Source code access patterns via commit authors |
| A.8.9 | commits, branches, tags | Configuration management via versioning discipline |
| A.8.32 | commits, merge_commits | Change management via history and review process |

The extractor reads `evidence_sources` from each control in the mapping. Only sources with `type: "git"` are processed. The `extractor` field determines which git command to run.

## Acceptance Criteria

**MUST:**
- AC1: EvidenceItem Pydantic model created and shared across all extractors
- AC2: GitEvidenceExtractor extracts commits as evidence for A.8.32 and A.8.9
- AC3: GitEvidenceExtractor extracts tags as evidence for A.8.9
- AC4: GitEvidenceExtractor extracts merge commits as evidence for A.8.32
- AC5: Date range filtering works (start_date, end_date params)

**SHOULD:**
- AC6: Branch extraction for workflow discipline evidence (A.8.9)

**MUST NOT:**
- No report generation (that's S479.5)
- No gate or session extraction (S479.3, S479.4)

## Risks

| Risk | Mitigation |
|------|------------|
| Git subprocess behavior varies across OS/versions | Use standard --format options; test with mock subprocess |
| Large repos produce many commits | Date range filtering limits scope; pagination not needed for v1 |

## Complexity

**Moderate** -- 3-4 components, custom git parsing logic, integration with S479.1 models. Subprocess management and date filtering add moderate complexity.

## Dependencies

- S479.1 (Control mapping) -- ControlMapping, ControlConfig, EvidenceSourceConfig models
