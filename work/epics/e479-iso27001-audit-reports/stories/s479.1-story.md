# S479.1: Control Mapping Configuration

## User Story

**As a** development team lead preparing for ISO 27001 recertification,
**I want** a YAML configuration mapping ISO 27001:2022 Annex A.8 controls to RaiSE evidence sources,
**so that** the audit report generator knows which artifacts to collect for each control.

## Acceptance Criteria

```gherkin
Feature: Control mapping configuration

  Scenario: Load default control mapping
    Given the default control mapping YAML exists
    When the mapping is loaded
    Then it contains exactly 8 A.8 controls
    And each control has at least one evidence source

  Scenario: Validate control mapping structure
    Given a control mapping YAML
    When it is parsed via Pydantic models
    Then each control has: id, name, description, evidence_sources
    And each evidence_source has: evidence_type (git|gate|session), extractor, description

  Scenario: Reject invalid mapping
    Given a control mapping YAML with missing required fields
    When it is parsed via Pydantic models
    Then a validation error is raised with clear message

  Scenario: Map A.8.4 — Access to source code
    Given the default mapping is loaded
    When I look up control "A.8.4"
    Then evidence_type is "git"
    And extractor references git permissions

  Scenario: Map A.8.25 — Secure development lifecycle
    Given the default mapping is loaded
    When I look up control "A.8.25"
    Then evidence_sources include gate-based evidence (tests, types, lint)

  Scenario: Map A.8.32 — Change management
    Given the default mapping is loaded
    When I look up control "A.8.32"
    Then evidence_sources include git commits, PRs, and backlog references
```

## Controls Mapped

|---------|-------------|---------------|----------------|

| A.8.9 | Configuration management | git | Git history, branches, tags |
