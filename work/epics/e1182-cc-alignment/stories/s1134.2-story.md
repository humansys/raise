# Story: Allowed-Tools Declaration for All Skills

## User Story
As a RaiSE developer using Claude Code,
I want every skill to declare its allowed-tools explicitly,
so that CC restricts tool access per skill, reducing permission prompts and limiting blast radius.

## Acceptance Criteria

### Scenario: Tool restriction enforced
```gherkin
Given a skill with allowed-tools declared in frontmatter
When CC executes the skill
Then only the declared tools are available during execution
And no unexpected permission prompts appear
```

### Scenario: Bash patterns at subcommand granularity
```gherkin
Given a skill that uses rai CLI commands
When its allowed-tools lists Bash patterns
Then patterns use subcommand globs like Bash(rai:*) not bare Bash
```

### Scenario: Tier-appropriate classification
```gherkin
Given a skill classified into a tool tier
When the allowed-tools are reviewed
Then the tools match the skill's actual needs (read-only, CLI, git, full-dev, research, orchestration)
```

## Examples (Specification by Example)

| Tier | Example Skill | Allowed Tools |
|------|---------------|---------------|
| Read-only | rai-session-start | Read, Glob, Grep, Bash(rai:*) |
| Read+CLI+Git | rai-story-close | Read, Glob, Grep, Edit, Write, Bash(rai:*), Bash(git:*) |
| Full-dev | rai-story-implement | Read, Glob, Grep, Edit, Write, Bash(rai:*), Bash(git:*), Bash(uv:*) |

## Notes
- D2 design in epic research defines the 5-tier classification
- 35 skills total, each classified into one tier
- Bash glob patterns follow CC conventions: `Bash(command:*)` syntax
- No skill content/instructions changes — frontmatter only
