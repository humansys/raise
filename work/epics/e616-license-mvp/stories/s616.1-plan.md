### T3: Auth module — verify_member + requires_plan + requires_role (M)

**What:** Rewrite `auth.py`: replace `verify_api_key` → `verify_member` with full joined
query chain (key → member → org in one query, then license lookup). Add `requires_plan(minimum)`
and `requires_role(role)` FastAPI dependencies. Update `last_used_at` inline.

**Test strategy:** Pure logic tests — mock the DB session to return pre-built model objects.
No actual DB needed. Tests verify: correct MemberContext construction, error handling (401/403),
plan/role gating logic. DB correctness verified by migration tests (T2) and smoke test (T5).

**Files:**
- modify: `packages/raise-server/src/raise_server/auth.py`
- create: `packages/raise-server/tests/test_auth.py`

**TDD:**
- RED: Tests for each scenario from design:
  - Valid key → MemberContext with correct plan/features
  - No license → community plan, empty features
  - Expired license → community plan (not matched by active query)
  - Inactive member → 401
  - Inactive key → 401
  - Missing/malformed header → 401
  - requires_plan: pro member on team endpoint → 403 with structured detail
  - requires_plan: team member on team endpoint → passes
  - requires_plan: enterprise on any → passes
  - requires_role: member role on admin endpoint → 403
  - requires_role: admin role on admin endpoint → passes
- GREEN: Implement verify_member (joined query), requires_plan, requires_role.
- REFACTOR: Ensure joined query is clean, error messages are structured.

## Execution Order

```
T1 (models + schemas + test infra)
  → T2 (migration)
  → T3 (auth module)  ← riskiest, do before routers
    → T4 (router updates)
      → T5 (integration smoke)
```

**Rationale:** Risk-first — T3 (auth) is the core logic and most likely to surface design issues.
T1/T2 are foundational. T4 is mechanical (find-and-replace pattern). T5 validates everything.
