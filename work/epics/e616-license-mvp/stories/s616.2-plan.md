## Execution Order

```
T1 (schemas + org) → T2 (members + seats) ─┐
                   → T3 (api keys)         ├→ T5 (seed + E2E)
                   → T4 (licenses)         ─┘
```

T2, T3, T4 all depend on T1 (schemas) but are independent of each other.
Executed sequentially for simplicity. Risk-first: org (base), members (seat logic),
keys (show-once), licenses (simplest CRUD), seed (integration).

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Show-once key leaks raw value in GET | High (security) | TDD: explicit test that GET response has no `key` field |
| Seat enforcement race condition | Low (MVP <10 clients) | Acceptable for MVP; DB-level constraint in V2 |
| Org-scoping bypass | High (data leak) | Every endpoint tests wrong-org access returns 403 |
