## Summary

Added `_validate_frontmatter()` private method to `FilesystemDocsTarget.publish()` that validates YAML frontmatter before writing to disk. Validates: parseable YAML, mapping type, required fields per artifact type (base, epic-level, story-level). Content without frontmatter passes through unchanged. AR and QR both ran, QR caught inconsistent error signaling (mix of exceptions and `PublishResult`) which was fixed before merge.

## What Went Well

1. **Jidoka applied at AR+QR gates.** AR passed with questions — Q2 about whether a custom exception was warranted was valid and prompted reflection. QR caught F5 (inconsistent error signaling: `publish()` returned `PublishResult(success=False)` for missing path but raised `FrontmatterValidationError` for bad frontmatter). Both findings were fixed before merge rather than deferred to a follow-up story. The lifecycle gates caught real design inconsistencies.

2. **Surgical insertion kept the diff small.** One private method (`_validate_frontmatter`) added, one call site in `publish()`. No protocol changes, no new modules, no dependency additions (`yaml` is stdlib). The feature was fully testable in isolation through `publish()` without any test infrastructure changes.

3. **QR finding led to simpler design.** Removing `FrontmatterValidationError` and unifying on `PublishResult(success=False)` for all error paths made the API simpler for callers. They check one thing (`result.success`) instead of handling both exceptions and result objects. The "fix" was actually a simplification.

4. **Design decisions were proportional.** D1 initially chose exceptions for validation errors. QR invalidated that choice. The pivot was cheap (single commit) because the method was self-contained. Design decisions should be easy to reverse when new information arrives.

## What To Improve

1. **Error signaling consistency should be a design-phase check.** The inconsistency between `PublishResult(success=False)` for missing path and `FrontmatterValidationError` for bad content was visible in the design doc's "Before/After" code blocks. A design review question like "does the new error path match the existing error convention?" would have caught this before implementation.

2. **T2 (integration verification) was implicit again.** Same observation as S1040.2 — the verification task was absorbed into the QR fix commit rather than executed as a separate step. For S-sized stories this is acceptable, but the pattern is recurring.

## Heutagogical Checkpoint

1. **What did I learn?**
   - Error signaling consistency is a first-class design concern. When a method already returns Result objects for errors, adding exceptions for new error paths creates a mixed API that forces callers to handle two error channels. The single-channel principle: pick one (Result or exception) and use it everywhere.
   - AR questions can surface design issues even when the verdict is PASS. Q2 ("do you need a custom exception?") was the seed that QR's F5 finding confirmed.

2. **What would I change about the process?**
   - Add "error channel consistency" to the design review checklist: does the new code use the same error signaling mechanism as the existing code in the same method/class? This would catch mixed exception/result patterns at design time.

3. **Are there improvements for the framework?**
   - The AR+QR pipeline worked as designed. AR surfaced a question, QR found the concrete defect, fix was applied before merge. No process change needed.
   - Consider adding "error signaling consistency" as a heuristic in the QR skill's semantic-bug checklist.

4. **What am I more capable of now?**
   - Clear pattern for adding validation to existing publish/write methods: private method, called before write, returns error result (not exception) matching existing convention.
   - Better intuition for when a custom exception is premature: if the only caller already speaks a Result type, use the Result type.
