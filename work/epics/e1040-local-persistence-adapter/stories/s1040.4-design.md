# S1040.4 Design: Harden FilesystemDocsTarget — Frontmatter Validation

## Problem & Value

`FilesystemDocsTarget.publish()` does a raw `path.write_text(content)` with no content validation. Malformed YAML frontmatter or missing required fields (title, status, story_id/epic_id) pass through silently, producing artifacts that break downstream readers (e.g. `artifacts/reader.py` dispatching on `artifact_type`). Catching these at write time gives clear, immediate error messages instead of mysterious failures later.

## Complexity: Simple

- 1 file to modify (`filesystem_docs.py`)
- 1 private method to add
- No protocol changes, no new dependencies (stdlib `yaml` already available)

## Approach

Add a `_validate_frontmatter(content: str, doc_type: str) -> None` private method to `FilesystemDocsTarget`. Called in `publish()` before `path.write_text()`. Raises `FrontmatterValidationError` on failure.

**Frontmatter detection:** Content starts with `---\n`, second `---\n` closes the block. If no frontmatter delimiters found, skip validation entirely (not all docs have frontmatter).

**Validation steps (when frontmatter exists):**
1. Extract text between first two `---` lines
2. Parse with `yaml.safe_load()` — if it fails, raise with parse error
3. Check required fields present — if missing, raise listing them

**Required fields by artifact level:**

| Level | Required Fields | How Detected |
|-------|----------------|--------------|
| Base (all artifacts with frontmatter) | `title`, `status` | Always checked |
| Story-level | `title`, `status`, `story_id`, `epic_id` | `story_id` present in frontmatter OR `doc_type` starts with `story-` |
| Epic-level | `title`, `status`, `epic_id` | `epic_id` present without `story_id`, OR `doc_type` starts with `epic-` |

Detection is best-effort: if we can infer the level from frontmatter keys or `doc_type`, we check the stricter set. Otherwise, only base fields are required.

## Error Type

```python
class FrontmatterValidationError(Exception):
    """Raised when content has invalid or incomplete frontmatter."""
    def __init__(self, message: str, missing_fields: list[str] | None = None) -> None:
        super().__init__(message)
        self.missing_fields = missing_fields or []
```

Defined in `filesystem_docs.py` (colocated with only caller). Exported for test imports.

## Before / After

### Before (current `publish()`):

```python
def publish(self, doc_type: str, content: str, metadata: dict[str, Any]) -> PublishResult:
    raw_path = metadata.get("path")
    if not raw_path:
        return PublishResult(success=False, message="metadata['path'] is required...")
    path = self._root / raw_path
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(content)  # <-- no validation
    return PublishResult(success=True, url=str(path), message="Written to filesystem")
```

### After:

```python
def publish(self, doc_type: str, content: str, metadata: dict[str, Any]) -> PublishResult:
    raw_path = metadata.get("path")
    if not raw_path:
        return PublishResult(success=False, message="metadata['path'] is required...")
    self._validate_frontmatter(content, doc_type)  # <-- raises on invalid
    path = self._root / raw_path
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(content)
    return PublishResult(success=True, url=str(path), message="Written to filesystem")
```

### `_validate_frontmatter()`:

```python
def _validate_frontmatter(self, content: str, doc_type: str) -> None:
    """Validate frontmatter YAML is parseable and has required fields."""
    if not content.startswith("---\n"):
        return  # No frontmatter — pass through

    end = content.find("\n---\n", 4)
    if end == -1:
        end = content.find("\n---", 4)
        if end == -1:
            return  # Malformed delimiters — treat as no frontmatter

    raw_yaml = content[4:end]
    try:
        data = yaml.safe_load(raw_yaml)
    except yaml.YAMLError as exc:
        raise FrontmatterValidationError(f"Unparseable YAML frontmatter: {exc}") from exc

    if not isinstance(data, dict):
        raise FrontmatterValidationError(
            f"Frontmatter must be a YAML mapping, got {type(data).__name__}"
        )

    # Determine required fields
    required: set[str] = {"title", "status"}
    is_story = "story_id" in data or doc_type.startswith("story-")
    is_epic = "epic_id" in data or doc_type.startswith("epic-")

    if is_story:
        required |= {"story_id", "epic_id"}
    elif is_epic:
        required.add("epic_id")

    missing = required - data.keys()
    if missing:
        raise FrontmatterValidationError(
            f"Missing required frontmatter fields: {sorted(missing)}",
            missing_fields=sorted(missing),
        )
```

## Decisions

| ID | Choice | Rationale | Alternatives |
|----|--------|-----------|-------------|
| D1 | Raise exception, not `PublishResult(success=False)` | ACs say "validation error is raised". Callers need to distinguish "no path in metadata" (soft) from "corrupt content" (hard). | Return failed PublishResult (masks the defect, caller may ignore) |
| D2 | `FrontmatterValidationError` in `filesystem_docs.py` | Only caller is `publish()`. No reason to put it elsewhere yet. | Shared exceptions module (premature) |
| D3 | No-frontmatter content passes through | Not all docs have frontmatter (e.g. plain governance docs, rendered artifacts). Scope explicitly says "pass through". | Require frontmatter always (breaks existing content) |
| D4 | Infer artifact level from frontmatter keys + `doc_type` | Best-effort heuristic — frontmatter keys are the primary signal, `doc_type` is fallback. Avoids a rigid registry. | Strict doc_type → fields mapping (fragile, requires maintenance) |

## Acceptance Criteria (refined from story)

### MUST
1. Valid frontmatter with required fields: file written normally (AC1)
2. Unparseable YAML: `FrontmatterValidationError` raised before write (AC2)
3. Missing required fields: `FrontmatterValidationError` with field list (AC3)
4. No frontmatter delimiters: content written as-is (AC4)
5. Epic artifacts require `epic_id` but not `story_id` (AC5)
6. Story artifacts require both `story_id` and `epic_id` (AC5)

### MUST NOT
1. Change `DocumentationTarget` protocol interface
2. Validate field values (only presence)
3. Add frontmatter to content that lacks it
