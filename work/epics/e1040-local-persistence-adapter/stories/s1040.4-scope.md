# S1040.4: Scope

## In Scope

- Parse frontmatter (YAML between `---` delimiters) from content before write
- Validate frontmatter is parseable YAML — reject with clear error if not
- Validate required fields per artifact type (title, status, epic_id/story_id)
- Raise validation error listing missing fields when required fields absent
- Pass through content without frontmatter delimiters unchanged (no validation)
- Full test coverage with TDD for all validation paths

## Out of Scope

- Changing the `DocumentationTarget` protocol interface
- Validating field values (e.g. status enum, date formats) — only presence checked
- Adding frontmatter to content that lacks it
- Modifying ConfluenceTarget or any other adapter
- Schema versioning or migration of existing artifacts
- Validating markdown body content beyond frontmatter

## Done Criteria

- [ ] Frontmatter extracted and parsed as YAML before file write
- [ ] Unparseable YAML frontmatter raises validation error before write
- [ ] Missing required fields (title, status, epic_id or story_id) raise error with field list
- [ ] Content without frontmatter delimiters written as-is without validation
- [ ] Required fields are context-aware (epic vs story artifact type)
- [ ] All tests pass (TDD, full coverage of validation paths)
- [ ] Type annotations on all code, pyright strict passes
- [ ] Linting passes
- [ ] No changes to DocumentationTarget protocol

## Technical Notes

- FilesystemDocsTarget already exists and does raw `path.write_text(content)`
- Validation happens inside FilesystemDocsTarget.publish(), before the write call
- Frontmatter detection: content starts with `---\n`, second `---\n` closes the block
- Use `yaml.safe_load()` for parsing — no custom constructors needed
- Artifact type can be inferred from frontmatter keys (epic_id without story_id = epic-level)
