# S1040.2: Migrate CLI Writes to FilesystemAdapter + Add Protections

## User Story (Connextra)

**As a** CLI user running session close, journal append, or memory writes,
**I want** all filesystem writes to go through FilesystemAdapter,
**so that** writes are atomic, encoding is consistent, and stale overwrites are prevented.

## Acceptance Criteria (Gherkin)

### AC1: Timestamp protection rejects stale session state overwrites

```gherkin
Scenario: Stale session state write is rejected
  Given a session state file exists with timestamp T2
  When save_session_state is called with data from timestamp T1 (T1 < T2)
  Then the write is rejected with a StaleWriteError
  And the existing file is unchanged
```

### AC2: Timestamp protection allows fresh session state overwrites

```gherkin
Scenario: Fresh session state write succeeds
  Given a session state file exists with timestamp T1
  When save_session_state is called with data from timestamp T2 (T2 > T1)
  Then the file is overwritten with the new data
```

### AC3: Pattern append uses FilesystemAdapter

```gherkin
Scenario: append_pattern writes atomically via adapter
  Given an existing patterns.jsonl file
  When append_pattern is called with a new pattern
  Then the pattern is appended atomically via FilesystemAdapter.append()
  And the file content is a valid JSONL file
```

### AC4: Session journal append uses FilesystemAdapter

```gherkin
Scenario: append_journal_entry writes atomically via adapter
  Given an existing session journal file
  When append_journal_entry is called with a new entry
  Then the entry is appended atomically via FilesystemAdapter.append()
```

### AC5: Session state write uses FilesystemAdapter

```gherkin
Scenario: save_session_state writes atomically via adapter
  Given an active session
  When save_session_state is called
  Then the state is written atomically via FilesystemAdapter.write()
  And the file content is identical to what write_text would produce
```

### AC6: Memory writer append uses FilesystemAdapter

```gherkin
Scenario: append_session in memory writer uses adapter
  Given an existing sessions.jsonl file
  When append_session is called with a session record
  Then the record is appended atomically via FilesystemAdapter.append()
```

### AC7: Developer profile writes use FilesystemAdapter

```gherkin
Scenario: Developer profile write uses adapter with home root
  Given a developer profile update
  When the profile is saved
  Then FilesystemAdapter is instantiated with root=~/.rai
  And the write goes through adapter.write()
```

### AC8: Crash during write preserves original file

```gherkin
Scenario: Process crash during write leaves original intact
  Given an existing session state file with valid content
  When a write operation is interrupted mid-way
  Then the original file content is unchanged
```

### AC9: Consistent encoding on all writes

```gherkin
Scenario: All writes use UTF-8 encoding
  Given any CLI write operation
  When the write is executed via FilesystemAdapter
  Then the file is encoded as UTF-8
```
