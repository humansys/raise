# S1040.4: Harden FilesystemDocsTarget — Frontmatter Validation

## User Story (Connextra)

**As a** developer publishing work artifacts (scope, design, plan, retro) via the documentation pipeline,
**I want** FilesystemDocsTarget to validate frontmatter before writing to disk,
**so that** malformed or incomplete artifacts are caught early with clear error messages.

## Acceptance Criteria (Gherkin)

### AC1: Valid frontmatter passes validation and file is written

```gherkin
Scenario: Valid frontmatter with all required fields
  Given content with parseable YAML frontmatter
  And the frontmatter contains all required fields for the artifact type
  When publish is called on FilesystemDocsTarget
  Then the file is written to disk normally
```

### AC2: Unparseable frontmatter raises validation error

```gherkin
Scenario: Malformed YAML in frontmatter block
  Given content with frontmatter delimiters (---) enclosing invalid YAML
  When publish is called on FilesystemDocsTarget
  Then a validation error is raised before any file write
  And the error message indicates the YAML parse failure
```

### AC3: Missing required fields raises validation error

```gherkin
Scenario: Frontmatter missing required fields
  Given content with valid YAML frontmatter
  But the frontmatter is missing required fields (e.g. no title)
  When publish is called on FilesystemDocsTarget
  Then a validation error is raised listing the missing fields
  And no file is written
```

### AC4: Content without frontmatter is written as-is

```gherkin
Scenario: Plain content with no frontmatter delimiters
  Given content with no --- delimiters at the start
  When publish is called on FilesystemDocsTarget
  Then the file is written to disk as-is without validation
```

### AC5: Required fields vary by artifact type

```gherkin
Scenario: Epic artifact requires epic_id but not story_id
  Given content with frontmatter containing epic_id and title and status
  But no story_id field
  When publish is called for an epic-level artifact
  Then the file is written successfully

Scenario: Story artifact requires story_id
  Given content with frontmatter containing title and status
  But no story_id field
  When publish is called for a story-level artifact
  Then a validation error is raised listing story_id as missing
```
