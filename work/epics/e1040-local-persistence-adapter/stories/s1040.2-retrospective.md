# Retrospective: S1040.2 — Migrate CLI Writes to FilesystemAdapter + Protections

**Story:** S1040.2 | **Epic:** E1040 | **Jira:** RAISE-1042
**Size:** M | **Branch:** `story/s1040.2/cli-adapter-migration`
**Date:** 2026-04-02

## Summary

Migrated 5 write sites across 4 files from raw `Path.write_text()`/`open("a")` to `FilesystemAdapter` atomic primitives. Added timestamp-based stale-overwrite protection for session state (closes RAISE-697). Added `StaleWriteError` exception and `last_modified` field to `SessionState`.

**Commits:** 7 (scope, design, plan, T1-T4 implementation, QR fix)
**Tests added:** 15 new tests (48 writer + 111 state + 20 journal + 47 profile = 226 in affected files)
**Final state:** 206 tests passing, pyright 0 errors, ruff clean

## What Went Well

1. **DD-8 pivot was the right call.** Recognizing the Steward layer would conflict with the Pluggable Domains vision (3.0) before building it saved an entire story (S1040.3) and avoided a temporary abstraction that would need dismantling. The pivot happened at the design phase — the cheapest possible point.

2. **Single migration point gave highest leverage.** Migrating `_append_jsonl` (T1) was a single function change that covered 3 callers (`append_pattern`, `append_calibration`, `append_session`). Identifying the choke point in the audit phase made T1 the obvious first task.

3. **Mechanical migration pattern was predictable.** Once T1 established the pattern (remove mkdir + raw write, replace with `FilesystemAdapter(root=X).write/append()`), T3 and T4 were near-identical. The plan correctly sequenced T1 first for this reason.

4. **QR caught a real bug.** String comparison of ISO 8601 timestamps (`"2026-04-02T10:00:00" < "2026-04-02T09:00:00"`) works coincidentally for same-length timestamps but breaks for mixed-format timestamps (e.g., with/without microseconds or timezone offsets). QR recommendation R1 fixed this with proper `datetime.fromisoformat()` parsing before comparison. This would have been a latent defect until legacy data appeared.

5. **Design doc as audit artifact.** The 5-site write audit in the design doc served dual purpose: it was both the analysis and the specification. Each site had "Current" and "After" code blocks, making the plan tasks trivially derivable.

## What To Improve

1. **Timestamp comparison should have been caught in TDD.** The tests used consistently-formatted timestamps, so the string comparison bug passed all tests. A test with mixed-format timestamps (e.g., one with microseconds, one without) would have caught it at RED phase. Lesson: when testing comparison logic, include edge cases with format variations, not just value variations.

2. **T5 (integration verification) was implicit.** The plan included T5 for cleanup and verification, but it was absorbed into the QR fix commit rather than executed as a separate step. For traceability, either execute T5 explicitly or note its absorption in the commit message.

3. **No learning records from earlier phases.** The design, plan, and implement phases did not emit learning records (introspection aspect not active). This means no PRIME/JIT pattern reuse signals were captured. Acceptable for this story but limits the introspection feedback loop.

## Heutagogical Checkpoint

1. **What did I learn?**
   - Strategic pivots at design phase are cheap and high-value. DD-8 removed an entire architectural layer (Steward) by recognizing alignment with a future vision (Pluggable Domains). The cost was one design-phase conversation; the saving was one story plus ongoing maintenance of a temporary abstraction.
   - ISO 8601 string comparison is unreliable across format variations. Always parse to `datetime` before comparing timestamps.

2. **What would I change about the process?**
   - Add "format variation" as an explicit edge case category for any comparison logic in TDD plans. The plan had tests for "older", "newer", and "missing" timestamps, but all used the same format. A "different format" test would have caught the QR finding during implementation.

3. **Are there improvements for the framework?**
   - The QR skill caught what TDD missed — confirming QR as a mandatory gate. No process change needed; the existing lifecycle worked correctly.
   - Consider adding a "comparison edge cases" checklist item to story-plan for tasks involving temporal or ordered comparisons.

4. **What am I more capable of now?**
   - Clear pattern for migrating raw I/O to adapter calls: identify choke points (shared functions like `_append_jsonl`), migrate bottom-up, verify callers don't change.
   - Confident in the adapter-at-call-site instantiation pattern — no DI needed for leaf functions with simple root resolution.
   - Better intuition for when an abstraction layer is premature vs. when infrastructure (like FilesystemAdapter) is permanent.
