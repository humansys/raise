# S350.2: Sanitize Git State

## User Story

**As a** new developer joining the project,
**I want** the repository to have clean git tracking with proper .gitignore rules,
**So that** I don't receive personal/ephemeral artifacts from other developers when I clone.

## Acceptance Criteria

```gherkin
Feature: Clean git state for multi-developer work

  Scenario: .claude/worktrees/ is gitignored
    Given the repository .gitignore
    When a developer creates worktrees via Claude Code
    Then the .claude/worktrees/ directory is not tracked by git

  Scenario: Legacy .claude/ files are removed from tracking
    Given the legacy files .claude/RAI.md, .claude/RAI-naming.md
    And the legacy archive .claude/rai.archive/
    When the cleanup is applied
    Then these files are removed from git tracking
    And they no longer appear in the repository

  Scenario: .raise/rai/personal/ directory exists after rai init
    Given a fresh clone of the repository
    When a developer runs `rai init`
    Then the .raise/rai/personal/ directory is created
    And it contains a .gitkeep file

  Scenario: .claude/projects/ machine-specific paths are gitignored
    Given the .claude/projects/ directory contains machine-specific paths
    When a developer clones the repository
    Then no machine-specific project memory is included
```

## Context

From S350.1 classification (mixed concerns M7, M8, M9, M10):
- **M10:** `.claude/worktrees/` not in .gitignore (per-dev ephemeral)
- **M7:** `.claude/RAI.md` legacy doc, superseded by CLAUDE.md + skills
- **M8:** `.claude/RAI-naming.md` legacy naming doc, no active use
- **M9:** `.claude/rai.archive/` historical archive (4 files), superseded
- `.claude/projects/` has machine-specific paths (per-dev memory)
