# S350.6 Retrospective — rai skill sync

## Summary

Added `rai skill sync` command that calls `scaffold_skills(dry_run=True)` to compare deployed skills against builtins using the dpkg three-hash algorithm. Outputs a status table (current/outdated/conflict) and exits 0 when all current, 1 when updates available.

Pure wiring — no new logic. The three-hash classification was already implemented in `scaffold_skills`. The command just needed to call it with `dry_run=True` and format the result.

3 new tests (mocking `scaffold_skills` for controlled output), all 28 skill tests passing. Type checks clean.

AR: PASS.

## Key Insight

When the underlying function already has a `dry_run` mode, the "check" command is trivial wiring. The design decision in S350.4 to add `dry_run=True` to `scaffold_skills` paid off here — zero new logic needed.

## Heutagogical Checkpoint

### 1. What did you learn?
XS stories don't need design phases. The plan itself was the design — "call existing function, format output." Overhead of a formal design doc would have exceeded the implementation effort.

### 2. What would you change?
Nothing. The story was correctly identified as pure wiring and executed in one pass.

### 3. Framework improvements?
Stories estimated XS-S that are pure wiring over existing dry_run modes could skip the design phase entirely. The plan suffices.

### 4. What are you more capable of?
Recognizing when prior stories' API design (dry_run parameter) creates trivial follow-up stories. Good API design compounds.
