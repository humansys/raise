## What Went Wrong
- **Missing `Path` import (PAT-E-597):** `from __future__ import annotations` masked a `NameError` in `_load_project_info`. The `Path` type annotation worked (lazy string) but `Path(str(...))` in the method body raised `NameError`, silently caught by bare `except Exception`. Found during review, not during implementation.
- The bare `except Exception` in error-isolation code (PAT-E-598) hid the bug. Should at minimum log the exception at debug level.

## Patterns Captured
- **PAT-E-597:** `from __future__ import annotations` masks NameError for unimported names in method bodies
- **PAT-E-598:** Bare `except Exception` in error-isolation patterns can hide import errors; always log the actual exception

## Heutagogical Reflection
The `from __future__ import annotations` trap is subtle and worth internalizing. The combination of lazy annotations + catch-all exception handling creates a particularly dangerous pattern where code appears to work but silently degrades. Review caught what TDD missed because the test fixture itself was correct — the bug was in the production code's import section, invisible to tests that import `Path` in their own scope.
