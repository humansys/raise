# S350.7: Jira Credentials Separation

- **Epic:** E350 — Rai Experience Portability
- **Tracker:** RAISE-223
- **Size:** S
- **Depends:** S350.2 (done)
- **Branch:** `story/s350.7/jira-credentials-separation`

## Problem

`.raise/jira.yaml` currently contains both project configuration (workflows, status mapping, team identifiers, issue types) and implicit credential references. While actual secrets live in environment variables, the file's structure doesn't clearly separate what is safe to commit from what might leak sensitive info. New developers need to know which env vars to set and where personal config belongs.

## Goal

Make `jira.yaml` safely committable by design:
1. Credentials come exclusively from environment variables (already the case, but make it explicit)
2. Per-developer overrides (e.g., personal identifier, default assignee) go to personal config
3. Document the credential/config boundary clearly

## Acceptance Criteria

- [ ] `jira.yaml` contains zero credential references — only project-level config
- [ ] Credentials documented as env vars: `JIRA_URL`, `JIRA_USERNAME`, `JIRA_API_TOKEN`
- [ ] Personal Jira overrides supported via `.raise/personal/jira.yaml` (gitignored)
- [ ] `rai init` creates `.raise/personal/` stub with jira.yaml template
- [ ] Existing `rai backlog` commands work unchanged after separation

## Non-Goals

- Changing the MCP bridge or adapter architecture
- Supporting multiple Jira instances per project
- OAuth or other auth methods beyond API token
