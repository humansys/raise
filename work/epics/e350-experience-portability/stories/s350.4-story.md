# S350.4: Pattern Destillation & 3-Level Model

## User Story (Connextra)

**As a** developer onboarding to a RaiSE project,
**I want** patterns organized in three levels (base/project/personal) with base patterns distilled from first principles,
**So that** I get a consistent Rai experience from base patterns, inherit project-specific patterns, and keep my personal patterns private.

## Context

The current pattern store has ~626 patterns accumulated over months in a single flat JSONL file. Many are duplicates, session-specific, or have been superseded by newer patterns. ADR-044 D3 defines a three-level model:

- **Base** (~50): First-principles patterns that ship with the `rai-cli` package. Stable, universal, version-controlled in source.
- **Project**: Patterns specific to a project, stored in `.raise/rai/memory/patterns.jsonl` (git-tracked).
- **Personal**: Per-developer patterns, stored in `.raise/rai/memory/personal/patterns.jsonl` (git-ignored).

This story distills the existing ~626 patterns into ~50 base first-principles, implements the three-level separation, and adjusts `rai pattern add` and `rai pattern promote` to respect the model.

## Acceptance Criteria (Gherkin)

### AC1: Base patterns ship with package
```gherkin
Given rai-cli is installed
When patterns are loaded at startup
Then base patterns from the package are available
And they are read-only (not modified by rai pattern add)
```

### AC2: Pattern add defaults to personal
```gherkin
Given a project with the three-level pattern model
When I run `rai pattern add "new pattern"`
Then the pattern is added to personal patterns
And it does not appear in project patterns
```

### AC3: Pattern promote moves personal to project
```gherkin
Given a personal pattern exists
When I run `rai pattern promote PAT-ID`
Then the pattern moves from personal to project level
And it is removed from personal patterns
And it is added to project patterns (git-tracked)
```

### AC4: Pattern loading merges all levels
```gherkin
Given base, project, and personal patterns exist
When patterns are loaded for context
Then all three levels are merged
And conflicts are resolved by specificity (personal > project > base)
```

### AC5: Base patterns distilled
```gherkin
Given the current ~626 patterns
When destillation is complete
Then ~50 base first-principles patterns exist in the package
And they cover the core behavioral, process, and technical domains
And each has a clear, non-overlapping scope
```
