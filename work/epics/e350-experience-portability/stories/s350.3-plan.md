# S350.3: MEMORY.md + CLAUDE.md Regeneration — Plan

## Design Decisions

1. **MEMORY.md** regenerates via hook on `graph:build` event
2. **CLAUDE.md** regenerates only in `rai init` (NOT on graph build)
3. **InstructionsGenerator** gets extended to include Rai-specific sections when `.raise/` exists
4. **One-time migration**: move hand-edited CLAUDE.md content to `.raise/` sources, then regenerate
5. **CLAUDE.local.md**: git-ignored file for per-developer overrides (AC5)

## Task Breakdown

### T1: MemoryHook — builtin hook for MEMORY.md regeneration
**File:** `src/rai_cli/hooks/builtin/memory_md_sync.py`
**Est:** 1h

Create a `MemoryMdSyncHook` class following the `BacklogHook` pattern:
- Implements the hook protocol with `events = ["graph:build"]`
- On `GraphBuildEvent`, calls `generate_memory_md()` from `rai_cli.onboarding.memory_md`
- Reads project name from `.raise/manifest.yaml`
- Reads development branch from manifest
- Writes to canonical path (`.raise/rai/memory/MEMORY.md`)
- Writes Claude Code copy (`get_claude_memory_path()`)
- Never raises — returns `HookResult` with status and message
- Register entry point in `pyproject.toml` under `[project.entry-points."rai.hooks"]`

**TDD tests** (`tests/hooks/test_memory_md_sync.py`):
- Hook subscribes to `graph:build` event
- Hook fires on `GraphBuildEvent` and writes both files
- Hook returns `HookResult(status="ok")` on success
- Hook returns `HookResult(status="error")` when manifest missing
- Hook is a no-op for non-GraphBuildEvent

**Depends on:** nothing

### T2: Extend InstructionsGenerator for Rai sections
**File:** `src/rai_cli/onboarding/instructions.py`
**Est:** 2h

Add a new code path in `InstructionsGenerator.generate()`:
- Detect `.raise/` directory existence at `project_path` (new param or passed flag)
- When `.raise/` exists, generate CLAUDE.md with structured sections:
  - **Rai Identity** — from `.raise/rai/identity/core.md` (read and include)
  - **Process Rules** — from `.raise/rai/framework/methodology.yaml`:
    - Work Lifecycle (epic/story/session flows)
    - Gates (blocking gates table)
    - Critical Rules (principles as numbered list)
  - **Branch Model** — from `.raise/manifest.yaml` (`branches` section)
  - **CLI Quick Reference** — from methodology.yaml `skills` section (compact table)
  - **External Integrations** — from `.raise/jira.yaml` (if exists): project key, workflow info
  - **File Operations** — standard instructions for file reading
  - **Post-Compaction** — standard instructions for context restoration
- Keep existing brownfield/greenfield logic as fallback for non-RaiSE projects
- Add `project_path: Path | None = None` parameter to `generate()`

**Implementation approach:**
- New method `_generate_raise_project()` parallel to `_generate_brownfield()` / `_generate_greenfield()`
- Reuse `MemoryMdGenerator` loaders for methodology.yaml parsing (DRY)
- Each section is a private method: `_add_rai_identity()`, `_add_rai_process_rules()`, etc.

**TDD tests** (`tests/onboarding/test_instructions_raise.py`):
- When `.raise/` exists with identity + methodology + manifest, output contains all sections
- When `.raise/` missing, falls back to existing brownfield/greenfield
- Each section renders correctly from source files
- Missing optional files (jira.yaml) produce no section, no error

**Depends on:** nothing

### T3: Wire CLAUDE.md regeneration into `rai init`
**File:** `src/rai_cli/cli/commands/init.py`
**Est:** 1h

Modify `init_command()` to always regenerate CLAUDE.md:
- After bootstrap and scaffolding, call `InstructionsGenerator.generate()` with `project_path`
- Write to `project_path / first_config.instructions_file` (typically `CLAUDE.md`)
- This replaces the current `--detect`-only CLAUDE.md generation
- The `--detect` path still runs convention detection + guardrails on top
- Add `.raise/` existence check: if `.raise/` exists, use Rai generator path; otherwise existing logic
- Show console message indicating CLAUDE.md was regenerated

**TDD tests** (`tests/cli/test_init_claudemd.py`):
- `rai init` on a project with `.raise/` produces CLAUDE.md with Rai sections
- `rai init` on a project without `.raise/` produces existing-style CLAUDE.md
- `rai init` overwrites existing hand-edited CLAUDE.md
- Warning is shown when overwriting (AC3)

**Depends on:** T2

### T4: CLAUDE.local.md + one-time CLAUDE.md migration
**File:** multiple (`.gitignore`, docs)
**Est:** 1h

Part A — CLAUDE.local.md support:
- Add `CLAUDE.local.md` to project `.gitignore`
- Document in generated CLAUDE.md footer that overrides go in `CLAUDE.local.md`
- Verify Claude Code loads both files (it does by convention)

Part B — One-time migration:
- Review current hand-edited `CLAUDE.md` content
- Identify content not covered by `.raise/` sources
- Move missing content to appropriate `.raise/` source files:
  - CLI reference details → methodology.yaml or new section
  - Common mistakes → methodology.yaml
  - File operations note → template in generator
  - Post-compaction protocol → template in generator
- Regenerate CLAUDE.md from sources
- Diff to verify nothing was lost

**TDD tests:**
- `.gitignore` contains `CLAUDE.local.md` entry
- Generated CLAUDE.md mentions `CLAUDE.local.md` as override mechanism

**Depends on:** T3

### T5: Integration verification
**Est:** 0.5h

End-to-end verification:
- Run `rai graph build` in project → verify MEMORY.md regenerated at both paths
- Run `rai init` → verify CLAUDE.md regenerated with all Rai sections
- Run full test suite: `uv run pytest`
- Run type check: `uv run mypy src/`
- Run lint: `uv run ruff check src/`
- Verify no regressions in existing hook tests

**Depends on:** T1, T2, T3, T4

## Risk Notes

- **CLAUDE.md migration** (T4) is the trickiest task — must not lose any content from the current hand-edited file. Diff verification is the gate.
- **InstructionsGenerator** (T2) is the largest task — methodology.yaml parsing is complex but can reuse `MemoryMdGenerator` loaders.
- **Hook registration** (T1) is straightforward — follow BacklogHook pattern exactly.
