# S350.1 Design: Inventory & Classify Artifacts

**Story:** S350.1 | **Epic:** E350 | **Complexity:** Simple
**Skill:** rai-story-design | **Created:** 2026-03-04

## Summary

Systematic audit and classification of every artifact that constitutes the Rai experience. Produces a classification document mapping each file/directory to one of four portability layers (base, project, personal, derived) with distribution mechanism and mixed-concern flags. Foundation for all other E350 stories.

## Approach

Walk the actual filesystem to inventory every file and directory that affects Rai behavior. The relevant trees are:

- `.raise/` -- project-level Rai configuration
- `.raise/rai/` -- Rai runtime state (identity, memory, framework, relationships, sync, telemetry, personal)
- `.claude/` -- IDE integration (settings, skills, scripts, legacy docs)
- `src/rai_cli/skills_base/` -- builtin skills (ship with package)
- `CLAUDE.md` -- project instructions (root)
- `.gitignore` -- controls distribution boundaries

No code changes. Output is a classification document.

## Layer Definitions

| Layer | Scope | Distribution | Boundary |
|-------|-------|-------------|----------|
| **Base** | Ships with `rai-cli` package, identical for all instances | PyPI package / `src/rai_cli/` | Defines Rai's universal identity, methodology, and capabilities |
| **Project** | Shared across team members working on the same repo | Git-tracked in `.raise/` or root | Project-specific config, patterns, integrations, backlog |
| **Personal** | Per-developer, never shared | `.gitignore`'d under `.raise/rai/personal/` | Sessions, calibration, personal telemetry |
| **Derived** | Generated from other sources, should be regenerable | Currently git-tracked (target: regenerated) | Graph artifacts, MEMORY.md, CLAUDE.md, skill projections |

## Acceptance Criteria

| ID | Criterion | Verifiable |
|----|-----------|:----------:|
| AC-1 | Every file/directory affecting Rai behavior is inventoried (verified by glob/grep audit) | Yes |
| AC-2 | Each artifact classified as exactly one of: base, project, personal, derived | Yes |
| AC-3 | At least one artifact in each layer; layer definitions documented with clear boundaries | Yes |
| AC-4 | Mixed-concern artifacts flagged with resolution recommendation | Yes |
| AC-5 | Classification document exists at expected path | Yes |
| AC-6 | No code changes -- documentation only | Yes |

## Decisions

| ID | Choice | Rationale |
|----|--------|-----------|
| D-1 | Four layers: base, project, personal, derived | Matches ADR-044 architecture. "Derived" layer distinguishes regenerable artifacts from source-of-truth. |
| D-2 | Classification document is markdown tables | Human readability is primary goal; tables are scannable and diff-friendly. |
| D-3 | Mixed-concern flags inline in table (Issues column) | Keeps flag co-located with artifact for context proximity. |
