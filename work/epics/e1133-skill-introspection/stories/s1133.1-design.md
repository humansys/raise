# S1133.1 Design: Introspection Aspect Definition

## Problem & Value

**Problem:** Lifecycle skills operate in cognitive isolation — no graph consultation before generating, no learning capture after, no measurement of effectiveness. Only story-review evaluates, too late for granular attribution.

**Value:** A shared introspection aspect makes learning invisible infrastructure. Skills just work; PRIME/JIT/LEARN happen around them. Learning records flow through the chain, enabling measurement and future self-improvement.

## Complexity: Moderate

One artifact (`aspects/introspection.md`), but non-trivial design decisions resolved through interactive exploration:

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Aspect model | Wrapper (stepping stone) | Learning is cognitive infrastructure, not skill responsibility |
| Silent nodes | Yes — mechanical skills skip records | Absence is signal. Eliminate waste (Poppendieck) |
| output_quality | Removed from schema | LEARN captures what it knows. Acceptance implicit in chain |
| Pattern capture | Gaps only in LEARN, capture in story-review | Don't capture speculative. Amplify the real |

## Approach

Write `aspects/introspection.md` as a **wrapper protocol** — it defines what happens before (PRIME) and after (LEARN) the skill body, plus what can happen during (JIT).

The aspect is designed for **stepping stone** integration: today, skills reference it via markers (`> **PRIME**`, `> **LEARN**`). Tomorrow, rai-agent composes it automatically.

### Components

| Component | Type | Location |
|-----------|------|----------|
| Introspection aspect | create | `aspects/introspection.md` |

The aspect file contains:
1. PRIME protocol (graph context loading, chain reads)
2. JIT protocol (decision bifurcation heuristic)
3. LEARN protocol (learning record production, downstream enrichment)
4. Learning record YAML schema
5. Chain mapping (which skill reads/enriches which)
6. Phase metadata specification
7. Evaluation semantics (recall failure vs use failure)

### Skill integration pattern (for S1133.3/S1133.4)

Skills declare participation via frontmatter metadata:
```yaml
metadata:
  raise.aspects: introspection
  raise.introspection:
    phase: story.design
    context_source: scope doc
    affected_modules: [mod-backlog, mod-graph]
    tier1_queries:
      - "patterns for {affected_modules} design decisions"
```

And add three markers in the body (no protocol reimplementation):
```markdown
> **PRIME**: Before Step 1, follow PRIME protocol in `aspects/introspection.md`

[... skill steps unchanged ...]

> **JIT**: [at decision bifurcation steps only]

> **LEARN**: After completing, follow LEARN protocol in `aspects/introspection.md`
```

## Examples

### Learning record schema (simplified)

```yaml
# .raise/rai/learnings/rai-story-design/S1133.1/record.yaml
skill: rai-story-design
version: "2.4.0"
work_id: S1133.1
timestamp: 2026-04-01T11:30:00

process_fidelity:
  tier1_primed: true
  tier1_queries:
    - query: "patterns for cross-cutting concerns in skill architecture"
      types: [approach, risk]
      results: 1
      useful: true
    - query: "prior story designs for docs/protocol stories"
      results: 0
  jit_queries:
    - query: "patterns for aspect-oriented design, wrapper vs reference"
      trigger: "Step 3 — two valid approaches"
      results: 0
      gap: true
  artifacts_produced: [s1133.1-design.md, s1133.1-design.yaml]

pattern_metrics:
  primed:
    - id: PAT-E-590
      vote: 1
      why: "hook extension pattern informed stepping stone design"
  gaps:
    - "No patterns for aspect-oriented composition in skill systems"

downstream_impact:
  next_skill: rai-story-plan
  plan_derivable: null       # filled by story-plan
  tasks_clear: null          # filled by story-plan
```

### Chain mapping

```
epic-design → [record] → epic-plan reads: "design gaps? sizing?"
                          → [record] → epic-close reads: "plan held?"

story-design → [record] → story-plan reads: "plan derivable? tasks clear?"
                           → [record] → story-implement reads: "tasks executable?"
                                         → enriches story-design: design_gaps_found
                                         → [record] → story-review reads all
```

Silent nodes (no record): story-start, story-close, epic-start.

### Downstream enrichment example

story-plan reads story-design's record and fills:
```yaml
downstream_impact:
  next_skill: rai-story-plan
  plan_derivable: true
  tasks_clear: true
```

## Acceptance Criteria

### MUST
1. `aspects/introspection.md` defines PRIME/JIT/LEARN as complete, self-contained protocol
2. Learning record schema has exactly 3 sections: `process_fidelity`, `pattern_metrics`, `downstream_impact`
3. Chain mapping documents what each skill reads from previous and what it enriches
4. JIT heuristic has concrete trigger and constraints (max queries, time budget)
5. LEARN protocol explicitly lists what it does NOT do (no pattern capture, no output quality eval)

### SHOULD
1. A skill author can read the aspect and understand the full protocol in <5 minutes
2. Guided query templates are generic enough to cover all phases with variables

### MUST NOT
1. Not reimplement protocol logic inside skills (markers + metadata only)
2. Not include `output_quality` in schema (acceptance implicit in chain)
3. Not capture candidate patterns in LEARN (only gaps; capture is story-review's job)
