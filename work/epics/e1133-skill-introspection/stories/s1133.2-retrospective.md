## What Went Well

1. **Interactive design caught a semantic mismatch early.** The initial aspect (S1133.1) used `--types approach,risk` in query templates, but `approach`/`risk` are *pattern subtypes*, not *node types*. The design conversation surfaced this: node types (`pattern`, `module`, `test`) are structural, subtypes (`approach`, `risk`, `process`) are semantic within a type. Without catching this, S1133.3 and S1133.4 would have built on a broken query interface.

2. **D4 eliminated a redundant concept.** The story originally scoped "story outcome metadata" as a separate schema. Design conversation revealed the learning record already captures everything a story outcome would: pattern votes, gaps, artifacts, downstream pointers. "The record IS the outcome" collapsed two schemas into one, removing an entire component.

3. **Pre-implement AR deferred scope creep.** AR flagged `enrich_downstream()` as premature — no consumers exist in this story. Deferring it to S1133.3/S1133.4 kept the implementation focused. Cost: ~10 min review. Savings: prevented building and testing dead code.

4. **QR found real improvements post-implement.** 4 recommendations, 3 applied (R1: warning log in `read_record` exception handler, R3: neighbor subtype filtering test, R4: Pydantic model equality instead of field-by-field comparison in round-trip test). R2 (accept `downstream` as intentionally open) was already documented via AR-R2.

5. **TDD kept the implementation clean.** RED-GREEN-REFACTOR on the subtypes filter and LearningRecord I/O meant every function had tests before code, and the test names served as the specification.

## What to Improve

1. **Design conversation took longer than expected for an S story.** The semantic mismatch (types vs subtypes) required back-and-forth to resolve. Next time, for stories that extend an existing taxonomy, explicitly list "what are the existing categories and what am I adding" in the scope doc to front-load this analysis.

2. **AR question resolution could be inline.** AR raised 4 questions (Q1-Q4) that were all resolved in the same session. For small stories where designer and reviewer are the same agent, consider a combined design+AR pass rather than two separate artifacts.

3. **Aspect update (F3) was documentation-only but coupled to code changes.** The query template fix (`--types approach,risk` to `--types pattern --subtypes approach,risk`) was a direct consequence of D1. Consider adding a checklist item: "update all references to modified interfaces" as part of the implementation task, not a separate doc task.

## Heutagogical Checkpoint

### What did you learn?
The distinction between *structural types* (node types in the graph: pattern, module, test, edge) and *semantic subtypes* (within a type: approach, risk, process) is fundamental to query correctness. Conflating them creates queries that either return nothing or return everything. This distinction was not explicit in the graph documentation before this story.

### What would you change about the process?
For infrastructure stories that extend a taxonomy, add a "taxonomy impact analysis" to the design template: list current values, proposed additions, and all query/filter points that reference them. This would have made the types-vs-subtypes mismatch visible before the conversation.

### Are there improvements for the framework?
The `--subtypes` filter pattern (field on Query model + filter in both `_keyword_search` and `_concept_lookup`) is a clean extension point. If we need `--tags` or `--scope` filters later, the same pattern applies. Consider documenting this as a graph query extension recipe.

### What are you more capable of now?
Better at identifying when a "new concept" in a story scope is actually an existing concept under a different name. D4 (learning record IS the outcome) is a general pattern: before adding a schema, check if an existing schema already captures the same information.
