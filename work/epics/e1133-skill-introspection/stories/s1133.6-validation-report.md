# S1133.6 Validation Report: Introspection Dogfood

## Executive Summary

The introspection system (PRIME/JIT/LEARN) was deployed across all 7 cognitive lifecycle skills and exercised in 3 épicas (E1133, E1134, E1064). The system **works when exercised** but **adoption is unreliable** — agents skip the protocol in most stories.

**Verdict:** System is architecturally sound. Adoption and persistence need hardening (→ E1136).

## Findings

### F1: Adoption Gap (Critical)

**Evidence:** E1134 ran 4 complete story cycles with introspection-enabled skills. Zero LEARN records produced. Retrospectives mention "No PRIME/JIT records — simple story" and "not generated (story too simple for PRIME/JIT)".

**Root cause:** PRIME/LEARN markers use suggestive language (`> **PRIME**: Before Step 1, follow PRIME protocol...`). The agent interprets this as optional and skips it for "simple" stories. But learning records should be produced for ALL cognitive skills — simplicity is not an exemption.

**Fix:** E1136 S1136.1 — Make markers mandatory instructions, not suggestions.

### F2: Schema Drift (Moderate)

**Evidence:** S1064.8 produced a record with nested schema (`prime:`, `jit:`, `learn:` sections with sub-fields) instead of the flat schema defined in `aspects/introspection.md` (~10 top-level fields).

**Root cause:** The agent doesn't read the aspect at LEARN time — it improvises the schema from memory of what a "learning record" should look like. The Pydantic model exists in code but there's no CLI command that validates.

**Fix:** E1136 S1136.2 — `rai learn write` CLI command that validates against the Pydantic model.

### F3: Records Don't Persist (Moderate)

**Evidence:** Records exist in worktree local disk but aren't committed automatically. Only the S1133.5 records (manually `git add -f`'d) are in the repo. Worktree-local records are lost on worktree cleanup.

**Root cause:** No automation for committing learnings. The path `.raise/rai/learnings/` is not gitignored but nothing adds it to staging.

**Fix:** E1136 S1136.3 — Auto-commit records with task commits. E1136 S1136.4 — Server persistence.

### F4: Infrastructure Bug (Resolved)

**Evidence:** `rai graph build` failed with duplicate epic IDs (e14, e3, e1001), completely blocking PRIME execution.

**Root cause:** Epic ID generation normalizes via `int()`, collapsing `e03`→`E3` and `e3`→`E3`.

**Fix:** Data fix applied in S1133.5 (3 epics renumbered). Prevention ticket RAISE-1199 created.

## What Worked

1. **PRIME queries execute correctly** when the agent follows the protocol — template substitution, graph query, results surfaced
2. **LEARN record format is simple enough** to write manually — flat YAML, ~10 fields
3. **Chain enrichment works** — story-plan enriched story-design's downstream fields
4. **JIT heuristic reframe** ("grounded decision" vs "bifurcation") is the right model — the agent queried naturally at decision points
5. **LearningRecord Pydantic model** validates correctly — `write_record()` and `read_record()` work as designed
6. **`--subtypes` query filter** works in the engine — just needs patterns tagged with `approach`/`risk`
7. **Missing records are handled gracefully** — story-review noted missing implement record (docs story) and continued

## What Needs Work (→ E1136)

1. **Make protocol non-skippable** — the agent needs to produce a record even for simple stories
2. **Provide CLI validation** — `rai learn write` to prevent schema improvisation
3. **Auto-persist records** — commit with task, push to server
4. **Populate graph** — acceptance rate improves as graph gets richer with `approach`/`risk` patterns
5. **Refine PRIME queries** — current templates return generic results, need domain-specific tuning after more data

## Conclusion

E1133 achieved its architectural goal: the introspection system (PRIME/JIT/LEARN, learning records, chain mapping, downstream enrichment) is designed, built, integrated into all 7 cognitive skills, and validated end-to-end. The system is sound.

The gap is operational: agents don't reliably exercise the protocol. This is a fixable adoption problem (clearer instructions, CLI tooling, auto-persistence), not an architectural one. E1136 addresses these fixes.

**E1133 is closeable.** The remaining work is E1136 (hardening) which builds on E1133's foundation.
