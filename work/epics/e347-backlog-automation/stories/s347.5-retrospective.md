## What Went Well

1. **Design-first paid off.** The design doc specified the exact data model,
   timeout strategy (ThreadPoolExecutor), and degraded paths before any code
   was written. Implementation followed the blueprint with minimal deviation.

2. **Sequential T1-T2-T3 decomposition worked cleanly.** T1 (model + stub) was
   trivial, T2 (fetch logic with timeout) was the risk-first task, T3 (integration)
   wired everything together. Each task had clear boundaries and testable output.

3. **Graceful degradation by design.** All three failure modes (timeout,
   SystemExit from resolve_adapter, generic Exception) were planned in the
   design and tested individually. The `_fetch_live_status()` function genuinely
   never raises.

4. **Quality review caught real issues.** Three fixes from QR:
   - Typing: `ProjectManagementAdapter` annotation on `resolve_adapter` return
   - Timeout boundary: moved `resolve_adapter` inside the thread so the 5s
     timeout covers adapter startup (MCP bridge init), not just `get_issue`
   - `SystemExit` catch: once `resolve_adapter` moved inside the thread,
     `SystemExit` (a BaseException) needed explicit handling since it escapes
     `except Exception`

## What to Improve

1. **The QR insight about SystemExit was non-obvious.** Moving `resolve_adapter`
   inside the thread was the correct design (per the design doc's timeout strategy),
   but the initial implementation had it outside. The QR process caught this, but
   ideally the plan should have been more explicit about "the entire operation
   runs inside the thread" from the start.

2. **Timeout test uses `time.sleep()`.** The timeout test relies on `time.sleep(6)`
   to simulate a slow adapter. This works but adds 6 real seconds to the test
   suite. A more sophisticated approach (mocking the ThreadPoolExecutor or using
   a shorter timeout in tests) would be faster. Acceptable for now given the 5s
   timeout is the production value being tested.

## Heutagogical Checkpoint

### 1. What did you learn?

- `SystemExit` is a **BaseException**, not an **Exception**. When code that
  calls `sys.exit()` runs inside a `ThreadPoolExecutor` thread, the `SystemExit`
  propagates through `future.result()` and escapes `except Exception` blocks.
  Must catch it explicitly.

- The timeout boundary decision (what goes inside vs. outside the thread) has
  cascading implications for exception handling. Moving `resolve_adapter` inside
  the thread was correct (design said "wrap the entire fetch") but changed which
  exceptions could surface.

### 2. What would you change about the process?

- The plan should have been more explicit about the "what runs inside the thread"
  boundary. A simple diagram showing "thread boundary: resolve_adapter + get_issue"
  would have prevented the initial implementation from putting resolve_adapter
  outside.

### 3. Are there improvements for the framework?

- Pattern PAT-E-621 captures the ThreadPoolExecutor + SystemExit interaction.
  This is reusable anywhere we wrap CLI-style code (that may `sys.exit`) in
  concurrent futures.

- Pattern PAT-E-622 captures the graceful degradation pattern (catch Exception
  AND BaseException subtypes separately, return degraded result with warning).

### 4. What are you more capable of now?

- Better intuition for Python's exception hierarchy in concurrent contexts.
  BaseException vs Exception is a distinction that matters when wrapping
  arbitrary code in threads/futures.

- Practical experience with the "never-fail function" pattern: design the
  function contract (never raises, always returns), then systematically
  enumerate all exception paths and handle each one.
