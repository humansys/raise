## What Went Well

1. **Design pivot was the right call.** The original story spec targeted markdown parsing (regex on scope docs, comment sections in story files, dependency columns). Human redirected to YAML file store during design phase. This eliminated all the fragile regex parsing and produced a much cleaner, more testable architecture.

2. **TDD kept each task tight.** Every task followed RED-GREEN-REFACTOR strictly. The 7-task plan mapped cleanly to test classes: `TestBacklogModels`, `TestYamlStoreGetIssue`, `TestYamlStoreSearch`, `TestYamlStoreCreate`, `TestYamlStoreTransition`/`Update`/`Batch`, `TestYamlStoreComments`/`Links`, `TestProtocolCompliance`/`TestIntegration`. No test was added without a failing RED phase first.

3. **Risk-first sequencing paid off.** T2 (YAML I/O core) was the riskiest task — if Pydantic YAML round-tripping failed, nothing downstream would work. Placing it second (after T1 models) meant we validated the core assumption early. Everything after T2 was mechanical load-modify-save.

4. **Architecture review + Quality review caught real issues.** AR flagged a `setattr` concern (Q2: what if someone passes `key` in update_issue fields?). QR recommended the `_IMMUTABLE_FIELDS` guard. Both were applied before story close.

5. **Legacy fallback preserved backward compatibility.** The `_use_yaml` property cleanly separates new YAML path from legacy markdown path. Existing users with `governance/backlog.md` but no `.raise/backlog/items/` directory continue working unchanged.

## What To Improve

1. **Story spec diverged significantly from implementation.** The original story doc describes markdown parsing scenarios (scope docs, comment sections in story files). The design doc describes YAML store. The story doc should have been updated after the pivot to match the actual design. This creates confusion for anyone reading the story artifact trail.

2. **Estimated vs actual timing not tracked.** The plan has a duration tracking table with empty "Actual" columns. Should have filled these in during implementation to feed calibration data. This is a recurring gap.

3. **Legacy code is still large.** The legacy markdown helpers (`_find_table_end`, `_format_row`, `_parse_row`, `_extract_scope_priority`, etc.) are 80+ lines of code that only serve the fallback path. Should be cleaned up in a future story once YAML store is validated in production use.

## Heutagogical Checkpoint

### 1. What did you learn?

- **YAML + Pydantic is a powerful local storage pattern.** `model_validate(yaml.safe_load(...))` for read, `yaml.safe_dump(model.model_dump(...))` for write. Clean round-tripping with validation on every load. The pattern is reusable for any `.raise/` configuration that needs CRUD semantics.

- **Feature flags via properties work well for incremental migration.** `@property _use_yaml -> bool` checking if the YAML directory exists means the adapter auto-selects the right code path. No configuration needed, no env vars, just directory presence.

- **`exclude_none=True` + cleaning empty collections produces readable YAML.** Without this, YAML files would have `parent: null` and `comments: []` everywhere. The cleanup in `_save_item` keeps files minimal.

### 2. What would you change about the process?

- **Update the story doc after a design pivot.** The story spec became misleading after the pivot to YAML. In future pivots, update the story acceptance criteria to match the actual design.

- **Track task durations in the plan.** The plan template has columns for estimated/actual — fill them in during implementation for calibration telemetry.

### 3. Are there improvements for the framework?

- **Skill improvement:** The `/rai-story-design` skill should include a step that says "if the design diverges from the story spec, update the story spec to match." This would prevent the story-spec-drift issue.

- **Pattern reuse:** The YAML file store pattern (PAT-E-616) could become a reusable recipe/template for any adapter that needs local-first storage.

### 4. What are you more capable of now?

- Building full CRUD adapters backed by YAML file stores with Pydantic validation.
- Designing incremental migrations with legacy fallback paths.
- Key generation patterns (scan existing files, extract max N, increment).

## Technical Decisions

| Decision | Rationale |
|----------|-----------|
| D1: YAML file store over markdown parsing | Human feedback: YAML is reliable, AI-first storage. Markdown parsing is fragile regex. |
| D2: One file per issue at `.raise/backlog/items/{KEY}.yaml` | Consistent with `.raise/mcp/*.yaml` pattern. Simple file-level CRUD. |
| D3: Comments and links embedded in issue YAML | No separate files. Load-modify-save is atomic per issue. |
| D4: Legacy fallback via `_use_yaml` property | Backward compat. No migration needed — YAML store is opt-in by directory creation. |
| D5: `_IMMUTABLE_FIELDS` guard on update_issue | QR recommendation. Prevents `key`, `created`, `comments`, `links` from being overwritten via generic field update. |
