# Design: S347.5 — Session-start live backlog query

## Problem

`assemble_context_bundle()` shows epic/story status from `session-state.yaml`,
which is only updated at session-close. Between sessions, backlog state drifts
(transitions via skills, Jira UI, or other developers). The developer starts
each session with potentially stale work status.

## Value

Live status at session-start means Rai and the developer share the same truth
about where work stands — no manual "let me check Jira" interruptions.

## Approach

Add a `_fetch_live_status()` helper to `bundle.py` that:

1. Reads `current_work.epic` and `current_work.story` from `SessionState`
2. Resolves the PM adapter via `resolve_adapter(None)` (uses manifest default)
3. Calls `adapter.get_issue(key)` for each non-empty key
4. Returns a `LiveBacklogStatus` Pydantic model with the results
5. On any failure (adapter unavailable, timeout, KeyError), returns a degraded
   result with a warning message — **never blocks session-start**

Integration point: `_format_work_section()` receives the live status and
annotates the existing work lines with live data when available.

### Components Affected

| Component | Change |
|-----------|--------|
| `src/rai_cli/session/bundle.py` | Modify — add `_fetch_live_status()`, `LiveBacklogStatus` model, update `_format_work_section()` and `assemble_orientation()` |
| `tests/session/test_bundle.py` | Modify — add tests for live status (success, timeout, unavailable, no-work) |

### Why Not subprocess (`rai backlog get`)?

The adapter is importable in-process. Subprocess adds ~500ms overhead, error
parsing complexity, and loses type safety. Direct adapter call via
`resolve_adapter()` → `adapter.get_issue()` is simpler and faster.

### Timeout Strategy

Python `signal.alarm` is Unix-only and not thread-safe.
`concurrent.futures.ThreadPoolExecutor` with `timeout` is portable and clean:

```python
from concurrent.futures import ThreadPoolExecutor, TimeoutError

with ThreadPoolExecutor(max_workers=1) as pool:
    future = pool.submit(adapter.get_issue, key)
    result = future.result(timeout=5.0)
```

If `resolve_adapter()` itself hangs (MCP server unreachable), wrap the entire
fetch (resolve + get_issue calls) in the single timeout boundary.

## Examples

### Data Structure

```python
class LiveBacklogStatus(BaseModel):
    """Live status fetched from backlog adapter during session-start."""

    epic_status: str = ""       # e.g., "in_progress", "done"
    epic_summary: str = ""      # e.g., "Backlog Automation"
    story_status: str = ""      # e.g., "in_progress"
    story_summary: str = ""     # e.g., "Session-start live query"
    warning: str = ""           # non-empty on degraded path
```

### Success Path — bundle output

```
Story: S347.5 [implement] — in_progress (live)
Epic: E347 — in_progress (live)
Branch: story/s347.5/session-start-live-query
```

The ` — in_progress (live)` suffix is appended only when live status differs
from or enriches the cached session-state.

### Degraded Path — adapter unavailable

```
Story: S347.5 [implement]
Epic: E347
Branch: story/s347.5/session-start-live-query
⚠ Backlog adapter unavailable — showing cached state
```

### No Current Work — no query attempted

When `state.current_work.epic` and `state.current_work.story` are both empty,
`_fetch_live_status()` returns `LiveBacklogStatus()` (all empty) immediately.
No adapter resolution or network call occurs.

### API Usage (internal)

```python
# In assemble_orientation():
live = _fetch_live_status(state, timeout=5.0)
sections.append(_format_work_section(state, release_node=release_node, live=live))
```

## Acceptance Criteria

### MUST

1. When adapter is available and current_work has epic/story keys,
   `assemble_context_bundle()` output includes live status from adapter
2. When adapter is unavailable (import error, MCP timeout, no adapter
   registered), bundle falls back to session-state values with a warning line
3. Timeout at 5 seconds — slow adapter does not block session-start
4. When `current_work.epic` and `current_work.story` are both empty,
   no adapter resolution or query is attempted
5. All 4 test scenarios covered: success, timeout, unavailable, no-work

### SHOULD

1. Live status annotation is visually distinct (e.g., `(live)` suffix)
2. Warning message names the failure reason (timeout vs. no adapter vs. error)

### MUST NOT

1. Must not change SessionState schema
2. Must not cache live results back to session-state.yaml
3. Must not retry on failure — single attempt, then fallback
4. Must not call adapter when there is no current work (avoid unnecessary MCP startup)
