# Plan: S347.7 Integration Tests + Dogfood

## Overview

Test-only story. Single test module (`tests/integration/test_backlog_integration.py`) with shared fixtures in `conftest.py`. No production code changes.

**Risk-first ordering:** Protocol parity (T1) first — if FileAdapter has bugs under real I/O, everything else fails. Hook flow (T2) depends on adapter working. Session query (T3) and sync guard (T4) are independent of each other but depend on T1 establishing that FileAdapter works. Final validation (T5) is always last.

## Tasks

## Execution Order

```
T1 (Protocol parity)
├── T2 (Hook→Adapter flow)  ← depends on T1
├── T3 (Session live query)  ← depends on T1
├── T4 (Sync guard)          ← depends on T1
└── T5 (Full validation)     ← depends on T1-T4
```

**Rationale:** T1 is risk-first — establishes that FileAdapter works under real I/O. T2/T3/T4 can theoretically run in parallel but will be implemented sequentially (single file, one committer). T5 is the integration gate.

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| FileAdapter bugs under real I/O | Low | T1 catches early; fix in FileAdapter if needed (counts as bug fix, not new production code) |
| Import path issues for `_fetch_live_status` (private function) | Medium | Check actual import path at Gemba; may need to test through public API instead |
| BacklogHook internal state assumptions | Low | Design examples are grounded in existing unit test patterns |
