# Story: BacklogHook Adapter-Aware

## User Story
As a developer using `rai signal emit-work`,
I want the BacklogHook to respect the adapter default from manifest.yaml and resolve work IDs by exact match,
so that lifecycle events route to the correct adapter without hardcoding "jira" and without fragile fuzzy JQL search.

## Acceptance Criteria

### Scenario: Adapter resolved from manifest default
```gherkin
Given manifest.yaml has backlog.adapter_default: jira
When a work:lifecycle event fires
Then BacklogHook resolves the adapter via resolve_adapter() without hardcoded "jira"
```

### Scenario: Adapter resolved with no manifest default (single adapter)
```gherkin
Given no backlog.adapter_default in manifest.yaml
And exactly one PM adapter is registered
When a work:lifecycle event fires
Then BacklogHook resolves the adapter via auto-detect
```

### Scenario: Exact key resolution via label
```gherkin
Given a story S347.4 has a Jira issue with label "rai:S347.4"
When BacklogHook searches for the issue
Then it uses exact label match (labels = "rai:S347.4") instead of fuzzy summary search
```

### Scenario: Fallback to summary search when label not found
```gherkin
Given a story S347.4 has a Jira issue without the rai: label
When BacklogHook searches by label and finds nothing
Then it falls back to summary search (summary ~ "S347.4")
And logs a warning recommending label adoption
```

### Scenario: Adapter unavailable
```gherkin
Given no adapter can be resolved
When a work:lifecycle event fires
Then BacklogHook returns HookResult(status="error") with clear message
And never raises an exception
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| event(story, S347.4, start), manifest default=jira | resolve_adapter(None) | Uses jira adapter (from manifest) |
| event(story, S347.4, start), no manifest default, 1 adapter | resolve_adapter(None) | Uses auto-detected adapter |
| work_id="S347.4", label="rai:S347.4" exists | search by label | Returns exact match |
| work_id="S347.4", no label, summary contains "S347.4" | search by summary | Returns match + warning |
| work_id="S347.4", no label, no summary match | search both | Returns None |

## Notes
- Current BacklogHook hardcodes `resolve_adapter("jira")` — must use `resolve_adapter(None)` to respect manifest default
- Current `_resolve_jira_key()` uses fuzzy JQL `summary ~ "S347.4"` — fragile, matches partial strings
- Design decision from design.md: exact match via `labels = "rai:S347.4"` with fallback to summary search
- Error isolation contract: `handle()` never raises — all adapter failures become `HookResult(status="error")`
- Dependency: S347.1 (adapter default in manifest) — already complete
