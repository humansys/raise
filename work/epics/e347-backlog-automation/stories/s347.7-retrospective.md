## What Went Well

- **Risk-first test ordering worked.** T1 (protocol parity, 6 tests) established that FileAdapter works under real I/O first. T2-T4 all built on that confidence — no time wasted debugging adapter bugs during hook/session/sync tests.
- **conftest.py extraction was clean.** Shared fixtures (`file_adapter`, `backlog_dir`, `jira_yaml_setup`, `write_yaml_item`) kept the 426-line test module focused on assertions, not setup boilerplate.
- **QR caught real issues.** Two concrete fixes: (1) `yaml.safe_load` instead of private `_load_item` for disk verification, (2) `write_yaml_item` as a fixture instead of importing from conftest module-level. Both improved encapsulation without changing behavior.
- **Dogfood review docstring.** The module docstring documenting E347 lifecycle evidence serves as executable documentation — anyone reading the test file understands what the epic built and how the tests prove it works.
- **Zero regressions.** 3502 tests pass, no skips added, no flaky tests.

## What to Improve

- **Private function access in tests.** `_fetch_live_status` is imported directly from `session.bundle` — this couples tests to internal API. The AR flagged this correctly. A public API (e.g., `query_live_status()`) would be more robust against refactoring. Acceptable for now since there's no public wrapper yet, but should be addressed if the function moves.
- **Patch path fragility.** Three different patch targets across the test module (`rai_cli.hooks.builtin.backlog.resolve_adapter`, `rai_cli.cli.commands._resolve.resolve_adapter`, `rai_cli.cli.commands._resolve.load_manifest`). If these modules are refactored, the tests break even though behavior hasn't changed. This is inherent to Python's import system but worth noting.
- **No parametric adapter coverage.** The design originally mentioned "both adapters, same assertions" but scope was narrowed to FileAdapter only (McpJiraAdapter requires live MCP, not CI-friendly). A future story could add a mock MCP bridge for parametric tests.

## Heutagogical Checkpoint

### 1. What did you learn?

- **yaml.safe_load for test verification is better than private API access.** When asserting persisted state, reading the YAML file directly is both more explicit and less coupled than calling `adapter._load_item()`. The raw YAML dict is the actual contract — whatever the adapter writes, the file must contain these fields.
- **conftest fixtures vs module-level helpers have different scoping rules.** A function defined in conftest is only available as a fixture (via injection), not as a callable import. The QR caught that `_write_yaml_item` was being imported as a module function — wrapping it in a fixture (`write_yaml_item`) makes the dependency explicit and pytest manages lifecycle.
- **Integration test categories map naturally to architectural boundaries.** The 4 categories (protocol parity, hook flow, session query, sync guard) each exercise a different cross-component boundary. This categorization made the test plan trivial to design and execute.

### 2. What would you change about the process?

- The AR and QR were productive — both caught issues that improved the code. However, both issues (private method access, conftest import) are patterns that could be part of a "testing hygiene" pre-check in the implementation skill. Adding a gate like "no private imports in integration tests" and "no module-level imports from conftest" would catch these before review.

### 3. Are there improvements for the framework?

- **Integration test template.** The conftest + categorized test classes pattern could become a scaffold: `rai test scaffold --type integration MODULE`. It would generate conftest.py with common fixtures and a test file with class stubs for each component boundary.
- The story-review skill continues to work well for test-only stories — the same structure (verify, reflect, patterns, document) applies even when there's no production code to analyze.

### 4. What are you more capable of now?

- Designing integration test suites that exercise cross-component boundaries without requiring external services. The key insight: patch only the resolution/discovery layer (resolve_adapter), use real implementations for everything else. This gives high confidence with minimal mocking.
- Better instinct for conftest organization — fixtures for setup, no module-level helpers that tests import directly.

## Epic-Level Reflection (E347 Backlog Automation)

This is the final story in E347. Looking back at the full 7-story arc:

### What the epic accomplished

E347 transformed `rai backlog` from a fragmented system (Jira, backlog.md, session-state as three disconnected worlds) into a unified adapter-based architecture:

| Story | What it built | Key decision |
|-------|--------------|--------------|
| S347.1 | Adapter default in manifest | Configuration over convention |
| S347.2 | FileAdapter full protocol parity | YAML store with atomic writes |
| S347.3 | Skills via `rai backlog` CLI | One path, auditable |
| S347.4 | BacklogHook adapter-aware | Exact key resolution, no fuzzy search |
| S347.5 | Session-start live query | Real state, not stale snapshot |
| S347.6 | `rai backlog sync` command | Dual-write with type guard |
| S347.7 | Integration tests | 13 tests proving components work together |

### Epic-level patterns

- **Walking skeleton + risk-first sequencing worked.** S347.1 (adapter default) was the walking skeleton — everything depended on it. S347.2 (FileAdapter parity) was risk-first — the markdown format for stories was the biggest uncertainty. This ordering eliminated blocking dependencies and let stories 3-6 proceed smoothly.
- **One source of truth is a design principle, not a feature.** The sync guard (S347.6), the adapter default (S347.1), and the session live query (S347.5) all enforce the same principle: there is one canonical state, and everything else is derived. This coherence emerged from the design, not from a checklist.
- **Test-only stories as capstones work.** S347.7 caught no adapter bugs (the unit tests were thorough), but proved that components compose correctly. The value isn't in finding bugs — it's in demonstrating that the architecture works as designed.
