# S1248.5 Plan: Session Doctor — Safe Operationalization

> **Story:** S1248.5
> **Size:** M (3-5 tasks)
> **Date:** 2026-04-03
> **Branch:** `story/s1248.5/session-doctor`
> **Scope:** `work/epics/e1248-git-first-session-state/s1248.5-scope.md`

## Overview

Replace silent gc() with an interactive Session Doctor that diagnoses issues, classifies them by risk, and only cleans what the user authorizes. Wire WorkstreamMonitor into session close for insights. Add standalone `rai session doctor` subcommand.

## Tasks

**Files:**
- CREATE `packages/raise-cli/src/raise_cli/session/doctor.py`
- CREATE `packages/raise-cli/tests/session/test_doctor.py`

**TDD Cycle:**

RED:
- `test_finding_model_frozen` — Finding is immutable Pydantic model
- `test_action_plan_model_frozen` — ActionPlan is immutable
- `test_diagnose_clean_state` — no findings when no issues
- `test_diagnose_zombie_no_content` — detects zombie pointer without narrative
- `test_diagnose_zombie_with_narrative` — detects zombie with narrative content, marks needs_consent
- `test_diagnose_stale_output` — detects stale session-output.yaml
- `test_diagnose_retention_exceeded` — detects dirs beyond limit
- `test_classify_separates_auto_and_consent` — correct split
- `test_execute_only_authorized` — skips needs_consent items without consent
- `test_execute_cleans_authorized` — cleans when consent given

GREEN:
- `SessionDoctor(project: Path)` with `_registry: LocalSessionRegistry`
- `diagnose() -> list[Finding]` — reuses gc() detection logic but NO side effects
- `classify(findings) -> ActionPlan` — splits by `safe_to_auto_clean`

- `Finding(BaseModel, frozen=True)` — category, severity, description, detail, safe_to_auto_clean, action
- `ActionPlan(BaseModel, frozen=True)` — auto_clean, needs_consent, info_only lists

REFACTOR: Extract detection logic from `registry.gc()` into shared helpers if duplication is excessive. Otherwise, doctor calls registry internals directly (they're in the same package).

## Execution Order

```
T1 (core) ──► T2 (start integration) ──► T5 (integration test)
          ├─► T3 (close insights)     ──┘
          └─► T4 (doctor subcommand)  ──┘
```

1. **T1** first — everything depends on SessionDoctor core
2. **T2, T3, T4** can run in sequence (same session, different areas)
3. **T5** last — validates everything together

Risk-first: T1 carries the most design risk (model + detection logic extraction). T2 is the riskiest integration (modifying session start flow).

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| gc() detection logic tightly coupled to side effects | Rework T1 | Read registry.py carefully — detection IS separable from cleanup |
| Session start flow already complex (C901) | Merge difficulty | Minimal changes: add doctor call before registration, keep it clean |
| WorkstreamMonitor git log unavailable in test | Flaky T3 | Mock subprocess, test formatting logic separately |
