# S3.6 Plan — Knowledge Query CLI

**Story:** S3.6 | **Size:** S | **Date:** 2026-03-21
**Design:** s3.6-design.md | **Tasks:** 4

## Data Flow Trace

```
CLI args (domain_name, query_str, format, limit)
  → load_domain(domain_dir) → (DomainManifest, GateConfig)
  → manifest.retrieval.adapter → _resolve_class() → DomainAdapter instance
  → manifest.retrieval.builder → _resolve_class() → GraphBuilder instance
  → config.node_dir → builder.build_from_directory(node_dir) → Graph
  → retrieve(graph, query_str, adapter, top_k=limit) → RetrievalResult
  → manifest.prompting → PromptingConfig(system_context, response_format)
  → format_query_result(result, prompting, format) → stdout
```

## Execution Order

```
T1 (models + domain) → T2 (formatters) → T3 (CLI wiring) → T4 (integration)
     ↓                      ↓                    ↓
  foundation           can parallel          needs T1+T2         capstone
```

T1 and T2 are independent and could run in parallel, but sequential is fine for S story.

**Rationale:** T1 is riskiest (dynamic import wiring) → first. T2 is pure formatting (low risk). T3 wires everything. T4 validates with real data.

| Risk | Mitigation |
|------|-----------|

| 447 nodes slow to build graph | Acceptable for S — optimization is future epic |
