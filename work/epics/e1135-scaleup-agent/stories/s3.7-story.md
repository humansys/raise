# S3.7 — CLI Gates for Extraction

## User Story

**As** Rai (the AI agent executing extraction skills),
**I want** deterministic CLI commands that validate extraction artifacts at each pipeline phase,
**So that** I can iterate until all gates pass before presenting results to the human — no invalid output escapes the pipeline.

## Acceptance Criteria

### Scenario: Validate extracted nodes against schema
```gherkin
Given a directory of extracted YAML node files
When I run `rai scaleup validate <dir>`
Then each file is validated against OntologyNode schema
And the output reports: valid count, invalid count, per-file errors
And exit code is 0 if all valid, 1 if any invalid
```

### Scenario: Reconcile cross-references
```gherkin
Given a directory of extracted YAML node files
When I run `rai scaleup reconcile <dir>`
Then phantom targets are reported (referenced but missing nodes)
And orphan nodes are reported (no relationships)
And cross-decision edges are reported
And exit code is 0 if no phantoms, 1 if phantoms exist
```

### Scenario: Check coverage against competency questions
```gherkin
Given extracted nodes and a competency questions YAML file
When I run `rai scaleup coverage <dir> --cqs <questions.yaml>`
Then coverage percentage is reported
And uncovered questions are listed
And exit code is 0 if ≥80% coverage, 1 otherwise
```

### Scenario: Chunk corpus into chapters
```gherkin
Given a markdown corpus file
When I run `rai scaleup chunk <corpus.md> --output <dir>`
Then individual chapter files are written to output dir
And a manifest.yaml with chapter metadata is produced
And exit code is 0
```

### Scenario: Build graph from node directory
```gherkin
Given a directory of valid YAML node files
When I run `rai scaleup graph <dir>`
Then a Graph is built and stats are printed (nodes, edges, by type)
And exit code is 0 if graph builds, 1 on errors
```

### Scenario: Generic interface — not ScaleUp-specific
```gherkin
Given any Pydantic model that validates YAML nodes
When the CLI tools receive (corpus_path, schema, output_dir) as args
Then the same validate/reconcile/coverage commands work
And no ScaleUp-specific logic leaks into the CLI layer
```

## SbE Examples

```bash
# Validate — all pass
$ rai scaleup validate .scaleup/knowledge/ontology/extracted/
Validated 68 nodes: 68 valid, 0 invalid
exit 0

# Validate — some fail
$ rai scaleup validate ./bad-nodes/
Validated 70 nodes: 65 valid, 5 invalid
  INVALID: node-foo.yaml — missing required field 'type'
  INVALID: node-bar.yaml — 'decision' must be one of people|strategy|execution|cash
exit 1

# Reconcile
$ rai scaleup reconcile .scaleup/knowledge/ontology/extracted/
Reconciled 68 nodes: 0 phantoms, 2 orphans, 3 cross-decision edges
exit 0

# Coverage
$ rai scaleup coverage .scaleup/knowledge/ontology/extracted/ --cqs competency_questions.yaml
Coverage: 21/23 questions (91.3%)
Uncovered: CQ-12 (What tools measure cash conversion?), CQ-19 (...)
exit 0

# Chunk
$ rai scaleup chunk scaling_up_llamaparse.md --output chapters/
Chunked into 14 chapters across 4 decision areas
Wrote manifest: chapters/manifest.yaml
exit 0

# Graph
$ rai scaleup graph .scaleup/knowledge/ontology/extracted/
Graph built: 68 nodes, 142 edges
  By type: decision=4, concept=22, tool=18, worksheet=14, metric=8, stage=2
exit 0
```
