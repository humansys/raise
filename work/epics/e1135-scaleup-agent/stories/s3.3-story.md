# Story: HITL Curation Skill

## User Story
As Eduardo (Scaling Up domain expert),
I want to validate extracted ontology nodes through a conversational skill,
so that the knowledge graph contains only expert-approved concepts with correct semantics.

## Acceptance Criteria

### Scenario: Node presentation for validation
```gherkin
Given a batch of extracted OntologyNode candidates from S3.2
When the curation session starts
Then each node is presented with its key fields (id, name, type, decision_area, description)
And Eduardo can accept, reject, or edit each node
```

### Scenario: Accept a node
```gherkin
Given a presented OntologyNode candidate
When Eduardo accepts the node
Then the node status is marked as "accepted"
And the node is persisted to the curated output YAML
And the session advances to the next node
```

### Scenario: Reject a node
```gherkin
Given a presented OntologyNode candidate
When Eduardo rejects the node with a reason
Then the node status is marked as "rejected" with the rejection reason
And the node is excluded from the curated output
And the session advances to the next node
```

### Scenario: Edit a node
```gherkin
Given a presented OntologyNode candidate
When Eduardo edits one or more fields (name, description, relationships, etc.)
Then the node is updated with Eduardo's corrections
And the modified node validates against the OntologyNode schema
And the node status is marked as "accepted" (with edits recorded)
And the session advances to the next node
```

### Scenario: Session persistence
```gherkin
Given a curation session in progress with 10 of 20 nodes reviewed
When the session ends (timeout or explicit stop)
Then the session state is saved (cursor position, all decisions so far)
And when resumed, the session continues from node 11
And previously accepted/rejected/edited nodes are not re-presented
```

### Scenario: Session summary
```gherkin
Given a completed or paused curation session
When the session ends
Then a summary is displayed: nodes accepted, rejected, edited, remaining
And the curated YAML file reflects all decisions
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| Node: `su-core-values` (type=methodology, decision=People) | Accept | Status: accepted, written to curated YAML |
| Node: `su-unknown-concept` (bad extraction) | Reject with "Not a real Scaling Up concept" | Status: rejected, reason recorded, excluded from output |
| Node: `su-topgrading` with wrong decision_area=Execution | Edit decision_area to People | Updated node validates, status: accepted (edited) |
| Session with 20 nodes, stopped at node 12 | Resume | Starts at node 13, shows "12/20 reviewed" |
| Session complete (all 20 nodes reviewed) | End | Summary: 15 accepted, 3 rejected, 2 edited |

## Notes
- Eduardo's time is the bottleneck (~8h total estimated, 14-21 sessions of 20-30min)
- Batch size: ~20 nodes per session to fit 20-30min window
- Depends on S3.2 extraction pipeline output (OntologyNode YAML files)
- Curated output feeds into S3.4 graph builder (already done, expects valid OntologyNode YAML)
- The skill is conversational (Claude Code session), not a web UI
- Session state must persist between Claude Code sessions (file-based)
