# Story: Schema Validation

## User Story
As a knowledge engineer building the ScaleUp ontology,
I want to validate Eduardo's schema against competency questions and corpus coverage,
so that I have confidence the schema can represent the methodology before building extraction on top.

## Acceptance Criteria

### Scenario: Competency questions covered
```gherkin
Given a set of ≥20 competency questions derived from Scaling Up methodology
When I check each question against the schema's node types and relationships
Then each question maps to at least one traversal path through the ontology
```

### Scenario: Corpus coverage check
```gherkin
Given the Scaling Up corpus (book + 4 workbooks)
And the schema's 6 node types (decision, concept, tool, worksheet, stage, metric)
When I check how many corpus sections map to at least one node type
Then coverage is ≥80% of major sections
And gaps are reported with suggested node type mappings
```

### Scenario: Schema consistency
```gherkin
Given the schema's 6 relationship types
When I validate that relationships reference valid node types
And check for orphan nodes (no relationships)
Then the schema has zero invalid references
And orphans are flagged for review
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| CQ: "What tools support the People decision?" | Check traversal: People → belongs-to → tools | Path exists: decision(people) → tools[topgrading, opsp, ...] |
| Corpus section: "Chapter 3: Core Values" | Map to node type | concept(core-values) — covered |
| Relationship: tool-opsp requires nonexistent-node | Validate references | Error: target "nonexistent-node" not found |
| Node with zero relationships | Check orphans | Warning: orphan node "metric-xyz" |

## Notes
- Eduardo's schema: `.scaleup/knowledge/ontology/` in scaleupagent repo (6 node types, 6 rel types, ~10 example nodes)
- Corpus: `scaling_up_llamaparse.md` (9,380 lines) + `metodologias_parsed/*.md` (4 workbooks, 12,044 lines)
- This is risk-first: if schema needs changes, we fix before building pipeline on top
- Output: ValidationReport with competency coverage, corpus coverage, consistency check
