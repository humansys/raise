# S3.4 Design: Graph Integration (Walking Skeleton)

## Problem & Value

**Problem:** Eduardo's ScaleUp ontology exists as standalone YAML files. RaiSE's knowledge graph (`rai graph`) doesn't know about them — the ScaleUp agent can't traverse methodology concepts alongside patterns, epics, and other RaiSE knowledge.

**Value:** Once mapped, `rai graph query "scaling up"` returns methodology nodes. The retrieval engine (S3.5) can do symbolic traversal across ScaleUp concepts. Multi-domain graph enables cross-cutting insights (e.g., a pattern linked to a ScaleUp concept).

## Approach

Map Eduardo's 6 YAML node types to GraphNode subclasses defined in `rai-agent` (not raise-commons). Use the existing `__init_subclass__(node_type=...)` auto-registration mechanism. Build a `ScaleUpGraphBuilder` that reads YAML files and produces `GraphNode` + `GraphEdge` instances.

### Key Design Decision

**Subclasses with typed fields** over metadata-only approach.

Why: Typed fields give pyright validation, IDE autocomplete, and domain-specific query filtering. The `GraphNode._registry` + `_reconstruct_node` already handle subclass reconstruction on deserialization — we get roundtrip for free.

### Mapping: Eduardo's YAML → GraphNode Subclasses

| Eduardo type | GraphNode subclass | `node_type=` | Typed fields |
|---|---|---|---|
| `decision` | `ScaleUpDecisionNode` | `scaleup.decision` | `key_question`, `chapter_range` |
| `concept` | `ScaleUpConceptNode` | `scaleup.concept` | `decision_area`, `difficulty`, `application` |
| `tool` | `ScaleUpToolNode` | `scaleup.tool` | `decision_area`, `difficulty`, `time_estimate`, `frequency` |
| `worksheet` | `ScaleUpWorksheetNode` | `scaleup.worksheet` | `decision_area`, `difficulty`, `time_estimate`, `frequency`, `fields_list`, `output` |
| `stage` | `ScaleUpStageNode` | `scaleup.stage` | `revenue_range`, `employee_range`, `characteristics` |
| `metric` | `ScaleUpMetricNode` | `scaleup.metric` | `decision_area`, `unit`, `direction`, `benchmark` |

All subclasses share a common base `ScaleUpNodeBase(GraphNode)` that adds:
- `name_es: str` — bilingual support
- `decision_area: str` — the decision this belongs to (people/strategy/execution/cash/cross)
- `source_chapter: str | None` — book chapter pointer
- `source_line: int | None` — book line pointer
- `tags: list[str]` — search keywords

### Mapping: Eduardo's Relationships → GraphEdge

| Eduardo rel type | GraphEdge type | Notes |
|---|---|---|
| `belongs-to` | `belongs_to` | Matches `CoreEdgeTypes.BELONGS_TO` |
| `requires` | `depends_on` | Matches `CoreEdgeTypes.DEPENDS_ON` |
| `feeds-into` | `feeds_into` | New ScaleUp-specific edge type |
| `measured-by` | `measured_by` | New ScaleUp-specific edge type |
| `prerequisite-of` | `depends_on` | Inverse: swap source/target, reuse `DEPENDS_ON` |
| `implements` | `implements` | Matches `CoreEdgeTypes.IMPLEMENTS` |

Note: Eduardo uses hyphens (`belongs-to`), RaiSE uses underscores (`belongs_to`). The builder normalizes.

### Data Flow

```
Eduardo's YAML files (.scaleup/knowledge/**/*.yaml)
         │
         ▼
   yaml.safe_load → OntologyNode (S3.1 Pydantic model, validates schema)
         │
         ▼
   ScaleUpGraphBuilder.build_node(ontology_node) → ScaleUpXxxNode (GraphNode subclass)
   ScaleUpGraphBuilder.build_edges(ontology_node) → list[GraphEdge]
         │
         ▼
   Graph.add_concept(node) + Graph.add_relationship(edge)
         │
         ▼
   QueryEngine.query(Query(types=["scaleup.concept"])) → QueryResult
```

## Examples

### Node creation
```python
from rai_agent.scaleup.graph.nodes import ScaleUpConceptNode

node = ScaleUpConceptNode(
    id="concept-right-people-right-seats",
    content="Right People, Right Seats: every role must be filled by someone who fits the company's core values (right person) AND has the skills and passion for their specific role (right seat).",
    created="2026-03-18",
    name_es="Personas Correctas en los Asientos Correctos",
    decision_area="people",
    difficulty="beginner",
    source_chapter="THE TEAM",
    source_line=1972,
    tags=["people", "hiring", "culture"],
)
assert node.type == "scaleup.concept"
assert node.decision_area == "people"
```

### Builder from YAML
```python
from rai_agent.scaleup.graph.builder import ScaleUpGraphBuilder

builder = ScaleUpGraphBuilder()
graph = builder.build_from_directory("/path/to/.scaleup/knowledge")

assert graph.node_count == 9  # Eduardo's existing nodes
assert graph.edge_count >= 5  # relationships declared in YAML

# Query by type
concepts = graph.get_concepts_by_type("scaleup.concept")
assert len(concepts) >= 1
assert concepts[0].type == "scaleup.concept"
```

### Roundtrip
```python
# Serialize
data = node.model_dump()
# Deserialize — reconstructs typed subclass
reconstructed = ScaleUpConceptNode.model_validate(data)
assert isinstance(reconstructed, ScaleUpConceptNode)
assert reconstructed.decision_area == "people"
```

### Multi-domain coexistence
```python
from raise_core.graph.engine import Graph
from raise_core.graph.models import PatternNode, GraphEdge

graph = Graph()
graph.add_concept(scaleup_node)
graph.add_concept(PatternNode(id="PAT-001", content="...", created="2026-01-01"))
graph.add_relationship(GraphEdge(source="PAT-001", target=scaleup_node.id, type="related_to"))

neighbors = graph.get_neighbors("PAT-001")
assert any(n.type == "scaleup.concept" for n in neighbors)
```

## Acceptance Criteria

**MUST:**
1. 6 GraphNode subclasses auto-registered with `scaleup.*` node types
2. Builder reads Eduardo's YAML directory → produces Graph with nodes + edges
3. Roundtrip: model_dump → model_validate preserves subclass type and typed fields
4. 9 existing Eduardo nodes successfully loaded and queryable
5. All tests pass, pyright strict, ruff clean

**SHOULD:**
1. Cross-domain test: ScaleUp + RaiSE nodes in same graph with edges between them
2. Builder reports skipped/invalid files without crashing (fail-soft per file, fail-fast per field)

**MUST NOT:**
1. Modify raise-commons code
2. Depend on `rai graph build` CLI integration (that's a future story)
3. Add vector/embedding-based retrieval
