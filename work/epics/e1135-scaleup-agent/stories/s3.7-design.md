# S3.7 Design — Knowledge Domain Infrastructure + CLI Gates

> **Revision 2** — 2026-03-19
> Incorporates AR feedback (R1, R2, Q1, Q2) and domain-centric redesign.

## Problem & Value

**Problem:** Rai needs deterministic tools to validate knowledge it extracts from any domain. Without CLI gates, quality depends 100% on inference. Without a domain abstraction, each new domain requires touching code.

**Value:** Pluggable knowledge domains with deterministic validation gates. New domain = new directory + `domain.yaml` manifest. No code changes. The agent iterates autonomously (extract → validate → fix) until gates pass. Research (128 sources) confirms this is the right architecture.

## Complexity

Moderate — domain model, 4 gates, CLI with domain discovery, scaffold command.

## Decisions (all human-approved)

### D1: CLI — `rai knowledge` via entry point
`rai.cli.commands` entry point. RAISE-594 implemented.

### D2: Output — Human default + `--json`
Exit code always 0/1. `--json` for structured GateResult.

### D3: Code layout — `knowledge/` generic + `scaleup/` domain-specific
Gates never import domain-specific types. Domain resolution via `domain.yaml`.

### D4: GateConfig — derived from `domain.yaml` (dynamic import with validation)
`domain.yaml` declares schema module + class. `load_domain()` imports dynamically with explicit validation (issubclass check). Safer than hardcoded imports because the validate gate catches model issues.

### D5: Domain home — `.raise/knowledge/`
Under RaiSE namespace. Consistent with `.raise/artifacts/`, `.raise/rai/`.

### D6: Chunk is NOT a gate
Chunk is a production step (split corpus → write files), not a validation step. Stays in extraction pipeline. Not exposed as CLI command.

### D7: Init scope — scaffold only (no discovery)
`rai knowledge init` creates directory + template `domain.yaml` + empty dirs. Schema discovery is S3.8.

### D8: Single `gates.py` file (AR recommendation R2)
4 gate functions (~20 lines each) in one file. Split if any grows beyond 50 lines.

## Architecture

### Domain as unit

```
.raise/knowledge/
  scaleup/                         ← one domain
    domain.yaml                    # manifest (schema, corpus, thresholds)
    competency_questions.yaml      # CQs for this domain
    extracted/                     # YAML nodes from extraction (Rai output)
    curated/                       # YAML nodes validated by human
  gtd/                             ← another domain (future)
    domain.yaml
    ...
```

### domain.yaml manifest

```yaml
name: scaleup
display_name: "Scaling Up (Verne Harnish)"
schema:
  module: rai_agent.scaleup.validation.models
  class: OntologyNode
corpus:
  - scaling_up_llamaparse.md
  - metodologias_parsed/
competency_questions: competency_questions.yaml
thresholds:
  cq_coverage: 80.0
required_types:
  - decision
  - concept
  - tool
  - worksheet
  - stage
  - metric
```

### CLI commands

```bash
# Check all gates for a domain
rai knowledge check --domain scaleup

# Check specific gate
rai knowledge check --domain scaleup --gate validate
rai knowledge check --domain scaleup --gate reconcile
rai knowledge check --domain scaleup --gate coverage
rai knowledge check --domain scaleup --gate graph

# Domain status (all domains, quick overview)
rai knowledge status

# Initialize new domain (scaffold)
rai knowledge init --domain gtd --corpus ~/Books/gtd.md

# JSON output
rai knowledge check --domain scaleup --json
```

### Code layout

```
src/rai_agent/knowledge/          ← GENERIC (no domain imports)
  __init__.py
  models.py                       # GateResult, GateConfig, DomainManifest
  domain.py                       # load_domain(), discover_domains(), scaffold_domain()
  gates.py                        # run_validate(), run_reconcile(), run_coverage(), run_graph()
  cli.py                          # Typer app: check, status, init
  formatter.py                    # Human-readable output for gate results

src/rai_agent/scaleup/            ← DOMAIN-SPECIFIC (existing)
  validation/models.py            # OntologyNode (the schema model)
  validation/schema_validator.py  # load_nodes, check_competency, check_consistency
  extraction/reconciler.py        # reconcile_nodes
  graph/builder.py                # ScaleUpGraphBuilder

.raise/knowledge/scaleup/         ← DOMAIN DATA
  domain.yaml                     # manifest
  competency_questions.yaml       # CQs (already exists, move here)
  extracted/                      # output of extraction
  curated/                        # output of curation
```

### Data flow

```
domain.yaml ──► load_domain() ──► DomainManifest ──► GateConfig
                                                         │
                    ┌────────────────────────────────────┘
                    ▼
              run_validate(node_dir, config) ──► GateResult
              run_reconcile(node_dir, config) ──► GateResult
              run_coverage(node_dir, config) ──► GateResult
              run_graph(node_dir, config) ──► GateResult
                    │
                    ▼
              CLI formats (human or JSON) + exit code
```

## Models

### DomainManifest

```python
class SchemaRef(BaseModel):
    module: str       # "rai_agent.scaleup.validation.models"
    class_name: str   # "OntologyNode" (field name avoids 'class' keyword)

class DomainManifest(BaseModel):
    name: str
    display_name: str
    schema: SchemaRef
    corpus: list[str] = []
    competency_questions: str | None = None
    thresholds: dict[str, float] = {"cq_coverage": 80.0}
    required_types: set[str] = set()
```

### GateConfig (derived from manifest at runtime)

```python
class GateConfig(BaseModel, arbitrary_types_allowed=True):
    node_model: type[BaseModel]
    cq_file: Path | None = None
    cq_threshold: float = 80.0
    required_types: set[str] = set()
    node_dir: Path              # .raise/knowledge/{domain}/curated/ (or extracted/)
```

### GateResult

```python
class GateResult(BaseModel):
    gate: str
    domain: str
    passed: bool
    metrics: dict[str, Any]
    errors: list[str]
    warnings: list[str]
    duration_ms: float
```

## Acceptance Criteria

**MUST:**
1. `rai knowledge check --domain X` runs 4 gates (validate, reconcile, coverage, graph), exits 0/1
2. `rai knowledge check --domain X --gate Y` runs single gate, exits 0/1
3. `rai knowledge status` lists registered domains with node counts
4. `rai knowledge init --domain X` scaffolds domain directory + template domain.yaml
5. All check commands support `--json` for structured GateResult output
6. `domain.yaml` is the single source of truth for domain config (schema, thresholds, CQs)
7. `knowledge/` modules never import domain-specific types — dynamic import via manifest
8. ScaleUp works as first domain with existing validation/extraction/graph code

**SHOULD:**
- Duration reported per gate
- Human output uses color/symbols (✓/✗) for quick scanning
- `--domain` defaults to the only registered domain if there's exactly one

**MUST NOT:**
- Import OntologyNode or any ScaleUp type inside `knowledge/` modules
- Add new dependencies
- Break existing `rai` CLI commands
- Include chunk as a gate (it's a production step)
