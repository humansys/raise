# S3.9 Design — Knowledge Pipeline Skills

> 4 interactive skills + optional orchestrator. Wraps existing code from S3.2/S3.7/S3.8 in HITL-friendly skills.

## Problem & Value

**Problem:** The knowledge pipeline (discover → extract → validate → diff) only runs as opaque Python scripts. No progress visibility, no HITL checkpoints, errors surface after long waits. S3.8 proved the code works but exposed the UX gap.

**Value:** Interactive skills let the operator (Emilio, Eduardo, future domain experts) run each phase with full visibility, intervene when needed, and build trust in the pipeline output before it enters the knowledge graph.

## Complexity: Moderate

- 4-5 skill files (SKILL.md each, following RaiSE skill pattern)
- No new algorithms — wrapping existing functions
- Key challenge: making LLM-calling code observable and interruptible within Claude Code skills

## Architecture

```
/rai-knowledge-discover  → discover_schema() → SchemaSpec → HITL review
        │
        ▼
/rai-knowledge-extract   → ExtractionPipeline.run() → nodes YAML → HITL review
        │
        ▼
/rai-knowledge-validate  → rai knowledge check → GateResults → HITL review
        │
        ▼
/rai-knowledge-diff      → diff_schemas() + diff_nodes() → reports → HITL review

/rai-knowledge-run       → chains all 4, pausing at each HITL gate
```

## Design Decisions

### D1: Skills are Claude Code skills (SKILL.md), not CLI commands

Skills run inside Claude Code's inference context — they can call tools, display progress inline, ask questions, and pause. CLI commands would be batch/opaque (the problem we're solving).

### D2: Each skill is self-contained — no shared state between phases

Each skill reads from disk (previous phase's output) and writes to disk. No in-memory pipeline object passed between skills. This means:
- Operator can run phases in any order (validate before diff, re-run extract)
- Session interruptions are safe (output is on disk)
- Each skill declares its inputs/outputs explicitly

### D3: Extraction skill uses the pipeline in "chapter mode" not "batch mode"

Instead of calling `pipeline.run()` (batch), the skill:
1. Creates the pipeline
2. Iterates chapters manually using `CorpusChunker`
3. Calls `_extract_chapter_nodes` per chapter
4. Reports progress after each chapter
5. Accumulates results, handles errors per-chapter

This requires the extraction function to be callable per-chapter (it already is — `_extract_chapter_nodes` is a module-level function).

### D4: Domain-agnostic — skills take domain name as parameter

Skills resolve domain config from `.raise/knowledge/{domain}/domain.yaml`. ScaleUp is the first domain but the skills work for any configured domain.

### D5: CLAUDECODE bypass handled inside invoke_structured

The skills run inside Claude Code sessions. LLM calls via `invoke_structured` need `CLAUDECODE=` unset. The `invoke_structured` function should handle this internally rather than requiring the operator to know about it.

## Skill Structure (per skill)

Each follows the RaiSE SKILL.md pattern:

```markdown
---
name: rai-knowledge-{phase}
description: ...
---

# Knowledge {Phase}

## Purpose
## Steps
### Step 1: Load Context
  - Resolve domain from `.raise/knowledge/{domain}/domain.yaml`
  - Check prerequisites (previous phase output exists)

### Step 2: Execute Phase
  - Call existing Python functions
  - Report progress inline

### Step 3: Present Results
  - Display summary to operator
  - Save outputs to disk

### Step 4: HITL Checkpoint
  - Ask operator to review before continuing
```

## Acceptance Criteria

### MUST
1. 4 skills (discover, extract, validate, diff) operational with ScaleUp domain
2. Extraction shows chapter-by-chapter progress inline
3. Each skill saves output to disk and reports paths
4. HITL pause after each phase
5. invoke_structured handles CLAUDECODE bypass internally

### SHOULD
1. Orchestrator skill (`/rai-knowledge-run`) chains all phases
2. Skills work with any domain in `.raise/knowledge/{domain}/`

### MUST NOT
1. No modifications to existing pipeline algorithms
2. No new CLI subcommands (these are skills, not CLI)
3. Skills must not crash on per-chapter extraction errors (skip + warn)
