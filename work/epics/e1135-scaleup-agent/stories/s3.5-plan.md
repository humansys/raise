### T2: Scoring functions + `retrieve()` orchestrator
**Size:** M | **Dependencies:** T1

**Create:**
- `src/rai_agent/knowledge/retrieval/engine.py`:
  - `spreading_activation(graph, seed_ids, edge_weights, decay, max_depth)` → `dict[str, float]` — pure function, BFS with exponential decay
  - `attribute_match(node, keywords)` → `float` — keyword overlap ratio
  - `type_relevance(node_type, type_boosts)` → `float` — lookup with default 0.5
  - `composite_score(sa, attr, type_rel, weights)` → `float` — weighted sum
  - `retrieve(graph, query, adapter, top_k)` → `RetrievalResult` — **function**, not class

**`retrieve()` logic:**
1. `adapter.interpret_query(query)` → `DomainHints` (fallback: empty `DomainHints(domain="unknown")`)
2. `adapter.advise_traversal(hints, available_types)` → `TraversalAdvice` (fallback: full-graph keyword search)
3. BFS traversal using advice (engine owns this, uses `graph.get_neighbors()`)
4. `adapter.annotate_results(traversed, hints)` → `list[ScoredNode]` with domain metadata (fallback: nodes with score=0.0)
5. Apply composite scoring using SA scores + annotation metadata, sort desc, return top-k

**Scoring weights** (named constants in engine.py, AR-Q3):
```python
W_SA = 0.5
W_ATTR = 0.3
W_TYPE = 0.2
```

**TDD:**
- RED: SA on 3-node chain (verify decay). `attribute_match` with known keywords. `type_relevance` lookup. `composite_score` weighted sum. `retrieve()` with mock adapter (verify 3-step flow). Fallback when adapter raises. Empty graph → empty result. top-k limiting.
- GREEN: Implement scoring + orchestrator
- REFACTOR: Pure functions, no graph mutation, no side effects

  - `EDGE_WEIGHTS: dict[str, float]` — class attribute (`requires=0.9`, `feeds_into=0.8`, etc.)

**Logic:**
- `interpret_query()`: keyword matching against decision areas + stages + node types → `ScaleUpHints`
- `advise_traversal()`: seed from `decision-{area}` anchor nodes, filter by edge types, depth=2
- `annotate_results()`: attach domain metadata (edge_weight, type_boost, decision_area_match) as scored nodes for engine composite scoring

## Architecture Review (pre-implementation)

**Verdict:** PASS WITH QUESTIONS — all resolved. Simplifications applied directly to tasks above.

| ID | Simplification | Effect |
|----|----------------|--------|
| R1 | 2 files in `knowledge/retrieval/` (`models.py` + `engine.py`) | T1+T2 scope, no `protocol.py` or `scoring.py` |
| R2 | `retrieve()` as function, not class | T2: no `RetrievalEngine` class |
| R3 | `annotate_results()` not `rank_results()` | Engine owns all scoring; adapter only provides metadata |
| Q1 | Keep 3 adapter methods | Extensibility over simplicity |
| Q2 | `ScaleUpHints` inside `adapter.py` | Small, coupled, split when it grows |
| Q3 | Weights as named constants in `engine.py` | Simple for MVP |
