# S3.1 Retrospective: Schema Validation

> **Story:** S3.1 | **Size:** S | **Date:** 2026-03-18
> **Estimated:** S (~45 min) | **Actual:** ~40 min

## Summary

Built a schema validation module (`src/rai_agent/scaleup/validation/`) with 3 checks against Eduardo's ScaleUp ontology: competency question traversal, corpus coverage, and consistency. 34 tests, pyright strict clean, ruff clean.

## What Went Well

- **TDD flow was smooth** — RED→GREEN→REFACTOR worked cleanly for all 3 tasks
- **Fixture-based testing** — 7 minimal YAML nodes + corpus excerpt gave full coverage of all validator paths without needing real data
- **Integration test against real data** gave immediate visibility into schema completeness (43.5% CQ coverage, 8 invalid refs)
- **Decision to use rai-agent repo** (not scaleupagent) was correct — avoided creating Python infra in a data-only repo

## What to Improve

- **D1 was wrong initially** — design placed code in scaleupagent without verifying it had Python project structure. Fixed during planning but wasted a design revision cycle
- **Corpus coverage H1 parsing** counts front matter (dedications, acknowledgements) as "sections" — 293 total when meaningful content sections are ~30. Could filter by position or content length

## Heutagogical Checkpoint

1. **Learned:** Regex-based path parsing works for CQ validation but is fragile. A formal traversal API would be better for S3.5 retrieval engine.
2. **Process change:** Always verify target repo infrastructure in design phase before assigning code location.
3. **Framework:** Document worktree-based workflow — no sub-branches within worktrees.
4. **Capability:** Can validate YAML ontologies against competency questions and corpus. load→check→report pattern is reusable.

## Patterns

- **PAT-E-001:** Verify target repo infrastructure before assigning code location in design
- **PAT-E-002:** Worktree = isolation boundary — work directly on worktree branch, no sub-branches
