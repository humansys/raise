# S3.3 Scope: HITL Curation Skill

## In Scope

- **Conversational curation skill** -- Claude Code skill (`/rai-scaleup-curate`) for Eduardo to validate extracted nodes
- **Batch presentation** -- present ~20 nodes per session, optimized for 20-30min review windows
- **Accept/reject/edit workflow** -- per-node decisions with schema validation on edits
- **Rejection reasons** -- capture why a node was rejected (bad extraction, duplicate, wrong type, etc.)
- **Edit tracking** -- record what fields were changed and original values
- **Session persistence** -- file-based state that survives between Claude Code sessions (JSON/YAML)
- **Session resume** -- pick up where Eduardo left off, no re-presenting reviewed nodes
- **Curated output serialization** -- accepted/edited nodes written to YAML compatible with S3.4 graph builder
- **Session summary** -- at session end, show counts (accepted, rejected, edited, remaining)
- **Input from S3.2** -- reads extracted OntologyNode YAML candidates as input

## Out of Scope

- Web UI or visual interface -- this is a conversational (CLI) skill
- Graph building or integration (S3.4 -- already done)
- Retrieval engine (S3.5)
- Extraction pipeline modifications (S3.2 -- already done)
- Bulk operations (accept-all, reject-all) -- every node gets individual review
- Relationship editing between nodes -- focus is on individual node validation
- Multi-user curation -- single curator (Eduardo)
- Automated quality scoring of nodes -- curation is human judgment

## Done Criteria

- [ ] Eduardo can start a curation session and see extracted nodes presented one by one
- [ ] Each node can be accepted, rejected (with reason), or edited
- [ ] Edited nodes validate against OntologyNode Pydantic schema
- [ ] Session state persists to disk and can be resumed in a new session
- [ ] Curated (accepted + edited) nodes are serialized to YAML
- [ ] Session summary shows accept/reject/edit/remaining counts
- [ ] All tests pass, pyright strict, ruff clean
