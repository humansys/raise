# S3.5 Design — Retrieval Engine (Domain-Agnostic + ScaleUp Adapter)

> Research-grounded design. 3 research lines (R1-R3), 20+ sources, 6 decisions HITL-approved.

## Problem & Value

**Problem:** Tenemos un grafo ScaleUp poblado (nodos de Eduardo) pero no hay forma de preguntar "dado que estoy en etapa de crecimiento con problemas en People, qué herramientas me aplican". Solo existe keyword search genérico en raise-commons.

**Value:** El agente da guía contextualizada en vez de respuestas genéricas. "Tu score en People=2, enfócate en Core Values antes de Topgrading" requiere traversar el grafo con contexto de negocio. La arquitectura domain-agnostic permite reutilizar el engine para futuros dominios (GTD, Life Areas).

## Complexity: Moderate

- 4 componentes: Protocol + engine + ScaleUp adapter + tests
- Integración con Graph engine existente de raise-commons
- Algoritmo custom (Spreading Activation) pero bien documentado

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│                  RetrievalEngine                        │
│  retrieve(graph, query, domain) → list[ScoredNode]     │
│                                                         │
│  1. adapter.interpret_query(text) → DomainHints         │
│  2. adapter.advise_traversal(hints) → TraversalAdvice   │
│  3. ENGINE executes traversal (BFS/SA)                  │
│  4. adapter.rank_results(nodes, hints) → ScoredNodes    │
│  5. Composite scoring: SA(0.5) + attr(0.3) + type(0.2) │
├─────────────────────────────────────────────────────────┤
│               DomainAdapter Protocol                    │
│  interpret_query() / advise_traversal() / rank_results()│
├──────────────┬──────────────────────────────────────────┤
│ ScaleUp      │ (future: GTD, Life Areas, ...)           │
│ Adapter      │                                          │
│ - edge weights: requires=0.9, feeds_into=0.8, ...      │
│ - seed logic: decision_area → decision-{area} node     │
│ - type boost: tool > concept > metric for "action"     │
├──────────────┴──────────────────────────────────────────┤
│          raise-commons Graph (NetworkX MultiDiGraph)    │
└─────────────────────────────────────────────────────────┘
```

## Design Decisions

### D1: Retrieval location → `knowledge/` (domain-agnostic)

**Choice:** Protocol, engine, models, and scoring live in `rai_agent/knowledge/retrieval/`. Domain implementations live in their own package (`scaleup/retrieval/`).

**Rationale:** Multi-domain retrieval es el objetivo. El engine debe ser reutilizable sin copiar código.

**Alternatives considered:**
- `scaleup/` only (simple, but YAGNI violation when second domain arrives)
- Extend raise-commons QueryEngine (mixes domain logic into framework)

### D2: Adapter interface → 3 methods

**Choice:** `interpret_query` / `advise_traversal` / `rank_results`

**Rationale:** Aislamiento de fallos (cada paso puede fallar independientemente con fallback), testabilidad unitaria sin grafo, observabilidad por paso. Patrón validado por TinkerPop TraversalStrategy y Neo4j GraphRAG Retriever.

**Alternatives considered:**
- 2 methods (advise + rank) — loses interpretation isolation
- 1 method (retrieve) — black box, no graceful degradation

**Key principle:** Adapter ADVISES, engine OWNS traversal. (R2, Finding 1 — "advise don't own" pattern dominates production systems)

### D3: Scoring → Spreading Activation + composite

**Choice:** SA as primary structural scoring, composite formula with 3 signals.

```
score = 0.5 * spreading_activation + 0.3 * attribute_match + 0.2 * type_relevance
```

**Rationale:** SA is purpose-built for associative retrieval in semantic networks. No convergence issues (unlike PPR on small graphs). Query-seeded, supports typed edge weights, exponential decay. ~50 lines, <5ms.

**Why not PPR:** Dangling nodes (leaves) create rank sinks, cycles accumulate rank, teleportation parameter has outsized effect on small graphs. (R3, Finding 9)

**Alternatives considered:**
- PPR (HippoRAG validated, but convergence issues at our scale)
- SA without composite (loses attribute and type signals)

**Sources:** SA-RAG (2024), HippoRAG (NeurIPS 2024), ACT-R activation model.

### D4: Query context → Hybrid Pydantic model

**Choice:** `DomainHints` base with `extra="allow"` + typed subclasses per known domain.

```python
class DomainHints(BaseModel):
    model_config = ConfigDict(extra="allow")
    domain: str

class ScaleUpHints(DomainHints):
    domain: Literal["scaleup"] = "scaleup"
    decision_area: str | None = None
    stage: str | None = None
    difficulty: str | None = None
```

**Rationale:** Typed subclasses give validation + autocomplete for known domains. `extra="allow"` lets unknown domains pass arbitrary fields without schema changes.

**Alternatives considered:**
- Discriminated unions only (each new domain requires code change)
- Open dict only (no validation, no IDE support)

### D5: Edge weights → Manual domain weights

**Choice:** Adapter declares weights per relationship type. Engine uses them in SA propagation.

```python
EDGE_WEIGHTS = {
    "requires": 0.9, "feeds_into": 0.8, "belongs_to": 0.7,
    "implements": 0.7, "measured_by": 0.6, "related_to": 0.3,
}
```

**Rationale:** Simple, explicable, domain expert provides semantic knowledge. IDF-style automatic weighting available as future enhancement.

**Alternatives considered:**
- Hybrid manual + IDF (more robust but more complex)
- IDF only (ignores domain semantics)

### D6: ScaleUp adapter location → `scaleup/retrieval/`

**Choice:** ScaleUp adapter implementation lives next to other ScaleUp code.

**Rationale:** Adapter IS domain code — it knows about decision areas, stages, ScaleUp node types. `knowledge/` only contains the Protocol and engine.

**Alternatives considered:**
- `knowledge/adapters/scaleup.py` (centralizes but mixes concerns)

## Examples

### API Usage

```python
from rai_agent.knowledge.retrieval.engine import RetrievalEngine
from rai_agent.scaleup.retrieval.adapter import ScaleUpAdapter

engine = RetrievalEngine()
adapter = ScaleUpAdapter()

# Structured query
results = engine.retrieve(
    graph=graph,
    query="cash flow tools for startups",
    adapter=adapter,
)

# results: list[ScoredNode]
for r in results:
    print(f"{r.node.id}: {r.score:.2f} — {r.explanation}")
# decision-cash: 0.92 — seed node (decision_area match)
# tool-ccc: 0.85 — SA=0.8, attr=0.9, type=1.0
# concept-power-of-one: 0.71 — SA=0.6, attr=0.7, type=0.8
```

### Adapter flow (ScaleUp)

```python
# Step 1: interpret_query
hints = adapter.interpret_query("cash flow tools for startups")
# → ScaleUpHints(domain="scaleup", decision_area="cash", stage="startup")

# Step 2: advise_traversal
advice = adapter.advise_traversal(hints, available_types=frozenset({"tool","concept","metric"}))
# → TraversalAdvice(
#     start_node_ids=["decision-cash"],
#     edge_type_filter=["belongs_to", "requires", "feeds_into"],
#     node_type_filter=["tool"],
#     max_depth=2,
#   )

# Step 3: engine traverses, then adapter ranks
scored = adapter.rank_results(traversed_nodes, hints)
# → [ScoredNode(node=<CCC>, score=0.85, explanation="SA=0.8, attr=0.9, type=1.0"), ...]
```

### Empty query / no match

```python
results = engine.retrieve(graph=graph, query="quantum physics", adapter=adapter)
# → [] (empty — no nodes match, no fabrication)
```

## Acceptance Criteria

### MUST

1. `RetrievalEngine.retrieve()` returns relevant ScaleUp nodes given decision area context
2. DomainAdapter Protocol with 3 methods (interpret/advise/rank) is defined and testeable
3. Spreading Activation scoring with configurable edge weights and decay
4. Composite scoring (SA + attribute + type) with configurable weights
5. Empty/no-match queries return empty list (no fabrication)

### SHOULD

1. ScaleUp adapter handles all 4 decision areas (people/strategy/execution/cash)
2. Results include explanation string (which signals contributed to score)
3. Edge weights declared by adapter, not hardcoded in engine

### MUST NOT

1. No LLM calls in retrieval path — pure graph operations
2. No vector embeddings — symbolic only
3. No modification to raise-commons — use existing Graph API

## Testing Strategy

- Unit tests per adapter method (no graph needed)
- Unit tests for SA algorithm (synthetic graph)
- Integration tests with real ScaleUp node structure
- Evaluation with existing competency questions (23 CQs from S3.1)
