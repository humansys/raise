# S3.2 Design: Extraction Pipeline

## Problem & Value

**Problem:** Eduardo has 9 manually authored ontology nodes. The Scaling Up corpus contains ~68 methodologies + ~34 worksheets + decisions, stages, and metrics. Manual YAML authoring doesn't scale.

**Value:** Automated extraction produces candidate nodes that Eduardo curates (S3.3), populating the knowledge graph for the ScaleUp agent. The pipeline is the bottleneck between validated schema (S3.1) and curation (S3.3).

## Approach

Use `instructor` library with the Anthropic API (Claude) to extract `OntologyNode` instances directly from corpus text. The corpus is chunked by chapter/decision area. Eduardo's 9 existing nodes serve as few-shot examples in the extraction prompt. Post-extraction reconciliation reports phantom relationship targets for Eduardo to resolve in S3.3.

### Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | instructor + Pydantic structured outputs | Simplest: OntologyNode IS the schema, no extra abstraction layer |
| D2 | Multi-pass by chapter | 9380-line corpus too long for single pass; chapter structure is natural chunking |
| D3 | `difficulty` as `Literal["beginner","intermediate","advanced"]` in OntologyNode | Validation at extraction boundary; GraphNode keeps `str` for flexibility |
| D4 | Phantom nodes: collect & reconcile, don't fail | Extraction shouldn't block on completeness; that's curation's job |

### Data Flow

```
scaling_up_llamaparse.md (9380 lines)
         |
         v
   CorpusChunker.split_by_chapter()
   -> list[Chapter] (People chs 3-5, Strategy 6-8, Execution 9-11, Cash 12-14)
         |
         v  (per chapter)
   instructor.from_anthropic(Claude) + extraction prompt + few-shot examples
   -> list[OntologyNode] (validated by Pydantic on extraction)
         |
         v
   Merge all chapters' nodes
         |
         v
   reconcile_nodes(all_nodes)
   -> ReconciliationReport (phantom_targets, orphan_nodes, cross_decision_edges)
         |
         v
   ExtractionResult(nodes, reconciliation, coverage)
         |
         v
   pipeline.write_yaml(nodes, output_dir)
   -> individual .yaml files matching Eduardo's format
```

### Key Models

```python
# --- Chunker output ---
class Chapter(BaseModel):
    title: str
    decision_area: DecisionArea
    text: str
    line_start: int
    line_end: int

# --- Reconciliation output ---
class ReconciliationReport(BaseModel):
    phantom_targets: list[str]       # IDs referenced but no node exists
    orphan_nodes: list[str]          # nodes with no relationships
    cross_decision_edges: list[tuple[str, str]]  # edges across decision areas

# --- Pipeline output ---
class CoverageReport(BaseModel):
    total_extracted: int
    by_type: dict[str, int]          # e.g. {"concept": 42, "tool": 15, ...}
    by_decision: dict[str, int]      # e.g. {"people": 28, "strategy": 22, ...}
    missing_source_refs: list[str]   # node IDs without source pointers

class ExtractionResult(BaseModel):
    nodes: list[OntologyNode]
    reconciliation: ReconciliationReport
    coverage: CoverageReport
```

### OntologyNode modification (D3)

Add `difficulty` field with Literal validation:

```python
# In OntologyNode (validation/models.py)
DifficultyLevel = Literal["beginner", "intermediate", "advanced"]

class OntologyNode(BaseModel):
    # ... existing fields ...
    difficulty: DifficultyLevel | None = None  # optional, not all node types have it
```

GraphNode subclasses (`ScaleUpConceptNode`, `ScaleUpToolNode`, `ScaleUpWorksheetNode`) keep `difficulty: str` — no change needed there.

### Prompt Strategy

Each extraction call receives:
1. **System prompt**: role (ontology extractor), output format (OntologyNode schema), constraints (valid node types, relationship types, ID format)
2. **Few-shot examples**: 2-3 of Eduardo's nodes matching the current decision area
3. **Chapter text**: the corpus chunk to extract from
4. **Task**: extract all methodologies, tools, worksheets, stages, and metrics from this chapter

instructor handles the structured output parsing — the LLM response IS the Pydantic model.

### Reproducibility

- `temperature=0` on all extraction calls
- Deterministic chunking (chapter boundaries from H1 headers)
- Sorted output (by node ID) before serialization
- `seed` parameter if supported by the model

## Examples

### Pipeline usage
```python
from rai_agent.scaleup.extraction.pipeline import ExtractionPipeline

pipeline = ExtractionPipeline(
    corpus_path=Path("/path/to/scaling_up_llamaparse.md"),
    examples_dir=Path("/path/to/.scaleup/knowledge/"),
    model="claude-sonnet-4-20250514",
)
result = pipeline.run()
# result.nodes: list[OntologyNode]
# result.reconciliation: ReconciliationReport
# result.coverage: CoverageReport

pipeline.write_yaml(result.nodes, output_dir=Path("./output/"))
```

## Acceptance Criteria

**MUST:**
1. Pipeline extracts OntologyNode instances from corpus chapters using instructor + Claude
2. Extracted nodes validate against OntologyNode Pydantic schema
3. Extraction prompts include Eduardo's nodes as few-shot examples
4. Corpus chunking splits by chapter/decision area
5. Post-extraction reconciliation reports phantom targets
6. Pipeline output is serializable to YAML matching Eduardo's format
7. Reproducible: same inputs produce structurally equivalent outputs

**SHOULD:**
8. Coverage >= 80% of corpus concepts
9. Source references (book_chapter, book_line) populated on each node

**MUST NOT:**
10. Must NOT require internet access during extraction (corpus is local, LLM API only)
11. Must NOT modify Eduardo's existing YAML nodes
12. Must NOT add LlamaIndex or KGGen dependencies
