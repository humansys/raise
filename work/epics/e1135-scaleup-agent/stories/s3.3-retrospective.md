# S3.3 Retrospective: HITL Curation Skill

> **Story:** S3.3 | **Size:** S | **Date:** 2026-03-19
> **Estimated:** 90min | **Actual:** ~16min | **Velocity:** 5.6x

## Summary

Built the HITL curation skill for ScaleUp ontology validation: curation models (decisions, session state, summary), session engine with YAML persistence and resume, node formatter for conversational display, skill definition, and integration tests. 52 tests, all passing. QR findings applied (bounds checks, skip logging, cursor validation).

## What Went Well

1. **Clean separation of concerns** — Models, session engine, formatter, and skill are independent components with well-defined interfaces. Each was testable in isolation with no mocking of adjacent components.
2. **Reuse of existing infrastructure** — `OntologyNode` from S3.2 validation models and `node_to_yaml_dict` from extraction utils eliminated duplicate serialization logic. The curated output is directly consumable by S3.4 graph builder.
3. **QR gate caught real defects** — Three issues found: silent drops on malformed files (BASE-046 violation), bare IndexError on cursor overflow, and out-of-order node_id acceptance. All would have caused session corruption in production use.
4. **TDD drove the API design** — Writing tests first for session load/resume/record_decision forced a clean state machine interface. The skip functionality emerged naturally from the test scenarios.

## What Could Be Better

1. **Massive over-estimation again** — 90min estimated vs ~16min actual (5.6x). S3.2 had 3.67x. Pattern is clear: stories with no LLM integration, no external APIs, and well-defined models execute much faster than estimated. Need to recalibrate S-size estimates for pure data-manipulation stories to ~20-30min.
2. **bubus dependency issue persists** — `uv run` fails in worktree due to `bubus>=2.3` resolution for Python 3.14 markers. Workaround is `PYTHONPATH=src` with main repo venv. This is tech debt accumulating across stories.
3. **AR questions from S3.2 still unresolved** — Q1 (skip semantics for curation) was partially answered by QR — skip was implemented but needed bounds validation. Q2 (resume validation) was answered cleanly by the cursor+node_order approach.

## Heutagogical Checkpoint

### 1. What did I learn?

HITL workflows have a specific class of bugs: state machine corruption from out-of-order operations. When a session tracks cursor position and node decisions, every mutation must validate that the operation is consistent with current state. The QR finding about `record_decision()` accepting arbitrary `node_id` values (not just the current node) would have caused silent cursor desync — the kind of bug that surfaces only after 10+ session resumes.

### 2. What would I change about the process?

Add explicit state-machine invariant tests to the TDD cycle for any stateful workflow. The happy-path tests covered load/record/write but not the edge cases of "what happens when operations arrive in wrong order?" This should be a standard practice for HITL components.

### 3. Are there improvements for the framework?

The QR gate continues to justify its cost (found 3 bugs across 3 consecutive stories). However, the estimation calibration is consistently off by 3-6x for pure-logic stories. The framework could benefit from a size-adjustment heuristic: if a story has zero external integrations (no LLM, no API, no I/O beyond files), apply a 0.3x multiplier to the time estimate.

### 4. What am I more capable of now?

Building stateful conversational workflows with YAML persistence. The session engine pattern (load/record/persist/resume) is now a tested, reusable approach for any multi-session HITL process. Also reinforced: category-at-a-time presentation as a UX pattern for domain expert review.
