# S3.8 Design — Full-Cycle Extraction from Zero

> 3 decisions HITL-approved. Zero absoluto: schema discovery + extraction + validation + diff.

## Problem & Value

**Problem:** Pipeline de extracción (S3.2) y gates (S3.7) nunca se han corrido juntos. No sabemos si el LLM puede descubrir la estructura del dominio sin schema previo, ni cómo su output se compara con el trabajo manual de Eduardo.

**Value:** Validación epistemológica del pipeline. Si la máquina descubre estructura similar al experto, el pipeline es confiable para futuros dominios. El diff report documenta dónde divergen y por qué.

## Complexity: Moderate

- 3 componentes nuevos: discovery, schema diff, content diff
- 1 modificación: pipeline acepta `examples_dir=None`
- Integración con infraestructura existente (S3.2 pipeline, S3.7 gates)
- LLM calls para discovery y extracción

## Architecture

```
  ▼
┌─────────────────────────┐

         │
         ▼
┌─────────────────────────┐

         │
         ▼
┌─────────────────────────┐
│  Extraction (S3.2)      │ ← Pipeline existente, examples_dir=None
│  → YAML nodes           │
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│  Gate Validation (S3.7) │ ← rai knowledge check scaleup

└────────┬────────────────┘
         │
         ▼

│  Content Diff (NEW)     │ ← extracted/ vs Eduardo's curated/
│  → NodeDiffReport       │
└─────────────────────────┘
```

## Design Decisions

### D1: Nivel de "from zero" → Zero absoluto
**Choice:** Descubrir schema automáticamente desde corpus, sin OntologyNode predefinido.
**Rationale:** El valor de S3.8 es epistemológico — ver qué descubre la máquina sola.
**Alternatives:** B) Zero de contenido (usar schema, sin examples), C) Solo validación del ciclo.

### D2: Enfoque de discovery → Discover → Compare → Extract
**Choice:** Discovery produce spec libre, comparamos con Eduardo, luego extraemos con OntologyNode.
**Rationale:** Dos comparaciones (schema + contenido) maximizan insight. Pragmático.
**Alternatives:** B) Generar Pydantic dinámico, C) Dos extracciones paralelas.

### D3: Modelo para extracción → Map to OntologyNode
**Choice:** Extracción usa OntologyNode existente (compatible con gates). Schema diff documenta diferencias.
**Rationale:** Gates requieren modelo conocido. El diff ya captura lo que el LLM descubrió diferente.
**Alternatives:** B) Pydantic dinámico en runtime, C) Dos extracciones (una con cada schema).

### Schema Diff
```python
from rai_agent.knowledge.discovery import diff_schemas
from rai_agent.scaleup.validation.models import OntologyNode

report = diff_schemas(discovered=spec, reference_model=OntologyNode)
# → SchemaDiffReport(
#     types_both=["concept", "tool", "decision"],
#     types_only_discovered=["principle"],
#     types_only_reference=["worksheet"],
```

### Extraction (zero examples)
```python
pipeline = ExtractionPipeline(
    corpus_path="scaling_up_llamaparse.md",
    examples_dir=None,  # zero: no few-shot examples
```


## Acceptance Criteria

### MUST
1. Schema discovery analiza corpus → produce spec de tipos + campos sin input previo
2. Schema diff report comparando descubierto vs OntologyNode de Eduardo
3. Extracción end-to-end sin few-shot examples produce nodos YAML
4. Gates de S3.7 corren sobre output extraído (pass o failures accionables)
5. Content diff report: nodos extraídos vs nodos curados (overlap + exclusive)

### SHOULD
1. Discovery corre en <5min (pocos LLM calls)
2. Content diff incluye análisis por decision area

### MUST NOT
1. No modificar los nodos curados de Eduardo
2. No modificar el pipeline de S3.2 (solo permitir examples_dir=None)
3. No generar modelos Pydantic dinámicos

## Testing Strategy

- Unit tests: SchemaSpec/SchemaDiffReport/NodeDiffReport construction + logic
- Unit tests: diff_schemas con specs sintéticas
- Unit tests: diff_nodes con directorios temporales de YAML
- Mock tests: discover_schema con LLM response mockeada
- Integration: pipeline.run(examples_dir=None) con corpus real (marcado slow)
