# S3.3 Design: HITL Curation Skill

## Problem & Value

**Problem:** S3.2 extraction pipeline produces ~100 candidate OntologyNode YAML files. These are LLM-generated and need expert validation before entering the knowledge graph. Without curation, the graph will contain hallucinated concepts, wrong decision areas, missing relationships, and duplicate nodes.

**Value:** Eduardo (domain expert) validates every node in 20-30min conversational sessions. The curated output is the authoritative ontology that feeds S3.4 graph integration. This is the quality gate between automated extraction and production knowledge.

## Complexity Assessment

| Criterion | Rating | Notes |
|-----------|--------|-------|
| Components | 3-4 (moderate) | Curation engine, session state, skill entry, YAML I/O |
| Story points | S | Sequential logic, no novel algorithms |
| External integrations | 1 | Reads S3.2 extracted YAML, writes curated YAML |
| Algorithm complexity | Trivial | Linear node review, file-based persistence |

**Overall: Moderate** (multiple components, each simple)

- **Risk gate:** Low -- file-based state, no LLM calls, pure data manipulation
- **UX gate:** Yes -- conversational workflow. Research done in epic design; Eduardo uses Claude Code sessions, batch-review pattern fits 20-30min windows
- **Integration gate:** Low -- reads `OntologyNode` YAML (S3.2 output), writes `OntologyNode` YAML (S3.4 input)

## Approach

Build a Claude Code slash command skill (`/rai-scaleup-curate`) that loads extracted nodes, presents them in batches to Eduardo, captures accept/reject/edit decisions, persists session state to YAML, and writes curated output.

### Data Flow

```
.scaleup/knowledge/ontology/extracted/*.yaml  (S3.2 output)
         |
         v
   CurationSession.load()
   -> load all extracted nodes + any prior session state
         |
         v  (per node, in batch)
   Present node to Eduardo (formatted display)
   -> Eduardo: accept / reject(reason) / edit(fields)
         |
         v
   CurationSession.record_decision(node_id, decision)
   -> update session state (cursor, decisions)
   -> auto-save to .scaleup/knowledge/ontology/curation-state.yaml
         |
         v  (on session end or all nodes done)
   CurationSession.write_curated()
   -> accepted + edited nodes → .scaleup/knowledge/ontology/curated/*.yaml
   -> summary: N accepted, N rejected, N edited, N remaining
```

### Key Models

```python
# --- Curation decision ---
class CurationDecision(str, Enum):
    ACCEPTED = "accepted"
    REJECTED = "rejected"
    EDITED = "edited"  # accepted with modifications

class NodeDecision(BaseModel):
    node_id: str
    decision: CurationDecision
    reason: str | None = None        # rejection reason
    edits: dict[str, Any] | None = None  # field: new_value for edited nodes
    original_hash: str               # SHA256 of original YAML for drift detection
    timestamp: datetime

# --- Session state (persisted) ---
class CurationSessionState(BaseModel):
    session_id: str                  # e.g. "curation-2026-03-19-001"
    source_dir: str                  # path to extracted YAML dir
    output_dir: str                  # path to curated YAML dir
    total_nodes: int
    cursor: int                      # index of next unreviewed node
    node_order: list[str]            # ordered list of node IDs
    decisions: dict[str, NodeDecision]  # node_id -> decision
    created: datetime
    updated: datetime

# --- Session summary ---
class CurationSummary(BaseModel):
    total: int
    accepted: int
    rejected: int
    edited: int
    remaining: int
```

## Design Decisions (auto-decided, pending human review)

### D1: Curation UX Pattern — category-at-a-time

**Choice:** Present nodes grouped by `decision` area (People, Strategy, Execution, Cash), one node at a time within each group.

**Rationale:** Eduardo is a domain expert who thinks in decision areas. Grouping by decision area maintains cognitive context -- all People concepts together, then Strategy, etc. Within each group, nodes are presented one at a time for focused attention. Batch size (~17 nodes per decision area) fits 20-30min sessions naturally.

**Alternatives considered:**
- *Batch-of-N (e.g., 20 at once):* Information overload. Eduardo would have to context-switch between node types and decision areas within a batch. Harder to maintain consistent quality.
- *One-at-a-time (ungrouped):* Misses the benefit of decision-area context. Eduardo would jump between People, Strategy, Cash randomly.
- *All-at-once table view:* Not feasible in conversational CLI. Better for web UI (out of scope).

### D2: Session Persistence — YAML state file

**Choice:** Single YAML file at `.scaleup/knowledge/ontology/curation-state.yaml` that stores `CurationSessionState`.

**Rationale:** YAML is human-readable (Eduardo or Rai can inspect state), consistent with the rest of the ScaleUp knowledge pipeline (all YAML), trivially serializable via Pydantic + PyYAML (already dependencies). No new deps. One file per curation campaign (not per session).

**Alternatives considered:**
- *SQLite:* Overkill for ~100 nodes + state. Adds dependency. Not human-readable. Would be right for 10K+ nodes.
- *JSON:* Works, but inconsistent with the YAML convention in `.scaleup/knowledge/`. Less readable for nested structures.
- *In-memory only:* Loses state between Claude Code sessions. Eduardo's 14-21 sessions would require restarting each time.

### D3: Curation Output — separate `curated/` directory

**Choice:** Write curated (accepted + edited) nodes to `.scaleup/knowledge/ontology/curated/`, keeping extracted nodes untouched in `extracted/`.

**Rationale:** Separation of concerns. Extracted nodes are the pipeline's output (S3.2). Curated nodes are Eduardo's validated output (S3.3). S3.4 graph builder reads from `curated/`. If re-extraction runs, it overwrites `extracted/` without losing curation decisions. Audit trail: diff between extracted and curated shows all Eduardo's changes.

**Alternatives considered:**
- *Modify in-place:* Destructive. Loses original extraction for comparison. Re-extraction would overwrite curation work.
- *Side-by-side annotations:* Adds complexity (every node needs an annotation file). Harder for S3.4 to consume.
- *Single directory with status field in YAML:* Mixes accepted/rejected/pending. S3.4 would need to filter.

### D4: Integration Approach — slash command skill only

**Choice:** Claude Code slash command `/rai-scaleup-curate` backed by a `.claude/skills/` markdown definition. The skill orchestrates the conversation. Python library provides the engine.

**Rationale:** Eduardo works in Claude Code sessions. A skill is the native interaction pattern. The Python library (`rai_agent.scaleup.curation`) provides the logic that the skill invokes via inline Python or CLI commands. No need for a separate `rai` CLI command -- the curation workflow is inherently conversational (present node, wait for human response, record decision).

**Alternatives considered:**
- *CLI command (`rai scaleup curate`):* Would need to be interactive (multi-step prompts). Doesn't fit the `rai` CLI's fire-and-forget pattern. Would require complex TUI.
- *Both CLI + skill:* Over-engineering. The CLI portion would only handle `--resume` and `--summary`, which the skill handles naturally.
- *Pure library (no skill):* Eduardo would need to know how to call Python functions directly. Skill provides the UX layer.

## Acceptance Criteria

**MUST:**
1. Skill `/rai-scaleup-curate` loads extracted nodes from `.scaleup/knowledge/ontology/extracted/` and presents them for review
2. Each node can be accepted, rejected (with reason), or edited (with schema validation)
3. Session state persists to YAML and resumes correctly across Claude Code sessions
4. Curated (accepted + edited) nodes are written to `.scaleup/knowledge/ontology/curated/` in valid OntologyNode YAML format
5. Session summary displays counts (accepted, rejected, edited, remaining)

**SHOULD:**
1. Nodes grouped by decision area for cognitive coherence
2. Edit tracking records original vs modified values for audit
3. Skip option to defer a node for later review

**MUST NOT:**
1. Must NOT modify extracted YAML files in `extracted/` directory
2. Must NOT require LLM API calls (pure data manipulation + human decisions)
3. Must NOT present already-reviewed nodes on session resume
