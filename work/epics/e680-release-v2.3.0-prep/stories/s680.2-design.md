# S680.2 Design — Changelog & Release Notes

## Approach

Create two documentation artifacts for the v2.3.0 release:

1. **CHANGELOG.md** — Add v2.3.0 entry in Keep a Changelog format with Added/Changed/Fixed/Security sections
2. **Release notes draft** — GitHub release notes with summary, highlights, known issues, and contributor recognition

## Input Sources

| Source | What it provides |
|--------|-----------------|
| `git log v2.2.3..dev` | Complete commit history (~200 commits) |
| Jira tickets tagged v2.3.0 | 36 tickets: features, bugs, security |
| Epic docs (S680.1 output) | E478, E494, E654 documentation |
| Epic design (E680) | Curated changelog draft, known issues list |

## CHANGELOG.md Structure

```
## [2.3.0] - 2026-03-30

### Added
- 5 entries (ACLI adapter, session identity, CLI extensions, ISO 27001, rai doctor)

### Changed
- 3 entries (MCP->ACLI migration, session data path, pattern scope default)

### Fixed
- 12+ entries (individual bugfixes with Jira refs)

### Security
- 3 entries (authlib, PyJWT, astro/cloudflare/undici CVEs)
```

Format: Keep a Changelog 1.1.0. Each entry includes a Jira/epic reference in parentheses.

## Release Notes Structure

```
# raise-cli v2.3.0

Summary paragraph

## Highlights
### New Features (from Added)
### Changes (from Changed)
### Security Fixes (from Security)

## Bug Fixes (from Fixed)

## Known Issues
- RAISE-539, RAISE-634, RAISE-213

## Contributors

## Upgrade Notes
- Breaking: MCP adapter replaced by ACLI
- Breaking: Session data path changed
```

## Complexity Assessment

**Simple** — known format, documentation only, no code changes.

- 2 artifacts (CHANGELOG.md modification, release notes draft creation)
- No external integrations
- No algorithm complexity
- Sources already gathered (S680.1 complete, epic design has curated draft)

## Acceptance Criteria

| ID | Criterion | Verifiable |
|----|-----------|:----------:|
| AC1 | CHANGELOG.md has v2.3.0 entry with Added/Changed/Fixed/Security | Yes |
| AC2 | Every entry traceable to commit, ticket, or epic | Yes |
| AC3 | Keep a Changelog format compliance | Yes |
| AC4 | Release notes draft ready for GitHub release | Yes |
| AC5 | Known issues documented | Yes |
| AC6 | No unreleased/v2.4.0 items included | Yes |

## Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Keep a Changelog format | Already established in CHANGELOG.md, industry standard |
| D2 | Manual curation over automated generation | ~200 commits need grouping; automation deferred to parking lot |
| D3 | Release date 2026-03-30 | Per epic plan; updatable if date shifts |

## Risks

| Risk | Mitigation |
|------|------------|
| Missing bugfix details | Use `git log v2.2.3..dev --oneline` to enumerate all changes |
| Format inconsistency | Follow existing CHANGELOG.md style + Keep a Changelog spec |
