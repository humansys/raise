## 1. Worked Example: Tracing `rai backlog search "RAISE-680"`

This traces the full path from CLI invocation to terminal output.

### Step 1: CLI entry — Typer command

```
src/raise_cli/cli/commands/backlog.py :: search()
```

Typer parses the command:

```
rai backlog search "RAISE-680"
→ query="RAISE-680", limit=50, adapter=None, format="human"
```

The function calls `resolve_adapter(None)` which discovers entry points under `rai.adapters.pm`. With `raise-pro` installed, the `jira` entry point resolves to `rai_pro.adapters.acli_jira:AcliJiraAdapter`. Since it's async, `resolve_adapter` wraps it in `SyncPMAdapter` (from `raise_cli.adapters.sync`).

```python
pm = resolve_adapter(None)  # → SyncPMAdapter(AcliJiraAdapter(...))
results = pm.search("RAISE-680", limit=50)
```

### Step 2: Sync→Async bridge

```
src/raise_cli/adapters/sync.py :: SyncPMAdapter.search()
```

`SyncPMAdapter` bridges the sync CLI call to the async adapter. It detects whether an event loop is running and either uses `asyncio.run()` directly or falls back to a thread-based executor.

### Step 3: Query normalization — `to_jql()`

```
packages/raise-pro/src/rai_pro/adapters/acli_jira.py :: AcliJiraAdapter.search()
```

The adapter normalizes the user query to JQL:

```python
jql = to_jql("RAISE-680")
# "RAISE-680" matches _ISSUE_KEY regex (^[A-Z][A-Z0-9_]+-\d+$)
# Result: "issue = RAISE-680"
```

Other query patterns:
- `"RAISE-680"` → `"issue = RAISE-680"` (issue key detected)
- `"project = RAISE AND status = Done"` → `'project = "RAISE" AND status = Done'` (JQL pass-through with project quoting)
- `"backlog cleanup"` → `'text ~ "backlog cleanup"'` (plain text → full-text search)

### Step 4: Site resolution

```python
site = self._resolve_site_from_jql("issue = RAISE-680")
# No "project = X" in JQL → falls back to default site
# self._default_site = "humansys.atlassian.net" (from jira.yaml)
```

For a query like `project = "RAISE" AND ...`, the adapter extracts `RAISE` from the JQL, looks it up in `self._projects`, finds `instance: humansys`, and resolves to `humansys.atlassian.net`.

### Step 5: Bridge call — subprocess execution

```
packages/raise-pro/src/rai_pro/adapters/acli_bridge.py :: AcliJiraBridge.call()
```

The adapter delegates to the bridge:

```python
result = await self._bridge.call(
    ["workitem", "search"],
    {"--jql": "issue = RAISE-680", "--limit": "50"},
    site="humansys.atlassian.net",
)
```

The bridge:
1. Checks if site differs from `self._current_site` → calls `acli jira auth switch --site humansys.atlassian.net` if needed
2. Builds command: `["acli", "jira", "workitem", "search", "--jql", "issue = RAISE-680", "--limit", "50", "--json"]`
3. Runs `asyncio.create_subprocess_exec(*cmd, stdout=PIPE, stderr=PIPE)`
4. Parses stdout as JSON
5. Emits logfire span: `acli.call workitem search site=humansys.atlassian.net latency=320ms success=true`

### Step 6: ACLI subprocess (external)

The `acli` binary executes against the Jira REST API and returns raw nested JSON:

```json
[
  {
    "key": "RAISE-680",
    "fields": {
      "summary": "Release v2.3.0 prep — epic documentation",
      "status": { "name": "In Progress" },
      "issuetype": { "name": "Epic" },
      "parent": null
    }
  }
]
```

Note: ACLI search returns a **top-level array**, not `{issues: [...]}`.

### Step 7: Response parsing

Back in `AcliJiraAdapter.search()`:

```python
issues = cast("list[dict[str, Any]]", result)  # it's a list
return [IssueSummary(**self._extract_issue_fields(i)) for i in issues]
```

`_extract_issue_fields` extracts from nested format:

```python
{
    "key": "RAISE-680",
    "summary": "Release v2.3.0 prep — epic documentation",
    "status": "In Progress",
    "issue_type": "Epic",
    "parent_key": None,
}
```

Result: `[IssueSummary(key="RAISE-680", summary="Release v2.3.0 prep...", status="In Progress", issue_type="Epic")]`

### Step 8: CLI display

Back in `backlog.py :: search()`:

```python
for issue in results:
    console.print(f"{issue.key} {issue.status:<12} {issue.summary}")
```

Output:
```
RAISE-680 In Progress  Release v2.3.0 prep — epic documentation
```

## 2. Extension Guide

### Adding a new backlog command

Example: adding `rai backlog assign KEY ASSIGNEE`.

**Step 1: Add protocol method** (if it doesn't exist)

Edit `src/raise_cli/adapters/protocols.py` — add to both sync and async protocols:

```python
# In ProjectManagementAdapter:
def assign_issue(self, key: str, assignee: str) -> IssueRef: ...

# In AsyncProjectManagementAdapter:
async def assign_issue(self, key: str, assignee: str) -> IssueRef: ...
```

**Step 2: Add sync wrapper delegation**

Edit `src/raise_cli/adapters/sync.py` — add to `SyncPMAdapter`:

```python
def assign_issue(self, key: str, assignee: str) -> IssueRef:
    return self._run(self._inner.assign_issue(key, assignee))
```

**Step 3: Implement in AcliJiraAdapter**

Edit `packages/raise-pro/src/rai_pro/adapters/acli_jira.py`:

```python
async def assign_issue(self, key: str, assignee: str) -> IssueRef:
    site = self._resolve_site_from_key(key)
    result = await self._bridge.call(
        ["workitem", "edit"],
        {"--key": key, "--assignee": assignee},
        site=site,
    )
    return self._parse_result_envelope(result)
```

**Step 4: Add CLI command**

Edit `src/raise_cli/cli/commands/backlog.py`:

```python
@backlog_app.command()
def assign(
    key: Annotated[str, typer.Argument(help="Issue key")],
    assignee: Annotated[str, typer.Argument(help="Assignee identifier")],
    adapter: AdapterOption = None,
) -> None:
    """Assign an issue to a team member."""
    pm = resolve_adapter(adapter)
    try:
        ref = pm.assign_issue(key, assignee)
    except Exception as exc:
        console.print(f"[red]Error:[/red] {exc}")
        raise typer.Exit(1) from exc
    console.print(f"{ref.key}: assigned → {assignee}")
```

**Step 5: Tests** — write unit tests mocking `AcliJiraBridge.call()`, then integration tests if ACLI is available.

### Supporting a new Jira field

Example: adding `sprint` to search results.

**Step 1:** Check ACLI JSON output includes the field (run `acli jira workitem view KEY --fields *all --json` and inspect).

**Step 2:** Add field to the relevant Pydantic model in `src/raise_cli/adapters/models.py`:

```python
class IssueSummary(BaseModel):
    # ... existing fields ...
    sprint: str | None = Field(default=None, description="Active sprint name")
```

**Step 3:** Extract in `_extract_issue_fields()`:

```python
"sprint": (fields.get("sprint") or {}).get("name"),
```

**Step 4:** Update CLI display format in `backlog.py` search command if desired.

### ACLI command mapping (all 11 protocol methods)

| Protocol method | ACLI command | Key flags | Response format |
|-----------------|-------------|-----------|-----------------|
| `create_issue` | `workitem create` | `--project`, `--summary`, `--type`, `--label` | Full issue `{key, fields}` |
| `get_issue` | `workitem view KEY` | `--fields *all` | Full issue `{key, fields}` |
| `update_issue` | `workitem edit` | `--key`, field flags | Envelope `{results, totalCount}` |
| `transition_issue` | `workitem transition` | `--key`, `--status NAME` | Envelope `{results, totalCount}` |
| `batch_transition` | `workitem transition` (per key) | `--key`, `--status` | Envelope per call |
| `link_to_parent` | `workitem edit` | `--key`, `--parent` | Envelope |
| `link_issues` | `workitem link create` | `--out`, `--in`, `--type` | Plain text (no `--json`) |
| `add_comment` | `workitem comment create` | `--key`, `--body` | Envelope |
| `get_comments` | `workitem comment list` | `--key`, `--limit` | `{comments: [{id, author, body}]}` |
| `search` | `workitem search` | `--jql`, `--limit` | Top-level array `[{key, fields}]` |
| `health` | `auth status` | (none) | Exit code + stdout message |

## 4. Invariants & Contracts

### BacklogAdapter protocol contract

The `AsyncProjectManagementAdapter` protocol (in `src/raise_cli/adapters/protocols.py`) defines 11 methods. `AcliJiraAdapter` implements all 11 via structural typing (no inheritance — just duck typing checked at runtime with `isinstance()`).

| Method | Signature | Return |
|--------|-----------|--------|
| `create_issue` | `(project_key: str, issue: IssueSpec) → IssueRef` | Key + URL |
| `get_issue` | `(key: str) → IssueDetail` | Full issue detail |
| `update_issue` | `(key: str, fields: dict) → IssueRef` | Key + URL |
| `transition_issue` | `(key: str, status: str) → IssueRef` | Key + URL |
| `batch_transition` | `(keys: list[str], status: str) → BatchResult` | Succeeded + failed |
| `link_to_parent` | `(child_key: str, parent_key: str) → None` | — |
| `link_issues` | `(source: str, target: str, link_type: str) → None` | — |
| `add_comment` | `(key: str, body: str) → CommentRef` | Comment ID |
| `get_comments` | `(key: str, limit: int) → list[Comment]` | Comments |
| `search` | `(query: str, limit: int) → list[IssueSummary]` | Summary list |
| `health` | `() → AdapterHealth` | Connectivity status |

Validation: `rai adapter check` verifies protocol compliance at runtime.

### jira.yaml schema requirements

```yaml
# Required top-level keys
default_instance: <string>       # Must match a key in instances
instances:                       # At least one instance required
  <name>:
    site: <string>               # FQDN (e.g., "humansys.atlassian.net")
    email: <string>              # Auth email for this instance
    projects: [<string>, ...]    # Project keys served by this instance

# Optional but expected
projects:
  <PROJECT_KEY>:
    instance: <string>           # Must match a key in instances
    name: <string>
    # Other fields optional (category, description, board_type)

workflow:
  status_mapping: ...            # Legacy (MCP era) — not used by ACLI adapter
  lifecycle_mapping: ...         # Used by hooks for automatic transitions
```

Invariants:
- `default_instance` must reference an existing key in `instances`
- Every project's `instance` field must reference an existing key in `instances`
- Projects listed in `instances.<name>.projects` should have matching entries in `projects` (warning if missing)
- The ACLI adapter uses **convention-based status resolution** (`normalize_status("in-progress") → "In Progress"`), not `status_mapping`

### ACLI binary requirements

- `acli` must be in `PATH`
- Must be authenticated: `acli jira auth status` must exit 0
- For multi-instance: `acli jira auth switch --site X` must succeed for each configured site
- Must support `--json` flag on workitem commands (ACLI v1.x+)
- `workitem link create` does NOT support `--json` — bridge uses `json_output=False`

### JSON output format expectations

ACLI returns **nested Jira API format** — always. The flat/sooperset format branch was deleted in S494.5.

- Issue: `{key: "RAISE-123", fields: {summary: "...", status: {name: "..."}, issuetype: {name: "..."}, ...}}`
- Search: top-level array `[{key, fields}, ...]` (NOT `{issues: [...]}`)
- Create: full issue object `{key, fields}`
- Edit/transition: envelope `{results: [{status, message, id}], totalCount, successCount}`
- Comment list: `{comments: [{id, author, body}]}`
- Link create: plain text (no JSON support)

## 5. Failure Modes

### FM1: ACLI binary not installed

**Symptom:** Any `rai backlog` command fails with:
```
Error: ACLI binary 'acli' not found in PATH. Install it from https://developer.atlassian.com/cli
```

**Detection:** `AcliJiraBridge.call()` catches `FileNotFoundError` from `asyncio.create_subprocess_exec` and raises `AcliBridgeError` with install instructions.

**Fix:** Install ACLI: `brew install atlassian-cli` (macOS) or download from Atlassian Developer site.

### FM2: Authentication expired or missing

**Symptom:** ACLI returns non-zero exit code with stderr like `"Not authenticated"` or `"Token expired"`.

**Detection:** `AcliJiraBridge.call()` checks `proc.returncode != 0` and raises `AcliBridgeError` with the stderr message. `rai backlog search` shows: `Error: Not authenticated`.

**Fix:** Re-authenticate: `acli jira auth login --site humansys.atlassian.net`

**Health check:** `rai adapter check` calls `AcliJiraBridge.health()` which runs `acli jira auth status` and returns `AdapterHealth(healthy=False, message="Not authenticated")`.

### FM3: Multi-instance site resolution fails

**Symptom:** Commands target the wrong Jira instance. Example: searching for `RAI-123` (on rai-agent) hits humansys instead.

**Root cause:** The project key `RAI` is not configured in `jira.yaml` `projects` section, so `_resolve_site()` falls back to `self._default_site` (humansys). A warning is logged: `"Unknown project 'RAI', using default instance"`.

**Detection:** Logger warning at DEBUG level. Issue not found (404 from wrong instance).

**Fix:** Add the project to `.raise/jira.yaml`:
```yaml
projects:
  RAI:
    instance: rai-agent
    name: Rai Dev
```

And ensure the instance is configured and uncommented in `instances:`.

### FM4: JQL syntax errors / reserved word conflicts

**Symptom:** `rai backlog search "project = RAISE AND status = Done"` fails because `RAISE` is a JQL reserved word.

**Detection:** ACLI returns non-zero exit with JQL parse error in stderr.

**Mitigation (built-in):** `to_jql()` calls `_quote_project_values()` which wraps unquoted project values in double quotes: `project = RAISE` → `project = "RAISE"`. This handles the most common case.

**Known gap:** Other reserved words in other positions are NOT auto-quoted. Workaround: manually quote in the query: `rai backlog search 'project = "RAISE" AND ...'`

### FM5: Timeout on large queries

**Symptom:** `rai backlog search` with broad JQL (e.g., all issues in a project) hangs for a long time, then returns partial results or times out.

**Root cause:** ACLI subprocess has no built-in timeout. `asyncio.create_subprocess_exec` waits indefinitely for `proc.communicate()`.

**Mitigation:** Use `--limit` flag: `rai backlog search "project = RAISE" -n 20`. The adapter passes `--limit` to ACLI.

**Future fix:** Add `asyncio.wait_for()` with configurable timeout in `AcliJiraBridge.call()`.

### FM6: Concurrent processes conflict on auth switch

**Symptom:** Multi-instance commands from two concurrent `rai` processes produce wrong results. Process A switches to site X, process B switches to site Y, process A's next command hits site Y.

**Root cause:** `acli jira auth switch` mutates global state in `~/.config/acli/`. The adapter tracks `self._current_site` per-instance, but cannot detect external mutations.

**Mitigation:** Document single-process assumption. In practice, concurrent `rai` processes are rare (one developer, one terminal).

**Future fix:** File lock on ACLI config directory before auth switch, or per-request `--site` flag if ACLI adds support.

### FM7: ACLI JSON output format change

**Symptom:** `json.JSONDecodeError` or unexpected `KeyError` in parsers after ACLI binary update.

**Detection:** `AcliJiraBridge.call()` catches `json.JSONDecodeError` and raises `AcliBridgeError("Failed to parse ACLI JSON output: ...")`.

**Mitigation:** Pin minimum ACLI version. Integration tests (15 E2E tests in `tests/integration/test_acli_jira_integration.py`) validate response format against real Jira.
