# E680: Release v2.3.0 Prep — Developer Documentation

This is a **process epic**, not a code epic. It established a structured, repeatable
release preparation workflow based on a 7-dimension framework derived from 13 industry
sources. The documentation below traces what actually happened, what we learned, and
how to replicate it.

## Section 1: Worked Example — v2.3.0 Release

### End-to-End Trace

The release preparation for v2.3.0 ran over 2 days (2026-03-23 to 2026-03-24) across
6 stories in 3 parallel streams. This section traces the actual sequence of events.

perimeter:

## Section 2: Extension Guide — How to Run Release Prep

### Prerequisites

- Access to Jira project (RAISE) with fixVersion management
- Confluence space (RaiSE1) for epic documentation
- GitLab repo (primary) and GitHub mirror
- `rai` CLI installed with all adapters configured
- `uv` for Python environment management

### Step-by-Step Process

#### Phase 0: Content Audit (NEW — learned from v2.3.0)

Before any documentation work:

1. **List all deliverables** for the release (features, docs, changelog entries)
2. **Tag each deliverable as Pro or Community**
3. **Exclude Pro content** from Community-facing artifacts (changelog, GitHub release notes, user docs)

This step was missing from the original matrix and caused a mid-story scope correction
in S680.3. It is now a required gate for all release types.

#### Phase 1: Scope Audit

```bash
# Query Jira for tickets in the target fixVersion
uv run rai backlog search "fixVersion = '2.3.0' ORDER BY type ASC" -a jira -n 100

# Compare against git log since last release
git log v2.2.3..HEAD --oneline | wc -l

# Tag any untagged tickets
uv run rai backlog update RAISE-XXX --fix-version "2.3.0"
```

**Note:** Jira fixVersion names do NOT have a `v` prefix. Use `2.3.0`, not `v2.3.0`.

#### Phase 2: Quality Gates

```bash
# Core quality (blocking)
uv run pytest                          # Full test suite
uv run pyright                         # Type checking (strict)
uv run pip-audit                       # Dependency CVE scan

# Framework quality (advisory for minor, blocking for major)
uv run ruff check .                    # Linting
uv run rai skill validate              # Skill structure
uv run rai graph build                 # Knowledge graph integrity

# Known issues with rai gate check:
# - `rai gate check --all` may hang (pytest gate timeout bug)
# - gate-lint passes invalid --check flag to ruff (config bug)
# - Run gates individually if --all hangs
```

**Expected non-blocking failures:**
- `rai release check` reports 2 expected failures: twine (not installed) and changelog
  unreleased section (empty when entries are pre-promoted to a version header)

#### Phase 3: Epic Documentation

For each completed epic since last release:

```bash
# Run /rai-epic-docs skill for each epic
# Produces 5 sections: worked example, extension guide, data flow, invariants, failure modes
# Publish to Confluence space RaiSE1
```

**Skip for patch releases.** Required for minor and major.

#### Phase 4: Changelog

Create or update `CHANGELOG.md` in Keep a Changelog 1.1.0 format:

- **Added** — New features (reference epic or ticket)
- **Changed** — Changes to existing features (note if breaking)
- **Fixed** — Bug fixes (reference ticket)
- **Security** — Vulnerability patches (reference CVE and ticket)

**Community filter:** Exclude Pro-only features (Jira/ACLI adapter, MCP servers,
Confluence adapters, ISO 27001).

#### Phase 5: User & Dev Docs

Update `/docs` for user-facing changes. For v2.3.0 this was 3 `.mdx` files.

**Apply the Community filter again.** Each doc file should serve one audience.

#### Phase 6: Smoke Tests

Use changelog Added/Changed entries as the verification matrix:

1. For each Added entry: verify the feature works E2E
2. For each Changed entry: verify the new behavior
3. Always include: clean install test, CLI help, session roundtrip

#### Phase 7: Release Publish

```bash
# 1. Sync skills to package
python scripts/sync-skills.py
# WATCH: Check for duplicate DISTRIBUTABLE_SKILLS list in __init__.py

# 2. Publish release (needs interactive confirmation)
yes | uv run rai release publish --bump minor
# Or: --bump patch, --bump major

# 3. Sync to GitHub mirror
bash scripts/sync-github.sh
# Excludes: packages/, .claude/, work/, scripts/

# 4. Push tag to GitHub (triggers PyPI publish via GitHub Actions)
cd /path/to/github-mirror
git push origin v2.3.0

# 5. Create GitHub Release with release notes
gh release create v2.3.0 --title "v2.3.0" --notes-file release-notes.md
```

#### Phase 8: Jira Cleanup

```bash
# Move incomplete tickets to next version
uv run rai backlog update RAISE-XXX --fix-version "2.4.0"

# Close completed tickets that weren't auto-transitioned
uv run rai backlog transition RAISE-XXX "Done"
```

#### Phase 9: GitHub Issues

- Close issues that are resolved by the release
- Comment on partially-resolved issues with status update

### Validated Release Prep Matrix

The original hypothesis (scope.md lines 101-117) was **mostly correct** for a minor
release. Below is the validated and adjusted matrix:

| Dimension | Patch (2.x.Y) | Minor (2.X.0) | Major (X.0.0) |
|-----------|:---:|:---:|:---:|
| **0. Pro/Community audit** (NEW) | skip | required | required |
| 1. Scope audit (Jira <> git) | auto | auto | auto |
| 2. Changelog (Keep a Changelog) | required | required | required |
| 3. Epic docs (`/rai-epic-docs`) | skip | run per epic | run per epic |
| 4. User/dev docs (`/docs`) | conditional | run (Community only) | run + migration guide |
| 5. Security audit (dep scan) | verify clean | verify clean | full audit |
| 6. Quality gates (test + lint + type) | unit + smoke | full suite + integration | full + install test |
| 7. Release mechanics | PyPI direct | PyPI direct | TestPyPI then PyPI |
| 8. Communication | changelog only | release notes | blog post + migration guide |
| 9. GitHub mirror sync (NEW) | push tag | push tag + release | push tag + release |
| RaiSE: skill validate | skip | run | run |
| RaiSE: graph build | skip | run | run |
| RaiSE: gate check --all | run | run | run |

**Changes from original hypothesis:**

1. **Dimension 0 added:** Pro/Community content audit. Not in original matrix. Discovered
   when S680.3 found Pro content leaking into Community docs mid-story.
2. **Dimension 9 added:** GitHub mirror sync. The GitLab-to-GitHub sync and tag push
   is a separate manual step, not automatic. Without it, PyPI publish does not trigger.
3. **Dimension 7 adjusted:** TestPyPI was correctly skipped for minor (CI is solid).
   Changed minor column from "TestPyPI -> PyPI" to "PyPI direct" based on actual practice.
4. **Dimension 4 clarified:** User docs for minor releases should be Community-only.
   Pro docs ship through separate channels.
5. **Changelog unreleased check:** `rai release check` flags an empty `[Unreleased]`
   section as a failure when entries have been pre-promoted to a version header. This
   is a false positive — document as non-blocking.

## Section 4: Invariants & Contracts

These rules must hold for every release. Violations produce either incorrect releases
or content leaks.

### INV-1: Community Changelog Must Not Reference Pro Features

**Scope:** CHANGELOG.md, GitHub release notes, PyPI description

**Rule:** No mention of Jira adapter, ACLI adapter, MCP servers, Confluence adapters,
or ISO 27001 in Community-facing artifacts.

**Rationale:** Community users cannot access Pro features. Documenting them in
Community artifacts creates confusion and support burden.

**Violated in v2.3.0:** Yes — caught and fixed during S680.3 when the original plan
included ACLI adapter migration docs in the Community user docs. The changelog draft
(design.md) initially included "ACLI Jira adapter with multi-instance support" and
"MCP Jira adapter replaced by ACLI adapter" as entries. These were removed from the
Community changelog and retained only in internal release documentation.

**Detection:** Manual review during Pro/Community content audit (Phase 0).

### INV-2: sync-skills.py Filters by Visibility

**Scope:** `scripts/sync-skills.py`

**Rule:** Only skills with `visibility: public` in their frontmatter are included in
the distributed package. Internal skills never ship.

**Detection:** Diff the `DISTRIBUTABLE_SKILLS` list against skill frontmatter after
running sync.

### INV-3: sync-github.sh Excludes Pro Code

**Scope:** `scripts/sync-github.sh`

**Rule:** The following directories are excluded from the GitHub mirror:
- `packages/` — Pro packages (raise-server, raise-pro)
- `.claude/` — Agent configuration and skills
- `work/` — Epics, stories, research
- `scripts/` — Internal tooling

**Detection:** After sync, verify excluded paths do not exist in the GitHub mirror.

### INV-4: pyproject.toml Is Patched During Sync

**Scope:** `scripts/sync-github.sh`

**Rule:** Workspace references in `pyproject.toml` (pointing to `packages/` subdirs)
are removed during GitHub sync because those packages do not exist in the mirror.

**Detection:** Verify `pyproject.toml` in GitHub mirror has no `path = "packages/..."` entries.

### INV-5: All Core Quality Gates Must Pass Before Release

**Scope:** pytest, pyright, pip-audit

**Rule:** Zero test failures, zero type errors, zero known CVEs. These are blocking
gates. Ruff lint, formatting, and skill validation are advisory for minor releases.

**Clarification:** `rai gate check --all` is the intended single command, but it has
a known timeout bug. Run tools individually as fallback.

### INV-6: Smoke Tests Verify Changelog Entries

**Scope:** S680.5 verification matrix

**Rule:** Each Added/Changed entry in the changelog should have a corresponding smoke
test verification. This ensures we ship what we claim to ship.

**Origin:** PAT-E-692 — discovered during v2.3.0 when expanding from 3 core smoke
tests to 6 changelog-driven verifications.

### INV-7: Tag Must Be Pushed to GitHub to Trigger PyPI Publish

**Scope:** Release mechanics

**Rule:** The git tag (e.g., `v2.3.0`) must be pushed to the **GitHub** remote, not
just GitLab. GitHub Actions triggers the PyPI publish pipeline on tag push events.
A tag that exists only on GitLab will not trigger publication.

**Detection:** After push, verify the tag exists on GitHub: `git ls-remote --tags github | grep v2.3.0`

## Section 5: Failure Mode Catalog

Failure modes discovered during the v2.3.0 release preparation, with symptoms,
detection methods, and fixes.

### FM-1: sync-skills.py Corrupts __init__.py

**Symptom:** The `DISTRIBUTABLE_SKILLS` list appears twice in the generated
`__init__.py`, causing duplicate skill registration or import errors.

**Root cause:** A bug in `sync-skills.py` that appends the list without checking if
it already exists.

**Detection:** After running `sync-skills.py`, inspect the generated `__init__.py`
for duplicate list declarations.

**Fix:** Check diff before committing. If duplicates found, manually remove the
duplicate block or fix the script's idempotency logic.

**Severity:** HIGH — corrupts the package if not caught.

### FM-2: Pro Content in Community Changelog

**Symptom:** Changelog entries reference Jira adapter, ACLI adapter, MCP server
features, Confluence adapter, or ISO 27001 — features not available to Community users.

**Root cause:** Epic design did not tag deliverables by audience (Pro vs Community).
The changelog was drafted from the full feature list without filtering.

**Detection:** Phase 0 Pro/Community content audit. Search changelog for keywords:
`jira`, `acli`, `mcp`, `confluence`, `iso 27001`, `pro`.

**Fix:** Remove Pro entries from Community-facing artifacts. Retain in internal
release documentation only.

**Severity:** MEDIUM — does not break functionality but creates user confusion.

**Pattern:** PAT-E-693

### FM-3: Local main Branch Diverges from origin/main

**Symptom:** `git checkout main` followed by merge causes conflicts because local
main has commits not on remote (or vice versa).

**Root cause:** Local main branch was not updated before starting release mechanics.

**Detection:** Before any merge to main: `git fetch origin && git diff main origin/main`

**Fix:** `git checkout main && git pull origin main` before merging dev.

**Severity:** LOW — easily resolved but wastes time during the publish step.

### FM-4: rai release publish Needs Interactive Confirmation

**Symptom:** `rai release publish --bump minor` prompts for confirmation and hangs
in automated/scripted contexts.

**Root cause:** The publish command has a safety confirmation prompt.

**Detection:** Observed when trying to script the release.

**Fix:** Pipe `yes` to the command: `yes | uv run rai release publish --bump minor`

**Severity:** LOW — easy workaround, but should be documented.

### FM-5: Tag Pushed to GitLab Only, Not GitHub

**Symptom:** Release tag exists on GitLab but PyPI publish never triggers. Package
version on PyPI remains at previous release.

**Root cause:** The release publish command creates the tag on the current remote
(GitLab). GitHub Actions only triggers on tag push events to the GitHub remote.

**Detection:** Check GitHub for tag: `git ls-remote --tags github | grep vX.Y.Z`.
Check PyPI for version: `pip index versions raise-cli`.

**Fix:** Explicitly push tag to GitHub after GitLab sync:
`cd /path/to/github-mirror && git push origin vX.Y.Z`

**Severity:** HIGH — release appears complete but package is not actually published.

### FM-6: Jira fixVersion Name Mismatch

**Symptom:** Jira queries return no results when searching for fixVersion `v2.3.0`.

**Root cause:** Jira fixVersion names do not use a `v` prefix. The correct name is
`2.3.0`, not `v2.3.0`.

**Detection:** Query fails silently (returns empty results, not an error).

**Fix:** Use `2.3.0` in all Jira queries and fixVersion assignments.

**Severity:** LOW — causes confusion during scope audit but no data loss.

### FM-7: MR Description Does Not Propagate to GitHub

**Symptom:** GitLab merge request description and discussion do not appear on GitHub.
The GitHub mirror only contains code, not GitLab metadata.

**Root cause:** `sync-github.sh` mirrors git content only, not platform metadata.

**Detection:** Expected behavior — not a bug, but a gap to be aware of.

**Fix:** Manually create GitHub Release with release notes. Do not rely on MR
description propagating.

**Severity:** LOW — no leak risk (Pro content in MR descriptions stays on GitLab),
but requires manual GitHub Release creation.

## Hypothesis Validation

### Original Hypothesis (scope.md lines 101-117)

The scope document proposed a "Release Prep by Type" matrix mapping 10 dimensions
across patch, minor, and major release types.

### Assessment: Mostly Correct, With Gaps

**What the matrix got right for v2.3.0 (minor):**

1. Scope audit = auto -- confirmed. Jira query + git log comparison.
2. Changelog = required -- confirmed. Keep a Changelog format worked well.
3. Epic docs = run per epic -- confirmed. 3 epics, 3 Confluence pages.
4. User/dev docs = run -- confirmed (with Community-only adjustment).
5. Security audit = verify clean -- confirmed. pip-audit found 0 CVEs.
6. Quality gates = full suite + integration -- confirmed. 3978 tests + 6 smoke verifications.
7. Skill validate = run -- confirmed. Found 2 errors, 11 warnings.
8. Graph build = run -- confirmed. 46k relationships built.
9. Gate check --all = run -- confirmed (with fallback to individual gates).

**What the matrix got wrong or missed:**

1. **Missing Dimension 0: Pro/Community content audit.** This is arguably the most
   important gate for any project with dual licensing or audience separation. It was
   discovered mid-release when S680.3 caught Pro content leaking into Community docs.
   Must be added as the FIRST dimension.

2. **Missing Dimension 9: GitHub mirror sync.** The matrix assumed release mechanics
   end at PyPI publish. In reality, the GitLab-to-GitHub sync and tag push are separate
   manual steps that gate PyPI publication. Without them, the release exists on GitLab
   but not on GitHub/PyPI.

3. **Dimension 7 (Release mechanics): TestPyPI was correctly skipped.** The original
   matrix said minor releases should use "PyPI direct." The research report (Claim 5)
   recommended TestPyPI for all releases but acknowledged this is MEDIUM confidence.
   v2.3.0 validated that PyPI direct is fine for minor releases when CI is solid.

4. **Changelog unreleased section: false positive.** `rai release check` flags an
   empty `[Unreleased]` section as a failure. This happens when changelog entries are
   pre-promoted to a version header (e.g., `## [2.3.0]`) before running the check.
   Document as non-blocking expected failure.

5. **Communication dimension underspecified.** The matrix says "release notes" for
   minor releases. In practice this means: GitHub Release page + Confluence page +
   Jira version closure. Three separate artifacts, not one.

### Revised Confidence

**Overall matrix confidence: HIGH** — the framework is sound. The two missing dimensions
(Pro/Community audit, GitHub mirror sync) are additive, not corrective. The core 7
dimensions from the research report held up under real-world execution.

### Key Learning: Pro/Community Separation Is a Release Gate

The single most important discovery from E680 is that **content audience separation
must be verified before any documentation or communication work begins.** This is not
a documentation concern — it is an architectural boundary that manifests in:

- Changelog entries (what features to list)
- User docs (what to document)
- Release notes (what to highlight)
- Package content (what skills to include)
- GitHub mirror (what code to sync)

The `sync-skills.py` and `sync-github.sh` scripts enforce the boundary for code. But
for documentation and communication, the boundary must be enforced by a manual content
audit (Phase 0) until tooling exists to automate it.
