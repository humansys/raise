## 1. Worked Example

### `rai session start "gemba research" --project . --context`

This traces the full end-to-end flow when a developer starts a named session with context output.

#### Step 1: Prefix Lookup

The CLI loads the developer profile from `~/.rai/developer.yaml` and extracts the developer's prefix via `profile.get_pattern_prefix()` (e.g., `"E"` for Emilio).

It then loads the prefix registry from `.raise/rai/prefixes.yaml`:

```yaml
E: {name: "Emilio Osorio", registered: "2026-03-22"}
```

If the prefix is new, `PrefixRegistry.register()` adds it (idempotent for same developer). If a different developer claims the same prefix, collision resolution suggests an extended prefix (e.g., `"EO"`).

**File:** `.raise/rai/prefixes.yaml` (committed to git)
**Module:** `src/raise_cli/session/prefix.py` → `PrefixRegistry.load()`, `.register()`, `.save()`

#### Step 2: ID Generation

`generate_session_id("E", now=datetime(2026, 3, 22, 14, 30))` produces:

```
S-E-260322-1430
```

Format: `S-{prefix}-{YYMMDD}-{HHMM}` — 16 characters, no coordination needed. Two sessions by the same developer in the same minute would collide, but that is prevented by the active session check.

**Module:** `src/raise_cli/session/identity.py` → `generate_session_id()`

#### Step 3: Active Session Pointer

An `ActiveSessionPointer` is written to `.raise/rai/personal/active-session`:

```json
{"id": "S-E-260322-1430", "name": "gemba research", "started": "2026-03-22T14:30:22.123456"}
```

This carries metadata from start to close — the close command reads it to build the `SessionIndexEntry` without needing to reconstruct state.

**File:** `.raise/rai/personal/active-session` (gitignored)
**Module:** `src/raise_cli/session/index.py` → `write_active_session()`

#### Step 4: Per-Session Directory

A working directory is created at `.raise/rai/personal/sessions/S-E-260322-1430/` for session-scoped files (state.yaml, signals.jsonl, journal.jsonl).

**Module:** `src/raise_cli/config/paths.py` → `get_session_dir()`

#### Step 5: Profile Update

The developer profile is updated: session count incremented, session added to `active_sessions` list. Stale sessions (>24h) trigger a warning.

**Module:** `src/raise_cli/onboarding/profile.py` → `increment_session()`, `start_session()`

#### Step 6: Context Bundle

Because `--context` was passed, `assemble_context_bundle()` runs. It reads:
- Developer profile (identity, communication prefs, coaching)
- Prior session state (work context, progress, narrative)
- Session index entries via `format_recent_sessions()` — reads `.raise/rai/personal/sessions/{prefix}/index.jsonl`
- Memory graph (governance primes, behavioral patterns)

Output is a token-optimized text bundle (~150 tokens) printed to stdout for AI consumption.

**Module:** `src/raise_cli/session/bundle.py` → `assemble_context_bundle()`
**Module:** `src/raise_cli/session/bundle_formatters.py` → `format_recent_sessions()`

#### Step 7: Session Close (later)

When the session ends, `rai session close --state-file /tmp/session-output.yaml --project .`:

1. Reads the active pointer → gets session ID, name, start time
2. Builds `SessionIndexEntry` with close time, summary, outcomes, branch
3. Appends to `.raise/rai/personal/sessions/E/index.jsonl`
4. Clears the active pointer (only if it matches this session ID)
5. Cleans up the per-session working directory

## 2. Extension Guide

### Adding a New Developer Prefix

Prefixes auto-register on first `rai session start`. No manual steps needed.

**What happens automatically:**
1. Developer runs `rai session start "my session" --project .`
2. CLI reads `~/.rai/developer.yaml` for developer name, extracts prefix (first letter of first name, uppercase)
3. `PrefixRegistry.load()` reads `.raise/rai/prefixes.yaml`
4. `PrefixRegistry.register(prefix, name)` checks for collisions:
   - Same developer + same prefix → no-op (idempotent)
   - Different developer + same prefix → raises `ValueError` with suggestion
5. On collision, CLI calls `resolve_collision()` which suggests `{prefix}{last_name_initial}` (e.g., `"E"` → `"EO"`)
6. Registry saved back to `prefixes.yaml`

**Manual override:** Edit `prefixes.yaml` directly (YAML, committed to git). Useful for teams choosing specific prefixes.

```yaml
E: {name: "Emilio Osorio", registered: "2026-03-22"}
J: {name: "Juan Pérez", registered: "2026-03-25"}
AL: {name: "Ana López", registered: "2026-04-01"}
```

### Adding a New Field to Session Index Entries

Session entries are defined by the `SessionIndexEntry` Pydantic model in `src/raise_cli/session/index.py`.

1. **Add the field to the model** with a default value (backward compatible):
   ```python
   class SessionIndexEntry(BaseModel, frozen=True):
       id: str
       name: str
       started: datetime
       closed: datetime | None = None
       type: str = "feature"
       summary: str = ""
       outcomes: list[str] = Field(default_factory=list)
       branch: str = ""
       tags: list[str] = Field(default_factory=list)  # NEW
   ```

2. **Populate the field at close time** in `src/raise_cli/cli/commands/session.py`, in the structured close block where `SessionIndexEntry(...)` is constructed (~line 653).

3. **Add tests** — the model is frozen (immutable), so test construction and serialization round-trip.

4. **Existing index files are safe** — `model_validate()` applies defaults to missing fields. No migration needed.

### Extending the Context Bundle

The context bundle is assembled in `src/raise_cli/session/bundle.py` via `assemble_context_bundle()`. Formatters live in `src/raise_cli/session/bundle_formatters.py`.

To add a new section:

1. **Write a formatter** in `bundle_formatters.py`:
   ```python
   def format_my_section(data: MyDataType) -> str:
       """Format my section for context bundle."""
       if not data:
           return ""
       return f"# My Section\n{data.summary}"
   ```

2. **Wire it into the bundle** in `bundle.py` — add it to the assembly sequence in `assemble_context_bundle()` or register it as a named section in `assemble_sections()`.

3. **If the section should be loadable via `rai session context --sections`**, add it to the section registry in `assemble_sections()`.

```
│   ├─ generate_session_id(prefix)     → "S-E-260322-1430"
│   ├─ write_active_session(pointer)   → .raise/rai/personal/active-session
│   ├─ migrate_flat_to_session()       → moves legacy flat files
│   ├─ mkdir per-session dir           → .raise/rai/personal/sessions/S-E-260322-1430/
│   ├─ start_session(profile)          → updates active_sessions list
│   ├─ save_developer_profile()        → ~/.rai/developer.yaml
│   └─ assemble_context_bundle()       → stdout (if --context)
│       ├─ format_developer_section()
│       ├─ format_work_section()

│       ├─ format_governance_primes()
│       └─ format_narrative()
│
├─ rai session close --state-file ... --project .
│   │
│   ├─ load_developer_profile()
│   ├─ read_active_session()           → .raise/rai/personal/active-session
│   ├─ process_session_close()         → patterns, corrections, state
│   ├─ SessionIndexEntry(...)          → build from pointer + close data
│   ├─ write_session_entry(prefix, entry) → .../sessions/{prefix}/index.jsonl
│   ├─ clear_active_session()          → removes active-session file
│   ├─ cleanup_session_dir()           → removes per-session working dir
│   └─ end_session(profile)            → removes from active_sessions
│
└─ rai session list --project .
    │
    ├─ load_developer_profile()

    └─ display entries                 → stdout
```

### Pydantic Models

| Model | Module | Purpose |
|-------|--------|---------|
| `PrefixEntry` | `session/prefix.py` | Single developer registration: name + date |
| `PrefixRegistry` | `session/prefix.py` | Full registry with register/collision/load/save |
| `SessionIndexEntry` | `session/index.py` | One session record: id, name, started, closed, type, summary, outcomes, branch |
| `ActiveSessionPointer` | `session/index.py` | Transient pointer: id, name, started — bridges start→close |

All models use `frozen=True` (immutable after construction).

### File Paths

| Path | Committed | Purpose |
|------|-----------|---------|
| `.raise/rai/prefixes.yaml` | Yes | Developer prefix registry |
| `.raise/rai/personal/active-session` | No (gitignored) | Current session pointer |
| `.raise/rai/personal/sessions/{prefix}/index.jsonl` | No (gitignored) | Session ledger per developer |
| `.raise/rai/personal/sessions/{session-id}/` | No (gitignored) | Per-session working state |
| `~/.rai/developer.yaml` | N/A (global) | Developer profile + active sessions list |

## 4. Invariants & Contracts

### Session ID Format

```
S-{prefix}-{YYMMDD}-{HHMM}
```

- **Prefix:** 1+ uppercase alphanumeric characters, registered in `prefixes.yaml`
- **Date/time:** UTC-agnostic (local time), minute resolution
- **Uniqueness:** Guaranteed within a developer prefix at minute granularity. Same-minute collision prevented by active session check (cannot start while one is active).

### Prefix Uniqueness

- Each prefix maps to exactly one developer name
- `PrefixRegistry.register()` is idempotent for same (prefix, name) pair
- Different developer + same prefix → `ValueError` with suggested alternative
- Collision resolution: `{prefix}{last_name_initial}` (e.g., `E` → `EO`)
- Path traversal characters (`..`, `/`, `\`) rejected at `get_developer_sessions_dir()`

### Active Pointer Consistency

- Written at session start, cleared at session close
- `clear_active_session(session_id=...)` only clears if the pointer matches the given ID — prevents one session from clearing another's pointer
- Stale pointers (>24h) trigger warnings at next start
- Pointer is JSON (not plain text) — carries name + start timestamp

### Privacy Boundary

| What | Where | Visibility |
|------|-------|------------|
| Prefix registry | `.raise/rai/prefixes.yaml` | Committed — all developers see all prefixes |
| Session index | `.raise/rai/personal/sessions/{prefix}/index.jsonl` | Gitignored — local only by default |
| Active pointer | `.raise/rai/personal/active-session` | Gitignored — local only |
| Working state | `.raise/rai/personal/sessions/{id}/` | Gitignored — local only |
| Developer profile | `~/.rai/developer.yaml` | Global, not in repo |

**Opt-in sharing:** Teams can modify `.gitignore` to commit session indexes. The per-developer directory structure ensures zero merge conflicts.

### Index File Contract

- Format: JSONL (one JSON object per line, newline-terminated)
- Append-only (no in-place edits)
- Malformed lines are skipped with a warning (defensive parsing)
- Missing fields get Pydantic defaults (backward compatible)

## 5. Failure Modes

### Prefix Collision

**Scenario:** Two developers with the same first initial (e.g., Emilio and Elena) join the same project.

**What happens:**
1. First developer registers `E` → succeeds
2. Second developer's `rai session start` calls `registry.register("E", "Elena García")`
3. `ValueError` raised: `"Prefix 'E' already registered to 'Emilio Osorio'. Use 'EG' instead."`
4. CLI catches the error, calls `resolve_collision("E", "Elena García")` → `"EG"`
5. Re-registers with `"EG"` → succeeds
6. Registry saved with both entries

**Risk:** Low. Automatic recovery. Developer sees no error.

### Stale Active Pointer

**Scenario:** Developer starts a session but the process crashes before close (terminal killed, machine restarted).

**What happens:**
- The active pointer file remains at `.raise/rai/personal/active-session`
- Next `rai session start` detects it via `profile.current_session.is_stale()` (>24h)
- Warning printed: "Stale session detected... Previous session was not closed."
- New session starts anyway — the old pointer is overwritten
- The stale session has no index entry (never closed) but working state files may exist

**Mitigation:** The session index only gets written on close. Incomplete sessions leave no trace in the shared registry.

### Missing Prefix File

**Scenario:** Fresh clone of a repository that has never had sessions, or `prefixes.yaml` was deleted.

**What happens:**
- `PrefixRegistry.load(path)` returns an empty registry when the file doesn't exist
- Next `rai session start` auto-registers the developer and creates the file
- No error, no user action needed

### Cross-Repo Leakage (The Original Bug — F4)

**Before E654:** Session IDs used a per-project counter stored in `.raise/rai/personal/sessions/index.jsonl`. The counter was environment-dependent — if you opened the same repo in a second terminal or worktree, the counter could diverge, producing duplicate IDs or gaps. Worse, the active session was tracked in `~/.rai/developer.yaml` (global), so starting a session in repo A could interfere with repo B.

**After E654:**
- Session IDs are timestamp-based — no shared counter, no coordination
- Active session pointer is per-project (`.raise/rai/personal/active-session`), not global
- Session index is per-developer-per-project, not shared across repos
- Two terminals in the same repo: same-minute collision prevented by active session check

### Migration from Old Format

**Decision:** S654.5 (migration) was dropped. Privacy-by-default means old sessions stay in their original location (`.raise/rai/personal/sessions/index.jsonl`). No data is moved or converted.

**What this means:**
- Old `SES-NNN` sessions remain readable by legacy code paths
- `format_recent_sessions()` in `bundle_formatters.py` has a fallback path that reads the legacy flat index
- New sessions use the new format exclusively
- There is no automatic migration — old and new coexist

**If you need old session data:** Read the legacy index directly at `.raise/rai/personal/sessions/index.jsonl` (JSONL, one entry per line, `{"id": "SES-177", "topic": "..."}` format).

## Appendix: KISS Refactor

The original design (from `design.md`) placed the session index in a committed directory (`.raise/rai/sessions/{prefix}/index.jsonl`). During implementation, a privacy review revealed this would make session data (names, summaries, times) visible to all repo collaborators by default.

**Resolution:** All session data moved to `personal/` (gitignored). Only `prefixes.yaml` is committed. Teams that want shared session visibility can opt-in via `.gitignore`. This also simplified the architecture: one directory tree instead of two, net -16 lines of code.
