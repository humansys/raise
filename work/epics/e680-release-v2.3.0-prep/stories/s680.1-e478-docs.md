## 1. Worked Example: Moving the Jira Adapter

This traces how the Jira MCP adapter was moved from `raise-cli` (community/OSS) to `raise-pro` (proprietary) in S478.1.

### Before (pre-E478)

```
raise-commons/
в”ңв”Җв”Җ src/raise_cli/
в”Ӯ   в”ңв”Җв”Җ adapters/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ mcp_jira.py          # Pro adapter вҖ” leaked into OSS wheel
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ mcp_confluence.py    # Pro adapter вҖ” leaked into OSS wheel
в”Ӯ   в”Ӯ   в””в”Җв”Җ ...
в”Ӯ   в””в”Җв”Җ hooks/builtin/
в”Ӯ       в””в”Җв”Җ jira_sync.py         # Pro hook вҖ” leaked into OSS wheel
в””в”Җв”Җ pyproject.toml               # Entry points registered pro code from raise_cli
    # [project.entry-points."rai.adapters.pm"]
    # jira = "raise_cli.adapters.mcp_jira:McpJiraAdapter"
```

Pro code was inside the `raise_cli` package. Running `pip install raise-cli` would install `atlassian-python-api`, `authlib`, `cryptography`, `requests`, `urllib3`, and `certifi` вҖ” none needed by community users.

### After (post-E478)

```

в”ңв”Җв”Җ src/raise_cli/
в”Ӯ   в”ңв”Җв”Җ adapters/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ filesystem.py        # Community adapter (only one)
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ registry.py          # Entry point discovery (unchanged)
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ models.py            # Shared data models
в”Ӯ   в”Ӯ   в””в”Җв”Җ protocols.py         # Protocol contracts (AsyncProjectManagementAdapter, etc.)
в”Ӯ   в””в”Җв”Җ hooks/builtin/
в”Ӯ       в”ңв”Җв”Җ backlog.py           # Community hooks only
в”Ӯ       в”ңв”Җв”Җ telemetry.py
в”Ӯ       в””в”Җв”Җ ...
в”ңв”Җв”Җ packages/raise-pro/
в”Ӯ   в”ңв”Җв”Җ pyproject.toml           # Own entry points, own dependencies
в”Ӯ   в””в”Җв”Җ src/rai_pro/
в”Ӯ       в”ңв”Җв”Җ adapters/
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ acli_jira.py     # Jira adapter (renamed from mcp_jira.py)
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ mcp_confluence.py
в”Ӯ       в”Ӯ   в””в”Җв”Җ acli_bridge.py   # Subprocess bridge for ACLI
в”Ӯ       в”ңв”Җв”Җ hooks/
в”Ӯ       в”Ӯ   в””в”Җв”Җ jira_sync.py     # Pro hook
в”Ӯ       в”ңв”Җв”Җ doctor/
в”Ӯ       в”Ӯ   в””в”Җв”Җ acli.py          # Pro doctor check
в”Ӯ       в””в”Җв”Җ providers/
в”Ӯ           в”ңв”Җв”Җ jira/             # Jira client, models, OAuth, sync
в”Ӯ           в””в”Җв”Җ auth/             # Credential management
в””в”Җв”Җ pyproject.toml               # No pro entry points, no pro deps
```

### What the move involved

1. **File relocation** (commit `7ca93387`): `git mv` of 3 files from `src/raise_cli/` to `src/rai_pro/`.
2. **Entry point migration** (S478.2): Pro entry points moved from root `pyproject.toml` to `packages/raise-pro/pyproject.toml`.
3. **Dependency cleanup** (S478.2): 6 dependencies removed from root `pyproject.toml`, added to `packages/raise-pro/pyproject.toml`.
4. **CLI cleanup** (S478.3): 207 lines of hardcoded Jira status logic removed from community CLI.

### Key insight (PAT-E-004)

Entry point discovery tests forced updating entry points in the same story as the code move (S478.1), not in S478.2 as originally planned. Tests that call `entry_points(group="rai.adapters.pm")` fail immediately if the registered module path doesn't resolve.

## 2. Extension Guide: Adding a New Pro-Only Adapter

### Step-by-step: Adding a hypothetical "Linear" PM adapter

**Step 1: Create the adapter module**

```
packages/raise-pro/src/rai_pro/adapters/linear.py
```

The adapter must implement the `AsyncProjectManagementAdapter` protocol from `raise_cli.adapters.protocols`. Import shared models from `raise_cli.adapters.models`:

```python
from raise_cli.adapters.models import IssueDetail, IssueRef, IssueSpec, IssueSummary
from raise_cli.adapters.protocols import AsyncProjectManagementAdapter
```

**Step 2: Register the entry point**

Edit `packages/raise-pro/pyproject.toml`:

```toml
[project.entry-points."rai.adapters.pm"]
jira = "rai_pro.adapters.acli_jira:AcliJiraAdapter"
linear = "rai_pro.adapters.linear:LinearAdapter"    # вҶҗ add this
```

**Step 3: Add dependencies**

Add any Linear-specific dependencies to `packages/raise-pro/pyproject.toml` under `[project.dependencies]`. Never add them to the root `pyproject.toml`.

**Step 4: Write tests**

Place tests in `tests/` (monorepo root test directory). Tests can import from both `raise_cli` and `rai_pro` since the dev environment installs both.

**Step 5: Verify discovery**

```bash
uv run rai adapter list
# Should show: filesystem (raise-cli), jira (raise-pro), linear (raise-pro)
```

### Entry point groups available for pro extensions

| Group | Protocol/Contract | Current Pro Registrations |
|-------|-------------------|--------------------------|
| `rai.adapters.pm` | `AsyncProjectManagementAdapter` | `jira` вҶ’ `AcliJiraAdapter` |
| `rai.docs.targets` | `AsyncDocTargetAdapter` | `confluence` вҶ’ `McpConfluenceAdapter` |
| `rai.doctor.checks` | `DoctorCheck` | `acli` вҶ’ `AcliCheck` |
| `rai.hooks` | `LifecycleHook` | `jira-sync` вҶ’ `JiraSyncHook` |

### Adding a new entry point group

If you need a group that doesn't exist (e.g., `rai.auth.providers`):

1. Define the protocol in `src/raise_cli/adapters/protocols.py` (community side вҖ” protocols are shared)
2. Add discovery constants in `src/raise_cli/adapters/registry.py`
3. Register the community default (if any) in root `pyproject.toml`
4. Register pro implementations in `packages/raise-pro/pyproject.toml`

### Import direction (one-way dependency)

```
rai_pro  в”Җв”Җimportsв”Җв”Җв–ә  raise_cli    вң…  (pro depends on community)
raise_cli  в”Җв”Җimportsв”Җв”Җв–ә  rai_pro    вқҢ  NEVER (would couple OSS to proprietary)
```

### Discovery flow at runtime

```
User runs: rai adapter list

1. raise_cli.cli dispatches command
2. raise_cli.adapters.registry._discover("rai.adapters.pm")
3. importlib.metadata.entry_points(group="rai.adapters.pm") returns:
   - "filesystem" вҶ’ raise_cli.adapters.filesystem:FilesystemPMAdapter  (always)
   - "jira" вҶ’ rai_pro.adapters.acli_jira:AcliJiraAdapter              (only if raise-pro installed)
4. Each entry point is loaded, validated (must be a class), registered
5. If rai_pro is NOT installed, "jira" entry point simply doesn't exist вҖ” no import error
```

## 4. Invariants & Contracts

### Hard invariants (must never be violated)

| # | Invariant | Enforcement |
|---|-----------|-------------|
| I1 | `raise-cli` source (`src/raise_cli/`, `src/raise_core/`) must never import from `rai_pro` | `import-linter` in dev dependencies; CI gate |
| I2 | `raise-cli` `pyproject.toml` must not list pro-only dependencies | Code review; `pip install raise-cli` smoke test in clean venv |
| I3 | `raise-cli` `pyproject.toml` must not register entry points pointing to `rai_pro.*` | Code review; entry point audit |
| I4 | `raise-pro` depends on `raise-cli`, never the reverse | `pyproject.toml` dependency direction |
| I5 | Protocol contracts live in `raise_cli.adapters.protocols` (shared) | Convention; pro adapters implement these protocols |
| I6 | Shared data models live in `raise_cli.adapters.models` (shared) | Convention; pro adapters import from here |

### Entry point contracts

- **Group names are stable API**: `rai.adapters.pm`, `rai.docs.targets`, `rai.hooks`, `rai.doctor.checks`, etc.
- **Entry point names must be unique across packages**: e.g., only one package can register `jira` under `rai.adapters.pm`
- **Entry points must resolve to classes** (not functions, not modules) вҖ” the registry validates this
- **Failed entry point loads are logged and skipped** вҖ” they never crash the CLI

### Workspace contract

- `packages/raise-pro/` is a UV workspace member via `[tool.uv.workspace] members = ["packages/*"]`
- Dev installs both packages: `raise-pro` is in `[project.optional-dependencies] dev`
- Production installs are separate: `pip install raise-cli` vs `pip install raise-pro`

## 5. Failure Modes

### FM1: raise-cli accidentally imports from rai_pro

**Symptom:** `pip install raise-cli` (without raise-pro) fails with `ModuleNotFoundError: No module named 'rai_pro'` on any CLI command that triggers the import path.

**Detection:** `import-linter` catches this statically. Also caught by installing raise-cli in an isolated venv.

**Fix:** Move the import to the pro package or behind a lazy/entry-point boundary.

### FM2: Pro dependency added to raise-cli pyproject.toml

**Symptom:** `pip install raise-cli` pulls unexpected packages (e.g., `atlassian-python-api`). Users on restricted networks or with security policies may reject the install.

**Detection:** Dependency audit in code review. `rai release check` quality gates.

**Fix:** Move the dependency to `packages/raise-pro/pyproject.toml`.

### FM3: Entry point registered in wrong pyproject.toml

**Symptom:** `rai adapter list` shows pro adapters even without raise-pro installed, but they fail to load with import errors. Or: pro adapters don't appear when raise-pro IS installed.

**Detection:** `rai adapter list` integration test; `rai adapter check` validates protocol contracts.

**Fix:** Ensure entry points in root `pyproject.toml` point only to `raise_cli.*` or `raise_core.*`. Entry points in `packages/raise-pro/pyproject.toml` point only to `rai_pro.*`.

### FM4: Shared model/protocol change breaks pro adapters

**Symptom:** Pro adapters fail `rai adapter check` or raise `TypeError` at runtime after a community release.

**Detection:** The dev environment installs both packages, so `pytest` catches protocol mismatches. `rai adapter check` validates at runtime.

**Mitigation:** Protocols in `raise_cli.adapters.protocols` are versioned by convention. Breaking changes require a coordinated release of both packages.

### FM5: Hatch includes pro code in community wheel

**Symptom:** `pip install raise-cli` wheel contains `rai_pro` files even though they live in `packages/raise-pro/`.

**Detection:** Inspect wheel contents: `unzip -l dist/raise_cli-*.whl | grep rai_pro`

**Root cause (PAT-E-005):** Hatch wheel config requires source code inside the declared package directory. As long as `rai_pro/` lives under `packages/raise-pro/src/` (not under `src/`), Hatch won't include it.

**Fix:** The `[tool.hatch.build.targets.wheel] packages = ["src/raise_cli", "src/raise_core"]` config explicitly limits what goes in the community wheel.

### FM6: Missing sdist configuration for raise-pro

**Noted in retrospective:** `packages/raise-pro/pyproject.toml` does not yet have `[tool.hatch.build.targets.sdist]` include configuration. This must be added before publishing raise-pro to PyPI. Without it, source distributions may include unexpected files or miss required ones.
