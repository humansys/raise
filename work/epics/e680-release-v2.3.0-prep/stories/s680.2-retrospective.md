## What Went Well

1. **E680 design doc had curated changelog draft** — reduced T1 from research to validation. The epic design phase paid off by front-loading categorization.
2. **Sequential task flow natural** — changelog → release notes → verify is the right order. Each task's output fed the next cleanly.
3. **Keep a Changelog format** — structured format made categorization straightforward and ensures consistency across releases.

## What To Change

1. **Automate changelog gathering** — `git log` + Jira query could generate a raw draft. Manual curation still needed for accuracy and context, but the initial gathering is automatable.
2. **Conventional commits would help** — if commit messages followed conventional commit format, categorization into Added/Changed/Fixed/Security could be partially automated.

## Learnings

1. **Manual curation catches context that automation misses** — commit messages often lack the "why" and user-facing impact. Changelog entries need human judgment to translate developer changes into user-relevant descriptions.
2. **Traceability to tickets is essential** — every changelog entry should reference its ticket/epic. This was enforced in T3 verification and caught one entry missing its reference.
3. **Release notes != changelog** — the changelog is a developer artifact (structured, complete). Release notes are a communication artifact (highlights, upgrade guidance, known issues). Different audiences, different content.

## Patterns Extracted

- **Changelog curation process:** Keep a Changelog entries should be manually curated from git log + Jira tickets for accuracy. Automated generation from conventional commits is a future optimization but manual curation catches context that commit messages miss.

## Framework Observations

- A `/rai-changelog` skill could automate the gathering + categorization step, leaving human curation for the final pass.
- Docs stories benefit from the full skill cycle — the plan kept T1/T2/T3 well-scoped even though "just write docs" feels simple.
