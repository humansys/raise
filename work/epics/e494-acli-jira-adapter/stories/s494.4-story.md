# S494.4: Multi-Instance Config and Site Switching

## User Story

As a **framework developer working across multiple Jira instances**,
I want the **ACLI adapter to resolve which Jira site to target per project key**,
so that **`rai backlog` commands work across humansys and rai-agent Jira instances from one config**.

## Acceptance Criteria

### Scenario: jira.yaml with instances section
```gherkin
Given jira.yaml has an `instances:` section mapping site names to URLs
And projects reference their instance via `instance:` field
When AcliJiraAdapter loads config
Then it builds a project → site lookup
```

### Scenario: Site resolution per project key
```gherkin
Given project RAISE maps to humansys.atlassian.net
And project RAI maps to rai-agent.atlassian.net
When I call search("project = RAISE ...")
Then the adapter resolves site from the project key in the query
And passes site to AcliJiraBridge.call()
```

### Scenario: ACLI auth switch when site differs
```gherkin
Given the current ACLI auth is for humansys
When a call targets rai-agent
Then the bridge runs `acli auth switch <site>` before the command
And caches the current site to avoid redundant switches
```

### Scenario: Single-instance backward compatibility
```gherkin
Given jira.yaml has no `instances:` section (current format)
When AcliJiraAdapter loads config
Then it works as before (no site resolution, no auth switching)
```

### Scenario: Unknown project key
```gherkin
Given a project key not listed in jira.yaml
When the adapter tries to resolve the site
Then it proceeds without site switching (best effort)
And logs a warning
```

## Notes

- Bridge already has `site` parameter reserved in `call()` and `health()`
- Site switching via `acli auth switch` mutates global ACLI state — cache to minimize
- Backward compatibility is critical: current single-instance configs must keep working
- `build_url()` already reads `site` from project config — extend, don't replace
