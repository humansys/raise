# S494.6: Integration Test Suite for ACLI Adapter — Design

## Complexity: Moderate

4 components, external integration (real Jira API), reusable pattern for future adapters.

## Problem

We have 61 unit tests for the ACLI adapter that mock the subprocess layer. These validate logic but cannot catch: wrong ACLI flags, JSON response format changes, auth/site switching failures, or Jira API contract mismatches. The epic done criteria require validation against real infrastructure.

## Value

1. **Validates E494 deliverable** — proves the adapter works end-to-end before merge to main
2. **Establishes a replicable pattern** — same structure applies to Confluence, declarative adapters, and future integrations
3. **Regression safety net** — catches ACLI version changes or Jira API drift on-demand

## Approach

**Pytest-native integration tests** using the existing `pytest.mark.integration` marker (already registered). Tests exercise `AcliJiraAdapter` directly (Python API level, not CLI), hitting real Jira via ACLI subprocess. Auto-skip when preconditions unmet.

### Design Decisions

| ID | Choice | Rationale | Alternatives Considered |
|----|--------|-----------|------------------------|
| D1 | Test at adapter level, not CLI level | Isolates adapter contract from CLI arg parsing (already tested). Faster, more focused assertions. | CLI-level via `typer.testing.CliRunner` — slower, harder to assert on model fields |
| D2 | Precondition skip via `pytest.fixture` + `pytest.skip()` | Standard pytest pattern. Composable — multiple preconditions combine naturally. Reusable for Confluence etc. | `skipIf` decorator — less flexible, can't share checks |
| D3 | Session-scoped adapter + test issue fixture | One ACLI auth setup per session. One test issue created/cleaned per session. Avoids Jira rate limits. | Function-scope — creates N issues per run, slower, risks rate limiting |
| D4 | Separate conftest in `tests/integration/jira/` | Keeps Jira-specific fixtures isolated. `tests/integration/conftest.py` stays for shared infra. Future: `tests/integration/confluence/`, etc. | Flat in `tests/integration/` — will get crowded with multiple adapters |
| D5 | Use RAI project for tests, not RAISE | RAI project avoids the JQL reserved word bug. Humansys instance works for multi-instance test via RAISE get (by key, not JQL). | RAISE project — blocked by JQL reserved word bug for search |

### Skip Logic (reusable pattern)

```
Precondition chain (any failure → skip all):
  1. shutil.which("acli") → "ACLI binary not in PATH"
  2. jira.yaml exists → "No Jira config at .raise/jira.yaml"
  3. jira.yaml has instances → "No instances configured"
```

Implemented as a session-scoped fixture that returns a configured `AcliJiraAdapter` or skips. Other adapter suites (Confluence) follow the same pattern: check binary → check config → return adapter or skip.

### Test Issue Lifecycle

```
Session setup:
  1. Create issue: "E494-INTEGRATION-TEST-{timestamp}" in RAI project
  2. Store key in fixture

Tests use the issue (order-independent where possible):
  - get_issue(key) → IssueDetail
  - add_comment(key, body) → CommentRef
  - get_comments(key) → [Comment]
  - update_issue(key, fields) → IssueRef
  - transition_issue(key, "in-progress") → IssueRef

Session teardown:
  1. Transition issue to Done (cleanup)
```

`create_issue` and `search` are tested independently (not tied to the shared fixture).

## Examples

### Precondition fixture (reusable pattern)

```python
import shutil
from pathlib import Path

import pytest

from rai_pro.adapters.acli_jira import AcliJiraAdapter

_JIRA_YAML = Path(".raise/jira.yaml")


@pytest.fixture(scope="session")
def acli_adapter() -> AcliJiraAdapter:
    """Session-scoped ACLI adapter. Skips if preconditions unmet."""
    if not shutil.which("acli"):
        pytest.skip("ACLI binary not in PATH")
    if not _JIRA_YAML.exists():
        pytest.skip(f"No Jira config at {_JIRA_YAML}")
    return AcliJiraAdapter()
```

### Test issue fixture (session lifecycle)

```python
import time

from raise_cli.adapters.models import IssueRef, IssueSpec


@pytest.fixture(scope="session")
def test_issue(acli_adapter: AcliJiraAdapter) -> IssueRef:
    """Create a test issue for the session. Cleaned up at session end."""
    import asyncio

    spec = IssueSpec(
        summary=f"E494-INTEGRATION-{int(time.time())}",
        issue_type="Task",
        labels=["e494-test", "auto-cleanup"],
    )
    ref = asyncio.run(acli_adapter.create_issue("RAI", spec))
    yield ref
    # Cleanup: transition to Done
    try:
        asyncio.run(acli_adapter.transition_issue(ref.key, "done"))
    except Exception:
        pass  # best-effort cleanup
```

### Test example

```python
import asyncio

from raise_cli.adapters.models import IssueDetail


class TestCoreOperations:

    @pytest.mark.integration
    def test_get_issue_returns_detail(
        self, acli_adapter: AcliJiraAdapter, test_issue: IssueRef
    ) -> None:
        detail = asyncio.run(acli_adapter.get_issue(test_issue.key))
        assert isinstance(detail, IssueDetail)
        assert detail.key == test_issue.key
        assert detail.summary.startswith("E494-INTEGRATION-")
        assert detail.status != ""
        assert detail.url.startswith("https://")
```

## Acceptance Criteria

### MUST
- AC1: `tests/integration/jira/test_acli_adapter_e2e.py` exists with tests for all 11 adapter methods
- AC2: Tests skip cleanly with clear message when ACLI or config unavailable
- AC3: `pytest -m integration` runs the suite; default `pytest` excludes it
- AC4: Multi-instance routing tested (at least RAI + RAISE cross-instance get)
- AC5: Test issue created/cleaned per session — no orphaned data

### SHOULD
- AC6: Pattern documented in conftest so Confluence/future adapters follow same structure
- AC7: Health check test included

### MUST NOT
- Touch existing unit tests or integration tests
- Add ACLI or Jira credentials to any committed file
- Make default `pytest` run slower or require external dependencies
