# S494.3: Design — Full Protocol Implementation

## Problem & Value

**Problem:** The 11 `AsyncProjectManagementAdapter` methods only have an MCP implementation (`McpJiraAdapter`). Need ACLI-based implementation to eliminate MCP dependency.

**Value:** `rai backlog` works through ACLI — simpler install, same CLI the user already has, no MCP subprocess in background.

## Complexity

Moderate — 3-4 components, M size, single external integration (ACLI subprocess via bridge).

## Approach

New `AcliJiraAdapter` in `acli_jira.py` delegating to the existing `AcliJiraBridge`. Three parser families:

1. **Read ops** (view, search): simplified `_extract_issue_fields()` (nested-only, no `is_flat` parameter — ACLI always returns nested Jira API format)
2. **Write ops** (create, edit, transition): new `_parse_result_envelope()` for `{results: [{id, status}]}`
3. **Comments**: own parser (ACLI format differs — author is string, created missing)

### Components

| Component | Change | File |
|-----------|--------|------|
| `AcliJiraAdapter` | create | `packages/raise-pro/src/rai_pro/adapters/acli_jira.py` |
| Tests | create | `tests/adapters/test_acli_jira.py` |

### Decisions

| ID | Choice | Rationale | Alternatives |
|----|--------|-----------|-------------|
| D1 | Copy `_extract_issue_fields` **nested-only** (no `is_flat` param) | ACLI always returns nested Jira API format — `is_flat` branching would be dead code. mcp_jira.py gets deleted in S494.5 | Full copy with is_flat (dead code), shared parser module (over-engineering), import from mcp_jira (coupling) |
| D2 | Transition resolves status name by **convention**: `status.replace("-", " ").title()` | Maps CLI slugs ("in-progress") to Jira names ("In Progress") without config lookup. `status_mapping` in jira.yaml is MCP legacy (transition IDs) — ACLI doesn't need it | Config lookup via workflow.states (unnecessary coupling), keep status_mapping (wrong abstraction for ACLI) |
| D3 | Comment.created = "" when ACLI omits it | Model allows empty string — don't invent timestamps | Fetch via workitem view (extra call for minor data) |
| D4 | URL constructed as `https://{site}/browse/{key}` | ACLI returns API URL, not web URL | Use API URL (not user-facing) |

## Method Mapping

| Protocol method | ACLI command | Key flags | Response format |
|-----------------|-------------|-----------|-----------------|
| `create_issue` | `workitem create` | `--project`, `--summary`, `--type` | Result envelope |
| `get_issue` | `workitem view` | `KEY` positional | Nested Jira API |
| `update_issue` | `workitem edit` | `--key`, field flags | Result envelope |
| `transition_issue` | `workitem transition` | `--key`, `--status NAME` | Result envelope |
| `batch_transition` | `workitem transition` (loop) | `--key`, `--status` | Result envelope per key |
| `link_to_parent` | `workitem edit` | `--key`, parent field | Result envelope |
| `link_issues` | `workitem link create` | `--out`, `--in`, `--type` | Result envelope |
| `add_comment` | `workitem comment create` | `--key`, `--body` | Result envelope |
| `get_comments` | `workitem comment list` | `--key`, `--limit` | ACLI comment format |
| `search` | `workitem search` | `--jql`, `--limit` | Array of nested issues |
| `health` | (delegates to bridge) | — | AdapterHealth |

## Examples

### Create issue → result envelope

```python
ref = await adapter.create_issue("RAISE", IssueSpec(summary="Test", issue_type="Task"))
# Bridge: acli jira workitem create --project RAISE --summary "Test" --type Task --json
# Returns: {"results": [{"status": "SUCCESS", "id": "RAISE-99"}], "totalCount": 1}
# Parsed: IssueRef(key="RAISE-99", url="https://humansys.atlassian.net/browse/RAISE-99")
```

### Get issue → nested Jira format

```python
detail = await adapter.get_issue("RAISE-99")
# Bridge: acli jira workitem view RAISE-99 --json
# Returns: {"key": "RAISE-99", "fields": {"summary": "Test", "status": {"name": "Backlog"}, ...}}
# Parsed: IssueDetail(key="RAISE-99", summary="Test", status="Backlog", ...)
```

### Search → top-level array

```python
results = await adapter.search("project = RAISE ORDER BY created DESC", limit=5)
# Bridge: acli jira workitem search --jql "..." --limit 5 --json
# Returns: [{"key": "RAISE-99", "fields": {...}}, {"key": "RAISE-98", ...}]
#   NOTE: array, NOT {"issues": [...]}
# Parsed: [IssueSummary(key="RAISE-99", ...), ...]
```

### Transition → by convention

```python
ref = await adapter.transition_issue("RAISE-99", "in-progress")
# Convention: "in-progress".replace("-", " ").title() → "In Progress"
# Bridge: acli jira workitem transition --key RAISE-99 --status "In Progress" --json
# Returns: {"results": [{"status": "SUCCESS", "id": "RAISE-99"}]}
# Parsed: IssueRef(key="RAISE-99", ...)
```

### Comments → ACLI format

```python
comments = await adapter.get_comments("RAISE-99", limit=5)
# Bridge: acli jira workitem comment list --key RAISE-99 --limit 5 --json
# Returns: {"comments": [{"id": "123", "author": "Emilio Osorio", "body": "test"}]}
# Parsed: [Comment(id="123", author="Emilio Osorio", body="test", created="")]
```

### Error: jira.yaml missing

```python
adapter = AcliJiraAdapter(project_root=Path("/tmp/empty"))
# Raises FileNotFoundError("Jira config not found: /tmp/empty/.raise/jira.yaml")
```

## Acceptance Criteria

### MUST

- **AC1:** All 11 `AsyncProjectManagementAdapter` methods implemented and tested against mocked ACLI responses
- **AC2:** Read ops (get_issue, search) parse nested Jira API format correctly
- **AC3:** Write ops (create, edit, transition) parse result envelope `{results: [{id, status}]}`
- **AC4:** Search handles top-level array (not `{issues: [...]}`)
- **AC5:** Config loaded from `.raise/jira.yaml` for project config; status name resolved by convention (`replace("-", " ").title()`), not config lookup

### SHOULD

- Construct web URL from site + key
- Reuse `_to_jql()` for query normalization

### MUST NOT

- Modify `AsyncProjectManagementAdapter` protocol
- Modify adapter models (`IssueSpec`, `IssueDetail`, etc.)
- Delete `mcp_jira.py` (S494.5)
- Register entry point (S494.5)
