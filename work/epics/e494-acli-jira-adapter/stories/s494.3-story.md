# S494.3: Full Protocol Implementation — AcliJiraAdapter

## User Story

As a **framework developer**,
I want an **AcliJiraAdapter that implements all 11 AsyncProjectManagementAdapter methods via ACLI subprocess calls**,
so that **`rai backlog` commands work through ACLI instead of MCP, with the same interface and observable telemetry**.

## Acceptance Criteria

### Scenario: Create issue
```gherkin
Given ACLI is authenticated and jira.yaml has project config
When I call create_issue with an IssueSpec
Then it executes `acli jira workitem create` with correct flags
And returns an IssueRef with the created key
```

### Scenario: Get issue with full detail
```gherkin
Given a Jira issue KEY-123 exists
When I call get_issue("KEY-123")
Then it executes `acli jira workitem view` with --json
And returns an IssueDetail parsed from nested Jira API format
```

### Scenario: Search with JQL
```gherkin
Given issues matching a JQL query exist
When I call search("project = RAI", limit=10)
Then it executes `acli jira workitem search --jql ... --limit 10 --json`
And returns a list of IssueSummary parsed from the array response
```

### Scenario: Transition by status name
```gherkin
Given an issue in "To Do" status
When I call transition_issue("KEY-123", "In Progress")
Then it executes `acli jira workitem transition --status "In Progress"`
And returns an IssueRef (no transition ID lookup needed)
```

### Scenario: Batch transition
```gherkin
Given three issues need transition
When I call batch_transition(["K-1", "K-2", "K-3"], "Done")
Then it executes transition for each key
And returns a BatchResult with successes and failures
```

### Scenario: Add and list comments
```gherkin
Given an issue KEY-123 exists
When I call add_comment("KEY-123", "test body")
Then it executes `acli jira workitem comment create`
And returns a CommentRef

When I call get_comments("KEY-123")
Then it returns a list of Comment (author as string, created may be None)
```

### Scenario: Link issues
```gherkin
Given two issues exist
When I call link_issues("K-1", "K-2", "blocks")
Then it executes `acli jira workitem link create --out K-1 --in K-2 --type blocks`

When I call link_to_parent("K-2", "K-1")
Then it executes `acli jira workitem edit` to set parent field
```

### Scenario: Health check
```gherkin
Given ACLI is installed and authenticated
When I call health()
Then it delegates to AcliJiraBridge.health()
And returns AdapterHealth with healthy=True
```

### Scenario: Config loading from jira.yaml
```gherkin
Given jira.yaml exists with project and workflow config
When AcliJiraAdapter is instantiated
Then it loads status_mapping and project config
And resolves site per project key for multi-instance support
```

## Notes

- Reuse nested-format parsers from mcp_jira.py (_extract_issue_fields, etc.)
- ACLI search returns array directly, not {issues: [...]} — handle this difference
- Comment `created` field is missing from ACLI output — set to None
- Status transition uses NAME not ID (spike finding D5)
- Delegates all subprocess work to AcliJiraBridge from S494.2
