# S494.1: Spike Report ‚Äî ACLI JSON Mapping

## Summary

**Recommendation: GO** ‚Äî ACLI JSON output maps to existing models with minor
adaptations. Two response formats exist: full-issue (nested Jira API) and
operation-result (success envelope). Both are parseable.

## Findings by Operation

### 1. Search (`workitem search --json`)

**Format:** Array of issues in nested Jira API format.

```
[{ key, id, self, fields: { summary, status: { name }, issuetype: { name },
   priority: { name }, assignee } }]
```

**Maps to:** `IssueSummary` ‚Äî direct mapping via `_extract_issue_fields(is_flat=False)`.

| Model field | JSON path | Status |
| ----------- | --------- | ------ |
| key | `[i].key` | OK |
| summary | `[i].fields.summary` | OK |
| status | `[i].fields.status.name` | OK |
| issue_type | `[i].fields.issuetype.name` | OK |
| parent_key | `[i].fields.parent.key` | OK (when parent exists) |

**Note:** Search returns an array at top level, not `{issues: [...]}` like the
MCP adapter expects. Parser needs adjustment: unwrap from array directly.

### 2. View (`workitem view KEY --json`)

**Format:** Single issue object, nested Jira API format.

```
{ key, id, self, fields: { summary, status, issuetype, priority, assignee,
  labels, parent, created, updated, description, comment: { comments } } }
```

**Maps to:** `IssueDetail` ‚Äî direct mapping via existing nested parsers.

| Model field | JSON path | Status |
| ----------- | --------- | ------ |
| key | `.key` | OK |
| summary | `.fields.summary` | OK |
| status | `.fields.status.name` | OK |
| issue_type | `.fields.issuetype.name` | OK |
| parent_key | `.fields.parent.key` | OK |
| assignee | `.fields.assignee.displayName` | OK (null when unassigned) |
| priority | `.fields.priority.name` | OK |
| labels | `.fields.labels` | OK (array of strings) |
| created | `.fields.created` | OK (ISO 8601) |
| updated | `.fields.updated` | OK (ISO 8601) |
| description | `.fields.description` | **ADF format** (not plain text) |
| url | `.self` | API URL (not web URL) |

**Gaps:**
- `description` is Atlassian Document Format (ADF), not plain text. Need to
  either: (a) extract text from ADF, or (b) pass raw ADF as string. Current
  MCP adapter stores it as-is ‚Äî keep same behavior.
- `url` is the REST API URL, not the web browse URL. Web URL would be
  `https://{site}/browse/{key}`. Minor ‚Äî construct from site + key.

### 3. Comment List (`workitem comment list --key KEY --json`)

**Format:** Different from Jira API nested format.

```
{ comments: [{ id, author, body, visibility }], isLast, maxResults, startAt, total }
```

**Maps to:** `Comment` ‚Äî mostly compatible.

| Model field | JSON path | Status |
| ----------- | --------- | ------ |
| id | `.comments[i].id` | OK |
| body | `.comments[i].body` | OK |
| author | `.comments[i].author` | OK (string, not nested object) |
| created | `.comments[i].created` | **MISSING** ‚Äî ACLI comment list omits `created` |

**Gaps:**
- `author` is a plain string in ACLI (e.g., "Emilio Osorio"), vs nested object
  in raw Jira API. Simpler ‚Äî no need for `.author.displayName` lookup.
- `created` timestamp is not returned by ACLI comment list. Options:
  (a) set to empty string (current model allows it), or
  (b) use `workitem view` with comment fields to get full data.

### 4. Create (`workitem create --json`)

**Format:** Operation result envelope.

```
{ results: [{ status: "SUCCESS", message: "...", id: "RAI-27" }],
  totalCount: 1, successCount: 1 }
```

**Maps to:** `IssueRef` ‚Äî needs adaptation.

| Model field | Source | Status |
| ----------- | ------ | ------ |
| key | `results[0].id` | OK (returns key like "RAI-27", not numeric id) |
| url | not returned | **MISSING** ‚Äî construct from site + key |

**Note:** Does NOT return the full issue. The `id` field in the envelope is
actually the issue key (e.g., "RAI-27"), not a numeric ID. This is sufficient
for `IssueRef`.

### 5. Edit (`workitem edit --key KEY --json`)

**Format:** Same operation result envelope as create.

```
{ results: [{ status: "SUCCESS", message: "...", id: "RAI-27" }],
  totalCount: 1, successCount: 1 }
```

**Maps to:** `IssueRef` ‚Äî same as create.

### 6. Transition (`workitem transition --key KEY --status NAME --json`)

**Format:** Same operation result envelope.

```
{ results: [{ status: "SUCCESS", message: "...", id: "RAI-27" }],
  totalCount: 1, successCount: 1 }
```

**Maps to:** `IssueRef` ‚Äî same as create/edit.

**Key simplification:** Uses status NAME directly ("In Progress"), not transition
ID. The `status_mapping` in jira.yaml becomes unnecessary for ACLI adapter.

## Response Format Summary

| Operation | Format | Wrapper needed |
| --------- | ------ | -------------- |
| search | Array of nested issues | Unwrap array (not `{issues: [...]}`) |
| view | Single nested issue | Direct parse (same as MCP nested branch) |
| comment list | ACLI-specific `{comments: [...]}` | Simple field extraction |
| create | Result envelope `{results: [...]}` | Extract key from `results[0].id` |
| edit | Result envelope | Same as create |
| transition | Result envelope | Same as create |

## Gaps and Mitigations

| Gap | Impact | Mitigation |
| --- | ------ | ---------- |
| Search returns array, not `{issues: [...]}` | Low | Detect array vs object in parser |
| Description is ADF, not plain text | Low | Store as-is (same as current behavior) |
| URL is API URL, not web URL | Low | Construct web URL: `https://{site}/browse/{key}` |
| Comment list missing `created` | Low | Default to empty string (model allows it) |
| Comment `author` is string, not object | Low | Simpler ‚Äî direct assignment |
| Write ops return envelope, not full issue | Low | Extract key from `results[0].id` |

## Architectural Implications

1. **Two response parsers needed:** (a) nested Jira API for read ops, (b) result
   envelope for write ops. Both are simple.
2. **Existing MCP parsers partially reusable:** `_extract_issue_fields` works for
   view/search. `_parse_comments` needs rewrite (different format). Write ops
   need new `_parse_result_envelope()`.
3. **`status_mapping` no longer required:** ACLI resolves transition by name.
   Can keep in config for backward compat but adapter doesn't need it.
4. **Search response shape differs:** Array vs `{issues: [...]}`. Handle with
   type check at parse entry.

## Conclusion

GO. No blockers. The JSON mapping is straightforward with minor adaptations.
The result envelope format for write operations is actually simpler than the
current MCP response parsing (no flat/nested detection needed).
