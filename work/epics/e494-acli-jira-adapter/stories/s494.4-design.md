# S494.4: Design — Multi-Instance Config and Site Switching

## Problem & Value

**Problem:** The ACLI adapter only works against one Jira site. Working across RAISE (humansys) and RAI (rai-agent) requires manual config changes.

**Value:** `rai backlog` commands auto-resolve the target Jira site from the project key — no manual switching.

## Complexity

**Moderate** — 3 components (config schema, adapter resolution, bridge switching), 1 external integration (ACLI subprocess).

## Approach

Extend `jira.yaml` with named instances and explicit project→instance routing. Adapter resolves project key → instance → site at call time. Bridge runs `acli jira auth switch --site X` when target site differs from cached current site.

No backward compatibility needed — no users of the adapter yet. `instances:` is required, fail fast on invalid config.

## Design Decisions

### D1: Schema — dict for `instances:`

Dict keyed by instance name, not list. Already proven in rai repo, enables direct lookup `instances[name]`.

### D2: Routing — A+C redundancy with cross-validation

Each project has `instance:` field pointing to its instance. Each instance has `projects:` listing its project keys. Cross-validation in `_load_config()` warns on mismatch. Prevents silent config drift as config grows.

### D3: Auth — `acli jira auth switch` with caching

ACLI has no per-request `--server` flag. Only mechanism is `acli jira auth switch --site X` (global state mutation). Mitigations:
- Cache current site in bridge, only switch when target differs
- Product-scoped switch (`acli jira auth switch`, not `acli auth switch`)
- No concurrency risk — `rai backlog` is sequential CLI

Research: 7/7 tools use per-instance auth, but ACLI doesn't support it. This is the pragmatic choice given the tool's constraints.

### D4: Site resolution from context

| Input type | Resolution |
|-----------|------------|
| `project_key` param | Direct lookup |
| Issue key `RAISE-123` | Extract prefix before `-` |
| JQL query | Regex `project\s*=\s*(\w+)`, fallback to default |
| Two keys (link) | Source key wins |
| No context (health) | Default instance |

### D5: No backward compatibility, fail fast

`instances:` is required. Project without `instance:` → validation error. `default_instance` at root level for ambiguous resolution (search without project, health).

## Schema Example

```yaml
default_instance: humansys

instances:
  humansys:
    site: humansys.atlassian.net
    email: emilio@humansys.ai
    projects: [RAISE]

  rai-agent:
    site: rai-agent.atlassian.net
    email: emilio@humansys.ai
    projects: [RAI]

projects:
  RAISE:
    instance: humansys
    name: RAISE
    category: Development
    description: RaiSE framework development
    board_type: scrum

  RAI:
    instance: rai-agent
    name: Rai Dev
    board_type: scrum
```

## Concrete Examples

### Config loading
```python
adapter = AcliJiraAdapter(project_root=Path("."))
# Reads .raise/jira.yaml
# Builds: _instances = {"humansys": {"site": "humansys.atlassian.net", ...}, ...}
# Builds: _projects = {"RAISE": {"instance": "humansys", ...}, ...}
# Sets: _default_instance = "humansys"
# Cross-validates: RAISE in instances["humansys"]["projects"] ✓
```

### Site resolution
```python
# From project_key
adapter._resolve_site("RAISE")  # → "humansys.atlassian.net"

# From issue key
adapter._resolve_site_from_key("RAISE-123")  # → "humansys.atlassian.net"
adapter._resolve_site_from_key("RAI-45")     # → "rai-agent.atlassian.net"

# From JQL
adapter._resolve_site_from_jql('project = RAISE AND status = "In Progress"')
# → "humansys.atlassian.net"

adapter._resolve_site_from_jql('text ~ "something"')
# → "humansys.atlassian.net" (default)

# Unknown project
adapter._resolve_site_from_key("UNKNOWN-1")
# → "humansys.atlassian.net" (default) + warning log
```

### Auth switching in bridge
```python
bridge = AcliJiraBridge()
# bridge._current_site = None (unknown at start)

await bridge.call(["workitem", "search"], {"--jql": "..."}, site="humansys.atlassian.net")
# → runs: acli jira auth switch --site humansys.atlassian.net
# → runs: acli jira workitem search --jql "..." --json
# → bridge._current_site = "humansys.atlassian.net"

await bridge.call(["workitem", "view", "RAISE-1"], {}, site="humansys.atlassian.net")
# → NO switch (same site cached)
# → runs: acli jira workitem view RAISE-1 --json

await bridge.call(["workitem", "search"], {"--jql": "..."}, site="rai-agent.atlassian.net")
# → runs: acli jira auth switch --site rai-agent.atlassian.net
# → runs: acli jira workitem search --jql "..." --json
# → bridge._current_site = "rai-agent.atlassian.net"
```

### Error cases
```python
# Missing instances: in jira.yaml
# → raises ConfigError("jira.yaml missing required 'instances' section")

# Project without instance: field
# → raises ConfigError("Project 'FOO' missing required 'instance' field")

# Cross-validation mismatch
# → logs warning: "Project RAISE declares instance 'humansys' but is not in instances.humansys.projects"

# Auth switch fails
# → AcliBridgeError propagates (no silent fallback)
```

## Acceptance Criteria

### MUST

- AC1: `jira.yaml` with `instances:` and `default_instance` parsed into adapter state
- AC2: Project key → instance → site resolution works for all adapter methods
- AC3: `acli jira auth switch --site X` called only when target site differs from cached
- AC4: Cross-validation warns on project↔instance mismatch
- AC5: Missing `instances:` or project without `instance:` raises clear error

### SHOULD

- AC6: `build_url()` uses site from instance config (not hardcoded)

### MUST NOT

- AC7: No silent fallback on config errors — fail fast
- AC8: No `acli auth switch` (global) — only `acli jira auth switch` (product-scoped)
