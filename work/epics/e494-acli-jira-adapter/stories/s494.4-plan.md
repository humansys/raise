## Execution Order

```
T1 (config) → T2 (resolution) → T3 (bridge switch) → T4 (integration)
```

Linear — each task builds on the previous. Risk-first: T1 validates the schema parsing before resolution logic; T3 is the riskiest (subprocess + state mutation) but grounded by T1+T2.

## Risks

| Risk | Mitigation |
|------|------------|
| `acli jira auth switch` has unexpected latency | Cache minimizes switches; measure in T3 |
| Existing tests break with new JIRA_YAML fixture | Update fixture in T1, all tests stay green |
| JQL regex misses edge cases | Fallback to default handles misses safely |
