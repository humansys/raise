# S1136.4 Design â€” `rai learn push` Server Persistence

## Decision: Reuse `agent_events` (Option B)

**Context:** raise-server already has `POST /api/v1/agent/events` (append-only, JSONB payload â‰¤100KB, org-scoped, auth via Bearer token, requires "team" plan). A dedicated `learning_records` table would give strict server-side schema enforcement but requires a new migration, new endpoint, and cross-branch coordination.

**Decision:** Push learning records as agent events with `event_type: "learning_record"` and the full `LearningRecord` dict as `payload`. Validation happens client-side (S1136.2 already enforces the Pydantic model). Server stores as JSONB â€” queryable later.

**Why:**
- Zero server changes â€” no Alembic migration, no new endpoint, no release/3.0.0 PR
- Client-side validation already enforced via `LearningRecord` Pydantic model
- append-only semantics match learning record lifecycle (write once, read many)
- Can evolve to dedicated table later if query patterns demand it

**Trade-off accepted:** No server-side schema enforcement. Mitigated by client validation + `event_type` field for filtering.

## Components

### 1. HTTP Client: `LearningPushClient`

**Location:** `packages/raise-cli/src/raise_cli/memory/push.py` (new file)

Follows the `ApiGraphBackend` pattern in `raise_cli/graph/backends/api.py`:
- `httpx.Client` with Bearer auth
- Timeout: connect=5s, read=30s
- Config from `RAI_SERVER_URL` + `RAI_API_KEY` env vars (same as graph backend)

```python
class LearningPushClient:
    def __init__(self, server_url: str, api_key: str) -> None: ...
    def push(self, record: LearningRecord) -> uuid.UUID: ...
    def close(self) -> None: ...
```

`push()` does:
1. Serialize `LearningRecord` to dict via `model_dump(mode="json")`
2. `POST /api/v1/agent/events` with `{"event_type": "learning_record", "payload": record_dict}`
3. Returns server-assigned UUID from `AgentEventResponse.id`
4. Raises `httpx.HTTPStatusError` on 4xx/5xx (caller handles)

### 2. Push Tracking: `.pushed` marker file

**Mechanism:** After successful push, write a marker file alongside the record:
- Record: `.raise/rai/learnings/{skill}/{work_id}/record.yaml`
- Marker: `.raise/rai/learnings/{skill}/{work_id}/.pushed`
- Content: `{server_id}\n{timestamp}\n` (for debugging, not parsed)

**Why marker file (not field in record.yaml):**
- Record is the canonical data â€” push status is metadata about delivery, not about the learning
- Avoids modifying record.yaml after commit (would dirty working tree)
- Simple existence check: `Path(".pushed").exists()` â†’ skip
- `.pushed` should be in `.gitignore` â€” push state is local per environment

### 3. CLI Command: `rai learn push`

**Location:** Add to existing `packages/raise-cli/src/raise_cli/cli/commands/learn.py`

```
rai learn push <path>           # Push single record
rai learn push --all            # Push all unpushed records
```

**Flags:**
- Positional `path` (optional): path to specific `record.yaml`
- `--all`: scan `.raise/rai/learnings/` for unpushed records
- `--project/-p`: project root (default: `.`)

**Behavior:**
1. Read env vars `RAI_SERVER_URL` + `RAI_API_KEY`
2. If missing â†’ clear error with setup hint, exit 1
3. Read + validate record via `LearningRecord.model_validate()`
4. Check `.pushed` marker â†’ skip if exists (with message)
5. Push via `LearningPushClient.push()`
6. Write `.pushed` marker on success
7. Report result (record ID from server)

**`--all` mode:**
1. `glob(".raise/rai/learnings/*/*/record.yaml")`
2. Filter out those with `.pushed` sibling
3. Push each, report summary (pushed N, skipped M, failed K)
4. Partial failure: report failures but continue others, exit 1 if any failed

### 4. `.gitignore` entry

Add `.pushed` to `.raise/rai/learnings/**/.pushed` in project `.gitignore` â€” push state is per-environment.

|-------|-------------|:---------:|
| `RAI_SERVER_URL` / `RAI_API_KEY` missing | "Server not configured. Set RAI_SERVER_URL and RAI_API_KEY." | 1 |

| 401 Unauthorized | "Authentication failed. Check RAI_API_KEY." | 1 |
| 403 Plan required | "Server requires 'team' plan. Check your license." | 1 |
| 422 Validation error | "Server rejected record: {details}" | 7 |
| Record not found | "No record at {path}" | 1 |
| Already pushed | "âŹ­ Already pushed: {skill}/{work_id}" (skip, exit 0) | 0 |

## Non-Goals

- No server-side changes (zero cross-branch coordination)
- No pull/sync (future story)
- No `--force` flag to re-push (future, if needed)
- No batch API call (one POST per record â€” volume is low, ~5-10 records per epic)
