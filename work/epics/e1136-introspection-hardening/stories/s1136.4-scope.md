# S1136.4 Scope

## In Scope

1. `rai learn push` CLI command (in raise-cli, release/2.4.0)
2. HTTP client to POST learning records to raise-server endpoint
3. Push tracking mechanism (mark records as pushed, skip on re-push)
4. `--file` flag for single record, `--all` flag for batch push
5. Error handling: server unreachable, validation failure, auth failure
6. Configuration: server URL from `.raise/manifest.yaml` or env var

## Out of Scope

- Server-side endpoint implementation (lives on release/3.0.0, separate PR)
- `rai learn list` / `rai learn show` (future)
- Aggregation or metrics dashboard
- Authentication infrastructure (use existing server auth if any)
- Pull/sync from server back to local

## Done Criteria

- [ ] `rai learn push --file <path>` sends record to server and reports result
- [ ] `rai learn push --all` pushes only unpushed records
- [ ] Already-pushed records are skipped (idempotent)
- [ ] Server errors produce clear messages (connection, validation, auth)
- [ ] Tests cover: happy path, server error, already-pushed, validation failure
- [ ] Gemba on raise-server completed before design (understand current state)

## Cross-Branch Note

Server code is on `release/3.0.0`. CLI code is on `release/2.4.0`.
Design phase must clarify whether the server endpoint already exists or needs creation.
If server work is needed, it will be a separate coordinated PR.
