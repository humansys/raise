### T3: `rai learn push` CLI command (M)

**Description:** Add `push` subcommand to `learn_app` typer group. Supports positional `path` for single record and `--all` for batch. Reads `RAI_SERVER_URL` + `RAI_API_KEY` from env.

**Files:**
- Modify: `packages/raise-cli/src/raise_cli/cli/commands/learn.py` (add `push` command)
- Modify: `packages/raise-cli/tests/cli/test_learn.py` (add push command tests)

**TDD cycle:**
- RED: Tests for:
  1. Single push happy path (mock client, assert output + exit 0)
  2. `--all` mode: pushes unpushed, skips pushed, reports summary
  3. Missing env vars: clear error message, exit 1
  4. Server connection error: error message, exit 1
  5. Already pushed record: skip message, exit 0
  6. 401/403 errors: appropriate messages
  7. 422 validation error: exit 7
- GREEN: Implement `push` command with env var reading, client creation, marker logic.
- REFACTOR: Extract env var reading helper if needed.

**AC ref:** All four Gherkin scenarios.

**Verification:**
```
uv run pytest --tb=short
uv run pyright
uv run ruff check
```

**Size:** M | **Dependencies:** T1, T2

### T4: Manual integration test (XS)

**Description:** Validate end-to-end with actual CLI invocation against a mock or real server.

**Steps:**

**Size:** XS | **Dependencies:** T3

## Execution Order

```
T1 (client) → T2 (markers) → T3 (CLI command) → T4 (integration)
```

**Rationale:** Risk-first. T1 is the riskiest (HTTP protocol, payload shape, error handling). T2 is low-risk filesystem. T3 wires everything together. T4 validates end-to-end.

Sequential -- no parallelism opportunities (each task builds on the previous).
