# Story: `rai learn push` -- Server Persistence

## User Story
As a RaiSE agent,
I want a `rai learn push` command that sends learning records to the raise-server endpoint,
so that learning data persists beyond local worktrees and accumulates for metrics.

## Acceptance Criteria

### Scenario: Push a validated learning record
```gherkin
Given a learning record exists at `.raise/rai/learnings/{skill}/{work_id}/record.yaml`
When I run `rai learn push --file <path>`
Then the record is sent to the raise-server `/api/learnings` endpoint
And the server responds with 201 Created
And the CLI reports success with the server record ID
```

### Scenario: Push all unpushed records
```gherkin
Given multiple learning records exist locally
And some have not been pushed to the server
When I run `rai learn push --all`
Then only unpushed records are sent to the server
And each record is marked as pushed locally (idempotent)
```

### Scenario: Server unreachable
```gherkin
Given the raise-server is not reachable
When I run `rai learn push`
Then the CLI reports a clear error with the server URL attempted
And the local record is not modified (retry later)
And the exit code is non-zero
```

### Scenario: Invalid record rejected by server
```gherkin
Given a learning record fails server-side validation
When I run `rai learn push --file <path>`
Then the CLI reports the validation errors from the server
And the exit code is non-zero
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| Valid record.yaml | `rai learn push --file record.yaml` | 201 Created, record ID printed |
| 3 local records, 1 already pushed | `rai learn push --all` | 2 records pushed, 1 skipped |
| Server down | `rai learn push --file record.yaml` | ConnectionError, exit 1 |
| Malformed record | `rai learn push --file bad.yaml` | Validation error, exit 1 |

## Notes

- **Server code lives in `packages/raise-server` on `release/3.0.0`**, not `release/2.4.0`.
  Gemba is needed before design to understand the current server endpoint state,
  available routes, and authentication model.
- Depends on S1136.2 (`rai learn write` + LearningRecord model) -- completed.
- The CLI command lives in `packages/raise-cli` on `release/2.4.0`.
- Server endpoint design may require a separate PR on `release/3.0.0` or coordination.
- Push tracking (marking records as pushed) needs a lightweight mechanism --
  e.g., a `.pushed` marker file or a `pushed_at` field in the record.
- Tracker: RAISE-1239
