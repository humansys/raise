# S476.1: Builtin Gates Read Manifest

## User Story

As a developer using RaiSE on a non-Python project,
I want the builtin gates (TestGate, LintGate, TypeGate, CoverageGate) to read their commands from `.raise/manifest.yaml`,
so that I can configure project-specific test/lint/typecheck commands without modifying Python code.

## Context

The manifest already defines `project.test_command`, `project.lint_command`, and `project.type_check_command`.
Five skills already use a manifest-first pattern with fallback. However, the four builtin gate classes
in Python code hardcode `pytest`, `ruff`, and `pyright` — they never consult the manifest.

## Acceptance Criteria

1. TestGate reads `project.test_command` from manifest; falls back to `pytest` if absent
2. LintGate reads `project.lint_command` from manifest; falls back to `ruff check .` if absent
3. TypeGate reads `project.type_check_command` from manifest; falls back to `pyright` if absent
4. CoverageGate reads `project.test_command` from manifest (coverage wraps test command); falls back to `pytest` if absent
5. When `.raise/manifest.yaml` does not exist, all gates behave exactly as they do today (backwards compatible)
6. All changes covered by TDD (RED-GREEN-REFACTOR)

## Size
