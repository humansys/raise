# Design: S338.9 — Stack-aware MCP recommendations

## Complexity: Moderate

Governance data file + skill updates + language detection from discovery artifacts.

## Problem

Developers don't know which MCP servers are useful for their stack. Missing tools = missing capabilities. This is a governance concern, not a convenience feature.

## Design Decisions

### D1: Catalog lives in governance (`.raise/mcp/catalog.yaml`)

The stack-to-MCP mapping is governance data — it defines what capabilities are available. Lives in `.raise/mcp/catalog.yaml` alongside the per-server configs.

### D2: Stack detection — artifact-first with scan fallback

1. Read `work/discovery/analysis.json` if it exists (instant)
2. If missing, infer languages from file extensions in the project (fast glob, no full scan)
3. Never fail — always produce recommendations

Note: `analysis.json` doesn't contain explicit languages — we derive them from file extensions in the `components[].file` field. Fallback uses `EXTENSION_TO_LANGUAGE` mapping directly via glob.

### D3: Skills read the catalog, not hardcode

- `/rai-mcp-add` reads `catalog.yaml` for package suggestions (replaces hardcoded table)
- `/rai-mcp-status` reads `catalog.yaml` + detected languages to show "missing capabilities"
- Single source of truth for known MCP servers

## Approach

### 1. Create `.raise/mcp/catalog.yaml`

```yaml
# MCP Server Catalog — known servers and stack recommendations
# Governance: missing servers impact agent capabilities
servers:
  context7:
    package: "@upstash/context7-mcp"
    type: npx
    description: "Documentation lookups for any library"
    recommended_for: all  # universal capability

  github:
    package: "@modelcontextprotocol/server-github"
    type: npx
    env: [GITHUB_TOKEN]
    description: "GitHub repository operations"
    recommended_for: all

  filesystem:
    package: "@modelcontextprotocol/server-filesystem"
    type: npx
    description: "Local file system access"
    recommended_for: []  # opt-in only

  fetch:
    package: "mcp-server-fetch"
    type: pip
    module: "mcp_server_fetch"
    description: "HTTP fetch for web content"
    recommended_for: [python]
```

### 2. Update `/rai-mcp-status` to show recommendations

After health table, add:

```
Missing capabilities for your stack (python, typescript):

| Server   | Description                    | Add with            |
|----------|--------------------------------|----------------------|
| context7 | Documentation lookups          | /rai-mcp-add context7 |
| github   | GitHub repository operations   | /rai-mcp-add github   |

Use /rai-mcp-add to install any of these.
```

### 3. Update `/rai-mcp-add` to read catalog

Replace hardcoded well-known servers table with catalog lookup. When human says "I want Context7", look up in `catalog.yaml` for package, type, env, module.

### Components

| Component | Change | Location |
|-----------|--------|----------|
| `catalog.yaml` | Create | `.raise/mcp/catalog.yaml` |
| `/rai-mcp-status` | Modify | Add recommendations section |
| `/rai-mcp-add` | Modify | Read catalog for suggestions |

## Acceptance Criteria

### MUST
- `catalog.yaml` exists in `.raise/mcp/` with known servers and `recommended_for` mapping
- `/rai-mcp-status` shows missing capabilities based on detected stack
- `/rai-mcp-add` reads catalog for package/type/env suggestions
- Already-registered servers excluded from recommendations

### SHOULD
- Detect languages from discovery artifact first, glob fallback
- `recommended_for: all` for universal tools (context7, github)

### MUST NOT
- Never auto-install — recommendations only, human decides
- Never fail if catalog is missing — degrade gracefully
