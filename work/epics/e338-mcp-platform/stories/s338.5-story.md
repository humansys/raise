# User Story: S338.5 — Declarative adapter `server.ref`

## Story

**As a** skill author defining a declarative MCP adapter,
**I want to** reference a registered MCP server by name via `server.ref`,
**so that** I don't duplicate connection config between `.raise/mcp/` and `.raise/adapters/`.

## Acceptance Criteria (Gherkin)

### Scenario 1: Adapter with server.ref resolves from registry

```gherkin
Given a registered MCP server "context7" in .raise/mcp/context7.yaml
And a declarative adapter YAML with server.ref: "context7"
When the adapter is constructed
Then it creates a bridge using the registry server's command, args, and env
```

### Scenario 2: Adapter with inline server still works (backwards compat)

```gherkin
Given a declarative adapter YAML with server.command: "uvx" and server.args: ["mcp-github"]
And no server.ref field
When the adapter is constructed
Then it creates a bridge using the inline command and args
```

### Scenario 3: Validation rejects config with neither ref nor command

```gherkin
Given a declarative adapter YAML with server.args: ["test"] only
And no server.ref and no server.command
When the YAML is parsed
Then a ValidationError is raised
```

### Scenario 4: Ref to nonexistent server raises error

```gherkin
Given a declarative adapter YAML with server.ref: "nonexistent"
And no server named "nonexistent" in the MCP registry
When the adapter is constructed
Then a McpBridgeError is raised with message containing "nonexistent"
```

## Size

S (Small) — schema model + adapter resolution + tests
