# S338.1 Design: MCP Infrastructure + Registry

## Problem

McpBridge is generic infrastructure but lives in `rai_cli.adapters/` and imports `AdapterHealth` from domain models. This couples infrastructure to domain, preventing a clean MCP registry layer.

## Value

After this story, `rai_cli.mcp` is the canonical home for MCP infrastructure. New code (S338.2-7) imports from there. Existing code is unbroken via re-export shim.

## Approach

Structural refactoring: create new package, move bridge, extract shared models, add registry. No new behavior except `discover_mcp_servers()`.

### Components

| Component | Change | Notes |
|-----------|--------|-------|
| `src/rai_cli/mcp/__init__.py` | CREATE | Public API exports |
| `src/rai_cli/mcp/bridge.py` | CREATE (move) | McpBridge, McpBridgeError from `adapters/mcp_bridge.py` |
| `src/rai_cli/mcp/models.py` | CREATE | McpHealthResult, McpToolResult, McpToolInfo (moved from bridge) |
| `src/rai_cli/mcp/schema.py` | CREATE | ServerConnection, McpServerConfig |
| `src/rai_cli/mcp/registry.py` | CREATE | discover_mcp_servers() |
| `src/rai_cli/adapters/mcp_bridge.py` | MODIFY | Becomes re-export shim |
| `src/rai_cli/adapters/declarative/schema.py` | MODIFY | ServerConfig → uses ServerConnection |
| `src/rai_cli/adapters/declarative/__init__.py` | MODIFY | Update re-exports |
| `src/rai_cli/adapters/mcp_jira.py` | MODIFY | Import from `rai_cli.mcp` |
| `src/rai_cli/adapters/mcp_confluence.py` | MODIFY | Import from `rai_cli.mcp` |
| `src/rai_cli/adapters/declarative/adapter.py` | MODIFY | Import from `rai_cli.mcp` |

### Consumer migration strategy

3 source consumers + 4 test files import from `adapters.mcp_bridge`. Two approaches:

**Option chosen:** Update source imports to `rai_cli.mcp.bridge`, keep test imports via shim (less churn in tests). Shim stays permanently for external consumers.

## Examples

### McpHealthResult (new, replaces AdapterHealth in bridge)

```python
# src/rai_cli/mcp/models.py
from pydantic import BaseModel, Field

class McpHealthResult(BaseModel):
    server_name: str = Field(..., description="MCP server identifier")
    healthy: bool = Field(..., description="Whether server responded")
    message: str = Field(default="", description="Status or error message")
    latency_ms: int | None = Field(default=None, description="Response latency in ms")
    tool_count: int = Field(default=0, description="Number of tools discovered")
```

### ServerConnection (shared between MCP and declarative)

```python
# src/rai_cli/mcp/schema.py
from pydantic import BaseModel, Field

class ServerConnection(BaseModel):
    command: str = Field(..., description="Server command (e.g. 'uvx', 'npx')")
    args: list[str] = Field(default_factory=list, description="Command arguments")
    env: list[str] | None = Field(default=None, description="Env var names to forward")
```

### McpServerConfig (new)

```python
# src/rai_cli/mcp/schema.py
class McpServerConfig(BaseModel):
    name: str = Field(..., description="Server name (used in rai mcp commands)")
    description: str | None = Field(default=None, description="Human-readable description")
    server: ServerConnection
```

## Acceptance Criteria

### MUST
- McpBridge importable from `rai_cli.mcp.bridge`
- Re-export shim at `rai_cli.adapters.mcp_bridge` works
- Bridge returns `McpHealthResult`, not `AdapterHealth`
- `ServerConnection` shared by mcp schema and declarative schema
- `discover_mcp_servers()` parses `.raise/mcp/*.yaml` correctly
- All existing tests pass

### SHOULD
- Source consumers (`mcp_jira`, `mcp_confluence`, `declarative/adapter`) migrated to new import path

### MUST NOT
- No behavior changes to McpBridge (only location + health return type)
- No changes to adapter protocols
- No new CLI commands (S338.2)
