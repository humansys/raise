# S338.1: MCP Infrastructure + Registry

## User Story (Connextra)

**As a** RaiSE developer,
**I want** the McpBridge to live in its own infrastructure package with its own models,
**So that** I can register and use any MCP server without coupling to domain adapters.

## Acceptance Criteria (Gherkin)

### AC1: Bridge moved to new package
```gherkin
Given McpBridge exists in rai_cli.adapters.mcp_bridge
When I move it to rai_cli.mcp.bridge
Then importing from rai_cli.mcp.bridge works
And importing from rai_cli.adapters.mcp_bridge still works (re-export shim)
And all existing tests pass
```

### AC2: Own health model
```gherkin
Given McpBridge.health() currently returns AdapterHealth
When I create McpHealthResult in rai_cli.mcp.models
Then McpBridge.health() returns McpHealthResult
And McpHealthResult has: server_name, healthy, message, latency_ms, tool_count
And the bridge no longer imports from rai_cli.adapters.models
```

### AC3: Shared ServerConnection
```gherkin
Given ServerConfig in declarative/schema.py has command, args, env
When I create ServerConnection in rai_cli.mcp.schema
Then declarative/schema.py imports and uses ServerConnection
And McpServerConfig uses ServerConnection
And no duplication of connection fields
```

### AC4: MCP server config schema
```gherkin
Given a YAML file at .raise/mcp/context7.yaml with name, description, server
When I parse it with McpServerConfig
Then validation passes for valid configs
And validation fails for missing required fields
```

### AC5: Registry discovery
```gherkin
Given .raise/mcp/ contains context7.yaml and snyk.yaml
When I call discover_mcp_servers()
Then it returns {"context7": McpServerConfig(...), "snyk": McpServerConfig(...)}
And invalid YAML files are skipped with warning
And non-existent directory returns empty dict
```
