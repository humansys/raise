# Feature: Minimal Agent Config — RaiSE Owns Its Context

> **Epic**: E15 - Ontology Graph Refinement
> **Complexity**: moderate | **SP**: 3

---

## 1. What & Why

**Problem**: CLAUDE.md (~300 lines) and CLAUDE.local.md (~77 lines) are manually-maintained copies of governance content that already lives in RaiSE artifacts. CLAUDE.local.md requires manual edits after every session — the #1 source of maintenance friction. The context bundle works but is too thin (~150 tokens) — deadlines, recent sessions, and progress metrics are missing because developer.yaml fields are empty.

**Value**: RaiSE owns its own AI agent configuration. CLAUDE.md becomes a ~20-line platform adapter. CLAUDE.local.md becomes a ~2-line bootstrap. The context bundle carries everything a session needs. Zero manual file edits between sessions. Platform-agnostic by construction.

---

## 2. Approach

**Two artifacts replace the monoliths:**

1. **`.raise/agent.md`** — RaiSE-owned agent configuration (code standards, architecture, toolchain, terminology). Generated by `raise init`. Static — changes only when governance changes.

2. **Rich context bundle** — Expanded from ~150 to ~400 tokens. Adds recent sessions, progress metrics. Dynamic — assembled fresh each session.

**CLAUDE.md** shrinks to safety rules + git practices + pointer to `/session-start`.
**CLAUDE.local.md** shrinks to a bootstrap pointer.

## 3. Design Decisions

### D1: What stays in CLAUDE.md?

Pure platform adapter — bootstrap pointer only:

```markdown
# RaiSE Project

Run `/session-start` to load context and governance.
```

3 lines. ALL governance (safety rules, git practices, code standards, architecture) lives in `.raise/agent.md`. Nothing should happen before `/session-start` runs — the pointer makes this explicit.

**Rationale**: CLAUDE.md is a platform adapter, not a governance carrier. RaiSE owns all governance in its own config. Platform agnosticism means we can't rely on CLAUDE.md for anything beyond bootstrapping.

### D2: What goes in `.raise/agent.md`?

Everything from current CLAUDE.md — ALL governance:
- Project identity + philosophy
- Development principles (constitution summary)
- Values table
- **Critical rules (NEVER/ALWAYS lists)**
- Full code standards (type safety, linting, testing, security, docs)
- Architecture (stack, pattern, directory structure)
- Toolchain (pre-commit, quality checks)
- **Git practices (platform, branching, co-authorship)**
- Terminology (deprecated → canonical)
- Golden data (sources of truth)
- Jidoka (stop on defects)
- References table
- Inference economy guidelines

**Loaded by**: session-start skill (Step 1.5 — read once per conversation).

### D3: How does the session-start skill load agent.md?

Two-step context loading:

```
Step 1: Load Context Bundle (dynamic — session state)
  uv run raise session start --project "$(pwd)" --context

Step 1.5: Load Agent Config (static — governance)
  Read .raise/agent.md

Step 2: Interpret & Present
```

**Rationale**: Bundle stays token-optimized (~400 tokens, dynamic). Agent.md is read as a file (static, ~200 lines). Clean separation of concerns.

### D4: What expands in the context bundle?

Add to `bundle.py`:

| New section | Source | Tokens |
|-------------|--------|--------|
| Recent sessions (last 3) | `sessions/index.jsonl` | ~100 |
| Epic progress (SP, %) | `session-state.yaml` | ~20 |
| Completed epics | `session-state.yaml` | ~20 |

New `SessionState` fields:
```python
class EpicProgress(BaseModel):
    epic: str           # "E15"
    stories_done: int   # 5
    stories_total: int  # 8
    sp_done: int        # 16
    sp_total: int       # 25

class SessionState(BaseModel):
    current_work: CurrentWork
    last_session: LastSession
    pending: PendingItems
    notes: str = ""
    # NEW
    progress: EpicProgress | None = None
    completed_epics: list[str] = []  # ["E1", "E2", ...]
```

### D5: How does session-close capture progress?

The session-close skill outputs progress in its state file:

```yaml
# Existing fields...
summary: "..."
current_work: {epic: E15, story: S15.8, ...}
# NEW fields
progress:
  epic: E15
  stories_done: 6
  stories_total: 8
  sp_done: 19
  sp_total: 25
completed_epics: [E1, E2, E3, E4, E7, E8, E9, E11, E12, E13, E14]
```

The skill computes this from epic scope during retrospective. CLI writes it to session-state.yaml.

### D6: Recent sessions — where to read from?

`sessions/index.jsonl` — already exists, append-only. Read last 3-5 entries in `bundle.py`.

### D7: CLAUDE.local.md — what stays?

```markdown

Run `/session-start` for context.
```

Two lines. Platform adapter tells Claude Code this is a RaiSE project.

## 5. Acceptance Criteria

### MUST

- [ ] Context bundle includes recent sessions (last 3), epic progress, completed epics
- [ ] `.raise/agent.md` exists with full governance config (code standards, architecture, toolchain)
- [ ] CLAUDE.md is ≤ 5 lines (bootstrap pointer only)
- [ ] CLAUDE.local.md is ≤ 3 lines (bootstrap pointer)
- [ ] `SessionState` model has `progress` and `completed_epics` fields
- [ ] Session-close writes progress to session-state.yaml
- [ ] Session-start skill loads agent.md
- [ ] Full lifecycle (start → work → close → start) with zero manual file edits
- [ ] Tests pass, >90% coverage on new code

### SHOULD

- [ ] Bundle stays under ~500 tokens total
- [ ] Deadlines populated in developer.yaml (via session-close or manual seed)

### MUST NOT

- Break existing session-start/close CLI interface
- Require hook changes
- Leave governance split between CLAUDE.md and agent.md (all governance in agent.md)
