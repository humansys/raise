## What Surprised Us

1. **Site evolved significantly** — now includes Starlight docs integration, not just the original landing page. Good context for S356.2 (Cloudflare deployment)
2. **QR false positive** — the quality review agent flagged `output: 'static'` in astro.config.mjs as incompatible with SSR pages. In Astro 5, static mode supports per-page opt-out via `export const prerender = false`. The build passing confirmed this was a false positive
3. **rai-epic-start skill bug** — discovered during this session that the skill was doing manual backlog.md edits instead of using `rai backlog` CLI

## Heutagogical Questions

### What did I learn?
QR agents operate with frozen knowledge — they may not know about framework version changes. Astro 5 changed how output modes work compared to Astro 4. Always verify critical QR findings against actual build/runtime behavior before investing time in fixes.

### What would I do differently?
Run the build verification earlier in the review cycle. The build succeeding immediately disproved the QR finding — if we had run it first, we could have dismissed the false positive without investigation.

### What pattern emerged?
PAT-E-005: QR agents may not know current framework versions. Verify critical findings against actual build/runtime before acting on them.

### What should I teach?
Framework version awareness matters for automated review. When a QR agent flags something, check: (1) does the build pass? (2) does the runtime behave correctly? (3) has the framework changed this behavior in recent versions? Trust the build over the agent's knowledge.
