# S356.3: Cloudflare Pages Deployment — Design

## Problem

The website source now lives in raise-commons (migrated in S356.1/S356.2) but deployment still targets raise-gtm. We need a CI/CD pipeline that deploys from this monorepo.

## Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Use `wrangler pages deploy` via GitHub Actions | More control than Cloudflare's built-in GitHub integration; works with monorepo (site/ subdirectory) |
| D2 | Trigger only on `site/**` path changes to main | Avoid deploying on Python-only changes; saves CI minutes |
| D3 | Secrets: `CLOUDFLARE_API_TOKEN` + `CLOUDFLARE_ACCOUNT_ID` | Standard Cloudflare auth pattern for CI; tokens scoped to Pages + D1 |

## Components

### 1. `.github/workflows/deploy-site.yml`

GitHub Actions workflow:
- **Trigger**: push to `main` with paths `site/**`
- **Steps**: checkout -> install Node 22 -> `npm ci` in site/ -> `npm run build` -> `npx wrangler pages deploy dist/ --project-name=raise-website`
- **Secrets**: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`
- **Working directory**: `site/`

### 2. `site/DEPLOYMENT.md`

Manual steps documentation:
- How to create Cloudflare API token (permissions: Pages edit, D1 edit)
- How to set GitHub secrets
- How to repoint Cloudflare Pages project from raise-gtm to raise-commons
- D1 database: no migration needed (same database_id in wrangler.jsonc)
- Manual deploy command for emergencies

### 3. `site/wrangler.jsonc` — Verification

Current config is correct:
- `name`: "raise-website" — matches Cloudflare project name
- `pages_build_output_dir`: "./dist" — matches Astro's default output
- `d1_databases`: binds "raise-waitlist" with existing database_id
- `compatibility_date` and `nodejs_compat` flag present

No changes needed to wrangler.jsonc.
