# S369.4 Retrospective -- Secrets & Hygiene Scan

## What Went Well

- Secrets scan was clean -- no hardcoded credentials anywhere in the codebase. Good hygiene already in place (env vars, parameter passing).
- Internal references scan caught exactly two hardcoded `/home/emilio` paths in docstring examples. Small fix, high value for open source credibility.
- Distribution audit confirmed clean packaging: 265 wheel files all under `raise_cli/`, no internal directories leaked.
- Plan's sequential T1-T4 ordering worked well -- each scan built confidence for the next.
- No TDD required (scanning/config story) -- correctly identified in planning, kept execution lean.

## What Could Improve

- Could have used trufflehog/gitleaks for full git history scan instead of grep-based approach. The grep approach covers current state well but doesn't scan history for rotated secrets. Acceptable trade-off given scope, but CI integration of a proper scanner should follow.
- IDE directory patterns (.idea/, .vscode/) should have been in .gitignore from project inception. These are standard patterns that every project needs.

## Epic-Level Observations (Final Story)

This was the last story of E369. The full epic arc worked well:
- S369.1-2 (XS quick wins) cleaned docs and metadata first, giving S369.4 a clean baseline to scan
- S369.3 hardened CI independently
- S369.4 provided final audit confidence before publish

The epic achieved its goal: a senior dev evaluating the repo in 15 minutes would find accurate docs, professional metadata, working CI, and no leaked internals. The remaining gap is coverage (29%), which was correctly scoped out of this epic.

## Parking Lot

## Patterns

- No new reusable patterns. Grep-based scanning is sufficient for current-state hygiene checks on small-to-medium repos. For git history scanning, a dedicated tool (trufflehog/gitleaks) is needed.
