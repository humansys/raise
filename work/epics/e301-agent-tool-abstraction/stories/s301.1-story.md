# S301.1: Protocols + Models

## User Story

**As** an adapter implementor,
**I want** extended protocol contracts and shared Pydantic models for PM and docs operations,
**So that** concrete adapters (Jira, Confluence) have a typed, stable interface to implement.

## Acceptance Criteria

```gherkin
Scenario: Extended PM protocol is backward-compatible
  Given the existing ProjectManagementAdapter protocol with 4 methods
  When S301.1 extends it with batch, comments, links, search, and health methods
  Then all existing implementations still satisfy the protocol
  And new methods are independently callable

Scenario: Extended Docs protocol adds read and search
  Given the existing DocumentationTarget protocol with 2 methods (can_publish, publish)
  When S301.1 extends it with get_page and search methods
  Then existing implementations still satisfy the protocol

Scenario: New models are importable and validated
  Given new Pydantic models (IssueDetail, IssueSummary, Comment, etc.)
  When adapter code imports them from rai_cli.adapters.models
  Then models validate input data and serialize to compact dict/JSON

Scenario: AdapterHealth model supports health checks
  Given an AdapterHealth model with name, healthy, message, latency_ms
  When an adapter reports its health
  Then the model captures connection status and optional latency
```

## SbE Examples

| Input | Expected |
|-------|----------|
| `IssueDetail(key="RAISE-1", summary="Test", status="Done", issue_type="Story")` | Valid model, `parent_key=None`, `labels=[]` |
| `BatchResult(succeeded=[IssueRef(key="A-1")], failed=["A-2: not found"])` | 1 succeeded, 1 failed string |
| `AdapterHealth(name="jira", healthy=True, latency_ms=42)` | `message=""` default |
| Existing `IssueRef(key="X-1")` | Still works unchanged |
