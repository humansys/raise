# Design: S301.2 — rai backlog CLI group

## Problem

No hay CLI para operar sobre el backlog a través del adapter abstraction (E301).
Los comandos actuales en `backlog.py` son un demo acoplado a `rai_pro` que no usan el patrón de adapters.

## Value

Fricción diaria reducida: los desarrolladores (y agentes AI) pueden crear, transicionar, buscar y gestionar issues desde terminal, desacoplados del PM concreto.

## Approach

Reescribir `backlog.py` limpio con 7 comandos que delegan 100% a `ProjectManagementAdapter` (protocolo OSS de S301.1). Los 4 comandos demo se eliminan.

### Components Affected

| Component                               | Change  | Notes                                          |
| --------------------------------------- | ------- | ---------------------------------------------- |
| `src/rai_cli/cli/commands/backlog.py` | Rewrite | Borrar demo, 7 nuevos comandos                 |
| `tests/cli/test_backlog_auth.py`      | Delete  | Tests del demo, ya no aplican                  |
| `tests/backlog/`                      | Create  | Tests de resolve_adapter, batch parsing, smoke |

## Design Decisions

### D1: Borrar comandos demo

Los 4 comandos existentes (auth, pull, push, status) hacen hard-import de `rai_pro`. Son un demo previo al adapter architecture. Se eliminan como parte de S301.2. No hay usuarios externos dependiendo de ellos.

### D2: Patrón open-core — Plugin Discovery (Pattern B)

Basado en [research de open-core CLI patterns](../../research/open-core-cli-patterns/evidence-catalog.md):

- Los comandos `rai backlog` son OSS, delegan a adapter descubierto vía entry points
- El adapter concreto (JiraAdapter, GitHubAdapter) viene en paquetes separados (rai-pro u otros)
- Mismo patrón que Docker CLI plugins, click-plugins, y que RaiSE ya usa para governance parsers y graph backends

### D3: Auto-detect adapter

`resolve_adapter()` busca entry points en grupo `rai.adapters.pm`:

- 1 registrado → lo usa automáticamente
- 0 registrados → error: "No PM adapter installed. Install rai-pro or register one via entry points."
- 2+ registrados → error: "Multiple PM adapters found: [names]. Use --adapter `<name>` to select."
- `--adapter <name>` flag en todos los comandos como override

Equivalente JavaEE: ServiceLocator con auto-wiring cuando hay un solo bean.

### D4: Output inline

Los modelos Pydantic (IssueSummary, IssueRef, etc.) ya son el filtro de señal/ruido. El CLI formatea con f-strings directas. Sin capa de formatter separada. La prioridad es señal/ruido, no tamaño fijo de tokens.

### D5: Testing — solo lógica real

Siguiendo PAT-E-444 (no testear glue/wrappers):

- `resolve_adapter()`: 4 cases (0 adapters, 1 adapter, 2+ adapters, flag override)
- Parsing de batch-transition (split de keys por coma, edge cases)
- 1 smoke test: `rai backlog --help` lista los 7 comandos
- NO: CliRunner test por cada comando (sería testear Typer, no nuestro código)

### D6: Stub injection — solo en tests

El stub (`StubPM` de S301.1) se inyecta vía monkeypatch en tests. El CLI de producción no conoce el stub. Equivalente JavaEE: `@MockBean` en Spring — inyección solo en contexto de test.

### D7: Firmas completas

Cada comando expone todos los campos relevantes como flags opcionales desde el inicio. No se requiere encadenar create + update. Equivalente: POST endpoint con body completo donde solo `summary` es obligatorio.

### D8: Project key obligatorio en create

`--project RAISE` es flag obligatorio en `create`. Sin magia de leer del manifest por ahora. Simple first — iteramos si la fricción lo justifica.

## Architecture Review Findings

### AR1: Blast radius confirmed safe

`backlog_app` only imported in `main.py:13` and registered at `main.py:52`. Other "backlog" imports in `governance/parsers/` are the markdown backlog parser — completely separate. Rewriting `cli/commands/backlog.py` has no side effects.

### AR2: test_backlog_auth.py must be deleted

Tests the demo commands (auth with mock of `rai_pro`). Orphaned when demo commands are removed.

### AR3: Registry returns classes, not instances

`get_pm_adapters()` returns `dict[str, type]` (e.g., `{"jira": JiraAdapter}`). `resolve_adapter()` must instantiate with no-arg constructor. If instantiation fails, error with clear message. Config injection is S301.3's responsibility — in S301.2, no real adapters are registered (only stub via monkeypatch in tests).

### AR4: `link` command uses only `link_issues()`

Protocol has both `link_to_parent()` and `link_issues()`. The `link` command exposes only `link_issues(source, target, link_type)` — generic linking. Parent linking is handled via `create --parent`. This keeps the CLI as pure glue with no branching logic.

### AR5: Query format is adapter-specific

`search` accepts `query: str` — which is JQL for Jira, different syntax for other PMs. Not a design problem, but `--help` text should note that query format depends on the configured adapter.

## Command Signatures

```
rai backlog create SUMMARY --project KEY [--type TYPE] [--labels L1,L2] [--parent KEY] [--description TEXT]
rai backlog transition KEY STATUS
rai backlog update KEY [--summary TEXT] [--labels L1,L2] [--priority TEXT] [--assignee TEXT]
rai backlog link SOURCE TARGET LINK_TYPE
rai backlog comment KEY BODY
rai backlog search QUERY [--limit N]
rai backlog batch-transition KEY1,KEY2,... STATUS
```

All commands accept `--adapter NAME` as override (D3).

## Examples

### Create

```
$ rai backlog create "Fix auth bug" --project RAISE
Created: RAISE-999

$ rai backlog create "Design API" --project RAISE --type Story --labels backend,urgent --parent RAISE-301
Created: RAISE-1000
```

### Transition

```
$ rai backlog transition RAISE-123 done
RAISE-123: transitioned → done
```

### Search

```
$ rai backlog search "status=Open" --limit 5
RAISE-100 Open         Fix auth bug
RAISE-101 Open         Add unit tests
RAISE-102 Open         Update docs
```

### Batch transition

```
$ rai backlog batch-transition RAISE-1,RAISE-2,RAISE-3 done
3/3 transitioned → done
```

### Error: no adapter

```
$ rai backlog search "status=Open"
Error: No PM adapter installed. Install rai-pro or register one via entry points.
```

### Error: multiple adapters

```
$ rai backlog search "status=Open"
Error: Multiple PM adapters found: jira, github. Use --adapter <name> to select.
```

## Acceptance Criteria

### MUST

1. `rai backlog --help` lists all 7 commands
2. All 7 commands delegate to ProjectManagementAdapter — zero business logic in CLI
3. resolve_adapter() auto-detects single adapter, errors on 0 or 2+, respects --adapter flag
4. Demo commands (auth, pull, push, status) removed
5. Tests pass: resolve_adapter (4 cases) + batch parsing + 1 smoke

### SHOULD

1. Error messages are clear and actionable (tell user what to do)
2. Output is one-liner per action (high signal/noise)

### MUST NOT

1. No import of rai_pro anywhere in backlog.py
2. No formatter layer or token-counting logic
3. No testing of Typer glue (CliRunner per command)
