# Story: McpBridge + McpJiraAdapter

## User Story
As a RaiSE CLI user,
I want `rai backlog` commands to execute against real Jira via an MCP bridge,
so that I can manage my backlog with compact CLI commands instead of raw MCP tool calls.

## Acceptance Criteria

### Scenario: Transition an issue via rai backlog
```gherkin
Given mcp-atlassian is installed and configured with Jira credentials
And issue RAISE-XXX exists in Jira with status "In Progress"
When I run `rai backlog transition RAISE-XXX done`
Then the issue transitions to "Done" in Jira
And the CLI prints compact output with key, status, and duration
```

### Scenario: Search issues via rai backlog
```gherkin
Given mcp-atlassian is configured
When I run `rai backlog search "project=RAISE AND status=Done" --limit 5`
Then compact issue summaries are returned (key, summary, status, type)
And token count is <500 for 5 results
```

### Scenario: MCP server not installed
```gherkin
Given mcp-atlassian is NOT installed on PATH
When I run any `rai backlog` command
Then a clear error is shown: "MCP server 'mcp-atlassian' not found. Install: pip install mcp-atlassian"
And the CLI exits with code 1
```

### Scenario: Health check
```gherkin
Given McpJiraAdapter is registered as entry point
When the adapter's health() method is called
Then it reports healthy=True if mcp-atlassian connects and responds
And it reports healthy=False with error message if connection fails
```

### Scenario: Telemetry spans emitted
```gherkin
Given logfire-api is installed (no-op shim)
When any MCP tool is called via McpBridge
Then a logfire span "mcp.tool_call" is created with server, tool, duration_ms, success attributes
And if logfire full SDK is not installed, the span is a no-op (zero overhead)
```

## Examples (Specification by Example)

| CLI Command | MCP Tool Called | Key Arguments |
|-------------|----------------|---------------|
| `rai backlog transition RAISE-301 done` | `jira_transition_issue` | `issue_key=RAISE-301, transition_id=41` |
| `rai backlog create --type Story "New feature"` | `jira_create_issue` | `project_key=RAISE, summary=..., issue_type=Story` |
| `rai backlog search "status=Done"` | `jira_search` | `jql=status=Done, limit=50` |
| `rai backlog comment RAISE-301 "WIP note"` | `jira_add_comment` | `issue_key=RAISE-301, body=WIP note` |
| `rai backlog get RAISE-301` (via update) | `jira_get_issue` | `issue_key=RAISE-301` |

## Notes
- Design decisions D1-D5 in `design.md` apply
- McpBridge is generic — used by S301.5 (McpConfluenceAdapter) too
- Status name → transition ID mapping via `.raise/jira.yaml` lifecycle_mapping
- `mcp-atlassian` is an external process, not a pip dependency of rai-cli
- logfire-api is base dep (no-op), logfire full SDK is optional [observability] extra
