# S301.7: E2E Dogfood — Design

## Complexity: Moderate
- 6 components, 2 external integrations (Jira Cloud, Confluence Cloud)
- Integration gate (PAT-E-539): real infrastructure required, no mocks

## What & Why

**Problem:** S301.1–S301.6 built the full agent tool abstraction stack (protocols, CLI, adapters, bridge, hooks), but it has never been exercised end-to-end against real Jira and Confluence. Contract mismatches, auth issues, and payload problems can only surface with real calls.

**Value:** Validates the epic's core promise — an AI agent can complete a full story lifecycle using only `rai backlog` and `rai docs` commands, with ≥10x token reduction vs raw MCP calls.

## Approach

This is a **validation story**, not a code story. The deliverable is:
1. A structured E2E test script executed manually against live infrastructure
2. Token measurement comparing CLI output vs raw MCP JSON
3. Bug fixes for anything discovered (fixed inline)
4. Friction log documenting gaps for S301.8/S301.9

No new features. No new tests (beyond bug fixes). The E2E run IS the validation.

### Components Exercised

| Component | Commands | Change Type |
|-----------|----------|-------------|
| `rai backlog` CLI | create, transition, comment, search | exercise (fix bugs if found) |
| `rai docs` CLI | publish, get, search | exercise (fix bugs if found) |
| McpJiraAdapter | all 11 PM methods via CLI | exercise |
| McpConfluenceAdapter | 5 docs methods via CLI | exercise |
| McpBridge | stdio transport, tool calls | exercise |
| JiraSyncHook | lifecycle_mapping events | exercise |

## Examples

### E2E Script: Jira Lifecycle

```bash
# 1. Create a dogfood issue
rai backlog create "S301.7 Dogfood Test" -p RAISE -t Task -d "E2E validation of agent tool abstraction"

# 2. Search to verify creation
rai backlog search 'project = "RAISE" AND summary ~ "S301.7 Dogfood"'

# 3. Transition to In Progress
rai backlog transition RAISE-XXX in-progress

# 4. Add a comment
rai backlog comment RAISE-XXX "Dogfood validation in progress — testing CLI abstraction layer"

# 5. Transition to Done
rai backlog transition RAISE-XXX done

# 6. Verify final state via search
rai backlog search 'project = "RAISE" AND key = RAISE-XXX'
```

Expected: Each command returns compact output (~100-200 tokens), issue moves through full lifecycle.

### E2E Script: Confluence Docs

```bash
# 1. Publish a governance artifact
rai docs publish roadmap --title "E301 Dogfood Test"

# 2. Search for the published page
rai docs search "E301 Dogfood"

# 3. Get the page content
rai docs get <page-id-from-search>
```

Expected: Page created/updated in Confluence, retrievable by search and direct get.

### E2E Script: Lifecycle Hooks

```bash
# Verify hook registration
uv run python -c "from importlib.metadata import entry_points; eps = entry_points(group='rai.lifecycle.hooks'); print([e.name for e in eps])"

# Hook fires on WorkStartEvent (verify via Jira state change)
# This happens implicitly when /rai-story-start runs — verify the transition happened
```

### Token Measurement

Compare output of:
```bash
# CLI (compact)
rai backlog search 'project = "RAISE" AND key = RAISE-XXX' 2>&1 | wc -c

# vs raw MCP equivalent (verbose) — capture from MCP tool call output size
# Reference: ~8,000 tokens for a typical Jira search result via raw MCP
# Target: ~200-500 tokens for equivalent CLI output
```

## Acceptance Criteria

### MUST
1. Full Jira lifecycle executed via CLI: create → search → transition(×2) → comment → verify
2. Confluence publish + search + get executed via CLI
3. All commands return without errors against live infrastructure
4. Token reduction documented with concrete measurements (target: ≥10x)
5. Any bugs found are fixed and committed with tests

### SHOULD
1. Lifecycle hooks verified (JiraSyncHook fires on event)
2. Friction log captured as input for S301.8

### MUST NOT
1. No new features added (scope creep)
2. No mock-based tests pretending to be E2E
3. Dogfood issue not left in inconsistent state (clean up after)
