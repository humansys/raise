## What Went Well

- **Research-grounded design.** Open-core CLI patterns research (7 sources) validated Pattern B and gave confidence to delete demo commands cleanly.
- **Interactive design session.** 8 decisions taken with human, each explained with JavaEE analogies. Developer learned Python concepts in context.
- **Arch review caught real issues.** AR3 (registry returns classes not instances) and AR4 (link_to_parent vs link_issues) would have caused confusion during implementation.
- **Clean blast radius.** Deleting 593 lines of demo code and its test file with zero side effects — confirmed by grep analysis.
- **PAT-E-444 applied.** 16 tests total, all testing real logic. Zero tests for Typer glue.

## What to Improve

- **Design questions need framing first.** 4 questions were rejected because they lacked context for a non-Python developer. Pattern captured: PAT-E-553 (explain-then-ask).
- **Lint tests too.** QR found 4 lint issues in test files because `ruff check` was only run on `src/`. Pattern captured: PAT-E-555.
- **StubPM duplication.** `_StubPM` in test_adapter_resolve.py duplicates StubPM from test_protocols.py. Accepted for now (1 usage), but worth watching if more test files need it.

## Heutagogical Checkpoint

1. **Learned:** Pattern B (Plugin Discovery) is the established open-core pattern. RaiSE already follows it. Auto-detect singleton is the simplest DX.
2. **Process change:** Frame design choices with "why this matters" before presenting options. Especially for stakeholders who don't know the tech stack.
3. **Framework improvement:** `resolve_adapter()` pattern is reusable for S301.4 (docs CLI) and any future entry point group.
4. **More capable:** Can now confidently design CLI groups that bridge OSS/pro without coupling.
