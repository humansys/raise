# S301.6: Skill Auto-Sync Hooks — Retrospective

> **Date:** 2026-03-01 (SES-306)
> **Size:** S (estimated 30 min, actual ~18 min)
> **Velocity:** 1.67x

## Summary

Built JiraSyncHook — auto-sync work lifecycle events to Jira transitions. Zero-friction PM sync: when lifecycle skills emit work:start/work:close events, Jira issues transition automatically based on lifecycle_mapping in jira.yaml.

## What Went Well

- **Design decisions eliminated ambiguity:** 4 interactive questions resolved all forks before implementation. Zero rework.
- **Hook system extensibility:** Adding a new hook was almost mechanical — the E248 architecture (ADR-039) has clean extension points.
- **Test-first paid off:** 16 unit tests + 2 integration tests, all green first try after implementation. No debugging cycles.
- **Module-level Path constant as test seam:** `_JIRA_YAML_PATH` made config readers testable without constructor injection complexity.

## What To Improve

- **TelemetryHook doesn't subscribe to work events:** New work:start/work:close events aren't tracked by telemetry. Not blocking, but a gap. Parking lot item.
- **Skill wiring deferred:** The hook infrastructure works, but no skill emits these events yet. S301.7 (dogfood) is the real validation.

## Heutagogical Checkpoint

1. **Learned:** Hook extension pattern is fully mechanical: typed event + subscriber + entry point + error isolation. No existing code changes needed.
2. **Process change:** None needed — interactive design + fast implementation worked perfectly.
3. **Framework improvement:** TelemetryHook should auto-subscribe to new event types, or at least have a review gate when new events are added.
4. **More capable of:** Building hook-based cross-cutting concerns. Pattern reusable for Slack, git automation, etc.
