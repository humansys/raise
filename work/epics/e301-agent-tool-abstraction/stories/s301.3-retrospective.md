# S301.3 Retrospective: McpBridge + McpJiraAdapter

> **Story:** S301.3 | **Size:** M | **Sessions:** SES-299 (T1), SES-300 (T2-T4)
> **Estimated:** ~2h | **Actual:** ~2 sessions (~2.5h combined)
> **Branch:** `story/s301.3/mcp-bridge-jira-adapter`

## Summary

Built the MCP bridge infrastructure and Jira adapter end-to-end:
- T1: McpBridge (generic async MCP client, 19 tests)
- T2: McpJiraAdapter (11 async methods, config, 18 tests)
- T3: resolve_adapter auto-wrap + entry point (2 tests)
- T4: Smoke test against real Jira — found and fixed 4 bugs

Total: 44 story-specific tests, 2990 full suite, ruff + pyright clean.

## What Went Well

- **TDD discipline held** — RED-GREEN-REFACTOR on all tasks, commit after each
- **Architecture review paid off** — AR-S3-1 (_parse_result), AR-S3-2 (link_to_parent), AR-S3-5 (comment_limit) all resolved cleanly during implementation
- **Dual-format parsers** — handling both sooperset and raw Jira API formats makes the adapter resilient to server changes
- **D6 architectural insight** — smoke test conversation led to valuable direction: adapter for domain logic, `rai mcp call` for pass-through ops

## What to Improve

- **Smoke test timing** — T4 should come earlier. A quick probe after T1 would have validated the real server response format before writing 18 mock-based tests. The mocks assumed raw Jira API format (`fields.summary`, `issuetype`) but sooperset uses flat format (`summary`, `issue_type`). Cost: ~20min rewriting parsers + test data.
- **Test YAML fixtures** — initial test YAML had `status_mapping` at top level, real file has it under `workflow`. Always copy from real file, don't simplify.
- **Bridge server invocation** — design said `McpBridge("mcp-atlassian")` but real server needs `uvx mcp-atlassian --jira-url ... --jira-token ...`. OQ2 (env vars only) was incomplete — server needs CLI args too.

## Heutagogical Checkpoint

1. **Learned:** `@runtime_checkable` Protocol isinstance() matches both sync and async — use `inspect.iscoroutinefunction()` instead. Sooperset mcp-atlassian has different response format than raw Jira API.
2. **Process change:** Smoke test real service before writing mocks to validate response format.
3. **Framework improvement:** Test fixtures should mirror real config file structure exactly.
4. **Capability gained:** End-to-end MCP adapter construction (bridge → adapter → auto-wrap → entry point → real Jira).

## Architectural Insight: D6

The PM adapter justifies its existence for domain logic (releases, sprint planning,
bulk ops with error capture). For simple pass-through operations (search, get_issue),
a future `rai mcp call` CLI would be more valuable — agent-agnostic MCP access from
skills without programming. Captured as parking lot item and D6 in epic design.
