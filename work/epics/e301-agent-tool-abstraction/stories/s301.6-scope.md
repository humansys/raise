# S301.6: Skill Auto-Sync Hooks — Scope

## In Scope

- New work lifecycle events: `WorkStartEvent`, `WorkCloseEvent` (frozen dataclasses in events.py)
- `JiraSyncHook` implementing `LifecycleHook` protocol — listens to `work:start`, `work:close`
- Read `lifecycle_mapping` from `.raise/jira.yaml` to resolve event → transition ID
- Use existing `resolve_adapter()` → adapter.transition_issue() for Jira calls
- Entry point registration: `rai.hooks = { jira-sync = "rai_cli.hooks.builtin.jira_sync:JiraSyncHook" }`
- Graceful no-op when lifecycle_mapping absent or adapter unavailable
- Tests for hook + events

## Out of Scope

- Bidirectional sync (webhooks, polling) → Phase 2
- Custom field updates during transition (e.g., setting resolution)
- Comment auto-posting on transition
- Modifying existing skills to emit events (wiring is a follow-up or part of S301.7 dogfood)
- `rai mcp call` generic CLI

## Done When

- JiraSyncHook passes LifecycleHook Protocol isinstance check
- WorkStartEvent("story", "RAISE-301") dispatched through emitter triggers hook
- Hook reads lifecycle_mapping from jira.yaml and calls adapter
- Graceful no-op when unconfigured (no crash, debug log)
- Graceful error when adapter fails (HookResult error, no crash)
- Entry point registered in pyproject.toml
- All tests pass, types check, lint clean
