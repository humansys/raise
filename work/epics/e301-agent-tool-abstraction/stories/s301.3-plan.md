**Risk-first:** Core async infrastructure, SDK interaction, session management.

**What to build:**
- `McpBridgeError(Exception)` — single exception with contextual messages
- `McpToolResult(BaseModel)` — text, data, is_error, error_message
- `McpToolInfo(BaseModel)` — name, description
- `McpBridge` class:
  - `__init__(server_command, server_args, env)` — store params, no connection

## T2: McpJiraAdapter (M)

**Depends on:** T1 (McpBridge exists)

**Files:**
- CREATE `src/rai_cli/adapters/mcp_jira.py`
- CREATE `tests/adapters/test_mcp_jira.py`
- MODIFY `.raise/jira.yaml` — add `status_mapping` section (AR-6)

**What to build:**
- `McpJiraAdapter` class implementing `AsyncProjectManagementAdapter`:
  - `__init__()` — read `.raise/jira.yaml`, validate `status_mapping`, create `McpBridge("mcp-atlassian")`
  - `_resolve_transition_id(status) → str` — lookup in status_mapping, clear error for unknown status
  - `_load_jira_config() → dict` — read and validate jira.yaml
  - 11 async methods mapping to mcp-atlassian tools:
    - `create_issue` → `jira_create_issue`
    - `get_issue` → `jira_get_issue`
    - `update_issue` → `jira_update_issue`
    - `transition_issue` → `jira_transition_issue` (with status_mapping resolution)
    - `batch_transition` → sequential `transition_issue` in single async context
    - `link_to_parent` → `jira_update_issue` with `{"parent": key}` (AR-S3-2)
    - `link_issues` → `jira_create_issue_link`
    - `add_comment` → `jira_add_comment`
    - `get_comments` → `jira_get_issue` with `comment_limit` param (AR-S3-5)
    - `search` → `jira_search`
    - `health` → lightweight JQL probe via `jira_search`
  - Each method: transform protocol args → MCP tool args, call bridge, transform `McpToolResult` → protocol model

**jira.yaml addition:**
```yaml
# Below lifecycle_mapping:
status_mapping:
  backlog: 11
  selected: 21
  in-progress: 31
  done: 41
```

**TDD cycle:**
- RED: test `__init__` reads jira.yaml and validates status_mapping
- RED: test `__init__` raises ValueError when status_mapping missing
- RED: test `_resolve_transition_id("done")` returns "41"
- RED: test `_resolve_transition_id("invalid")` raises with available statuses
- RED: test `transition_issue` calls bridge with correct tool + args
- RED: test `create_issue` maps IssueSpec fields to jira_create_issue args
- RED: test `get_issue` parses McpToolResult.data into IssueDetail
- RED: test `search` passes JQL and limit to jira_search
- RED: test `get_comments` passes limit as comment_limit (AR-S3-5)
- RED: test `link_to_parent` uses jira_update_issue with parent field (AR-S3-2)
- RED: test `batch_transition` loops N calls sequentially
- RED: test `health` returns AdapterHealth from JQL probe
- GREEN: implement each
- REFACTOR: extract response parsing helpers

**Mock strategy:** Mock `McpBridge.call()` — return pre-built `McpToolResult` objects. Do NOT mock `ClientSession` here — that's T1's concern.

## T4: Integration smoke test (S)

**Depends on:** T3 (full wiring complete)

**What to validate (manual, against real Jira):**

1. Install deps: `pip install -e ".[mcp]"` + verify `mcp-atlassian` on PATH
2. Verify entry point loads: `python -c "from rai_cli.adapters.registry import get_pm_adapters; print(get_pm_adapters())"`
3. Verify resolve_adapter works: `python -c "from rai_cli.cli.commands._adapter_resolve import resolve_adapter; a = resolve_adapter(None); print(type(a))"`
4. Run `rai backlog search 'project = "RAISE" AND status = Done' --limit 3` — expect compact results
5. Run `rai backlog transition RAISE-XXX done` on a test issue — expect success
6. Run `rai backlog get RAISE-XXX` — expect issue detail
7. Verify all tests pass: `pytest tests/adapters/test_mcp_bridge.py tests/adapters/test_mcp_jira.py tests/cli/test_adapter_resolve.py -v`
8. Full quality gate: `ruff check . && pyright && pytest`

**Also verify (AR-S3-3):** `StdioServerParameters.env=None` inherits parent env. If not, fix in McpBridge and add test.

**AC coverage:** AC1-AC5 (end-to-end validation), SHOULD-1/2/3
