# S301.5: McpConfluenceAdapter — Design

> **Complexity:** Simple-Moderate (2 components, 1 integration, proven pattern)
> **Gates:** No risk, UX, or integration gates triggered.

## What & Why

**Problem:** `rai docs publish/get/search` (S301.4) has CLI surface but no concrete adapter. Entry point `rai.docs.targets` has zero registrations — commands fail with "No docs target installed."

**Value:** Completes M2 (Docs MVP). Enables publishing governance artifacts to Confluence and querying pages without raw MCP calls. Same token reduction pattern proven in M1 (~24x for backlog).

## Approach

Thin mapping layer: `McpConfluenceAdapter` implements `AsyncDocumentationTarget` (5 methods) by delegating to `mcp-atlassian` Confluence tools via the existing `McpBridge` (S301.3). Replicates the McpJiraAdapter pattern (compounding, PAT-E-442).

### Components

| Component | Change | Notes |
|-----------|--------|-------|
| `src/rai_cli/adapters/mcp_confluence.py` | **Create** | Adapter class, response parsers |
| `src/rai_cli/adapters/mcp_bridge.py` | **Modify** | Array parsing in `_parse_result()` |
| `pyproject.toml` | **Modify** | Entry point `rai.docs.targets` |
| `.raise/confluence.yaml` | **Create** | Project config (space_key) |

## Design Decisions

### D1: Bridge per adapter (independent lifecycle)

Each adapter owns its own `McpBridge` instance. Even though `mcp-atlassian` serves both Jira and Confluence, the bridge instances are separate — decoupled lifecycle, no cross-adapter dependency. If Confluence migrates to a different MCP server tomorrow, only this adapter changes.

### D2: MCP tool mapping

| Protocol method | MCP tool | Key params |
|----------------|----------|-----------|
| `can_publish` | *(no MCP call)* | Returns `True` always |
| `publish` (create) | `confluence_create_page` | space_key, title, content |
| `publish` (update) | `confluence_update_page` | page_id, title, content |
| `get_page` | `confluence_get_page` | page_id |
| `search` | `confluence_search` | query, limit |
| `health` | `confluence_search` | query="type=page", limit=1 |

### D3: Publish — create vs update via metadata tracking

Deterministic page tracking via `.raise/confluence-pages.yaml`:

```yaml
# .raise/confluence-pages.yaml (auto-managed by adapter)
roadmap: "3087892481"
adr: "3087925249"
```

- First publish: `confluence_create_page` → save page_id to yaml
- Subsequent: `confluence_update_page` with stored page_id
- **Auto-heal (PAT-E-523):** if update fails (page deleted in Confluence), remove entry from yaml and fall back to create

No search-by-title ambiguity. Reliable for a framework that says "reliable AI."

### D4: Config — one yaml per adapter

Config lives in `.raise/confluence.yaml` (not jira.yaml):

```yaml
# .raise/confluence.yaml
space_key: rAIse
```

- Credentials from env vars: `CONFLUENCE_URL`, `CONFLUENCE_USERNAME`, `CONFLUENCE_API_TOKEN`
- Space key from yaml, fallback to `CONFLUENCE_SPACE_KEY` env var
- Fail fast if neither exists

### D5: Response formats (probed against real Confluence)

Probed 2026-03-01 against `mcp-atlassian` via `uvx`. Single format per tool (no dual-format like Jira):

**`confluence_search`** — returns JSON **array** (not dict):
```json
[
  {"id": "9240577", "title": "Guardrails", "type": "page",
   "url": "https://humansys.atlassian.net/wiki/spaces/LA/pages/9240577",
   "space": {"key": "LA", "name": "Lean-Agile"},
   "content": {"value": "...", "format": "view"}}
]
```

**`confluence_get_page`** — dict wrapped under `metadata`:
```json
{"metadata": {"id": "9240577", "title": "...", "url": "...",
  "space": {"key": "LA", "name": "..."}, "version": 1,
  "content": {"value": "...", "format": "markdown"}}}
```

**`confluence_create_page`** / **`confluence_update_page`** — dict with `message` + `page`:
```json
{"message": "Page created successfully",
 "page": {"id": "3087892481", "title": "...", "url": "...",
  "space": {"key": "rAIse", "name": "rAIse"}, "version": 1,
  "content": {"value": "...", "format": "markdown"}}}
```

**Key finding:** `created`/`updated` timestamps come back empty — `PageSummary.updated` will be `""`.

### D6: Bridge array parsing

`McpBridge._parse_result()` today only parses JSON dicts. Confluence search returns arrays. Fix:

```python
parsed = json.loads(text)
if isinstance(parsed, dict):
    data = parsed
elif isinstance(parsed, list):
    data = {"items": parsed}
```

Wraps arrays in `{"items": [...]}`. Adapter reads `result.data["items"]`. Additive change — no impact on existing Jira adapter.

## Examples

### Publish (first time — create)

```python
adapter = McpConfluenceAdapter(project_root=Path("/home/emilio/Code/raise-commons"))

result = await adapter.publish(
    doc_type="roadmap",
    content="# Roadmap\n\n## Q1 2026\n- Agent Tool Abstraction",
    metadata={"title": "Roadmap"},
)
# PublishResult(success=True, url="https://humansys.atlassian.net/wiki/spaces/rAIse/pages/3087892481")
# Side effect: .raise/confluence-pages.yaml now contains roadmap: "3087892481"
```

### Publish (update — page exists)

```python
# .raise/confluence-pages.yaml has: roadmap: "3087892481"
result = await adapter.publish(
    doc_type="roadmap",
    content="# Roadmap v2\n\nUpdated.",
    metadata={"title": "Roadmap"},
)
# Calls confluence_update_page with page_id="3087892481"
# PublishResult(success=True, url="https://...")
```

### Publish (auto-heal — page was deleted)

```python
# .raise/confluence-pages.yaml has: roadmap: "3087892481" but page was deleted
result = await adapter.publish(
    doc_type="roadmap",
    content="# Roadmap\n\nRecreated.",
    metadata={"title": "Roadmap"},
)
# update_page fails → remove "roadmap" from yaml → create_page → save new page_id
# PublishResult(success=True, url="https://...new-id...")
```

### Get page

```python
page = await adapter.get_page("3087892481")
# PageContent(
#     id="3087892481", title="S301.5 Test Page",
#     content="# Test\n\nThis is a test page.",
#     url="https://humansys.atlassian.net/wiki/spaces/rAIse/pages/3087892481",
#     space_key="rAIse", version=2,

```

### Search

```python
results = await adapter.search("roadmap", limit=5)
# [
#     PageSummary(id="3087892481", title="Roadmap", url="https://...", space_key="rAIse", updated=""),
#     PageSummary(id="9240577", title="Guardrails", url="https://...", space_key="LA", updated=""),
# ]
```

### Health

```python
health = await adapter.health()
# AdapterHealth(name="confluence", healthy=True, message="OK", latency_ms=340)
```

### Can publish

```python
ok = await adapter.can_publish("roadmap", {})
# True
```

### Error — config missing

```python
adapter = McpConfluenceAdapter(project_root=Path("/tmp/no-config"))
# FileNotFoundError: "Confluence config not found: /tmp/no-config/.raise/confluence.yaml"
```

## Acceptance Criteria

### MUST

1. `McpConfluenceAdapter` implements all 5 `AsyncDocumentationTarget` methods + `aclose()`
2. Publish uses create/update via `.raise/confluence-pages.yaml` tracking, with auto-heal on deleted pages (PAT-E-523)
3. Response parsers handle probed mcp-atlassian format — `metadata.*` for get, `page.*` for create/update, array for search
4. `McpBridge._parse_result()` wraps JSON arrays in `{"items": [...]}` without breaking existing Jira flows
5. Entry point `rai.docs.targets = confluence` registered in pyproject.toml, discoverable via `resolve_docs_target()`

### SHOULD

1. Config from `.raise/confluence.yaml` with `CONFLUENCE_SPACE_KEY` env var fallback
2. Clear error messages for: config missing, credentials missing, connectivity failure

### MUST NOT

1. No new CLI commands — S301.4 already provides publish/get/search
2. No modifications to `jira.yaml` — Confluence has its own config file
3. No breaking changes to McpJiraAdapter or existing bridge consumers
