# Design: S325.4 — Backlog-Aware Skills

**Epic:** E325 Agent-Orchestrated Workflow
**Jira:** RAISE-333
**Size:** M
**Date:** 2026-03-01
**Complexity:** Moderate (3 components, 1 external integration, new hook type)

---

## What & Why

**Problem:** Lifecycle skills emit work signals (`rai signal emit-work`) but Jira stays out of sync. The developer manually runs `rai backlog transition` after each skill — process overhead the orchestrator (S325.6) can't automate.

**Value:** After this story, `rai signal emit-work story X --event start` automatically creates/transitions the Jira issue. The orchestrator chains skills and Jira reflects state without human intervention.

## Approach

Hook-based, not instruction-based. Mechanical, not behavioral.

```
rai signal emit-work story X --event start
    │
    ├─ (1) Writes telemetry JSONL (exists)
    │
    └─ (2) NEW: Fires WorkLifecycleEvent → hook system
              │
              └─ BacklogHook.handle()
                   ├─ Reads lifecycle_mapping from jira.yaml
                   ├─ Maps event → backlog action
                   └─ Calls adapter.transition_issue() or adapter.create_issue()
```

### Design Decision: Adapter Resolution (Opción A)

Resolve adapter inside `handle()` on every call. Simplest, aligns with ADR-039 ("hooks never crash"). If adapter unavailable, hook returns `HookResult(status="error")` and CLI continues. Testable via mocking `resolve_adapter`.

### Event-to-Backlog Mapping

Uses `lifecycle_mapping` from `.raise/jira.yaml`:

| emit-work event | Backlog action | lifecycle_mapping key |
|-----------------|---------------|----------------------|
| `story start` | `create` (if needed) + `transition in-progress` | `story_start: 31` |
| `story complete` | `transition done` | `story_close: 41` |
| `epic start` | `create` (if needed) + `transition in-progress` | `epic_start: 31` |
| `epic complete` | `transition done` | `epic_close: 41` |

Other events (`blocked`, `unblocked`, `abandoned`) → no backlog action (pass-through).

### Jira Key Resolution

The `work_id` in emit-work is a RaiSE ID (e.g., "S325.4"), not a Jira key (e.g., "RAISE-333"). BacklogHook needs to resolve this:

- `rai backlog search 'summary ~ "S325.4"' --format agent` → parses `KEY|STATUS|SUMMARY`
- If no result and event is `start` → `rai backlog create` with summary containing the work_id
- If no result and event is `complete` → warn and skip (can't transition what doesn't exist)

### Graceful Degradation

Per ADR-039, hook failure is logged and skipped:
- No jira.yaml → skip (no lifecycle_mapping)
- Adapter unavailable → `HookResult(status="error")`
- Search/create/transition fails → log warning, return error result
- CLI never crashes from hook failure

## Examples

```python
# WorkLifecycleEvent
@dataclass(frozen=True)
class WorkLifecycleEvent(HookEvent):
    event_name: Literal["work:lifecycle"] = field(default="work:lifecycle", init=False)
    work_type: str = ""      # "story" or "epic"
    work_id: str = ""        # "S325.4" or "E325"
    event: str = ""          # "start", "complete", "blocked", etc.
    phase: str = ""          # "design", "implement", etc.

# BacklogHook
class BacklogHook:
    events = ["work:lifecycle"]
    priority = 0

    def handle(self, event: HookEvent) -> HookResult:
        if not isinstance(event, WorkLifecycleEvent):
            return HookResult(status="ok")

        try:
            adapter = resolve_adapter()
        except Exception:
            return HookResult(status="error", message="adapter unavailable")

        mapping = load_lifecycle_mapping()  # from jira.yaml
        if mapping is None:
            return HookResult(status="error", message="no jira.yaml")

        action = derive_action(event, mapping)
        if action is None:
            return HookResult(status="ok")  # no backlog action for this event

        execute_action(adapter, action)
        return HookResult(status="ok")

# Bridge in signal.py emit_work()
# After telemetry write, add:
from rai_cli.hooks.emitter import create_emitter
emitter = create_emitter()
emitter.emit(WorkLifecycleEvent(
    work_type=work_type_lower,
    work_id=work_id,
    event=event_type,
    phase=phase,
))
```

## Acceptance Criteria

**MUST:**
1. `WorkLifecycleEvent` typed event in hook system
2. `emit-work` fires event to hook system after telemetry write
3. `BacklogHook` creates Jira issue on `start` if none exists
4. `BacklogHook` transitions Jira on `start` (in-progress) and `complete` (done)
5. Hook failure does not crash CLI (ADR-039 error isolation)
6. All gates pass (pytest, pyright, ruff)

**SHOULD:**
- Use `--format agent` (S325.3) for search output parsing

**MUST NOT:**
- Make backlog sync a gate (hook failure = warning, not error)
- Modify SKILL.md files (hooks handle it mechanically)
