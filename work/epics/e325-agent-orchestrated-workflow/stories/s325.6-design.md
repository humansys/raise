# S325.6 — Story Orchestrator: Design

**Date:** 2026-03-01
**Complexity:** Moderate-to-Complex (3 components, 3 integrations, custom logic)
**Type:** SKILL.md-only (no Python code changes)

---

## Problem

Developers must manually invoke 6 skills in sequence to run a full story cycle. Each invocation requires knowing the correct order, passing outputs as inputs, and remembering where they left off if interrupted. This process overhead contradicts E325's goal of agent-orchestrated workflow.

## Value

`/rai-story-run S325.6` → agent executes the full cycle, pausing only at natural gates. Process overhead drops from ~40% to ~10% of interaction time. Resumption is automatic — re-invoke after interruption and the orchestrator picks up where it left off.

---

## Approach

Create a single `/rai-story-run` meta-skill (SKILL.md) that:

1. **Detects phase** from git artifacts (declarative table, no state file — Q5 decision)
2. **Chains skills** in order: start → design → plan → implement → review → close
3. **Applies delegation** at 2 natural gates (post-design, post-implement)
4. **Stops on failure** — developer re-invokes to resume via phase detection

### Components

| Component | Action | Location |
|-----------|--------|----------|
| `/rai-story-run` SKILL.md | Create | `src/rai_cli/skills_base/rai-story-run/SKILL.md` |
| `.claude/skills` mirror | Create | `.claude/skills/rai-story-run/SKILL.md` |
| Skill validation | Verify | `rai skill validate` |

---

## Design Decisions

### D1: Delegation at natural gates only (not every step)

**Decision:** Check delegation level at 2 points: post-design and post-implement.

**Rationale:** Start/plan/review/close are mechanical — interrupting them adds friction without reducing risk. Design and implement are where human judgment adds value (architectural decisions, code quality).

**Behavior by level:**

| Level | Post-Design | Post-Implement |
|-------|-------------|----------------|
| REVIEW | Pause, show design summary, wait for approval | Pause, show implementation summary, wait for approval |
| NOTIFY | Show summary, continue unless user intervenes | Show summary, continue unless user intervenes |
| AUTO | Continue (gates still block on test failure) | Continue (gates still block on test failure) |

**Resolution:** Read `delegation` from `~/.rai/developer.yaml`, resolve via `resolve_delegation(profile, skill_name)` from S325.2. If no profile: default to REVIEW (safest).

### D2: Declarative phase detection table

**Decision:** Artifact presence → phase mapping as a table in the SKILL.md.

**Rationale:** Deterministic, unambiguous — agent reads table, checks file existence, resumes from latest phase. No narrative interpretation, no bash fragility.

**Detection table:**

| Check (in order) | Artifact Path | If exists → resume at |
|-------------------|--------------|----------------------|
| 5 | `stories/s{N}.{M}-retrospective.md` | close |
| 4 | `stories/s{N}.{M}-plan.md` | implement |
| 3 | `stories/s{N}.{M}-design.md` | plan |
| 2 | `stories/s{N}.{M}-story.md` | design |
| 1 | story branch exists | start (branch exists but no artifacts) |
| 0 | (nothing) | start (from scratch) |

**Important:** Check in reverse order (5→0) — take the most advanced phase.

### D3: Stop on failure, resume via phase detection

**Decision:** If any skill fails, stop immediately and report. Developer re-invokes `/rai-story-run` which resumes from the last completed phase.

**Rationale:** Zero cascading errors. Phase detection makes resumption free. No retry logic to maintain.

## Examples

### Example 1: Fresh story, Ha developer (delegation=NOTIFY)

```
User: /rai-story-run S999.1

Orchestrator:
  1. Phase detection → no artifacts → start from beginning
  2. Load delegation: Ha → NOTIFY
  3. Invoke /rai-story-start S999.1
     → story_md, scope_md created
  4. Invoke /rai-story-design S999.1
     → design_md created
  5. ── GATE: post-design ──
     "Design complete. Summary: [approach]. Continuing to plan..."
  6. Invoke /rai-story-plan S999.1
     → plan_md created
  7. Invoke /rai-story-implement S999.1
     → code committed, tests pass
  8. ── GATE: post-implement ──
     "Implementation complete. Summary: [tasks done, tests pass]. Continuing to review..."
  9. Invoke /rai-story-review S999.1
     → retrospective_md created
  10. Invoke /rai-story-close S999.1
      → merged, branch deleted
  Done.
```

### Example 2: Resuming mid-cycle

```
User: /rai-story-run S999.1
  (interrupted during implement — context expired)

User: /rai-story-run S999.1  (new session)

Orchestrator:
  1. Phase detection:
     - retrospective.md? No
     - plan.md? Yes → resume at implement
  2. "Resuming S999.1 from implement phase (plan.md found)"
  3. Invoke /rai-story-implement S999.1
     → continues from next uncompleted task in plan
  ... (continues normally)
```

### Example 3: Shu developer (delegation=REVIEW)

```
User: /rai-story-run S999.1

Orchestrator:
  ...
  5. ── GATE: post-design ──
     "Design complete. Summary: [approach]."
     "Review the design and approve to continue, or provide feedback."
     → WAITS for explicit approval
  ...
  8. ── GATE: post-implement ──
     "Implementation complete. Summary: [tasks, tests]."
     "Review and approve to continue."
     → WAITS for explicit approval
```

### Example 4: Failure mid-cycle

```
User: /rai-story-run S999.1

Orchestrator:
  ...
  7. Invoke /rai-story-implement S999.1
     → tests fail after T3
  8. STOP: "Implementation failed at task T3. Tests failing."
     "Fix the issue and re-invoke /rai-story-run S999.1 to resume."
  (No further skills invoked)
```

## Acceptance Criteria

### MUST

1. `/rai-story-run {story_id}` chains all 6 lifecycle skills in correct order
2. Phase detection from artifact presence resumes at correct phase (table-based, reverse order)
3. Delegation gates at post-design and post-implement respect DelegationLevel (REVIEW/NOTIFY/AUTO)
4. Failure stops the cycle immediately with clear report
5. SKILL.md validates with `rai skill validate` (≤150 body lines)

### SHOULD

1. Show brief progress indicator between skills ("Phase 3/6: plan...")
2. Summary at cycle end showing total phases completed and key artifacts

### MUST NOT

1. Create any state file — phase detection is git-derived only
2. Modify existing lifecycle skills — they remain independently invocable
3. Add Python code — this is SKILL.md-only
