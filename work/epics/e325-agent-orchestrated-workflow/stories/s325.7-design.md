# S325.7 — Epic Orchestrator: Design

**Date:** 2026-03-01
**Complexity:** Complex (3 components, 3 integrations, novel story iteration logic)
**Type:** SKILL.md-only (no Python code changes)

---

## Problem

Running an epic requires manually invoking 4 epic skills in sequence, then manually iterating through each story calling `/rai-story-run` for each. The developer must track which stories are done, which are next, and when to close the epic. This is the last manual orchestration layer.

## Value

`/rai-epic-run E325` → agent executes the full epic lifecycle including story iteration, pausing only at delegation gates. Developer focuses on decisions (story breakdown, sequencing) while the agent handles process. Completes E325's objective: "developer says 'implement story X' and the agent executes the full cycle."

---

## Approach

Create `/rai-epic-run` meta-skill following the same pattern as `/rai-story-run` (PAT-E-602, PAT-E-603):

1. **Detect phase** from git artifacts — broader artifact set than story level (D1)
2. **Chain epic skills** in order: start → design → plan → [story iterations] → close
3. **Iterate stories** by reading Progress Tracking table from scope.md (D2)
4. **Apply delegation** at 2 natural gates: post-design and post-plan (D3)
5. **Stop on failure** — resume via re-invocation and phase detection (D4)

| Component | Action | Location |
|-----------|--------|----------|

## Design Decisions

### D1: Epic phase detection table

Check artifacts in reverse order — take the most advanced phase:

| Check | Artifact | If exists → resume at |
|:-----:|----------|----------------------|
| 4 | All stories in Progress Tracking = "Done" | **close** |
| 3 | scope.md has "Implementation Plan" section | **stories** (iterate pending) |
| 2 | scope.md exists (no Implementation Plan section) | **plan** |
| 1 | epic branch exists, no scope.md | **design** |
| 0 | (nothing) | **start** |

**Key distinction (check 3 vs 4):** Parse the Progress Tracking table in scope.md. If any story has status != "Done" → resume at **stories**. If all stories are "Done" → resume at **close**.

### D2: Story iteration from scope.md

Read the Progress Tracking table in scope.md to determine story execution:

1. Parse table rows — each has Story ID, Size, Status columns
2. Filter stories where Status != "Done"
3. Execute in table order (plan already resolved dependencies in sequencing)
4. Each story invoked via `/rai-story-run {story_id}`
5. After each story completes, scope.md is already updated by `/rai-story-close`

No dependency graph parsing — the plan table order IS the resolved dependency order.

### D3: Delegation gates at post-design and post-plan

| Gate | Why |
|------|-----|
| Post-design | Story breakdown is the highest-value human decision in an epic |
| Post-plan | Sequencing and milestones deserve review before committing to execution |

No gate between stories — each `/rai-story-run` has its own internal gates (post-design, post-implement). Adding inter-story gates would be excessive.

### D4: Stop on failure, resume via phase detection

Same as S325.6: if any skill or story fails, stop immediately. Developer fixes and re-invokes `/rai-epic-run`. Phase detection picks up at the correct phase — story iteration resumes from the next pending story.

## Examples

### Example 1: Fresh epic, Ha developer (delegation=NOTIFY)

```
User: /rai-epic-run E999

Orchestrator:
  1. Phase detection → no artifacts → start from beginning
  2. Delegation: Ha → NOTIFY
  3. Invoke /rai-epic-start E999
     → epic branch, scope.md created
  4. Invoke /rai-epic-design E999
     → stories defined in scope.md
  5. ── GATE: post-design ──
     "Design complete. 5 stories defined: S999.1-S999.5. Continuing to plan..."
  6. Invoke /rai-epic-plan E999
     → implementation plan, milestones, sequencing added to scope.md
  7. ── GATE: post-plan ──
     "Plan complete. 3 milestones, 5 stories sequenced. Starting story iteration..."
  8. Stories phase:
     ── Story 1/5: S999.1 ──
     Invoke /rai-story-run S999.1 → (full story cycle)
     ── Story 2/5: S999.2 ──
     Invoke /rai-story-run S999.2 → (full story cycle)
     ... (continues for each pending story)
  9. All stories done → invoke /rai-epic-close E999
     → retrospective, merge to dev, branch cleanup
  Done.
```

### Example 2: Resuming mid-epic (3/5 stories done)

```
User: /rai-epic-run E999

Orchestrator:
  1. Phase detection:
     - All stories Done? No (S999.4, S999.5 pending)
     - Implementation Plan section? Yes → stories
  2. "Resuming E999 at stories phase (3/5 done, 2 remaining)"
  3. ── Story 4/5: S999.4 ──
     Invoke /rai-story-run S999.4
     ── Story 5/5: S999.5 ──
     Invoke /rai-story-run S999.5
  4. All stories done → invoke /rai-epic-close E999
  Done.
```

### Example 3: Shu developer (delegation=REVIEW) at design gate

```
User: /rai-epic-run E999

Orchestrator:
  ...
  5. ── GATE: post-design ──
     "Design complete. Stories defined:"
     "S999.1 (S): Foundation setup"
     "S999.2 (M): Core feature"
     "S999.3 (M): Integration"
     "Review the story breakdown and approve to continue, or provide feedback."
     → WAITS for explicit approval
```

### Example 4: Story fails mid-iteration

```
User: /rai-epic-run E999

Orchestrator:
  ...
  8. ── Story 3/5: S999.3 ──
     Invoke /rai-story-run S999.3
     → Tests fail during implement
  9. STOP: "Story S999.3 failed during implement. 2/5 stories complete."
     "Fix the issue and re-invoke /rai-epic-run E999 to resume."
```

## Acceptance Criteria

### MUST

1. `/rai-epic-run {epic_id}` chains all 4 epic skills plus story iteration
2. Phase detection resumes at correct epic phase (table-based, 5 states)
3. Story iteration reads Progress Tracking from scope.md, executes pending stories via `/rai-story-run`
4. Delegation gates at post-design and post-plan respect DelegationLevel
5. SKILL.md validates with `rai skill validate` (≤150 body lines)

### SHOULD

1. Show story progress indicator ("Story 3/5: S999.3")
2. Summary at epic end showing stories completed, velocity, patterns collected

### MUST NOT

1. Create any state file — phase detection is git-derived only
2. Modify existing epic or story skills
3. Parse dependency graphs — table order is sufficient
