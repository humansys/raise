# Design: S325.1 — Skill I/O Contract Definition

**Story:** S325.1
**Epic:** E325
**Complexity:** Simple-to-Moderate (schema design is the work, edits are mechanical)

---

## Problem

Lifecycle skills have no typed declaration of what data they need and produce. An orchestrator must hardcode or guess every data flow.

## Value

With I/O contracts in frontmatter, the orchestrator agent reads metadata to understand skill chaining. Data flows become explicit and inspectable.

## Approach

Add two new metadata fields to SKILL.md frontmatter: `raise.inputs` and `raise.outputs`. These are YAML lists of typed entries. The agent reads them; no code enforces them.

### Schema

```yaml
metadata:
  # ... existing fields ...
  raise.inputs: |
    - story_id: string, required, argument
    - epic_scope: file_path, optional, previous_skill
  raise.outputs: |
    - story_branch: string, next_skill
    - scope_commit: string, git
```

Each entry follows the pattern: `name: type, required|optional, source|destination`

**Types:** `string`, `file_path`, `boolean`, `list`
**Sources (inputs):** `argument` (human/orchestrator provides), `previous_skill` (output of prior skill), `cli` (from rai CLI), `config` (from manifest/developer.yaml), `git` (derived from git state)
**Destinations (outputs):** `next_skill` (consumed by next in chain), `file` (written to disk), `git` (branch/commit), `cli` (emitted via rai CLI)

### Why string-encoded YAML lists (not nested objects)

SKILL.md frontmatter metadata values are all strings (current convention: `raise.fase: "3"`, `raise.gate: "gate-plan"`). Nested YAML objects in frontmatter break the parser expectations. String-encoded lists maintain consistency and the validator doesn't need schema changes for nested structures.

### Components Affected

| Component | Change | Files |
|-----------|--------|-------|
| 12 lifecycle SKILL.md (builtin) | Modify — add raise.inputs/raise.outputs | `src/rai_cli/skills_base/rai-*/SKILL.md` |
| 12 lifecycle SKILL.md (deployment) | Sync — copy from builtin | `.claude/skills/rai-*/SKILL.md` |
| Validator | No change needed — unknown metadata keys are already ignored | `src/rai_cli/skills/validator.py` |

## Concrete I/O Contracts per Skill

### Epic Chain

**rai-epic-start:**
```yaml
  raise.inputs: |
    - epic_id: string, required, argument
    - epic_slug: string, required, argument
    - dev_branch: string, required, config
  raise.outputs: |
    - epic_branch: string, next_skill
    - brief: file_path, next_skill
    - scope: file_path, next_skill
```

**rai-epic-design:**
```yaml
  raise.inputs: |
    - brief: file_path, optional, previous_skill
    - scope: file_path, required, previous_skill
  raise.outputs: |
    - scope: file_path, next_skill
    - design: file_path, optional, next_skill
```

**rai-epic-plan:**
```yaml
  raise.inputs: |
    - scope: file_path, required, previous_skill
  raise.outputs: |
    - scope: file_path, next_skill
```

**rai-epic-close:**
```yaml
  raise.inputs: |
    - scope: file_path, required, previous_skill
    - all_retrospectives: boolean, required, git
    - dev_branch: string, required, config
  raise.outputs: |
    - retrospective: file_path, file
    - merge_commit: string, git
```

## Acceptance Criteria

### MUST
- All 12 lifecycle skills have `raise.inputs` and `raise.outputs` in frontmatter
- Schema is consistent (same field names, types, sources across all skills)
- `rai skill validate` passes on all modified skills
- Non-lifecycle skills validate without I/O fields (backward compat)
- Builtin and deployment copies identical

### SHOULD
- Contracts accurately reflect actual data flow (verified against skill body)

### MUST NOT
- Break existing metadata parsing
- Add code enforcement (metadata only)
- Modify non-lifecycle skills
