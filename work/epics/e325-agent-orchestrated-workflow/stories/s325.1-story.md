# Story: Skill I/O Contract Definition

## User Story
As an orchestrator skill,
I want typed inputs/outputs declared in each lifecycle skill's SKILL.md frontmatter,
so that I can chain skills with explicit data flow instead of implicit conversation state.

## Acceptance Criteria

### Scenario: Skill declares inputs and outputs
```gherkin
Given a lifecycle skill SKILL.md (e.g., rai-story-start)
When I read its frontmatter metadata
Then I find raise.inputs listing required/optional typed inputs with source
And I find raise.outputs listing typed outputs with destination
```

### Scenario: Validator accepts I/O contract metadata
```gherkin
Given a SKILL.md with raise.inputs and raise.outputs in metadata
When I run the skill validator
Then validation passes without errors related to I/O fields
```

### Scenario: Existing skills still validate
```gherkin
Given a SKILL.md without raise.inputs/raise.outputs (non-lifecycle skill)
When I run the skill validator
Then validation still passes (I/O fields are optional)
```

## Examples (Specification by Example)

| Skill | Sample Input | Sample Output |
|-------|-------------|---------------|
| rai-story-start | story_id (string, required, argument) | story_branch (string, next_skill) |
| rai-story-design | story_id (string, required, previous_skill) | design_artifact (file_path, next_skill) |
| rai-story-close | story_id (string, required, previous_skill) | merge_commit (string, state_file) |

## Notes
- I/O contracts are metadata documentation, NOT code-enforced (architecture review Q2)
- Keep contracts lean — the orchestrator knows the chain order, metadata clarifies data flow
- All 12 lifecycle skills get contracts, non-lifecycle skills are untouched
- Validator should accept but not require I/O fields (backward compatible)
- Sync builtin → deployment after all edits
