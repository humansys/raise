## Summary

Built a BacklogHook that listens to `work:lifecycle` events (fired by `rai signal emit-work`) and automatically syncs Jira — creating issues on `start` if none exists, transitioning to in-progress/done. Architecture: WorkLifecycleEvent dataclass → bridge in emit-work CLI → BacklogHook → ProjectManagementAdapter. Graceful degradation per ADR-039.

## What Went Well

- **User caught wrong approach** — initial scope was SKILL.md instruction edits. User remembered the hooks strategy, which is mechanical and testable vs behavioral and fragile.
- **Scope revised twice before implementation** — added `create` (from M2 criteria), then pivoted to hooks. Both corrections were cheap because they happened at scope/design level.
- **Hook system (E248) was ready** — LifecycleHook protocol, EventEmitter, HookRegistry, entry points all existed. Only needed to add one event type and one hook class.
- **TDD cycle was clean** — RED caught ImportError, GREEN was minimal, all 21 tests passed first try after implementation.

## What to Improve

- **Initial scope was wrong twice** — should have re-read M2 criteria and checked for existing hook strategy before writing scope. Would have saved 2 revision commits.
- **QR false positive** — flagged `AssertionError` as a typo when it's correct Python. Always verify findings against runtime before reporting.
- **`uv sync` vs `uv sync --extra dev`** — forgot dev extras after pyproject.toml change, caused pytest not found. Muscle memory gap.

## Heutagogical Checkpoint

1. **Learned:** Hook-based integration > instruction-based for deterministic side effects.
2. **Process:** Scope corrections are cheap before implementation. Revise aggressively.
3. **Framework:** QR is fallible — verify before fixing. False positives erode trust.
4. **Capability:** Builtin hook pattern: event → bridge → hook → entry point. Replicable.
