# Design: S325.5 — Discovery/Onboard Compaction

## Problem

The discovery/onboard pipeline requires ~11 manual HITL steps across 4 skills. Many are mechanical (run find, approve obvious components, choose to read docs). M2 criteria: ≤50% manual steps.

## Value

Onboarding a brownfield project drops from ~11 HITL decisions to ~4. Developer focuses on actual decisions (confirm gaps, edit wrong descriptions) instead of process mechanics.

## Approach

SKILL.md-only changes to 4 skills. Add I/O contracts (raise.inputs/raise.outputs) to discovery skills matching S325.1 format. Compact manual steps by automating mechanical decisions.

### Components Affected

| File | Change | What |
|------|--------|------|
| `src/rai_cli/skills_base/rai-discover-start/SKILL.md` | Modify | Replace find commands with `rai discover scan -o summary`, auto-populate context |
| `src/rai_cli/skills_base/rai-discover-scan/SKILL.md` | Modify | Add LLM batch synthesis instruction for medium-confidence, auto-accept high |
| `src/rai_cli/skills_base/rai-discover-validate/SKILL.md` | Modify | Remove "ask user" for high-confidence, streamline medium batch flow |
| `src/rai_cli/skills_base/rai-project-onboard/SKILL.md` | Modify | Auto-read found docs, pre-populate from discovery, ask only for gaps |
| `.claude/skills/` (4 files) | Modify | Mirror builtin changes |

### Step-by-Step Compaction

#### discover-start (3 manual → 0)

**Before:** 6 find commands per language + manual directory inspection + manual entry point identification.

**After:** Single `rai discover scan . -o summary` auto-detects languages, directories, entry points. Write context.yaml automatically.

**HITL removed:** Language detection, directory identification, entry point identification.
**HITL kept:** None — fully automated phase.

#### discover-scan (2 manual → 1)

**Before:** Human synthesizes purpose/category for every medium/low component.

**After:**
- High (≥70): Use auto_purpose directly (already in skill, just emphasize no human needed).
- Medium (40-69): Present by module batch with LLM-suggested descriptions. Human approves/edits batch (1 decision per module, not per component).
- Low (<40): Keep individual review (these genuinely need human judgment).

**HITL removed:** Per-component synthesis for medium confidence.
**HITL kept:** Module batch approve/edit (reduced from O(components) to O(modules)).

#### discover-validate (3 manual → 1)

**Before:** Ask to auto-validate high → batch review medium → individual review low.

**After:**
- High: Auto-validate without asking (silent, logged).
- Medium: Present module batches with descriptions from discover-scan. Approve/edit.
- Low: Keep scale gate + individual review.

**HITL removed:** "Approve all high?" question (always yes → just do it).
**HITL kept:** Medium batch review, low individual review (real decisions).

#### project-onboard (3 manual → 1)

**Before:** Ask whether to read found docs → present discovery + ask confirm → fill gaps.

**After:**
- Auto-read all found documentation (README, ARCHITECTURE, CONTRIBUTING, etc.) without asking.
- Pre-populate governance fields from discovery + docs.
- Present pre-populated summary. Ask only for unfilled gaps.

**HITL removed:** "Read these docs?" question (always yes), redundant confirm of discovery.
**HITL kept:** Fill actual gaps not found by discovery.

### I/O Contracts for Discovery Skills

Add `raise.inputs` and `raise.outputs` to frontmatter, matching S325.1 format:

```yaml
# discover-start
raise.inputs: |
  - project_root: path, required, argument
raise.outputs: |
  - context_yaml: file_path, next_skill (work/discovery/context.yaml)

# discover-scan
raise.inputs: |
  - context_yaml: file_path, required, from discover-start
raise.outputs: |
  - analysis_json: file_path, next_skill (work/discovery/analysis.json)
  - components_draft: file_path, next_skill (work/discovery/components-draft.yaml)

# discover-validate
raise.inputs: |
  - analysis_json: file_path, required, from discover-scan
  - components_draft: file_path, required, from discover-scan
raise.outputs: |
  - components_validated: file_path, next_skill (work/discovery/components-validated.json)

# project-onboard
raise.inputs: |
  - project_root: path, required, argument
raise.outputs: |
  - governance_docs: file_path[], next_skill (governance/*.md)
  - knowledge_graph: file_path, next_skill (.raise/rai/memory/index.json)
```

### Manual Step Count

| Skill | Before | After | Reduction |
|-------|:------:|:-----:|:---------:|
| discover-start | 3 | 0 | 100% |
| discover-scan | 2 | 1 | 50% |
| discover-validate | 3 | 1-2 | 50-67% |
| project-onboard | 3 | 1 | 67% |
| **Total** | **11** | **3-4** | **64-73%** |

Target was ≤50% → achieved.

## Acceptance Criteria

### MUST

1. discover-start fully automated (no find commands, no manual directory ID)
2. discover-validate auto-validates high-confidence silently (no "approve all?" question)
3. project-onboard auto-reads found docs (no "read these?" question)
4. I/O contracts in frontmatter for all 4 discovery skills
5. Total HITL steps ≤6 (≤50% of original 11)

### SHOULD

1. discover-scan presents medium-confidence by module batch with LLM-suggested descriptions
2. project-onboard pre-populates governance from discovery + docs before asking

### MUST NOT

1. Break manual skill invocation (backward compatible)
2. Change any Python code (SKILL.md only)
3. Remove scale gate for low-confidence validation
