# Design: S325.5 — Discovery/Onboard Compaction

## Problem

The discovery pipeline is split across 5 skills (discover-start → scan → validate → document → graph build) requiring ~15 manual HITL steps. The pipeline is always linear — no real use case for running step 3 without steps 1-2. The split is accidental complexity from incremental development.

Meanwhile, project-onboard duplicates parts of this pipeline and has its own ~3 manual steps.

M2 criteria: discovery completes with ≤50% of current manual steps.

## Value

Replace 5 discovery skills + onboard redundancy with a single `/rai-discover` that runs the full pipeline end-to-end. Developer invokes once, pauses only at real decision gates. Total HITL drops from ~15 to ~3-4.

## Approach

Create one new `/rai-discover` skill that consolidates the entire pipeline. Mark the 4 individual discover-* skills as deprecated (kept for backward compat, not promoted). Update project-onboard to delegate discovery to `/rai-discover`.

### Before vs After

**Before (5 skills, ~15 HITL steps):**
```
/rai-discover-start    → 3 HITL (find cmds, dir inspection, entry points)
/rai-discover-scan     → 3 HITL (run scan, synthesize medium, synthesize low)
/rai-discover-validate → 3 HITL (approve high, batch medium, review low)
/rai-discover-document → 3 HITL (verify modules, review prose, confirm docs)
rai graph build        → 0 HITL
/rai-project-onboard   → 3 HITL (read docs?, confirm discovery, fill gaps)
```

**After (1 skill, ~3-4 HITL steps):**
```
/rai-discover (unified pipeline):
  Phase 1: Detect    → 0 HITL (auto-detect via rai discover scan -o summary)
  Phase 2: Extract   → 0 HITL (rai discover scan → analysis.json)
  Phase 3: Describe  → 1-2 HITL (present medium/low by module batch, approve/edit)
  Phase 4: Document  → 1 HITL (review generated module docs + system docs)
  Phase 5: Build     → 0 HITL (rai graph build, verify)

/rai-project-onboard (updated):
  Step 2: delegates to /rai-discover (instead of inline scan)
  Step 3: 1 HITL (fill governance gaps not found by discovery)
```

### `/rai-discover` Skill Design

**Phases (consolidates 5 skills):**

| Phase | Source | HITL | What happens |
|-------|--------|:----:|------|
| 1. Detect | discover-start | 0 | `rai discover scan . -o summary` → auto-detect languages, dirs, entry points → write context.yaml |
| 2. Extract | discover-scan | 0 | `rai discover scan {dirs} -o json` → analysis.json with confidence scores |
| 3. Describe | scan + validate merged | 1-2 | High (≥70): auto-accept silently. Medium (40-69): present by module batch with LLM-suggested descriptions → approve/edit. Low (<40): scale gate → pattern/layer/key modes |
| 4. Document | discover-document | 1 | Generate module docs (frontmatter + prose), system docs (context, design, domain model, index). Present for review. |
| 5. Build | graph build | 0 | `rai graph build` → verify modules in graph, no stale refs |

**Key merges:**
- scan step 3 (synthesize) + validate steps 2-4 → single "Describe" phase. No reason to synthesize then separately validate — do both in one pass.
- discover-start steps 1-2 (find commands) → replaced by `rai discover scan -o summary`.

**ShuHaRi adaptation:**
- **Shu**: All phases shown, explain each, pause after describe + document
- **Ha**: Auto-run phases 1-2, pause at describe, auto-run document if <10 modules
- **Ri**: Full pipeline with minimal pauses, approve module batches inline

**I/O Contract:**
```yaml
raise.inputs: |
  - project_root: path, required, argument
  - language: string, optional, argument (auto-detected if omitted)
raise.outputs: |
  - context_yaml: file_path (work/discovery/context.yaml)
  - components_validated: file_path (work/discovery/components-validated.json)
  - module_docs: file_path[] (governance/architecture/modules/*.md)
  - system_docs: file_path[] (governance/architecture/*.md)
  - graph: side_effect (rai graph build)
```

### project-onboard Update

Replace Step 2 (inline `rai discover scan`) with: "Run `/rai-discover` if not already done."

This avoids duplicating the pipeline. If developer already ran `/rai-discover`, onboard skips directly to governance gap-filling (Step 3).

### Manual Step Count

| Phase | Before (across 5 skills) | After (unified) | Reduction |
|-------|:------------------------:|:----------------:|:---------:|
| Detect | 3 | 0 | 100% |
| Extract | 1 | 0 | 100% |
| Describe | 5 (synthesize + validate) | 1-2 | 60-80% |

| Onboard | 3 | 1 | 67% |

Target was ≤50% → far exceeded.

## Acceptance Criteria

### MUST

1. Single `/rai-discover` skill runs full pipeline (detect → extract → describe → document → build)
2. High-confidence components auto-accepted silently (no HITL)
3. Medium-confidence presented by module batch (not per-component)
4. Module + system docs generated in same flow
5. `rai graph build` runs as final step
6. I/O contract in frontmatter
7. Total HITL ≤4 steps for the full pipeline

### SHOULD

1. project-onboard delegates to `/rai-discover` instead of inline scan
2. Individual discover-* skills marked deprecated (not removed, backward compat)

### MUST NOT

1. Change any Python code (SKILL.md only)
2. Remove scale gate for low-confidence validation
3. Break `/rai-project-onboard` flow
