# S474.5 Design: Document CI=Skills Guardrail

## Complexity Assessment

**Simple** -- docs-only story, no code changes, single deliverable document plus cross-references.

## Approach

Create a governance guardrail document that codifies the CI=Skills parity principle:
**CI and local gates MUST check the same things, driven by a single source of truth.**

### Deliverable: `.raise/governance/guardrails/ci-skills-parity.md`

The document will cover:

1. **Principle statement** -- CI pipeline and local gates (`rai gate check`, pre-commit hook) execute the same commands because they all read from `.raise/manifest.yaml`.

2. **Architecture flow** -- How the pieces connect:
   ```
   .raise/manifest.yaml (single source of truth)
       |
       +---> rai gate check --all  (local, on-demand)
       +---> pre-commit hook       (local, on-commit)
       +---> .gitlab-ci.yml        (remote, on-MR/push)
   ```

3. **Current commands** -- Table mapping manifest keys to actual commands and where they run (pre-commit, CI, `rai gate check`).

4. **How to add/modify a gate command** -- Developer checklist:
   - Update `manifest.yaml`
   - Verify `rai gate check --all` passes
   - Verify CI pipeline reflects the same command
   - If pre-commit relevant, verify hook picks it up

5. **Known asymmetries** (documented, not bugs):
   - Pre-commit hook skips `test_command` (too slow for commit-time)
   - CI runs security scanning (Snyk, SAST, SonarQube) beyond quality gates
   - Pre-commit includes housekeeping hooks (trailing whitespace, YAML check) not in manifest

6. **Cross-references** to `gate-code.md`, pre-commit config, CI pipeline, manifest.

### Cross-reference updates

- `gate-code.md`: Add a "See also" link to the new guardrail document.
- No CLAUDE.md changes needed -- the guardrail is discoverable via governance directory.

## Decisions

| ID | Choice | Rationale |
|----|--------|-----------|
| D1 | Place in `.raise/governance/guardrails/` (new dir) | Governance guardrails are distinct from gate definitions; creates a clear home for future guardrails |
| D2 | Document known asymmetries explicitly | Prevents confusion -- readers see the principle AND the intentional exceptions |
| D3 | No CLAUDE.md update | The guardrail is a governance artifact, not a process rule; discoverable via directory structure |
