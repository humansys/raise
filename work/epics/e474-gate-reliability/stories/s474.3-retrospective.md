# Retrospective: S474.3 -- Code Audit Against S370.1 Standard

**Story:** S474.3 | **Epic:** E474 Gate Reliability
**Size:** M (estimated 180 min) | **Actual:** ~120 min
**Type:** analysis

## Heutagogical Checkpoint

### 1. What did you learn?

- **Tiered fix strategy is essential for large violation counts.** 247 violations sounds overwhelming, but triaging into auto-fix (63), global ignore (66), per-file ignore (77), manual (9), and C901 noqa (32) made it manageable. The key insight: most "violations" from new rules are either already covered by another tool (ANN401 vs pyright strict) or are noise in specific contexts (test docstrings).

- **Per-file ignores are the right granularity.** Global ignores suppress rules everywhere; inline noqa is tedious per-file. Per-file ignores in pyproject.toml target specific directories (tests/, protocol stubs) where rules genuinely don't apply. This is a Ruff best practice that earned its place.

- **McCabe complexity 11-15 is a gray zone.** 32 functions fell in this range. Reviewing each one confirmed they are primarily CLI command handlers and parsers where splitting would reduce readability. The `# noqa: C901 -- complexity N, refactor deferred` convention documents the debt without hiding it.

### 2. What would you change about the process?

- **Run `--statistics` before `--fix`.** I should have captured violation counts per rule code before any fixes, to have a cleaner baseline. The gap analysis (T1) covered this conceptually but the numbers came from running fix iteratively.

- **Batch commits by rule family.** The plan suggested this as a risk mitigation ("if violations exceed ~50, split by rule family"). In practice, ANN and D fixes were interleaved. A cleaner approach would be: commit after ANN clean, then commit after D clean. This story had 3 implementation commits which is acceptable but could have been more granular.

### 3. Are there improvements for the framework?

- **Code audit skill could automate triage.** A `rai code-audit` skill that runs expanded rules, categorizes violations (auto-fixable, per-file ignore candidates, manual), and generates a triage report would save significant time. The S370.3 skill already scaffolded this; this story validated the workflow it should automate.

- **Pattern for pyproject.toml per-file-ignores format.** The Ruff per-file-ignores syntax (`"tests/**" = ["ANN001", ...]`) is non-obvious. A template or convention in the code standard would help.

### 4. What are you more capable of now?

- **Ruff rule triage at scale.** Confident in evaluating which rules to enable globally, which to per-file ignore, and which to globally ignore with documented rationale. The ANN401 decision (pyright handles real type safety, ANN401 is redundant) is a reusable judgment.

- **Audit documentation.** The 5-dimension audit results format (criterion/enforcement/status/evidence) proved clear and actionable. Can apply this pattern to future standards audits.
