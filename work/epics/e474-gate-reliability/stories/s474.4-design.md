# Design: S474.4 -- Pre-commit Hook for Lint+Format+Types

**Story:** S474.4 | **Epic:** E474 Gate Reliability
**Complexity:** Small (single CLI command + Python hook script)

## Problem

Developers can commit code that fails lint, format, or type-check gates. These failures are only caught later by `rai gate check` or CI. A pre-commit hook would shift detection left to commit time, preventing bad code from entering the repository.

## Value

Blocks commits that would fail CI, reducing feedback loops. Uses the same manifest-driven commands as gates, ensuring consistency between local and CI checks.

## Approach

### Option A: Shell script that parses YAML (rejected)

Parsing YAML in bash is fragile. Requires `yq` or `grep`/`sed` hacks. Not portable.

### Option B: `pre-commit` framework (rejected)

Scope explicitly excludes external frameworks. Would also not read from manifest natively.

### Option C: Python hook script installed by `rai gate install-hook` (chosen)

A thin Python script that reuses `raise_cli.onboarding.manifest.load_manifest()` and `raise_cli.gates.builtin._runner.run_manifest_command()` infrastructure. Installed via a new CLI subcommand on `gate_app`.

## Design

### Component 1: Hook Script (`src/raise_cli/gates/hook.py`)

A standalone module with a `main()` entry point that:

1. Loads manifest via `load_manifest(Path.cwd())`
2. Reads `lint_command`, `format_command`, `type_check_command` from `manifest.project`
3. Runs each configured command via `subprocess.run()` (reusing the same split+cwd pattern from `_runner.py`)
4. Prints clear pass/fail per check with the check name
5. Exits 0 if all pass, 1 if any fail
6. Skips unconfigured commands (no manifest key = skip)

The hook does NOT run `test_command` (too slow for commit-time, per scope).

Key design choice: This is a Python script, not a shell script. The git hook at `.git/hooks/pre-commit` will be a thin shim that calls `uv run python -m raise_cli.gates.hook`. This means:
- Full access to manifest loader (no YAML parsing in bash)
- Consistent command resolution with gates
- Testable with standard pytest

### Component 2: CLI Commands (`rai gate install-hook` / `rai gate uninstall-hook`)

Two new subcommands on the existing `gate_app` Typer:

**`rai gate install-hook`:**
1. Resolves `.git/hooks/pre-commit` path from project root
2. Checks if hook already exists; if so, warns and exits (no silent overwrite)
3. Writes a shim script:
   ```bash
   #!/usr/bin/env bash
   # Installed by: rai gate install-hook
   # Remove with:  rai gate uninstall-hook
   uv run python -m raise_cli.gates.hook
   ```
4. Makes it executable (`chmod +x`)
5. Prints confirmation message
6. Option: `--force` flag to overwrite existing hook

**`rai gate uninstall-hook`:**
1. Checks if `.git/hooks/pre-commit` exists and was installed by rai (checks for marker comment)
2. Removes the file
3. Prints confirmation
4. If hook exists but was not installed by rai, warns and exits without removing

### Component 3: Tests

- `tests/gates/test_hook.py` -- Unit tests for the hook module:
  - Test: all three checks pass -> exit 0
  - Test: lint fails -> exit 1, clear message
  - Test: missing manifest -> exit 1
  - Test: unconfigured command -> skip gracefully
  - Test: test_command is never executed

- `tests/cli/test_gate_hook_commands.py` -- CLI integration tests:
  - Test: install-hook creates file with correct content and permissions
  - Test: install-hook refuses to overwrite existing hook (no --force)
  - Test: install-hook --force overwrites
  - Test: uninstall-hook removes rai-installed hook
  - Test: uninstall-hook refuses to remove non-rai hook

## Decisions

| ID | Choice | Rationale | Alternatives |
|----|--------|-----------|-------------|
| D1 | Python hook module, not shell script | Reuses manifest loader, testable, no YAML parsing in bash | Shell script with yq (fragile), shell with grep (brittle) |
| D2 | Thin bash shim in .git/hooks | Git requires the hook file there; shim delegates to Python | Symlink to script (breaks if repo moves), direct Python shebang (needs correct venv) |
| D3 | `uv run python -m raise_cli.gates.hook` as invocation | Works with project's uv environment, no PATH assumptions | `python -m` (might pick wrong Python), direct import (needs installed package) |
| D4 | Marker comment for uninstall safety | Prevents accidentally deleting user's custom hook | Always delete (dangerous), backup before delete (complexity) |
| D5 | No auto-fix, check-only | Scope requirement. Developer fixes consciously | Auto-fix with ruff format (scope creep, surprising behavior) |
| D6 | Skip test_command deliberately | Tests are too slow for commit-time. Scope excludes them | Run tests too (violates scope, bad DX) |

## Integration Points

| Module | Change Type | Files |
|--------|------------|-------|
| gates/hook | new | `src/raise_cli/gates/hook.py` |
| cli/gate | modification | `src/raise_cli/cli/commands/gate.py` |
| tests | new | `tests/gates/test_hook.py`, `tests/cli/test_gate_hook_commands.py` |

## Acceptance Criteria

- **AC1:** `rai gate install-hook` creates `.git/hooks/pre-commit` with correct shim
- **AC2:** The hook runs lint, format, and type-check commands from manifest
- **AC3:** The hook does NOT run test_command
- **AC4:** Commit is blocked (exit 1) when any check fails
- **AC5:** Clear output shows which check passed/failed
- **AC6:** `rai gate uninstall-hook` removes only rai-installed hooks
- **AC7:** `--no-verify` bypass works (native git behavior, no code needed)
- **AC8:** Hook is opt-in: never auto-installed
