# Retrospective: S474.2 — FormatGate + Configurable Gates

**Story ID:** S474.2
**Epic:** E474 Gate Reliability
**Size:** S (estimated 120 min, actual ~76 min)
**Date:** 2026-03-06
**Status:** Complete

## Summary

Created a shared `_runner.py` helper that reads gate commands from `.raise/manifest.yaml`, refactored all 4 existing gates (Test, Lint, Type, Coverage) to use it, created a new FormatGate, registered it as an entry point, updated `gate-code.md` governance doc, and fixed a QR finding about missing stderr in failure details.

## What Went Well

- **Design-first paid off.** The design document clearly specified the shared helper pattern, which made implementation mechanical. All 5 gates ended up with ~5-line `evaluate()` methods.
- **TDD caught the stderr gap.** The QR review identified that failure details only included stdout. The fix was straightforward because the runner was already centralized — one change fixed all 5 gates.
- **Scope expansion was controlled.** The scope doc said "out of scope: other gates" but the design correctly expanded to refactor all 4 existing gates. The plan noted this deviation explicitly, maintaining traceability.
- **Faster than estimated.** 76 min vs 120 min estimate. The shared helper pattern reduced per-gate implementation to near-trivial.

## What to Improve

- **Scope vs design alignment.** The scope was written before design explored the full implications. In future, either update scope after design or accept that design supersedes scope. The plan's note about the deviation was the right call.
- **CoverageGate DRY concern.** The AR noted CoverageGate has special logic (appending `--cov` flags to `test_command`). This is acceptable complexity but should be watched — if more gates need command composition, extract a pattern.

## Heutagogical Checkpoint

### 1. What did you learn?

The shared helper pattern for gates is a clean application of DRY that transforms each gate from a 30-line class with subprocess management into a 5-line delegate. The key insight is the "skip when not configured" behavior — it makes gates composable across different tech stacks without requiring all tools.

### 2. What would you change about the process?

The scope document should either be updated after design (when the full implications are understood) or explicitly marked as "pre-design" so readers know the design is authoritative. In this story, the plan caught the divergence, but it could confuse future readers.

### 3. Are there improvements for the framework?

The gate system is now manifest-driven, which is a framework improvement itself. One potential next step: the `_runner.py` helper could support command templates (e.g., `{src_dir}` expansion) for gates that need path-aware commands. Not needed now, but worth noting for when non-Python projects adopt RaiSE.

### 4. What are you more capable of now?

More capable of recognizing when a "new feature" story (FormatGate) actually implies a refactoring opportunity (all existing gates). The design phase correctly identified this — the pattern of "add one, refactor all" is worth applying when adding to a set of similar components.
