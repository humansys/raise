# S353.4 Design: Quality Validation

## Problem

The Checkpoint & Fork pattern was implemented (S353.2) and extended (S353.3), but there is no objective measurement proving that forked orchestration produces quality comparable to standalone execution. Without this validation, the epic's core claim -- eliminating the 4.6x quality gap -- is unsubstantiated.

## Value

A documented validation with methodology, measurements, and pass/fail declaration provides the evidence needed to trust orchestrated execution and close the epic. Reproducible methodology lets future changes be re-validated.

## Approach

This is a measurement/analysis story, not a code story. The deliverable is a **validation report** that:

1. Defines what metrics to measure and why
2. Gathers measurements from S353.3's forked execution (the first story to use all 8 phases forked)
3. Compares against the baseline (standalone vs old orchestrator)
4. Declares pass/fail against >80% parity target

### Evidence Availability

Key constraint: AR and QR are **inline-only phases** (per the artifact I/O contract in epic design). They return verdicts via Agent tool return value but do NOT write files to disk. This means:

- AR/QR raw output (char counts, full text) is **not available** for post-hoc measurement -- it lived in the forked subagent context, which was discarded after execution
- What IS available: the S353.3 retrospective, which documents AR/QR outcomes (verdicts, findings, applied fixes)
- What IS available: artifact-producing phases wrote files to disk (design.md, plan.md, retrospective.md)

This constraint shapes the methodology: we measure **observable outcomes** (artifact quality, verdict depth, applied findings) rather than raw output volume.

## Validation Methodology

### Metrics Framework

| Metric | What It Measures | How Measured | Why It Matters |
|--------|-----------------|--------------|----------------|
| **Artifact completeness** | Did all 8 phases produce their expected outputs? | Check file existence + content sections | Basic proof that forking doesn't break output |
| **AR verdict depth** | Did AR produce substantive architectural findings? | Count findings, heuristics evaluated, questions raised (from retrospective evidence) | AR in old orchestrator was reduced to near-zero; this is the quality gap |
| **QR verdict depth** | Did QR produce actionable recommendations? | Count recommendations, observations, cross-references (from retrospective evidence) | QR in old orchestrator: 2,196 chars / 1 tool call vs standalone 10,064 / 28 |
| **Applied findings** | Were AR/QR findings actionable enough to drive changes? | Count commits that reference AR/QR findings | Strongest quality signal: findings that actually changed the code |
| **Review depth** | Did story-review produce patterns and calibration? | Count patterns added/reinforced, calibration data present | Review was the most context-sensitive phase |
| **End-to-end success** | Did the orchestrated story complete successfully? | Story status = Done, all done criteria met | Integration test of the full pipeline |

### Threshold Rationale

The >80% target comes from the epic scope. This is appropriate because:
- 100% parity is unrealistic: standalone execution has full conversation context, forked does not
- 80% means the critical quality signals (actionable findings, pattern extraction, artifact depth) must be present
- Below 80% would indicate the fork pattern loses essential quality that makes AR/QR valuable

### Caveats and Limitations

1. **Single-story sample:** S353.3 is a documentation-only story (1 file modified, no code). AR/QR naturally produce fewer findings for simpler stories. A code-heavy story would be a stronger test.
2. **No raw output comparison:** AR/QR output volume (char count, tool calls) cannot be compared because inline-only phases don't persist their output. Future improvement: log AR/QR output to a file for post-hoc analysis.
3. **Retrospective-based measurement:** We measure from the retrospective's description of AR/QR outcomes, not from raw output. This is indirect but reliable -- the retrospective was written by the review subagent, which is itself forked and thus has its own fresh context.
