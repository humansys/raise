# S353.3 Design: epic-run Checkpoint

## Problem

epic-run Step 3 (story iteration) invokes story-run inline but has no inter-story checkpoint protocol. There is no summary read between stories, no progress tracking update within the loop, and no documentation of the F5 depth constraint that mandates inline invocation. Without explicit checkpoint logic, the main thread accumulates unbounded context across stories.

## Value

Thin checkpoints between stories keep the main thread context lightweight across multi-story epics. Explicit F5 documentation prevents future regressions where someone might attempt to fork story-run.

## Approach

Modify epic-run SKILL.md Step 3 (Iterate Stories) to add three things:

1. **F5 depth constraint documentation** -- a callout explaining why story-run MUST stay inline
2. **Thin checkpoint protocol** -- after each story completes, read summary from story artifacts, update progress tracking in scope.md, apply inter-story gate
3. **Context management guidance** -- explicit note that the main thread sees only summaries between stories

Epic-level phases (Step 2 phases 1-4 and phase 6) are unchanged. The invocation mechanism (Skill tool for story-run) is already correct -- this story documents and formalizes the checkpoint behavior that should happen between story boundaries.

## Concrete Changes to SKILL.md

### 1. Replace Step 3 body (Iterate Stories)

Current Step 3 is 8 lines. Replace with an expanded version that includes:

**F5 depth constraint callout:**

```markdown
> **F5 Constraint — story-run MUST execute inline (main thread).**
> story-run forks its heavy phases via Agent tool (depth 1). If epic-run
> forked story-run as a subagent, story-run would be at depth 1 and its
> own forks would need depth 2 — blocked by the F5 limit (subagents
> cannot spawn subagents). Always invoke story-run via Skill tool, never
> via Agent tool.
```

**Expanded story iteration loop with checkpoint:**

```markdown
For each pending story (Status != "Done"), in table order:

1. Show `── Story {N}/{total}: {story_id} ──`
2. Invoke `/rai-story-run {story_id}` via **Skill tool** (inline, main thread)
3. story-run executes — it forks its own heavy phases via Agent tool
4. **Checkpoint** (after story-run returns):
   a. Read the story's retrospective (`s{N}.{M}-retrospective.md`) or scope for summary
   b. Update the `### Progress Tracking` table in `scope.md`: set Status → "Done", fill Actual/Velocity/Notes
   c. Present one-line summary: "Story {story_id}: Done — {key outcome}"
5. Proceed to next story
```

**Context management note:**

```markdown
**Context management:** The main thread accumulates only checkpoint
summaries between stories — not the full tool-call history of each
phase (those live in forked subagent contexts that are discarded after
each phase). This keeps the orchestrator responsive through multi-story
epics.
```

### 2. Add inter-story gate (optional, within checkpoint)

No new gate type needed. The existing delegation gates within story-run handle per-story pauses. The epic-run checkpoint is purely informational -- it updates progress and presents a one-liner. If a story fails, story-run already stops immediately, which surfaces to epic-run.

### 3. Update Quality Checklist

Add two items:

- `[ ] Story-run invoked via Skill tool (inline), NEVER via Agent tool (F5 constraint)`
- `[ ] Checkpoint between stories: summary read + progress update + one-line report`

### 4. Update References

Add reference to E353 design and S353.2.
