# Story: epic-run Checkpoint

## User Story
As a developer running /rai-epic-run,
I want story-run to execute inline (main thread) with thin checkpoints between stories,
so that story-run can fork its heavy phases to subagents (depth 1) without hitting the F5 depth limit.

## Acceptance Criteria

### Scenario: story-run executes inline in main thread
```gherkin
Given I invoke /rai-epic-run for an epic with multiple stories
When the orchestrator reaches the story iteration phase
Then it invokes /rai-story-run inline (main thread, not forked)
And story-run can fork its heavy phases via Agent tool (depth 1)
```

### Scenario: Thin checkpoint between stories
```gherkin
Given a story completes via /rai-story-run
When the orchestrator transitions to the next story
Then it reads the completed story's summary (retrospective or scope artifacts)
And it updates the Progress Tracking table in scope.md
And it applies any inter-story gate
```

### Scenario: Context stays lightweight across stories
```gherkin
Given the epic has 4+ stories
When the orchestrator iterates through stories
Then the main thread accumulates only summaries between stories
And heavy phase output lives in discarded subagent contexts
And the orchestrator remains responsive through the last story
```

### Scenario: Epic-level phases stay inline
```gherkin
Given I invoke /rai-epic-run for an epic
When the orchestrator reaches epic-level phases (start, design, AR, plan, close)
Then those phases execute inline (same as current behavior)
And no forking occurs for epic-level phases
```

## Notes
- Critical constraint: epic-run MUST NOT fork story-run (would make story-run depth 1, blocking its own forks at depth 2 -- F5 limit)
- S353.2 already modified story-run to fork heavy phases; this story ensures epic-run stays compatible
- Checkpoint is "thin": read summary, update progress, apply gate -- not a full state serialization
- Epic-level phases (start, design, AR, plan, close) are unchanged in behavior
