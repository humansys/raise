# S3.1 Design: Schema Validation

## Complexity: Simple

3 concerns (competency questions, corpus coverage, schema consistency), no external integrations, no novel algorithms. Straightforward validation logic with Pydantic models for reporting.

## Problem & Value

Eduardo designed a 6-type ontology schema with ~10 example nodes, but we don't know if it can represent the full Scaling Up methodology. If the schema has gaps, everything downstream (extraction, curation, retrieval) fails silently — producing a graph that *looks* complete but can't answer the questions an empresario would ask.

**Value:** Confidence to build on the schema, or a concrete list of what to fix before investing in pipeline.

## Approach

Build a validation module that runs 3 checks against Eduardo's schema and corpus, producing a structured `ValidationReport`. This is a read-only analysis — it does NOT modify the schema, only reports.

### Components

| Component | Change | Description |
|-----------|--------|-------------|
| `src/rai_agent/scaleup/validation/models.py` | **create** | Pydantic models: `CompetencyQuestion`, `CoverageGap`, `ConsistencyError`, `ValidationReport` |
| `src/rai_agent/scaleup/validation/schema_validator.py` | **create** | Core validation: load YAML nodes, run 3 checks, produce report |
| `src/rai_agent/scaleup/validation/competency_questions.yaml` | **create** | Catalog of ≥20 CQs derived from Scaling Up methodology |
| `tests/scaleup/validation/` | **create** | Test suite for validator |

### Decisions

**D1: Where does this code live?**
- Choice: `src/rai_agent/scaleup/validation/` in the rai-agent repo
- Rationale: ScaleUp pipeline is a component of rai-agent. The scaleupagent repo is data-only (corpus + schema YAML). All code lives in rai.

**D2: How to define competency questions?**
- Choice: YAML catalog with structured fields (question, expected traversal path, decision area)
- Rationale: Inspectable, editable by Eduardo, versionable. Not hardcoded in tests.
- Alternative: Hardcoded in Python → not editable by domain expert

**D3: How to check corpus coverage?**
- Choice: Parse chapter headings from corpus markdown, map each to node types using keyword matching
- Rationale: Simple heuristic sufficient — we're checking coverage, not extracting nodes (that's S3.2). LLM-based mapping would be overkill for a validation gate.
- Alternative: LLM-based semantic mapping → overkill, saves that for S3.2

## Examples

### Competency Question format (YAML)
```yaml
- id: CQ-001
  question: "What tools support the People decision?"
  decision: people
  expected_path: "decision(people) ← belongs-to ← tool[*]"
  expected_min_results: 2

- id: CQ-002
  question: "What metrics measure execution effectiveness?"
  decision: execution
  expected_path: "concept/tool[execution] → measured-by → metric[*]"
  expected_min_results: 1

- id: CQ-003
  question: "What worksheets implement the OPSP?"
  decision: strategy
  expected_path: "tool(opsp) ← implements ← worksheet[*]"
  expected_min_results: 1
```

### Validation invocation
```python
from scaleupagent.validation.schema_validator import validate_schema

report = validate_schema(
    knowledge_dir=Path(".scaleup/knowledge"),
    corpus_paths=[
        Path("scaling_up_llamaparse.md"),
        Path("metodologias_parsed/people_workbook.md"),
        Path("metodologias_parsed/strategy_workbook.md"),
        Path("metodologias_parsed/execution_workbook.md"),
        Path("metodologias_parsed/cash_workbook.md"),
    ],
    questions_path=Path("validation/competency_questions.yaml"),
)
```

### ValidationReport output (success)
```python
ValidationReport(
    competency=CompetencyResult(
        total=20,
        covered=18,
        uncovered=[
            UncoveredQuestion(id="CQ-015", question="...", reason="No metric nodes for cash decision"),
            UncoveredQuestion(id="CQ-019", question="...", reason="No stage→tool relationships"),
        ],
        coverage_pct=90.0,
    ),
    corpus_coverage=CoverageResult(
        total_sections=25,
        covered=22,
        gaps=[
            CoverageGap(section="THE BARRIERS", line=770, suggested_type="concept"),
            CoverageGap(section="Grow Your Talent", line=2439, suggested_type="concept"),
            CoverageGap(section="The Power of One", line=6800, suggested_type="tool"),
        ],
        coverage_pct=88.0,
    ),
    consistency=ConsistencyResult(
        invalid_references=[],
        orphan_nodes=["metric-xyz"],
        type_violations=[],
        valid=True,
    ),
    overall_valid=True,  # all 3 checks pass thresholds
)
```

### ValidationReport output (error case)
```python
ValidationReport(
    competency=CompetencyResult(total=20, covered=12, ..., coverage_pct=60.0),
    corpus_coverage=CoverageResult(total_sections=25, covered=15, ..., coverage_pct=60.0),
    consistency=ConsistencyResult(
        invalid_references=[
            InvalidReference(source="tool-opsp", rel_type="requires", target="tool-core-values", reason="Target not found"),
        ],
        orphan_nodes=["stage-expansion"],
        valid=False,
    ),
    overall_valid=False,
)
```

## Acceptance Criteria

### MUST
1. ≥20 competency questions defined covering all 4 decisions + cross-cutting
2. Schema traversal check reports which CQs are covered/uncovered with reasons
3. Corpus coverage check maps major sections (H1 headings) to node types, reports gaps
4. Consistency check validates relationship references exist and flags orphans
5. `ValidationReport` is a Pydantic model with structured, serializable output

### SHOULD
1. Competency questions editable as YAML (not hardcoded)
2. Report includes suggested node types for uncovered corpus sections

### MUST NOT
- Modify any existing ontology YAML files
- Use LLM for validation (this is deterministic/structural)
- Depend on RaiSE framework code (this is scaleupagent-local)
