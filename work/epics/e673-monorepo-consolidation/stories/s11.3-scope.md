# S11.3 Scope

## In Scope

- Move `src/raise_core/` → `packages/raise-core/src/raise_core/` with own pyproject.toml
- Move `src/raise_cli/` → `packages/raise-cli/src/raise_cli/` with own pyproject.toml
- Convert root pyproject.toml to workspace-only (no longer a package itself)
- Update all tool config (ruff, pyright, pytest, coverage, import-linter) for new paths
- Verify `uv sync` resolves all workspace members
- Verify all existing tests pass
- Verify editable install works

## Out of Scope

- Adding packages/rai-agent/ (that's S11.4)
- CI/CD changes (that's S11.5)
- Docker/distribution packaging (that's S11.6)
- Publishing to PyPI
- Changing the `rai` entry point behavior
- Updating the Confluence Dev Environment Setup page

## Done Criteria

1. `packages/raise-core/` exists with own pyproject.toml and `src/raise_core/`
2. `packages/raise-cli/` exists with own pyproject.toml and `src/raise_cli/`
3. Root pyproject.toml is workspace-only (not a package)
4. `uv sync` succeeds
5. `uv run pytest` passes (all existing tests)
6. `uv run ruff check` passes
7. `uv run pyright` passes
8. Editable install resolves both packages

## Risks

- TN-003 gotcha: global `rai` command breaks when raise-core is extracted (mitigated by documenting in scope commit, not blocking)
- Hatch build config needs careful migration — raise-cli wheel must still include only raise_cli, not raise_core
- Entry points (rai.governance.parsers, rai.hooks, rai.gates, etc.) all live in raise-cli — must stay there
- raise-pro depends on raise-cli via workspace — verify dependency chain still works
