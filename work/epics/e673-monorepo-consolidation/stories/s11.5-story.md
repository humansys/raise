# S11.5: CI/CD Workspace — Selective Pipeline per Package

## User Story (Connextra)

As a **raise-commons maintainer**,
I want **CI/CD pipelines that detect which packages changed and run only their validation gates**,
so that **PRs get fast, focused feedback and releases publish the correct package**.

## Acceptance Criteria (Gherkin)

```gherkin
Feature: Selective CI/CD Pipeline

  Scenario: Change in one package triggers only that package's checks
    Given a PR that modifies only files under packages/raise-cli/
    When the CI workflow runs
    Then only raise-cli tests, lint, and type checks execute
    And other package jobs are skipped

  Scenario: Root-level change triggers all packages
    Given a PR that modifies pyproject.toml or conftest.py
    When the CI workflow runs
    Then all package pipelines execute

  Scenario: Multi-package change triggers affected packages
    Given a PR that modifies packages/raise-core/ and packages/raise-cli/
    When the CI workflow runs
    Then raise-core and raise-cli checks execute
    And raise-pro, raise-server, rai-agent are skipped

  Scenario: Release tag publishes specific package
    Given a git tag matching pattern {package}-v{version}
    When the release workflow triggers
    Then only that package is built and published to PyPI

  Scenario: All quality gates pass per package
    Given a package's CI job is triggered
    When the job runs
    Then it executes: pytest, ruff check, ruff format --check, pyright
    And reports status independently
```

## Context

- 5 workspace packages: raise-core, raise-cli, raise-pro, raise-server, rai-agent
- Current ci.yml runs everything monolithically (single pytest, single ruff, single pyright)
- Current release.yml does `uv build` without specifying `--package` — only works for single-package
- rai-agent has extra deps (claude-agent-sdk, telegram, etc.) that need separate sync
