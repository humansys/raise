# S11.5 Design: CI/CD Workspace — Selective Pipeline per Package

## Problem & Value

Current CI/CD workflows are monolithic — any change triggers all tests and the release
workflow does `uv build` without `--package`, only working for single-package repos.

Selective pipelines give faster PR feedback (only affected packages) and independent
per-package releases with dedicated tags.

## Approach

### D1: Change Detection — Native `paths` filters

Use GitHub Actions native `paths` filter on `push`/`pull_request` triggers. Each package
gets its own caller workflow with paths scoped to its directory + root files. Transitive
dependencies handled by including upstream package paths in downstream callers.

**Why not dorny/paths-filter or custom script:** Native filters are declarative, zero
dependencies, and sufficient for our 5-package workspace. Transitive deps are known
and static — no need for dynamic detection.

### D2: Workflow Structure — Reusable workflow + per-package callers

- `.github/workflows/ci-package.yml` — reusable workflow accepting `package`, `test-path`, `src-path`
- `.github/workflows/ci-{package}.yml` — one caller per package with path filters
- Validation steps (pytest, ruff, pyright) defined once in reusable workflow

**Why not matrix:** Matrix with conditional path filters is awkward in GHA — separate
callers are more explicit and each package's trigger conditions are independently readable.

### D3: Release Tags — `{package}-v{version}` pattern

Tag format: `raise-core-v2.4.0`, `rai-agent-v0.1.0`. Release workflow parses tag to
extract package name, builds with `uv build --package {name}`, publishes to PyPI.

## Components Affected

| Component | Change | Notes |
|-----------|--------|-------|
| `.github/workflows/ci-package.yml` | **create** | Reusable workflow |
| `.github/workflows/ci-raise-core.yml` | **create** | Caller with paths filter |
| `.github/workflows/ci-raise-cli.yml` | **create** | Caller, includes raise-core paths |
| `.github/workflows/ci-raise-pro.yml` | **create** | Caller, includes raise-core paths |
| `.github/workflows/ci-raise-server.yml` | **create** | Caller, minimal paths |
| `.github/workflows/ci-rai-agent.yml` | **create** | Caller, includes raise-core paths |
| `.github/workflows/ci.yml` | **delete** | Replaced by per-package callers |
| `.github/workflows/release.yml` | **modify** | Tag parsing + `--package` flag |

## Transitive Dependency Map

```
raise-core  ← raise-cli, raise-pro, rai-agent depend on it
raise-cli   ← raise-pro depends on it (extends CLI)
raise-server ← standalone (no workspace deps)
```

Callers include paths of their workspace dependencies:

| Package | Own paths | Also watches |
|---------|-----------|-------------|
| raise-core | `packages/raise-core/**` | — |
| raise-cli | `packages/raise-cli/**` | `packages/raise-core/**` |
| raise-pro | `packages/raise-pro/**` | `packages/raise-core/**`, `packages/raise-cli/**` |
| rai-agent | `packages/rai-agent/**` | `packages/raise-core/**` |
| raise-server | `packages/raise-server/**` | — |

All callers also watch: `pyproject.toml`, `uv.lock`, `conftest.py`.

## Architecture Review Decisions

### Q1: Dependency installation strategy → `uv sync` (plain)

Parity with local dev. No `--all-extras` (installs unused deps), no `--package`
(premature isolation for 5-package team). Tests mock external services; extras
not needed in CI.

### Q2: Pyright scope → global (`uv run pyright`)

Run without path args — uses `pyrightconfig.json` includes. Same as local.
Negligible time difference at current scale.

### Q3: PyPI environment → single `pypi` environment

One shared environment for all packages. Per-package environments add config
overhead with no benefit at team size of 5. Tag-based workflow already scopes
which package publishes.

**Prerequisite:** Before first release of a new package, manually create the
project on PyPI and add Trusted Publisher pointing to this repo + `release.yml`.

## Acceptance Criteria

### MUST
- AC1: PR touching only `packages/raise-cli/` triggers only raise-cli CI
- AC2: PR touching `pyproject.toml` or `uv.lock` triggers all 5 package CIs
- AC3: Tag `raise-core-v2.4.0` builds and publishes only raise-core
- AC4: Per-package jobs run pytest, ruff check, ruff format --check, pyright
- AC5: Transitive deps — raise-core change triggers raise-cli, raise-pro, rai-agent

### SHOULD
- AC6: Zero duplication — validation steps defined once in reusable workflow

### MUST NOT
- AC7: No changes to codeql.yml or deploy-site.yml
- AC8: No hard-coded Python versions in callers (only in reusable workflow)
