# S11.5 Scope

## In Scope

- Refactor `.github/workflows/ci.yml` to detect changed packages via path filters
- Per-package CI jobs: pytest, ruff check, ruff format --check, pyright
- Root-level changes (pyproject.toml, conftest.py, uv.lock) trigger all packages
- Refactor `.github/workflows/release.yml` for per-package publishing with `{package}-v{version}` tag pattern
- Each package publish uses `uv build --package {name}`
- Matrix or reusable workflow to avoid duplication across 5 packages

## Out of Scope

- Docker packaging (S11.6 / RAISE-677)
- License files, CONTRIBUTING.md, README (S11.6 / RAISE-677)
- CodeQL or deploy-site workflows (unchanged)
- PyPI Trusted Publisher setup for new packages (manual config, not code)
- Branch protection rules (GitHub settings, not workflow code)

## Done Criteria

1. PR with changes only in `packages/raise-core/` triggers only raise-core CI job
2. PR with root changes triggers all package CI jobs
3. `uv run pytest`, `ruff check`, `ruff format --check`, `pyright` run per-package
4. Tag `raise-core-v2.4.0` triggers build + publish only for raise-core
5. Tag `rai-agent-v0.1.0` triggers build + publish only for rai-agent
6. All existing tests still pass on the workflow changes
7. No workflow duplication — reusable workflow or matrix strategy

## Risks

- Path filter detection may miss transitive dependency changes (raise-cli depends on raise-core)
- PyPI Trusted Publisher needs per-package environment setup (manual, not automatable via workflow)
- rai-agent extras ([telegram], [scheduling]) may need special sync in CI
