# S11.3: uv Workspace Setup

## User Story (Connextra)

As a **RaiSE platform developer**,
I want **raise-commons restructured as a uv workspace monorepo with each package in packages/**,
so that **each package can be versioned, tested, and published independently while sharing a single lockfile**.

## Acceptance Criteria (Gherkin)

```gherkin
Scenario: Workspace resolves all members
  Given raise-commons has packages/raise-core/, packages/raise-cli/, packages/raise-pro/, packages/raise-server/
  And the root pyproject.toml defines [tool.uv.workspace] with all members
  When I run `uv sync`
  Then all workspace members resolve without errors
  And a single uv.lock is generated

Scenario: raise-cli depends on raise-core as workspace member
  Given packages/raise-cli/pyproject.toml lists raise-core as a workspace dependency
  When I run `uv run rai --version`
  Then raise-cli resolves raise-core from the workspace (not PyPI)

Scenario: Existing tests pass after restructure
  Given all source code has been moved to packages/
  When I run `uv run pytest`
  Then all existing tests pass with no import errors

Scenario: Editable install works
  Given the workspace is set up
  When I run `pip install -e packages/raise-cli -e packages/raise-core`
  Then both packages are installed in editable mode
  And `from raise_core.graph import models` resolves correctly
```

## Context

- raise-core was extracted to packages/ in S463.1, then reverted in S516/T1 back to src/
- TN-003 documents the `rai` global command issue from that extraction
- raise-pro and raise-server already exist in packages/ as workspace members
- Root pyproject.toml currently IS the raise-cli package (hatch builds src/raise_cli + src/raise_core)
