## T1: Audit Core Layer

**Scope:** `core/`, `config/`, `schemas/`, `rai_base/` (~20 files)

**Modules:**
- `core/` -- shared utilities, constants, types
- `config/` -- Pydantic Settings, paths, loaders
- `schemas/` -- shared data models
- `rai_base/` -- base classes, protocols

**Primary focus:** D3 (API surface, types), D4 (no mutable state)
**Secondary focus:** D5 (docstrings)

**Steps:**
1. Read all `.py` files in each Core module
2. Evaluate against 26 HUMAN/BOTH criteria (skip 11 TOOL-only)
3. Score each module: pass/partial/fail per dimension (D1-D5)
4. Record findings with: criterion ID, severity, file:line, description, fix
5. Note exemplary code patterns
6. Write scorecards to audit-results.md Core section

**Verdict rules:**
- **pass:** 0 critical or must-fix findings in the dimension
- **partial:** 0 critical, 1+ must-fix findings
- **fail:** 1+ critical findings

**Verification:** 4 module scorecards written. Every finding has criterion ID + file:line.

**Commit:** `feat(s370.4): audit Core layer (config, schemas, core, rai_base)`

## T8: Test Sampling

**Scope:** 2-3 representative test files per module across all layers (~20-30 files sampled from ~251 total)

**Focus:** D5.3 (test assertion quality and naming), Appendix A5 (vague test assertions)

**Strategy:**
- For each source module audited in T1-T7, read 2-3 test files from the corresponding `tests/` directory
- Evaluate test naming: behavior-spec names (`test_parse_jql_returns_empty_on_invalid_input`) vs generic (`test_parse`)
- Evaluate assertion specificity: `assert result.name == "expected"` vs `assert result is not None`
- Check for AI tells (A5: vague assertions, A6: uniform structure)
- If 3+ sampled test files share the same issue, mark it as **systemic** (applies to all test files in that module)

**Sampling plan:**
| Layer | Modules to sample | Test files per module |
|-------|-------------------|----------------------|
| Core | config, schemas | 2-3 each |
| Domain (small) | discovery, telemetry | 2 each |
| Domain (medium) | graph, skills | 2-3 each |
| Domain (large) | gates, hooks | 2-3 each |
| Domain (gov) | governance | 2 |
| Application | session, handlers | 2-3 each |
| Presentation | cli | 3 |

**Verification:** At least 2 test files sampled per layer. Systemic patterns identified where 3+ files share issues.

**Commit:** `feat(s370.4): audit test sampling across all layers`

## Execution Notes

- **No TDD cycle:** This is an analysis story. Tasks produce findings in audit-results.md, not code.
- **No code modifications:** MN1 constraint -- only audit-results.md and plan artifacts are modified.
- **TOOL-only criteria skipped:** MN2 constraint -- only 26 HUMAN/BOTH criteria evaluated.
- **Session continuity:** Tasks T1-T7 are independent of each other (all depend only on T0). They can be executed in any order or split across sessions.
- **Progress tracking:** Each task commits its batch, enabling resume from any point.
- **Layer order:** Design decision D2 prescribes Core first, then Domain, Application, Presentation. This lets core patterns inform upper-layer evaluation.
