## What Went Well

1. **What/How/Why triple as forcing function.** The structured format prevented vague platitudes. Every criterion had to answer three concrete questions, which naturally filtered out criteria that sounded good but were unverifiable.

2. **TOOL/HUMAN/BOTH classification.** Adding a third category (BOTH) emerged during writing when criteria like D1.1 (exception handling) had a tool-checkable component (E722) plus a judgment component (Exception broadness). This honest classification is more useful than forcing binary.

3. **Research-to-governance pipeline.** The 24 sources and 8 triangulated claims from the research phase gave the standard an evidence base. Criteria reference specific sources, which makes them defensible rather than opinion-based.

4. **Sequential dimension tasks.** Breaking the work into one task per dimension kept commits focused and reviewable. Each dimension built on the document structure established in T1.

5. **DRAFT-first approach.** Marking the document as DRAFT with explicit revision expectations (S370.4 audit) prevents premature rigidity. The standard will evolve from practice, not from theory alone.

## What to Improve

1. **Criterion count balance.** Dimensions range from 6 to 8 criteria. Some dimensions (D2: Readability) could have been trimmed — 8 criteria for readability is a lot to check manually. The audit (S370.4) will reveal which criteria actually matter in practice.

2. **AI-tells section is speculative.** The anti-patterns appendix draws from research but has not been validated against our actual codebase. It may need significant revision after the audit.

## Heutagogical Checkpoint

### 1. What did you learn?

The What/How/Why triple is a general-purpose pattern for governance criteria, not just code standards. Any governance artifact (guardrails, review checklists, quality gates) benefits from forcing each rule to be concrete (What), actionable (How), and justified (Why). Vague criteria like "write clean code" fail all three tests.

### 2. What would you change about the process?

Nothing significant. The research-first approach (E370 started with research before codifying) paid off — every criterion traces to evidence. The only minor friction was deciding between TOOL and HUMAN classification for borderline cases, which resolved naturally by adding BOTH.

### 3. Are there improvements for the framework?

The skill pipeline (research -> design -> plan -> implement -> review) worked well for governance artifacts, not just code. This suggests governance stories can follow the same skill cycle as code stories without adaptation. The framework already supports this.

### 4. What are you more capable of now?

Translating research evidence into structured governance. The specific skill: taking N sources with overlapping claims and distilling them into M concrete, verifiable criteria with clear classification for automation potential.

## Improvements Applied

- None needed for framework or skills. The existing skill cycle handled a governance-only story without friction.
