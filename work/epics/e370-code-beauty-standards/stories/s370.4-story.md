# Story: Codebase Audit

## User Story
As a project maintainer,
I want a systematic audit of every module against our code quality standard,
so that I know exactly what needs fixing before publishing open source.

## Acceptance Criteria

### Scenario: Full codebase coverage
```gherkin
Given the code-standards-draft.md with 37 criteria across 5 dimensions
And the /rai-code-audit skill is available
When I run the audit across all src/raise_cli/ and tests/ modules
Then every module has a scorecard with pass/partial/fail per dimension
```

### Scenario: Findings categorized by severity
```gherkin
Given the audit produces findings across modules
When the results are compiled
Then each finding is categorized as critical/must-fix/recommended/observation
And findings are grouped by ADR-001 architectural layer
```

### Scenario: Actionable output for S370.5
```gherkin
Given the audit-results.md is complete
When a developer reads it
Then critical and must-fix items are specific enough to act on without re-analysis
And each finding references the criterion it violates
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| `src/raise_cli/config.py` | Audit against D1-D5 | Scorecard: D1=pass, D2=partial, D3=pass, D4=pass, D5=pass |
| `tests/test_config.py` | Audit against D1-D5 | Scorecard + finding: "test names don't follow given-when-then" |
| All 444 files | Compile results | audit-results.md with summary table + per-layer detail |
