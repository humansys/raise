# S370.5b Design: Extract Helpers

## Problem

25+ functions exceed 100 lines across CLI commands (6 files), governance parsers (3 files), discovery scanner, hooks emitter, and MCP Jira adapter. Long functions hinder readability, testability, and maintainability. They accumulate cyclomatic complexity that makes code review unreliable.

## Value

- Functions become single-responsibility, testable, and readable at a glance
- Removes all `noqa: C901` suppressions from targeted modules
- Passes senior developer code review for open source publication
- Prepares modules for deeper restructuring in S370.5c/d/e

## Approach

Extract long functions into named private helpers within the same module. No architectural changes, no file moves -- just function decomposition. Each extraction follows RED-GREEN-REFACTOR: existing tests must pass before and after.

### GovernanceParser Protocol (R8)

9 parser classes already implement the same interface:
- `can_parse(locator: ArtifactLocator) -> bool`
- `parse(locator: ArtifactLocator) -> list[GraphNode]`

Define a `GovernanceParser` Protocol in `governance/parsers/__init__.py`. All 9 classes (EpicScopeParser, AdrParser, RoadmapParser, VisionParser, ConstitutionParser, BacklogParser, PrdParser, GlossaryParser, GuardrailsParser) already conform -- no code changes needed beyond the Protocol definition and typing the extractor to use it.

Additionally: justify all `except Exception` in `governance/extractor.py` (11 occurrences) with inline comments explaining why broad catch is intentional (error isolation for multi-file extraction).

### ThreadPoolExecutor Lifecycle (R9)

Current `_call_hook_with_timeout` creates a new `ThreadPoolExecutor(max_workers=1)` per hook call with `shutdown(wait=False, cancel_futures=True)`. Fix: use context manager pattern:

```python
with ThreadPoolExecutor(max_workers=1) as executor:
    future = executor.submit(hook.handle, event)
    return future.result(timeout=timeout)
```

This ensures proper thread cleanup even on timeout/exception paths.

### MCP Jira Format Handlers (R10)

`_parse_issue_detail` and `_parse_search_results` both branch on `is_flat` with near-identical field extraction. Extract a shared `_extract_issue_fields(data: dict, is_flat: bool) -> dict` helper that returns normalized field values. Both parsers then construct their domain model from the normalized dict.

### Global Output Format (R4)

`_current_output_format` in `cli/main.py` is a module-level global written by `main()` callback and read by `__main__.py` error handler. The error handler runs outside Typer context, so it cannot access `ctx.obj`. Options:

1. **Store format on RaiError** -- error handler reads `error.output_format`
2. **Use ContextVar** -- thread-safe, but adds framework concept
3. **Wrap app() to capture format** -- catch-and-rethrow pattern

Decision: analyze during implementation, pick simplest option that removes the global. If architecture change is too risky, document the global with a justifying comment and defer removal.
