# S370.5d: Bundle Split

## User Story (Connextra)

As a contributor reading raise-commons, I want session/bundle.py split into focused modules so that data fetching, formatting, and assembly are separate responsibilities with clear boundaries.

## Acceptance Criteria (Gherkin)

```gherkin
Scenario: Bundle assembly produces identical output
  Given the refactored module structure
  When assemble_context_bundle() is called
  Then it returns the same output as before the split

Scenario: Formatters are isolated
  Given bundle_formatters.py exists
  When I inspect its contents
  Then it contains only _format_* functions with no data fetching

Scenario: Data fetchers are isolated
  Given bundle_data.py exists
  When I inspect its contents
  Then it contains only data-loading functions (graph queries, live status)

Scenario: Assembly orchestrates
  Given bundle.py is the orchestrator
  When I inspect its contents
  Then it imports from formatters and data, and contains only assembly logic

Scenario: All tests pass unchanged
  Given the refactored modules
  When I run the full test suite
  Then all existing tests pass

Scenario: Type checks and lint pass
  Given the refactored code
  When I run pyright and ruff
  Then 0 errors reported
```
