# Story Design: S370.1 — Draft Code Standard

## Summary

Codify the 5-dimension code quality framework from research (24 sources, 8 triangulated claims) into a concrete governance draft at `.raise/governance/code-standards-draft.md`. Each criterion has a What/How/Why triple and is classified as Tool-Verifiable or Human-Judgment.

## Complexity: Moderate

- Single new file (governance document)
- No external integrations or algorithm complexity
- Requires synthesizing research into concrete, verifiable criteria

## Problem and Value

**Problem:** We have research findings (24 sources, 5 dimensions) but no codified standard to evaluate our code against.

**Value:** Without a concrete standard, "beautiful code" is subjective. With it, we can audit systematically (S370.4) and build tooling (S370.2) and review skills (S370.3) around it.

## Approach

The standard document will:

1. Cover all 5 dimensions from research, maintaining the same structure for traceability
2. Each criterion has: **What** to check (specific), **How** to verify (tool or checklist), **Why** it matters (sourced)
3. Classify criteria as **Tool-Verifiable** (Ruff rule, Pyright check, import-linter contract) vs **Human-Judgment** (naming quality, idiom elegance, test specification quality)
4. Architecture dimension references ADR-001 layers explicitly
5. Type safety dimension references ADR-002 Pydantic pattern
6. Include AI-generated code tells as anti-patterns
7. Marked as **DRAFT** -- will evolve after codebase audit (S370.4)

## Criterion Format Example

Each criterion follows this structure:

```markdown
### D1.1: No bare except clauses

- **What:** Every `except` clause catches a specific exception type, never bare `except:` or `except Exception:`
- **How:** `ruff check --select E722` (bare except) + manual review for overly broad `except Exception`
- **Why:** "Errors should never pass silently" (PEP 20). Bare excepts hide bugs, catch KeyboardInterrupt, mask real errors. (Sources: PEP 20, Slatkin Items 80-88, Google Style Guide)
- **Verifiable by:** Tool (E722) + Human (Exception broadness)
```

## Key Decisions

### D1: DRAFT status -- standard evolves from practice

**Choice:** Mark as DRAFT, expect revision after audit.
**Rationale:** Cannot know which criteria matter until we audit real code (S370.4). Premature finalization risks criteria that don't match codebase reality.
**Alternatives rejected:** Final standard from research alone; no standard (ad hoc review).

### D2: Tool-Verifiable vs Human-Judgment classification

**Choice:** Every criterion classified into one of two categories.
**Rationale:** Enables S370.2 (tooling) to know what to automate and S370.3 (skill) to know what to evaluate manually. Clear separation of concerns.
**Alternatives rejected:** Single flat checklist; separate tool config and review guide.

### D3: What/How/Why triple for every criterion

**Choice:** Structured triple rather than prose.
**Rationale:** What makes it concrete, How makes it actionable, Why traces to evidence. Without all three, criteria become vague platitudes.
**Alternatives rejected:** Simple rule list; detailed prose per dimension.

### D4: Maintain research dimension structure

**Choice:** 5 dimensions match research report exactly.
**Rationale:** Preserves traceability from evidence catalog through research report to governance artifact. Changing structure would lose source attribution.
**Alternatives rejected:** Reorganize by tool; reorganize by severity.
