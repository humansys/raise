# Story S370.1: Draft Code Standard

## User Story

**As** a developer contributing to raise-commons,
**I want** a concrete, verifiable code quality standard organized by dimension,
**So that** code reviews have objective criteria instead of subjective opinions.

## Size: M

## Acceptance Criteria

```gherkin
Feature: Code Quality Standard Draft

  Scenario: Standard covers all 5 dimensions
    Given the research report with 24 sources and 8 triangulated claims
    When the standard is codified
    Then `.raise/governance/code-standards-draft.md` exists
    And it contains sections for all 5 dimensions:
      | Dimension                       |
      | Correctness and Safety          |
      | Readability and Idiom           |
      | Type Safety and API Surface     |
      | Architecture and Design         |
      | Collaboration and Maintenance   |

  Scenario: Each criterion is verifiable
    Given a criterion in any dimension
    Then it has a "What to check" description (specific, not vague)
    And it has a "How to verify" method (tool command or review checklist)
    And it has a "Why it matters" rationale (sourced from research)

  Scenario: Tool-verifiable vs human-judgment classification
    Given a criterion in the standard
    Then it is classified as either "tool-verifiable" or "human-judgment"
    And tool-verifiable criteria reference specific Ruff rules, Pyright checks, or CLI commands

  Scenario: Document is marked as draft
    Given the code-standards-draft.md file
    Then the filename contains "draft"
    And the document header states it is a draft subject to revision after audit
```
