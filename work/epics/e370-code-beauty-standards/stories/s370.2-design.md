# Story Design: S370.2 -- Expand Tooling Gates

**Story:** S370.2 | **Epic:** E370 Code Beauty Standards
**Complexity:** MODERATE
**Date:** 2026-03-06

---

## Problem

Our Ruff configuration only covers 9 rule sets (E, W, F, I, N, UP, B, C4, SIM). The code-standards-draft (S370.1 output) identifies 11 additional rule sets that should be enabled to automate Dimension 1 (Correctness & Safety) and parts of Dimension 2 (Readability & Idiom). We also have no architecture enforcement -- ADR-001 defines clear layers but nothing prevents imports from violating them.

## Value

After this story, every `git commit` automatically enforces:
- Security hygiene (S/bandit rules)
- Complexity cap (C90, max-complexity = 10)
- Return consistency, unused args, performance anti-patterns
- Test style consistency
- Architecture layer contracts (import-linter)

The "senior review" bar becomes partially automated. Human reviewers can focus on D2-D5 judgment criteria instead of catching things tools should catch.

## Approach

Three changes, executed incrementally.

### Change 1: Ruff Expansion (+11 Rule Sets)

Enable one rule set at a time in risk-first order. For each:
1. Add to `[tool.ruff.lint] select`
2. Run `ruff check src/ tests/`
3. Fix violations (prefer fix over noqa)
4. Add justified noqa only for: false positives, intentional patterns (PAT-E-589), or fixes requiring larger refactor (defer to S370.5)
5. Commit

**Order (risk-first):**

| # | Rule Set | Category | Expected Impact |
|---|----------|----------|-----------------|
| 1 | S | flake8-bandit (security) | Hardcoded secrets, eval, pickle |
| 2 | C90 | McCabe complexity | Functions > 10 cyclomatic complexity |
| 3 | PL | Pylint subset | Broad convention/error/refactor |
| 4 | PERF | Perflint | Unnecessary allocations |
| 5 | FURB | refurb | Modern Python idioms |
| 6 | RET | flake8-return | Mixed return styles |
| 7 | ARG | flake8-unused-arguments | Dead parameters |
| 8 | ERA | eradicate | Commented-out code |
| 9 | PT | flake8-pytest-style | Test style consistency |
| 10 | TC | flake8-type-checking | TYPE_CHECKING imports |
| 11 | A | flake8-builtins | Builtin shadowing |

**Per-file-ignores needed upfront:**
- `tests/**/*.py`: `S101` (assert is how pytest works), `ARG001` (unused fixture parameters are normal)

### Change 2: import-linter

Add `import-linter` as dev dependency. Configure layer contracts in `pyproject.toml`:

```toml
[tool.importlinter]
root_packages = ["raise_cli"]

[[tool.importlinter.contracts]]
name = "ADR-001 Layer Contract"
type = "layers"
layers = [
    "raise_cli.cli | raise_cli.output | raise_cli.viz",
    "raise_cli.session | raise_cli.context | raise_cli.onboarding | raise_cli.handlers | raise_cli.doctor | raise_cli.governance | raise_cli.artifacts | raise_cli.skills | raise_cli.backlog | raise_cli.hooks | raise_cli.gates | raise_cli.publish | raise_cli.agents",
    "raise_cli.engines | raise_cli.discovery | raise_cli.graph | raise_cli.tier | raise_cli.memory",
    "raise_cli.core | raise_cli.config | raise_cli.schemas | raise_cli.adapters | raise_cli.rai_base",
]
```

The exact layer mapping will be validated during implementation -- if `lint-imports` reveals unexpected cross-layer imports, we fix the imports (not the architecture), per scope.md.

### Change 3: McCabe Complexity Cap

Part of C90 enablement:

```toml
[tool.ruff.lint.mccabe]
max-complexity = 10
```

Functions exceeding complexity 10 will be decomposed into smaller functions.

## Key Decisions

### D1: Fix Over Ignore

Prefer fixing violations over adding noqa. Only noqa when:
- False positive
- Fix requires larger refactor (defer to S370.5)
- Intentional pattern (e.g., PAT-E-589 module-level Path constants)

**Rationale:** noqa suppresses permanently. Fixing raises quality.

### D2: Risk-First Ordering

Security (S) first, cosmetic (A) last. Finding a hardcoded secret is more valuable than finding a shadowed builtin.

### D3: Layer Contracts Match Actual Modules

ADR-001 defines 4 conceptual layers. The contracts map the real ~25 raise_cli subpackages to those layers. This preserves ADR-001 intent while matching actual code structure.

### D4: import-linter as Dev Dependency

Development/CI tool only. Not needed at runtime. Justified per D5.6: standard tool for Python architecture enforcement, referenced in code-standards-draft.md.

## Acceptance Criteria

**MUST:**
- AC1: All 11 new Ruff rule sets enabled in pyproject.toml
- AC2: `ruff check src/ tests/` passes with zero violations
- AC3: import-linter contracts defined per ADR-001 layers
- AC4: `lint-imports` passes with zero violations
- AC5: McCabe max-complexity = 10 configured and passing

**SHOULD:**
- SC1: Violations fixed rather than ignored (noqa count < 20 new additions)
- SC2: Each rule set enabled in a separate commit for traceability

**MUST NOT:**
- MN1: Must NOT add noqa without justification comment
- MN2: Must NOT change functional behavior (only config + style fixes)

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|-----------|
| Hundreds of violations from PL rules | Medium | Per-rule ignores for noisy sub-rules (PLR09xx thresholds) |
| import-linter reveals systemic layer violations | Low | Fix imports not architecture; escalate if systemic |
| C90 violations require non-trivial decomposition | Low | Extract helper functions; complex cases deferred to S370.5 |
| S rules flag false positives (e.g., test passwords) | Medium | Per-file-ignores for test files |

## Verification

```bash

grep -c "noqa" src/ tests/ -r            # SC1: < 20 new
```
