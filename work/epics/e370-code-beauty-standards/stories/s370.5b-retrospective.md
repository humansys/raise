## What Went Well

1. **Risk-first ordering paid off.** T1-T3 (small, independent) built confidence before the riskier T4 (architecture change) and bulk extractions (T5-T8). No cascading failures.
2. **Dramatically faster than estimated.** 1h45m vs 5-8h estimate. Pure extraction refactoring with good existing test coverage is fast when the plan is precise. The detailed extraction inventory in the design doc made each task mechanical.
3. **Review process caught real issues.** The duplicate Protocol (AR C1) and the blocking ThreadPoolExecutor (QR R3) would have been latent bugs without post-implementation review.
4. **ContextVar was the right call for R4.** Clean 1:1 replacement of the module global -- no architectural changes needed, thread-safe by default.

## What to Improve

1. **Check existing codebase before defining new Protocols.** T2 added a GovernanceParser Protocol that already existed in adapters/protocols.py. The design phase should have grep'd for existing Protocol definitions. Cost: one fix commit + wasted review time.
2. **Context manager is not always the right cleanup pattern.** ThreadPoolExecutor's `__exit__` calls `shutdown(wait=True)`, which blocks until the timed-out thread finishes. For fire-and-forget timeout patterns, explicit `shutdown(wait=False)` is correct. Design doc assumed context manager was universally better.
3. **Estimation for pure-extraction refactoring is consistently too high.** S370.5a was also faster than estimated. When tests exist and the extraction inventory is detailed, the work is largely mechanical. For future stories: estimate pure extraction at 30-50% of "normal" refactoring time.

## Heutagogical Checkpoint

**1. What did I learn?**
- `ThreadPoolExecutor.__exit__` calls `shutdown(wait=True)` -- context managers are not universally safe for timeout patterns. The blocking behavior is by design (CPython source confirms).
- Protocol definitions should be searched for before creating new ones. `grep -r "class.*Protocol" src/` should be a pre-task check for any Protocol work.
- ContextVar is a clean drop-in replacement for module-level globals when the only need is cross-module read access within a single call stack.

**2. What would I change about the process?**
- Add a "check existing patterns" step to the design phase for any task that introduces a Protocol, base class, or shared abstraction. A 30-second grep avoids a duplicate + fix commit cycle.
- Adjust estimation heuristics: pure extraction with existing tests = 20-30 min per file, not 1 hour.

**3. Are there improvements for the framework?**
- The design template could include a "check for existing abstractions" prompt when the task involves defining new Protocols or interfaces.
- Estimation guidance could distinguish between "refactoring with existing tests" vs "refactoring requiring new tests" with different multipliers.

**4. What am I more capable of now?**
- Confident in large-scale function extraction across 13+ files in a single story without breaking anything. The key is a detailed inventory + commit-per-task + existing test coverage.
- Better judgment about when context managers are appropriate vs explicit resource management.

## Patterns

### New

- **PAT-pending-1:** Before defining a new Protocol or abstract base class, grep the codebase for existing ones (`grep -r "class.*Protocol" src/`). Duplicates create confusion and require fix commits.
- **PAT-pending-2:** ThreadPoolExecutor context manager calls `shutdown(wait=True)` on exit, which blocks until threads finish. For timeout patterns where you want to abandon the thread, use explicit `shutdown(wait=False, cancel_futures=True)` instead.
- **PAT-pending-3:** Pure extraction refactoring with existing test coverage takes 30-50% of estimated time when the extraction inventory is detailed. Adjust estimates accordingly.
