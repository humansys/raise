# Retrospective: S370.2 — Expand Tooling Gates

**Story:** S370.2 | **Epic:** E370 Code Beauty Standards

## Summary

Enabled 11 new Ruff rule sets (S, C90, PL, PERF, FURB, RET, ARG, ERA, PT, TC, A) and added import-linter with ADR-001 architecture contracts. Started with 2036 violations measured in T1; all resolved to zero remaining.

## What Went Well

- **Early signal task (T1)** dimensioned the real work: 2036 violations measured before fixing. This informed the strategy — global ignores for rules with 100+ violations from legitimate patterns vs per-file fixes for the rest.
- **Risk-first ordering** worked: security (S) and complexity (C90) first, style (A) last.
- **One commit per rule set** made each change reviewable and revertible.
- **Global ignore strategy** was pragmatic: PLC0415 (inline imports: 628), PLR2004 (magic values in tests: 494), ARG002 (protocol conformance: 196) — global ignores with TOML comments beat hundreds of per-line noqa.
- **TC (type checking imports) globally ignored** — correct decision given Pydantic runtime dependency on imports that TC001-TC004 would move to TYPE_CHECKING blocks.
- **import-linter pragmatism:** Two focused contracts (foundation independence + no upward CLI dependency) enforce the most important ADR-001 invariants without being brittle. Full 4-layer strict contract was too rigid.

## What to Improve

- **max-complexity deviation:** Design said 10, implementation used 15. At max-complexity=10, ~20+ more functions would need noqa. At 15, only 9 need it. Decision: start at 15, tighten to 10 as part of S370.5 (function decomposition). AR flagged this correctly.
- **noqa budget slightly exceeded:** 22 noqa vs design's budget of 20 (SC1). QR flagged this. Two over budget is acceptable but signals the budget was tight.
- **C901 deferrals:** 9 functions with complexity 11-15 deferred to S370.5. These are real debt, not ignored debt — tracked and scoped.
- **CI gate parity gap discovered post-implementation.** Story passed local tests but CI failed on `ruff check` (import sorting) and `ruff format` (56 files). Led to E474 creation.
- **Epic scope commit on wrong branch.** E474 initialization happened on this story branch instead of dev because the epic-start skill didn't verify the current branch.

## Heutagogical Checkpoint

1. **What did I learn?** Incremental linter expansion with measurement-first approach prevents surprise scope. The early signal pattern (measure before fixing) is generalizable beyond lint — any large-scale change benefits from dimensioning first. Also learned that import-linter's full strict layer contracts are brittle; focused contracts targeting the highest-value invariants are more sustainable.

2. **What would I change about the process?** The design should have included a decision tree for max-complexity threshold rather than a fixed number. When you don't know the violation landscape, specify a range (e.g., 10-15) with criteria for choosing within it, rather than a single number that may need deviation.

3. **Are there improvements for the framework?** Skills must run all CI gates, not just tests (created E474). Skills need branch guards before committing (PAT-E-010). The story-review skill Step 1 only verifies tests — should verify full gate suite.

4. **What am I more capable of now?** Ruff configuration strategy at scale (20 rule sets). import-linter contract design — knowing when strict layers help vs when focused contracts are better. Systematic tooling expansion with measurement-first approach.
