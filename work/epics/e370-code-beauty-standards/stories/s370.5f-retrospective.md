# S370.5f Retrospective: Migrate Loaders to YAML

**Story:** S370.5f | **Size:** M | **Date:** 2026-03-06
**Jira:** RAISE-500

## Summary

Replaced fragile markdown regex/frontmatter parsing in identity and architecture loaders with deterministic YAML reading. Clean cut — YAML is source of truth, MD is never parsed.

## What Went Well

- **TDD cycle** worked perfectly — RED confirmed fixture changes broke things, GREEN confirmed loader rewrites fixed them
- **Identity loader** reduction was dramatic: 200 lines of regex → 102 lines of YAML reading
- **Architecture loader** simplification was clean: dropped frontmatter extraction, just `yaml.safe_load()` directly
- **Onboarding** consolidation: 3 regex-parsing methods → 1 clean YAML method, `re` import removed entirely
- **Design decision D3** (clean cut, no fallback) kept the code simple

## What to Improve

- **Test fixture migration scope** was underestimated. T2 plan said "~3 test files" but actual work touched 33 fixture occurrences across 5 test files + contract tests. Should have counted occurrences, not files.
- **Regex-based file transformation** failed 3 times before switching to state machine. Should have started with state machine for multi-line block transformations.

## Heutagogical Checkpoint

1. **Learned:** State-machine approach for test fixture migration; `yaml.safe_load()` + `cast()` pattern for pyright
2. **Process change:** Count fixture occurrences, not just files, when estimating migration tasks
3. **Framework:** No changes needed — TDD and skill cycle worked well
4. **New capability:** Confident pattern for YAML-based loaders across the codebase

## Patterns

- **PAT-E-658:** yaml.safe_load() + cast() for pyright strict
- **PAT-E-659:** State machine over regex for bulk test fixture migration
- **PAT-E-597:** Reinforced (future annotations + Any from yaml)
