# Story S370.3: Scope

## In Scope

Create `/rai-code-audit` skill in `.raise/skills/`. The skill:

- Takes a module path or list of modules as input
- Evaluates against code-standards-draft.md dimensions (D1-D5)
- Focuses on HUMAN and BOTH criteria that tools cannot catch
- Produces a scorecard with findings categorized by severity (critical, must-fix, should-fix, consider)
- Follows the existing skill YAML format (see run-gate.yaml, check-compliance.yaml)
- Includes Jidoka stops for incoherent input or evaluation failures

### Dimensions Covered

| Dimension | Focus | Example Criteria |
|-----------|-------|-----------------|
| D1: Correctness & Safety | BOTH criteria (broad exception review, resource cleanup judgment) | Exception broadness justification, custom resource managers |
| D2: Readability & Idiom | HUMAN criteria (naming, idiom quality, docstring substance) | Intention-revealing names, Pythonic patterns, docstring value |
| D3: Type Safety & API Surface | HUMAN criteria (API minimality, Pydantic model design) | Public interface size, model field validation |
| D4: Architecture & Design | HUMAN criteria (SRP, dependency direction, module cohesion) | Responsibility clarity, layer contract intent |
| D5: Collaboration & Maintenance | HUMAN criteria (test specification, PR readiness) | Tests as specification, change history clarity |

## Out of Scope

- Running the audit against modules (S370.4)
- Fixing findings discovered by audit (S370.5)
- Modifying existing `/rai-quality-review` skill (S370.6 reconciliation)
- Automating TOOL-classified criteria (already covered by Ruff/Pyright/import-linter)

## Done When

1. Skill file exists at `.raise/skills/code-audit.yaml`
2. Validates via `rai skill validate`
3. Follows skill template structure consistent with existing skills
4. Covers all 5 dimensions with focus on HUMAN criteria
5. Input/output contracts are well-defined
6. Severity classification scheme is documented in the skill
