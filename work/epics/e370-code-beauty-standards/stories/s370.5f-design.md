# S370.5f Design: Migrate Loaders to YAML

## Problem & Value

Two loaders parse free-form markdown with regex (identity) or frontmatter extraction (architecture) instead of reading structured YAML. This is fragile — any formatting change in core.md breaks the identity loader's 150 lines of regex. The artifacts loader already reads pure YAML and is the target pattern.

**Value:** Deterministic parsing, ~80% code reduction in identity loader, consistency across all loaders.

## Approach

### A. Identity Loader (core.md → core.yaml)

Create `core.yaml` as structured source of truth. The identity loader drops from 200 lines of regex to ~30 lines of YAML reading.

**core.yaml structure:**

```yaml
values:
  - number: 1
    name: "Honesty over Agreement"
    description: "tell you when you're wrong, push back on bad ideas, admit when I don't know"
  - number: 2
    name: "Simplicity over Cleverness"
    description: "the simple solution that works > the elegant solution that's complex, do we really need this abstraction?"
  # ...

boundaries:
  will:
    - "push back on bad ideas"
    - "stop when I detect incoherence, ambiguity, or drift"
    # ...
  wont:
    - "pretend certainty I don't have"
    - "validate ideas just because they were proposed"
    # ...

principles:
  - name: "Simplicity over Completeness"
    embodiment: "i push back on over-engineering. \"Do we really need this?\""
  # ...
```

The loader reads this and produces the same `GraphNode` list (same IDs, same types, same metadata).

### B. Architecture Loader (frontmatter-in-md → pure YAML)

Architecture docs already have all data in YAML frontmatter — the markdown body is human documentation. The loader only reads the frontmatter anyway.

**Approach:** Migrate `governance/architecture/*.md` and `governance/architecture/modules/*.md` to `.yaml` extension with the same YAML content (no frontmatter delimiters). The markdown body becomes a `description` field or is dropped (loader never used it).

The `_parse_architecture_doc` function simplifies: no frontmatter extraction, just `yaml.safe_load()`.

### C. Generators (rai init / onboarding)

- `onboarding/bootstrap.py` — change `identity_files = ["core.md"]` → `["core.yaml"]`
- `onboarding/instructions.py` — reads core.md for CLAUDE.md generation; adapt to read core.yaml
- Template for core.yaml replaces template for core.md

### D. Existing file migration

- Convert `.raise/rai/identity/core.md` → `core.yaml` in this repo
- Convert `governance/architecture/*.md` → `*.yaml` in this repo

## Decisions

| ID | Choice | Rationale | Alternatives |
|----|--------|-----------|--------------|
| D1 | core.yaml flat structure (values/boundaries/principles) | Matches loader consumption — one file, 3 sections | Separate files per section (overkill) |
| D2 | Keep architecture MD alongside YAML for humans | Human docs useful; loader reads YAML only, MD is generated/maintained separately | Delete MD entirely (loses documentation) |
| D3 | Clean cut — YAML only, no fallback to .md | Simpler code, no legacy paths; MD is never parsed | Backward compat fallback (adds complexity for no gain) |

## Acceptance Criteria

- **AC1**: `load_identity()` reads `core.yaml`, produces same GraphNode IDs/types/metadata
- **AC2**: `load_architecture()` reads `*.yaml` files, produces same GraphNode IDs/types/metadata
- **AC3**: Identity loader has no `re` import
- **AC4**: `rai init` generates `core.yaml` for new projects
- **AC5**: All existing tests pass (adapted for YAML fixtures)
- **AC6**: pyright 0 errors, ruff clean
