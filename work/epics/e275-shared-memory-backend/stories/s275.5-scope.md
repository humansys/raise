# S275.5 — Skills & Templates sync + serve — Scope

## In Scope
- `POST /api/v1/skills/sync` — CLI uploads skills (replace strategy)
- `GET /api/v1/skills` — list available skills (name + description)
- `GET /api/v1/skills/{name}` — get full skill content (markdown)
- `POST /api/v1/templates/sync` — CLI uploads templates (replace strategy)
- `GET /api/v1/templates` — list available templates (name + description)
- `GET /api/v1/templates/{name}` — get full template content (markdown)
- Pydantic schemas for sync payloads and responses
- DB tables or storage for skills and templates (per org)
- Service layer functions
- FastAPI route definitions with auth (OrgContext)
- 404 for missing skill/template
- Tests: schemas, db, services, API integration

## Out of Scope
- Skill execution (server-side inference) → future story if Rovo can't handle
- Skill versioning / history → not needed for POC
- Template rendering / variable substitution → Rovo does this with its inference
- CLI sync command implementation → separate story (S275.6 or new)
- Trace/impact/constraints endpoints → moved to parking lot

## Done When
- `POST /api/v1/skills/sync` stores skills, returns count
- `GET /api/v1/skills` lists skills with name + description
- `GET /api/v1/skills/{name}` returns full markdown content
- `POST /api/v1/templates/sync` stores templates, returns count
- `GET /api/v1/templates` lists templates with name + description
- `GET /api/v1/templates/{name}` returns full markdown content
- 404 for nonexistent skill/template
- All tests pass (schemas, db, services, API)
- Type checks + lint pass
- Three-layer pattern followed (routes → services → db/queries)
