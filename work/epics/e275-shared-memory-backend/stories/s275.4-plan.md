## Execution Order

```
T1 (schemas) → T2 (db/queries) → T3 (service) → T4 (routes) → T5 (integration)
```

**Rationale:** Bottom-up — models first, then data layer, then business logic, then transport. Each layer tested in isolation before wiring. Risk-first: T2 (upsert SQL) is the riskiest task, positioned early.

## Risks

| Risk | L/I | Mitigation |
|------|:---:|------------|
| ON CONFLICT upsert syntax complexity with SA 2.0 | M/M | Use `postgresql.insert()` dialect-specific function. Well-documented pattern. |
| to_tsvector performance at scale | L/L | Acceptable for <10K nodes. Materialized tsvector column in parking lot. |
| Edge FK resolution (domain node_id → DB UUID) | M/M | Resolve in service layer: build node_id→UUID map after upsert, use for edge insertion. |
