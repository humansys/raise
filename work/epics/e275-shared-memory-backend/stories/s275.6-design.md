# S275.6 — ApiGraphBackend + DualWrite — Design

## Complexity: Moderate (3 components, 1 external integration)

## Problem

The CLI persists graphs only to local filesystem. Teams cannot share knowledge graphs across repos. The server endpoints exist (S275.4) but no CLI client connects to them.

## Value

With `RAI_SERVER_URL` set, `rai graph build` pushes the graph to the shared server — enabling cross-repo queries, team intelligence, and Rovo agent access. Zero workflow change for the developer.

## Approach

Three new components in **rai-cli** (not rai-core — avoids polluting core's minimal dependency tree with httpx), one modification to existing CLI code:

| Component | Change | Package / Location |
|-----------|--------|----------|
| `ApiGraphBackend` | **Create** | `rai-cli/graph/backends/api.py` |
| `DualWriteBackend` | **Create** | `rai-cli/graph/backends/dual.py` |
| `get_active_backend()` | **Create** | `rai-cli/graph/backends/__init__.py` |
| CLI commands | **Modify** | `rai-cli/cli/commands/graph.py` (import from new factory) |

rai-core keeps: protocol, `BackendHealth`, `FilesystemGraphBackend`, and its own simple `get_active_backend()` (filesystem-only). rai-cli's factory wraps it with env-var-based selection.

**Why not rai-core?** rai-core is the shared domain with minimal deps (pydantic + networkx). httpx is an HTTP client dependency that only the CLI consumer needs. Keeping rai-core lean means community tier doesn't pull unnecessary deps.

**Key decisions:**

1. **httpx (sync)** — Protocol methods are synchronous (`persist()`, `load()`, `health()`). httpx supports sync mode natively. No async needed. httpx added to rai-cli deps (already has rich, typer, etc.).
2. **Env vars** — `RAI_SERVER_URL` (base URL), `RAI_API_KEY` (API key). Consistent with 12-factor and existing server auth (`rsk_` prefix keys).
3. **DualWrite strategy** — Local first (always succeeds), then remote (best-effort). `load()` reads from local only — local is source of truth. Remote failure = warning log, never exception.
4. **Serialization** — `Graph` → sync payload: iterate `iter_concepts()` and `iter_relationships()` to build node/edge dicts. Server payload matches existing schemas from S275.4.
5. **CLI factory** — `rai_cli.graph.backends.get_active_backend()` checks env vars, returns `DualWriteBackend` if server configured, else delegates to rai-core's `FilesystemGraphBackend`. CLI commands import from rai-cli factory.

## Examples

### ApiGraphBackend usage

```python
from rai_cli.graph.backends.api import ApiGraphBackend

backend = ApiGraphBackend(
    server_url="http://localhost:8000",
    api_key="rsk_test_abc123",
    project_id="raise-commons",
)

# Persist
backend.persist(graph)  # POST /api/v1/graph/sync

# Health
health = backend.health()  # GET /health
# BackendHealth(status="healthy", message="...", metadata={"backend": "api", ...})

# Load — not supported standalone (local is source of truth)
backend.load()  # raises NotImplementedError
```

### DualWriteBackend usage

```python
from rai_cli.graph.backends.dual import DualWriteBackend

dual = DualWriteBackend(local=filesystem_backend, remote=api_backend)

# Persist writes to both — remote failure is warning
dual.persist(graph)
# INFO: Persisted to local filesystem
# INFO: Synced to remote server (42 nodes, 18 edges)
# — or if server down —
# INFO: Persisted to local filesystem
# WARNING: Remote sync failed: ConnectionError(...). Graph saved locally only.

# Load reads from local only
graph = dual.load()  # delegates to local.load()

# Health shows both
health = dual.health()
# BackendHealth(status="degraded", message="local: healthy, remote: unavailable", ...)
```

### get_active_backend() selection (CLI factory)

```python
import os
from rai_cli.graph.backends import get_active_backend

# Without server config → FilesystemGraphBackend (from rai-core)
backend = get_active_backend(path=Path(".raise/rai/memory/index.json"))

# With server config → DualWriteBackend (local + remote)
os.environ["RAI_SERVER_URL"] = "http://localhost:8000"
os.environ["RAI_API_KEY"] = "rsk_test_abc123"
backend = get_active_backend(path=Path(".raise/rai/memory/index.json"))
```

### Graph → sync payload serialization

```python
# Inside ApiGraphBackend.persist():
nodes = [
    {
        "node_id": node.id,
        "node_type": node.type,
        "scope": "project",
        "content": node.content,
        "source_file": node.source_file,
        "properties": node.metadata,
    }
    for node in graph.iter_concepts()
]
edges = [
    {
        "source_node_id": edge.source,
        "target_node_id": edge.target,
        "edge_type": edge.type,
        "weight": edge.weight,
        "properties": edge.metadata,
    }
    for edge in graph.iter_relationships()
]
payload = {
    "project_id": self.project_id,
    "nodes": nodes,
    "edges": edges,
}
```

### Error cases

```python
# Server unreachable during persist
backend.persist(graph)
# WARNING: Remote sync failed: httpx.ConnectError: ... — graph saved locally only

# Server unreachable during health
health = backend.health()
# BackendHealth(status="unavailable", message="Connection refused", metadata={"backend": "api"})

# Missing API key with server URL set
# get_active_backend logs warning, falls back to FilesystemGraphBackend
```

## Load Strategy

The `load()` on `ApiGraphBackend` needs to retrieve all nodes+edges. Two options:

- **Option A:** GET /api/v1/graph/query with a wildcard — but query endpoint does text search, not "get all".
- **Option B:** Add a GET /api/v1/graph/nodes endpoint on server (S275.6 scope creep).

**Decision:** For now, `ApiGraphBackend.load()` is not the primary path — `DualWriteBackend.load()` reads from local. `ApiGraphBackend.load()` raises `NotImplementedError` with a clear message. If needed later (S275.7 offline sync), we add a server endpoint then.

## Acceptance Criteria

**MUST:**
1. `ApiGraphBackend.persist(graph)` sends correct sync payload to server
2. `ApiGraphBackend.health()` returns healthy/unavailable based on server status
3. `DualWriteBackend.persist()` writes to both; remote failure = warning, not exception
4. `DualWriteBackend.load()` reads from local only
5. `get_active_backend()` in rai-cli returns `DualWriteBackend` when env vars set, `FilesystemGraphBackend` otherwise
6. CLI commands use rai-cli factory (not rai-core's)

**SHOULD:**
1. `DualWriteBackend.health()` aggregates both backend statuses
2. Timeout on HTTP calls (5s connect, 30s read for sync of large graphs)

**MUST NOT:**
- Add httpx to rai-core dependencies
- Raise exceptions on remote failures during persist
- Change existing `FilesystemGraphBackend` behavior
- Require async in the protocol (keep synchronous)
