# S275.1: Extract rai-core Package

**JIRA:** [RAISE-276](https://humansys.atlassian.net/browse/RAISE-276)
**Absorbs:** [RAISE-145](https://humansys.atlassian.net/browse/RAISE-145) (drop "Unified" prefix)
**Size:** S
**Epic:** E275 — Shared Memory Backend

## User Story

As a RaiSE platform developer,
I want the graph domain extracted into a standalone `rai-core` package,
so that both rai-cli (COMMUNITY) and rai-server (PRO) can share models, protocols, and backends without circular dependencies.

## Acceptance Criteria

```gherkin
Scenario: rai-core package exists as uv workspace member
  Given the monorepo root pyproject.toml
  When I run `uv sync`
  Then rai-core is resolved as a workspace package
  And rai-cli depends on rai-core with exact version pin

Scenario: Graph domain lives in rai_core.graph
  Given the rai_core package
  When I import from rai_core.graph.models
  Then GraphNode, GraphEdge, and all 18 node subclasses are available
  And when I import from rai_core.graph.engine
  Then Graph (formerly UnifiedGraph) is available
  And when I import from rai_core.graph.query
  Then QueryEngine (formerly UnifiedQueryEngine) is available

Scenario: Backward compatibility via re-exports
  Given existing code importing from rai_cli.context.models
  When I run the full test suite
  Then all tests pass without import changes
  And old names (UnifiedGraph, UnifiedQueryEngine) still resolve

Scenario: Only graph domain moved to core
  Given rai_cli.adapters.protocols
  Then ProjectManagementAdapter, GovernanceSchemaProvider, GovernanceParser, DocumentationTarget still live there
  And only KnowledgeGraphBackend moved to rai_core

Scenario: Placeholder structure for future domains
  Given rai_core package
  Then rai_core/workflow/ exists with __init__.py and docstring
  And rai_core/governance/ exists with __init__.py and docstring

Scenario: All quality gates pass
  When I run pytest, pyright, and ruff
  Then all pass with zero regressions
```
