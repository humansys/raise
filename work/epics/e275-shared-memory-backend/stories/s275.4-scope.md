# S275.4 Scope: Graph Sync + Query Endpoints

## In Scope
- `POST /api/v1/graph/sync` — idempotent full-graph upsert (nodes + edges)
- `GET /api/v1/graph/query` — keyword search using PG GIN full-text index
- `db/queries.py` — specialized SQL (upsert ON CONFLICT, GIN ts_vector search)
- `services/graph.py` — service layer orchestrating PG ↔ domain logic
- `api/v1/graph.py` — thin route handlers (validation + delegation to service)
- Pydantic request/response models for both endpoints
- Org isolation (WHERE org_id filter on all queries)
- OpenAPI spec auto-generated (deliverable to Fernando)
- Tests for all layers (routes, service, queries)

## Out of Scope
- Trace/impact/constraints endpoints (S275.5)
- CLI integration / ApiGraphBackend (S275.6)
- Alembic migration (schema unchanged from S275.2)
- Rate limiting / throttling
- Pagination on sync (full graph fits in single request for now)
- Advanced search (embeddings, fuzzy matching beyond GIN)
- Graph algorithm execution on server side

## Done When
- `POST /api/v1/graph/sync` upserts nodes+edges, returns counts
- `GET /api/v1/graph/query?q=keyword` returns matching nodes with scores
- All queries are org-scoped (no cross-org data leaks)
- OpenAPI spec is clean and deliverable to Fernando
- All tests pass (new + existing)
- pyright + ruff clean
