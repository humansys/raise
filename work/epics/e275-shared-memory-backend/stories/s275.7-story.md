# S275.7 — Dogfood + Offline Fallback

## User Story

**As a** developer using raise-cli with DualWriteBackend,
**I want** failed remote writes to be queued and retried on reconnect,
**So that** my workflow isn't disrupted by network issues and data eventually reaches the server.

## Acceptance Criteria

```gherkin
Feature: Dogfood + Offline Fallback

  Scenario: Real graph syncs through DualWrite pipeline
    Given raise-commons knowledge graph is built with rai graph build
    And RAI_SERVER_URL and RAI_API_KEY are configured
    When I run rai graph build
    Then the graph persists locally AND syncs to the remote server
    And the server responds with sync confirmation

  Scenario: Failed remote write creates pending_sync marker
    Given RAI_SERVER_URL is configured but the server is unreachable
    When I run rai graph build
    Then the graph persists locally successfully
    And a pending_sync marker file is created in .raise/
    And the marker contains the timestamp and graph hash

  Scenario: Sync pending data on reconnect
    Given a pending_sync marker exists from a previous offline build
    And the server is now reachable
    When I run rai graph sync-pending (or next rai graph build)
    Then the pending graph data is synced to the server
    And the pending_sync marker is removed

  Scenario: No marker when remote succeeds
    Given RAI_SERVER_URL is configured and reachable
    When I run rai graph build
    Then no pending_sync marker is created
    And existing pending_sync markers are cleared after successful sync
```
