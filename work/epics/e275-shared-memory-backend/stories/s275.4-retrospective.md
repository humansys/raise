# S275.4 Retrospective: Graph Sync + Query Endpoints

**Story:** S275.4 — Graph Sync + Query Endpoints
**Dates:** 2026-02-26 (single session)
**Estimated:** ~100 min | **Actual:** ~90 min | **Velocity:** ~1.1x

## Summary

Delivered two domain-level endpoints (`POST /graph/sync`, `GET /graph/query`) with three-layer architecture (routes → services → db/queries). Quality review caught 2 critical and 4 recommended issues, all resolved before close. OpenAPI spec clean and deliverable to Fernando.

## What Went Well

- **Bottom-up plan held perfectly** — schemas → queries → service → routes. No backtracking.
- **Quality review found real bugs** — C1 (silent edge drop) was a semantic correctness issue that no automated gate would catch. C2 (FK precondition) documented a latent bug.
- **Test isolation by layer** — each layer mocked its dependencies cleanly. 83 tests total, no integration test needed for correctness.
- **SA 2.0 upsert worked first try** — `pg_insert().on_conflict_do_update()` on named constraint.

## What to Improve

- **Design should list failure modes for data ops** — the silent edge drop (C1) should have been caught at design time, not QR. A "what if inputs reference missing entities?" checklist item would have caught it.
- **FastAPI dependency mocking** — wasted ~10 min discovering that `unittest.mock.patch` doesn't work for `Depends()`. Now captured as PAT-E-522.
- **async_sessionmaker mocking** — another ~5 min on the mock pattern. Now captured as PAT-E-524.

## Heutagogical Checkpoint

1. **Learned:** FastAPI Depends captures at registration (PAT-E-522), SA rowcount doesn't distinguish insert/update, async CM mocking pattern (PAT-E-524)
2. **Process change:** Add "failure modes" section to design doc for data mutation stories
3. **Framework:** QR before close continues to justify itself. Consider design checklist addition.
4. **Capability:** Three-layer async FastAPI pattern now established — S275.5 will be mechanical
