# S275.2: Retrospective — PostgreSQL Schema + Alembic

**Story:** S275.2 | **Date:** 2026-02-25 | **Size:** S
**Estimated:** ~1.25h | **Actual:** ~1.1h | **Velocity:** ~1.1x

## Summary

Created `rai-server` as uv workspace package with 4 SQLAlchemy 2.0 models (Organization, ApiKey, GraphNodeRow, GraphEdgeRow), Alembic initial migration, and Docker Compose with PG 16 + server placeholder. Quality review caught 2 `default=` vs `server_default=` bugs before they could propagate to S275.3+.

## What Went Well

- **Interactive design paid off.** 7 decisions made with human input — no rework during implementation.
- **Concrete design examples → mechanical implementation.** Models were nearly copy-paste from design doc.
- **Quality review caught real bugs.** `default=True` on `is_active` and `default=1.0` on `weight` would have caused NULL failures on raw SQL inserts. Fixed before they could propagate.
- **TDD with inspection API.** 24 structural tests validate schema without needing a real DB — fast, reliable, CI-friendly.
- **Zero regressions.** 2773 existing tests unaffected.

## What to Improve

- **Design examples should be reviewed for consistency.** The `default=` vs `server_default=` inconsistency originated in the design doc's examples. A "consistency scan" in design phase would have caught this earlier.
- **`uv run --package rai-server` pattern.** Took a moment to figure out that standalone workspace packages need `--package` flag. Worth documenting for S275.3.

## Heutagogical Checkpoint

1. **Learned:** `default=` vs `server_default=` distinction in SA. Quality review is the right gate for semantic bugs that linters miss.
2. **Process change:** Add consistency check to design examples (all columns should use server_default uniformly for a DB-first approach).
3. **Framework improvement:** None needed — the quality review skill worked exactly as designed.
4. **More capable at:** uv multi-package workspace patterns, SA 2.0 Mapped[] columns, Alembic async migrations.
