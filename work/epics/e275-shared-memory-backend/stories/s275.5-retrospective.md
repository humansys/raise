## What to Improve

- **Consumer research before scope** — S275.5 was scoped 3 times because we didn't understand Rovo's capabilities upfront
- **UX research gate for APIs** — PAT-E-263 didn't trigger because the story was framed as "backend endpoints", but the consumer (Rovo) needed understanding first
- **Scope commit timing** — the initial scope commit had trace/impact/constraints; ideally scope commit comes after consumer validation for API stories

## Heutagogical Checkpoint

1. **Learned:** Rovo agents = prompt + actions. Skills live in Confluence (Rovo reads natively). Our API provides context + telemetry, not content serving.
2. **Change:** Do consumer research before story scope for API stories.
3. **Framework:** Add consumer-validation gate to story-start for API stories. Expand UX gate (PAT-E-263) to cover API consumers.
4. **More capable:** Rovo/Forge architecture understanding. Three-layer pattern is now zero-friction.
