# S275.4 Design: Graph Sync + Query Endpoints

**JIRA:** [RAISE-279](https://humansys.atlassian.net/browse/RAISE-279)
**Size:** M | **Complexity:** Moderate (4 components, custom SQL)

## Problem

The shared memory server (S275.3) has auth and health but no graph operations. CLI users can't sync or query knowledge graphs. Fernando is blocked waiting for the OpenAPI spec.

## Value

After this story, a CLI `rai graph build` can push its graph to the server, and any team member can query it. Fernando gets the OpenAPI spec to start Forge actions.

## Approach

Three-layer architecture (thin routes → service → SQL queries), two endpoints:

1. **POST /api/v1/graph/sync** — full-graph replace per (org, project). Upsert nodes, replace edges, prune orphan nodes. Single transaction.
2. **GET /api/v1/graph/query** — keyword search across node_id, node_type, content using `to_tsvector()` on the fly (no schema change needed).

### Key Design Decision: Sync Strategy

**Replace, not merge.** Each sync sends the complete graph for a project. The server:
1. Upserts nodes via `ON CONFLICT (org_id, repo_id, node_id) DO UPDATE`
2. Deletes all edges for (org_id, repo_id), bulk inserts new ones
3. Deletes orphan nodes (existing nodes for this org+repo not in incoming set)

Why replace: the CLI is the source of truth (`rai graph build` is deterministic from code). Incremental merge adds complexity without value — the full graph is small (<10K nodes).

### Key Design Decision: Search Without Migration

The current schema has GIN on `properties` JSONB but no `tsvector` column. Options:

| Option | Pros | Cons |
|--------|------|------|
| A. `to_tsvector()` in WHERE (no migration) | Zero schema change, works now | No index, full scan |
| B. Add tsvector generated column + GIN | Indexed, fast | Requires Alembic migration |
| C. ILIKE on content | Simplest | No ranking, no stemming |

**Choice: A** — use `to_tsvector('english', node_id || ' ' || node_type || ' ' || content)` in queries. At our scale (<10K nodes/org), this is fast enough. Add the materialized column (B) later if performance needs it.

            "node_type": "module",

            "properties": {"language": "python", "lines": 245}

        {
            "node_id": "mod-graph",
            "node_type": "module",

            "content": "Knowledge graph building and querying",

        }

        {
            "source_node_id": "mod-memory",

## Pydantic Models

```python
# schemas/graph.py

class NodeInput(BaseModel):
    node_id: str                          # Domain ID (e.g., "mod-memory")
    node_type: str                        # "module", "class", "function", etc.
    scope: str = "project"                # "project", "personal", "org"
    content: str                          # Description / serialized content
    source_file: str | None = None
    properties: dict[str, Any] = {}

class EdgeInput(BaseModel):
    source_node_id: str                   # Domain ID of source node
    target_node_id: str                   # Domain ID of target node
    edge_type: str                        # "depends_on", "contains", etc.
    weight: float = 1.0
    properties: dict[str, Any] = {}

class GraphSyncRequest(BaseModel):
    project_id: str                       # Repo/project identifier
    nodes: list[NodeInput]
    edges: list[EdgeInput]

class GraphSyncResponse(BaseModel):
    status: str = "ok"
    project_id: str
    nodes_created: int
    nodes_updated: int
    edges_created: int
    nodes_pruned: int

class NodeResult(BaseModel):
    node_id: str
    node_type: str
    scope: str
    content: str
    source_file: str | None
    properties: dict[str, Any]
    rank: float

class GraphQueryResponse(BaseModel):
    results: list[NodeResult]
    total: int
    query: str
    limit: int
```

## Acceptance Criteria

**MUST:**
1. `POST /api/v1/graph/sync` upserts nodes, replaces edges, prunes orphans — all in one transaction
2. `GET /api/v1/graph/query?q=keyword` returns matching nodes ranked by ts_rank
3. All data is org-scoped (no cross-org leaks)
4. Invalid payloads return 422 with clear errors
5. OpenAPI spec auto-generated and clean

**SHOULD:**
1. Sync response includes accurate created/updated/pruned counts
2. Query supports `limit` parameter (default 20, max 100)

**MUST NOT:**
- Expose internal DB UUIDs (use domain node_id)
- Allow sync without authentication
- Require Alembic migration
