# S275.7 — Dogfood + Offline Fallback — Design

## Problem

When `DualWriteBackend` fails to sync remotely, the failure is logged and forgotten. There's no way to know a sync is pending or to retry it later.

## Value

Developers get reliable eventual consistency — offline work syncs automatically on next successful build, with visible markers showing pending state.

## Approach

Modify `DualWriteBackend` to manage a `pending_sync` marker file when remote fails. On next successful remote persist, clear the marker. No new CLI commands — `rai graph build` is the retry mechanism.

**Components:**
- `src/rai_cli/graph/backends/dual.py` — modify: add marker create/clear logic
- `src/rai_cli/graph/backends/pending.py` — create: `PendingSyncMarker` model + read/write helpers
- `tests/graph/test_pending_sync.py` — create: marker lifecycle tests
- `tests/graph/test_dual_backend.py` — modify: add marker integration tests

## Examples

### Marker file (`.raise/pending_sync.json`)

```json
{
  "timestamp": "2026-02-26T14:30:00Z",
  "graph_path": ".raise/rai/graph-index.json",
  "node_count": 42,
  "edge_count": 67,
  "error": "Connection refused"
}
```

### DualWriteBackend.persist() — happy path (server up)

```python
backend = DualWriteBackend(local=fs_backend, remote=api_backend)
backend.persist(graph)
# → local persists ✓
# → remote persists ✓
# → pending_sync.json deleted if it existed
```

### DualWriteBackend.persist() — offline (server down)

```python
backend = DualWriteBackend(local=fs_backend, remote=api_backend)
backend.persist(graph)
# → local persists ✓
# → remote fails → warning logged
# → .raise/pending_sync.json created with error summary
```

### Next build — server back up

```python
backend.persist(graph)  # next rai graph build
# → local persists ✓
# → remote persists ✓ (server recovered)
# → .raise/pending_sync.json deleted
# → log: "Cleared pending sync marker — remote sync successful"
```

### PendingSyncMarker model

```python
from pydantic import BaseModel
from datetime import datetime

class PendingSyncMarker(BaseModel):
    timestamp: datetime
    graph_path: str
    node_count: int
    edge_count: int
    error: str
```

### Helper functions

```python
def write_pending_marker(raise_dir: Path, marker: PendingSyncMarker) -> None:
    """Write pending_sync.json to .raise/ directory."""

def read_pending_marker(raise_dir: Path) -> PendingSyncMarker | None:
    """Read pending_sync.json, return None if not present."""

def clear_pending_marker(raise_dir: Path) -> bool:
    """Delete pending_sync.json. Return True if file existed."""
```

## Acceptance Criteria

**MUST:**
1. Failed remote persist creates `.raise/pending_sync.json` with timestamp, counts, and error
2. Successful remote persist deletes `.raise/pending_sync.json` if it exists
3. Marker model is Pydantic with typed fields
4. All existing tests pass (no regressions)
5. Type checks + lint pass

**SHOULD:**
- Log message when clearing a pending marker ("Cleared pending sync marker")
- Log message when creating marker ("Remote sync failed, marked as pending")

**MUST NOT:**
- Change the local persist behavior (always succeeds or raises)
- Add new CLI commands (retry = just run `rai graph build` again)
- Add background retry / daemon / scheduler
