# S1130.5 Retrospective

## Summary

Implemented `generate_jira_config()` pure function with workflow state merging, status mapping, and issue type generation. 15 tests, all passing. Dogfood against live Jira caught a bug in S1130.2's `get_issue_types()` — API returns `issueTypes` key, not `values`.

## What went well

- S1130.4 pattern transferred directly — core generator was fast (T1 in ~5 min)
- Merge/dedup logic for multi-project workflows and issue types was clean
- Dogfood caught a real API bug that all 21 mocked tests in S1130.2 missed
- AR/QR were clean — only one minor refactor (consolidate set computation)

## What to improve

- S1130.2 mocked tests assumed `values` key without verifying against live API response
- Need a convention for optional live integration tests (`@pytest.mark.live`) to catch API shape mismatches
- Estimated 25 min, actual ~30 min — dogfood time adds ~20% overhead but catches real bugs

## Heutagogical Checkpoint

1. **Learned:** API v2 vs v3 response format differences — `issueTypes` vs `values`. Mocks that don't mirror real responses are a liability.
2. **Process change:** Always dogfood discovery-dependent code before closing the story, not as optional extra.
3. **Framework improvement:** `@pytest.mark.live` convention for API shape validation (tracked, not blocking).
4. **Capability:** Multi-section config generation with cross-project dedup. Merge pattern is reusable.
