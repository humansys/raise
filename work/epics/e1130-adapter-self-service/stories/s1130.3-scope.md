# S1130.3 Scope: Adapter Doctor Check

## In Scope

- `AdapterDoctorCheck` class implementing `DoctorCheck` Protocol
- Category: `"adapters"`, check_id: `"adapters"`
- Three-level checks per adapter (Jira + Confluence):
  1. Config file exists (.raise/jira.yaml, .raise/confluence.yaml)
  2. Required env vars set (API tokens, usernames)
  3. Live backend health (online mode only — `requires_online = True` for live checks)
- Registered via `rai.doctor.checks` entry point in pyproject.toml
- Actionable fix hints for each failure mode

## Out of Scope

- Config content validation beyond existence (schema validation is in config loaders)
- Auto-fix capabilities (doctor informs, doesn't repair)
- Discovery or config generation (S1130.1-5 handle that)
- Changes to existing doctor infrastructure (protocol, runner, report)

## Done Criteria

1. `AdapterDoctorCheck.evaluate()` returns results for both adapters
2. Config-missing → WARN with hint to run `/rai-adapter-setup`
3. Env-var-missing → ERROR with specific variable name
4. Live health check → PASS/ERROR based on connectivity (online only)
5. Registered in pyproject.toml entry points
6. All tests pass, types check, linting clean
7. Module: `packages/raise-cli/src/raise_cli/doctor/checks/adapters.py`
