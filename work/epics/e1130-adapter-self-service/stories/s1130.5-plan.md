# S1130.5 Implementation Plan

## Overview

Pure function `generate_jira_config()` that takes `JiraProjectMap` + user selections and returns a dict passing `JiraConfig.model_validate()`. Applies pattern from S1130.4 with richer output (workflow states, issue types, status mapping).

## Tasks

### T1: Core generator + schema validation (S)

**RED:** Test that `generate_jira_config()` with minimal inputs returns a dict that passes `JiraConfig.model_validate()`.

**GREEN:** Implement `generate_jira_config()` producing `default_instance`, `instances`, `projects` sections.

**REFACTOR:** Clean up.

- **Files:** `jira_config_gen.py` (create), `test_jira_config_gen.py` (create)
- **AC:** Scenarios 1, 4, 5
- **Tests:** minimal config validates, invalid project raises ValueError, multiple projects
- **Verify:** `uv run pytest --tb=short`, `uv run ruff check src/ tests/`, `uv run ruff format --check src/ tests/`, `uv run pyright`

### T2: Workflow states + status mapping (S)

**RED:** Test that generated config includes `workflow.states` and `workflow.status_mapping` from discovered `WorkflowState` data.

**GREEN:** Add workflow generation logic — map `WorkflowState` → states list + derive `status_mapping` from state names.

**REFACTOR:** Extract helper if needed.

- **Files:** `jira_config_gen.py`, `test_jira_config_gen.py`
- **AC:** Scenario 2
- **Tests:** workflow states present, status mapping keys derived from names, empty workflows handled
- **Verify:** same gates
- **Depends on:** T1

### T3: Issue types + integration test (S)

**RED:** Test that generated config includes `issue_types` section from discovered `IssueTypeInfo` data.

**GREEN:** Add issue types generation.

**REFACTOR:** Final cleanup. Manual integration test against live Jira.

- **Files:** `jira_config_gen.py`, `test_jira_config_gen.py`
- **AC:** Scenario 3
- **Tests:** issue types present, empty issue types handled
- **Verify:** same gates + manual dogfood
- **Depends on:** T1

## Execution Order

```
T1 (core + validation) → T2 (workflows) ──┐
                         T3 (issue types) ─┘ (T2, T3 independent but sequential for cleaner commits)
```

Risk-first: T1 validates the schema contract (highest risk). T2/T3 are additive.

## Risks

| Risk | Mitigation |
|------|------------|
| JiraConfig extra fields not validated by model_validate | Test that workflow/issue_types keys survive round-trip |
| Status mapping key derivation from names | Use simple slugify (lowercase, spaces→hyphens) |
