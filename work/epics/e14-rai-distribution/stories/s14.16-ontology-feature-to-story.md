# S14.16: Ontology Rename — Feature to Story

> **Size:** L (5 SP)
> **Epic:** E14 Rai Distribution
> **Dependencies:** None (blocking: all future work uses new terminology)
> **Priority:** Critical — ship clean ontology for F&F release (Feb 9)

## Rationale

Our backlog ontology mixes cultures: Epics (agile) containing Features (pre-agile/SAFe).
Analysis of 38 features across 8 epics shows:
- 47% are S-sized (2-3 SP) — behave like stories, not features
- Decomposition is always Feature → Tasks (never Feature → Sub-features)
- Jira mapping (E8) already says `RaiSE Feature → Jira Story`
- Glossary contradicts itself about the hierarchy

**New ontology:** `Epic → Story → Task` (aligns with Jira's native hierarchy)

### Boundary: What Changes vs What Stays

**RENAME to "story":**
- All process terminology: "feature cycle", "feature branch", "/feature-*" skills
- Work unit references: "feature lifecycle", "feature scope", "feature design"
- Directory names: `features/` → `stories/`, `feature-start/` → `story-start/`
- CLI enums and schemas: `work_type="feature"` → `work_type="story"`
- Code symbols: `extract_features()` → `extract_stories()`, `ConceptType.FEATURE` → `ConceptType.STORY`
- Tagline: "feature by feature" → "one story at a time"

**KEEP as "feature":**
- Historical IDs: F14.1, F14.2, F7.3 etc. (they were features)
- Archive files (read-only history)
- Product/commercial usage: "enterprise features", "team features" (roadmap terms)
- Code concepts unrelated to work lifecycle (if any future "feature flag" usage)
