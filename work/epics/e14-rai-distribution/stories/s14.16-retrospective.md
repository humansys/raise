## What Went Well

- **Phased approach worked:** Green-to-green commits meant every phase was independently recoverable. Phase 1 (dirs, 116 files), Phase 2+3 (docs, 203 files), Phase 4 (src+tests, 39 files) each left tests passing.
- **TDD caught real bugs:** 5 distinct test failures in Phase 4 caught issues (NameError, assertion mismatches, backward compat gaps) that would have been silent in production.
- **Backward compat was deliberate:** The loader/builder reading old `"feature"` keys from historical JSONL was identified upfront in design (D1) and implemented cleanly.
- **Comprehensive final audit:** The grep-based integration test found remaining references that multi-pass tools missed (CLI param names, variable names, docstring prose).
- **Decisions upfront (D1-D7):** Having clear design decisions before any code eliminated ambiguity during implementation. No "should I rename this?" moments.

## What Could Improve

- **repren/fastmod weren't actually used:** Despite planning around deterministic tooling, the actual execution relied on inference-based edits (Read + Edit) due to context window carrying the full scope. The plan was correct in principle but the tooling overhead (installing, pattern files, dry-run cycles) exceeded the cost of careful inference-based edits on a codebase we know well.
- **Multiple cleanup passes:** Phases 4c and 4d (CLI help text, deep cleanup) were additional passes that should have been caught in the initial Phase 4. Root cause: the initial repren batch + inference pass focused on code symbols but missed prose in CLI help strings and base pattern files.
- **Stale unified graph:** After renaming `ConceptType.FEATURE` → `ConceptType.STORY`, the cached `unified.json` became invalid because it contained `type: "feature"` nodes. This wasn't anticipated — we need a pattern for "rebuild graph after schema changes."
- **Context window pressure:** The story spanned 3 sessions (context continuations), partly because the scope was genuinely L-sized and partly because each cleanup pass discovered more references.

## Heutagogical Checkpoint

### What did you learn?
- **Large-scale renames have a long tail.** The 80/20 rule applies: 80% of references are caught by the first batch pass, but the remaining 20% require multiple targeted passes through different layers (code symbols → CLI params → help text → docstrings → comments → base data files).
- **Graph caches are schema-dependent.** When you change a Literal type constraint, any cached data with old values becomes a validation bomb.
- **Backward compat is a first-class concern for JSONL data.** Unlike databases with migrations, JSONL files are append-only and historical entries can't be retroactively updated.

### What would you change about the process?
- **Add "rebuild indexes" to the ontology rename checklist.** Any rename that touches Pydantic Literal types should include a task to rebuild/invalidate cached graphs.
- **Single deep pass > multiple shallow passes.** For Phase 4, one comprehensive audit (src/ grep for `\bfeature\b` at the start) would have prevented the 4c/4d cleanup commits.

### Are there improvements for the framework?
- **New pattern: Ontology renames need graph rebuild.** When schema Literals change, stale caches break query. Add to checklist.
- **New pattern: Long-tail cleanup for cross-cutting renames.** Plan for 3 passes: batch, interactive, comprehensive audit.

### What are you more capable of now?
- **Executing large cross-cutting refactors** with confidence (315 files, 12 commits, zero regressions).
- **Backward compat patterns** for append-only JSONL data stores.
- **Schema migration awareness** — knowing where cached data breaks when types change.
