# S250.1 Design: Contract Infrastructure + Pilot

## What & Why

**Problem:** 23 skills grew organically — inconsistent structure, 35-75% substance ratio, duplicated cross-cutting content across files. Agents comply less as instruction count grows (p(each)^n).

**Value:** Establish a reusable pattern (shared preamble + contract template) and validate it on 3 diverse skills before batch application. If the pattern works here, S250.2/S250.3 become mechanical.

## Approach

Three deliverables:

### 1. Shared Preamble (`skills_base/preamble.md`)

Extract content that appears in 3+ skills into a single file loaded once per session via CLAUDE.md. Skills reference it, don't duplicate it.

**Content to extract:**
- ShuHaRi level definitions (currently in 21/23 skills, slightly different each time)
- Gate declaration semantics (what prerequisites/gates mean, how to check)
- Common verification pattern (the "Verification: X / If blocked: Y" format)
- Graph query pattern (`rai graph query` usage)

**What stays in each skill:** A 3-line ShuHaRi section with behavior deltas specific to that skill, not the universal definitions.

### 2. Contract Template (`skills_base/contract-template.md`)

A reference template implementing ADR-040's 7-section canonical structure:

```markdown
# {Skill Title}

## Purpose
One sentence: what + success criteria.

## Mastery Levels (ShuHaRi)
≤5 lines. Behavior deltas only (definitions are in preamble).

## Context
When to use / skip. Inputs. Prerequisites (gates).
Decision tables for conditionals.

## Steps
### Step N: {Verb Phrase}
- What to do
- Verification: how to check
- If blocked: recovery

## Output
Artifacts produced. State changes. Next skill.

## Quality Checklist
Atomic, verifiable items. Last thing agent reads.

## References
Links only. File paths, ADRs, related skills.
```

### 3. Pilot Refactoring (3 skills)

| Skill | Current | Target | Key Compression |
|-------|:-------:|:------:|-----------------|
| `rai-session-start` | 148 lines | ≤130 | Remove Notes section, add Quality Checklist, keep Steps lean (already close) |
| `rai-session-close` | 176 lines | ≤150 | Complement of session-start — validates preamble extraction for the pair |
| `rai-story-implement` | 236 lines | ≤150 | Remove Progress Template (bloat), compress Notes→Quality Checklist, merge Steps 0+0.1+1 |

**Compression strategies (from ADR-040):**
1. **Remove philosophy** — motivation paragraphs don't improve compliance
2. **Templates to references** — inline templates are the biggest bloat; link to them instead
3. **Merge fractional steps** — 0, 0.1, 0.2, 1, 1.5 → clean integer sequence
4. **Tables over prose** — decision heuristics as tables, not paragraphs
5. **ShuHaRi to preamble** — universal definitions extracted, only deltas remain
6. **Notes→Quality Checklist** — useful notes become checklist items; philosophy deleted

## Acceptance Criteria

**MUST:**
1. Shared preamble exists at `skills_base/preamble.md` with ShuHaRi definitions + common patterns
2. Contract template exists at `skills_base/contract-template.md` with 7 sections
3. All 3 pilot skills have exactly 7 sections in ADR-040 canonical order
4. All 3 pilot skills ≤150 lines (excluding YAML frontmatter)
5. All 3 pilot skills have ≥80% substance ratio (no philosophy, no inline templates, no duplicate content)

**SHOULD:**
- Affirmative phrasing throughout (≤3 negative phrases per skill)
- Decision tables replacing prose conditionals

**MUST NOT:**
- Lose critical step content — every step's core action must survive compression
- Break skill functionality — refactored skills must still guide the agent correctly
- Add new content — this is compression, not enhancement
