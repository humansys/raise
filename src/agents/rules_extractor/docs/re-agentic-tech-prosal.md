# PydanticAI-Based Business Rule Extraction Agent Proposal

## System Architecture Overview

 ([image]()) *High-level architecture of the PydanticAI-based rule extraction pipeline.*  
The proposed solution employs a modular pipeline that ingests legacy AS/400 artifacts, applies an LLM-driven analysis, and outputs structured business rules. At a high level, the system consists of an **LLM Extraction Agent**, a **Schema Formatter & Validator**, and a **Metadata/Knowledge Store** orchestrated by a multi-agent framework. We leverage **PydanticAI** – an agent framework by the Pydantic team – to manage these components due to its built-in support for type-safe outputs and multi-agent workflows ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=The%20Pydantic%20team%2C%20known%20for,attention%20in%20the%20developer%20community)) ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=PydanticAI%20introduces%20an%20agent,share%20state%20and%20communicate%20seamlessly))】. The agent treats source code as data, first parsing it into an abstract syntax structure (where possible) before invoking the LLM; this ensures we capture the code’s intrinsic structure rather than treating it as plain tex ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=CodeConcise%20is%20inspired%20by%20the,stored%20in%20a%20graph%20database))】. The LLM extraction agent identifies business logic in code, which is then converted into structured rule objects via Pydantic models. A validation module checks each extracted rule for accuracy and assigns confidence scores. All results (the rule objects and associated metadata) are stored in a versioned repository or database for traceability and later consumption by development agents. The entire pipeline is designed to be **platform-agnostic** – running in a containerized or serverless environment – and prioritizes quick deployment by utilizing off-the-shelf AI services and frameworks.

**Components Overview:**

- **LLM Extraction Agent:** The “brain” of the system, powered by a large language model, which reads legacy code and extracts candidate business rules along with contextual explanations.
- **Schema Formatter & Validator:** A module (implemented via Pydantic models) that converts the LLM’s output into a well-defined JSON/YAML schema and validates that the content conforms to expected types/fields. This prevents malformed or incomplete rule dat ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=3,safe%2C%20predictable%20results))】.
- **Multi-Agent Orchestrator:** Orchestration logic that coordinates multiple specialized agents (extractor, formatter, validator, etc.). PydanticAI supports such an agent-based architecture out-of-the-box, allowing agents to share state and communicate seamlessl ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=PydanticAI%20introduces%20an%20agent,share%20state%20and%20communicate%20seamlessly))】. This orchestrator manages the flow of data through pipeline stages and handles re-tries or human escalation as needed.
- **Metadata Store:** A repository for storing extracted rules and their metadata. This can be a document database or version-controlled file store containing JSON/YAML rule definitions. It also includes a simple **lineage store** mapping each rule back to its source location (program name, line numbers) and an index for quick lookup. Optionally, a graph database can serve as a knowledge base linking rules to business entities (see *Knowledge Graph Integration* below).

In summary, the architecture balances **rapid development** (by using a powerful LLM and PydanticAI for quick implementation) with **structured output** (via enforced schemas and metadata storage). No BRMS or external rule engine is needed at this stage – rules are simply materialized as data – keeping the design lightweight and easy to integrate into subsequent modernization steps.

## Modular Pipeline Stages

To extract business rules from RPG/COBOL code, the process is broken into clear, modular stages. Each stage in the pipeline transforms the input incrementally, which makes the system easier to debug, extend, and parallelize if needed. Below are the major stages in sequence:

1. **Preprocessing Legacy Code:** The pipeline begins by gathering and normalizing all source artifacts – RPG code, COBOL programs, CL/JCL scripts, database schemas, screen definitions, and even code comments. Preprocessing parses these sources into an intermediate representation. For example, we use language-specific parsers to generate Abstract Syntax Trees (ASTs) for COBOL and RPG where possibl ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=CodeConcise%20is%20inspired%20by%20the,stored%20in%20a%20graph%20database))】. Non-executable elements like comments or DDS (display file) definitions are also parsed to capture any business terminology. This stage may remove or flag non-business logic (e.g. technical utility code or boilerplate) so that subsequent steps can focus on true business rules. Known IBM analysis tools like Application Discovery (ADDI) or IBM Rational’s parsers can assist in identifying code structure and dependencies (e.g. call graphs, copybook expansions). The output of preprocessing is a structured, navigable form of the legacy code (AST or pseudo-code) and a map of where each piece of logic resides (for traceability).

2. **LLM-Based Rule Identification:** Next, a Large Language Model agent scans the preprocessed code to identify business rules embedded in logic. This involves detecting conditional statements, computations, data validations, and other decision points that represent business policies. The agent processes code in manageable chunks (such as one program or module at a time, or even smaller function/procedure level) to stay within context length. For each chunk, it generates a list of candidate rules in natural language form, along with evidence (e.g. which lines or paragraphs led to that rule). We prompt the LLM in a **chain-of-thought** style, asking it to reason step-by-step through the code and explain what each significant code path represents in business terms. This is analogous to “mining the code to discover business rules” as done in traditional analysis tool ([Mainframe modernization with Generative AI | CROZ](https://croz.net/mighty-mainframe-modernization-generative-ai/#:~:text=%E2%87%92%C2%A0Analysis%20of%20all%20components%20of,for%20further%20analysis%20and%20synthesis))】. For example, an IF…ELSE block that calculates a fee waiver might be extracted as a rule “If customer age < 18, then waive the account maintenance fee.” Comments and variable names are leveraged to assign meaningful business terminology to each rule. The output of this stage is a set of raw, unstructured rule descriptions with references to source code.

3. **Rule Classification:** The identified rules are then categorized into predefined types: **decision rules**, **calculation rules**, **validation rules**, or **workflow rules**. Classification can be done by the LLM agent (e.g. by analyzing the linguistic cue or structure of the rule) or by simple heuristics. For instance, rules that enforce an input constraint (“must be X or else error”) are tagged as *validation* rules, those that compute or derive a value (“compute X = Y+Z”) as *calculation* rules, and those selecting among different paths or outcomes as *decision* rules (similar to DMN decision logic). Workflow or process flow rules (e.g. “if transaction type = X, route to Queue A”) are tagged separately. This classification step is important because each rule type will be formalized into a slightly different schema. It also helps prioritize – for example, decision and validation rules may be of higher interest initially. The classification is stored as a field in the rule metadata. (If the LLM is used, the prompt provides examples of each category so it can label each rule accordingly).

4. **Structured Formalization (PydanticAI Modeling):** In this stage, the raw text of each rule is converted into a structured object that follows a predefined Pydantic model (see next section for schema design). We define a Pydantic `BaseModel` for each rule type (DecisionRule, CalculationRule, ValidationRule, WorkflowRule, etc.) with fields capturing the logical structure. The agent (or a dedicated Formatter agent) takes each rule description and populates the schema: e.g., separating the rule into `conditions`, `outcome`, `trigger_event`, etc., and adding metadata like `source_reference`. Because PydanticAI allows us to specify the desired output schema and have the LLM directly generate a JSON that conforms to it, we can combine this with the extraction step. In practice, the LLM could be prompted to output JSON in the required format immediately after describing the rule, but we have a validation step next as a safeguard. By formalizing rules, we ensure machine-consumability – the rules become data that other tools or agents can read. For example, a decision rule might be represented as a row in a decision table or as a JSON object with explicit condition fields.

5. **Validation & Confidence Scoring:** Each structured rule is then validated and assigned a confidence score. **Schema validation** is automatically enforced by Pydantic – if the LLM’s output is missing a field or has a wrong type, it will be flagged and can be corrected either by a retry or by post-processing (for example, if a field meant to be numeric is output as a string, we can attempt a conversion or reprompt the LLM for that field). Beyond schema, we implement a **semantic validation** layer: an *LLM-as-judge* (or “referee” agent) re-checks the rule against the original source code snippet to verify correctness. This agent could prompt the LLM with the code and the extracted rule, asking “Is this rule an accurate description of the code? Answer with a correctness score 0–100.” Using a separate verification step with an LLM is inspired by self-verification techniques where the model validates its own answer ([Self-Verify LLM Responses - Instructor](https://python.useinstructor.com/prompting/self_criticism/self_verification/#:~:text=The%20self,The%20process%20follows%20two%20stages))】. Additionally, we apply heuristics: for instance, if the rule was derived from a very straightforward `IF` statement, confidence is inherently higher than if it was inferred from a complex, multi-paragraph flow. Any rule that the validation deems low-confidence or mismatched is flagged for human review (or queued for an iterative refinement by the agents). The output of this stage is a set of **validated, structured rules** each with a confidence score and any notes about uncertainties.

6. **Output Serialization and Traceability:** Finally, the structured rules are serialized to machine-readable formats (JSON and/or YAML) and saved to the metadata store. Each rule entry includes a unique identifier and version, allowing the set of rules to be version-controlled. We include traceability info such as timestamps and source lineage: e.g. which program and line range the rule came from, and a hash or version tag of the source code. This ensures that as the legacy code evolves (or our understanding improves), new extractions can be diffed against old ones. The rules can be stored as individual JSON files (one per rule or per program) or aggregated by domain. A simple example of a serialized rule might be: `{"id": "RULE-001", "type": "validation", "condition": "Account balance < 0", "action": "reject transaction", "source": "COBOLProgA: line 200-210", "confidence": 0.95, "extracted_at": "2025-04-05T12:00:00Z"}`. In addition to JSON/YAML files, the system can auto-generate documentation (for example, a Markdown or HTML report) that lists all rules, organized by type and module, including the context and any comments. This provides human-friendly documentation side by side with the raw rule data.

Each stage of the pipeline is designed to be **modular** and can be improved or replaced independently. For example, if a more formal static analysis tool is later introduced, it could augment the Preprocessing stage to feed better input to the LLM. Similarly, if fine-tuning the LLM greatly improves accuracy, the Validation stage can be simplified. This modular approach, combined with multi-agent orchestration, aligns with best practices in GenAI-enabled modernizatio ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=Model,and%20description%20of%20business%20rules))】, but avoids deep coupling with any specific vendor tool, thus keeping the solution flexible and fast to adapt.

## LLM Strategy

Choosing the right Large Language Model and prompt strategy is critical for fast, accurate extraction of business rules from legacy code. Our approach prioritizes **fast time-to-market**, so we will begin with state-of-the-art general models via API, then consider domain-specific fine-tuning in parallel.

**Model Selection:** We recommend using a top-tier model known for strong code understanding and reasoning. For example, OpenAI’s GPT-4 or Anthropic’s Claude v3 (Opus tier) are ideal initial choices. Claude 3 Opus is specifically noted for handling complex, multi-step tasks and strong coding abilitie ([Meet Claude - Anthropic](https://www.anthropic.com/claude#:~:text=Opus.%20Our%20highest,order%20math%20and%20coding%20tasks))】, which suggests it can navigate legacy COBOL/RPG logic effectively. These models have the advantage of being immediately available (via cloud API or platforms like Amazon Bedrock) and highly capable, which accelerates initial development. If data privacy or on-prem deployment is a concern, an open-source code model like **Code Llama** or **StarCoder** could be fine-tuned and used behind the firewall, albeit with likely lower baseline performance. Another promising avenue is specialized models: for instance, researchers have introduced *XMainframe*, a code LLM tuned with mainframe COBOL knowledge that outperforms general code models on COBOL task ([XMainframe: A Large Language Model for Mainframe Modernization](https://arxiv.org/html/2408.04660v1#:~:text=Contributions%3A%20As%20we%20believe%20that,data%20related%20to%20mainframes%20and))】. If such a model (or IBM’s **Granite** model for Z) is available, it could be evaluated for this project. In summary, start with the most powerful model accessible to get good results quickly, then iterate.

**Prompting Approach:** We will employ an iterative prompting strategy combining **few-shot examples** and **chain-of-thought reasoning**. The prompt to the LLM will include: (a) a brief system role definition (e.g. *“You are a code analysis assistant specialized in IBM i (AS/400) business logic extraction.”*), (b) one or two examples of code (perhaps simplified) with the expected JSON rule output (this helps the model align to the output schema), and (c) the target code snippet or program text to analyze. We will encourage step-by-step analysis by instructing the model to first explain the code logic in its own words (this can be done in a hidden “scratchpad”), then output only the structured rules JSON. This technique helps the model not to skip reasoning steps. In practice, PydanticAI allows us to define a `result_type` (the Pydantic model) and will validate the LLM output against it, so our prompt will focus on guiding content while the framework handles structur ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=3,safe%2C%20predictable%20results))】. We also incorporate *structured prompt templates*: e.g., “List all business rules in the following code. Use the format: [JSON list of Rule objects]. Only include rules, no other text.” to reinforce the desired output shape. 

For RPG code, which can be more cryptic (especially older fixed-format RPG), we will include guidance in the prompt about RPG specifics or even use a preprocessor to translate RPG to pseudo-code. Similarly, COBOL copybooks for data definitions will be provided to the model alongside the code, so it understands the fields (we can prepend a section like “DATA DICTIONARY: Customer-Record has fields...” if needed). This contextual info can prevent misinterpretation of field meanings.

**Chain-of-Thought & Verification:** We plan to use a chain-of-thought approach not only for extraction but also for self-verification. For example, after the model produces a candidate rule, we may ask it (in a follow-up prompt) to double-check if that rule is fully supported by the code, as part of the same call if possible. Some LLMs (like GPT-4) can follow complex multi-turn instructions well, which we can utilize to reduce errors. The prompting may involve asking the model to think of counterexamples or ensure the inverse conditions don’t produce a rule it missed.

**Fine-Tuning Plan:** While initial prototypes will rely on zero-shot or few-shot prompting of a powerful model, we recognize that fine-tuning an LLM on domain-specific code can improve accuracy and reduce needed prompt complexity. We propose a parallel effort to curate a fine-tuning dataset:
- Collect sample **RPG and COBOL code** with known business rules. This could include open-source COBOL programs (e.g., from government or academic repositories), or sample code provided by IBM in manuals. We can augment this with synthetic code: for instance, write small COBOL/RPG programs that implement a few clear business rules (loan eligibility, insurance premium calc, etc.), then use those as training pairs (code → labeled rules).
- Include any available **proprietary code** from the client (if allowed) in the fine-tuning data. If direct code can’t be used for IP reasons, extracting an abstract representation (like replace business terms with generics but keep structure) is an option.
- Leverage existing model fine-tuning efforts: IBM’s approach with Granite included fine-tuning on thousands of pairs of COBOL and Java program ([Application modernization with IBM generative AI - IBM Research](https://research.ibm.com/blog/cobol-java-ibm-z#:~:text=IBM%20trained%20the%20base%20model,generated%20synthetic%20code))】 – while that was aimed at translation, those datasets could implicitly help rule extraction understanding. Open Mainframe Project’s datasets or the MainframeBench from XMainframe work could also be sources of COBOL knowledge.

A fine-tuned model (even if it’s a smaller 7-13B parameter open model) could then be used in the pipeline to reduce reliance on an external API and run on-prem if needed. The fine-tuning objective would be to have the model directly output structured rules given code, effectively teaching it the “pattern” of business rule extraction. This is ambitious, but even partial fine-tuning (e.g. on understanding RPG syntax) could be beneficial. We will evaluate ROI of fine-tuning after the initial agent is working – if the base model already performs well (which is possible with GPT-4-level models), fine-tuning can be a second-phase optimization rather than a blocker for MVP. 

**Summary:** Use the best available LLM to jump-start development, craft prompts with examples and step-by-step reasoning to maximize correctness, and plan for fine-tuning with COBOL/RPG-specific data to further improve accuracy and self-sufficiency. By combining these strategies, we can achieve high quality rule extraction quickly while laying the groundwork for continuous improvement.

## PydanticAI Schema Design

A core part of this solution is expressing extracted business rules in a **machine-consumable logical format**. We define a set of Pydantic data models (schemas) to formalize different types of business rules. This provides a clear contract for what constitutes a “rule object,” enabling downstream software agents to consume the rules easily. It also ensures consistency and validation of the LLM outputs (since Pydantic will enforce types and required fields).

**Rule Type Schemas:** For each category of rule identified, we create a schema class. The primary ones are:

- **DecisionRule:** Represents a business decision logic, typically based on one or more conditions leading to a choice among outcomes. Fields might include:
  - `id` (unique rule identifier, string),
  - `description` (human-readable summary),
  - `conditions` (list of boolean conditions that must be satisfied, each condition could be a structured object with operands, operator, and an optional field reference),
  - `outcomes` (list of possible outcomes or a mapping of condition-combinations to outcome; for a simple IF/ELSE, there might be two outcomes: one when conditions true, one for else),
  - `outcome_type` (if outcomes are e.g. assignments vs workflow branches, etc., but for pure decision maybe not needed),
  - `source_reference` (metadata linking to code: program name, section, line numbers),
  - `lineage` (if this decision rule is composed from multiple code pieces or calls, list them),
  - plus common metadata fields like `confidence`, `extraction_timestamp`, `system_version`.

- **CalculationRule:** Represents a formula or derivation. Fields:
  - `id`, `description`,
  - `formula` (perhaps a structured representation of the calculation, e.g. an expression tree or a simplified algebraic string "TARGET = F1 * 0.05 if F2 = 'Y' else F1 * 0.1"),
  - `target_field` (the business field being calculated, if identifiable),
  - `source_fields` (list of input fields used in the calculation),
  - `conditions` (if the calculation only applies under certain conditions),
  - `source_reference`, `confidence`, etc.

- **ValidationRule:** Represents a check or constraint on data.
  - `id`, `description`,
  - `condition` (the condition that triggers the validation failure, e.g. "AccountBalance < 0"),
  - `action` (what happens when condition is met – often this is an error or warning; we can formalize it as `level: error/warning` and maybe an `message` or `code`),
  - `field` (optional, if this is a single-field validation rule like "FIELD X must satisfy condition Y"),
  - `source_reference`, `confidence`, etc.
  - Possibly a `severity` or `priority` field for the rule.

- **WorkflowRule:** Represents a routing or process flow decision.
  - `id`, `description`,
  - `trigger` (event or state that triggers this rule, e.g. "Order status changed to Shipped"),
  - `condition` (if any, e.g. "if customer is VIP"),
  - `action` or `next_step` (what workflow action to take, e.g. "send approval request", "route to queue ABC"),
  - `source_reference`, etc.

All rule types inherit from a base class `BusinessRule` that has common metadata fields: `id`, `type` (enum of rule type), `source_reference` (which can include the source file/module and possibly a pointer into that source like line numbers or a byte offset if from compiled code), `lineage` (if the rule was composed from multiple code elements or if it relates to other rules), `extraction_timestamp`, `system_version` (an identifier for the codebase version or extraction run). **Lineage** is important in cases where a business rule is not explicitly in one place in code – e.g., part of the rule is in a main program and part in a subroutine. In such cases, the rule object might list both source locations. We also include a `confidence` score (float 0 to 1) and possibly a `reviewed` flag to indicate if a human SME has confirmed the rule.

**Metadata Fields Rationale:**  
- *Source trace/lineage:* This links back to the legacy code, enabling verification and future maintainers to locate where the rule came from. For example, `source_reference: {"program": "CUST0001RPG", "lines": "230-265"}`. If multiple sources, it could be a list.  
- *Confidence:* Allows prioritizing which rules might need human attention and provides a measure of extraction quality.  
- *Extraction timestamp:* Helpful for auditing and ensuring we know when the rule was pulled (e.g., after a code change, the timestamp would update).  
- *System version:* If the legacy system has versions or if we tag each extraction with a commit ID or build number, storing it here helps track rule evolution (e.g., rules might be slightly different between version 1.0 and 1.1 of the application).

**Example Outputs:** Below are simplified examples of how a rule might be represented in **YAML** and **JSON**, conforming to the schema definitions:

- *Example 1: A Validation Rule (YAML)*  
```yaml
id: RULE-V001  
type: validation  
description: "Customer age must be 18 or above to create an account."  
condition: "customer_age < 18"  
action:  
  level: error  
  message: "Applicant must be at least 18 years old."  
source_reference:  
  program: "OPENACCT"  
  lines: "120-130"  
lineage: ["OPENACCT:120-130"]  
confidence: 0.98  
extraction_timestamp: "2025-04-05T12:20:30Z"  
system_version: "AS400_PROD_2025_04"  
```

- *Example 2: A Decision Rule (JSON)*  
```json
{
  "id": "RULE-D005",
  "type": "decision",
  "description": "Determine interest rate tier based on credit score.",
  "conditions": [
    { "field": "credit_score", "operator": ">=", "value": 750, "label": "HighScore" },
    { "field": "credit_score", "operator": "<", "value": 750, "label": "LowScore" }
  ],
  "outcomes": [
    { "when": "HighScore", "rate": 0.05 },
    { "when": "LowScore", "rate": 0.08 }
  ],
  "outcome_type": "interest_rate",
  "source_reference": {
    "program": "LOANPROC",
    "paragraph": "CALC_RATE",
    "lines": "200-250"
  },
  "confidence": 0.90,
  "extraction_timestamp": "2025-04-05T12:21:00Z",
  "system_version": "AS400_PROD_2025_04"
}
```

In the decision rule example, the rule essentially says: if credit_score ≥ 750, interest_rate = 5%, else interest_rate = 8%. We represented it with two condition entries and outcomes keyed by a label. (This format is analogous to a DMN decision table with two rules; in a DMN-like representation it could also be one rule with two columns, but JSON here is one way to do it.)

We will refine the schema as we iterate, but the key is that **every field is explicitly defined and validated**. Pydantic’s validation will reject any LLM output that doesn’t match (for example, if the LLM omits `id` or uses a wrong data type somewhere), enforcing reliability in the outputs ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=3,safe%2C%20predictable%20results)). This structured approach also means these JSON/YAML rule files could later be converted to a standard like **DMN (Decision Model and Notation)** if integration with a rules engine becomes desirable. For now, they serve as a contract for software agents: e.g., a future “code generation agent” can read the JSON rule and know exactly the logic to implement in a new system.

**DMN-like Decision Tables:** It’s worth noting that for decision rules with multiple permutations, we could output them in a decision table format. In JSON, this could be an array of rules with conditions and outcomes. Our schema design can accommodate that (as shown in the outcomes array above). If needed, we can produce an actual DMN XML or use YAML to tabulate it. However, since DMN is an XML standard, we’ll stick to JSON/YAML for simplicity and transform to DMN later if needed. The focus is on logical clarity and completeness of rule representation.

**Documentation and Schema Evolution:** Each schema class will include a docstring or metadata that describes the rule in plain English. This is useful for auto-generating documentation. For example, the `ValidationRule` model can have a description like “Represents a business validation that must hold true; if condition evaluates to true, a violation occurs triggering an action.” We will maintain these definitions in a **data dictionary** for the rules themselves. This helps new developers or SMEs understand what each field means. As new types of rules are discovered (perhaps we find a kind of rule not anticipated, like a **timing rule**: e.g., “process X must happen within N days of event Y”), we can extend the schema definitions accordingly. The use of Pydantic makes this relatively easy – we add a new model or new fields and regenerate the data structures. Versioning of schema will be managed (e.g., the `system_version` in metadata can also double as a schema version if needed, or we maintain a separate version number for the schema itself in the output files).

Overall, the PydanticAI schema design ensures that from unstructured legacy code we move to **highly structured, typed, and annotated rule objects**. This not only serves the immediate modernization goals but creates a knowledge asset for the organization – a repository of business rules that can be queried, tested, and transformed independent of the old source code.

## Multi-Agent Workflow Design

To efficiently coordinate the complex task of extracting and validating rules, we adopt a **multi-agent workflow**. Each agent in the system has a specialized role, and together they operate as a team to perform extraction with checks and balances. By dividing responsibilities, we can parallelize certain tasks and apply different prompting strategies tuned to each role. PydanticAI’s architecture inherently supports multi-agent collaboration with shared state ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=PydanticAI%20introduces%20an%20agent,share%20state%20and%20communicate%20seamlessly)), but the design can also be implemented with other frameworks (LangChain, CrewAI, etc.) if needed.

**Agent Roles and Responsibilities:**

- **Code Extractor Agent:** This agent’s role is to read through a given source (e.g., one program or a module chunk) and output raw business rule candidates. It focuses on comprehension and recall. Internally, it might use the LLM with prompts that encourage detailed reasoning about the code logic. It produces a preliminary list of rules (in either narrative or draft JSON form) along with any explanatory notes. Essentially, this is the “primary” agent doing the heavy lifting of understanding code.

- **Rule Formatter Agent:** The formatter takes the extractor’s output and ensures it conforms to the required schema. If the extractor already attempted JSON output, the formatter validates it using the Pydantic models. If the extractor gave a narrative or partially structured output, the formatter agent reorganizes it into the final JSON/YAML structure. This agent might call the LLM with a prompt like: *“Given the following rule description, fill in this JSON template…”* or directly use Pydantic parsing. It handles adding any missing metadata (timestamps, IDs). This separation is useful to keep extraction prompts simpler (the extractor can focus on finding rules, not JSON syntax) and then let the formatter enforce structure.

- **Validation/Verifier Agent:** The validator acts as a quality gate. It reviews each structured rule against the source code context. This agent might use a different prompting approach, for example: *“You are a code reviewer. Verify if the rule accurately represents the code snippet.”* It could even use a different model or tool (for example, a simple interpreter or test harness, if we generate one, to simulate the code). Primarily, though, an LLM agent here can do a second-pass explanation and point out mismatches. It then annotates the rule with a confidence score or flags it for issues. In some implementations, we could have two validator agents for redundancy (maybe run the rule and code through two different LLMs and compare judgments).

- **Memory Manager/Context Agent:** This agent is responsible for maintaining the shared context and knowledge base that all other agents use. For instance, it can store the knowledge graph or glossary (see next section) and provide definitions to the extractor agent on demand (like “Agent asks: what is field CUST_STATUS? Memory agent looks it up in the glossary and returns: CUST_STATUS = Customer account status code.”). It also keeps track of which parts of the code have been processed, preventing duplicate work. If the extraction is done module by module, the memory agent can note progress and also handle inter-module links (e.g., if a rule spans a call, it can remind the extractor agent when it reaches the called program). Technically, this could be implemented by using a shared database or context object that all agents can query via tool interfaces.

- **Orchestrator:** While not a separate persona agent, the orchestrator logic oversees the sequence of steps. It could be a simple loop in code or a higher-level agent that delegates tasks. For example, for each program: orchestrator invokes Extractor → passes result to Formatter → passes to Validator → either accepts and stores the result or triggers a refinement. The orchestrator also manages **human-in-the-loop checkpoints**: if the Validator flags a rule, the orchestrator might pause and request human review for that item. 

**Interaction Model:** The agents communicate through well-defined channels – typically by passing the result objects along the pipeline. Using PydanticAI, we can have agents produce outputs of certain model types which become the input for the next agent, ensuring a smooth handoff. The **Extractor** produces `DraftRule` objects, the **Formatter** expects those and produces `BusinessRule` objects, the **Validator** appends a verification report or adjusts the `confidence`. The **Memory agent** can be invoked by others at any time (for instance, the Extractor might use a tool call to query the glossary: PydanticAI and other frameworks allow agents to call functions/tools mid-prompt ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=4,making)) – we can register a function like `lookup_term(term) -> definition` that the Extractor agent can call when needed).

**Parallelism and Iteration:** Some tasks can be parallel. If multiple programs or modules can be processed independently, we could spin up multiple Extractor agents in parallel (subject to available compute/LLM throughput). The orchestrator ensures each section of code is assigned to an extractor agent. The Memory agent helps ensure they don’t conflict (like if two agents want to update the knowledge graph concurrently, the memory agent serializes those updates). After initial extraction, there might be an iteration: if the Validator finds an issue, we could either:
- Have the Extractor agent run again on that section with a refined prompt (perhaps giving it the feedback from the Validator).
- Or have a specialized **Refinement agent** that attempts to fix the rule (this could be the Formatter agent re-running with more context).

The multi-agent setup enables a kind of feedback loop: Extract → Validate → Refine (if needed) → finalize. This loop continues until all code is processed and all rules are either validated or marked for human review.

**State Persistence:** All intermediate and final results are persisted (possibly by the Memory manager). We maintain a state, for example, in a simple JSON file or in memory that lists what programs have been processed, how many rules extracted, etc. In case of a system crash or restart, this state allows the process to resume from where it left off, which is important for large codebases. We could also log the conversations (prompts and responses) of each agent for auditability using PydanticAI’s logging or an external logger (which can be helpful if we need to trace why an agent made a certain decision).

**Human-in-the-Loop Checkpoints:** Despite automation, involving human experts (SMEs) at critical points greatly increases trust and accuracy ([Automated Framework to Extract Software Requirements from Source Code](https://dl.acm.org/doi/pdf/10.1145/3639233.3639242#:~:text=legacy%20software,This%20allows%20the%20development%20and)). The workflow will include optional pauses where a human can review. For example:
- After extraction of a module, a BA or SME can quickly scan the extracted rules (perhaps via a dashboard interface) to confirm if they “make sense” before they are validated and finalized.
- After validation, any rule below a confidence threshold (say <0.7) is queued for SME review. The SME can then either approve it as-is, edit the rule (correct the logic or wording), or mark it as a false extraction to be discarded.
- The human feedback is then fed back. In PydanticAI, this could even be interactive: an agent could be a “HumanAgent” where the orchestrator asks for input. Or simpler, the system could output a file of “pending review” rules for a person to edit and then read those edits back in.

By structuring the workflow with agents, we isolate concerns: each agent can be developed and optimized independently. For instance, if we find the extraction agent misses certain patterns, we tweak its prompt or logic without affecting the validator. This design also mirrors real-world team collaboration (one person finds information, another checks it) which tends to improve outcomes ([PydanticAI: Simplifying LLM-Driven Development with Agent Frameworks](https://www.linkedin.com/pulse/pydanticai-simplifying-llm-driven-development-agent-frameworks-fa2gf#:~:text=driven%20responses)). CrewAI or similar frameworks could be used here as well – e.g., CrewAI allows definition of multiple agent personas and their conversation. In our case, agents communicate mostly through data passing rather than long dialogues, so a simpler orchestrator might suffice. 

**Logging and Monitoring:** The multi-agent system will include logging of actions for transparency. Each agent can log what it did (e.g., “Extractor: found 5 rules in program X”, “Validator: rule Y confidence downgraded due to missing condition”). This provides a trace of the pipeline which is useful for debugging and for users to understand what the AI did (which is often a requirement in sensitive modernization projects).

In summary, the multi-agent workflow is designed for **collaboration and oversight**. By splitting roles, we achieve a checks-and-balances system: the Extractor’s work is checked by the Validator, and formatting issues are handled explicitly by the Formatter, etc. This reduces the chance of a single-point failure (e.g., one monolithic prompt hallucinating without check) and aligns with how complex tasks are handled in advanced LLM applications (breaking tasks into smaller, more controllable steps). The end result is a more reliable and maintainable pipeline, aligning human and AI strengths.

## Validation and Verification Layer

Ensuring the extracted business rules are correct and complete is just as important as extracting them. The Validation & Verification layer introduces quality control through automated checks, LLM-based judgment, and opportunities for human review. The goal is to have a high degree of confidence in the rule set before it’s handed off to developers or other systems.

**LLM-as-Judge (Self-Verification):** As mentioned, we employ a strategy where the LLM itself verifies or scores the extracted rules. This is inspired by recent research into using LLMs to critique or validate their own outputs ([Self-Verify LLM Responses - Instructor](https://python.useinstructor.com/prompting/self_criticism/self_verification/#:~:text=The%20self,The%20process%20follows%20two%20stages)). In practice, after a rule is extracted and formatted, we feed the LLM a prompt like: *“Here is a segment of code and a proposed rule extracted from it. Is this rule an accurate and complete description of what the code does? Answer with reasoning and a correctness score.”* The LLM then performs a backward verification: it takes the rule (declarative form) and essentially “replays” it against the code. For example, if the rule says "X must be > 0", the LLM looks at code to see if indeed there’s a check `IF X <= 0 THEN ...error`. This process acts as an independent check ([Self-Verify LLM Responses - Instructor](https://python.useinstructor.com/prompting/self_criticism/self_verification/#:~:text=The%20self,The%20process%20follows%20two%20stages)). We might generate multiple variants of the rule and ask the LLM which is correct (a form of comparison). If the LLM judge finds a discrepancy (say the rule missed a condition or mis-stated a threshold), it will either correct it or flag it.

**Heuristic Checks:** Alongside the LLM judge, we implement a series of deterministic checks:
- **Consistency Check:** If a rule was extracted multiple times from different places (e.g., the same rule logic appears in two programs), ensure they are consistent. If not, flag them (it could indicate a subtle difference or an extraction error).
- **Contradiction Check:** Using the structured format, check if two rules from the same domain contradict each other (this is rudimentary conflict detection – e.g., one rule says “if status = A then do X” and another says “if status = A then do Y”, possibly a conflict unless X and Y are complementary or ordered).
- **Completeness Check:** For decision tables, ensure coverage. For example, if a decision rule has conditions that split a scenario, do we have outcomes for all relevant cases? (DMN engines often have validations for missing rules or overlapping rules – we can mimic some of that logic).
- **Data Type/Schema Cross-check:** If we have access to schema (DB2 field definitions), verify that the rule conditions make sense with field types. E.g., if a field is numeric, a rule checking for “= 'ABC'” is probably incorrect or indicates a parsing issue.
- **Range/Boundary Analysis:** For numeric conditions, check boundaries. If code says “> 100” we might also see if there's an implicit “<= 100” handling elsewhere. This is more analysis than pure check, but it can catch if maybe an else branch wasn’t captured as a rule.

We can automate some of these by simple Python logic on the JSON rule outputs.

**Confidence Scoring:** We aggregate the results of LLM judge and heuristics into a unified confidence score for each rule. For instance, start with the self-reported confidence from extraction (the LLM might implicitly give one or we assign high confidence if straightforward), then deduct points for any issues found (or boost if cross-verified by multiple methods). The scoring mechanism can be tuned with weightings. The end result is each rule gets a score 0.0 to 1.0. We consider anything above, say, 0.8 as high confidence (green), 0.5-0.8 medium (yellow, needs maybe a quick look), below 0.5 low (red, needs review).

**Review Dashboard:** All validated rules will be presented in a dashboard/report that highlights any low-confidence items. The dashboard can group rules by module or business domain, show the confidence color, and allow a human reviewer to click on a rule to see details: the rule itself, the source code snippet that led to it, and the LLM’s rationale (we can include the chain-of-thought from the verification step here, effectively showing “why the AI thought this was correct or not”). This kind of interface significantly aids an SME in reviewing AI outputs ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=chatbot%20featured%20in%20CodeConcise%2C%20which,face%20when%20working%20with%20them)) because it’s focused and context-rich.

**Automated Test Cases:** Where feasible, we will generate simple test cases to verify rules. This can be done by leveraging the LLM to create input data that triggers certain rules. For example, if a rule is “if salary > $100k then require VP approval”, we can ask the LLM to produce two fictional records: one with salary 120k (expect rule triggers) and one with 80k (expect rule does not trigger). If we had a way to execute the legacy code (or a partial simulation of it) with those inputs, we could confirm the behavior. However, executing legacy COBOL/RPG isn’t trivial in an automated test harness without the full environment. Instead, we might create a **pseudo-code simulator**: since we have AST or structured logic, we could evaluate the condition expressions ourselves on the test data. This is more of a stretch goal, but it could increase confidence for critical rules. At minimum, documenting such example scenarios in the output helps humans validate the logic.

**Formal Verification (Optional):** For some rules, especially calculation formulas or critical business invariants, we could use formal methods. One idea is to translate rules into a constraint solver format (like Z3 or another SMT solver) to check for certain properties (e.g., ensure that two rules don't produce conflicting outcomes for any possible input). Another idea is to integrate with a **business rules validator** in the style of DMN TCK (Technology Compatibility Kit) which checks if the rules cover all combinations, etc. These are advanced steps and might be beyond MVP, but the structured output positions us well to apply such tools later. For example, a set of ValidationRules could be turned into logical assertions and we could see if any assertion is unreachable or always true, indicating a redundant rule.

**Feedback Loop:** When the verification finds an issue, the system doesn’t just flag it – it also feeds it back into improvement:
- If a rule is low confidence because, say, the LLM was unsure, we might re-prompt the Extractor with a more specific query, or break the code into smaller pieces and try again.
- If a human corrects a rule, we capture that correction and potentially use it to fine-tune the model or at least as an additional few-shot example for similar cases. Over time, this makes the system smarter (learning from mistakes).
- We maintain a log of common errors (e.g., “LLM often misses rule when code uses GO TO to jump over a section”) and develop mitigations for those patterns in preprocessing or prompting.

**Benchmarking Accuracy:** As part of verification, we will measure the system’s precision/recall on known rules if any are available. For instance, if the client can provide a handful of known business rules (from documentation or SME knowledge), we check if each is present in the extracted set (recall) and if any extracted ones are not true (false positives). This helps quantify improvements as we tweak the pipeline.

Using an LLM as a judge is a novel but increasingly accepted practice to improve reliability of LLM outputs ([Self-Verify LLM Responses - Instructor](https://python.useinstructor.com/prompting/self_criticism/self_verification/#:~:text=The%20self,The%20process%20follows%20two%20stages)). Combined with traditional QA methods, this hybrid approach should significantly reduce hallucinations and errors. The end result of the Validation layer is that stakeholders can trust the extracted rule set as a correct reflection of the legacy code’s business logic. Furthermore, the verification artifacts (scores, rationale, test cases) serve as documentation evidence supporting the modernization effort – useful in audits or in convincing business users that the rules are accurate.

## Knowledge Graph and Glossary Integration

To bridge the gap between low-level code and high-level business understanding, we integrate a **Knowledge Graph (KG)** and business glossary into the rule extraction process. The aim is to link each extracted rule and its components (fields, values, conditions) to well-defined business concepts, ensuring consistency and aiding disambiguation. This also creates an enriched knowledge base that can be queried independently (for example, to ask “show me all rules related to customer eligibility”).

**Ontology and Glossary:** We will either use an existing enterprise ontology (if the organization has one) or build a simple glossary of key business terms, data entities, and their relationships. For instance, define entities like *Customer*, *Account*, *Order*, and link them to fields in code (e.g., `CUST-ID` in COBOL corresponds to the *Customer* entity’s identifier). We also include business domain definitions (e.g., what does “VIP status” mean). This glossary can start small (maybe extracted from schema names and known business terms) and grow over time as the LLM encounters new terms.

**Linking during Extraction:** When the Extractor agent or Formatter is processing a rule, it will attempt to annotate it with links to the glossary. For example, if a rule involves field `POLICY_TYPE`, the system will recognize this likely relates to the concept *Policy* (if such a concept is in the glossary). The rule object could then include a reference like `concepts: ["Policy"]`. We can accomplish this by having the Memory/KG agent provide suggestions. PydanticAI (or a custom tool) can be used to call a glossary lookup: for each identifier or literal in the rule, search the glossary for matches or synonyms. If found, attach the concept. If not found, that might indicate a new business term – we can optionally prompt the LLM to explain it or flag it for SME clarification. (For example, code might have abbreviations; LLM could guess or ask, and an SME might confirm "Oh, 'LTV' stands for Loan-to-Value ratio.").

**Maintaining Ontological Consistency:** By linking to a KG, we ensure that all rules use consistent terminology. If one program calls something “CustNbr” and another “CustomerID”, the glossary would map both to the concept *Customer Identifier*. We will prefer using business-friendly names in the rule descriptions where possible (the LLM can be guided to use glossary terms: e.g., if it sees `CSTMR_CAT = 'P'`, use “Preferred Customer” if the glossary says 'P' means Preferred). This prevents proliferation of inconsistent language in the extracted rules and will make the next modernization steps (like writing new code or documentation) much smoother.

**Knowledge Graph Implementation:** We can implement the KG using a graph database like Neo4j, as this allows storing nodes for **Entities**, **Fields**, and **Rules**, and relationships like *Rule uses Field*, *Field belongs to Entity*, *Entity is related to Entity*. Thoughtworks’ CodeConcise approach reports success in storing AST and rule relationships in a graph for querying ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=CodeConcise%20is%20inspired%20by%20the,stored%20in%20a%20graph%20database)) ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=Neo4j%2C%20our%20graph%20database%20of,leveraging%20the%20graph%E2%80%99s%20structure%2C%20allowing)). For example, we will store a node for each extracted rule (with its type and description), and link it to nodes representing the data elements it involves. We also add nodes for business concepts. If a rule’s condition is "account_balance < 0", we link the condition to the *Account* concept (because account_balance is an attribute of Account) and possibly to a *BusinessPolicy* concept like *OverdraftPolicy*. Later, an analyst could query the graph: “find all rules related to OverdraftPolicy” and get them. 

**LLM and KG Synergy:** The LLM can both contribute to and benefit from the knowledge graph:
- *Contribute:* As mentioned, the LLM can populate relationships by interpreting code. It might infer, for instance, that a constant “EMP_TYPE = 'X'” corresponds to *Contractor* vs *Employee* based on context, adding that info to the glossary node for EMP_TYPE. We will be cautious and usually confirm such inferences with an SME, but the LLM can draft it.
- *Benefit:* In turn, when the LLM is uncertain about a term, we can let it query the KG. This can be done by a tool invocation. E.g., if code uses field `MEMB_STAT_CD`, the LLM can call a tool `lookup("MEMB_STAT_CD")` which searches the KG. If the KG knows that `MEMB_STAT_CD` is “Member Status Code with possible values A=Active, I=Inactive, S=Suspended”, that info is returned to the LLM, which then can correctly interpret rules involving that field. This is essentially Retrieval-Augmented Generation (RAG), where the KG (plus possibly a vector store) is the retrieval mechanism ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=Neo4j%2C%20our%20graph%20database%20of,leveraging%20the%20graph%E2%80%99s%20structure%2C%20allowing)). 

To facilitate this, we will index the glossary in a vector database as well, so semantic search can be done on terms. Neo4j now supports vector search in its graph DB ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=Neo4j%2C%20our%20graph%20database%20of,leveraging%20the%20graph%E2%80%99s%20structure%2C%20allowing)), which we can leverage, or use an external vector store linked to the glossary text.

**Glossary Updates:** When a human SME clarifies a term or when we extract a new one from code comments (e.g., a comment says “/* X is the customer segment code */”), we update the central glossary. Over time, the knowledge graph becomes richer – not only do we have rules, but also context like *“Rule R1 deals with Customer of type Gold”*. Maintaining this is an investment that pays off in later phases, especially if the organization plans to build a knowledge portal or use this for training new developers.

**Ontological Consistency Example:** Suppose the legacy system uses a code “T” to mean “Trial account”. In one place, a COBOL paragraph might enforce “IF ACCT_CODE = 'T' THEN set TrialPeriod=90 days”. The rule extracted would say something about trial accounts. The glossary would have an entry `AccountType` with value 'T' = “Trial Account”. Our KG linking ensures the rule’s condition is tied to AccountType:Trial. If elsewhere another program checks `if account_type == 'TRIAL'` (maybe a newer module using full words), the KG lets us know that's actually the same concept. Thus we avoid listing essentially the same rule twice under different guises. We also detect if something might be misinterpreted – if the LLM guessed wrong on a term, consistency checks with KG might reveal it.

**Querying the Knowledge Graph:** With rules in a KG, interesting queries become possible:
- “List all rules that use concept X” (e.g., all rules related to *Interest Rate*).
- “What business terms does program Y touch?” (traverse from program node to its rules to concepts).
- “Are there any rules that involve both concept A and B?” (maybe find hidden interactions).
- These queries can be done via Cypher (if using Neo4j) or via an API. We might not implement a full query interface in the MVP, but designing the data model now enables these later.

**SME Q&A via KG:** A stretch goal is to allow a chatbot interface where users can ask questions about the legacy business logic in plain English and get answers derived from the extracted rules and knowledge graph (somewhat like the CodeConcise chatbot idea ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=chatbot%20featured%20in%20CodeConcise%2C%20which,face%20when%20working%20with%20them))). For example, “Under what conditions is an account automatically closed?” The system would search rules for any outcome about closing account, then use the conditions to respond. With the context stored, an LLM could answer by citing the rule. Even if we don’t implement this fully, structuring the knowledge in this way makes such capabilities possible. It also offloads some reliance on the few SMEs who understand the old system, as the KG+LLM combo can answer certain queries similarly to an SME ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=chatbot%20featured%20in%20CodeConcise%2C%20which,face%20when%20working%20with%20them)).

**Maintaining the KG:** We will implement routines to update the KG every time we run the extraction. If the code base updates or if we refine rules, we sync those changes to the graph. The KG becomes a living documentation of the system. We will also include the glossary entries and rule definitions in the final outputs (documentation) to ensure even without the graph DB, the knowledge is captured. For instance, embed the glossary as a section in the rule documentation.

In essence, the Knowledge Graph and glossary integration serve as the **semantic glue** between code and business language. It reduces misinterpretation (by providing definitions), promotes reuse of terms (everyone uses the same phrase for the same concept), and sets the stage for more advanced analysis and querying of the extracted rules. It’s an extra upfront effort but aligns with best practices seen in industry where LLM outputs are combined with knowledge bases for reliability ([Legacy Modernization meets GenAI](https://martinfowler.com/articles/legacy-modernization-gen-ai.html#:~:text=Neo4j%2C%20our%20graph%20database%20of,leveraging%20the%20graph%E2%80%99s%20structure%2C%20allowing)). By doing this, we ensure the modernized rules aren’t just technically correct, but also make sense to business stakeholders in their own language.

## Required AS/400 Inputs

Although the user currently does not have the AS/400 artifacts on hand, it’s crucial to plan for **which inputs** from the legacy system we will need to successfully perform rule extraction. For an IBM AS/400 (IBM i) environment, the following sources should be gathered:

- **Source Code for All Business Programs:** This includes all RPG code (whether RPG III, RPG/400 or modern RPG IV / ILE RPG) and COBOL programs on the system. Typically, these reside in source physical files (e.g., QRPGSRC, QCBLSRC libraries). We will need the latest version of every program that contains business logic. If there are other languages in use (e.g., CL – Control Language scripts, or even Java on IBM i), those should be included as well because they might implement business rules or orchestrate processes.

- **Copybooks and Include Files:** In COBOL, any COPY statements referencing copybooks (or in RPG, /COPY or /INCLUDE source members) should be collected. These often contain data structure definitions or common logic that is essential for understanding the rules. Without copybooks, the code might not compile or make sense (missing context for fields). For RPG, this may include copybooks for data structures or calculation subroutines. Ensuring we have all includes allows the preprocessing stage to expand them so the LLM sees the full picture.

- **Database Schema Definitions:** The AS/400 likely uses DB2 for i (previously known as DB2/400) as the database. We need the definitions of physical and logical files (tables and indexes) – essentially the data schema. On IBM i, these could be in the form of DDS (Data Description Specifications) for physical files, or SQL DDL if the database was defined with SQL. Having the field names, types, and sometimes even field text descriptions is extremely useful. For example, knowing that field `CSTATR` is a 1-character field with text "Customer Status (A=Active, I=Inactive)" would let the LLM and our system recognize values and meaning. Schema definitions also tell us relationships (foreign keys if documented, or logical file join relationships), which can hint at rules spanning multiple files.

- **Screen and Report Definitions:** Many AS/400 applications have 5250 screens defined via DDS for display files (DSPF) or UIM files. We should obtain the source for any interactive screen that might contain validation logic or hints. Often, DDS for display files includes keywords for field validation (like `CHECK` or `VALUES` clauses) that enforce business rules at the UI level (e.g., valid values for a field). These are business rules too, and should be extracted. Similarly, report definitions (printer files) could contain conditional formatting that implies rules, though less critical. At least, screen field labels and prompt texts are very useful to decipher the meaning of fields in the code. For instance, if the screen label says "Enter Account Type (P=Personal, B=Business)", that maps the code’s `ACCT_TYPE` field to business meaning.

- **Job Control Scripts:** If the environment uses batch jobs orchestrated by CL programs or job scheduler entries, those can contain business logic sequencing (like “run Program A then B; if B fails, send alert”). We should gather CL program sources (.CLLE etc.). On mainframe we’d ask for JCL; on IBM i, CL is the analogous piece. They might not contain explicit business rules, but they show the flow of processes (which could be considered workflow rules). For modernization, understanding job flows is important. So, any **JCL (if COBOL is from mainframe)** or **CL** (for IBM i jobs) scripts involved in the system should be included.

- **Operational Logs or Decision Tables (if any):** In some systems, especially if they had quasi-rules-engine, there might be externalized tables or config files that store rules (e.g., a file of interest rates by tier). If such data-driven rules exist (like parameter files), those should be identified. The user may not know of these upfront, but during code analysis we might notice, for example, a table that seems to hold rules. We’d then need the data from those tables to fully capture the logic. It’s not typical for pure RPG/COBOL shops to have external rule tables (unless they used a product for it), but worth keeping in mind.

- **Existing Documentation (if available):** While not code, any existing functional specs, requirement docs, or even help text can be useful to verify rules. The user might not have this, but we should ask if any *user manuals* or *comments in code* exist that describe business logic. Comments in the source code are especially important and are already covered by including source, but we highlight that we won’t strip comments; we’ll use them.

In summary, **complete source code and definitions** are needed. If any piece is missing, the extraction might miss rules or produce wrong results. For example, missing a copybook that defines a condition value could cause the LLM to misinterpret a constant. Thus, the effort should include exporting all relevant source from the AS/400. IBM’s Rational Developer for i (RDi) or older tools like PDM can be used to mass-export source members. We might recommend using **IBM RDi** to connect to the IBM i and pull all source members into text files, which can then feed into our pipeline. This ensures proper encoding (EBCDIC to ASCII if needed) is handled.

**Tools for Extracting and Parsing Input:**
- *IBM Rational Developer for i (RDi):* An IDE that can easily browse and extract sources from the AS/400. It can also do some analysis like cross-referencing where fields are used, which could assist preprocessing (e.g., building a call graph or expanding copybooks).
- *IBM Application Discovery (ADDI):* Though primarily for mainframe, IBM may have similar analysis tools for IBM i. ADDI on z/OS provides a repository of code structure; if a similar tool exists for IBM i, it could be leveraged to get an initial map of the application.
- *Micro Focus COBOL Analyzer:* If the COBOL code is extensive, Micro Focus’s analyzer tools (as referenced in their docs ([Extracting Business Rules](https://www.microfocus.com/documentation/cobol-analyzer/ca50/CA/GUID-64A974DE-F7E2-4D71-AEAC-45397C1B1996.html#:~:text=Business%20Rule%20Manager%20,project%20glossary%3B%20and%20much%20more))) can parse COBOL and even extract some rules manually. For IBM i COBOL and RPG, Micro Focus might not cover RPG, but it covers COBOL well. We could use it to identify paragraphs that contain many conditions, etc., as starting points.
- *Fresche Solutions X-Analysis:* This is a modernization tool specifically for IBM i (RPG/COBOL) which builds an application model, including business rules. If available, it could give a quick inventory of rules or at least help parse RPG, which is notoriously tricky. While we are not integrating with a BRMS, using such a tool just to get parse trees or call graphs could accelerate development.
- *Open-Source Parsers:* There are ANTLR grammars for COBOL (for example, the Open Mainframe project or others have defined COBOL parsers). For RPG, open-source options are fewer, but some exist for RPG Free format. We may integrate an ANTLR-based parser in our preprocessing for syntax accuracy. This would ensure we can programmatically navigate the code (e.g., find all IF statements). However, given time constraints, leveraging existing commercial tools might be faster initially.
- *IBM i Data Export Tools:* IBM i allows outputting DDS to DDS source or generating SQL from DDS (`DSPFFD` command displays field descriptions, etc.). We might use system commands to dump schema info. For screens, `DSPFD` and `DSPF` commands can list field constraints.

By assembling all these inputs, we essentially recreate a read-only copy of the application’s source and metadata off the IBM i. This is our playground for analysis. The completeness of these inputs will directly influence the quality of the extracted rules. So, a key early task is to work with the AS/400 administrators to extract:
  - All relevant source members (perhaps thousands of them) into a repository.
  - All data definitions and possibly one-time dumps of any code tables.

We should communicate to the user that obtaining these is a prerequisite for the automated extraction – the AI can’t guess rules that aren’t in the provided source.

Lastly, as the system runs, we might discover we need more context (e.g., sample data to understand typical values). While not strictly required, having some production or test data examples could help the LLM infer certain business logic nuances (for example, if we see typical values of a field, it might clarify what that field represents). If accessible, logs or databases can be used in a read-only manner to supplement understanding, but this is optional.

To recap, **the required inputs are: full source code (RPG, COBOL, CL, etc.), copybooks/includes, database (DB2) schema definitions, screen/display file definitions, and any existing documentation or code-related assets**. Equipped with those, our agent can thoroughly analyze the legacy system. Notably, these inputs align with what traditional analyses require as well – you essentially need everything that went into building the system to successfully reverse-engineer it ([](https://cmfirstgroup.com/wp-content/uploads/2017/09/CM_evolveIT_Capturing_Business_Rules_in_COBOL_Systems.pdf#:~:text=1,business%20expects%20them%20to%20do)) (with the code itself being the ultimate source of truth).

## Output Format and Documentation Standards

The end deliverables of this extraction process are as much about **documentation** as they are about raw data. We will produce outputs in structured formats (JSON/YAML) for machine consumption, and also ensure there are human-friendly documentation artifacts explaining the rules in context. Clarity and traceability are top priorities for these outputs.

**Machine-Readable Rule Output:** The primary output will be a set of JSON and/or YAML files containing the extracted rules, following the schema described earlier. We propose to organize these files logically, for example:
- By application module or domain: e.g., `rules_customer.yaml` might contain all rules related to customer management, `rules_billing.yaml` for billing, etc. or 
- By program: e.g., `PROGRAM_X_rules.json` containing all rules extracted from program X.
Alternatively, we could have one large file, but splitting them improves navigability and version control diffs. We will likely output both JSON and YAML (since YAML is easier to read/edit manually, while JSON might be easier for certain tools to parse). These can be generated in one go (YAML is basically a formatted view of the JSON data structure in Python).

Each rule entry in the JSON/YAML includes all fields as per schema. We ensure that lists and nested structures are properly indented/serialized. If using YAML, we’ll include comments or tags if helpful (YAML supports comments, which we could use to embed the original code snippet as a comment above the rule, providing immediate context in the YAML file itself for a reader).

We also plan to include a **traceability tag** in each rule object linking to the source location. For example, a rule might have `"source_id": "COBOL:ORDPROC:PARA123"` which could correspond to a bookmark in the code. If we set up a convention (like `program:label` or line range), a developer could quickly find the origin. In documentation, we can hyperlink this if we have code listings available, but at least it’s recorded.

**Documentation for Humans:** In addition to raw YAML/JSON, we will create a documentation deliverable, likely in Markdown or HTML, that presents the business rules in a friendly way:
- **Contextual Grouping:** Organize rules by business domain or user functionality. This might be a hierarchy of sections. For example:
  - *Customer Account Rules* (with sub-sections for validations, calculations, etc. related to Customer Accounts),
  - *Loan Processing Rules*, 
  - *Insurance Policy Rules*, etc., depending on the system.
  If the domain breakdown isn’t obvious, we might organize by the source program (since often programs are named by domain).
- **Rule Listings:** Under each section, list the rules (perhaps in a table or bullet list form). Each rule listing will include the rule’s description in natural language, the type (validation/decision/etc.), and possibly key conditions/outcomes in brief. Next to it, we provide the source reference and confidence. For example: “**Rule:** Applicants under 18 are not eligible for account creation. **Type:** Validation. **Source:** OPENACCT RPG (line 120). **Confidence:** High (0.98).”
- **Traceability Links:** If possible, we’ll cross-reference to the JSON/YAML or code. E.g., the documentation might have a hyperlink that when clicked, shows the original code snippet (we can generate an HTML snippet for each referenced code segment). This might be facilitated by tools like Sphinx or MkDocs where we can embed code with anchors. If not in initial scope, we will at least provide line numbers and file names as text so someone can manually find it.
- **Comments and Annotations:** We will carry over any relevant code comments into the documentation. If a code comment said “*// Biz rule: if cust age <18, reject*”, we will include that as a note in the documentation to reinforce that this was an intended rule. We may also add explanatory notes (via the LLM or SMEs) for complicated rules. For example, if a rule is very technical, we might add: “*(This rule calculates the amortization schedule. It’s complex because of X...)*” – giving context beyond the raw logic.

**Versioned Documentation:** Each iteration of rule extraction can produce a versioned document (e.g., *BusinessRules_v1.0.docx* or markdown in a Git repo). We plan to maintain a change log of the rules. E.g., if in the next extraction some rules changed (due to code changes or improved extraction), the documentation will highlight new/modified rules, making it easy for reviewers to see what’s updated.

**Templates and Standards:** We will follow any existing documentation standards the organization has (if there’s a template for business rules, we can adopt it). If none, we’ll create a clean template with the company branding (if desired) and a clear introduction explaining what the document is (basically a modern “requirements document” reverse-engineered). Each rule will have:
- A unique identifier (as given in JSON, like RULE-001, etc.) so it can be referred to unambiguously in discussions.
- A one-line summary (like the description field).
- A detailed description if necessary (for multi-condition rules, we might break it down).
- The origin (legacy system reference).
- Any notes (like “Note: This rule might be overridden by parameter X in production” if we discovered such a thing).

We aim for the documentation to be understandable by business analysts, not just developers. This means avoiding or explaining technical terms. For instance, instead of writing “If CUST_STAT = 'I' then...” we’ll write “If a customer’s status is 'Inactive' then ...” The glossary integration helps here, ensuring we replace code jargon with business terms as much as possible in the documentation.

**Example Documentation Excerpt:**  
*Section: Account Closure Rules*  
- **Rule AC-1:** *Accounts that have been inactive for over 12 months are automatically closed.* (Decision Rule) If an account’s last activity date is more than 1 year in the past, then the system will mark the account as closed on the next batch cycle. **Source:** COBOL program `ACCTCLSE`, lines 220-250. **Confidence:** 0.9. *(This rule helps reduce maintenance of long-dormant accounts.)*  

- **Rule AC-2:** *VIP accounts are exempt from automatic closure.* (Override Rule) If the account type is VIP, the inactivity closure rule does not apply. **Source:** COBOL `ACCTCLSE`, lines 255-260. **Confidence:** 0.95.

*(The above are illustrative – in actual output we’d cite the real sources.)*

Notice we included a rationale note for AC-1. We might get these rationales from code comments or SME input. If not, we keep it purely descriptive.

**Traceability Matrix:** We can also produce a matrix mapping each program to the rules found in it, and each rule to the programs that implement it. This is helpful if the organization wants to ensure every piece of code is accounted for by some rule or technical function. This matrix could be an appendix in the documentation or a separate CSV file. It’s basically a two-way mapping: Program -> Rules, and Rule -> Program(s). Because our data is structured, generating such a matrix is straightforward.

**Adherence to Standards:** If the output needs to adhere to something like ISO documentation standards or IEEE for requirements, we can format accordingly (with shall statements etc.). But since this is internal modernization, a simpler format is likely fine. The key is clarity and completeness.

**YAML/JSON format details:** We will document the schema (perhaps in a README or at top of the YAML file as comments). For example, in the YAML file we might put:
```yaml
# Schema: ValidationRule { id: str, description: str, condition: str, action: {level: str, message: str}, source_reference: {program: str, lines: str}, confidence: float, ... }
```
This way anyone reading the YAML knows what each field means. We’ll also ensure uniform formatting (indentation etc.), and no overly long lines (for readability).

**No Secrets in Output:** Since this might be shared widely, we’ll be careful that the output doesn’t inadvertently leak something sensitive (like literal passwords or keys, which sometimes are in code). Typically business rules don’t involve secrets, but we’ll scan the output for any such thing just in case.

In summary, the output will consist of:
- **Structured rule files (JSON/YAML)** – the authoritative machine-readable representation.
- **Human-oriented documentation (Markdown/HTML/Word)** – organized explanation of rules with trace links and possibly code excerpts.
- **Supporting files** like glossary, trace matrices, etc., as needed to provide full context.

This dual output meets the needs of both software agents (which can directly ingest JSON/YAML) and stakeholders/humans (who need to understand and validate the rules). By following these documentation standards, we ensure the extracted knowledge is accessible and actionable, not just a dump of data. It essentially becomes a living requirements document for the legacy system, bridging to the future system development.

## System Architecture Overview

 *High-level architecture of the PydanticAI-based rule extraction pipeline.*  
The proposed solution employs a modular pipeline that ingests legacy AS/400 artifacts, applies an LLM-driven analysis, and outputs structured business rules. At a high level, the system consists of an **LLM Extraction Agent**, a **Schema Formatter & Validator**, and a **Metadata/Knowledge Store** orchestrated by a multi-agent framework. We leverage **PydanticAI** – an agent framework by the Pydantic team – to manage these components due to its built-in support for type-safe outputs and multi-agent workflows. The agent treats source code as data, first parsing it into an abstract syntax structure (where possible) before invoking the LLM; this ensures we capture the code’s intrinsic structure rather than treating it as plain text. The LLM extraction agent identifies business logic in code, which is then converted into structured rule objects via Pydantic models. A validation module checks each extracted rule for accuracy and assigns confidence scores. All results (the rule objects and associated metadata) are stored in a versioned repository or database for traceability and later consumption by development agents. The entire pipeline is designed to be **platform-agnostic** – running in a containerized or serverless environment – and prioritizes quick deployment by utilizing off-the-shelf AI services and frameworks.

**Components Overview:**

- **LLM Extraction Agent:** The “brain” of the system, powered by a large language model, which reads legacy code and extracts candidate business rules along with contextual explanations.
- **Schema Formatter & Validator:** A module (implemented via Pydantic models) that converts the LLM’s output into a well-defined JSON/YAML schema and validates that the content conforms to expected types/fields. This prevents malformed or incomplete rule data.
- **Multi-Agent Orchestrator:** Orchestration logic that coordinates multiple specialized agents (extractor, formatter, validator, etc.). PydanticAI supports such an agent-based architecture out-of-the-box, allowing agents to share state and communicate seamlessly. This orchestrator manages the flow of data through pipeline stages and handles re-tries or human escalation as needed.
- **Metadata Store:** A repository for storing extracted rules and their metadata. This can be a document database or version-controlled file store containing JSON/YAML rule definitions. It also includes a simple **lineage store** mapping each rule back to its source location (program name, line numbers) and an index for quick lookup. Optionally, a graph database can serve as a knowledge base linking rules to business entities (see *Knowledge Graph Integration* below).

In summary, the architecture balances **rapid development** (by using a powerful LLM and PydanticAI for quick implementation) with **structured output** (via enforced schemas  ([From COBOL to Business Rules—Extracting Business Rules from Legacy Code](https://research.vu.nl/files/231254036/From_COBOL_to_Business_Rules_Extracting_Business_Rules_from_Legacy_Code.pdf#:~:text=%E2%80%A2%20A%20copy%20table%20depicting,code%20leading%20to%20that%20output))torage). No BRMS or external rule engine is needed at this stage – rules are simply materialized as data – keeping the design lightweight and easy to integrate into subsequent modernization steps.

## Modular Pipeline Stages

To extract business rules from RPG/COBOL code, the process is broken into clear, modular stages. Each stage in the pipeline transforms the input incrementally, which makes the system easier to debug, extend, and parallelize if needed. Below are the major stages in sequence:

1. **Preprocessing Legacy Code:** The pipeline begins by gathering and normalizing all source artifacts – RPG code, COBOL programs, CL/JCL scripts, database schemas, screen definitions, and even code comments. Preprocessing parses these sources into an intermediate representation. For example, we use language-specific parsers to generate Abstract Syntax Trees (ASTs) for COBOL and RPG where possible. Non-executable elements like comments or DDS (display file) definitions are also parsed to capture any business terminology. This stage may remove or flag non-business logic (e.g. technical utility code or boilerplate) so that subsequent steps can focus on true business rules. Known IBM analysis tools like Application Discovery (ADDI) or IBM Rational’s parsers can assist in identifying code structure and dependencies (e.g. call graphs, copybook expansions). The output of preprocessing is a structured, navigable form of the legacy code (AST or pseudo-code) and a map of where each piece of logic resides (for traceability).

2. **LLM-Based Rule Identification:** Next, a Large Language Model agent scans the preprocessed code to identify business rules embedded in logic. This involves detecting conditional statements, computations, data validations, and other decision points that represent business policies. The agent processes code in manageable chunks (such as one program or module at a time, or even smaller function/procedure level) to stay within context length. For each chunk, it generates a list of candidate rules in natural language form, along with evidence (e.g. which lines or paragraphs led to that rule). We prompt the LLM in a **chain-of-thought** style, asking it to reason step-by-step through the code and explain what each significant code path represents in business terms. This is analogous to “mining the code to discover business rules” as done in traditional analysis tools. For example, an IF…ELSE block that calculates a fee waiver might be extracted as a rule “If customer age < 18, then waive the account maintenance fee.” Comments and variable names are leveraged to assign meaningful business terminology to each rule. The output of this stage is a set of raw, unstructured rule descriptions with references to source code.

3. **Rule Classification:** The identified rules are then categorized into predefined types: **decision rules**, **calculation rules**, **validation rules**, or **workflow rules**. Classification can be done by the LLM agent (e.g. by analyzing the linguistic cue or structure of the rule) or by simple heuristics. For instance, rules that enforce an input constraint (“must be X or else error”) are tagged as *validation* rules, those that compute or derive a value (“compute X = Y+Z”) as *calculation* rules, and those selecting among different paths or outcomes as *decision* rules (similar to DMN decision logic). Workflow or process flow rules (e.g. “if transaction type = X, route to Queue A”) are tagged separately. This classification step is important because each rule type will be formalized into a slightly different schema. It also helps prioritize – for example, decision and validation rules may be of higher interest initially. The classification is stored as a field in the rule metadata. (If the LLM is used, the prompt provides examples of each category so it can label each rule accordingly).

4. **Structured Formalization (PydanticAI Modeling):** In this stage, the raw text of each rule is converted into a structured object that follows a predefined Pydantic model (see next section for schema design). We define a Pydantic `BaseModel` for each rule type (DecisionRule, CalculationRule, ValidationRule, WorkflowRule, etc.) with fields capturing the logical structure. The agent (or a dedicated Formatter agent) takes each rule description and populates the schema: e.g., separating the rule into `conditions`, `outcome`, `trigger_event`, etc., and adding metadata like `source_reference`. Because PydanticAI allows us to specify the desired output schema and have the LLM directly generate a JSON that conforms to it, we can combine this with the extraction step. In practice, the LLM could be prompted to output JSON in the required format immediately after describing the rule, but we have a validation step next as a safeguard. By formalizing rules, we ensure machine-consumability – the rules become data that other tools or agents can read. For example, a decision rule might be represented as a row in a decision table or as a JSON object with explicit condition fields.

5. **Validation & Confidence Scoring:** Each structured rule is then validated and assigned a confidence score. **Schema validation** is automatically enforced by Pydantic – if the LLM’s output is missing a field or has a wrong type, it will be flagged and can be corrected either by a retry or by post-processing (for example, if a field meant to be numeric is output as a string, we can attempt a conversion or reprompt the LLM for that field). Beyond schema, we implement a **semantic validation** layer: an *LLM-as-judge* (or “referee” agent) re-checks the rule against the original source code snippet to verify correctness. This agent could prompt the LLM with the code and the extracted rule, asking “Is this rule an accurate description of the code? Answer with a correctness score 0–100.” Using a separate verification step with an LLM is inspired by self-verification techniques where the model validates its own answers. Additionally, we apply heuristics: for instance, if the rule was derived from a very straightforward `IF` statement, confidence is inherently higher than if it was inferred from a complex, multi-paragraph flow. Any rule that the validation deems low-confidence or mismatched is flagged for human review (or queued for an iterative refinement by the agents). The output of this stage is a set of **validated, structured rules** each with a confidence score and any notes about uncertainties.

6. **Output Serialization and Traceability:** Finally, the structured rules are serialized to machine-readable formats (JSON and/or YAML) and saved to the metadata store. Each rule entry includes a unique identifier and version, allowing the set of rules to be version-controlled. We include traceability info such as timestamps and source lineage: e.g. which program and line range the rule came from, and a hash or version tag of the source code. This ensures that as the legacy code evolves (or our understanding improves), new extractions can be diffed against old ones. The rules can be stored as individual JSON files (one per rule or per program) or aggregated by domain. A simple example of a serialized rule might be: `{"id": "RULE-001", "type": "validation", "condition": "Account balance < 0", "action": "reject transaction", "source": "COBOLProgA: line 200-210", "confidence": 0.95, "extracted_at": "2025-04-05T12:00:00Z"}`. In addition to JSON/YAML files, the system can auto-generate documentation (for example, a Markdown or HTML report) that lists all rules, organized by type and module, including the context and any comments. This provides human-friendly documentation side by side with the raw rule data.

Each stage of the pipeline is designed to be **modular** and can be improved or replaced independently. For example, if a more formal static analysis tool is later introduced, it could augment the Preprocessing stage to feed better input to the LLM. Similarly, if fine-tuning the LLM greatly improves accuracy, the Validation stage can be simplified. This modular approach, combined with multi-agent orchestration, aligns with best practices in GenAI-enabled modernization, but avoids deep coupling with any specific vendor tool, thus keeping the solution flexible and fast to adapt.

## LLM Strategy

Choosing the right Large Language Model and prompt strategy is critical for fast, accurate extraction of business rules from legacy code. Our approach prioritizes **fast time-to-market**, so we will begin with state-of-the-art general models via API, then consider domain-specific fine-tuning in parallel.

**Model Selection:** We recommend using a top-tier model known for strong code understanding and reasoning. For example, OpenAI’s GPT-4 or Anthropic’s Claude v3 (Opus tier) are ideal initial choices. Claude 3 Opus is specifically noted for handling complex, multi-step tasks and strong coding abilities, which suggests it can navigate legacy COBOL/RPG logic effectively. These models have the advantage of being immediately available (via cloud API or platforms like Amazon Bedrock) and highly capable, which accelerates initial development. If data privacy or on-prem deployment is a concern, an open-source code model like **Code Llama** or **StarCoder** could be fine-tuned and used behind the firewall, albeit with likely lower baseline performance. Another promising avenue is specialized models: for instance, researchers have introduced *XMainframe*, a code LLM tuned with mainframe COBOL knowledge that outperforms general code models on COBOL tasks. If such a model (or IBM’s **Granite** model for Z) is available, it could be evaluated for this project. In summary, start with the most powerful model accessible to get good results quickly, then iterate.

**Prompting Approach:** We will employ an iterative prompting strategy combining **few-shot examples** and **chain-of-thought reasoning**. The prompt to the LLM will include: (a) a brief system role definition (e.g. *“You are a code analysis assistant specialized in IBM i (AS/400) business logic extraction.”*), (b) one or two examples of code (perhaps simplified) with the expected JSON rule output (this helps the model align to the output schema), and (c) the target code snippet or program text to analyze. We will encourage step-by-step analysis by instructing the model to first explain the code logic in its own words (this can be done in a hidden “scratchpad”), then output only the structured rules JSON. This technique helps the model not to skip reasoning steps. In practice, PydanticAI allows us to define a `result_type` (the Pydantic model) and will validate the LLM output against it, so our prompt will focus on guiding content while the framework handles structure. We also incorporate *structured prompt templates*: e.g., “List all business rules in the following code. Use the format: [JSON list of Rule objects]. Only include rules, no other text.” to reinforce the desired output shape. 

For RPG code, which can be more cryptic (especially older fixed-format RPG), we will include guidance in the prompt about RPG specifics or even use a preprocessor to translate RPG to pseudo-code. Similarly, COBOL copybooks for data definitions will be provided to the model alongside the code, so it understands the fields (we can prepend a section like “DATA DICTIONARY: Customer-Record has fields...” if needed). This contextual info can prevent misinterpretation of field meanings.

**Chain-of-Thought & Verification:** We plan to use a chain-of-thought approach not only for extraction but also for self-verification. For example, after the model produces a candidate rule, we may ask it (in a follow-up prompt) to double-check if that rule is fully supported by the code, as part of the same call if possible. Some LLMs (like GPT-4) can follow complex multi-turn instructions well, which we can utilize to reduce errors. The prompting may involve asking the model to think of counterexamples or ensure the inverse conditions don’t produce a rule it missed.

**Fine-Tuning Plan:** While initial prototypes will rely on zero-shot or few-shot prompting of a powerful model, we recognize that fine-tuning an LLM on domain-specific code can improve accuracy and reduce needed prompt complexity. We propose a parallel effort to curate a fine-tuning dataset:
- Collect sample **RPG and COBOL code** with known business rules. This could include open-source COBOL programs (e.g., from government or academic repositories), or sample code provided by IBM in manuals. We can augment this with synthetic code: for instance, write small COBOL/RPG programs that implement a few clear business rules (loan eligibility, insurance premium calc, etc.), then use those as training pairs (code → labeled rules).
- Include any available **proprietary code** from the client (if allowed) in the fine-tuning data. If direct code can’t be used for IP reasons, extracting an abstract representation (like replace business terms with generics but keep structure) is an option.
- Leverage existing model fine-tuning efforts: IBM’s approach with Granite included fine-tuning on thousands of pairs of COBOL and Java programs – while that was aimed at translation, those datasets could implicitly help rule extraction understanding. Open Mainframe Project’s datasets or the MainframeBench from XMainframe work could also be sources of COBOL knowledge.

A fine-tuned model (even if it’s a smaller 7-13B parameter open model) could then be used in the pipeline to reduce reliance on an external API and run on-prem if needed. The fine-tuning objective would be to have the model directly output structured rules given code, effectively teaching it the “pattern” of business rule extraction. This is ambitious, but even partial fine-tuning (e.g. on understanding RPG syntax) could be beneficial. We will evaluate ROI of fine-tuning after the initial agent is working – if the base model already performs well (which is possible with GPT-4-level models), fine-tuning can be a second-phase optimization rather than a blocker for MVP. 

**Summary:** Use the best available LLM to jump-start development, craft prompts with examples and step-by-step reasoning to maximize correctness, and plan for fine-tuning with COBOL/RPG-specific data to further improve accuracy and self-sufficiency. By combining these strategies, we can achieve high quality rule extraction quickly while laying the groundwork for continuous improvement.

## PydanticAI Schema Design

A core part of this solution is expressing extracted business rules in a **machine-consumable logical format**. We define a set of Pydantic data models (schemas) to formalize different types of business rules. This provides a clear contract for what constitutes a “rule object,” enabling downstream software agents to consume the rules easily. It also ensures consistency and validation of the LLM outputs (since Pydantic will enforce types and required fields).

**Rule Type Schemas:** For each category of rule identified, we create a schema class. The primary ones are:

- **DecisionRule:** Represents a business decision logic, typically based on one or more conditions leading to a choice among outcomes. Fields might include:
  - `id` (unique rule identifier, string),
  - `description` (human-readable summary),
  - `conditions` (list of boolean conditions that must be satisfied, each condition could be a structured object with operands, operator, and an optional field reference),
  - `outcomes` (list of possible outcomes or a mapping of condition-combinations to outcome; for a simple IF/ELSE, there might be two outcomes: one when conditions true, one for else),
  - `outcome_type` (if outcomes are e.g. assignments vs workflow branches, etc., but for pure decision maybe not needed),
  - `source_reference` (metadata linking to code: program name, section, line numbers),
  - `lineage` (if this decision rule is composed from multiple code pieces or calls, list them),
  - plus common metadata fields like `confidence`, `extraction_timestamp`, `system_version`.

- **CalculationRule:** Represents a formula or derivation. Fields:
  - `id`, `description`,
  - `formula` (perhaps a structured representation of the calculation, e.g. an expression tree or a simplified algebraic string "TARGET = F1 * 0.05 if F2 = 'Y' else F1 * 0.1"),
  - `target_field` (the business field being calculated, if identifiable),
  - `source_fields` (list of input fields used in the calculation),
  - `conditions` (if the calculation only applies under certain conditions),
  - `source_reference`, `confidence`, etc.

- **ValidationRule:** Represents a check or constraint on data.
  - `id`, `description`,
  - `condition` (the condition that triggers the validation failure, e.g. "account_balance < 0"),
  - `action` (what happens when condition is met – often this is an error or warning; we can formalize it as `level: error/warning` and maybe an `message` or `code`),
  - `field` (optional, if this is a single-field validation rule like "FIELD X must satisfy condition Y"),
  - `source_reference`, `confidence`, etc.
  - Possibly a `severity` or `priority` field for the rule.

- **WorkflowRule:** Represents a routing or process flow decision.
  - `id`, `description`,
  - `trigger` (event or state that triggers this rule, e.g. "Order status changed to Shipped"),
  - `condition` (if any, e.g. "if customer is VIP"),
  - `action` or `next_step` (what workflow action to take, e.g. "send approval request", "route to queue ABC"),
  - `source_reference`, etc.

All rule types inherit from a base class `BusinessRule` that has common metadata fields: `id`, `type` (enum of rule type), `source_reference` (which can include the source file/module and possibly a pointer into that source like line numbers or a byte offset if from compiled code), `lineage` (if the rule was composed from multiple code elements or if it relates to other rules), `extraction_timestamp`, `system_version` (an identifier for the codebase version or extraction run). **Lineage** is important in cases where a business rule is not explicitly in one place in code – e.g., part of the rule is in a main program and part in a subroutine. In such cases, the rule object might list both source locations. We also include a `confidence` score (float 0 to 1) and possibly a `reviewed` flag to indicate if a human SME has confirmed the rule.

**Metadata Fields Rationale:**  
- *Source trace/lineage:* This links back to the legacy code, enabling verification and future maintainers to locate where the rule came from. For example, `source_reference: {"program": "CUST0001RPG", "lines": "230-265"}`. If multiple sources, it could be a list.  
- *Confidence:* Allows prioritizing which rules might need human attention and provides a measure of extraction quality.  
- *Extraction timestamp:* Helpful for auditing and ensuring we know when the rule was pulled (e.g., after a code change, the timestamp would update).  
- *System version:* If the legacy system has versions or if we tag each extraction with a commit ID or build number, storing it here helps track rule evolution (e.g., rules might be slightly different between version 1.0 and 1.1 of the application).

**Example Outputs:** Below are simplified examples of how a rule might be represented in **YAML** and **JSON**, conforming to the schema definitions:

- *Example 1: A Validation Rule (YAML)*  
```yaml
id: RULE-V001  
type: validation  
description: "Customer age must be 18 or above to create an account."  
condition: "customer_age < 18"  
action:  
  level: error  
  message: "Applicant must be at least 18 years old."  
source_reference:  
  program: "OPENACCT"  
  lines: "120-130"  
lineage: ["OPENACCT:120-130"]  
confidence: 0.98  
extraction_timestamp: "2025-04-05T12:20:30Z"  
system_version: "AS400_PROD_2025_04"  
```

- *Example 2: A Decision Rule (JSON)*  
```json
{
  "id": "RULE-D005",
  "type": "decision",
  "description": "Determine interest rate tier based on credit score.",
  "conditions": [
    { "field": "credit_score", "operator": ">=", "value": 750, "label": "HighScore" },
    { "field": "credit_score", "operator": "<", "value": 750, "label": "LowScore" }
  ],
  "outcomes": [
    { "when": "HighScore", "rate": 0.05 },
    { "when": "LowScore", "rate": 0.08 }
  ],
  "outcome_type": "interest_rate",
  "source_reference": {
    "program": "LOANPROC",
    "paragraph": "CALC_RATE",
    "lines": "200-250"
  },
  "confidence": 0.90,
  "extraction_timestamp": "2025-04-05T12:21:00Z",
  "system_version": "AS400_PROD_2025_04"
}
```

In the decision rule example, the rule essentially says: if credit_score ≥ 750, interest_rate = 5%, else interest_rate = 8%. We represented it with two condition entries and outcomes keyed by a label. (This format is analogous to a DMN decision table with two rules; in a DMN-like representation it could also be one rule with two columns, but JSON here is one way to do it.)

We will refine the schema as we iterate, but the key is that **every field is explicitly defined and validated**. Pydantic’s validation will reject any LLM output that doesn’t match (for example, if the LLM omits `id` or uses a wrong data type somewhere), enforcing reliability in the outputs. This structured approach also means these JSON/YAML rule files could later be converted to a standard like **DMN (Decision Model and Notation)** if integration with a rules engine becomes desirable. For now, they serve as a contract for software agents: e.g., a future “code generation agent” can read the JSON rule and know exactly the logic to implement in a new system.

**DMN-like Decision Tables:** It’s worth noting that for decision rules with multiple permutations, we could output them in a decision table format. In JSON, this could be an array of rules with conditions and outcomes. Our schema design can accommodate that (as shown in the outcomes array above). If needed, we can produce an actual DMN XML or use YAML to tabulate it. However, since DMN is an XML standard, we’ll stick to JSON/YAML for simplicity and transform to DMN later if needed. The focus is on logical clarity and completeness of rule representation.

**Documentation and Schema Evolution:** Each schema class will include a docstring or metadata that describes the rule in plain English. This is useful for auto-generating documentation. For example, the `ValidationRule` model can have a description like “Represents a business validation that must hold true; if condition evaluates to true, a violation occurs triggering an action.” We will maintain these definitions in a **data dictionary** for the rules themselves. This helps new developers or SMEs understand what each field means. As new types of rules are discovered (perhaps we find a kind of rule not anticipated, like a **timing rule**: e.g., “process X must happen within N days of event Y”), we can extend the schema definitions accordingly. The use of Pydantic makes this relatively easy – we add a new model or new fields and regenerate the data structures. Versioning of schema will be managed (e.g., the `system_version` in metadata can also double as a schema version if needed, or we maintain a separate version number for the schema itself in the output files).

Overall, the PydanticAI schema design ensures that from unstructured legacy code we move to **highly structured, typed, and annotated rule objects**. This not only serves the immediate modernization goals but creates a knowledge asset for the organization – a repository of business rules that can be queried, tested, and transformed independent of the old source code.

## Multi-Agent Workflow Design

To efficiently coordinate the complex task of extracting and validating rules, we adopt a **multi-agent workflow**. Each agent in the system has a specialized role, and together they operate as a team to perform extraction with checks and balances. By dividing responsibilities, we can parallelize certain tasks and apply different prompting strategies tuned to each role. PydanticAI’s architecture inherently supports multi-agent collaboration with shared state, but the design can also be implemented with other frameworks (LangChain, CrewAI, etc.) if needed.

**Agent Roles and Responsibilities:**

- **Code Extractor Agent:** This agent’s role is to read through a given source (e.g., one program or a module chunk) and output raw business rule candidates. It focuses on comprehension and recall. Internally, it might use the LLM with prompts that encourage detailed reasoning about the code logic. It produces a preliminary list of rules (in either narrative or draft JSON form) along with any explanatory notes. Essentially, this is the “primary” agent doing the heavy lifting of understanding code.

- **Rule Formatter Agent:** The formatter takes the extractor’s output and ensures it conforms to the required schema. If the extractor already attempted JSON output, the formatter validates it using the Pydantic models. If the extractor gave a narrative or partially structured output, the formatter agent reorganizes it into the final JSON/YAML structure. This agent might call the LLM with a prompt like: *“Given the following rule description, fill in this JSON template…”* or directly use Pydantic parsing. It handles adding any missing metadata (timestamps, IDs). This separation is useful to keep extraction prompts simpler (the extractor can focus on finding rules, not JSON syntax) and then let the formatter enforce structure.

- **Validation/Verifier Agent:** The validator acts as a quality gate. It reviews each structured rule against the source code context. This agent might use a different prompting approach, for example: *“You are a code reviewer. Verify if the rule accurately represents the code snippet.”* It could even use a different model or tool (for example, a simple interpreter or test harness, if we generate one, to simulate the code). Primarily, though, an LLM agent here can do a second-pass explanation and point out mismatches. It then annotates the rule with a confidence score or flags it for issues. In some implementations, we could have two validator agents for redundancy (maybe run the rule and code through two different LLMs and compare judgments).

- **Memory Manager/Context Agent:** This agent is responsible for maintaining the shared context and knowledge base that all other agents use. For instance, it can store the knowledge graph or glossary (see next section) and provide definitions to the extractor agent on demand (like “Agent asks: what is field CUST_STATUS? Memory agent looks it up in the glossary and returns: CUST_STATUS = Customer account status code.”). It also keeps track of which parts of the code have been processed, preventing duplicate work. If the extraction is done module by module, the memory agent can note progress and also handle inter-module links (e.g., if a rule spans a call, it can remind the extractor agent when it reaches the called program). Technically, this could be implemented by using a shared database or context object that all agents can query via tool interfaces.

- **Orchestrator:** While not a separate persona agent, the orchestrator logic oversees the sequence of steps. It could be a simple loop in code or a higher-level agent that delegates tasks. For example, for each program: orchestrator invokes Extractor → passes result to Formatter → passes to Validator → either accepts and stores the result or triggers a refinement. The orchestrator also manages **human-in-the-loop checkpoints**: if the Validator flags a rule, the orchestrator might pause and request human review for that item. 

**Interaction Model:** The agents communicate through well-defined channels – typically by passing the result objects along the pipeline. Using PydanticAI, we can have agents produce outputs of certain model types which become the input for the next agent, ensuring a smooth handoff. The **Extractor** produces `DraftRule` objects, the **Formatter** expects those and produces `BusinessRule` objects, the **Validator** appends a verification report or adjusts the `confidence`. The **Memory agent** can be invoked by others at any time (for instance, the Extractor might use a tool call to query the glossary: PydanticAI and other frameworks allow agents to call functions/tools mid-prompt – we can register a function like `lookup_term(term) -> definition` that the Extractor agent can call when needed).

**Parallelism and Iteration:** Some tasks can be parallel. If multiple programs or modules can be processed independently, we could spin up multiple Extractor agents in parallel (subject to available compute/LLM throughput). The orchestrator ensures each section of code is assigned to an extractor agent. The Memory agent helps ensure they don’t conflict (like if two agents want to update the knowledge graph concurrently, the memory agent serializes those updates). After initial extraction, there might be an iteration: if the Validator finds an issue, we could either:
- Have the Extractor agent run again on that section with a refined prompt (perhaps giving it the feedback from the Validator).
- Or have a specialized **Refinement agent** that attempts to fix the rule (this could be the Formatter agent re-running with more context).

The multi-agent setup enables a kind of feedback loop: Extract → Validate → Refine (if needed) → finalize. This loop continues until all code is processed and all rules are either validated or marked for human review.

**State Persistence:** All intermediate and final results are persisted (possibly by the Memory manager). We maintain a state, for example, in a simple JSON file or in memory that lists what programs have been processed, how many rules extracted, etc. In case of a system crash or restart, this state allows the process to resume from where it left off, which is important for large codebases. We could also log the conversations (prompts and responses) of each agent for auditability using PydanticAI’s logging or an external logger (which can be helpful if we need to trace why an agent made a certain decision).

**Human-in-the-Loop Checkpoints:** Despite automation, involving human experts (SMEs) at critical points greatly increases trust and accuracy. The workflow will include optional pauses where a human can review. For example:
- After extraction of a module, a BA or SME can quickly scan the extracted rules (perhaps via a dashboard interface) to confirm if they “make sense” before they are validated and finalized.
- After validation, any rule below a confidence threshold (say <0.7) is queued for SME review. The SME can then either approve it as-is, edit the rule (correct the logic or wording), or mark it as a false extraction to be discarded.
- The human feedback is then fed back. In PydanticAI, this could even be interactive: an agent could be a “HumanAgent” where the orchestrator asks for input. Or simpler, the system could output a file of “pending review” rules for a person to edit and then read those edits back in.

By structuring the workflow with agents, we isolate concerns: each agent can be developed and optimized independently. For instance, if we find the extraction agent misses certain patterns, we tweak its prompt or logic without affecting the validator. This design also mirrors real-world team collaboration (one person finds information, another checks it) which tends to improve outcomes. CrewAI or similar frameworks could be used here as well – e.g., CrewAI allows definition of multiple agent personas and their conversation. In our case, agents communicate mostly through data passing rather than long dialogues, so a simpler orchestrator might suffice. 

**Logging and Monitoring:** The multi-agent system will include logging of actions for transparency. Each agent can log what it did (e.g., “Extractor: found 5 rules in program X”, “Validator: rule Y confidence downgraded due to missing condition”). This provides a trace of the pipeline which is useful for debugging and for users to understand what the AI did (which is often a requirement in sensitive modernization projects).

In summary, the multi-agent workflow is designed for **collaboration and oversight**. By splitting roles, we achieve a checks-and-balances system: the Extractor’s work is checked by the Validator, and formatting issues are handled explicitly by the Formatter, etc. This reduces the chance of a single-point failure (e.g., one monolithic prompt hallucinating without check) and aligns with how complex tasks are handled in advanced LLM applications (breaking tasks into smaller, more controllable steps). The end result is a more reliable and maintainable pipeline, aligning human and AI strengths.

## Validation and Verification Layer

Ensuring the extracted business rules are correct and complete is just as important as extracting them. The Validation & Verification layer introduces quality control through automated checks, LLM-based judgment, and opportunities for human review. The goal is to have a high degree of confidence in the rule set before it’s handed off to developers or other systems.

**LLM-as-Judge (Self-Verification):** As mentioned, we employ a strategy where the LLM itself verifies or scores the extracted rules. This is inspired by recent research into using LLMs to critique or validate their own outputs. In practice, after a rule is extracted and formatted, we feed the LLM a prompt like: *“Here is a segment of code and a proposed rule extracted from it. Is this rule an accurate and complete description of what the code does? Answer with reasoning and a correctness score.”* The LLM then performs a backward verification: it takes the rule (declarative form) and essentially “replays” it against the code. For example, if the rule says "X must be > 0", the LLM looks at code to see if indeed there’s a check `IF X <= 0 THEN ...error`. This process acts as an independent check. We might generate multiple variants of the rule and ask the LLM which is correct (a form of comparison). If the LLM judge finds a discrepancy (say the rule missed a condition or mis-stated a threshold), it will either correct it or flag it.

**Heuristic Checks:** Alongside the LLM judge, we implement a series of deterministic checks:
- **Consistency Check:** If a rule was extracted multiple times from different places (e.g., the same rule logic appears in two programs), ensure they are consistent. If not, flag them (it could indicate a subtle difference or an extraction error).
- **Contradiction Check:** Using the structured format, check if two rules from the same domain contradict each other (this is rudimentary conflict detection – e.g., one rule says “if status = A then do X” and another says “if status = A then do Y”, possibly a conflict unless X and Y are complementary or ordered).
- **Completeness Check:** For decision tables, ensure coverage. For example, if a decision rule has conditions that split a scenario, do we have outcomes for all relevant cases? (DMN engines often have validations for missing rules or overlapping rules – we can mimic some of that logic).
- **Data Type/Schema Cross-check:** If we have access to schema (DB2 field definitions), verify that the rule conditions make sense with field types. E.g., if a field is numeric, a rule checking for “= 'ABC'” is probably incorrect or indicates a parsing issue.
- **Range/Boundary Analysis:** For numeric conditions, check boundaries. If code says “> 100” we might also see if there's an implicit “<= 100” handling elsewhere. This is more analysis than pure check, but it can catch if maybe an else branch wasn’t captured as a rule.

We can automate some of these by simple Python logic on the JSON rule outputs.

**Confidence Scoring:** We aggregate the results of LLM judge and heuristics into a unified confidence score for each rule. For instance, start with the self-reported confidence from extraction (the LLM might implicitly give one or we assign high confidence if straightforward), then deduct points for any issues found (or boost if cross-verified by multiple methods). The scoring mechanism can be tuned with weightings. The end result is each rule gets a score 0.0 to 1.0. We consider anything above, say, 0.8 as high confidence (green), 0.5-0.8 medium (yellow, needs maybe a quick look), below 0.5 low (red, needs review).

**Review Dashboard:** All validated rules will be presented in a dashboard/report that highlights any low-confidence items. The dashboard can group rules by module or business domain, show the confidence color, and allow a human reviewer to click on a rule to see details: the rule itself, the source code snippet that led to it, and the LLM’s rationale (we can include the chain-of-thought from the verification step here, effectively showing “why the AI thought this was correct or not”). This kind of interface significantly aids an SME in reviewing AI outputs because it’s focused and context-rich.

**Automated Test Cases:** Where feasible, we will generate simple test cases to verify rules. This can be done by leveraging the LLM to create input data that triggers certain rules. For example, if a rule is “if salary > $100k then require VP approval”, we can ask the LLM to produce two fictional records: one with salary 120k (expect rule triggers) and one with 80k (expect rule does not trigger). If we had a way to execute the legacy code (or a partial simulation of it) with those inputs, we could confirm the behavior. However, executing legacy COBOL/RPG isn’t trivial in an automated test harness without the full environment. Instead, we might create a **pseudo-code simulator**: since we have AST or structured logic, we could evaluate the condition expressions ourselves on the test data. This is more of a stretch goal, but it could increase confidence for critical rules. At minimum, documenting such example scenarios in the output helps humans validate the logic.

**Formal Verification (Optional):** For some rules, especially calculation formulas or critical business invariants, we could use formal methods. One idea is to translate rules into a constraint solver format (like Z3 or another SMT solver) to check for certain properties (e.g., ensure that two rules don't produce conflicting outcomes for any possible input). Another idea is to integrate with a **business rules validator** in the style of DMN TCK (Technology Compatibility Kit) which checks if the rules cover all combinations, etc. These are advanced steps and might be beyond MVP, but the structured output positions us well to apply such tools later. For example, a set of ValidationRules could be turned into logical assertions and we could see if any assertion is unreachable or always true, indicating a redundant rule.

**Feedback Loop:** When the verification finds an issue, the system doesn’t just flag it – it also feeds it back into improvement:
- If a rule is low confidence because, say, the LLM was unsure, we might re-prompt the Extractor with a more specific query, or break the code into smaller pieces and try again.
- If a human corrects a rule, we capture that correction and potentially use it to fine-tune the model or at least as an additional few-shot example for similar cases. Over time, this makes the system smarter (learning from mistakes).
- We maintain a log of common errors (e.g., “LLM often misses rule when code uses GO TO to jump over a section”) and develop mitigations for those patterns in preprocessing or prompting.

**Benchmarking Accuracy:** As part of verification, we will measure the system’s precision/recall on known rules if any are available. For instance, if the client can provide a handful of known business rules (from documentation or SME knowledge), we check if each is present in the extracted set (recall) and if any extracted ones are not true (false positives). This helps quantify improvements as we tweak the pipeline.

Using an LLM as a judge is a novel but increasingly accepted practice to improve reliability of LLM outputs. Combined with traditional QA methods, this hybrid approach should significantly reduce hallucinations and errors. The end result of the Validation layer is that stakeholders can trust the extracted rule set as a correct reflection of the legacy code’s business logic. Furthermore, the verification artifacts (scores, rationale, test cases) serve as documentation evidence supporting the modernization effort – useful in audits or in convincing business users that the rules are accurate.

## Knowledge Graph and Glossary Integration

To bridge the gap between low-level code and high-level business understanding, we integrate a **Knowledge Graph (KG)** and business glossary into the rule extraction process. The aim is to link each extracted rule and its components (fields, values, conditions) to well-defined business concepts, ensuring consistency and aiding disambiguation. This also creates an enriched knowledge base that can be queried independently (for example, to ask “show me all rules related to customer eligibility”).

**Ontology and Glossary:** We will either use an existing enterprise ontology (if the organization has one) or build a simple glossary of key business terms, data entities, and their relationships. For instance, define entities like *Customer*, *Account*, *Order*, and link them to fields in code (e.g., `CUST-ID` in COBOL corresponds to the *Customer* entity’s identifier). We also include business domain definitions (e.g., what does “VIP status” mean). This glossary can start small (maybe extracted from schema names and known business terms) and grow over time as the LLM encounters new terms.

**Linking during Extraction:** When the Extractor agent or Formatter is processing a rule, it will attempt to annotate it with links to the glossary. For example, if a rule involves field `POLICY_TYPE`, the system will recognize this likely relates to the concept *Policy* (if such a concept is in the glossary). The rule object could then include a reference like `concepts: ["Policy"]`. We can accomplish this by having the Memory/KG agent provide suggestions. PydanticAI (or a custom tool) can be used to call a glossary lookup: for each identifier or literal in the rule, search the glossary for matches or synonyms. If found, attach the concept. If not found, that might indicate a new business term – we can optionally prompt the LLM to explain it or flag it for SME clarification. (For example, code might have abbreviations; LLM could guess or ask, and an SME might confirm "Oh, 'LTV' stands for Loan-to-Value ratio.").

**Maintaining Ontological Consistency:** By linking to a KG, we ensure that all rules use consistent terminology. If one program calls something “CustNbr” and another “CustomerID”, the glossary would map both to the concept *Customer Identifier*. We will prefer using business-friendly names in the rule descriptions where possible (the LLM can be guided to use glossary terms: e.g., if it sees `CSTMR_CAT = 'P'`, use “Preferred Customer” if the glossary says 'P' means Preferred). This prevents proliferation of inconsistent language in the extracted rules and will make the next modernization steps (like writing new code or documentation) much smoother.

**Knowledge Graph Implementation:** We can implement the KG using a graph database like Neo4j, as this allows storing nodes for **Entities**, **Fields**, and **Rules**, and relationships like *Rule uses Field*, *Field belongs to Entity*, *Entity is related to Entity*. Thoughtworks’ CodeConcise approach reports success in storing AST and rule relationships in a graph for querying. For example, we will store a node for each extracted rule (with its type and description), and link it to nodes representing the data elements it involves. We also add nodes for business concepts. If a rule’s condition is "account_balance < 0", we link the condition to the *Account* concept (because account_balance is an attribute of Account) and possibly to a *BusinessPolicy* concept like *OverdraftPolicy*. Later, an analyst could query the graph: “find all rules related to OverdraftPolicy” and get them. 

**LLM and KG Synergy:** The LLM can both contribute to and benefit from the knowledge graph:
- *Contribute:* As mentioned, the LLM can populate relationships by interpreting code. It might infer, for instance, that a constant “EMP_TYPE = 'X'” corresponds to *Contractor* vs *Employee* based on context, adding that info to the glossary node for EMP_TYPE. We will be cautious and usually confirm such inferences with an SME, but the LLM can draft it.
- *Benefit:* In turn, when the LLM is uncertain about a term, we can let it query the KG. This can be done by a tool invocation. E.g., if code uses field `MEMB_STAT_CD`, the LLM can call a tool `lookup("MEMB_STAT_CD")` which searches the KG. If the KG knows that `MEMB_STAT_CD` is “Member Status Code with possible values A=Active, I=Inactive, S=Suspended”, that info is returned to the LLM, which then can correctly interpret rules involving that field. This is essentially Retrieval-Augmented Generation (RAG), where the KG (plus possibly a vector store) is the retrieval mechanism. 

To facilitate this, we will index the glossary in a vector database as well, so semantic search can be done on terms. Neo4j now supports vector search in its graph DB, which we can leverage, or use an external vector store linked to the glossary text.

**Glossary Updates:** When a human SME clarifies a term or when we extract a new one from code comments (e.g., a comment says “/* X is the customer segment code */”), we update the central glossary. Over time, the knowledge graph becomes richer – not only do we have rules, but also context like *“Rule R1 deals with Customer of type Gold”*. Maintaining this is an investment that pays off in later phases, especially if the organization plans to build a knowledge portal or use this for training new developers.

**Ontological Consistency Example:** Suppose the legacy system uses a code “T” to mean “Trial account”. In one place, a COBOL paragraph might enforce “IF ACCT_CODE = 'T' THEN set TrialPeriod=90 days”. The rule extracted would say something about trial accounts. The glossary would have an entry `AccountType` with value 'T' = “Trial Account”. Our KG linking ensures the rule’s condition is tied to AccountType:Trial. If elsewhere another program checks `if account_type == 'TRIAL'` (maybe a newer module using full words), the KG lets us know that's actually the same concept. Thus we avoid listing essentially the same rule twice under different guises. We also detect if something might be misinterpreted – if the LLM guessed wrong on a term, consistency checks with KG might reveal it.

**Querying the Knowledge Graph:** With rules in a KG, interesting queries become possible:
- “List all rules that use concept X” (e.g., all rules related to *Interest Rate*).
- “What business terms does program Y touch?” (traverse from program node to its rules to concepts).
- “Are there any rules that involve both concept A and B?” (maybe find hidden interactions).
- These queries can be done via Cypher (if using Neo4j) or via an API. We might not implement a full query interface in the MVP, but designing the data model now enables these later.

**SME Q&A via KG:** A stretch goal is to allow a chatbot interface where users can ask questions about the legacy business logic in plain English and get answers derived from the extracted rules and knowledge graph (somewhat like the CodeConcise chatbot idea). For example, “Under what conditions is an account automatically closed?” The system would search rules for any outcome about closing account, then use the conditions to respond. With the context stored, an LLM could answer by citing the rule. Even if we don’t implement this fully, structuring the knowledge in this way makes such capabilities possible. It also offloads some reliance on the few SMEs who understand the old system, as the KG+LLM combo can answer certain queries similarly to an SME.

**Maintaining the KG:** We will implement routines to update the KG every time we run the extraction. If the code base updates or if we refine rules, we sync those changes to the graph. The KG becomes a living documentation of the system. We will also include the glossary entries and rule definitions in the final outputs (documentation) to ensure even without the graph DB, the knowledge is captured. For instance, embed the glossary as a section in the rule documentation.

In essence, the Knowledge Graph and glossary integration serve as the **semantic glue** between code and business language. It reduces misinterpretation (by providing definitions), promotes reuse of terms (everyone uses the same phrase for the same concept), and sets the stage for more advanced analysis and querying of the extracted rules. It’s an extra upfront effort but aligns with best practices seen in industry where LLM outputs are combined with knowledge bases for reliability. By doing this, we ensure the modernized rules aren’t just technically correct, but also make sense to business stakeholders in their own language.

## Required AS/400 Inputs

Although the user currently does not have the AS/400 artifacts on hand, it’s crucial to plan for **which inputs** from the legacy system we will need to successfully perform rule extraction. For an IBM AS/400 (IBM i) environment, the following sources should be gathered:

- **Source Code for All Business Programs:** This includes all RPG code (whether RPG III, RPG/400 or modern RPG IV / ILE RPG) and COBOL programs on the system. Typically, these reside in source physical files (e.g., QRPGSRC, QCBLSRC libraries). We will need the latest version of every program that contains business logic. If there are other languages in use (e.g., CL – Control Language scripts, or even Java on IBM i), those should be included as well because they might implement business rules or orchestrate processes.

- **Copybooks and Include Files:** In COBOL, any COPY statements referencing copybooks (or in RPG, /COPY or /INCLUDE source members) should be collected. These often contain data structure definitions or common logic that is essential for understanding the rules. Without copybooks, the code might not compile or make sense (missing context for fields). For RPG, this may include copybooks for data structures or calculation subroutines. Ensuring we have all includes allows the preprocessing stage to expand them so the LLM sees the full picture.

- **Database Schema Definitions:** The AS/400 likely uses DB2 for i (previously known as DB2/400) as the database. We need the definitions of physical and logical files (tables and indexes) – essentially the data schema. On IBM i, these could be in the form of DDS (Data Description Specifications) for physical files, or SQL DDL if the database was defined with SQL. Having the field names, types, and sometimes even field text descriptions is extremely useful. For example, knowing that field `CSTATR` is a 1-character field with text "Customer Status (A=Active, I=Inactive)" would let the LLM and our system recognize values and meaning. Schema definitions also tell us relationships (foreign keys if documented, or logical file join relationships), which can hint at rules spanning multiple files.

- **Screen and Report Definitions:** Many AS/400 applications have 5250 screens defined via DDS for display files (DSPF) or UIM files. We should obtain the source for any interactive screen that might contain validation logic or hints. Often, DDS for display files includes keywords for field validation (like `CHECK` or `VALUES` clauses) that enforce business rules at the UI level (e.g., valid values for a field). These are business rules too, and should be extracted. Similarly, report definitions (printer files) could contain conditional formatting that implies rules, though less critical. At least, screen field labels and prompt texts are very useful to decipher the meaning of fields in the code. For instance, if the screen label says "Enter Account Type (P=Personal, B=Business)", that maps the code’s `ACCT_TYPE` field to business meaning.

- **Job Control Scripts:** If the environment uses batch jobs orchestrated by CL programs or job scheduler entries, those can contain business logic sequencing (like “run Program A then B; if B fails, send alert”). We should gather CL program sources (.CLLE etc.). On mainframe we’d ask for JCL; on IBM i, CL is the analogous piece. They might not contain explicit business rules, but they show the flow of processes (which could be considered workflow rules). For modernization, understanding job flows is important. So, any **JCL (if COBOL is from mainframe)** or **CL** (for IBM i jobs) scripts involved in the system should be included.

- **Operational Logs or Decision Tables (if any):** In some systems, especially if they had quasi-rules-engine, there might be externalized tables or config files that store rules (e.g., a file of interest rates by tier). If such data-driven rules exist (like parameter files), those should be identified. The user may not know of these upfront, but during code analysis we might notice, for example, a table that seems to hold rules. We’d then need the data from those tables to fully capture the logic. It’s not typical for pure RPG/COBOL shops to have external rule tables (unless they used a product for it), but worth keeping in mind.

- **Existing Documentation (if available):** While not code, any existing functional specs, requirement docs, or even help text can be useful to verify rules. The user might not have this, but we should ask if any *user manuals* or *comments in code* exist that describe business logic. Comments in the source code are especially important and are already covered by including source, but we highlight that we won’t strip comments; we’ll use them.

In summary, **complete source code and definitions** are needed. If any piece is missing, the extraction might miss rules or produce wrong results. For example, missing a copybook that defines a condition value could cause the LLM to misinterpret a constant. Thus, the effort should include exporting all relevant source from the AS/400. IBM’s Rational Developer for i (RDi) or older tools like PDM can be used to mass-export source members. We might recommend using **IBM RDi** to connect to the IBM i and pull all source members into text files, which can then feed into our pipeline. This ensures proper encoding (EBCDIC to ASCII if needed) is handled.

**Tools for Extracting and Parsing Input:**
- *IBM Rational Developer for i (RDi):* An IDE that can easily browse and extract sources from the AS/400. It can also do some analysis like cross-referencing where fields are used, which could assist preprocessing (e.g., building a call graph or expanding copybooks).
- *IBM Application Discovery (ADDI):* Though primarily for mainframe, IBM may have similar analysis tools for IBM i. ADDI on z/OS provides a repository of code structure; if a similar tool exists for IBM i, it could be leveraged to get an initial map of the application.
- *Micro Focus COBOL Analyzer:* If the COBOL code is extensive, Micro Focus’s analyzer tools (as referenced in their docs) can parse COBOL and even extract some rules manually. For IBM i COBOL and RPG, Micro Focus might not cover RPG, but it covers COBOL well. We could use it to identify paragraphs that contain many conditions, etc., as starting points.
- *Fresche Solutions X-Analysis:* This is a modernization tool specifically for IBM i (RPG/COBOL) which builds an application model, including business rules. If available, it could give a quick inventory of rules or at least help parse RPG, which is notoriously tricky. While we are not integrating with a BRMS, using such a tool just to get parse trees or call graphs could accelerate development.
- *Open-Source Parsers:* There are ANTLR grammars for COBOL (for example, the Open Mainframe project or others have defined COBOL parsers). For RPG, open-source options are fewer, but some exist for RPG Free format. We may integrate an ANTLR-based parser in our preprocessing for syntax accuracy. This would ensure we can programmatically navigate the code (e.g., find all IF statements). However, given time constraints, leveraging existing commercial tools might be faster initially.
- *IBM i Data Export Tools:* IBM i allows outputting DDS to DDS source or generating SQL from DDS (`DSPFFD` command displays field descriptions, etc.). We might use system commands to dump schema info. For screens, `DSPFD` and `DSPF` commands can list field constraints.

By assembling all these inputs, we essentially recreate a read-only copy of the application’s source and metadata off the IBM i. This is our playground for analysis. The completeness of these inputs will directly influence the quality of the extracted rules. So, a key early task is to work with the AS/400 administrators to extract:
  - All relevant source members (perhaps thousands of them) into a repository.
  - All data definitions and possibly one-time dumps of any code tables.

We should communicate to the user that obtaining these is a prerequisite for the automated extraction – the AI can’t guess rules that aren’t in the provided source.

Lastly, as the system runs, we might discover we need more context (e.g., sample data to understand typical values). While not strictly required, having some production or test data examples could help the LLM infer certain business logic nuances (for example, if we see typical values of a field, it might clarify what that field represents). If accessible, logs or databases can be used in a read-only manner to supplement understanding, but this is optional.

To recap, **the required inputs are: full source code (RPG, COBOL, CL, etc.), copybooks/includes, database (DB2) schema definitions, screen/display file definitions, and any existing documentation or code-related assets**. Equipped with those, our agent can thoroughly analyze the legacy system. Notably, these inputs align with what traditional analyses require as well – you essentially need everything that went into building the system to successfully reverse-engineer it (with the code itself being the ultimate source of truth).

## Tooling and Platform Recommendations

To maximize development speed and ensure maintainability, we will use a suite of modern tools and frameworks. These tools cover code parsing, LLM orchestration, data storage, and version control, all chosen to be platform-agnostic and accelerators for our use case.

**Development/Analysis Tools:**
- **IDE and Parser Utilities:** Using an IDE like *IBM RDi* or *Visual Studio Code* with IBM i extensions will simplify browsing the legacy code during development of the pipeline. RDi provides features like RPG and COBOL syntax highlighting, outline views, and search, which help manual understanding and verification. **Continuous Improvement:** Because this system will likely evolve, we’ll set up a Git repository with CI (Continuous Integration) that can run automated tests of the extraction on a small sample of code whenever we make changes, to catch regressions. This is more software-engineering process, but it's noteworthy.

In summary, the toolchain is: **Python + PydanticAI** at the core, with optional use of **LangChain/CrewAI/DSPy** for orchestration enhancements, **Graph database (Neo4j)** for knowledge graph, **Vector store** for semantic lookup, **IBM/Micro Focus tools** for parsing assistance, and **standard devops tools** (Git, Docker, pytest) to manage development and deployment. All components are chosen for compatibility with cloud or on-prem, and many are open-source or easily licensable, aligning with the project needs. This setup ensures we can move fast (by using pre-built frameworks) but also produce robust, traceable outputs (with version control and validation tools).

## Output Format and Documentation Standards

The end deliverables of this extraction process are as much about **documentation** as they are about raw data. We will produce outputs in structured formats (JSON/YAML) for machine consumption, and also ensure there are human-friendly documentation artifacts explaining the rules in context. Clarity and traceability are top priorities for these outputs.

**Machine-Readable Rule Output:** The primary output will be a set of JSON and/or YAML files containing the extracted rules, following the schema described earlier. We propose to organize these files logically, for example:
- By application module or domain: e.g., `rules_customer.yaml` might contain all rules related to customer management, `rules_billing.yaml` for billing, etc. or 
- By program: e.g., `PROGRAM_X_rules.json` containing all rules extracted from program X.
Alternatively, we could have one large file, but splitting them improves navigability and version control diffs. We will likely output both JSON and YAML (since YAML is easier to read/edit manually, while JSON might be easier for certain tools to parse). These can be generated in one go (YAML is basically a formatted view of the JSON data structure in Python).

Each rule entry in the JSON/YAML includes all fields as per schema. We ensure that lists and nested structures are properly indented/serialized. If using YAML, we’ll include comments or tags if helpful (YAML supports comments, which we could use to embed the original code snippet as a comment above the rule, providing immediate context in the YAML file itself for a reader).

We also plan to include a **traceability tag** in each rule object linking to the source location. For example, a rule might have `"source_id": "COBOL:ORDPROC:PARA123"` which could correspond to a bookmark in the code. If we set up a convention (like `program:label` or line range), a developer could quickly find the origin. In documentation, we can hyperlink this if we have code listings available, but at least it’s recorded.

**Documentation for Humans:** In addition to raw YAML/JSON, we will create a documentation deliverable, likely in Markdown or HTML, that presents the business rules in a friendly way:
- **Contextual Grouping:** Organize rules by business domain or user functionality. This might be a hierarchy of sections. For example:
  - *Customer Account Rules* (with sub-sections for validations, calculations, etc. related to Customer Accounts),
  - *Loan Processing Rules*, 
  - *Insurance Policy Rules*, etc., depending on the system.
  If the domain breakdown isn’t obvious, we might organize by the source program (since often programs are named by domain).
- **Rule Listings:** Under each section, list the rules (perhaps in a table or bullet list form). Each rule listing will include the rule’s description in natural language, the type (validation/decision/etc.), and possibly key conditions/outcomes in brief. Next to it, we provide the source reference and confidence. For example: “**Rule:** Applicants under 18 are not eligible for account creation. **Type:** Validation. **Source:** OPENACCT RPG (line 120). **Confidence:** High (0.98).”
- **Traceability Links:** If possible, we’ll cross-reference to the JSON/YAML or code. E.g., the documentation might have a hyperlink that when clicked, shows the original code snippet (we can generate an HTML snippet for each referenced code segment). This might be facilitated by tools like Sphinx or MkDocs where we can embed code with anchors. If not in initial scope, we will at least provide line numbers and file names as text so someone can manually find it.
- **Comments and Annotations:** We will carry over any relevant code comments into the documentation. If a code comment said “*// Biz rule: if cust age <18, reject*”, we will include that as a note in the documentation to reinforce that this was an intended rule. We may also add explanatory notes (via the LLM or SMEs) for complicated rules. For example, if a rule is very technical, we might add: “*(This rule calculates the amortization schedule. It’s complex because of X...)*” – giving context beyond the raw logic.

**Versioned Documentation:** Each iteration of rule extraction can produce a versioned document (e.g., *BusinessRules_v1.0.docx* or markdown in a Git repo). We plan to maintain a change log of the rules. E.g., if in the next extraction some rules changed (due to code changes or improved extraction), the documentation will highlight new/modified rules, making it easy for reviewers to see what’s updated.

**Templates and Standards:** We will follow any existing documentation standards the organization has (if there’s a template for business rules, we can adopt it). If none, we’ll create a clean template with the company branding (if desired) and a clear introduction explaining what the document is (basically a modern “requirements document” reverse-engineered). Each rule will have:
- A unique identifier (as given in JSON, like RULE-001, etc.) so it can be referred to unambiguously in discussions.
- A one-line summary (like the description field).
- A detailed description if necessary (for multi-condition rules, we might break it down).
- The origin (legacy system reference).
- Any notes (like “Note: This rule might be overridden by parameter X in production” if we discovered such a thing).

We aim for the documentation to be understandable by business analysts, not just developers. This means avoiding or explaining technical terms. For instance, instead of writing “If CUST_STAT = 'I' then...” we’ll write “If a customer’s status is 'Inactive' then ...” The glossary integration helps here, ensuring we replace code jargon with business terms as much as possible in the documentation.

**Example Documentation Excerpt:**  
*Section: Account Closure Rules*  
- **Rule AC-1:** *Accounts that have been inactive for over 12 months are automatically closed.* (Decision Rule) If an account’s last activity date is more than 1 year in the past, then the system will mark the account as closed on the next batch cycle. **Source:** COBOL program `ACCTCLSE`, lines 220-250. **Confidence:** 0.9. *(This rule helps reduce maintenance of long-dormant accounts.)*  

- **Rule AC-2:** *VIP accounts are exempt from automatic closure.* (Override Rule) If the account type is VIP, the inactivity closure rule does not apply. **Source:** COBOL `ACCTCLSE`, lines 255-260. **Confidence:** 0.95.

*(The above are illustrative – in actual output we’d cite the real sources.)*

Notice we included a rationale note for AC-1. We might get these rationales from code comments or SME input. If not, we keep it purely descriptive.

**Traceability Matrix:** We can also produce a matrix mapping each program to the rules found in it, and each rule to the programs that implement it. This is helpful if the organization wants to ensure every piece of code is accounted for by some rule or technical function. This matrix could be an appendix in the documentation or a separate CSV file. It’s basically a two-way mapping: Program -> Rules, and Rule -> Program(s). Because our data is structured, generating such a matrix is straightforward.

**Adherence to Standards:** If the output needs to adhere to something like ISO documentation standards or IEEE for requirements, we can format accordingly (with shall statements etc.). But since this is internal modernization, a simpler format is likely fine. The key is clarity and completeness.

**YAML/JSON format details:** We will document the schema (perhaps in a README or at top of the YAML file as comments). For example, in the YAML file we might put:
```yaml
# Schema: ValidationRule { id: str, description: str, condition: str, action: {level: str, message: str}, source_reference: {program: str, lines: str}, confidence: float, ... }
```
This way anyone reading the YAML knows what each field means. We’ll also ensure uniform formatting (indentation etc.), and no overly long lines (for readability).

**No Secrets in Output:** Since this might be shared widely, we’ll be careful that the output doesn’t inadvertently leak something sensitive (like literal passwords or keys, which sometimes are in code). Typically business rules don’t involve secrets, but we’ll scan the output for any such thing just in case.

In summary, the output will consist of:
- **Structured rule files (JSON/YAML)** – the authoritative machine-readable representation.
- **Human-oriented documentation (Markdown/HTML/Word)** – organized explanation of rules with trace links and possibly code excerpts.
- **Supporting files** like glossary, trace matrices, etc., as needed to provide full context.

This dual output meets the needs of both software agents (which can directly ingest JSON/YAML) and stakeholders/humans (who need to understand and validate the rules). By following these documentation standards, we ensure the extracted knowledge is accessible and actionable, not just a dump of data. It essentially becomes a living requirements document for the legacy system, bridging to the future system development.

## Implementation Roadmap and Challenges

Implementing this solution will be done in phases to deliver value quickly while managing risks and complexity. Below is a proposed roadmap from MVP to a robust production system, along with anticipated challenges and our mitigation strategies:

**Phase 1: MVP – Core Extraction on Sample Code (Weeks 1-4)**  
*Goal:* Deliver a basic working prototype that can extract rules from a small subset of code end-to-end.  
- Set up the environment and tools: Python project with PydanticAI, access to GPT-4/Claude API, etc. Extract a handful of representative programs from AS/400 (or use sample code if actual not yet available).  
- Implement the basic pipeline: read code, prompt LLM, get rules, output JSON. Initially, do this in a simplified way (maybe one agent that does it all in a prompt, just to see results).  
- Define initial Pydantic models for one rule type (say, ValidationRule and DecisionRule to start).  
- Quick evaluation: Have an SME review the output for those sample programs. This will give feedback on accuracy and any obvious misses.  
- **Fastest time to market** focus: At this stage, we favor manual or simple approaches if needed. For example, we might manually break the code into chunks for the LLM rather than implementing a parser right away. The aim is to demonstrate feasibility and get early buy-in.  
- *Deliverable:* A demo of the agent extracting a few rules correctly, outputs shown in JSON/YAML, and a short report of findings.

**Phase 2: Expand Coverage & Rigor (Weeks 5-8)**  
*Goal:* Scale up to handle the entire codebase and introduce the multi-agent and validation components for better accuracy.  
- Onboard the full set of legacy code into the pipeline. This might involve writing scripts to load all source members and iterate through them. Ensure copybooks are inlined (develop or integrate a lightweight preprocessor for COBOL and RPG).  
- Implement the **Preprocessing** module properly (maybe using a COBOL parser library and a rudimentary RPG parser). This will help identify code blocks for the LLM and filter out non-rule code.  
- Integrate the **multi-agent workflow**: Split the logic into Extractor and Formatter at least. Possibly also Validator as a separate step (could still be sequential in one script, but logically separated).  
- Add the **validation layer**: after extraction, run the LLM judge prompt on each rule. Start collecting confidence metrics. Also compute a few heuristic checks (like duplicate rule detection or easy contradictions).  
- Begin building the **knowledge glossary**: perhaps manually seed it with known terms. As we run on more programs, start populating it. If time, integrate Neo4j and store some relationships. If not, at least maintain a Python dictionary of term mappings for the LLM to use.  
- Fine-tune prompt strategies: With more code, we’ll refine our few-shot examples and instructions. Possibly implement caching of LLM responses for identical prompts (to save cost/time if re-running multiple times).  
- **Challenge:** *Hallucinations* – The LLM might sometimes invent a rule that seems plausible but isn’t in code. We mitigate this in Phase 2 by the validation agent, and by grounding the prompt with “Answer only based on code. If unsure, say so.” We also keep track of any hallucinated content and adjust prompts (e.g., reduce creativity by adjusting temperature down on the model).  
- *Deliverable:* A full run on the entire system’s code that produces a first complete draft of all rules. There will be errors, but we treat this as a baseline. Also deliver a list of low-confidence areas that need review.

**Phase 3: Human Review & Iterative Improvement (Weeks 9-12)**  
*Goal:* Involve SMEs to verify and correct rules, and feed those improvements back to the system.  
- Set up review sessions where SMEs/developers go through the extracted rules (using the dashboard or docs). Collect their feedback (e.g., “Rule X is wrong because the code actually means something else” or “We have a missing rule about Y that the code implies indirectly”).  
- Update the extraction logic to address the feedback: This could mean adding new patterns for the LLM to catch (maybe we add a custom heuristic: “anytime you see indicator variable on, that implies rule XYZ”). Or it could mean adjusting the schema (maybe we realize a new metadata field is needed to capture nuance).  
- Potentially perform a **fine-tuning** of the LLM if we have accumulated enough corrected examples. If not full fine-tune, we at least add these as few-shot exemplars for future runs.  
- Expand the **Knowledge Graph** with real data: by now, we have many rules and their linked terms. Load them into Neo4j and run some consistency queries. For example, see if any concept has two different rules that conflict. If found, highlight to SMEs. Sometimes conflicts are legitimate (one overrides another under conditions), but it’s worth checking.  
- **Challenge:** *Legacy code obfuscation* – Some business rules might be “spread out” or not written in a straightforward manner. We might find that our approach missed rules that aren’t explicitly in an IF statement but rather emerge from a series of computations. To mitigate this, at this phase we might incorporate **program slicing** techniques. For instance, take each output field and trace back what influences it (like the SoftRules approach in research). This could uncover rules that the straightforward pass missed. Implementing slicing for RPG/COBOL is complex, but we can use a simplified approach: look at all conditions that lead to writes to key fields (like account status being set to "Closed").  
- **Challenge:** *Implicit knowledge not in code* – There might be business rules that are not coded but assumed (e.g., “we never allow negative price” might be ensured by data entry UI or just policy). Our system wouldn’t get those since they aren’t in code. We document these as out-of-scope, but we will list any such rules mentioned by SMEs during review, so the new system’s developers are aware of them.  
- *Deliverable:* Revised rule set (JSON/YAML) with much higher accuracy, and a vetted documentation. Essentially, this is the version to be used for actual modernization input. Possibly a metric like “we achieved 95% recall of known rules and 98% precision” – meaning most rules are correct and we got almost all of them.

**Phase 4: Integration into Modernization Pipeline (Weeks 13-16)**  
*Goal:* Prepare the extracted rules for consumption by development teams or automated code generation tools.  
- Ensure the output format is compatible with the next stage (if the next stage is software agents that generate new code, confirm they can read our JSON or we adapt to their expected input format). For example, if using a tool that prefers DMN, we might here add a converter from our JSON to DMN XML for decision rules. Or if developers want them in Excel, perhaps output an Excel sheet of rules as well.  
- Lock down the versioning: baseline the rules corresponding to the current production system. If code changes in the legacy, plan how to update rules (the extraction can be rerun incrementally on changed programs). Possibly set up a process with the customer that whenever the legacy code updates, they run our extraction again to keep rules in sync – at least until they fully move off the legacy.  
- **Performance Tuning:** If the codebase is huge (millions of lines), the approach might be slow or costly with the LLM. In this phase, we profile it. Perhaps we identify hotspots: maybe one program with 10k lines takes multiple LLM calls. We could attempt to optimize by splitting tasks or using a simpler model for trivial parts. Also consider concurrency – run multiple LLM calls in parallel to utilize the available API limits fully. If cost is an issue, perhaps use GPT-4 for complex programs and GPT-3.5 for simpler ones, etc. The orchestrator can have logic for that (like, measure complexity of code by number of IF statements; choose model accordingly).  
- **Robustness:** Add more safety checks, e.g., if the LLM fails or times out for a program, have a retry mechanism or fall-back (maybe break the program into smaller parts and try again). Ensure the system can run to completion without manual intervention.  
- **Documentation for Developers:** Work with the modernization team who will use these rules (if that’s a separate team) to ensure they understand how to read them. Possibly do a knowledge transfer session using the documentation we created. We might produce a mapping of rules to proposed new system’s modules or user stories, effectively translating the rules into new-system requirements.

- **Challenge:** *Model and Integration Risks:* If using external APIs for LLM (like OpenAI), consider what happens if network issues or API changes occur. By Phase 4, we might mitigate this by having an offline model option ready (maybe the fine-tuned local model), even if it’s a backup. Also, consider data privacy – ensure no sensitive data was sent to external APIs without agreement. If that’s a concern, pivot to the on-prem model by this phase.
- **Challenge:** *Evolution of Code during Project:* If the legacy system is still actively maintained, new changes might introduce new rules. We plan for this by making the pipeline re-runnable and possibly integrating with version control as mentioned. That way, if code changes, we can diff the rule outputs and catch new/modified rules quickly (which can then be merged into the documentation).

**Phase 5: Ongoing Monitoring and Multi-Agent Collaboration (beyond 16 weeks)**  
This covers further enhancements:
- Possibly set up a continuous integration on the legacy code: whenever a change is promoted, run the extractor and alert if any business rule changed unexpectedly (this can act as a guard against accidental rule changes).
- Develop the SME chatbot using the knowledge graph, to field questions during the redevelopment process (this uses the infrastructure we built, just an application of it).
- Use the extracted rules to generate unit tests for the new system. For instance, each rule can become one or more test cases in the target platform to ensure the new implementation respects it. We could automate generating these test stubs.
- Explore integration with a BRMS: even though not initially planned, if later they decide to put rules in a rule engine, our output can be the basis for writing those rules formally. We might provide a mapping from our JSON to Drools or IBM ODM format as a deliverable if needed.

Throughout these phases, we maintain a focus on **fast implementability**. That means we will use existing tools and frameworks rather than reinventing wheels (e.g., we lean on PydanticAI for structure, and on LLM’s own reasoning instead of writing complex code analyzers from scratch, unless necessary). We also keep the output **agent-ready** – structured and clean.

To manage **hallucinations**, our layered approach (with validation and human review) is the main mitigation. We also use conservative prompting. In testing, if we find certain phrasings cause the model to guess beyond the code, we’ll adjust them. Experience shows that instructing the model to quote exact code for each rule and then interpret helps ensure it sticks to facts. We might do that: have the model output a snippet of code and the rule for that snippet, iteratively.

For **legacy obfuscation** (spaghetti code, heavy use of GOTO, etc.), we mitigate by preprocessing (e.g., converting GO TO flows into structured form via an algorithm or tool). We saw in one reference that converting COBOL flow into a structogram helped identify conditions. We could implement a simple flow analyzer for such cases. This is complex, but not all programs will require it – we’ll prioritize if a critical module is too unstructured for the LLM, then invest effort there.

**Implicit rule ambiguity** is largely mitigated by involving SMEs and using the knowledge graph to piece things together. If something is ambiguous, we won’t have the AI guess quietly; rather it can flag “unclear” and we bring an SME or explicitly document the assumption.

In conclusion, this roadmap ensures a **phased delivery**: an early MVP to show value, followed by scaling up and tightening accuracy, and culminating in a reliable, integrated rule extraction system that feeds the modernization effort. Each phase tackles different challenges, with risk mitigations in place – from using multi-agent checks to human oversight and knowledge integration. By the end, we expect to have a comprehensive set of machine-consumable business rules extracted from the AS/400, thoroughly validated, and ready to drive the next stage of the legacy system modernization.
