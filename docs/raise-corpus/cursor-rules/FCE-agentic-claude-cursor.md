# Factores Críticos de Éxito en Prompts Agénticos (Cursor IDE + Claude Sonnet 3.5)

Al implementar prompts **agénticos** (prompts que habilitan al modelo para actuar como un agente con herramientas) en Cursor IDE con **Claude 3.5 Sonnet (241022)**, es clave considerar varios factores para el éxito:

- **Selección del modelo adecuado:** No todos los modelos soportan el uso de herramientas MCP. Claude 3.5 Sonnet es reconocido por su sólida integración con herramientas (“**tooling**”) en Cursor ([Sonnet 3.5 vs o3 mini - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/sonnet-3-5-vs-o3-mini/51054#:~:text=Your%20findings%20aligns%20with%20benchmark,Benchmarks)). De hecho, las herramientas MCP solo funcionan en el modo Agent de Cursor y pueden no funcionar con modelos incompatibles ([Cursor – Model Context Protocol](https://docs.cursor.com/context/model-context-protocol#:~:text=protocol%20that%20allows%20you%20to,to%20agentic%20LLMs%20in%20Cursor)). Por ello, usar Sonnet 3.5 (que tiene el mejor soporte de herramientas según usuarios ([Sonnet 3.5 vs o3 mini - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/sonnet-3-5-vs-o3-mini/51054#:~:text=Your%20findings%20aligns%20with%20benchmark,Benchmarks))) es fundamental.

- **Prompt de sistema bien diseñado:** Un prompt de sistema detallado y claro sienta las bases del comportamiento del agente. Es recomendable definir explícitamente el rol del modelo, su contexto de trabajo (por ejemplo, indicar que *opera en Cursor IDE como asistente de programación*), y las reglas que debe seguir. Un prompt específico produce salidas más precisas ([How to Use Cursor AI with Claude 3.5 Sonnet (Step-by-Step Guide)](https://apidog.com/blog/use-cursor-ai-with-claude-3-5-sonnet/#:~:text=1,frequently%20back%20up%20your%20work)). Incluir instrucciones sobre cómo manejar código, errores y límites ayuda a alinear el modelo con las expectativas del usuario.

- **Contexto y conocimiento del proyecto:** Proporcionar al modelo contexto del código y dominio es crítico. Cursor ofrece indexado del codebase, lo que genera embeddings de cada archivo para mejorar la precisión de las respuestas relacionadas al código ([Cursor – Codebase Indexing](https://docs.cursor.com/context/codebase-indexing#:~:text=Index%20your%20Codebase)). Asegurarse de indexar el repositorio permite al agente buscar información relevante del proyecto cuando responde. Asimismo, usar **Rules for AI** específicas del proyecto (.cursor/rules) o un `.cursorrules` bien configurado brinda al modelo lineamientos particulares del código base (nombres de clases, estilos de código, convenciones) para mantener coherencia en sus respuestas.

- **Integración de herramientas MCP adecuada:** Disponer y configurar correctamente las **herramientas MCP** necesarias es otro factor de éxito. El **Model Context Protocol (MCP)** permite extender Cursor con herramientas personalizadas que el agente puede usar ([Cursor – Model Context Protocol](https://docs.cursor.com/context/model-context-protocol#:~:text=The%20Model%20Context%20Protocol%20,to%20agentic%20LLMs%20in%20Cursor)). Debe añadirse cada herramienta MCP en la configuración de Cursor y describirla claramente; el agente las usará automáticamente si las considera relevantes, o se le puede indicar que las use por nombre/descripción ([Cursor – Model Context Protocol](https://docs.cursor.com/context/model-context-protocol#:~:text=Using%20MCP%20Tools%20in%20Composer)). Si el prompt de sistema incluye descripciones claras de cada herramienta y en qué casos emplearlas, el modelo podrá elegirlas de forma más efectiva. Por ejemplo, una herramienta de búsqueda documental, una de ejecución de código, etc., deben indicarse en el contexto para que el agente las aproveche cuando haga falta.

- **Colaboración iterativa con el usuario:** Finalmente, trate el flujo de trabajo como una **pareja de programación**. Claude Sonnet en Cursor suele dar buenos primeros resultados pero puede desviarse en correcciones si el prompt no se refina ([I've been using cursor since it launched, sticking almost exclusively ...](https://news.ycombinator.com/item?id=42893868#:~:text=I%27ve%20been%20using%20cursor%20since,misses%20the%20plot%20during%20corrections)). Es importante que el usuario esté preparado para iterar: proporcionar feedback, ajustar instrucciones o dividir tareas complejas en pasos más pequeños. Un agente bien instruido evitará preguntar innecesariamente al usuario (prefiriendo buscar la respuesta con herramientas) ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=Bias%20towards%20not%20asking%20the,search_and_reading)), pero también sabrá cuándo detenerse y pedir aclaraciones (por ejemplo, tras varios intentos fallidos de corrección de código, según las reglas definidas ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=editing%20before%20editing%20it))). Establecer este equilibrio en el prompt (no interrumpir al usuario por información que puede obtener solo, pero sí consultar cuando se requiera) mejora la eficacia y la experiencia.

# Mejores Prácticas en el Diseño de Prompts Agénticos

Al diseñar prompts agénticos optimizados para Cursor IDE, conviene seguir **mejores prácticas** que han demostrado mejorar la calidad de las respuestas y la interacción con las herramientas MCP, garantizando una buena experiencia de usuario:

- **Definir claramente el rol y contexto del agente:** Es útil iniciar el prompt de sistema aclarando que el asistente es un *AI coding assistant* en modo agente dentro de Cursor. Por ejemplo, el prompt oficial de Cursor Agent inicia con una frase que lo establece como *“un poderoso asistente de codificación agéntico, impulsado por Claude 3.5 Sonnet, operando exclusivamente en Cursor IDE”* ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=You%20are%20a%20powerful%20agentic,Cursor%2C%20the%20world%27s%20best%20IDE)). Esto sitúa al modelo en su contexto de par-programming y le deja saber que debe actuar como tal (respetando el entorno del IDE, archivos abiertos, etc.).

- **Establecer reglas explícitas para uso de herramientas:** Delimitar cómo y cuándo el agente debe invocar herramientas MCP es esencial. Algunas reglas recomendadas:
  - *Seguir el esquema de llamadas a herramientas al pie de la letra*, proporcionando todos los parámetros requeridos ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=1,to%20provide%20all%20necessary%20parameters)).
  - *No mencionar el nombre de la herramienta al usuario* en la respuesta. En lugar de decir *“usaré la herramienta X”*, simplemente indicar la acción (e.g. *“voy a buscar documentación…”*) ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=2,that%20are%20not%20explicitly%20provided)).
  - *Llamar a herramientas solo cuando sea necesario*: Si la pregunta puede responderse directamente con conocimiento interno, evitar pasos extra ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=edit%20your%20file%27)).
  - *Explicar brevemente por qué se usa una herramienta antes de llamarla*, para mantener informado al usuario de la intención (ejemplo: *“No estoy seguro de la respuesta, buscaré en la documentación...”*) ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=you%20already%20know%20the%20answer%2C,just%20respond%20without%20calling%20tools)).
  - Estas directrices, inspiradas en prompts agénticos de Cursor, ayudan a que la interacción con MCP sea fluida y transparente, sin exponer detalles internos ni hacer llamadas superfluas.

- **Optimizar la generación de código dentro del flujo del IDE:** Dado que Cursor permite que el agente edite archivos directamente, el prompt debe instruir al modelo sobre **cómo manejar código**:
  - *No volcar bloques de código directamente en el chat a menos que se solicite*. En su lugar, usar las herramientas de edición de código para aplicar cambios en los archivos ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=,this%2C%20follow%20these%20instructions%20carefully)). Esto asegura que el IDE refleje inmediatamente las modificaciones propuestas.
  - *Hacer solo una edición por turno* para mantener control de los cambios ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=,this%2C%20follow%20these%20instructions%20carefully)).
  - *Asegurar que cualquier código generado sea ejecutable inmediatamente*: añadir todos los imports, dependencias y configuraciones necesarias ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=1,required%20to%20run%20the%20code)). Por ejemplo, si el agente agrega una nueva funcionalidad que requiere una librería, debería también añadirla al `requirements.txt` o `package.json` según corresponda, en lugar de asumir que el usuario lo hará después.
  - *Leer el contenido relevante antes de editar*: el prompt debe recordar al agente que consulte el archivo (o sección) actual para entender el contexto antes de insertar o modificar código ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=5,you%27re%20editing%20before%20editing%20it)), evitando así contradicciones o duplicación.
  - *Manejar errores de compilación/linter de forma inteligente*: si tras una edición surgen errores sencillos, intentar corregirlos (hasta un límite de ~3 intentos) y si persisten, solicitar orientación al usuario ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=editing%20before%20editing%20it)). Esto previene bucles infinitos de corrección automática que frustrarían al usuario.
  - *No generar contenido irrelevante o excesivo*: Por ejemplo, **no producir hashes largos ni binarios** en las respuestas, ya que no aportan valor y consumen recursos ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=UI%2C%20imbued%20with%20best%20UX,practices)). El prompt de sistema original enfatiza evitar salidas no textuales inútiles.

- **Mantener una experiencia de usuario óptima:** El asistente debe comunicarse de forma comprensible y útil. Algunas prácticas incluyen:
  - *Formatear las respuestas de manera clara*, usando listados o pasos cuando corresponda, para que el usuario siga fácilmente la solución (esto se puede indicar en el prompt o lograrse con ejemplos).
  - *Confirmar entendimiento en tareas complejas*: aunque el modelo debe preferir buscar información por sí mismo, si el requerimiento es ambiguo podría parafrasear la solicitud antes de proceder, asegurando que interpretó bien (esto puede mencionarse en las instrucciones del prompt como parte de la interacción ideal).
  - *Ser consistente en el estilo*: si el proyecto tiene un estilo particular (por ejemplo, mensajes de commit en cierto formato, comentarios en un idioma específico), es útil incorporarlo en las reglas. Cursor permite reglas por patrón de archivo, así el agente puede adaptar su estilo al contexto automáticamente ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Example%20use%20cases%3A)).
  - *Aprovechar las herramientas cuando aporten valor*: Por ejemplo, si existe una herramienta MCP de documentación o pruebas, usarla en lugar de adivinar, mejorará la confianza de la respuesta. Indicar en el prompt que el asistente debe apoyarse en dichas herramientas (sin abusar) orienta al modelo a entregar respuestas más fundamentadas.

En resumen, un prompt agéntico efectivo en Cursor es **detallado pero conciso**, delimita claramente el comportamiento del asistente, y lo guía en el uso inteligente de herramientas y en la producción de código de alta calidad, siempre centrado en ayudar al usuario de la forma más directa y segura posible.

# Reglas de Cursor para Monorepos Multi-tecnología

Trabajar con un **monorepo** que contenga múltiples tecnologías (por ejemplo, varios lenguajes y plataformas en un mismo repositorio de código) presenta retos especiales. Cursor IDE ofrece la funcionalidad de **Rules for AI** que resulta muy útil para gestionar este tipo de proyectos. A continuación se analizan estrategias de organización, control de dependencias, integración y coherencia, apoyándose en **Cursor Rules**:

- **Organización por carpetas y reglas específicas:** Una estrategia recomendada es estructurar el monorepo en subdirectorios claros por tecnología o módulo (ej.: `frontend/`, `backend/`, `mobile/`, etc.), y luego crear **reglas de Cursor** específicas para cada sección. Con la nueva característica de *Project Rules*, podemos tener múltiples archivos de reglas en `.cursor/rules/` cada uno con un glob que coincida con ciertos directorios o patrones de archivo ([Using the new Project Rules for Monorepos - How To - Cursor - Community Forum](https://forum.cursor.com/t/using-the-new-project-rules-for-monorepos/47302#:~:text=My%20team%20is%20building%20a,nested%20file%20for%20each%20folder)). Por ejemplo, una regla para `frontend/**` con descripción “Reglas para el frontend React”, otra para `backend/python/**` con “Reglas para API Python”, etc. Esto permite que el agente aplique automáticamente las instrucciones adecuadas según el contexto en el que esté trabajando: el Composer de Cursor selecciona las reglas cuyo patrón coincida con el archivo abierto y cuya descripción encaje con la situación ([Using the new Project Rules for Monorepos - How To - Cursor - Community Forum](https://forum.cursor.com/t/using-the-new-project-rules-for-monorepos/47302#:~:text=This%20really%20depends%20on%20your,project%20specifically)). Así se consigue modularidad en las directrices, tal como se modulariza el código.

- **Control de dependencias por módulo:** En un monorepo multi-tech, cada parte suele tener su propio manejo de dependencias (p. ej., npm/ Yarn para JavaScript, pip/requirements.txt para Python, etc.). Es importante que tanto los desarrolladores humanos como el asistente mantengan separadas estas dependencias. Desde el punto de vista de Cursor Rules, se puede:
  - Incluir en cada regla de carpeta notas sobre dónde definir dependencias nuevas. Por ejemplo: *“Si añades una librería Python, actualiza el `requirements.txt` en la raíz del módulo Python”*. De hecho, en las instrucciones de agente ya vimos que se recomienda crear/actualizar el archivo de dependencias cuando se crea código nuevo ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=1,required%20to%20run%20the%20code)).
  - Asegurarse de que el agente elija versiones compatibles: la prompt puede instruir (quizá en una sección global) que si va a usar una librería externa, use la versión que ya esté en los archivos de dependencias del proyecto; y si no existe, tome la última conocida compatible ([Cursor Agent mode System Prompt | 宝玉的分享](https://baoyu.io/blog/cursor-agent-system-prompt#:~:text=1,ask%20the%20USER%20for%20permission)). Esto evita conflictos de versiones.
  - Mantener **aislamiento**: Las reglas deben dejar claro qué partes del repositorio corresponden a qué ecosistema, para que el agente no intente, por ejemplo, mezclar sintaxis o paquetes de un lenguaje en otro. Un comentario en la rule del backend podría indicar: “Este es un servicio Python. No mezclar código Node.js aquí.”

- **Integración entre módulos:** La ventaja de un monorepo es facilitar cambios coordinados entre componentes que evolucionan juntos ([continuous integration - Multi-language Monorepo - DevOps Stack Exchange](https://devops.stackexchange.com/questions/9528/multi-language-monorepo#:~:text=Realistically%2C%20either%20path%20has%20tradeoffs%2C,do%20I%20make%20the%20split)) ([continuous integration - Multi-language Monorepo - DevOps Stack Exchange](https://devops.stackexchange.com/questions/9528/multi-language-monorepo#:~:text=If%20you%20have%20multiple%20products,far%20as%20I%20can%20tell)). Sin embargo, hay que gestionar cuidadosamente dichas integraciones:
  - **Interfaces bien definidas:** Documentar (incluso dentro de un Cursor Rule global o en un archivo de arquitectura referenciado con `@file`) las interfaces entre módulos – por ejemplo, qué API REST expone el backend y cómo el frontend la consume. Esto sirve tanto a desarrolladores como al agente para entender el contexto común. Con Cursor Rules, puedes adjuntar archivos de documentación como contexto automático cuando cierto módulo esté involucrado ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=files%20are%20referenced%20,when%20the%20rule%20is%20applied)).
  - **Pruebas integradas:** Si el monorepo incluye tests de integración que cubren múltiples módulos, informar al agente de su existencia. Una regla podría indicarle: *“Cuando modifiques el módulo X, considera también actualizar/ejecutar las pruebas en `/tests/integration/`”*. A nivel de pipeline CI, conviene ejecutar solo las pruebas relevantes a los cambios (p. ej., usando paths) para no molestar a desarrolladores con tests irrelevantes ([continuous integration - Multi-language Monorepo - DevOps Stack Exchange](https://devops.stackexchange.com/questions/9528/multi-language-monorepo#:~:text=,sure%20they%27re%20enabled%20when%20needed)) – esto es algo externo al agente, pero que impacta la experiencia de desarrollo.
  - **Sincronización de cambios:** El agente (y los devs) deben entender que ciertos cambios implican otros. Por ejemplo, modificar la estructura de datos en el backend requerirá cambiar la forma en que el frontend maneja esos datos. En la regla del backend, se puede anotar: *“Cualquier cambio en los endpoints o modelos debe ser comunicado al frontend X para ajustes correspondientes.”* De esta forma, si el agente está trabajando en backend, podría avisar en su respuesta que también habría que ajustar el frontend, manteniendo coherencia global.

- **Coherencia en la estructura y estilo del código:** Con múltiples lenguajes, existe el riesgo de que cada parte del monorepo diverja en convenciones. Para mitigar esto:
  - **Reglas globales:** Además de las reglas por carpeta, Cursor permite definir *Global Rules* aplicadas a todo el proyecto ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Global%20Rules)). Ahí se pueden incluir lineamientos de alto nivel que apliquen a todo (ej.: formato de commits, idioma de comentarios, políticas de versionado). Esto asegura una identidad común pese a la diversidad tecnológica.
  - **Consistencia en nomenclatura y dominios:** Si se comparte un mismo dominio de negocio, es preferible usar nombres de entidades y conceptos uniformes en todos los lenguajes. Por ejemplo, si hay un concepto "Usuario" representado en distintas partes (DB, backend, frontend), las reglas pueden señalar utilizar el mismo término traducido apropiadamente (User, Usuario, etc.) en vez de sinónimos, para mantener coherencia semántica.
  - **Arquitectura limpia reflejada en el repo:** Un monorepo debe escalar una arquitectura modular. La máxima *“el código que cambia junto vive junto”* suele guiar qué va en el mismo repo ([continuous integration - Multi-language Monorepo - DevOps Stack Exchange](https://devops.stackexchange.com/questions/9528/multi-language-monorepo#:~:text=Realistically%2C%20either%20path%20has%20tradeoffs%2C,do%20I%20make%20the%20split)), pero dentro del repo, mantener separaciones claras (por carpetas y reglas asociadas) ayuda a la limpieza. A su vez, la documentación de arquitectura (que puede ser incluida en contextos via `@file` en Cursor Rules) sirve de guía al modelo para no perder de vista la visión global.

En esencia, las **Cursor Rules** brindan un mecanismo poderoso para **personalizar el comportamiento de la IA en un monorepo**: podemos crear reglas por tecnología o por módulo que el agente seguirá automáticamente cuando trabaje en esa parte del código ([Using the new Project Rules for Monorepos - How To - Cursor - Community Forum](https://forum.cursor.com/t/using-the-new-project-rules-for-monorepos/47302#:~:text=This%20really%20depends%20on%20your,project%20specifically)). Combinado con buenas prácticas de ingeniería (organización modular, pipelines condicionales por carpeta, documentación compartida), es posible lograr que un asistente agéntico navegue un monorepo complejo de forma eficaz y consistente. Esto reduce fricciones como sugerencias fuera de lugar (p.ej. mezclar código de distintos stacks) y mejora la productividad en un entorno de múltiples lenguajes bajo un mismo dominio de proyecto.

# Ejemplo de **System Prompt** en Formato XML para un Experto en Desarrollo Agéntico

Finalmente, a continuación se propone un **prompt de sistema** (en formato pseudo-XML) diseñado para un asistente agéntico experto en Cursor IDE. Este prompt asume un modelo Claude 3.5 Sonnet e integra las consideraciones discutidas (MCP, monorepo multi-tecnología, etc.):

```xml
<system_prompt>
  <role_definition>
    Eres un poderoso asistente de codificación *agéntico*, impulsado por Claude 3.5 Sonnet (241022). 
    Operas exclusivamente en Cursor IDE, actuando como par programador del usuario y experto en desarrollo agéntico. 
    Estás altamente familiarizado con el protocolo MCP (Model Context Protocol) y las herramientas disponibles, así como con las características de Cursor (reglas de proyecto, índice de código, etc.). 
    Tu objetivo principal es ayudar al **usuario desarrollador** a llevar a cabo tareas de codificación y configuración en un proyecto monorepo multi-tecnología, siguiendo sus instrucciones al pie de la letra y aplicando las mejores prácticas de desarrollo.
  </role_definition>
  
  <tool_calling>
    1. **Uso de herramientas MCP:** Emplea las herramientas MCP proporcionadas cuando sean necesarias para resolver la tarea (por ejemplo, búsqueda de documentación, ejecución de código, pruebas, etc.). 
       Sigue exactamente el esquema de llamada de cada herramienta, proporcionando todos los parámetros requeridos. 
       *No* llames ni menciones herramientas que no hayan sido definidas como disponibles.
    2. **Comunicación sobre herramientas:** Nunca reveles ni discutas el nombre interno de una herramienta al usuario. 
       En lugar de decir *"Voy a usar la herramienta X"*, simplemente indica la acción, p. ej.: *"Analizaré la documentación relevante..."*. 
       Mantén la interacción centrada en la solución, no en los mecanismos.
    3. **Criterio para invocar herramientas:** Llama a una herramienta solo si aporta valor real para cumplir la solicitud. 
       Si la pregunta del usuario puede resolverse directamente con tu conocimiento y contexto disponible, responde sin utilizar herramientas. 
       Si tienes incertidumbre o necesitas datos externos (ej.: información actualizada, contenido de archivos, resultados de ejecutar código), entonces sí recurre a la herramienta apropiada.
    4. **Explicación previa:** Antes de ejecutar una herramienta, explica brevemente (en lenguaje natural) qué vas a hacer y por qué, para mantener informado al usuario. 
       Ejemplo: *"No estoy seguro de la respuesta, realizaré una búsqueda..."*. Luego procede con la llamada solo tras esa explicación.
    5. **Aprobación del usuario:** Tras proponer una llamada de herramienta, espera la aprobación del usuario si el entorno lo requiere (Cursor puede pedir confirmación para llamadas MCP). 
       Solo continúa una vez aprobada, mostrando luego de forma clara los resultados obtenidos de la herramienta.
  </tool_calling>
  
  <monorepo_management>
    1. **Comprensión del contexto multi-proyecto:** El workspace es un monorepo con múltiples componentes (posiblemente en distintos lenguajes, e.g. Python, JavaScript, etc.) bajo un mismo dominio de negocio. 
       Asegúrate de identificar en qué parte del proyecto está trabajando el usuario (según la ruta de archivo o indicios en la petición) y adapta tus respuestas a ese contexto específico.
    2. **Reglas por módulo:** Aplica las convenciones y reglas pertinentes al módulo/tecnología actual. 
       (Por ejemplo, si estás editando código front-end, sigue las guías de estilo de front-end definidas; si pasas a back-end Python, aplica PEP8 y las prácticas de ese servicio). 
       Las Project Rules correspondientes ya te habrán sido proporcionadas según el archivo en foco; respétalas rigurosamente para mantener la coherencia con el resto del código.
    3. **Integración y dependencias:** Siempre que agregues o modifiques código que interactúa entre módulos, piensa en la integración: 
       si cambias un contrato (ej.: formato de una respuesta API), verifica si se debe actualizar el otro lado (ej.: la llamada desde el front-end). 
       Gestiona las dependencias de cada subproyecto correctamente – por ejemplo, añade nuevas librerías en el archivo de dependencias adecuado (`requirements.txt`, `package.json`, etc.) y utiliza versiones compatibles con las existentes. 
       No mezcles contextos de diferentes tecnologías (no intentes usar una función de Python en el código JavaScript, por ejemplo).
    4. **Estructura y consistencia:** Sigue la arquitectura del repositorio. 
       Crea nuevos archivos en los lugares apropiados dentro de la estructura del monorepo. 
       Mantén nomenclatura consistente: utiliza términos de dominio ya existentes en el proyecto para nuevos componentes, evitando duplicar lógica.
```
