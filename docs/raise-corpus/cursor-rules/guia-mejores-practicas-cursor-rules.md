Voy a investigar las mejores prácticas para construir Cursor Rules en el IDE de Cursor.com, utilizando información general del repositorio. Me centraré en cómo optimizar la generación de reglas a partir del código existente, asegurando un balance entre precisión, eficiencia y flexibilidad.

Te avisaré cuando tenga la información lista.

# Guía de mejores prácticas para construir **Cursor Rules** en Cursor IDE

## 1. Principios generales  
Las **Cursor Rules** permiten personalizar el comportamiento de la IA en Cursor, actuando como instrucciones de alto nivel para el modelo. Es fundamental estructurarlas de forma eficiente y mantenible. Algunas recomendaciones generales son: 

- **Organización modular**: En lugar de una única regla monolítica, divide las reglas por tema o contexto. Con la nueva estructura de proyecto (directorio `.cursor/rules`), puedes crear múltiples archivos de reglas con un encabezado YAML que indique una descripción y patrones de archivos a los que aplican ([Can anyone help me use this new .cursor/rules functionality? - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/can-anyone-help-me-use-this-new-cursor-rules-functionality/45692#:~:text=I%E2%80%99m%20on%20v0,and%20a%20list%20of%20rules)) ([Can anyone help me use this new .cursor/rules functionality? - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/can-anyone-help-me-use-this-new-cursor-rules-functionality/45692#:~:text=,description%E2%80%A6)). Esto brinda control granular, haciendo que cada regla se aplique solo en las partes relevantes del código ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Here%E2%80%99s%20how%20they%20work)). Por ejemplo, podrías tener reglas separadas para *frontend*, *backend*, *base de datos*, etc., cada una activándose solo en sus rutas o tipos de archivo designados. Esto mejora la mantenibilidad al permitir actualizar secciones específicas sin afectar al resto.

- **Contexto del proyecto primero**: Comienza cada archivo de reglas proporcionando una visión general de tu proyecto. Describe brevemente de qué se trata, su stack tecnológico y cualquier característica global relevante ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=1)). Esta descripción inicial sirve para situar al modelo en el dominio correcto. Por ejemplo, *“Esta es una aplicación web en React con TypeScript que sigue la guía de estilo de Airbnb”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=1)). Con este contexto, la IA entenderá el entorno general sobre el cual debe basar sus sugerencias.

- **Estándares de código claros**: Después del contexto general, incluye las convenciones de codificación de tu equipo. Define lineamientos de estilo y mejores prácticas que deben cumplirse en el código generado ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=2)). Por ejemplo, puedes indicar reglas como *“Usar componentes funcionales con Hooks en lugar de componentes de clase”* y *“Preferir `const` sobre `let` para declaraciones de variables”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Please%20follow%20these%20coding%20standards%3A,names%2C%20PascalCase%20for%20component%20names)). También especifica convenciones de nombres (por ejemplo, camelCase para variables y PascalCase para clases o componentes) y cualquier otra norma (orden de importaciones, longitud máxima de línea, etc.). Estos estándares guiarán al autocompletado para que el código sugerido se alinee con el estilo del proyecto.

- **Incluye solo lo necesario**: Aunque es tentador volcar mucha información en las reglas, es mejor ser conciso. Cada instrucción adicional consume espacio en el contexto que la IA procesa. Prioriza las reglas más importantes y evita redundancias. Instrucciones breves y directas facilitan que el modelo las **“absorba”** correctamente. Por ejemplo, en lugar de un párrafo largo explicando un patrón, es más efectivo usar viñetas o puntos enumerados con frases concretas. Las reglas escritas con markdown permiten formatear secciones con encabezados y listas, lo que mejora la legibilidad tanto para humanos como para la IA ([Can anyone help me use this new .cursor/rules functionality? - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/can-anyone-help-me-use-this-new-cursor-rules-functionality/45692#:~:text=I%E2%80%99m%20on%20v0,and%20a%20list%20of%20rules)). Una estructura limpia (secciones como *Arquitectura*, *Estándares*, *Errores comunes*, etc.) ayuda a mantener el archivo de reglas organizado.

- **Mantenlas versionadas y actualizadas**: Al ser archivos de texto en tu repositorio, las reglas deben incluirse en control de versiones junto con el código. Así, cualquier cambio en la arquitectura o convenciones del proyecto puede reflejarse en las reglas de manera sincronizada. Revisa periódicamente el contenido de las reglas y **ajústalo según evolucione el proyecto** ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=By%20implementing%20these%20best%20cursor,aligned%20with%20your%20current%20requirements)). Por ejemplo, si adoptan una nueva biblioteca o cambian el estilo de código, actualiza la regla correspondiente. Esta atención continua garantiza que la IA siempre reciba instrucciones relevantes y correctas.

En resumen, unas reglas bien estructuradas deben presentar primero la *visión global* del proyecto, luego las *normas específicas* (estilo, bibliotecas, estructura) y estar divididas lógicamente para facilitar su mantenimiento. Siguiendo estos principios, lograrás una base sólida sobre la cual las reglas especializadas pueden operar.

## 2. Extracción de información del repositorio  
Para crear reglas efectivas, debes **alimentarlas con la información clave de tu propio código fuente**. Esto implica extraer conocimientos del repositorio para incorporarlos como contexto en las reglas. Algunas mejores prácticas en esta etapa son:

- **Revisa la documentación existente**: Comienza por los archivos de documentación de tu proyecto (README, guías de estilo, archivos de configuración). Rescata de allí detalles importantes: propósito del proyecto, frameworks utilizados, versión de lenguaje, convenciones ya establecidas, etc. Incluir un resumen de esta información en las reglas ayudará a la IA a entender el entorno. Si tu proyecto es grande, sintetiza la *“vista de alto nivel”* del sistema en las reglas ([Definitive Rules - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/definitive-rules/45282#:~:text=ONE,in%20Rules%20for%20AI%20section)). Un miembro de la comunidad indica que en su caso la mayor parte de `.cursorrules` consiste en explicar la arquitectura general de un gran código base, dando esa visión panorámica al modelo ([Definitive Rules - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/definitive-rules/45282#:~:text=ONE,in%20Rules%20for%20AI%20section)).

- **Analiza la base de código**: Explora el código fuente para detectar patrones y prácticas actuales. Por ejemplo, identifica cómo están estructurados los directorios (¿hay separación por módulos o capas como *controllers*, *services*, *models*?), observa si se siguen convenciones como cierto estilo de nomenclatura o diseño, y detecta qué bibliotecas se usan con más frecuencia. Esta “minería” de información puede ser manual o asistida por herramientas. El IDE de Cursor ofrece indexado de la base de código ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=,when%20the%20rule%20is%20applied)), por lo que podrías usar la búsqueda (`@Code` o `@Files`) para encontrar, por ejemplo, todas las clases que heredan de una cierta clase base, o todas las funciones marcadas con TODO, etc., y decidir si alguna convención surge de esos resultados para documentar en las reglas. 

- **Utiliza la información de configuración**: Si tu proyecto tiene linters, formatters o herramientas de compilación configuradas (por ejemplo, un archivo ESLint, Prettier, `pyproject.toml` para Black, o directrices de compilación), extrae sus reglas relevantes. Muchas de estas configuraciones representan decisiones de estilo o restricciones que debes trasladar a las Cursor Rules. Por ejemplo, si el linter exige comillas simples en cadenas o una longitud máxima de línea, vale la pena mencionarlo en las reglas para que la IA lo tenga en cuenta al autocompletar. Del mismo modo, si la configuración indica ciertas *best practices* (como evitar funciones depreciadas), inclúyelo en forma de instrucción.

- **Incorpora conocimiento de dominio**: Más allá del código, considera la *información de negocio o de dominio* presente en el repositorio. Por ejemplo, documentación de arquitectura (ADR), modelos de datos, glosarios de términos de negocio, etc. Estos elementos pueden no ser código, pero entenderlos es crucial para producir código correcto. Una práctica útil es resumir esos documentos en un archivo de reglas específico (por ejemplo `project_info.mdc`) y luego referenciarlo desde otras reglas con `@file` si tu regla lo soporta ([Can anyone help me use this new .cursor/rules functionality? - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/can-anyone-help-me-use-this-new-cursor-rules-functionality/45692#:~:text=General%20Rules)). De esta forma, no saturas todas las reglas con contexto extenso, sino que mantienes un archivo de referencia que el modelo puede consultar cuando sea necesario.

- **Ejemplo de extracción iterativa**: Un usuario relata su proceso trabajando con un lenguaje DSL propio: primero **ingestó la documentación del DSL** para ponerla a disposición del modelo, luego inició un proyecto de prueba y **observó la salida que generaba la IA**, para finalmente añadir reglas que corrigieran o guiaran al modelo, *“como si le hablara a un desarrollador junior”*, con explicaciones en Markdown ([Providing the best context - How To - Cursor - Community Forum](https://forum.cursor.com/t/providing-the-best-context/28099#:~:text=We%20work%20in%20an%20in,junior%20developer%20using%20markdown%20syntax)). Esta anécdota muestra un enfoque: utilizar la IA inicialmente sin muchas reglas, ver qué malentendidos o desviaciones ocurren, y **extraer de esa experiencia** qué información adicional del repositorio o del dominio debe volcarse en las reglas. Por ejemplo, si la IA confundió la responsabilidad de ciertos módulos, puedes agregar en las reglas una sección aclarando la arquitectura modular. Si el modelo sugiere una biblioteca no utilizada en el proyecto, asegúrate de haber listado todas las bibliotecas permitidas (y quizás explícitamente mencionar las no deseadas).

En síntesis, extraer información del repositorio implica **identificar los elementos clave de tu proyecto** (tecnología, estilo, arquitectura, dominio) y **convertirlos en instrucciones** que vivan en las reglas. Cuanto más fiel sea esa transferencia de conocimiento, más contextuadas y pertinentes serán las sugerencias de la IA.

## 3. Tipos de reglas: autocompletado, formateo, sugerencias y validaciones  
Al redactar las Cursor Rules, es útil pensar en diferentes **categorías de reglas**, según el aspecto del proceso de desarrollo al que afectan. Las cuatro categorías principales son: reglas de **autocompletado**, de **formateo**, de **sugerencias** y de **validaciones**. A continuación se explica cada tipo y cómo abordarlas en las reglas:

### Reglas de *autocompletado*  
Estas reglas buscan orientar **cómo el modelo completa el código** a medida que escribes. Su objetivo es que el código generado automáticamente encaje bien con la base de código existente y las expectativas del desarrollador. Las mejores prácticas para reglas de autocompletado incluyen:

- **Patrones de código y plantillas**: Si en tu proyecto existen patrones recurrentes (como bloques de código estándar, estructuras típicas de funciones, clases base comunes, etc.), descríbelos en las reglas. Por ejemplo, en un proyecto Node.js podrías indicar: *"Al crear una nueva ruta Express, incluir siempre la gestión de errores con `try/catch` y responder en formato JSON"*. Esto induce al autocompletado a generar esos fragmentos completos en vez de dejarlos fuera. En general, indica al modelo qué *esqueleto* debe seguir el código generado en ciertos contextos.

- **Uso correcto de APIs y frameworks**: Es frecuente que el modelo de lenguaje pueda sugerir llamadas o funciones que *suenan* razonables pero que no existen en una API particular, o utilizar de forma inapropiada una biblioteca. Para evitarlo, especifica en las reglas cómo usar las APIs de tu proyecto. Por ejemplo, *“Cuando sugieras código para la base de datos, usa únicamente nuestros métodos del repositorio X en lugar de llamadas SQL directas”*. Otro ejemplo: en un proyecto React con hooks, podríamos tener una regla: *“Siempre que sea posible, utiliza Hooks de React en lugar de componentes de clase”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Please%20follow%20these%20coding%20standards%3A,names%2C%20PascalCase%20for%20component%20names)) (lo cual el modelo respetará al autocompletar componentes). Estas indicaciones alinean la finalización automática con las prácticas esperadas.

- **Convenciones de estilo como guía**: Muchas reglas de autocompletado se traslapan con estándares de estilo. Por ejemplo, si la convención es *usar programación funcional y evitar clases*, al enunciar esto en las reglas estás moldeando las completaciones para que sigan ese paradigma. En un proyecto Python FastAPI podríamos decir: *“Emplear programación funcional/declarativa; evitar clases en lo posible”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation)). De igual modo, indicar *“usar nombres descriptivos con snake_case para variables y archivos”* guía al modelo a completar con nombres coherentes con esa convención ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation)). En resumen, cada pauta de estilo que incluyes se convierte en un sesgo positivo en las sugerencias de código.

- **Integridad del código**: Puedes instruir a la IA a *completar de forma más completa*. Por ejemplo, si al autocompletar una función es importante que incluya ciertos pasos (validaciones de entrada, comentarios de documentación, etc.), menciónalo en las reglas. *“Toda nueva función debe incluir un docstring que explique sus parámetros y retorno”* sería una regla que impulsa al modelo a generar ese docstring al autocompletar la función. Otra táctica útil es pedir que agregue las importaciones necesarias automáticamente. Si bien Cursor a veces gestiona las importaciones, recalcar *“Incluir los imports requeridos para cada snippet sugerido”* puede ayudar a que la autocompletación no omita esas líneas, evitando que el desarrollador tenga que agregarlas manualmente después.

En esencia, las reglas de autocompletado **alinean las sugerencias de código con tu estilo y patrones esperados**, de modo que al aceptar una completitud, el código insertado requiera mínimas correcciones. Un proyecto con buenas reglas verá que el autocompletado produce código casi listo para usarse, coherente con la base existente.

### Reglas de *formateo*  
Estas reglas se centran en la **presentación y estilo del código** que genera la IA. Aunque el formateo puede ser corregido por herramientas automatizadas (prettier, black, etc.), es beneficioso que la IA ya sugiera código formateado correctamente para ahorrar pasos y reducir fricción. Buenas prácticas en reglas de formateo:

- **Especificar la guía de estilo**: Indica explícitamente si sigues una guía conocida (PEP 8 para Python, el estilo Airbnb para JavaScript, Google Style para Java, PSR-12 para PHP, etc.). Por ejemplo: *“El código debe seguir PEP 8 en cuanto a espacios, nombres y organización”*. En un caso real, un desarrollador de TypeScript señaló en sus reglas que se use la guía de estilo de Airbnb ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=1)), lo que implica, entre otras cosas, comillas simples, identación de 2 espacios, etc. Al mencionar la guía o estándar, el modelo tenderá a imitar ese estilo en su formato de salida.

- **Detallar convenciones de formato**: Si hay reglas específicas de formato en tu proyecto, inclúyelas. Por ejemplo, *“Usar llaves en la misma línea para las funciones y bucles (estilo **One True Brace**)”* ([.CursorRules](https://dotcursorrules.com/#:~:text=,Classes%2C%20loops%2C%20and%20If%20statements)), o *“Siempre incluir punto y coma al final de las declaraciones (en JavaScript)”*. Otro ejemplo: *“En YAML, usar 2 espacios en vez de tabulaciones”*. Estas precisiones harán que el código generado ya cumpla con ellas. Un fragmento real de una regla indica: *“Usar formato One True Brace para funciones, clases, bucles e if”* ([.CursorRules](https://dotcursorrules.com/#:~:text=,Classes%2C%20loops%2C%20and%20If%20statements)), asegurando que el modelo coloque correctamente las llaves en esos casos.

- **Integración con formateadores**: Si usas una herramienta de formateo automático (por ejemplo, Prettier, Black, clang-format), puedes mencionarlo en las reglas para que la IA intente respetar ese formato. En un blog, un usuario comentó que utiliza Prettier y ESLint, y definió como requerimiento que *“todo el código debe pasar ESLint con la configuración del proyecto”* ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=All%20code%20must%3A)). Esto implica que el modelo debe generar código que no viole reglas de linting/format. De hecho, en sus *Quality Requirements* especificó: *“Todo el código debe mantener tipado estricto de TypeScript, pasar ESLint, e incluir manejo de errores”* ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=All%20code%20must%3A)). Así, formatea y organiza el código de salida pensando ya en satisfacer esas reglas. 

- **Consistencia en la salida**: También es útil instruir a la IA a mantener un formato consistente durante la conversación. Por ejemplo, una regla global podría ser: *“Siempre formatear las respuestas de código con bloques de código triple backtick, especificando el lenguaje”*. Esto garantiza que, incluso al dar sugerencias en chat, la IA presentará el código de forma bien formateada y fácil de copiar. Asimismo, si prefieres que las explicaciones vengan comentadas en el código (o separadas afuera), acláralo para mantener uniformidad en las respuestas.

En resumen, las reglas de formateo **orientan al modelo sobre cómo debe verse el código**, disminuyendo la necesidad de reformatear después. Esto incluye tanto lineamientos de estilo concretos (llaves, espacios, comillas) como recordatorios de cumplir con herramientas de formato ya configuradas en el proyecto.

### Reglas de *sugerencias*  
Bajo esta categoría entran las reglas que afectan **cómo la IA brinda sugerencias o recomendaciones** más allá del código en sí. Es decir, cómo el asistente conversacional de Cursor responde en contexto de chat o agente, qué tipo de explicaciones o consejos da, y de qué manera los presenta. Algunas mejores prácticas para estas reglas:

- **Controlar la verbosidad y el tono**: Define si quieres que las respuestas de la IA sean detalladas o concisas, formales o informales. Por ejemplo, puedes instruir: *“Responde de manera concisa, sin dar ejemplos de código adicionales a menos que se soliciten”*. Un usuario comentó que en sus reglas globales indica a la IA *ser concisa y no dar ejemplos a menos que los pida*, para ahorrar espacio de contexto ([Definitive Rules - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/definitive-rules/45282#:~:text=ONE,in%20Rules%20for%20AI%20section)). Esto es útil en entornos donde el prompt tiene límite de tokens: mantiene las sugerencias enfocadas. Si, por el contrario, valoras explicaciones elaboradas, podrías indicar: *“Proporciona explicaciones claras para las decisiones arquitectónicas, acompañadas de referencias a secciones específicas del código o documentación”* ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=1,and%20maintainability%20in%20all%20suggestions)). Así, el modelo sabrá que debe justificar sus recomendaciones (por ejemplo: *“Uso X aquí debido a [referencia a diseño Y]”*). Ajustar el tono también es importante: *“Responder siempre en primera persona plural y con un tono profesional”* podría ser otra regla de estilo en las sugerencias.

- **Enriquecer las recomendaciones**: Puedes guiar a la IA para que sus sugerencias incluyan consideraciones adicionales valiosas. Por ejemplo: *“Al sugerir una solución, mencionar implicaciones en seguridad, rendimiento y mantenibilidad”*. En una regla real se definió que el asistente debe **considerar aspectos de seguridad, escalabilidad y mantenibilidad en todas las sugerencias** ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=1,the%20quality%20requirements%20defined%20herein)). Esto implica que si el desarrollador pide una implementación, la IA no solo dará el código, sino que podría agregar: “ten en cuenta que esta solución escala hasta X peticiones por segundo” o “esta técnica expone tal riesgo de seguridad”. De igual manera, podrías pedir que sugiera optimizaciones cuando sea pertinente: *“Si detectas que el código puede mejorarse en eficiencia, propón la mejora”*. El objetivo es que la IA actúe como un par programador experimentado, no solo cumpliendo la orden mínima sino ofreciendo *insights* adicionales útiles.

- **Formato de las sugerencias**: Otra vertiente es instruir cómo estructurar las respuestas. Por ejemplo, *“Si la respuesta es muy larga, presentarla en secciones con encabezados”*, o *“Usar listas numeradas para pasos a seguir en una explicación”*. Esto mejora la legibilidad de las sugerencias complejas. Si integras documentación externa mediante referencias, podrías estandarizar frases como: *“Según la documentación oficial…”* cuando la IA traiga información de `@Docs` o `@Web`. Todas estas guías hacen más consistente la interacción con el asistente.

- **Ejemplos y contraejemplos**: Puedes incluso influir en si el asistente debe mostrar ejemplos alternativos. Por ejemplo, *“Cuando sugieras una solución, comenta también una posible alternativa si existe un enfoque diferente significativo”*. Esto podría llevar al modelo a decir *“Alternativamente, podrías usar la librería Z para lograr lo mismo”*. Solo utiliza este tipo de regla si realmente es deseable en tu flujo de trabajo (podría serlo en etapas de diseño, pero quizá no cuando se busca una respuesta directa).

En suma, las reglas de sugerencias se enfocan en **cómo la IA comunica sus ideas**: el nivel de detalle, los ángulos que considera (seguridad, rendimiento, etc.), el formato y tono de la respuesta, e incluso la inclusión o exclusión de ciertos contenidos en sus recomendaciones. Bien afinadas, estas reglas hacen que la ayuda de la IA sea más valiosa y adaptada a tus preferencias de trabajo.

### Reglas de *validaciones*  
Este tipo de reglas instruye al modelo a **verificar o asegurar ciertas condiciones de calidad o corrección** en el código que genera. Funcionan como una capa de autocontrol que la IA aplica antes de considerar su trabajo “listo”. Incorporar reglas de validación puede elevar la confiabilidad del código sugerido. Buenas prácticas:

- **Verificaciones sintácticas y de compilación**: Instrúyele que *revise la sintaxis y la corrección del código antes de presentarlo*. Por ejemplo: *“Verifica que el código generado compile sin errores y cumple las validaciones de tipos”*. En un conjunto de reglas compartido, se indica al agente revisar la sintaxis y dependencias antes y después de hacer ediciones ([Tips for Agent (It's very powerful!) - How To - Cursor - Community Forum](https://forum.cursor.com/t/tips-for-agent-its-very-powerful/33111#:~:text=%3E%20%20%20,Ensure%20consistent%20formatting)) ([Tips for Agent (It's very powerful!) - How To - Cursor - Community Forum](https://forum.cursor.com/t/tips-for-agent-its-very-powerful/33111#:~:text=%3E%20%20%20,Ensure%20formatting%20consistency)). Esencialmente, el modelo debe hacerse la pregunta “¿esto compila y pasa las pruebas?” antes de sugerir el código. Si incluyes *“asegúrate de que el código respeta el tipado estricto”* o *“valida que todas las variables usadas estén definidas”*, estarás forzando a la IA a un chequeo mental similar al de un linter/compilador. 

- **Conformidad con requisitos de calidad**: Define criterios que todo cambio o fragmento propuesto debe cumplir. Por ejemplo: *“El código no debe introducir vulnerabilidades conocidas (XSS, inyección SQL, etc.)”*, o *“Todas las funciones nuevas deben incluir manejo de errores para casos excepcionales”*. Un ejemplo tomado de una regla real: *“El asistente debe validar el código contra los requisitos de calidad aquí definidos”* ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=1,the%20quality%20requirements%20defined%20herein)). Esto abarca desde estilo (ya cubierto) hasta buenas prácticas generales. Si en tu proyecto, por ejemplo, **ninguna consulta SQL va sin parámetros** (para prevenir inyección), puedes ponerlo en las reglas. Entonces, si la IA genera una consulta concatenando strings, estaría rompiendo la regla y debería corregirla antes de mostrarla.

- **Uso de tests o ejemplos para validar**: Aunque el modelo no puede realmente ejecutar código, puedes hacer que razone sobre casos de prueba. Una técnica es incluir en las reglas algo como: *“Imaginar al menos un caso de prueba para cada función sugerida y verificar mentalmente que la implementación lo pasa”*. Esto suena abstracto, pero puede llevar al modelo a detectar errores lógicos simples. Por ejemplo, si sugiere una función de división, podría “pensar” en un caso con divisor = 0 y ajustar la sugerencia para manejarlo porque la regla le recordó contemplar casos. También podrías decir: *“No sugerir código que falle en los tests existentes del proyecto”* si tienes pruebas; sin embargo, esto último solo funciona si el contexto del agente incluye saber qué tests existen, lo cual suele requerir que los tests mismos estén cargados o indexados.

- **Validaciones de consistencia**: Otra validación útil es la consistencia con el código existente. Por ejemplo: *“Si se introduce un nuevo método en una clase, verificar que su nombre y comportamiento sean consistentes con métodos similares en la clase/proyecto”*. Esto anima a la IA a comprobar patrones. O *“No duplicar funciones ya existentes; buscar referencias antes de crear algo nuevo”* — una suerte de autovalidación para evitar código repetido. De hecho, se puede combinar con el uso del índice: Cursor puede buscar símbolos, pero aquí la regla es más una recomendación para que la IA no reimplemente funcionalidad existente inadvertidamente.

Resumiendo, las reglas de validación **instan al modelo a “auto-revisar” el resultado** antes de entregarlo, aplicando filtros de calidad similares a los que haría un revisor humano o herramientas automáticas. Esto reduce la carga de corregir errores tontos o problemas evidentes en las sugerencias, ya que idealmente la IA los detectará y ajustará en su propio proceso de generación.

## 4. Optimización de rendimiento  
Al trabajar con Cursor Rules es importante considerar su **impacto en el rendimiento del IDE y del asistente de IA**. Aunque las reglas enriquecen el contexto, también consumen parte de la ventana de contexto del modelo y podrían, si son excesivas, ralentizar las respuestas. Aquí van algunas mejores prácticas para optimizar rendimiento:

- **Reglas breves y relevantes**: Mantén cada regla lo más corta posible, sin sacrificar claridad. Evita incluir información irrelevante o demasiado detallada que el modelo quizá no necesite en cada consulta. Cada palabra de más compite con el código del usuario por espacio en el prompt. Piensa que las reglas son un *“presupuesto de tokens”* que debes administrar. Si logras enunciar una política en 5 palabras en vez de 15, hazlo. Por ejemplo, en lugar de escribir *“El desarrollador debe procurar que las variables globales no sean utilizadas debido a posibles condiciones de carrera y dificultades de mantenimiento”*, podrías poner *“Evitar variables globales; usar inyección de dependencias”*. Es más telegráfico pero igual de efectivo para la IA. Un consejo frecuente es **ser conciso para ser eficiente** ([Definitive Rules - Discussion - Cursor - Community Forum](https://forum.cursor.com/t/definitive-rules/45282#:~:text=ONE,in%20Rules%20for%20AI%20section)).

- **Uso estratégico de *Project Rules* vs Global Rules**: Cursor te permite tener reglas globales (en *Settings > Rules for AI*) y reglas de proyecto (archivo(s) en `.cursor/rules`). Las globales se adjuntan siempre, mientras que las de proyecto solo cuando corresponde. Para rendimiento, coloca en global solo lo imprescindible para todos los proyectos (por ejemplo, “sé conciso” puede ser global) y deja lo específico en reglas de proyecto. Esto evita cargar reglas innecesarias en cada consulta. Las reglas de proyecto mismas, como ya se mencionó, **deberían tener condiciones (globs)** para aplicarse solo donde importan ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=Here%E2%80%99s%20how%20they%20work)). Así, si tienes reglas extensas para manejo de *SQL*, esas solo se incluyen cuando trabajas en archivos SQL o de base de datos, y no cuando estás editando, digamos, un archivo CSS. Esta modularidad reduce la carga cognitiva para el modelo en cada acción.

- **Limitar el encadenamiento excesivo**: Es posible referenciar archivos dentro de las reglas (usando `@file` u otros símbolos) para agregar contexto adicional dinámicamente ([Cursor – Rules for AI](https://docs.cursor.com/context/rules-for-ai#:~:text=,when%20the%20rule%20is%20applied)). Si bien esto es poderoso, úsalo con moderación. Incluir automáticamente fragmentos muy grandes (como un archivo completo de documentación) en cada consulta puede **degradar el rendimiento** notablemente. En su lugar, opta por resúmenes. Si sientes que necesitas referenciar un documento extenso constantemente, considera extraer de él los puntos clave y ponerlos fijos en las reglas. De esa forma evitas transferir bloques enormes repetidamente. 

- **Evitar conflictos y confusiones**: Asegúrate de que tus reglas no contengan directivas contradictorias, ya que eso puede hacer que el modelo pierda tiempo “decidiendo” qué seguir. Por ejemplo, no quieras mezclar en la misma regla mensajes como "*sé muy detallado en las explicaciones*" y en otro punto "*sé conciso*". Esa ambigüedad no solo confunde a la IA sino que puede llevar a respuestas más lentas mientras intenta reconciliar las instrucciones. Mantén un mensaje coherente y sin ambigüedades.

- **Monitorea el tiempo de respuesta**: Una buena práctica es observar, tras implementar tus reglas, cómo afecta eso la velocidad de las sugerencias. Si notas retardos significativos, podrías experimentar quitando o simplificando algunas reglas para ver el impacto. Cursor IDE en sí ejecuta las peticiones al modelo, por lo que si la entrada (que incluye tu consulta + contexto + reglas) es muy grande, la respuesta del modelo tardará más. No hay un número mágico de tokens que debas usar, pero como regla general empírica, mantener las reglas por debajo de unos pocos miles de caracteres suele ser prudente. 

En resumen, la optimización de rendimiento con Cursor Rules se logra **manteniendo las reglas concisas, contextuales y bien segmentadas**, de modo que el modelo siempre procese únicamente lo necesario. Así obtendrás respuestas más rápidas y con menor probabilidad de agotar el contexto disponible.

## 5. Mejora iterativa de las reglas  
Las primeras versiones de tus Cursor Rules no tienen por qué ser perfectas. De hecho, es recomendable adoptar un **enfoque iterativo**, ajustando las reglas con el tiempo a medida que recibes *feedback* y observas métricas de éxito. ¿Cómo llevar a cabo esta mejora continua? Aquí algunas pautas:

- **Observa el comportamiento de la IA**: Después de implementar o cambiar reglas, presta atención a las respuestas de la IA. ¿Las sugerencias de código ahora se alinean mejor con tus expectativas? ¿Sigue habiendo errores o estilos no deseados? Toma nota de los casos en que el asistente **no** se comporta como debería según las reglas. Esos casos son oro puro para saber qué falta afinar. Por ejemplo, si añadiste una regla de evitar cierto antipatrón pero aún así el modelo lo usa en alguna sugerencia, puede que necesites reformular la regla más claramente o moverla a un lugar más prominente (por ejemplo, más arriba en el archivo, o a reglas globales si es muy importante).

- **Incorpora el feedback de otros desarrolladores**: Si trabajas en equipo, invita a tus colegas a utilizar Cursor con las reglas y que te comenten su experiencia. Quizá ellos detecten molestias o mejoras. Por ejemplo, podrían decirte *“Cursor sigue usando la función antigua de autenticación en vez de la nueva”*. Esto te indica que tal vez debas enfatizar en las reglas qué mecanismo de autenticación usar. Tener la perspectiva de varios usuarios te ayuda a cubrir más casos. Cada vez que alguien encuentre que la IA hizo algo incorrecto o subóptimo, **considera si una regla adicional o modificada podría prevenirlo en el futuro**.

- **Usa métricas cuando sea posible**: Aunque es difícil obtener métricas cuantitativas directas de “calidad de sugerencias”, puedes medir indirectamente cosas como: reducción en el número de ediciones manuales tras aceptar una sugerencia, porcentaje de compilaciones exitosas sin correcciones cuando el código fue generado por IA, o simplemente tu satisfacción en una escala. Si llevas un registro (incluso mental) de “antes de las reglas vs después de las reglas”, verás el progreso. Si ciertas métricas empeoran (por ejemplo, notas que ahora tardas más en obtener la respuesta deseada), analiza si alguna regla está entorpeciendo más de lo que ayuda.

- **Refina con casos concretos**: Una táctica valiosa es, ante un fallo de la IA, agregar una instrucción específica en las reglas para cubrir ese caso. Un experto en prompt engineering sugiere: *“si ocurren problemas, anótalos en tu archivo .cursorrules para afinar tu enfoque”* ([Maximizing Your Cursor Use: Advanced Prompting, Cursor Rules ...](https://medium.com/@extremelysunnyyk/maximizing-your-cursor-use-advanced-prompting-cursor-rules-and-tooling-integration-496181fa919c#:~:text=Maximizing%20Your%20Cursor%20Use%3A%20Advanced,file%20to%20refine%20your%20approach)). Es decir, documenta el problema en forma de regla. Imagina que la IA borró accidentalmente un archivo crítico en una sesión anterior; podrías añadir una regla: *“Nunca eliminar archivos de configuración importantes a menos que el usuario lo confirme explícitamente”*. De esta manera conviertes un aprendizaje (incidente) en prevención futura. 

- **Itera gradualmente**: No es recomendable cambiar demasiadas reglas a la vez sin probar, porque si luego la IA mejora o empeora, no sabrás exactamente qué cambio lo causó. Es mejor tratar las reglas como experimentos controlados: modifica o agrega una o dos cosas, prueba durante un tiempo, evalúa resultados, y luego sigue con otras. Esta iteración controlada te permite entender el impacto de cada regla.

- **Mantén un histórico**: Dado que las reglas están en control de versiones, puedes revisar cambios pasados y quizás correlacionarlos con eventos (e.j., *“Después de agregar la regla X, el asistente dejó de usar tal librería no deseada”*). Conservar comentarios en el commit sobre por qué se hizo cierto ajuste de regla puede ser útil para el futuro o para otros en tu equipo.

Resumiendo, mejorar las Cursor Rules es un proceso **continuo de afinación**. Se basa en *feedback* real (tanto humano como observado en el comportamiento del modelo) y en la disposición a ajustar las instrucciones. Con el tiempo, tu archivo de reglas irá madurando y cubriendo cada vez más escenarios, convirtiéndose en una parte integral de la documentación viva de tu proyecto para la IA.

## 6. Compatibilidad con otras herramientas  
Las Cursor Rules no operan en el vacío; tu flujo de desarrollo involucra linters, formateadores, sistemas de compilación, pruebas automatizadas, etc. Por ello, es importante que las reglas **se complementen bien con estas herramientas**, evitando conflictos y aprovechando sinergias. Aquí van las mejores prácticas para lograr compatibilidad:

- **Alineación con linters**: Si en tu proyecto usas un linter (ESLint, Flake8, RuboCop, etc.), configura las reglas para que la IA siga las mismas normas que el linter va a exigir. Esto reduce iteraciones, ya que el código sugerido ya vendrá “lintiado” correctamente. Un ejemplo claro: *"El código sugerido debe pasar ESLint con la configuración del proyecto"* ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=All%20code%20must%3A)). En la práctica, esto se traduce a que la IA respete todas las reglas de estilo y posibles restricciones que tu ESLint impondría. Si ESLint prohíbe el uso de `var` o funciones `eval()`, especifícalo también en las reglas para que el asistente ni siquiera las proponga. De esta manera, **lo que la IA genere no disparará errores de linting**, y podrás integrarlo al repositorio más rápidamente.

- **Respeto por el formateador**: Similar al punto anterior, si cuentas con una herramienta de formateo automático (por ejemplo Prettier para JS/TS, Black para Python, gofmt para Go), menciona en las reglas que el código debe adherir a ese formato. Esto previene que el diff tras formatear sea grande. En el caso de Prettier, podrías listar algunas de sus configuraciones clave si son diferentes del estándar (e.g., *“usar comillas simples”, “no usar punto y coma”*, etc., dependiendo de tu `.prettierrc`). En un blog se citaba: *“Uso Prettier sobre ESLint para formateo”*, seguido de reglas de calidad que incluyen *“todo código debe mantener tipado estricto y pasar ESLint”* ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=Language%20and%20Style)) ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=All%20code%20must%3A)). Esto indica que el formateo (controlado por Prettier/ESLint) está integrado en los requerimientos de código. En otras palabras, la IA considerará esas normas de formateo como parte de sus obligaciones al generar código.

- **Consistencia con el compilador**: Para lenguajes compilados (Java, C#, C++), orienta las reglas a que produzcan código que compile a la primera. Instrucciones como *“Usar únicamente APIs disponibles en Java 11”* (si tu proyecto está en Java 11, por ejemplo) son importantes para que el modelo no sugiera sintaxis de versiones más nuevas que romperían la compilación. Igualmente, *“Asegurarse de que todos los caminos de ejecución devuelven el tipo requerido”* puede ayudar a prevenir errores de compilación comunes (como olvidarse un return). Si tu proyecto usa un compilador estricto o flags especiales (ej: `-Wall -Werror` en C++), deja claro que **no se toleran warnings**. Así el modelo será más cuidadoso en detalles. Piensa que el linter y el compilador son en sí validaciones automáticas; si alineas las reglas con ellos, es casi como tener el compilador “dentro” de la IA validando antes de sugerir.

- **Integración con herramientas de prueba**: Esto puede ser llevado un paso más allá: si tienes un sólido suite de tests, puedes animar al modelo a producir código que pase esos tests. Por ejemplo: *“Seguir TDD: primero pensar en los tests. Todo código sugerido debe considerar pasar los casos de prueba existentes”*. Además, en las reglas puedes enumerar qué frameworks de prueba se usan para que la IA los utilice. En un ejemplo, un desarrollador indicó en sus reglas las herramientas de testing: Jest, React Testing Library, etc. ([Cursor: Rules for AI Code Editor | Sebastian Müller, Hamburg - sbstjn.com](https://sbstjn.com/blog/ai-code-companion-cursor-rules/#:~:text=,mocks%20for%20cloud%20resource%20testing)). Esto permite que, si le pides al asistente “escribe una prueba para X”, sepa exactamente qué librerías usar y cómo estructurarla, manteniendo compatibilidad con tu entorno de pruebas. 

- **Indicaciones sobre herramientas externas**: Si tu flujo incluye otras herramientas (ejemplo: Docker, CI/CD pipelines, generadores de código), puedes documentar en las reglas cómo interactuar con ellas. Por ej.: *“Si se modifica el esquema de BD, actualizar el archivo `db_structure.md`”*. De hecho, vimos una referencia donde una regla indicaba: *“Actualizar `db_structure.md` cuando ocurran cambios en el esquema, asegurando que se mantenga como la fuente de verdad”* ([Show me your general prompt for Rules for AI from Cursor settings](https://www.reddit.com/r/cursor/comments/1faf2rw/show_me_your_general_prompt_for_rules_for_ai_from/#:~:text=settings%20www,it%20remains%20the%20authoritative)). Esto integra una tarea de documentación (herramienta externa al código en sí) dentro de las sugerencias de la IA. La IA, al seguir esta regla, podría recordarte esa actualización o incluso generar el cambio correspondiente en el archivo de documentación.

- **Evitar duplicidad/conflicto con herramientas**: Si ya tienes una herramienta que se encarga de algo, tus reglas no deben contradecirla. Por ejemplo, si usas un formateador automático, no es necesario que tus reglas detallen cada aspecto del formato (la herramienta lo hará de todos modos), pero sí debe asegurarse de **no contradecir**. Sería contraproducente decir "*no poner punto y coma al final de línea*" en reglas si luego Prettier los va a agregar; ese tipo de conflicto solo confundiría al modelo. En su lugar, alinea la regla con lo que la herramienta hará (en este caso, indica *“usar punto y coma al final”* si Prettier así lo exige, o simplemente no lo menciones y deja que Prettier actúe). En general, las reglas y las demás herramientas deben empujar en la misma dirección.

En conclusión, piensa en las Cursor Rules como parte de un *ecosistema* junto con linters, formatters y compiladores. **La meta es que la IA produzca código que ya venga correcto** bajo esas métricas, reduciendo retrabajo. Al explicitar en las reglas los mismos criterios que evalúan esas herramientas, el asistente puede autoevaluar sus respuestas con ese estándar. Un beneficio colateral es que las sugerencias se sienten más “nativas” del proyecto: usan las mismas librerías que usaría un desarrollador humano (evitando dependencias ajenas) ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Guide%20the%20AI%20to%20use,project%27s%20chosen%20libraries%20and%20frameworks)), cumplen las mismas validaciones y encajan sin problemas en la tubería de integración continua existente.

## 7. Ejemplos detallados de reglas por lenguaje  
A continuación, se presentan ejemplos ilustrativos de **Cursor Rules optimizadas** para distintos entornos de programación. Estos ejemplos muestran cómo traducir las mejores prácticas anteriores en reglas concretas adaptadas a un lenguaje o stack específico.

### Ejemplo: Aplicación web en *React* con *TypeScript*  
Supongamos un proyecto front-end escrito en TypeScript usando React. Las reglas podrían estructurarse así (en un archivo `.mdc` con su encabezado YAML apropiado):

- **Contexto del Proyecto**: *“Esta es una aplicación web **React** escrita en **TypeScript**. El proyecto sigue la guía de estilo Airbnb y utiliza una arquitectura modular por componentes.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=1)). (Brinda al modelo una visión general de la tecnología y convenciones globales).

- **Estándares de Código**: 
  - *“Usar componentes funcionales con Hooks en lugar de componentes de clase.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Please%20follow%20these%20coding%20standards%3A,names%2C%20PascalCase%20for%20component%20names)) (Así, la IA siempre sugerirá `function Component()` o componentes con `useState`/`useEffect` en vez de `React.Component` tradicionales).  
  - *“Preferir `const` sobre `let` para declaraciones de variables.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Please%20follow%20these%20coding%20standards%3A,names%2C%20PascalCase%20for%20component%20names)) (El modelo tenderá a declarar variables inmutables cuando sea posible).  
  - *“Emplear **camelCase** para nombres de variables y funciones, y **PascalCase** para nombres de componentes y clases.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Please%20follow%20these%20coding%20standards%3A,names%2C%20PascalCase%20for%20component%20names)) (Garantiza consistencia en la nomenclatura).  
  - *“Evitar el uso de any; usar tipado estricto en todo momento con interfaces o types.”* (Alineado con TypeScript estricto, evitando perder tipado).

- **Librerías y Frameworks Preferidos**: 
  - *“Al sugerir código, utiliza **React Router** para navegación en vez de otras librerías.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Guide%20the%20AI%20to%20use,project%27s%20chosen%20libraries%20and%20frameworks))  
  - *“Para estado global, utilizar **Redux** (con Toolkit) en lugar de Context API pura u otras soluciones.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Guide%20the%20AI%20to%20use,project%27s%20chosen%20libraries%20and%20frameworks))  
  - *“Para estilos, emplear **styled-components**, evitando CSS inline o librerías no adoptadas en el proyecto.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=When%20suggesting%20code%2C%20please%20use,JS%20styling))  
  (Con esto, si la IA debe sugerir una ruta, sabrá que tiene `<Routes>` de React Router disponible; si sugiere manejo de estado, sabrá que debe pensar en `configureStore` de Redux, etc., manteniéndose consistente con las dependencias reales del proyecto).

- **Estructura de Archivos**: *“Estructura del proyecto: 
  `/src/components` contiene componentes reutilizables, 
  `/src/pages` contiene componentes de página o vistas, 
  `/src/utils` funciones utilitarias, 
  `/src/api` lógica de llamadas a API.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Help%20the%20AI%20understand%20your,project%27s%20organization)).  
  (Esto orienta al asistente sobre dónde viven las cosas, de modo que por ejemplo, si sugiere crear un nuevo componente utilitario, pueda inferir que debería ir en `src/components` y quizás mencionarlo en la respuesta).

- **Pautas de Rendimiento**: 
  - *“Usar `React.memo` en componentes puros para evitar renders innecesarios.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Emphasize%20performance%20considerations%3A))  
  - *“Implementar carga diferida (lazy loading) para módulos de ruta mediante `React.lazy` y `Suspense`.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Performance%20guidelines%3A%20,renders))  
  - *“Optimizar las dependencias de los Hooks (useEffect, useCallback) para prevenir re-ejecuciones redundantes.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Emphasize%20performance%20considerations%3A))  
  (Estas reglas hacen que la IA, al proponer soluciones, considere siempre el ángulo de rendimiento: por ejemplo, podría envolver automáticamente un componente exportado con `React.memo` si ve que no depende de props dinámicas).

- **Requisitos de Pruebas**: 
  - *“Escribir pruebas unitarias para cada nuevo componente o función usando **Jest** + **React Testing Library**.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=6))  
  - *“Apuntar a una cobertura mínima del 80% en pruebas.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Outline%20your%20testing%20approach%3A))  
  - *“Incluir pruebas de snapshot para componentes de UI importantes.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=6))  
  (De esta manera, si le pides a la IA que genere un componente complejo, quizás también te sugiera esbozar una prueba unitaria correspondiente, o al menos recordar agregar pruebas, porque está en las reglas). 

- **Lineamientos de Documentación**: 
  - *“Usar comentarios JSDoc en funciones y componentes públicos, describiendo sus props y retornos.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=7))  
  - *“Utilizar `PropTypes` en cada componente React para validar las props (además del tipado TS).”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=7))  
  - *“Mantener un archivo README.md en cada directorio principal explicando su contenido si éste es complejo.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Documentation%20guidelines%3A%20,file%20in%20each%20major%20directory))  
  (Con esto, la IA podría incluir un comentario JSDoc automáticamente cuando genera una nueva función, o recordarle al usuario que actualice cierto README si se agregó un módulo notable).

- **Manejo de Errores**: 
  - *“En operaciones asíncronas, envolver en try/catch y manejar errores de forma adecuada.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=8))  
  - *“Implementar un componente de límite de error (Error Boundary) global para capturar errores de render.”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Error%20handling%3A%20,Sentry))  
  - *“Loguear los errores al servicio de monitoreo (p.ej., enviar a Sentry los detalles de errores inesperados).”* ([Best Cursor Rules: Mastering .cursorrules for Cursor IDE](https://dotcursorrules.com/blog/best-cursor-rules-mastering-cursorrules-for-cursor-ide#:~:text=Error%20handling%3A%20,Sentry))  
  (Así, cualquier código sugerido para, digamos, una llamada `fetch` sabrá que debería llevar su try/catch. O si el usuario pregunta “¿Cómo manejo errores globales?”, la IA verá en las reglas que la respuesta debe incluir un Error Boundary y logging). 

Estas reglas de ejemplo para React/TypeScript cubren desde lo general (contexto, estilo) hasta detalles específicos (herramientas de testing, logging de errores). Al aplicarlas, el desarrollador recibiría sugerencias alineadas con todo lo anterior, lo que acelera la integración del código generado a la base existente con mínimo retrabajo.

### Ejemplo: API backend en *Python* con *FastAPI*  
Ahora consideremos un proyecto backend en Python, utilizando FastAPI para construir APIs REST. Las Cursor Rules podrían ser algo como lo siguiente:

- **Contexto y Rol de la IA**: *“Eres un asistente experto en **Python** y **FastAPI**, enfocado en desarrollo de APIs escalables.”* (Muchas reglas .cursorrules comienzan estableciendo el rol/expertise del modelo, para sesgar sus respuestas). *“Responde de forma concisa y técnica, siempre proporcionando ejemplos de código válidos en Python.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=You%20are%20an%20expert%20in,FastAPI%2C%20and%20scalable%20API%20development)). Esto último asegura que las respuestas vengan con ejemplos concretos y al grano.

- **Estilo y Paradigma**: 
  - *“Utilizar programación funcional y declarativa; evitar definir clases si no es necesario (usar funciones y dependencias inyectables de FastAPI).”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation))  
  - *“Prefirir la iteración y la composición de funciones sobre la duplicación de código.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation))  
  - *“Escribir código *Pythonic*: claro y legible, siguiendo PEP 8.”* (Por ejemplo, en vez de programar al estilo Java dentro de Python, seguir convenciones de Python en excepciones, context managers, etc.)

- **Convenciones de Nomenclatura**: 
  - *“Usar nombres descriptivos con **snake_case** para variables, funciones y nombres de archivos.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation))  
  - *“Para variables booleanas, utilizar verbos auxiliares (ej.: `is_active`, `has_permission`).”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation))  
  - *“Los endpoints (rutas) de la API deben nombrarse en plural en inglés (ej. `/users`, `/orders`).”* (Esto último sería una convención específica de la API que queremos que la IA respete).

- **Estructura de Proyecto**: *“Estructura de archivos: separar rutas (routers) por módulo de negocio, utilidades comunes en un paquete `utils/`, modelos de datos y esquemas Pydantic en `models/` o `schemas/`, archivos estáticos en `static/`.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Pydantic%20models%20over%20raw%20dictionaries,for%20input%20validation)). Con esto, la IA al sugerir *“crear un nuevo endpoint para X”* podría indicar en su respuesta qué archivo router.py tocar o cómo organizar el código, reflejando la estructura establecida.

- **Tipado y Validación**: 
  - *“Usar anotaciones de tipo en **todas** las definiciones de función y método.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Return%20an%20Object%20%28RORO%29%20pattern,raw%20dictionaries%20for%20input%20validation)) (Esto fuerza a que cada función sugerida venga tipada).  
  - *“Emplear modelos **Pydantic** (BaseModel) para la entrada y salida de datos en la API, en lugar de diccionarios crudos.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=underscores%20for%20directories%20and%20files,raw%20dictionaries%20for%20input%20validation)) ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=I%2FO,Minimize)). Por ejemplo, si se crea un endpoint, que la IA defina un modelo Pydantic para el request/response en vez de manejar dicts sin esquema.  
  - *“Aprovechar tipos async: usar `async def` para operaciones de I/O o aquellas que interactúan con la base de datos de forma asíncrona; usar `def` normal solo para cómputo síncrono intensivo en CPU.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation)). Esto hará que el modelo sugiera funciones asíncronas adecuadamente (FastAPI soporta ambas, pero es importante marcar la diferencia según el caso).

- **Manejo de Rutas y Controladores**: 
  - *“Cada router de FastAPI debe tener un prefix y tags definidos; agrupar endpoints lógicamente.”* (por ejemplo, rutas de usuarios bajo `/users`).  
  - *“Preferir *named exports* para importar routers y utilidades (por ejemplo, `from routers.users import router as UsersRouter`).”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation))  
  - *“Seguir el patrón **RORO** (Receive an Object, Return an Object): recibir modelos Pydantic como entrada y retornar modelos Pydantic como respuesta, evitando tipos primitivos sueltos.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=Write%20concise%2C%20technical%20responses%20with,raw%20dictionaries%20for%20input%20validation)). Esto garantiza consistencia en cómo se estructuran las funciones de ruta.

- **Errores y Excepciones**: 
  - *“Usar `HTTPException` de FastAPI para errores esperados (p.ej., NotFound, Unauthorized), devolviendo el código HTTP apropiado.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=I%2FO,Minimize))  
  - *“No usar `raise Exception` genérico; en su lugar definir excepciones específicas o usar las proporcionadas por frameworks.”*  
  - *“Manejar errores inesperados con middleware global: registrar el error (log) y retornar una respuesta 500 genérica.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=I%2FO,Minimize)). Por ejemplo, incluir en reglas la existencia de un middleware de captura de excepciones asegurará que la IA no replique manejo de excepción en cada endpoint de forma redundante, sino que confíe en ese middleware global para lo no controlado.

- **Rendimiento y Buenas Prácticas de Backend**: 
  - *“Optimizar las consultas a base de datos: usar consultas asíncronas (por ej. con `asyncpg` o `sqlalchemy` async) para I/O.”* ([awesome-cursorrules/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules at main · PatrickJS/awesome-cursorrules · GitHub](https://github.com/PatrickJS/awesome-cursorrules/blob/main/rules/python-fastapi-best-practices-cursorrules-prompt-f/.cursorrules#:~:text=SQLAlchemy%202,features))  
  - *“Implementar caching donde aplique (p.ej., resultados de lectura muy frecuentes) para mejorar la respuesta.”*  
  - *“Evitar bloqueos en la event loop: no realizar operaciones sincrónicas largas en rutas async (usar subprocesos o delegar a tareas background si es necesario).”*  
  - *“Liberar recursos apropiadamente: cerrar conexiones de base de datos en eventos de apagado (`@app.on_event("shutdown")`).”* (Esto recuerda a la IA manejar contextos correctamente, aunque no lo vea, sabrá que debe mencionar el cierre de recursos si crea uno nuevo).  

- **Integración con Herramientas/Frameworks**: 
  - *“Usar **Alembic** para migraciones de base de datos (no generar SQL manual para alterar esquemas).”* (Si el proyecto la usa, claro está).  
  - *“Seguir las prácticas de **FastAPI Dependency Injection** para obtener sesiones de DB u otros componentes (ej. `Depends(get_db)`).”*  
  - *“Escribir tests con **PyTest** utilizando la herramienta `TestClient` de FastAPI para probar endpoints.”* (Esto le indica a la IA cómo encaminar sugerencias de pruebas de API).  

Como vemos, estas reglas para Python/FastAPI cubren estilo de código, estructura, uso correcto del framework y consideraciones de robustez. Están escritas de forma que la IA actúe prácticamente como un **guarda de arquitectura**: cualquier desviación (sea de estilo, de uso de librería, de manejo de errores) está explícitamente prohibida o recomendada, de forma que el modelo la detecte y se autocorrija. Por ejemplo, gracias a la regla de `HTTPException`, difícilmente la IA sugeriría devolver manualmente un `"404 not found"` en texto plano, sino que usaría la excepción adecuada de FastAPI. Y con la insistencia en modelos Pydantic, el modelo sabrá que debe generar clases que heredan de `BaseModel` para la entrada/salida, integrándose perfectamente con el ecosistema FastAPI.
