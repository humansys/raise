# Migration Guide: v2.4 → v2.5

## Overview

Version 2.5 introduces the **Three-Directory Model** (ADR-011), which reorganizes how RaiSE structures project directories and separates concerns between configuration, governance, and work.

**Key Changes:**
- `specs/` renamed to `work/`
- New `governance/` directory for approved artifacts
- New `framework/` directory (for raise-commons and framework definition)
- Context files moved to `framework/context/`
- Guardrails file renamed from `governance.md` to `guardrails.md`

## The Three-Directory Model

### Core Concept

```
project/
├── .raise/       # Configuration (the engine) — HOW to govern
├── governance/   # Authority (the library) — WHAT governs
└── work/         # Activity (the workbench) — WHAT we're doing
```

| Directory | Purpose | Lifecycle |
|-----------|---------|-----------|
| `.raise/` | Framework engine (katas, gates, templates) | Stable, synced |
| `governance/` | Approved artifacts (visions, designs, ADRs) | Curated, after gates pass |
| `work/` | Work-in-progress (features, proposals, research) | Transient |

### Promotion Flow

Artifacts move from `work/` to `governance/` when they pass validation gates:

```
work/                              governance/
─────                              ───────────

work/proposals/adr-012.md    →     governance/decisions/adr-012.md
     (draft)                            (accepted)

work/projects/foo/vision.md  →     governance/projects/foo/vision.md
     (draft)                            (approved)

work/features/NNN/           →     (nothing — ephemeral)
     (feature work)
```

### Governance Index

`governance/index.yaml` is a manifest for agents:

```yaml
schema_version: "1.0.0"
solution:
  name: "My Product"
  status: active

artifacts:
  - path: solution/vision.md
    level: solution
    type: vision
    version: 1.0.0
    status: approved
```

## Migration Steps

### Step 1: Create New Directories

```bash
mkdir -p governance/{solution,projects,decisions,context}
mkdir -p work/{features,proposals,research}
```

### Step 2: Move Solution-Level Artifacts

```bash
# Business Case
mv specs/main/business_case.md governance/solution/business_case.md

# Solution Vision
mv specs/main/solution_vision.md governance/solution/vision.md

# Guardrails (renamed from governance.md)
mv .raise/governance/governance.md governance/solution/guardrails.md
```

### Step 3: Move Project-Level Artifacts

```bash
# Create project directory
mkdir -p governance/projects/main

# Move project artifacts
mv specs/main/project_vision.md governance/projects/main/vision.md
mv specs/main/tech_design.md governance/projects/main/design.md
mv specs/main/project_backlog.md governance/projects/main/backlog.md
```

### Step 4: Move Feature Work

```bash
# Move feature specs
mv specs/[0-9][0-9][0-9]-*/ work/features/
```

### Step 5: Create Governance Index

Create `governance/index.yaml`:

```yaml
schema_version: "1.0.0"
solution:
  name: "Your Solution Name"
  status: active

artifacts:
  - path: solution/vision.md
    level: solution
    type: vision
    version: 1.0.0
    status: approved

  - path: solution/guardrails.md
    level: solution
    type: guardrails
    version: 1.0.0
    status: approved

  # Add project artifacts as needed
```

### Step 6: Update References

```bash
# Find old references
grep -r "specs/main/" .
grep -r "specs/raise/" .
grep -r ".raise/context/" .

# Update paths in your files
# specs/main/ → governance/solution/ or governance/projects/{name}/
# specs/raise/ → framework/
# .raise/context/ → framework/context/
```

### Step 7: Clean Up (After Verification)

```bash
# Only after verifying everything works
rm -rf specs/main/
rm -rf .raise/context/  # If moved to framework/
```

## Gate Changes

Gates now validate artifacts in their new locations and include promotion instructions:

| Gate | New Validation Path |
|------|---------------------|
| `gate-vision` | `governance/projects/{project}/vision.md` |
| `gate-design` | `governance/projects/{project}/design.md` |
| `gate-discovery` | `governance/projects/{project}/prd.md` |

**Post-Gate Action**: When a gate passes, artifacts are promoted from `work/` to `governance/`.

## Glossary Changes

New terms added in v2.5:

- **Governance Directory**: Authoritative source of approved artifacts
- **Work Directory**: Workbench for work-in-progress
- **Governance Index**: Manifest file (`governance/index.yaml`)
- **Governance Promotion**: Moving artifacts from `work/` to `governance/`
- **Golden Data Drift**: When governance artifacts become stale

Updated terms:

- **Guardrails**: Now stored in `governance/solution/guardrails.md` (renamed from `governance.md`)

## Breaking Changes

1. **Directory rename**: `specs/` → `work/` for feature work
2. **Artifact locations**: Solution/project artifacts move to `governance/`
3. **File rename**: `governance.md` → `guardrails.md`
4. **Context location**: `.raise/context/` → `framework/context/` (for raise-commons)

## Non-Breaking Changes

1. `.raise/` structure (katas, gates, templates) remains the same
2. Kata execution flow unchanged
3. Gate validation logic unchanged (only paths updated)

## FAQ

### Do I need to migrate immediately?

No. v2.5 is backward-compatible during a transition period. You can migrate gradually.

### What if I have work in `specs/NNN-feature/`?

Move them to `work/features/NNN-feature/`. The content structure remains the same.

### Why rename `governance.md` to `guardrails.md`?

To avoid naming collision with the new `governance/` directory. `guardrails.md` is also more precise — it contains the specific constraints, while "governance" is the broader concept.

### What's the purpose of `governance/index.yaml`?

It's a manifest that enables agents to understand the project without exploring the file tree. Agents read the index first, then fetch only the artifacts they need.
