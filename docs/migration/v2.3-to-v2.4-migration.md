# Migration Guide: v2.3 → v2.4

## Overview

Version 2.4 introduces the **Three-Level Artifact Hierarchy** (ADR-010), which reorganizes how RaiSE structures development artifacts.

**Key Change:** "Solution Vision" at project level is renamed to **"Project Vision"**.

---

## What Changed

### New Hierarchy

```
ANTES (v2.3)                          DESPUÉS (v2.4)
═══════════════                       ═══════════════

[Sin nivel solución]                  SOLUTION LEVEL (Sistema)
                                      ├── Business Case
                                      ├── Solution Vision
                                      └── Governance

PROJECT LEVEL                         PROJECT LEVEL (Proyecto)
├── PRD                               ├── PRD
├── Solution Vision  ─────────────→   ├── Project Vision  (RENOMBRADO)
├── Tech Design                       ├── Tech Design
└── Backlog                           └── Backlog
```

### Renamed Artifacts

| Antes | Después | Nivel |
|-------|---------|-------|
| `solution_vision.md` (proyecto) | `project_vision.md` | Project |
| *(no existía)* | `solution_vision.md` | Solution |
| *(no existía)* | `business_case.md` | Solution |

### New Katas

| Kata | Propósito | Output |
|------|-----------|--------|
| `solution/discovery` | Business Case creation | `specs/main/business_case.md` |
| `solution/vision` | Solution Vision definition | `specs/main/solution_vision.md` |

### Updated Katas

| Kata | Cambio |
|------|--------|
| `project/vision` | Output renombrado a `project_vision.md` |
| `project/design` | Input cambiado a `project_vision.md` |
| `setup/governance` | Prerequisito cambiado a `solution/vision` |

## Migration Steps

### For Existing Projects

#### Step 1: Rename Project-Level Vision

```bash
# Si tienes un archivo solution_vision.md a nivel proyecto
mv specs/main/solution_vision.md specs/main/project_vision.md
```

#### Step 2: Update References

Buscar y reemplazar en tu codebase:

```bash
# Buscar referencias al nombre antiguo
grep -r "solution_vision.md" specs/ docs/

# Actualizar manualmente o con sed
# (asegurarse de que sea a nivel proyecto, no solución)
```

#### Step 3: Create Solution-Level Artifacts (Opcional)

Si deseas adoptar la jerarquía completa:

```bash
# 1. Crear Business Case
raise kata solution/discovery

# 2. Crear Solution Vision
raise kata solution/vision

# 3. Ejecutar Governance
raise kata setup/governance
```

### For New Projects

Los nuevos proyectos deben seguir el flujo completo:

```
1. solution/discovery  → Business Case
2. solution/vision     → Solution Vision
3. setup/governance    → Governance/Guardrails
4. project/discovery   → PRD
5. project/vision      → Project Vision (antes: Solution Vision)
6. project/design      → Tech Design
7. project/backlog     → Backlog
```

## Glossary Changes

Nuevos términos añadidos al glosario v2.4:

- **Solution Level**: Nivel de artefactos del sistema (perdura entre proyectos)
- **Project Level**: Nivel de artefactos de proyecto (time-bound)
- **Business Case**: Justificación de negocio a nivel solución
- **Project Vision**: Visión técnica a nivel proyecto (renombrado)

Términos actualizados:

- **Solution Vision**: Ahora exclusivamente a nivel solución
- **Work Cycle**: Añadido ciclo `solution/`

---

## Backward Compatibility

### Breaking Changes

1. **Output path cambiado**: `project/vision` kata ahora genera `project_vision.md` en lugar de `solution_vision.md`
2. **Gate validation path**: `gate-vision` ahora busca `project_vision.md`

### Non-Breaking Changes

1. Nuevas katas (`solution/discovery`, `solution/vision`) son opcionales
2. Proyectos existentes pueden seguir funcionando si renombran el archivo

---

## FAQ

### ¿Debo crear Business Case y Solution Vision para proyectos existentes?

**No obligatorio.** Si tu proyecto ya tiene un `solution_vision.md` (ahora `project_vision.md`), puedes:
1. Renombrarlo a `project_vision.md` y seguir adelante
2. Opcionalmente, crear artefactos de nivel solución si lo necesitas

### ¿Qué pasa si tengo múltiples proyectos en el mismo sistema?

El nivel **solución** es ideal para este caso:
- Un **Business Case** y **Solution Vision** definen el sistema
- Múltiples **Project Visions** definen cada proyecto/iniciativa

### ¿La governance ahora requiere Solution Vision?

Sí. El kata `setup/governance` ahora tiene `solution/vision` como prerequisito. Esto asegura que la governance derive de una visión clara del sistema.
