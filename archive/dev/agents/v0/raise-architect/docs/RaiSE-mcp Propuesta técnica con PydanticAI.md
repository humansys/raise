# Propuesta técnica: Servidor MCP con PydanticAI y FastAPI para asistencia agéntica en desarrollo de software

## Introducción

La **Model Context Protocol (MCP)** es un protocolo abierto estandarizado (basado en JSON-RPC 2.0) que permite a aplicaciones de IA conectarse con fuentes de datos y herramientas externas de forma segura ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=The%20protocol%20is%20based%20on,and%20error%20reporting%20enhancing%20reliability)). En esencia, un servidor MCP actúa como un *backend* inteligente (similar a un “puerto USB-C” para IA ([Introduction \- Model Context Protocol](https://modelcontextprotocol.io/introduction#:~:text=MCP%20is%20an%20open%20protocol,different%20data%20sources%20and%20tools))) que expone datos, herramientas o plantillas de prompt a asistentes de IA (por ejemplo, integrados en un IDE) ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=,AI%E2%80%99s%20behavior%20in%20specific%20scenarios)). Esto posibilita que los modelos de lenguaje puedan realizar acciones más allá de generar texto, como acceder a archivos locales, consultar servicios externos o seguir plantillas predefinidas ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=services%20like%20GitHub%2C%20all%20in,a%20secure%20and%20standardized%20way)) ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=,AI%E2%80%99s%20behavior%20in%20specific%20scenarios)).

En este contexto, se propone el desarrollo de **RaiSE** (Remote AI Software Engineer), un servidor MCP construido con **FastAPI** (framework web asíncrono en Python) y **PydanticAI** (framework orientado a agentes LLM del equipo de Pydantic) para asistir a desarrolladores dentro de su flujo de trabajo. RaiSE funcionará como backend para clientes MCP como Cursor IDE o Claude Desktop, habilitando **flujos agénticos** de asistencia durante el desarrollo de software. Es decir, permitirá que un agente de IA autónomo asista al programador en tareas complejas, siguiendo políticas y estándares definidos, mediante interacciones multi-paso con modelos de lenguaje y herramientas.

El caso de uso principal a ilustrar es: *“El desarrollador, desde su IDE, activa un agente en modo cursor (‘agent mode’) solicitando una acción, por ejemplo generar una historia de usuario. El cliente MCP (en el IDE) envía esta solicitud al servidor RaiSE. El servidor entonces determina la plantilla adecuada para la historia de usuario (según el contexto, estándares de la empresa, estilo, etc.), realiza las llamadas a modelos de lenguaje y APIs necesarias mediante flujos agénticos, y devuelve una respuesta estructurada al cliente para mostrarla al usuario.”*

A continuación, se detalla la arquitectura propuesta, los flujos de ejecución representativos, consideraciones sobre el uso de n8n como conector de servicios DevSecOps, el *stack* tecnológico y patrón de desarrollo recomendados, así como aspectos de seguridad, extensibilidad y gobernanza de la solución.

## Arquitectura propuesta del sistema

**Componentes principales:** La solución se compone de los siguientes bloques lógicos:

- **Cliente MCP (IDE o interfaz de usuario):** Aplicación host (p. ej. Cursor IDE, Claude Desktop) donde el desarrollador interactúa. Este cliente actúa como *MCP host* o *MCP client*, conectándose al servidor RaiSE mediante el protocolo MCP ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=to%20leverage%20external%20data.%20,file%20system%20server%20might%20allow)). Desde el IDE, el usuario puede invocar acciones agénticas (como “generar historia de usuario”) ya sea escribiendo una solicitud o utilizando una opción de la interfaz. El cliente MCP transmite la solicitud al servidor RaiSE y espera la respuesta estructurada.  
    
- **Servidor MCP (RaiSE – FastAPI \+ PydanticAI):** Servicio web liviano (contenedor Docker desplegable en cloud o *on-premise*) que implementa el protocolo MCP. Este servidor expone capacidades específicas para asistir en desarrollo, clasificadas en términos de MCP como **Tools (herramientas)** y **Prompts (plantillas)** ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=,AI%E2%80%99s%20behavior%20in%20specific%20scenarios)). RaiSE contiene la lógica de **agente de IA** que recibe las peticiones del IDE, las interpreta, decide los pasos a seguir (p. ej., seleccionar plantilla de historia de usuario, llamar a un modelo de lenguaje, acceder a información adicional) y finalmente produce una respuesta. Todo el ciclo de resolución – desde el prompt inicial, llamadas a herramientas/LLM, hasta la composición de la respuesta – ocurre internamente en el servidor. Esto descarga trabajo del cliente y centraliza la lógica en un entorno controlado.  
    
- **Modelos de Lenguaje (LLMs):** RaiSE integrará uno o varios LLMs para generar contenido y decisiones. Se privilegiarán modelos **open-source** para las tareas de lenguaje natural (por ejemplo, modelos estilo Llama 2 fine-tuneados para generación de texto empresarial), alojados localmente o en un servicio autosuficiente, de forma que los datos sensibles permanezcan bajo control de la organización. Para tareas específicas de código, se utilizará **Claude 3.7 Sonnet** de Anthropic (un modelo de vanguardia especializado en coding ([Claude 3.7 Sonnet \- Anthropic](https://www.anthropic.com/claude/sonnet#:~:text=Claude%203.7%20Sonnet%20,taste%20while%20maintaining%20context)) ([Guides: Get started with Claude 3.7 Sonnet \- AI SDK](https://sdk.vercel.ai/docs/guides/sonnet-3-7#:~:text=Guides%3A%20Get%20started%20with%20Claude,complex%20reasoning%2C%20and%20content%20generation))) vía API. La selección dinámica del modelo permitirá que, por ejemplo, al generar una historia de usuario (texto), se use un LLM abierto en servidor local, mientras que si el agente debe generar o analizar código, delegue a Sonnet 3.7 para mayor calidad. Esta combinación equilibra la preferencia por *open source* con el rendimiento superior de Sonnet en casos de código.  
    
- **Conectores DevSecOps (APIs/Jira/Git/GitHub/...):** Para acciones que involucren servicios externos de la cadena de desarrollo (crear un ticket en Jira, abrir un *pull request* en GitHub, lanzar una *pipeline* CI/CD, etc.), el servidor incluirá **herramientas integradas** que encapsulen llamadas a esas APIs. Estas herramientas pueden implementarse directamente en Python (usando las bibliotecas oficiales o peticiones HTTP) o mediante un orquestador externo como **n8n**. La arquitectura evaluará ambas opciones: llamadas directas vs. invocación de flujos n8n (ver sección dedicada a n8n). En cualquier caso, estos conectores estarán aislados del núcleo lógico (por ejemplo en módulos separados), y RaiSE gestionará credenciales de acceso de forma segura (variables de entorno, vault, etc.).  
    
- **Almacenamiento y contexto (opcional):** Si el agente requiere información persistente – p. ej., plantillas de hist. de usuario corporativas, glosarios de términos, políticas de seguridad para filtrar respuestas – se podrá integrar un pequeño repositorio de datos. Esto podría ser desde archivos de configuración en JSON/YAML, una base de datos ligera, o incluso recursos estáticos expuestos vía MCP **Resources** (otra categoría de capacidades MCP para datos ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=,AI%E2%80%99s%20behavior%20in%20specific%20scenarios))). Por ejemplo, un recurso “company://user\_story\_template” podría proveer al agente un texto base de cómo debe formatearse una historia de usuario según estándares internos. Alternativamente, el agente puede tener ese conocimiento codificado en su *prompt* de sistema.

**Diagrama de alto nivel:** A continuación se describe el flujo entre componentes (en pseudodiagrama):

- **1\)** El IDE (Cliente MCP) inicia una petición al servidor RaiSE mediante MCP (JSON-RPC sobre HTTP/S o SSE). Por ejemplo, el usuario selecciona “Generar Historia de Usuario” en la interfaz de Cursor IDE; internamente el cliente envía un llamado al método o *tool* correspondiente expuesto por RaiSE (p. ej., una función `generate_user_story`).  
- **2\)** FastAPI en el servidor recibe la solicitud en un endpoint MCP. Gracias a la integración MCP, la petición ya viene estructurada y se encamina al handler apropiado (la función/herramienta solicitada).  
- **3\)** Dentro de RaiSE, la función invocada actúa como *wrapper* de la lógica agéntica: crea o utiliza un **agente PydanticAI** para manejar la tarea. Este agente tiene un *prompt* de sistema predefinido (por ejemplo, instrucciones sobre formato y estilo de historias de usuario), y está configurado con las **tools internas** que pudiera necesitar (por ejemplo, si requiere consultar Jira o código, aunque en este caso solo redacción). El agente recibe como entrada el pedido concreto (p. ej. descripción de requisitos de la nueva funcionalidad) y comienza su flujo.  
- **4\)** El **agente PydanticAI** ejecuta su lógica: podría primero seleccionar una plantilla adecuada (esto puede estar implementado simplemente como parte del *prompt* inicial, o si fuera complejo, incluso llamando a una herramienta interna que retorna la plantilla según el contexto del proyecto). Luego el agente invoca el **LLM** correspondiente para redactar el contenido: en este caso, un modelo open-source para generar texto de la historia de usuario, asegurándose de seguir la estructura (“Como , quiero , para …”) y tono definidos. Si el agente necesitara varios pasos (por ejemplo, validar que la salida cumple criterios o enriquecerla con criterios de aceptación), podría realizar múltiples llamadas de modelo o post-procesar con funciones Python. PydanticAI facilita este flujo, ya que permite definir \*\*herramientas (funciones)\*\* que el LLM puede llamar durante su ejecución, e incluso \*\*tipos estructurados de resultado\*\* que deben cumplirse (\[PydanticAI: A Comprehensive Guide to Building Production-Ready AI Applications \- DEV Community\](https://dev.to/yashddesai/pydanticai-a-comprehensive-guide-to-building-production-ready-ai-applications-20me\#:\~:text=Agents%20are%20the%20primary%20interface,container%20for%20various%20components%2C%20including)). En nuestro caso, podríamos definir un modelo Pydantic para la historia de usuario (con campos como \*título\*, \*descripción\*, \*criterios de aceptación\*) y exigir al agente que devuelva un objeto válido; PydanticAI validará la salida o reintentará si no concuerda con el esquema esperado, garantizando que la respuesta sea \*estructurada\* y conforme a lo solicitado.  
- **5\)** Una vez el agente genera la solución (aquí, el texto de la historia de usuario formateada), la función/tool del servidor retorna esa respuesta estructurada. FastAPI serializa la respuesta (por JSON-RPC) de vuelta hacia el cliente MCP.  
- **6\)** El cliente IDE recibe la respuesta. Si es un resultado estructurado (ej. campos JSON), el cliente puede formatearlo amigablemente (por ejemplo mostrar la historia de usuario con campos identificados). En la mayoría de casos, el IDE simplemente inserta o muestra el texto formateado en su interfaz (p. ej., en un panel lateral o directamente en un archivo, dependiendo de la acción). Para el usuario, el resultado es que la historia de usuario completa aparece en segundos, generada conforme a estándares.

Este flujo completo ocurre de forma transparente y **sin intervención manual del desarrollador más allá de la solicitud inicial**, ejemplificando un flujo agéntico de alto nivel. A diferencia de un simple plugin de IA en el IDE que genere texto de un prompt directo, aquí el servidor aplica lógica adicional (plantillas, consultas, validaciones) antes de usar el LLM, resultando en respuestas más pertinentes y con governance controlada por la empresa.

## Implementación con FastAPI y PydanticAI

La elección de **FastAPI** y **PydanticAI** como base tecnológica está orientada a maximizar la productividad y robustez del desarrollo: PydanticAI es descrito como un *framework* “diseñado para que construir aplicaciones de IA generativa de grado producción sea menos doloroso, aportando al mundo de IA el diseño ergonómico e innovador de FastAPI” ([PydanticAI: A Comprehensive Guide to Building Production-Ready AI Applications \- DEV Community](https://dev.to/yashddesai/pydanticai-a-comprehensive-guide-to-building-production-ready-ai-applications-20me#:~:text=PydanticAI%20is%20a%20powerful%20Python,integration%20with%20other%20Python%20tools)). Esto significa que PydanticAI adopta principios de *type-safety*, modularidad y reutilización similares a FastAPI, pero aplicados a agentes de IA. Sus conceptos clave – **Agents, Tools, System Prompts, Structured Results, Dependencies** ([PydanticAI: A Comprehensive Guide to Building Production-Ready AI Applications \- DEV Community](https://dev.to/yashddesai/pydanticai-a-comprehensive-guide-to-building-production-ready-ai-applications-20me#:~:text=Agents%20are%20the%20primary%20interface,container%20for%20various%20components%2C%20including)) – encajan perfectamente con la lógica de un servidor MCP agéntico.

**Agentes PydanticAI:** En RaiSE definiremos uno o varios **`Agent`** de PydanticAI, que servirán para distintas categorías de tareas. Por ejemplo, un agente especializado en **historias de usuario** tendrá:

- Un **prompt de sistema** que incluye las instrucciones de estilo y formato de historia de usuario (“Usa el formato: Como \[rol\], quiero..., etc. siguiendo las guías de UX de la empresa XYZ…”). Este prompt puede ser estático o generado dinámicamente a partir de contexto (p. ej., adaptando vocabulario según el proyecto actual, obtenido de la descripción del repositorio).  
- Posibles **herramientas funcionales** disponibles durante la ejecución del agente. En el caso de generar historias de usuario, quizás no sean necesarias herramientas adicionales (basta el LLM con el prompt). En otros agentes, podrían incluirse funciones como `search_jira(query)` o `get_repo_info()` para que el LLM las invoque si necesita datos. PydanticAI permite registrar funciones Python como *tools* que el LLM puede llamar dentro de la conversación (similar al mecanismo de *function calling* de OpenAI).  
- Un **modelo estructurado de resultado** (ejemplo: una clase Pydantic `UserStory` con campos `titulo`, `descripcion`, `criterios_aceptacion`). Esto obliga al LLM a retornar información bien estructurada que luego Pydantic valida. Si la respuesta del modelo no se ajusta, PydanticAI puede detectar el error y reintentar o corregir. Así garantizamos, por ejemplo, que siempre haya un título y al menos un criterio de aceptación.  
- La configuración del **modelo de lenguaje** a usar. En PydanticAI, al instanciar un Agent se puede especificar qué LLM usar en cada ejecución. Aquí podríamos por defecto asociar un modelo open-source (p.ej., `openai:gpt-3.5-turbo` si se tiene local o un `llama2:7b-chat` local) para texto general, pero permitiendo override. Incluso podríamos cambiar dinámicamente: si el agente detecta que la tarea involucra código (mediante un parámetro o analizando la solicitud), podría conmutar a usar Anthropic Sonnet para esa ejecución – aunque más limpio sería tener otro agente/herramienta para código.

**Exposición vía FastAPI/MCP:** Con FastAPI, exponer estas capacidades es sencillo. Existen utilidades comunitarias que integran FastAPI con MCP de forma automática, como `fastapi_mcp` o `fastmcp`, que montan un servidor MCP a partir de endpoints FastAPI con mínimo *boilerplate* ([tadata-org/fastapi\_mcp: A zero-configuration tool for ... \- GitHub](https://github.com/tadata-org/fastapi_mcp#:~:text=GitHub%20github,app%20and%20it%20works)). En nuestro caso, podemos definir directamente *endpoints* o funciones con decoradores MCP: por ejemplo, usando la clase `FastMCP` del SDK oficial:

```py
from mcp.server.fastmcp import FastMCP
from pydantic_ai import Agent

mcp_server = FastMCP("RaiSE Server")

# Configurar agente de historia de usuario
user_story_agent = Agent(model="openllm:llama2-13b-chat", system_prompt="...guía de estilo...")
@mcp_server.tool()
def generate_user_story(req: str) -> dict:
    """Genera una historia de usuario a partir de una descripción dada."""
    result = user_story_agent.run(f"Redacta una historia de usuario para: {req}")
    return result.data  # .data contendrá el dict/objeto estructurado resultado
```

En este esquema, el decorador `@mcp_server.tool()` registra la función como herramienta MCP. La docstring sirve como descripción que los clientes MCP pueden usar para descubrirla (por ejemplo, el IDE puede listar “Genera una historia de usuario a partir de una descripción dada.” como capacidad disponible) ([Building an MCP Server \- by Avi Chawla](https://blog.dailydoseofds.com/p/building-an-mcp-server#:~:text=Say%20through%20our%20MCP%20server%2C,s%20in%20a%20string)). Dentro, usamos `user_story_agent.run(...)` para ejecutar el agente PydanticAI con el prompt adecuado. El agente internamente llamará al LLM configurado y retornará un objeto (por ejemplo, instancia de `UserStory` o simplemente texto estructurado) accesible vía `result.data` ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=%40server,theme%7D%27%29%20return%20r.data)).

**Nota:** El ejemplo anterior es ilustrativo; en implementación real usaríamos `async def` y posiblemente *dependencies* de FastAPI/PydanticAI para inyectar configuración. Además, podríamos manejar errores (si el LLM falla) y tiempos de espera para evitar que una petición bloquee demasiado.

**Múltiples herramientas y flujos:** Podemos seguir el patrón anterior para cada funcionalidad agéntica deseada. Por ejemplo, otra herramienta `@mcp_server.tool()` podría ser `open_jira_issue(title: str, description: str)` que crea un ticket en Jira. Internamente esta función podría invocar un flujo diferente: quizás primero usar el agente de historia de usuario para formatear el título/desc, luego llamar a la API de Jira (sea directamente con `requests` o delegando a n8n). Cada herramienta expuesta se documenta con su docstring, y el cliente MCP (como Cursor) puede sugerir al asistente LLM su uso cuando el usuario lo requiera ([Building an MCP Server \- by Avi Chawla](https://blog.dailydoseofds.com/p/building-an-mcp-server#:~:text=As%20shown%20in%20the%20video,s%20in%20strawberry)). Alternativamente, ciertas herramientas no serán llamadas directamente por el LLM del cliente sino por órdenes explícitas (como cuando el usuario selecciona una acción de menú en la UI, que llama esa herramienta). RaiSE manejará ambas situaciones.

**Formato de respuesta:** Dado que el protocolo MCP soporta respuestas estructuradas vía JSON-RPC, podemos aprovecharlo para retornar objetos JSON con campos bien definidos ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=,AI%E2%80%99s%20behavior%20in%20specific%20scenarios)). Por ejemplo, `generate_user_story` podría devolver un dict JSON con keys `titulo`, `como`, `quiero`, `para`, etc. Si el cliente es capaz de formatearlo, genial; si no, al menos el asistente LLM podría traducirlo a texto legible. En cualquier caso, gracias a PydanticAI mantenemos consistencia en la respuesta.

**Estado y sesión:** MCP siendo un protocolo con sesiones permite mantener cierto estado contexto en la interacción ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=The%20protocol%20is%20based%20on,and%20error%20reporting%20enhancing%20reliability)). RaiSE podría aprovechar esto para *cachear* contexto de una petición a otra (ej: recordar el último requerimiento que el usuario proporcionó para la historia de usuario), aunque en general mantendremos el servidor *stateless* entre peticiones a menos que un caso requiera lo contrario, para simplicidad y escalabilidad.

## Flujo de ejecución representativo

A continuación describimos un flujo concreto de extremo a extremo para el caso de uso principal (generación de una historia de usuario), integrando los componentes mencionados:

1. **Invocación desde el IDE:** El desarrollador elige la opción “Generar historia de usuario” en Cursor IDE (por ejemplo, mediante un comando especial en modo agente). El IDE reúne el contexto necesario – posiblemente el texto de requisitos seleccionados o una descripción proporcionada por el usuario – y envía una solicitud MCP al servidor RaiSE. Este llamado podría ser al *tool* `generate_user_story` con un parámetro que contiene la descripción del requisito.  
     
2. **Recepción en servidor y selección de plantilla:** En RaiSE, la función `generate_user_story` es ejecutada. Supongamos que incluye lógica adicional antes de invocar al LLM: por ejemplo, elegir una plantilla específica según el tipo de proyecto. Podríamos tener varias plantillas (p. ej. para frontend vs backend, o con formato Gherkin si se usa BDD). Esta lógica se puede implementar de manera sencilla: quizás `generate_user_story` examina la descripción recibida o algún flag y elige un *prompt* de sistema adecuado para el agente (o incluso selecciona cuál agente usar si definimos agentes distintos por tipo de historia). También podría consultar una base de conocimiento: por ejemplo, un recurso MCP `company://story_guidelines` que devuelve las directrices de redacción, asegurando siempre estar actualizado.  
     
3. **Llamada al LLM (agente en acción):** Con la plantilla/prompt decidida, `generate_user_story` utiliza el agente PydanticAI configurado. El *prompt final* entregado al modelo incluirá: la plantilla base, el contexto del requerimiento específico, y cualquier instrucción adicional (por ejemplo “asegúrate de incluir 3 criterios de aceptación en formato bullet”). El modelo de lenguaje open-source asignado procesa la solicitud y genera un borrador de la historia de usuario. Supongamos que la salida del modelo es un bloque de texto estructurado con secciones. PydanticAI intentará parsearlo contra el modelo de datos `UserStory`. Si falta algo (digamos no detecta criterios de aceptación porque el modelo no los listó explícitamente), el agente podría tener lógica para reintentar pidiendo específicamente “Agregar criterios de aceptación”. Todo esto ocurre rápidamente en memoria.  
     
4. **Post-procesamiento y validación:** Tras obtener el resultado del LLM, RaiSE realiza validaciones finales. Por ejemplo, correr una verificación sencilla de QA: longitud dentro de límites (ni demasiado corta ni excesivamente larga), lenguaje apropiado (sin palabras prohibidas), etc. De no cumplir, se podría ajustar (p. ej. truncar texto demasiado largo) o en casos extremos, retornar un error manejado al cliente. Este es un punto donde entra en juego la **gobernanza**: se aplican reglas corporativas a las salidas del modelo antes de exponerlas al usuario.  
     
5. **Respuesta al cliente:** La función devuelve la historia de usuario final (estructurada). El servidor MCP envía la respuesta de vuelta al cliente mediante JSON-RPC. Gracias a la conexión establecida, el IDE la recibe casi inmediatamente. Si usamos **Server-Sent Events (SSE)** para el transporte (opción recomendada para streaming ([Building a Server-Sent Events (SSE) MCP Server with FastAPI \- Ragie](https://www.ragie.ai/blog/building-a-server-sent-events-sse-mcp-server-with-fastapi#:~:text=Building%20a%20Server,However%2C%20due%20to%20limited))), podríamos incluso proveer retroalimentación parcial o tokens a medida que se generan (por ejemplo, mostrando el texto de la historia progresivamente). Para nuestra necesidad, probablemente la generación es lo bastante rápida para mandar el resultado completo.  
     
6. **Presentación en IDE:** El IDE inserta la historia generada en el contexto adecuado. Tal vez abre un editor con el texto prellenado, listo para que el desarrollador lo revise y edite. Como la respuesta venía estructurada (ej. JSON), el cliente podría aplicarle un formato enriquecido (campos con negrita para “Como ..., quiero ..., para ...”), facilitando la lectura. El desarrollador ve una historia de usuario bien formada que puede ajustar o aceptar tal cual.

Este flujo ejemplifica cómo **todo el razonamiento y orquestación se realizaron en el servidor** (prompting, selección de herramientas/modelos, validaciones), mientras el cliente solo actuó como invocador y presentador. Esto tiene ventajas claras: la lógica puede evolucionar en el servidor sin necesitar actualizar el cliente; además, los datos sensibles (p.ej. estándares internos) nunca se envían al modelo del cliente ni a servicios externos sin pasar por controles en RaiSE.

Otro flujo representativo podría ser **creación de un ticket en Jira a partir de contexto de código**: el dev escribe en el chat del IDE “Abre un ticket de bug con la traza de error actual”. El agente del IDE identifica que debe usar una herramienta MCP (quizá `create_jira_issue`) expuesta por RaiSE. El IDE envía la petición con los detalles (título sugerido, descripción con la traza). RaiSE recibe, el agente quizá formatea mejor la descripción, el conector Jira (sea código directo o vía n8n) crea el ticket y devuelve el ID/URL. RaiSE responde con “Ticket CRE-1234 creado exitosamente”. Este tipo de interacciones combinan LLM \+ acción sobre APIs, y RaiSE puede manejarlas de forma muy similar al ejemplo anterior, simplemente incorporando el paso de integración externa.

## Evaluación del uso de n8n en la arquitectura

**n8n** es una plataforma de automatización de flujos de trabajo (similar a Zapier/Make) que permite orquestar integraciones entre servicios con bajo código. En nuestra arquitectura, contemplamos usar n8n *exclusivamente* como proveedor de conectores DevSecOps, es decir, para realizar las integraciones con herramientas como Jira, GitHub, GitLab, Slack, etc., pero **no** para manejar la lógica principal de IA ni la orquestación interna del agente. A continuación, se discuten ventajas y desventajas de esta inclusión:

- **Ventajas de incluir n8n:**  
    
  - *Aceleración de integraciones:* n8n ofrece nodos preconstruidos para decenas de servicios DevOps/SecOps. En lugar de codificar manualmente la autenticación OAuth, endpoints REST, manejo de paginación, etc., podríamos configurar flujos n8n que realizan esas tareas (p.ej., un flujo "Crear issue en Jira" que recibe campos y usa el nodo Jira para crear el ticket). Esto reduce el tiempo de desarrollo para soportar múltiples herramientas externas.  
  - *Mantenibilidad y reutilización:* Los flujos en n8n son visuales y configurativos; un analista podría ajustarlos sin profundizar en el código de RaiSE. Por ejemplo, si mañana cambia la API de Jira o se desea agregar campos personalizados, se modifica el flujo en n8n en lugar de desplegar una nueva versión del servidor MCP. Además, los mismos flujos podrían reutilizarse fuera de RaiSE si se desea (por ejemplo, en otras automatizaciones corporativas).  
  - *Seguridad de credenciales:* n8n maneja credenciales de conexión mediante un sistema propio (credenciales almacenadas y referenciadas en nodos). Al delegar integración a n8n, RaiSE no necesita almacenar tokens sensibles; simplemente llama al *webhook* de n8n asociado al flujo, pasando datos mínimos. Esto podría simplificar el compliance, ya que las credenciales residen en un solo lugar (el servidor de n8n) con sus mecanismos de cifrado.


- **Desventajas de incluir n8n:**  
    
  - *Complejidad adicional:* Introducir n8n significa un componente extra desplegado (otro contenedor/servicio) y una capa adicional en el flujo de llamada. Cada petición desde RaiSE a n8n añade latencia (aunque normalmente es pequeña, unos cientos de milisegundos por llamada HTTP externa). Además, hay que mantener la instancia de n8n, vigilar su disponibilidad, actualizaciones, etc.  
  - *Doble gestión de lógica:* Si bien n8n simplifica la integración, también fragmenta la solución: parte de la lógica de negocio (por mínima que sea dentro de los flujos) vive fuera del código principal. Esto puede dificultar depurar end-to-end, ya que un error en un conector se ve en n8n y no directamente en logs de RaiSE. Se requeriría coordinar cambios en ambos lados y tener buena documentación de estos flujos.  
  - *Escalabilidad y rendimiento:* n8n, al ser general, puede no ser tan eficiente como un código Python dedicado para tareas muy intensivas. Si imaginamos un caso de alta frecuencia (ej: abrir 50 issues en lote), n8n podría ser un cuello de botella comparado con llamadas directas con una librería optimizada. Sin embargo, para el uso interactivo puntual que tendríamos (acciones disparadas por usuario), esto no sería crítico.  
  - *Consideraciones de seguridad adicionales:* Aunque n8n manejaría credenciales, ahora hay que asegurar que la comunicación entre RaiSE y n8n es segura (idealmente interna en la red, no expuesta pública). También confiar en que n8n esté correctamente configurado (p. ej., flujos no accesibles sin autorización). Es agregar una superficie más a asegurar.

**Recomendación:** Incluir n8n **solo si** el equipo busca rapidez de integración con muchas herramientas y acepta la complejidad añadida. En un MVP, se podría **no usar n8n inicialmente** y codificar directamente 1-2 integraciones clave (por ejemplo, Jira y GitHub) usando sus SDKs o APIs REST dentro de RaiSE, para reducir dependencias. Una vez validada la utilidad, si surge la necesidad de ampliar a más servicios rápidamente, n8n puede incorporarse. Alternativamente, se puede diseñar RaiSE con una interfaz de integración abstracta: que llame a “conector Jira” sin importar si debajo es una función Python o un webhook n8n. Así podríamos cambiar la implementación sin impactar al agente. En términos de arquitectura extensible, esta abstracción es beneficiosa.

En cualquier caso, si se usa n8n, se mantendrá la **lógica principal en RaiSE**: es RaiSE quien decide *cuándo* llamar a un flujo n8n y cómo manejar la respuesta. n8n no iniciará procesos por sí solo (no hay triggers autónomos) sino que actuará bajo demanda. Esto preserva el control agéntico centralizado y evita solapamientos: MCP y n8n se complementan, no compiten (recordemos: MCP permite al *AI assistant* hablar con herramientas directamente ([How is MCP different from Zapier/Make/n8n? : r/ClaudeAI \- Reddit](https://www.reddit.com/r/ClaudeAI/comments/1hdarn1/how_is_mcp_different_from_zapiermaken8n/#:~:text=How%20is%20MCP%20different%20from,to%20data%20sources%20and%20tools)), mientras n8n usualmente encadena eventos; aquí lo usamos solo como librería remota de integraciones).

## Stack tecnológico propuesto y patrón de desarrollo

A partir de lo anterior, resumimos el **stack** tecnológico y algunas prácticas de desarrollo para implementar RaiSE:

- **Lenguaje y Framework web:** Python 3.11+ con FastAPI. Este combo ofrece rapidez de desarrollo, soporte asíncrono (importante para llamadas concurrentes a LLMs o APIs sin bloquear) y fácil despliegue. FastAPI también aporta validación de datos (vía Pydantic) para las entradas/salidas HTTP, complementando la validación interna de PydanticAI.  
    
- **Framework de agentes:** PydanticAI, aprovechando su integración nativa con Pydantic y su filosofía declarativa (similar a FastAPI en cuanto a definir componentes claramente). Esto permite definir agentes con pocos pasos y asegurando consistencia tipada en las respuestas ([PydanticAI: A Comprehensive Guide to Building Production-Ready AI Applications \- DEV Community](https://dev.to/yashddesai/pydanticai-a-comprehensive-guide-to-building-production-ready-ai-applications-20me#:~:text=PydanticAI%20is%20a%20powerful%20Python,integration%20with%20other%20Python%20tools)). Adicionalmente, PydanticAI tiene soporte integrado para múltiples proveedores de LLM (OpenAI, Anthropic, Bedrock, Cohere, etc.) ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=,pydantic_ai.models.test)), lo que alinea con la necesidad de usar tanto un modelo Anthropic (Sonnet) como potencialmente otros modelos open-source.  
    
- **Modelo(s) de lenguaje:**  
    
  - Para **open-source**: Se evaluará algún modelo de la familia LLaMA 2 o similar, posiblemente fine-tuneado para español o para el dominio de historias de usuario si se encuentra disponible. Una opción podría ser **Llama-2 13B Chat** desplegado en un servidor interno usando libs como HuggingFace Transformers o llamando a un endpoint de inferencia (por ejemplo, via `text-generation-inference` server or an API like OpenRouter if it hosts open models). PydanticAI puede envolver un modelo local mediante su clase `OpenAIModel` apuntando a un endpoint compatible, o mediante un *provider* específico si lo hay. Otra opción ligera sería **GPT4All** variants or **Mistral 7B** for quick responses, though quality might be lower. Se hará pruebas de calidad para escoger un modelo que genere texto coherente en español y entienda instrucciones.  
  - Para **modelo de código (Sonnet 3.7)**: Se usará la API de Anthropic Claude (Claude Instant/Claude 2). Dado que PydanticAI ya en el ejemplo usa `anthropic:claude-3-5-haiku-latest` ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=server_agent%20%3D%20Agent%28%20%27anthropic%3Aclaude,reply%20in%20rhyme%27)), podremos configurar un agente con `anthropic:claude-3-7-sonnet-latest` (si la nomenclatura coincide con su SDK) o integrar vía el `anthropic` provider by specifying the model ID for Sonnet. Esto requerirá la clave de API de Anthropic y cumplir sus requisitos de uso. Sonnet 3.7 aportará sus *capacidades avanzadas de generación de código y razonamiento* ([Claude 3.7 Sonnet \- Anthropic](https://www.anthropic.com/claude/sonnet#:~:text=Claude%203,taste%20while%20maintaining%20context)) cuando RaiSE lo necesite (p. ej., si implementamos en el futuro una herramienta "refactor code" o "analiza este error").


- **Contenedorización:** Se creará un Dockerfile basado en una imagen ligera (por ejemplo, `python:3.11-slim`). El contenedor incluirá la aplicación FastAPI (posiblemente iniciada con Uvicorn/Gunicorn for production). Si se usan modelos locales grandes, podría haber un trade-off: incluirlos en la imagen (aumentando su tamaño significativamente) o montarlos en volumen aparte. Quizá la mejor práctica es mantener RaiSE independiente y consumir modelos alojados en otro servicio or container (microservicio de modelo). Sin embargo, para despliegue sencillo self-hosted, podríamos permitir un modo opcional donde el modelo open-source se carga en memoria dentro de RaiSE (asumiendo la máquina tiene la RAM/GPU necesaria). En cualquier caso, **desplegar en cualquier entorno Docker** es viable dado que todos los componentes (FastAPI, PydanticAI, libs) son Python puro con dependencias manejables. Para reproducibilidad, se versionarán los requirements (congelando versiones de PydanticAI, etc., dado lo rápido que evolucionan).  
    
- **SDK MCP / implementación protocolo:** En lugar de implementar manualmente JSON-RPC, usaremos componentes existentes. Por ejemplo, **FastMCP** (proyecto open-source) facilita exponer un servidor MCP completo con mínimos pasos ([GitHub \- jlowin/fastmcp: The fast, Pythonic way to build Model Context Protocol servers](https://github.com/jlowin/fastmcp#:~:text=Let%27s%20create%20a%20simple%20MCP,calculator%20tool%20and%20some%20data) ) ([GitHub \- jlowin/fastmcp: The fast, Pythonic way to build Model Context Protocol servers](https://github.com/jlowin/fastmcp#:~:text=%40mcp,return%20a%20%2B%20b) ). También el módulo `mcp` oficial (como en el snippet de PydanticAI docs) provee clases para iniciar la servidor (FastMCP) y registrar herramientas ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=from%20mcp)) ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=%40server,theme%7D%27%29%20return%20r.data)). Adoptar estas bibliotecas acelera el desarrollo (*time-to-market*) y asegura cumplimiento del estándar MCP v0.x sin profundizar en su especificación de bajo nivel. Asimismo, nos beneficiaremos de funcionalidades ya incluidas como: negociación de capacidades, manejo de llamadas concurrentes, cancelación de peticiones, etc., que el protocolo soporta ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=The%20protocol%20is%20based%20on,and%20error%20reporting%20enhancing%20reliability)).  
    
- **Observabilidad:** Para un producto de nivel empresarial es crucial contar con **observabilidad** de las operaciones del agente. Proponemos integrar:  
    
  - **Logging estructurado** de las interacciones: Cada petición MCP que llegue se logueará con un ID, tipo de acción solicitada, y métricas como duración del proceso y modelo usado. Las salidas de LLM (o partes relevantes) también podrían registrarse en nivel debug para diagnosticar problemas de calidad. Utilizaremos la capacidad de *logging* de FastAPI/Uvicorn combinado con logs internos del agente. PydanticAI podría tener integraciones con OpenTelemetry (observamos algo de `pydantic_evals.otel` en sus módulos ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=,68))), lo que permitiría trazar cada ejecución de agente como span, incluso correlacionando sub-llamadas a LLMs y herramientas. De ser posible, habilitaremos esa telemetría para tener visibilidad fina (p. ej., en Jaeger o Grafana Tempo).  
  - **Monitoreo de métricas**: Podríamos exponer métricas Prometheus (e.g., conteo de peticiones por tipo de herramienta, latencia promedio, uso de distintos modelos, etc.). FastAPI con Starlette permite añadir middlewares de métricas fácilmente, o usar libs como `prometheus-fastapi-instrumentator`. Esto ayuda a detectar cuellos de botella (p. ej., si Sonnet está tardando demasiado) y patrones de uso.  
  - **Herramientas de depuración de agentes**: Durante desarrollo, nos apoyaremos en utilidades como *MCP Inspector* (una herramienta para probar servidores MCP de forma aislada ([GitHub \- jlowin/fastmcp: The fast, Pythonic way to build Model Context Protocol servers](https://github.com/jlowin/fastmcp#:~:text=fastmcp%20install%20server) )) o en simulaciones con un cliente Python MCP ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=from%20mcp%20import%20ClientSession%2C%20StdioServerParameters,stdio%20import%20stdio_client)) ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=await%20session,our%20toes%20from%20feeling%20dread)). Esto permite testear cómo el agente responde a distintos inputs fuera del IDE, acelerando la iteración. Podríamos incluir en el repositorio scripts de prueba para cada herramienta (integrando quizás en CI/CD pruebas automatizadas de las salidas del agente dado ciertos inputs, usando *snapshots* esperados).


- **Patrón de desarrollo modular:** La arquitectura interna del código seguirá principios de **responsabilidad única** y modularidad:  
    
  - Separaremos la definición de **agentes** (prompts, lógica) de la capa de **exposición FastAPI/MCP**. Por ejemplo, podríamos tener un módulo `agents/user_story_agent.py` donde se instancia el Agent PydanticAI y quizás funciones auxiliares, y otro módulo `api/tools.py` donde se registran las herramientas `generate_user_story` etc., invocando al agente. Así, si en un futuro quisiéramos usar el agente de forma distinta (ej: en una CLI), podríamos sin dificultad.  
  - Centralizaremos configuraciones en un archivo de **settings** (posiblemente usando Pydantic Settings management). Ahí definiremos: URLs o claves de APIs externas (Jira, etc.), elección de modelos (p. ej. switch para usar un modelo open-source distinto), *feature flags* como activar/desactivar n8n, etc. Esto facilita gobernanza porque un admin puede cambiar comportamientos sin tocar múltiples lugares en el código.  
  - Aplicaremos principios de **Clean Architecture** en la medida de lo posible: las herramientas/integraciones externas (p. ej., funciones que llaman a Jira API) se tratarán como infraestructura, separadas de la lógica de orquestación del agente (dominio). PydanticAI ya nos guía en eso al tener *dependencies* que se pueden inyectar; podríamos usarlo para proveer al agente, por ejemplo, un cliente de Jira ya autenticado como dependencia, en lugar de llamar directamente dentro de la lógica. Esto mejora testabilidad (podemos mockear ese cliente en tests).


- **Governanza y control de versiones:** Cada “flujo agéntico” (por ej., generación de historia de usuario) puede versionarse en cuanto a prompt y comportamiento. Se recomienda mantener un registro versionado de las plantillas de prompt o guías (incluso almacenándolas en Git). Si en el tiempo se ajustan los estándares de historias, podríamos implementar un mecanismo para que el agente use la versión apropiada según contexto o configuración. Asimismo, para gobernanza, podríamos integrar un sistema de **aprobación humana** opcional para ciertas acciones sensibles: por ejemplo, antes de que el agente realice una acción irreversible (como *merge* código), requerir confirmación explícita. Estas políticas se configurarían en RaiSE.

## Seguridad, extensibilidad y gobernanza

**Seguridad:** Siendo una herramienta que potencialmente accede a información sensible (código fuente, descripciones de requisitos, credenciales de APIs), RaiSE debe incorporar seguridad en varios niveles:

- *Comunicación:* El servidor MCP debería ejecutarse bajo HTTPS si está desplegado remotamente, o al menos en una red privada/VPN si es on-premise, para evitar interceptación de tráfico. JSON-RPC sobre HTTP puede cifrarse con TLS como cualquier API. Además, podríamos requerir autenticación del cliente MCP – por ejemplo, un token API compartido que el IDE de confianza envía en headers – para evitar que clientes no autorizados accedan a las herramientas (esto es relevante si se despliega en un servidor central al que se conectan múltiples IDEs de usuarios; si es local no tanto).  
- *Autorización y aislamiento:* Cada herramienta expuesta puede conllevar permisos. Por ejemplo, quizá no todos los desarrolladores deben poder activar cierta automatización crítica. Podríamos integrar un simple control de roles: p.ej., RaiSE verifica la identidad del solicitante (si el IDE le pasa un ID de usuario) antes de realizar una acción en ciertos sistemas (esto requeriría extender el protocolo o usar múltiples endpoints HTTP con auth, ya que MCP en sí no define roles). A nivel interno, las herramientas solo acceden a lo necesario: por diseño, un servidor MCP delimita sus recursos claramente ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=MCP%20servers%20are%20designed%20to,boundaries%20to%20prevent%20unauthorized%20access)). RaiSE sólo tendrá acceso a aquello que programemos (p. ej., llamadas a Jira con un usuario técnico específico). Esto limita el alcance de posibles abusos; un exploit en el agente difícilmente podría leer archivos arbitrarios a menos que expongamos un recurso para ello.  
- *Manejo de datos sensibles:* Cuando se llame a LLMs externos (Anthropic Sonnet), se deben filtrar o anonimizar datos sensibles en los prompts en la medida de lo posible, dado que pasan a un servicio externo. Por ejemplo, evitar enviar código propietario completo a la nube; quizá sólo resúmenes o porciones necesarios para contexto. Además, respetar las políticas de uso de ese servicio (p. ej. Anthropic tiene cláusulas de no enviar ciertas categorías de datos). Por el lado del cliente, educar a los usuarios a no pedir al agente acciones que expongan secretos. En este aspecto, incluir **filtros de contenido** automáticos en RaiSE es aconsejable: PydanticAI/Anthropic suelen tener sistemas de moderación; RaiSE podría detectar peticiones peligrosas (como “borra todo el repo”) y vetarlas o requerir confirmación.  
- *Actualizaciones y parches:* Mantener actualizadas las dependencias (FastAPI, PydanticAI) para incorporar parches de seguridad es parte de la gobernanza técnica. Se podría establecer un ciclo de revisión periódica y pruebas regresivas antes de actualizar versiones, dado que PydanticAI es relativamente nuevo y evoluciona rápido.

**Extensibilidad:** RaiSE está concebido para ser **modular y extensible**. Algunas consideraciones:

- *Añadir nuevas herramientas/agentes:* Gracias a la estructura basada en FastAPI y decoradores MCP, sumar una nueva capacidad es tan sencillo como escribir una nueva función, decorar con `@mcp_server.tool()` y registrarla. Mientras la docstring esté clara, los clientes podrán descubrirla. Internamente, podemos desarrollar nuevos agentes PydanticAI for diferentes casos: por ej., un agente de **documentación de código** que dado un archivo produce documentación en formato específico, o un agente de **traducción** de textos. Cada uno con su prompt y modelo (quizá un modelo especializado en traducción). La arquitectura soporta cargar múltiples agentes en paralelo. De ser necesario, también se pueden exponer **Prompts predefinidos** vía MCP (la especificación permite exponer plantillas reutilizables ([What Are MCP Servers? Clearly Explained](https://apidog.com/blog/mcp-servers-explained/#:~:text=commands,AI%E2%80%99s%20behavior%20in%20specific%20scenarios))), aunque hasta ahora nos centramos en Tools que encapsulan todo.  
- *Sustituir modelos o proveedores:* La abstracción de PydanticAI sobre LLMs permite cambiar el modelo con mínima fricción. Si en el futuro aparece un modelo open-source mejor que el actual, podríamos configurarlo en el agente sin cambiar la lógica. Incluso podríamos hacer *switch* dinámico según disponibilidad (ej: usar un modelo más chico si el servidor tiene poca RAM, etc., configurable). Lo mismo con integraciones: si se decide reemplazar n8n por otro sistema, al estar encapsulado en la capa de integración, no afecta la orquestación central.  
- *Escalabilidad horizontal:* FastAPI permite escalar replicando instancias detrás de un balanceador. Dado que RaiSE podría intensivamente usar CPU/GPU (por los modelos), escalar horizontalmente es válido: múltiples instancias podrían compartir una cola de requests (usando a distributed RPC mechanism or simply multiple API pods with load balancing). Habrá que asegurarse que el estado de agentes no sea crítico de compartir (preferiblemente stateless). Para datos comunes (p. ej., cache de plantillas o resultados recientes), podríamos usar un almacén en memoria distribuido o base de datos compartida. PydanticAI Agents siendo reutilizables, podríamos inicializarlos en cada instancia al arranque, listos para servir peticiones.

**Gobernanza:** Por último, la gobernanza se refiere a cómo controlamos la evolución y uso adecuado del sistema:

- *Políticas de uso:* Definir claramente qué puede y qué no puede hacer el agente. Por ejemplo, quizás limitar que ciertas acciones (borrar recursos, modificar infraestructura) no sean automatizadas por seguridad. Estas políticas se implementan en RaiSE a nivel de qué herramientas exponemos y con qué permisividad.  
- *Registro y auditoría:* Mantener logs auditables de qué acciones tomó la IA, cuándo y a petición de quién, es importante en entorno corporativo. RaiSE deberá loguear cada operación (ej: “Usuario U solicitó generar historia, se usó modelo X, resultado entregado…”, “Usuario V creó ticket Jira PROJ-123”). Estos registros pueden almacenarse en un sistema central de logs seguro. Ante cualquier comportamiento inesperado, se podrá auditar la secuencia.  
- *Feedback loop:* Permitir que los desarrolladores califiquen o corrijan las salidas del agente y alimentar esas observaciones de vuelta al equipo de IA. Esto no es necesariamente online learning, pero sí gobernanza de calidad: por ejemplo, si varias personas reportan que las historias de usuario son muy genéricas, el equipo puede decidir ajustar el prompt base. Versionar y comunicar esos cambios (quizá incluso tener feature flags para activarlos progresivamente) es parte de gobernanza del producto de IA. PydanticAI facilita evaluación con sus módulos de *evals* y reportes ([Server \- PydanticAI](https://ai.pydantic.dev/mcp/server/#:~:text=,pydantic_evals.generation)), que podríamos usar para medir aciertos.

En conclusión, la arquitectura propuesta de **RaiSE** combina la potencia de un servidor MCP (extensible, estándar) con la flexibilidad de agentes PydanticAI y la solidez de FastAPI. Este diseño permitirá ofrecer a los desarrolladores una experiencia de asistencia inteligente integrada en sus IDEs, para tareas que van desde la generación de historias de usuario hasta potencialmente otras actividades de DevSecOps, manteniendo a la vez controles empresariales sobre cómo opera la IA. La utilización cuidadosa de componentes existentes (ej. *fastmcp* para MCP, conectores n8n o librerías para APIs, modelos pre-entrenados open source) acelerará el desarrollo sin comprometer calidad, posicionando a RaiSE como un backend inteligente de rápida evolución y alto valor para la productividad en el desarrollo de software.
