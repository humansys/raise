# raise-server

Backend REST API for the RaiSE shared knowledge graph. Provides persistent storage for graph data, agent memory, and telemetry across all clients of an organization.

## Prerequisites

- Docker (with your user in the `docker` group)
- uv
- Python 3.12+

> **First time with Docker?** Add your user to the docker group and activate it:
> ```bash
> sudo usermod -aG docker $USER
> newgrp docker
> ```

---

## Local Setup

All commands run from the **repo root** (`raise-commons/`) unless stated otherwise.

### 1. Start postgres + server

```bash
docker compose up --build
```

Starts:
- `postgres:16-alpine` on port `5432` (waits for healthy before server starts)
- `raise-server` on port `8000`

The postgres data persists in a Docker volume (`pgdata`). To reset from scratch: `docker compose down -v`.

### 2. Run migrations (first time, or after `down -v`)

```bash
cd packages/raise-server
RAI_DATABASE_URL=postgresql+asyncpg://rai:rai_dev@localhost:5432/rai uv run alembic upgrade head
```

Creates tables: `organizations`, `api_keys`, `graph_nodes`, `graph_edges`, `agent_events`, `memory_patterns`.

### 3. Seed dev data

From the repo root:

```bash
docker compose exec -T postgres psql -U rai -d rai < packages/raise-server/scripts/seed_dev.sql
```

Creates:
- Org: `RaiSE Dev` (slug: `raise-dev`)
- API key: `rsk_dev_test_key_12345`

### 4. Verify

```bash
curl http://localhost:8000/health
# {"status":"ok","database":"connected","version":"0.1.0"}
```

## Environment Variables

All variables use the `RAI_` prefix (read by pydantic-settings).

| Variable | Required | Default | Description |
|---|---|---|---|
| `RAI_DATABASE_URL` | yes | — | PostgreSQL async URL (`postgresql+asyncpg://...`) |
| `RAI_LOG_LEVEL` | no | `INFO` | Uvicorn log level |
| `RAI_HOST` | no | `0.0.0.0` | Bind host |
| `RAI_PORT` | no | `8000` | Bind port |

In Docker Compose, `RAI_DATABASE_URL` is set automatically. For running outside Docker, set it manually (as shown in step 2).

## Authentication

All endpoints except `/health` require an API key:

```
Authorization: Bearer rsk_<key>
```

The server hashes the raw key with SHA-256 and validates it against the `api_keys` table. The raw key is never stored.

Dev key: `rsk_dev_test_key_12345`

### Memory — `/api/v1/memory`

Learned patterns from agents or CLI, scoped to the org.

#### `GET /api/v1/memory/patterns`

List patterns for the org.

Query params: `limit` (1–100, default 50)

```bash
curl -H "Authorization: Bearer rsk_dev_test_key_12345" http://localhost:8000/api/v1/memory/patterns
```

Response:
```json
{"patterns": [], "count": 0}
```

#### `POST /api/v1/memory/patterns`

Add a learned pattern.

```bash
curl -X POST \
  -H "Authorization: Bearer rsk_dev_test_key_12345" \
  -H "Content-Type: application/json" \
  -d '{"content": "Always run migrations before seed", "context": ["setup"], "properties": {}}' \
  http://localhost:8000/api/v1/memory/patterns
```

Request body:

| Field | Type | Required | Description |
|---|---|---|---|
| `content` | string | yes | Pattern text (1–10000 chars) |
| `context` | string[] | no | Keywords for retrieval |
| `properties` | object | no | Arbitrary metadata |

Response: `{"id": "<uuid>", "status": "ok"}`

### Graph — `/api/v1/graph`

Knowledge graph nodes and edges, scoped to org + project.

#### `POST /api/v1/graph/sync`

Full graph sync for a project. Idempotent upsert — replaces all nodes and edges for the given `project_id`.

Request body:

| Field | Type | Required | Description |
|---|---|---|---|
| `project_id` | string | yes | Repo/project identifier |
| `nodes` | NodeInput[] | yes | Graph nodes |
| `edges` | EdgeInput[] | yes | Graph edges |

`NodeInput` fields: `node_id`, `node_type`, `scope` (default `"project"`), `content`, `source_file` (optional), `properties`.

`EdgeInput` fields: `source_node_id`, `target_node_id`, `edge_type`, `weight` (default 1.0), `properties`.

Response: `{"status": "ok", "project_id": "...", "nodes_upserted": N, "edges_created": N, "edges_skipped": N, "nodes_pruned": N}`

#### `GET /api/v1/graph/query`

Keyword search across the org's knowledge graph.

Query params: `q` (required, min 1 char), `limit` (1–100, default 20)

```bash
curl -H "Authorization: Bearer rsk_dev_test_key_12345" "http://localhost:8000/api/v1/graph/query?q=auth&limit=5"
```

Response: `{"results": [...], "total": N, "query": "auth", "limit": 5}`

---

### Agent Telemetry — `/api/v1/agent`

Append-only event log from agents and CLI actions, scoped to the org.

#### `POST /api/v1/agent/events`

Record a telemetry event. Payload max size: 100 KB.

```bash
curl -X POST \
  -H "Authorization: Bearer rsk_dev_test_key_12345" \
  -H "Content-Type: application/json" \
  -d '{"event_type": "story.started", "payload": {"story_id": "s1.1"}}' \
  http://localhost:8000/api/v1/agent/events
```

Request body:

| Field | Type | Required | Description |
|---|---|---|---|
| `event_type` | string | yes | Event identifier (max 100 chars) |
| `payload` | object | no | Arbitrary event data (max 100 KB) |

Response: `{"id": "<uuid>", "status": "ok"}`

#### `GET /api/v1/agent/events`

List recent events for the org.

Query params: `limit` (1–100, default 20)

```bash
curl -H "Authorization: Bearer rsk_dev_test_key_12345" http://localhost:8000/api/v1/agent/events
```

Response: `{"events": [...], "count": N}`

## Architecture

### Three-package structure

```
raise-commons/
  packages/
    raise-core/     ← domain models, graph logic, memory abstractions
    raise-cli/      ← CLI client (rai command)
    raise-server/   ← this package: REST API + DB persistence
```

`raise-server` depends on `raise-core` for domain models. `raise-cli` calls `raise-server` when a remote backend is configured.

### Multi-tenant model

All data is scoped to an `org_id`. A single raise-server instance serves multiple organizations — each org has its own API keys, graph, memory, and telemetry.

### Auth model

1. Client sends `Authorization: Bearer rsk_<raw_key>`
2. Server computes `SHA-256(raw_key)` and queries `api_keys` table
3. If found and `is_active=true`, returns `OrgContext(org_id, org_name)` injected into the handler
4. Raw key is never stored

### Migrations

Managed with Alembic. Migration files live in `alembic/versions/`:

| Migration | Description |
|---|---|
| `001_initial_schema` | organizations, api_keys, graph_nodes, graph_edges |
| `002_agent_memory` | agent_events, memory_patterns |

To create a new migration:
```bash
cd packages/raise-server
RAI_DATABASE_URL=postgresql+asyncpg://rai:rai_dev@localhost:5432/rai uv run alembic revision --autogenerate -m "description"
```
